● Overview

매치메이킹 — 게임 운영의 가장 정치적인 시스템

매치메이킹은 단순 알고리즘이 아니라 대기·공정성·latency·재미·매출이 부딪히는 정책 결정. SRE는 이 결정의 데이터·도구·관측을 책임진다. F1·F2가 토폴로지·네트워크였다면 F3는 매치 알고리즘과 세션 운영의 깊이를 다룬다. (2026-05 기준)

매치메이킹의 3대 목표 — 영원한 트레이드오프 삼각형

⏱ 짧은 대기

큐 진입 후 빨리 매치 시작. 사용자 체감 1순위.

⚖ 공정성

비슷한 실력끼리. 한쪽이 압도하면 둘 다 재미 없음.

📡 낮은 latency

같은 region·가까운 ping. 멀면 lag.

★ 이 셋은 동시 만족 불가능
대기 줄이려면 풀 확장 → MMR 차이 + 먼 region 허용. 공정성 강화하면 풀 좁아짐 → 대기 폭증. Latency 좁히면 글로벌 풀 작아짐 → 대기 + MMR 격차. 매치메이킹 정책은 이 셋의 비중을 어떻게 잡느냐의 문제.

매치 quality — 어떻게 정의하나

"매치 quality 좋다"는 추상적. 실제 운영에서 측정 가능한 지표로 정의하지 않으면 개선 불가능.

MMR variance

한 매치 안 플레이어 MMR 표준편차. 작을수록 균형. 보통 1 σ 안.

예측 승률 차이

매치 시작 전 양 팀 예측 승률 차. 50:50에 가까울수록 좋음. TrueSkill 같은 알고리즘이 직접 계산.

실제 결과 분포

매치 종료 시 점수차 분포. blowout (예: 5:0) 비율이 높으면 매치 quality 의심.

플레이어 만족도

매치 후 설문·재큐 비율·이탈률. quality 정량 메트릭 + 정성 신호 모두 본다.

퍼블리셔 SRE의 매치메이킹 관여 범위

SRE 책임

매치메이킹 인프라 (Open Match·FlexMatch 운영), 큐 메트릭 수집·대시보드, 서버 fleet 캐파, 매칭 latency, 장애 대응

게임사 책임

매치 알고리즘 룰, MMR 시스템, 역할 큐 정책, 신규 placement, smurf 탐지 룰

공동 (회색지대)

큐 정책 변경 (이벤트 시), MMR 시스템 마이그레이션, abuse 대응, 매치 quality 정의·KPI, SBMM 강도 결정

★ SRE의 핵심 가치
매치메이킹 알고리즘 자체를 SRE가 짜진 않지만, "현재 큐 분포가 어떻게 변화 중인지" "변경 후 매치 quality가 어떻게 변했는지" 데이터를 제공하는 것이 SRE 역할. 이게 없으면 정책 결정이 추측에 의존.
● Algorithms

레이팅 & 매치 알고리즘

ELO에서 출발해 Glicko-2·TrueSkill·OpenSkill까지. 어떤 게임에 어떤 알고리즘이 맞는지, SBMM 논쟁의 양면을 정리한다.

ELO — 모든 레이팅의 출발점

1960년 Arpad Elo가 체스 위해 개발. 두 플레이어 점수 차이 → 예측 승률 → 결과로 점수 조정. 1대1 게임에 표준.

ELO 업데이트 공식
예측 승률 P_A = 1 / (1 + 10^((R_B - R_A) / 400))
새 레이팅 R_A' = R_A + K × (S_A - P_A)

R_A, R_B: 각 플레이어 레이팅
S_A: 실제 결과 (승=1, 무=0.5, 패=0)
K: 변동폭 상수 (체스: 16~40, 게임: 보통 32)

장점

단순, 직관적, 1대1에 잘 동작. 체스·격투·1v1 RTS 표준.

한계

팀 게임 처리 어려움 (각 플레이어 기여도 평균?), 신규 플레이어 placement 느림, 활동 빈도 미반영.

Glicko / Glicko-2 — 불확실성 (Rating Deviation)

Mark Glickman 개발. ELO에 RD (Rating Deviation) 추가 — "이 사람 레이팅을 얼마나 확신하나". 신규/오래 안 한 플레이어는 RD 큼 → 더 많이 변동.

Glicko-2 추가

Volatility (변동성) 추가 — 일관된 플레이어 vs 들쭉날쭉한 플레이어 구분.

장점

신규/복귀 플레이어 빠른 적응, 활동 안 한 기간 반영 (RD 다시 증가).

적용

League of Legends 일부 시스템, Counter-Strike 일부, 다수 인디 게임. 1v1·소팀에 적합.

한계

여전히 팀 게임 (5v5 등) 처리는 보조 로직 필요.

TrueSkill 1·2 — Microsoft Research, 팀 기반 표준

2005년 Halo 2 출시 위해 Microsoft Research가 개발. μ (skill 추정)과 σ (확신도) 두 변수. Bayesian 추론으로 매치 결과 → 모든 플레이어 레이팅 동시 업데이트. 팀 게임 (n vs n, free-for-all 등) 자연스럽게 처리.

TrueSkill 1 (2005~)

Halo 시리즈에서 검증, Xbox Live 표준. 한 매치마다 모든 참가자 μ·σ 업데이트.

TrueSkill 2 (2018)

Halo 5 매치 예측 정확도 68% (TrueSkill 1은 52%). 게임 내 통계 (KDA·점수 등) 추가 활용. TrueMatch 시스템으로 Gears 5에 launch, Gears of War 4·Halo 5에 통합.

장점

팀 매치 자연스러움, free-for-all 지원, μ·σ 표시 가능 (사용자 진척감), 신규 플레이어 placement 빠름.

한계

구현 복잡 (Bayesian 추론), TrueSkill 1은 Microsoft 특허 — 상용 사용 시 라이선스 (TrueSkill 2 별도 paper). OpenSkill·기타 대안 사용 일반화.

OpenSkill — 오픈소스 대안 (2024~)

Vivek Joshy 개발, 2024년 JOSS 논문. Python 라이브러리 (PyPI v6.0.2 활발 유지). Plackett-Luce 모델 추천. n-team·n-player 비대칭 매치 처리.

성능

Python TrueSkill 구현 대비 3배 빠름. 대규모 매치메이킹 시스템에 유리.

모델 다양성

5개 모델 — Plackett-Luce, Bradley-Terry, ThurstoneMostellerFull, ThurstoneMostellerPart, BradleyTerryFull. 게임 특성에 맞춰 선택.

라이선스

오픈소스 (MIT). 특허 부담 없이 상용 게임에 사용 가능.

적용 사례

인디 게임·MMORPG 일부·스포츠 통계·데이팅 앱 등 다양. AAA 사례 적지만 빠르게 확산.

★ TrueSkill을 못 쓸 때
특허 부담·라이선스 협상이 부담이면 OpenSkill이 사실상 표준 대안. Bayesian inference 기반이라 통계적 성질 유사하면서 오픈소스. 신규 게임 시작 시 우선 검토 가치.

신규 플레이어 placement

Placement 매치

신규에게 일정 매치 (보통 5~10) 동안 σ를 크게 유지 → 결과에 따라 빠른 레이팅 조정. 시즌 시작 시 동일 처리.

Smurf 방지

신규로 위장한 고숙련자 (smurf) 방지 — 비정상적으로 placement 매치 압도하면 빠른 MMR 점프.

Tier 보호

placement 동안 별도 풀에서 매칭, 정식 풀에 들어가기 전 적정 tier 결정.

UX 트레이드오프

placement 길면 정식 진입까지 오래 걸림 (이탈), 짧으면 평가 부정확. 보통 5~10 매치.

SBMM 논쟁 — 산업 동향 (2026년 기준)

"Skill-Based Matchmaking을 캐주얼 모드에도 적용해야 하나?"는 FPS 산업의 영원한 논쟁. 사용자는 "내 실력대로 평가받기 싫다", 개발사는 "신규·라이트가 학살당해 떠난다" — 양측 모두 일리.

코어 사용자 입장

"실력 향상의 보상이 없다" — SBMM이 강하면 잘해도 항상 비슷한 상대 → 실력 늘어도 승률 50% 고정.

개발사 입장

신규·라이트 사용자 retention. 학살당하면 즉시 이탈, 결국 코어만 남아 게임 죽음.

2026 동향

Black Ops 7에서 SBMM을 default에서 제거 → open matchmaking 전환. 사용자 retention 데이터에 산업 시선 집중. Activision은 white paper에서 "skill·connection·wait time만 factor"라고 주장.

EOMM 의혹

Activision/Sony 일부 patent에 EOMM (Engagement-Optimized Matchmaking) 가능성 시사 — 결제 행동 + 매치 결과 연계 가능성. Treyarch Design Director는 "skill-based damage" 의혹은 공식 부인. 커뮤니티 회의론 지속.

★ SRE 입장
SBMM은 정책 결정 — SRE가 강도를 정하는 건 아니지만 (1) 변경 전후 매치 quality·retention·매출 분석 (2) abuse 케이스 (smurf·dodge) 데이터 (3) 실험 (A/B) 인프라 — 이 3가지가 SRE 역할. 정책 결정의 데이터 인프라 제공자.

● Pool & Queue

매치 풀 & 큐 설계

매치메이킹 정책의 절반은 "어떤 사람들을 같은 풀에 묶을 것인가". 모드·MMR 대역·플랫폼·역할·파티 크기 — 너무 잘게 자르면 매치 안 만들어지고, 너무 크게 잡으면 quality 떨어진다.

매치 풀 분리 설계

모드별 분리

경쟁·캐주얼·이벤트·아케이드 등 모드마다 별도 풀. 큐 분산 효과 + 모드 정체성.

MMR 대역 (tier)

Bronze·Silver·Gold·Platinum·Diamond·Master. 각 tier 안에서 매칭. 인접 tier 풀은 큐 길어지면 합류.

플랫폼 분리/통합

크로스 플레이 켜면 대기 줄지만 platform별 입력 차이 (마우스 vs 패드) 공정성 이슈. 옵션 노출 일반화.

지역 + region

같은 region 내 매칭 → 인접 region 허용 → 글로벌. 큐 시간 따라 단계적 확장.

★ 풀 분할의 함정
"우리 게임 풀이 작아서 매칭 안 돼요"는 보통 풀을 너무 잘게 자른 결과. 모드 5개 × MMR 7 tier × 3 region = 105개 풀. 풀당 20명만 있어도 매치 만들기 어려움. 풀 통합·확장 정책이 핵심.

솔로 vs 듀오 vs 풀파티 격차

한 팀 5명 중 (1) 솔로 5명 (2) 듀오+솔로3 (3) 풀파티 5명 — 같은 MMR이라도 호흡 차이로 풀파티가 압도적 유리. 이 격차 보정이 매치메이킹의 큰 숙제.

완전 분리

솔로 큐 vs 듀오 큐 vs 풀파티 큐 분리. 가장 공정, 풀 분산 부담.

매칭 격차 보정

풀파티 vs 솔로 매칭 시 솔로 측에 MMR 보너스 추가 — 결과적으로 비슷한 승률 만들기.

파티 크기 제한

경쟁 모드에서 듀오까지만 허용, 풀파티 금지 (LoL 솔로 큐 같이). 코어 사용자 만족도 ↑.

혼합 매칭 (현실)

완전 분리 어려우면 같은 풀에서 파티 크기 매칭 시도 (양 팀 풀파티 vs 풀파티). 안 되면 격차 보정.

역할 큐 (Role Queue) · Autofill

MOBA에서 도입된 패턴 (Overwatch·LoL·Valorant). 미리 역할 (탱커·딜러·서포터·정글·미드) 선택 → 각 역할별 풀로 분산. 게임 시작 시 역할 다툼 사라짐.

장점

매치 시작 시 역할 분쟁 없음, "원하는 역할 못 했다" 불만 ↓.

큐 시간 늘어남

인기 역할 (딜러) 큐가 항상 길고 비인기 (탱커·서포터) 짧음. 역할 균형 안 맞으면 큐 stall.

Autofill

인기 역할 큐가 길면 비인기 역할로 자동 배정 (보상 제공). 싫어하는 사용자도 있음.

운영 메트릭

역할별 큐 분포·대기 시간. 항상 deficit 역할에 보상 동적 조정.

큐 wait expansion — 시간 따라 매치 풀 확대

초기엔 좁은 풀 (정확한 MMR·같은 region) → 1분 지나도 매치 못 만들면 풀 확대 → 3분 지나면 더 확대. 이 expansion 룰이 매치메이킹의 핵심.

FlexMatch rule expression 예시 (간략)
{
  "name": "MMR-tight-then-expand",
  "ruleType": "distance",
  "measurements": [{ "name": "Player.attributes.skill" }],
  "maxDistance": 100,                     # 초기 100점 차
  "expansions": [
    { "afterSeconds": 30, "maxDistance": 200 },   # 30s 후 200점
    { "afterSeconds": 60, "maxDistance": 400 },
    { "afterSeconds": 120, "maxDistance": 800 }
  ]
}
★ AWS GameLift FlexMatch · Open Match 모두 동일 컨셉
FlexMatch는 룰 셋에 expansion·distance 명시. Open Match는 match function 안 코드로 표현. 핵심은 "시간 따라 룰 완화" — 사용자 대기 시간 cap.

친구 · 파티 매칭

친구 초대 (Lobby)

Steam ISteamMatchmaking::CreateLobby + JoinLobby 같은 API. 친구가 한 lobby 모이고 매치 큐 진입.

파티 MMR 계산

파티 평균 MMR? 최고 MMR 가중치? 단순 평균은 boosting 악용. 보통 (평균 + 최고)/2 같은 가중치.

크로스 region 친구

한국·일본 친구 매치 → 한쪽 region 선택 → 다른 쪽 lag. UX 트레이드오프 명시.

크로스 플랫폼 파티

PC + 콘솔 파티 → 매칭 풀이 두 플랫폼 합쳐짐. 입력 차이 보정 정책 필요.

Smurf · Duo-boost 탐지

Smurf (보조 계정)

고숙련자가 신규 계정으로 저 tier 학살. 신규 만족도·retention 파괴. placement 매치 압도 패턴 탐지.

Duo-boost

고MMR 친구 + 저MMR 친구 듀오 큐 → 저MMR 친구의 MMR 인위적 상승. 비대칭 KDA 패턴 탐지.

탐지 신호

placement 비정상 승률·KDA, 계정 나이 vs MMR 상승률, 동일 IP·디바이스 다중 계정.

대응

탐지 시 (1) 빠른 MMR jump (2) 의심 계정끼리 별도 풀 (3) 신뢰도 페널티 (4) 명백한 abuse는 ban.

● Server Allocation

Warmup pool — 항상 따끈한 인스턴스 유지

매치 들어왔는데 서버 cold start하면 30초~수 분 lag. 미리 일정 수 (Buffer) 인스턴스를 Ready 상태로 띄워둠.

Buffer 정책

"Ready 인스턴스 N개 항상 유지" — Agones BufferPolicy, GameLift Min Available. N은 매치 생성 rate 따라 결정.

예측적 warmup

이벤트·평일 저녁 피크에 사전 확장. 트래픽 패턴 학습 후 자동 스케일.

비용

Ready 인스턴스도 비용 발생. Buffer 너무 크면 낭비, 작으면 cold start lag. 트레이드오프.

Hot pool 변형

Ready보다 한 단계 더 따뜻한 "이미 게임 데이터 로드한" 상태. 매치 시작까지 1~2초.

Cold start 함정 — 30초 vs 5분

"인스턴스 띄우는데 5분 걸린다"는 흔한 함정. 단계별로 분해하면 다음과 같다.

노드 준비20~120s
이미지 풀10~60s
컨테이너 시작5~30s
Asset 로드10~120s
초기화5~30s
Ready총 ~5min

완화 1: 노드 미리 띄우기

cluster autoscaler buffer로 빈 노드 항상 N개. 노드 준비 시간 0.

완화 2: 이미지 사전 캐시

모든 노드에 이미지 사전 풀 (image pre-puller). 이미지 풀 시간 단축.

완화 3: Asset 분리

큰 asset은 별도 mount (PVC), 컨테이너는 작게. 스타트업 빠름.

완화 4: Warmup pool

위 모든 단계 끝낸 인스턴스를 Ready 상태로 미리 띄움. 매치 들어오면 즉시 allocate.

동적 region 할당

Latency 기반

매치 참가자들의 ping 평균이 가장 작은 region 선택. 글로벌 매치라면 중간 region (예: 한·일 매치 → 일본).

Capacity health 고려

최적 region이 fleet 포화면 차순위로. FlexMatch Queue가 우선순위 + health 둘 다 고려.

Cost 가중치

Spot 가용·인스턴스 가격 차이 반영. FleetIQ 같은 도구가 자동.

비상 fail-over

region 전체 장애 시 다른 region으로 자동. RPO·RTO 정의 필요.

Open Match 2 · FlexMatch 패턴 깊이

Open Match 2 (om-core)

OM1 마이크로서비스 → OM2는 단일 binary `om-core`로 통합. 운영 단순. Match Function (MMF)와 Director 분리 책임.

Match Function (MMF)

티켓 평가·매치 만드는 로직. 게임사가 작성. OM2 core가 MMF 호출하고 결과 받음.

Director

Backend API 사용해서 어떤 MMF 실행할지 결정. OM2는 Director가 모든 assignment 처리. 매치 결과 → DGS 직접 송신.

FlexMatch 룰셋

JSON 룰 표현식. distance·count·comparison·collection 룰 조합. expansion으로 시간 따라 완화. auto backfill priority 설정.

● Session Lifecycle

세션 lifecycle 운영 — 6단계 깊이

F1에서 lifecycle 6단계 개요를 다뤘다. 여기선 각 단계의 운영 디테일 — lobby state·hot-swap·late joiner·backfill·hand-off·reconnect 모두.

Lobby state — 매치 시작 전 상태

Lobby 생성

친구 매칭·파티 모이기·게임 모드 선택. Steam은 ISteamMatchmaking::CreateLobby. 자체 게임은 lobby service 운영.

Ready check

매치 잡힌 후 모든 참가자에게 "수락" 묻기. 수락 안 하면 매치 폐기, 다음 큐로.

Pick/ban phase (MOBA)

매치 들어가기 전 챔피언 선택·금지. 매치마다 1~2분 추가 시간.

Loading screen

모든 클라가 게임 데이터 로드 완료 대기. 가장 느린 클라가 병목. timeout 시 dropper.

Late joiner (BR · 캐주얼)

BR (Battle Royale) drop-in

매치 시작 후 일정 시간 (예: 첫 storm 전) 동안 추가 입장 허용. 매치 시작 전 풀이 다 안 차도 시작 가능.

캐주얼 in-progress join

매치 진행 중 빈 슬롯에 추가 입장. 진영 균형 유지에 도움.

경쟁 모드 disable

경쟁 매치는 late join 비추 — 공정성 깨짐. 처음 5명 vs 5명만 인정.

UX 처리

Late join 사용자에게 "현재 매치 X분 진행 중" 안내, 점수 차이 보상 옵션.

Match backfill 깊이 — FlexMatch 패턴

BR·캐주얼에서 매치 시작 후 누군가 leave → 빈 슬롯 발생. 자동으로 새 플레이어 채우는 시스템.

Manual backfill

게임 서버가 직접 backfill request. 자체 로직으로 언제·몇 명 채울지 결정.

Automatic backfill

FlexMatch 자동 — 매치 시작 시 빈 슬롯 있으면 즉시 백필 ticket 생성. AUTOMATIC 모드.

Backfill priority

backfillPriority 룰 — 백필 ticket을 신규 매치보다 우선 처리. 진행 중 매치 빨리 채워줌.

Backfill 룰 동일성

원 매치 만든 룰셋 그대로 사용 → MMR·역할·region 동일 기준 매칭.

★ Backfill의 함정
매치 거의 끝났는데 (5분 남음) 백필 → 들어오자마자 매치 종료. 사용자 짜증. 매치 진행도에 따라 backfill 차단 시간 (last-N분 차단) 설정 필요.

Hand-off · 인스턴스 이관

노드 점검

k8s node 점검 시 GameServer drain → 새 매치 안 받음, 진행 중은 끝까지. 이관 없음 (재시작 안 함).

Spot 회수

2분 통보 후 강제 종료. 매치 강제 종료 + 사용자 통보 + 보상 (게임사 정책).

Region fail-over

region 장애 시 진행 중 매치 보존 어려움. 사실상 모두 종료, 다른 region에서 재큐.

State 복원 시도

일부 게임 (MMO 채널 이전)은 진행 상태를 다른 노드로 이관 시도. 복잡·느림. 보통 안 함.

Reconnect — 네트워크 끊김 후 복귀

모바일 백그라운드

OS가 앱 죽일 수 있음. 5초~5분 안 reconnect 허용. 토큰·세션 ID 유지.

네트워크 전환

5G ↔ Wi-Fi 전환 시 IP 변경. QUIC connection ID 같은 메커니즘으로 연결 유지.

매치 보존

진행 중 매치를 N초 동안 슬롯 비워두고 reconnect 대기. 시간 초과 시 AI bot 또는 매치 종료.

UX 안내

"Reconnecting..." 명확 표시. 시도·재시도·실패 단계별 메시지.

Abandon · Dodge rate

Dodge (큐 거부)

매치 ready check에서 거부. 일정 비율은 정상 (1~5%). 폭증 시 매칭 알고리즘 또는 quality 의심.

Abandon (게임 중 이탈)

매치 진행 중 의도적 leave. 다른 사용자 매치 망침. 강한 페널티 + 신뢰도 시스템.

Disconnect (사고)

네트워크·크래시로 비자발 이탈. abandon과 구분 필요. reconnect 허용 시간 처리.

SRE 분석

같은 사용자가 dodge/abandon 패턴 보이면 별도 풀 격리. 신뢰도 낮은 사용자끼리 매칭 (저신뢰 풀).

Allocation latency — 매치 → 서버 할당까지

목표

p99 1초 이내. Agones·GameLift 모두 sub-second 가능하지만 fleet warmup 부족 시 cold start lag.

측정

매치 결정 시각 → GameServer Allocated 시각. histogram 분포.

p99 spike 원인

(1) Buffer 부족 → cold start (2) k8s API server 느림 (3) Allocator 자체 부하.

알림

p99 >1s 5분 지속 시 P1 알림. Buffer 또는 fleet 자동 확장 트리거.

큐 abuse 방어

Multi-account farming

같은 사람이 여러 계정으로 큐 → 듀오 boost·ranking abuse. IP·디바이스 fingerprint 탐지.

Match dodge cycle

매치 잡힘 → 화면 보고 마음에 안 들면 dodge → 다시 큐. 강한 dodge penalty + 누적 시 큐 lock 시간 증가.

Throw farming

의도적 패배로 MMR 낮추고 저tier 학살. 이상 KDA·승률 패턴 탐지.

Bot 큐

자동 매크로로 큐 진입 → AFK. 행동 패턴 (입력 다양성·반응 시간) 분석.

운영 도구 · 이벤트 정책

큐 강제 만료

오래 매칭 안 되면 강제 cancel + 사용자 통보. 무한 대기 방지.

매치 정책 핫 변경

이벤트 시 (큐 정원 확장·MMR 룰 변경 등) 코드 수정 없이 설정 변경. configmap 또는 feature flag.

플레이어 격리

운영 도구로 특정 사용자 매칭에서 분리 (CS 케이스·테스터 격리).

이벤트 풀 추가

주말 더블 XP·경쟁 이벤트 시 별도 풀 추가. 정상 풀과 격리.

A/B 실험

매치메이킹 정책 변경 시 일부 사용자 대상 실험 후 결과 비교. SRE가 실험 인프라 운영.

장애 시 폴백

매치메이커 장애 시 (1) 큐 일시 정지 (2) 단순 풀 (random matching) 폴백 (3) 큐 시간 cap 강화. 최소한의 게임 가능.
