Launch & Spike
= The Biggest Test

● Overview

신작 런칭 캐파시티 & 트래픽

퍼블리셔의 가장 큰 이벤트. 평소 대비 10~100배의 스파이크가 단시간에 몰리고, 모든 시스템(계정·결제·게임서버·텔레메트리)이 동시에 시험대에 오른다. 사고 시 사용자 첫인상도 망가진다.

게임 트래픽의 특수성

예측 가능한 비균질성

일반 웹 서비스처럼 평탄하지 않다. 신작 런칭/이벤트/시즌 시점이 미리 알려져 있고 그때 트래픽이 평소의 10~100배. 예측 가능 = 사전 준비 가능.

매우 짧은 피크

피크가 몇 시간(런칭 첫날) 또는 몇 분(이벤트 시작 직후)에 집중. autoscaling이 따라가지 못하는 속도. 사전 warm-up이 필수.

기다리는 사용자

일반 웹은 사용자가 흩어져 들어오지만 게임은 같은 시각에 모이는. 사전예약자들이 정각에 동시 접속.

모든 시스템 동시 부하

로그인 → 계정 → 게임서버 → 결제 → 텔레메트리 모두 동시 폭증. 가장 약한 시스템이 전체 게임 다운으로 이어짐(Amdahl).

신작 런칭이 왜 가장 위험한가

① 첫인상 = 평판

런칭 첫날 다운되면 사용자 이탈 + SNS 부정 평판 + 스토어 별점 하락. Diablo 4(2023-06)·여러 대형 게임이 런칭 사고로 사과한 사례 다수. 회복에 수 주~수 개월.

② 검증되지 않은 인프라

출시 전엔 부하 테스트로만 검증. 실제 사용자 트래픽 패턴은 정확히 예측 불가능. "우리 게임은 어떨지" 진짜 데이터가 없음.

③ 외부 의존성

Apple/Google/Steam 결제, 푸시 인프라, CDN, 외부 인증 — 우리가 통제 못하는 영역도 동시에 부하받음. 스토어가 trending 페이지에 띄우는 순간 트래픽 폭증.

④ 마케팅 비용 매몰

UA 광고비를 미리 태웠으니 런칭이 망하면 비용은 그대로 + 매출은 0. 손익 분기점 도달이 늦어지면 게임 수명 자체에 영향.

SRE 책임 영역

① 캐파시티 플래닝

예상 트래픽 산출, 인프라 사전 확보, headroom 정책. 클라우드 capacity reservation으로 인스턴스 확보까지.

② 부하 테스트

실제 트래픽 패턴을 모사한 부하 테스트. 병목 발견 → 보강 → 재테스트의 반복.

③ 런칭 워룸

런칭 D-day 24시간 워룸. 양사·전 부서 인력 대기. 실시간 모니터링 + 즉시 대응.

④ Graceful degradation

캐파 부족 시 즉시 다운보다 대기열·부분 점검·기능 끔으로 부드럽게 전환. 사용자에 명확한 안내.

● Traffic Patterns

트래픽 패턴

게임 도메인의 4가지 주요 트래픽 패턴 — 일일/주간/시즌, 신작 런칭, 이벤트 스파이크, EA vs 정식 출시.

일일·주간·시즌 패턴

일일

피크: 평일 저녁 19~23시, 주말 14~24시(KST). region별 시차 적용. 새벽 5~7시가 보통 가장 한산.

주간

금·토·일 트래픽이 평일 대비 1.5~2배. 주간 점검은 보통 화·수 새벽에. 금요일 점검·배포 회피는 업계 룰.

월간

시즌 시작·종료 시점에 스파이크. 월말 결제 한도 리셋 시점에 결제 트래픽 증가.

연간

방학·연휴(설·추석·크리스마스) 트래픽 증가. 신학기·휴가 끝 시점 감소. 연 1회 1주년·기념 이벤트 대형 스파이크.

오픈 정각 스파이크

사전예약자가 모두 정각에 시도 → 첫 1분이 가장 위험. 로그인·캐릭터 생성 폭주. 사전 warm-up + 대기열 시스템 필수.

D+1~7 검증

실제 사용자가 다양한 콘텐츠를 진행하면서 미발견 버그·캐파 병목이 드러남. 핫픽스 cadence를 평소보다 짧게 운영.

이벤트 스파이크

이벤트 시작 시각

사용자가 정각에 몰림. 평소 대비 5~20배. 이벤트 보상이 한정 수량이면 더 심함.

이벤트 종료 직전

"마감 직전 보상 확보" 행동. 마지막 30분~1시간 트래픽 폭증. 종료 시각 정확성 중요(서버 시간 동기).

대형 가챠 출시

인기 신캐릭터 가챠 출시 시 결제 + 가챠 시스템 + 인벤토리 + 텔레메트리 모두 폭증. 결제 시스템이 병목이 되기 쉬움.

인플루언서·언론 노출

예측 못한 외부 노출(스트리머·뉴스)도 트래픽 spike 요인. 마케팅팀과 정기 동기화로 사전 정보 공유.

● Capacity Planning

캐파시티 플래닝

예상 트래픽을 산출하고 인프라를 사전 확보하는 작업. 부족하면 다운, 과하면 비용 낭비. 항상 안전 쪽으로 마진을 잡되 정확도를 높여간다.

사전예약 → 런칭 트래픽 환산

사전예약 수가 캐파시티 산정의 첫 입력. 모든 사전예약자가 정각에 들어오는 건 아니지만 "최악의 경우"를 가정해야 한다.

Headroom (여유율) 정책

표준 권장

일상 운영 시 최대 캐파의 60~70%가 평균 사용량. 30~40% 여유 확보. 갑작스런 spike에 대응 가능.

런칭·이벤트

예상 피크의 1.5~2배로 확보. 예측이 빗나가도 대응 가능. 이벤트 종료 후 다시 일상 수준으로 축소.

비용 trade-off

여유율이 높으면 비용 ↑. 너무 빡빡하면 사고 위험 ↑. 게임 비즈니스에서는 "비용 < 사고 영향"이라 보통 비용을 감수.

capacity reservation

AWS On-Demand Capacity Reservations 등으로 사전 예약. 인스턴스 부족(out-of-stock)으로 scale-out 실패 방지.

Autoscaling의 한계

"오토스케일링이 알아서 해준다"는 환상. 실제로는 한계가 많다.

warm-up 시간

새 인스턴스 부팅 + 의존성 다운로드 + 헬스체크 통과까지 몇 초~몇 분. 게임 런칭 1분 안에 폭발하는 트래픽엔 못 따라감.

인스턴스 부족

한 가용영역의 특정 인스턴스 타입이 부족하면 launch 실패. 10개+ 인스턴스 타입 다양화 권장.

예측 vs 반응

일반 autoscaling은 reactive(부하 감지 → 확장). 게임은 predictive scaling(예상된 시각에 사전 확장) 또는 scheduled scaling(이벤트 시각에 맞춰 사전 capacity 추가) 필요.

downstream 한계

웹/API 서버는 자동 확장돼도 DB·외부 API는 안 됨. 병목이 어디인지 평소에 파악하고 그 부분을 우선 보강.

♥ KEY
게임 런칭은 "오토스케일링 + 사전 warm-up + capacity reservation + scheduled scaling" 4가지 조합. 어느 하나만으로는 부족.

외부 의존성 캐파시티

관리형 DB

RDS·Cloud SQL은 자동 확장 한계 있음. 사전에 instance class·read replica 확보. DB connection pool exhaustion이 가장 흔한 런칭 사고.

외부 API (스토어·인증)

Apple/Google 결제, LINE/Apple SSO 등 외부 API의 rate limit 확인. 우리 트래픽이 폭증해도 외부 limit이 병목이 될 수 있음.

CDN

다운로드 트래픽 폭증. CDN 사전 워밍(런칭 전 일부 콘텐츠 캐시 유도). 비용 사전 산출.

푸시·이메일

FCM·APNs·SES 같은 외부 서비스도 limit 있음. 사전 quota 확장 신청.

● Load Testing

부하 테스트

실제 트래픽이 들어오기 전에 시스템 한계를 측정. 병목 발견 → 보강 → 재테스트의 반복으로 자신감을 쌓는다.

테스트 종류

Smoke Test

매우 작은 부하로 시스템이 기본적으로 동작하는지. CI/CD 파이프라인에 포함. 몇 초~몇 분.

Load Test

예상 평소 부하를 일정 시간 가함. 정상 동작 확인. SLO 만족 여부 검증.

Stress Test

예상 피크 또는 그 이상 부하 → 시스템이 깨지는 한계 측정. "우리 게임은 몇 명까지 받을 수 있는가"의 답.

Spike Test

짧은 시간에 부하를 급격히 늘림. 런칭 정각 시나리오 모사. autoscaling이 따라오는지 검증.

Soak Test (Endurance)

중간 정도 부하를 장시간(수 시간~수 일) 지속. 메모리 누수·DB 연결 누수·로그 폭증 등 long-running 이슈 발견.

Capacity Test

점진적으로 부하 증가시켜 SLO 깨지는 시점 측정. "안전 운영 한계"를 정량적으로 결정.

게임 도메인 부하 시나리오

단순 API 호출만 테스트하면 실제 사용자 패턴과 다름. 전형적 사용자 여정을 시나리오로 작성해야 한다.

① 신규 가입 → 튜토리얼

회원가입 → 캐릭터 생성 → 튜토리얼 → 첫 결제 권유. 런칭 시 가장 흔한 시나리오. 인증·계정·인벤토리 모두 부하.

② 로그인 → 게임 진입

로그인 → 공지 확인 → 캐릭터 선택 → 게임서버 접속. 매일 반복되는 가장 빈번한 흐름.

③ 결제 → 가챠 → 인벤토리

상점 → 결제 → 영수증 검증 → 가챠 → 결과 → 인벤토리 추가. 매출 critical path. 정합성 검증.

④ 매치메이킹 → 게임 플레이

매치 큐 → 매칭 → 게임서버 할당 → 플레이 → 결과 보고. 실시간 게임의 핵심 부하 패턴.

★ TIP
실제 트래픽 비율을 모방. 예: "로그인 70% / 결제 5% / 매치 25%" 같은 mix를 한 시나리오에 함께. 단일 패턴만 테스트하면 실제 운영의 병목을 못 봄.

결과 해석 · 보강

핵심 메트릭

p50/p95/p99 latency(평균만 보지 말 것), 실패율, throughput, 자원 사용률(CPU/메모리/DB conn). p99가 가장 사용자 체감 지표.

병목 식별

특정 지점(DB·캐시·외부 API)에서 latency 급증 → 그곳이 병목. 부하 증가시켜 어디가 먼저 깨지는지 확인.

SLO 검증

"p99 500ms 이내, 실패율 0.1% 이내" 같은 SLO를 사전에 정의 → 부하 테스트로 검증. 미달 시 보강 후 재테스트.

반복 테스트

한 번으로 끝나지 않음. 보강 → 재테스트 → 새 병목 → 보강의 반복. 런칭 D-30부터 D-1까지 매주 진행.

⚠ PITFALL
① 스테이징 캐파 ≠ 프로덕션이면 의미 없음. 같은 또는 비례한 환경에서 측정. ② 실제 사용자 데이터(다양한 캐릭터·아이템)가 없으면 결과 왜곡. 익명화된 프로덕션 데이터 또는 합성 데이터 활용.

런칭 사고 대응

아무리 준비해도 100% 안전하지 않다. 사고가 났을 때 빠르게 사용자 영향을 줄이고 신뢰를 잃지 않는 것이 핵심.

런칭 day 워룸

구성

전 부서 인력 24시간 대기. SRE·개발사·DBA·네트워크·CS·운영·법무·임원. 외부 개발사도 동일 대기.

역할 분담 (IC 모델)

Incident Commander(전체 결정), Communications Lead(외부 안내), Operations Lead(기술 대응), Scribe(기록). Google SRE Book의 표준 모델.

모니터링 화면

실시간 대시보드(접속자·결제·에러율·지연), Statuspage 운영 화면, 외부 모니터링(SNS·커뮤니티 반응 모니터링).

커뮤니케이션 채널

전용 슬랙/디스코드 워룸 채널, 외부 개발사 동시 접속, 임원 보고 별도 채널. 상태 업데이트는 30분~1시간마다.

대기열 · 점검 전환 (Graceful Degradation)

캐파 부족 시 즉시 다운보다 단계적 degradation이 사용자 신뢰에 좋다. "못 들어오는 것"보다 "기다리면 들어가는 것"이 백배 낫다.

대기열입장 제한
+ 안내
기능 축소이벤트·상점

대기열 시스템

접속 인원 임계 초과 시 신규 접속을 큐에 대기. 예상 대기 시간 표시. 기존 접속자는 영향 없음.

기능별 차단

비핵심 기능(이벤트·상점·랭킹) 끄기 → 핵심(로그인·게임 플레이) 살리기. Feature Flag로 즉시 토글(P5 참조).

사용자 안내

모든 단계마다 명확한 안내. "현재 접속자 폭증으로 대기 중", "이벤트 일시 중단" 등. 침묵은 사용자 분노의 원인 #1.

점검 보상

긴급 점검 후 보상 정책 사전 합의. 일반적으로 점검 시간에 비례한 캐시·아이템.

Postmortem (사후 회고)

비난 없는(Blameless) 회고

"누구 잘못"이 아닌 "왜 시스템이 그 잘못을 막지 못했나"에 집중. 개인 비난은 유사 사고를 숨기게 만들어 더 큰 화근.

구조

(1) 요약·영향, (2) 타임라인(분 단위), (3) 근본 원인, (4) 무엇이 잘 됐나, (5) 무엇이 실패했나, (6) 액션 아이템 (담당자·기한 명시).

외부 공유

심각한 사고는 사용자에 공개 postmortem 발행. 신뢰 회복에 효과적. 너무 기술적이지 않게 ("사용자에게 영향 + 우리가 한 일 + 앞으로의 약속").

액션 아이템 추적

postmortem에서 나온 액션 아이템은 반드시 추적·완료 확인. 안 한 채 다음 사고 → 같은 원인 반복.

♥ KEY
대형 게임도 런칭 사고는 흔하다 (Diablo 4도 2023-06 런칭 week에 서버 장애 + Blizzard 사과). 중요한 건 사고 자체가 아니라 그 후의 회복과 학습이다.
