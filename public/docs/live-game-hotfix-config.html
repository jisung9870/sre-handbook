● Overview

라이브 게임의 변경 전달 3계층

라이브 게임은 정상 패치 사이클로 못 기다리는 변경이 매일 발생한다. 변경의 성격에 따라 서버 핫픽스 / 컨피그 푸시 / 클라이언트 OTA를 선택적으로 사용해 MTTR을 분 단위로 만든다. 2026-05 기준

3계층 변경 모델

"변경"은 단일 개념이 아니다. 어디를 바꾸느냐에 따라 전달 경로·심사·MTTR이 다 다르다. 라이브 게임 운영자는 변경마다 "이건 어느 계층에서 처리할까"를 즉시 판단할 수 있어야 한다.

① 서버 사이드 핫픽스

백엔드 코드·DB·게임 데이터 변경. 스토어 심사 무관 — 분 단위 MTTR. 드롭률 조정, 버그 수정(서버 측), 어뷰저 차단, 이벤트 활성화. 라이브 운영의 90%가 여기서 해결된다.

② 컨피그 푸시

이미 배포된 클라/서버의 동작을 설정값으로 변경. Remote Config·Feature Flag 도구. 초 단위 반영. A/B test, kill switch, 점진 노출, 강제 업데이트 트리거 모두 같은 메커니즘.

③ 클라이언트 변경

실행 파일·DLL·엔진·UI 코드. 모바일은 스토어 심사 24~48시간, Steam은 즉시. OTA(JS bundle, 에셋 핫스왑)로 일부 우회 가능하나 스토어 정책의 한계 안에서만 허용.

★ 운영 원칙
"코드로 빼낼 수 있는 건 컨피그로, 컨피그로 빼낼 수 있는 건 서버로." 클라이언트 핫픽스가 필요한 변경 자체를 줄이는 게 라이브 게임 운영의 본질이다. 처음 설계 시점에 드롭률·밸런스·이벤트 일정·UI 텍스트는 모두 서버 컨피그로 빼두면 클라이언트 빌드 cadence와 무관하게 운영 가능.

MTTR 목표값 — 계층별 합의

사고 발생 → 사용자 영향 차단까지 걸리는 시간(MTTR)은 계층마다 본질적으로 다르다. 같은 사고도 어느 계층에서 처리할 수 있는지에 따라 MTTR이 100배 이상 차이난다.

● Server-side Hotfix

서버 사이드 핫픽스

백엔드 코드·게임 데이터의 긴급 변경. 스토어 심사·클라 재설치 무관 — 라이브 운영자가 가장 자주 다루는 도구다. 점진 배포, 세션 인지 롤링, 자동 롤백이 안전성의 3축.

서버 핫픽스란

대상

(1) 백엔드 서비스 코드: API 서버, 매치메이커, 결제 검증, 푸시. (2) 게임 데이터 테이블: 드롭률, 아이템 스탯, 이벤트 일정. (3) 어뷰저 차단 룰: 패턴 매칭, IP 블록.

일반 배포와의 차이

정기 배포는 PR → CI → staging → prod의 풀 사이클. 핫픽스는 심사 단축·테스트 축소가 들어감. 그만큼 롤백 가능성 검증이 핵심 — 핫픽스로 더 큰 사고 만드는 게 흔한 안티패턴.

3계층 변경별 위험

코드 변경: 빌드/배포 필요, 위험 ↑. 데이터 변경: 빌드 없이 DB UPDATE, 위험 ↓ but 회귀 어려움. 설정 변경: 컨피그 푸시로 분리(별도 탭) — 가장 안전.

표준 흐름

탐지 → 진단 → 핫픽스 PR(축약) → 1차 검증(staging) → 점진 배포(샤드 단위) → 모니터링 → 전 샤드 → 사후 회고. 처음부터 100% 배포는 절대 금지.

샤드 / 리전 점진 배포

"전 샤드에 동시 배포"는 핫픽스의 가장 흔한 실수다. 검증 안 된 코드를 전체에 동시 푸시하면 한 번 실패에 모든 사용자가 영향받는다. 일부 샤드/리전 → 모니터링 → 확대 패턴이 표준.

Internal QA1 shard, 5분
Canary1~2 shards, 15분
Region 110% 트래픽, 30분
Region 1 전체100% R1
Global나머지 region

샤드 단위 분리

게임 백엔드는 보통 샤드(서버 그룹) 단위로 운영. 100 샤드 중 5개 먼저 → 메트릭 30분 관찰 → 50개 → 100개. 각 단계에서 자동 게이트(에러율, 응답 시간) 통과 시에만 다음 단계.

리전 단위 분리

글로벌 게임은 KR → JP → SEA → NA → EU 순. KR에서 문제 발생 시 다른 리전 사용자는 영향 X. 리전별 메트릭 격리 필수 — 글로벌 평균만 보면 한 리전의 문제가 묻힌다.

자동 게이트 — 통과 조건

각 단계에서 다음 메트릭이 baseline 대비 일정 임계 이내일 때만 진행: error_rate +0.1% 이내, p99_latency 1.5x 이내, crash_rate +0.05% 이내, session_drop +1% 이내. 한 항목이라도 임계 초과 → 자동 일시정지 + 운영자 알림.

Argo Rollouts 핫픽스 단계 정의 예시
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata: { name: game-api-hotfix }
spec:
  strategy:
    canary:
      steps:
      - setWeight: 5          # 5% 트래픽 (1~2 샤드)
      - pause: { duration: 15m }
      - analysis:
          templates: [{ templateName: error-rate }]
      - setWeight: 25         # 한 region 일부
      - pause: { duration: 30m }
      - analysis:
          templates: [{ templateName: error-rate }]
          templates: [{ templateName: crash-rate }]
      - setWeight: 100        # 글로벌

세션 인지 롤링 — drain pattern

게임 서버 배포는 일반 HTTP 서비스와 결정적으로 다르다. 활성 세션이 있는 인스턴스를 그냥 종료하면 진행 중인 매치가 끊긴다. drain pattern은 신규 세션 받기를 멈추고, 기존 세션이 자연 종료될 때까지 기다리는 패턴.

Agones / GameLift 기본 지원

k8s 기반 게임 서버 운영 도구는 모두 drain 단계를 지원. Agones GameServer.Status.State의 Allocated → Shutdown 전환 시 신규 매치 라우팅 차단. 플레이어가 자연스럽게 떠난 후 인스턴스 회수.

API 서버는 다른 패턴

매치 인스턴스가 아닌 일반 API 서버는 Connection Draining + readiness false. LB가 새 요청 안 보내도록 막고, 기존 in-flight 요청 완료 대기. preStop hook + terminationGracePeriod 60초가 표준.

"긴 세션"의 함정 — MMO·MOBA

MOBA 한 매치 30~40분, MMO 인스턴스 던전 1시간 이상. drain만으로는 부족. 대안 (1): 매치 종료 후 다음 매치 시작 전 incremental update. 대안 (2): 핫리로드 가능한 게임 데이터(스탯·드롭률)는 데이터만 변경, 코드는 정기 패치. 대안 (3): "다음 정기 점검" 안내 후 강제 종료.

롤백 안전망

핫픽스가 새로운 사고를 만들면 영향이 두 배다. 롤백 메커니즘이 평소에 검증된 상태여야 한다 — 사고 시점에 처음 시도하는 롤백은 위험.

코드 롤백

Argo Rollouts·Spinnaker는 이전 버전을 항상 보관(scaleDownDelaySeconds). 1분 안에 트래픽 스위치만으로 롤백. 핫픽스 머지된 main에서 revert PR 자동 생성하는 봇 추천.

데이터 롤백

DB UPDATE 핫픽스는 코드보다 어렵다. 변경 전 스냅샷 의무: UPDATE items SET drop_rate=0.05 WHERE id=42 전에 SELECT id, drop_rate INTO backup_table. 일괄 롤백 스크립트 미리 준비.

스키마 변경의 함정

핫픽스로 컬럼 추가·제거하는 건 위험. 구버전이 새 스키마 못 읽거나 그 반대. 핫픽스에서는 스키마 변경 금지 — Expand/Migrate/Contract 패턴은 정기 패치 사이클로.

분기마다 롤백 리허설

"롤백은 안전"은 검증 안 된 가정. 분기마다 staging에서 강제 롤백 시뮬레이션 — 자동화 스크립트가 정상 작동하는지, 데이터 정합성 깨지지 않는지. 실제 사고 시 처음 시도하면 안 된다.

✕ FORWARD FIX의 유혹
"롤백하면 어차피 다시 핫픽스 해야 하니, 그냥 한 번 더 핫픽스로 고치자"는 forward fix는 위험. 새 변경이 또 사고를 만들 확률이 있다. 원칙: 사고 시 일단 롤백 → 분석 → 정식 해결. forward fix는 (1) 데이터 마이그레이션이 이미 진행, (2) 보안 패치, (3) 외부 호환성 변경 같은 예외 상황에만.

핫픽스 런북 패턴

사고 발생 시 의사 결정의 70%는 미리 합의된 룰로 자동화 가능. 새벽 3시에 누가 무엇을 어떻게 할지 즉답 가능해야 한다.

● Config Push

컨피그 & Remote Config 운영

가장 안전하고 가장 빠른 변경 메커니즘. 코드 빌드 없이 초 단위로 동작 변경 가능. Firebase·Unity·PlayFab·LaunchDarkly 등 도구 선택, 안전성·정합성 운영, kill switch와 컨피그 부채 관리까지.

컨피그 vs 코드 — 무엇을 컨피그로 빼낼까

컨피그로 빼낼 후보

이벤트 일정, 드롭률, 아이템 가격, UI 텍스트, 점검 공지, 푸시 메시지 템플릿, 강제 업데이트 트리거, kill switch, 신규 기능 활성화 토글, A/B test 분기. "숫자·문자열·boolean"으로 표현 가능한 비즈니스 룰 대부분.

코드에 둬야 할 것

알고리즘 로직, 게임 메커니즘, 데이터 구조, 보안 검증 로직. 컨피그로 표현하면 가독성 떨어지고 회귀 어려운 영역. "컨피그가 코드 흉내 내기 시작하면" 그건 잘못된 추상화.

경계가 모호한 영역

드롭 테이블의 가중치는 컨피그 OK. 드롭 알고리즘 자체는 코드. "기획자가 직접 만질 수 있는가"가 1차 기준. 기획자가 수정해야 하는 값은 다 컨피그. 단, 그 컨피그가 코드 buggy해지면 책임은 코드 작성자에게 있어야 함 — 검증 로직은 코드.

★ 분리 설계 원칙
컨피그를 처음 만들 때 "이 키는 뭘 의미하나, 누가 만지나, 잘못된 값일 때 어떻게 되나"를 동시에 정의한다. 이 3가지를 답할 수 없는 키는 만들지 않는다. "일단 컨피그로 빼두자"는 컨피그 부채의 시작이다.

컨피그 안전성 — 잘못된 값으로부터의 방어

컨피그가 코드보다 안전한 게 아니다. 잘못된 컨피그가 잘못된 코드보다 더 빠르고 더 광범위한 사고를 만든다 (배포가 빠르니까). 컨피그 푸시도 검증·점진 배포·롤백이 필요하다.

1. 스키마 검증

컨피그 키마다 타입·범위·열거값을 명시. JSON Schema 또는 protobuf로 정의. drop_rate는 number, [0, 1]로 강제. 0.5를 50으로 잘못 넣어도 검증 단계에서 차단.

2. 점진 배포

컨피그도 100% 즉시 푸시 금지. 1% → 10% → 50% → 100% 점진 적용. Firebase Remote Config은 condition으로 사용자 비율 분기. LaunchDarkly는 percentage rollout.

3. 자동 롤백 트리거

컨피그 푸시 후 N분 동안 에러율·crash율 모니터. 임계 초과 시 자동으로 이전 컨피그로 복원. AWS AppConfig·LaunchDarkly는 내장 — 자체 시스템도 같은 구조 권장.

4. 클라 측 안전 default

컨피그 fetch 실패 시 SDK는 안전한 default 값으로 동작해야. flags.isOn('feature', false) — 컨피그 시스템 죽어도 안전. SDK 캐시 + 마지막 성공 값 폴백.

Kill Switch — 장애 시 1차 방어선

Kill switch는 "신기능 토글"이 아니다. 이미 운영 중인 핵심 기능을 비상시 비활성화하는 컨피그. 결제·외부 추천·푸시·매치메이킹 같은 영역에 미리 박아두는 게 라이브 운영의 기본.

박아둘 곳

외부 의존: 외부 API 호출 (그 API 죽으면 우리도 함께 죽음). 비싼 연산: ML 추천, 무거운 쿼리. 최근 변경: 신규 기능 6개월간 kill switch 유지. 핵심 게이트: 결제, 매치메이킹 — 더 큰 영향 막는 마지막 카드.

Graceful Degradation

Kill switch on 시 단순 에러가 아니라 "덜 좋지만 동작하는 모드"로 폴백: 추천 엔진 죽으면 "인기 아이템 top 10" 정적 리스트, 결제 외부 의존 죽으면 "잠시 후 재시도" 안내 + 자동 큐잉.

운영 — 검증 필수

Kill switch는 평소에 안 쓴다 → 사고 시점에 처음 토글하면 동작 안 할 위험. 분기마다 staging에서 모든 kill switch 토글 테스트. runbook에 "어떤 상황에 어떤 스위치"를 명시 — 새벽 3시에 어떤 스위치를 누를지 모르면 의미 없음.

Kill switch 패턴 — 게임 매치메이킹
async function findMatch(player) {
  // kill switch — 매치메이킹 전체 비활성화 (보통 false)
  if (await config.getBool('matchmaking.disabled', false)) {
    metrics.increment('matchmaking.disabled');
    return { status: 'maintenance', message: '잠시 후 다시 시도해주세요' };
  }

  // kill switch — 새 매칭 알고리즘 (롤아웃 중)
  const useNewAlgo = await config.getBool('matchmaking.use_v2', false);

  try {
    return useNewAlgo
      ? await matchV2(player)
      : await matchV1(player);
  } catch (err) {
    // v2 실패 시 v1로 폴백 (graceful degradation)
    if (useNewAlgo) return await matchV1(player);
    throw err;
  }
}

컨피그 부채 — 늘어나는 키, 사라지지 않는 부채

컨피그는 강력한 만큼 부채화 쉬운 도구다. 100개 release flag가 코드에 남아 있으면 모든 함수가 if-else 지옥. 1년 후 어떤 키가 살아있는지 아무도 모른다.

유형별 수명 관리

Release toggle (점진 노출): 100% 도달 후 30일 내 코드에서 제거. Experiment toggle (A/B): 실험 결정 후 즉시. Ops toggle (kill switch): 영구. Permission (게임별 분기): 영구.

탐지 — 죽은 키

· 100% rollout 30일 이상 — 그냥 코드에 박아도 됨
· 0% rollout 90일 이상 — 안 쓰는 코드, 그냥 삭제
· 마지막 evaluation 60일 — 코드에서 호출 안 됨
도구의 stale flag 리포트 활용 (LaunchDarkly·Unleash 모두 지원).

방지 — 만들 때부터 청산 계획

(1) 만료일 필수: 키 생성 시 expiration 입력, release toggle 기본 30일. (2) Owner 명시: CODEOWNERS 매핑, 만료 시 자동 PR. (3) 분기당 정리 데이: 분기마다 reading day 1일 = 죽은 키 제거. "나중에 정리하자"는 영원히 정리 안 한다.

● Client OTA

클라이언트 핫픽스 & OTA

실행 파일·DLL·게임 코드 변경. 모바일 스토어 정책의 한계가 가장 강한 영역. Apple Guideline 4.7, CodePush 종료 후 대안, Lua 스크립팅, 에셋 핫스왑까지.

Apple Guideline 4.7 — 한계와 정공법

Apple App Store Review Guideline 4.7은 "interpreted code"의 동적 다운로드/실행을 통제. 2025-11 업데이트로 HTML5/JS 미니앱·미니게임도 전체 App Store 심사 대상이 되어 더 엄격해졌다.

4.7 핵심 규칙

"interpreted code"의 동적 다운로드는 (1) 앱의 핵심 기능 변경 불가, (2) 새 기능 추가 불가, (3) 디지털 상품 판매·결제 우회 불가. WebKit + JavaScript Core 외 엔진 사용 금지.

4.7.4 — manifest 의무 (2025-11~)

앱이 동적 콘텐츠(HTML5/JS)를 가지고 있으면 모든 소프트웨어와 메타데이터의 인덱스를 제출 의무. universal links 포함. Apple이 심사 시 이 manifest로 컴플라이언스 검증.

네이티브 API 노출 금지 (4.7.2)

HTML5/JS 콘텐츠가 카메라·푸시·파일시스템·연락처·위치 등 네이티브 iOS 기능에 접근 불가. 브릿지·래퍼·커스텀 프레임워크로 우회 시도하면 거부.

실무 영향

"앱 안에서 새 미니게임 다운로드해서 실행"은 grey zone. 앱 본질을 변경하지 않는 데이터 업데이트는 OK: 캐릭터 스탯, 이벤트 일정, 새 스킨 등은 데이터 다운로드로 처리. 새 게임 모드·새 메커니즘은 본 빌드 패치 사이클로.

⚠ 정책 변동 주의
Apple 4.7은 자주 업데이트된다. 2025-11 큰 변경 후 2026 추가 강화 가능성. 주요 변경 시점에 게임 운영 정책 재검토 필수. WWDC와 Apple Developer News 모니터링.

Lua 스크립팅 — 게임의 가장 일반적 OTA

많은 게임이 게임 로직 일부를 Lua 같은 스크립팅 언어로 작성하고, 빌드에 인터프리터와 스크립트를 함께 번들. 새 스킬·이벤트 로직을 클라 패치 없이 변경 가능.

번들된 Lua는 OK

Lua 인터프리터 + Lua 스크립트가 앱 번들 안에 포함되어 출시 시 함께 심사받으면 Apple 정책상 안전. Solar2D 같은 엔진의 핵심 운영 모델.

다운로드 Lua는 grey area

"앱 시작 시 서버에서 Lua 스크립트 다운로드해서 실행"은 4.7의 "minor way for minor features" 한계. 앱의 핵심 동작을 바꾸지 않는 minor 변경은 통과되곤 하지만 정식 보장 없음.

실무 — 안전한 사용

(1) Lua는 게임 로직 표현용, 새 기능 추가가 아닌 설정·튜닝 수준으로 사용. (2) 스크립트는 게임 콘텐츠 데이터로 분류 (캐릭터 스탯·드롭 룰). (3) 새 게임 모드를 Lua로 다운로드하는 건 피한다 — 클라 패치로. (4) Lua 검증·sandbox 필수 — 서버에서 받은 스크립트가 임의 시스템 호출 불가.

★ 한국 모바일 게임에서 흔한 패턴
리니지 류 MMORPG, 미소녀 게임, 카드 게임이 Lua 또는 자체 스크립팅 언어를 사용해 이벤트·신규 캐릭터 스킬을 클라 패치 없이 운영. 새 캐릭터 추가도 에셋 + 스크립트로 처리하는 게 일반적. 단 게임 메커니즘 자체는 클라에 박힘.

OTA 업데이트 도구 — 2026 현황

React Native·Flutter·하이브리드 앱의 JS bundle 핫 업데이트 도구. Microsoft App Center가 2025-03-31 종료되며 CodePush도 함께 종료 — 대안 도구로 마이그레이션 필요.

네이티브 게임은?

Unity·Unreal 모바일 게임은 OTA 도구 직접 사용 X. 대신 에셋 번들 핫스왑(다음 섹션) + Lua 스크립팅을 조합. RN/하이브리드 게임에만 OTA 도구 적용.

EAS의 percentage rollout

OTA 자체도 점진 배포 가능. 1% → 10% → 50% → 100%. 새 bundle에서 crash 발생 시 republish로 즉시 이전 bundle로 롤백 (사용자 측 효과).

Android In-App Update API

Google Play Core의 AppUpdateManager는 앱 안에서 강제 업데이트 UI를 제공. iOS와 달리 Google Play Store가 직접 강제 업데이트 메커니즘을 제공한다.

Flexible Update

사용자가 백그라운드에서 다운로드 → 본인이 설치 시점 선택. 부드러운 사용자 경험. 비긴급 업데이트에 적합.

Immediate Update

전체 화면 차단 — 업데이트 완료까지 앱 사용 불가. 보안 패치, 호환성 깨지는 변경에 사용. 사용자 경험은 나쁘지만 강제력 강함.

iOS는 자체 구현 필요

iOS는 동등한 시스템 API 없음. 앱 시작 시 서버에 버전 체크 API 호출 → 응답에 따라 업데이트 안내 화면 표시가 표준 패턴. App Store URL로 이동시킴. 자세한 패턴은 "강제 업데이트" 탭 참고.

Android In-App Update — 기본 흐름
val appUpdateManager = AppUpdateManagerFactory.create(context)

appUpdateManager.appUpdateInfo.addOnSuccessListener { info ->
  if (info.updateAvailability() == UPDATE_AVAILABLE) {
    // 우선순위(서버에서 결정)에 따라 immediate vs flexible
    val priority = info.updatePriority()
    val mode = if (priority >= 4) IMMEDIATE else FLEXIBLE

    appUpdateManager.startUpdateFlowForResult(
      info, mode, activity, REQUEST_CODE
    )
  }
}

에셋 번들 핫스왑 — Unity·Unreal 게임의 OTA

Unity Addressables, Unreal Pak 패치, 자체 에셋 매니저. 실행 파일은 그대로 두고 에셋만 교체. 모바일 스토어 정책상 안전한 영역 — 새 캐릭터 모델, 새 UI, 새 사운드 등.

Unity Addressables

에셋을 grouping → 빌드 시 번들로 분리 → CDN에서 다운로드. 게임 시작 시 catalog.json으로 버전 확인 → 변경된 번들만 다운. UpdateCatalog + DownloadDependenciesAsync.

Unreal Pak Patch

Pak 파일 단위 차분 패치. Patch_v2.pak이 base.pak 위에 덮어씌움. UE의 IO Store로 더 효율적 패키징 (UE 5.x 권장).

운영 — 에셋 핫픽스의 한계

(1) 코드 변경 불가 — 새 메커니즘은 클라 빌드 패치 필요. (2) 에셋 호환성 — 구버전 클라가 신버전 에셋 못 읽으면 크래시. catalog 버전 매칭 + 강제 업데이트 트리거 조합. (3) CDN 비용 — 에셋 크기가 크면 다운로드 비용 큼. (4) 첫 실행 경험 — 큰 에셋 다운로드 대기는 사용자 이탈. 핵심 에셋만 즉시 다운, 부가 에셋은 백그라운드.

♥ 분리 원칙
"클라 패치"와 "에셋 패치"를 분리해 운영하면 cadence 자유도가 커진다. 클라는 격주, 에셋은 매일 가능. 단 호환성 매트릭스 관리 필수 — 구 클라가 신 에셋을 읽거나 그 반대일 때 graceful degradation 또는 강제 업데이트.
● Force Update

강제 업데이트 & 버전 게이트

모바일 OS는 자동 강제 업데이트 메커니즘이 없다. 앱이 자체 구현. 버전 게이트 API, 그레이스 윈도우, 점진적 강제, 모바일 vs Steam 차이.

버전 게이트 API — 자체 구현 패턴

앱 시작 시 서버에 "이 버전 OK?" 묻는 API. 서버가 minimum/recommended 버전 응답 → 클라가 분기. 이 API가 모든 강제 업데이트의 근간.

버전 게이트 API — 응답 스키마
// GET /api/v1/version-check?platform=ios&version=1.2.3
{
  "status": "force_update",    // ok | recommend_update | force_update | maintenance
  "minimum_version": "1.3.0",
  "latest_version": "1.5.2",
  "store_url": "https://apps.apple.com/.../id12345",
  "message": {
    "ko": "안전한 게임 플레이를 위해 업데이트가 필요합니다",
    "en": "Update required for security"
  },
  "deadline": "2026-06-15T00:00:00Z",    // 그레이스 마감
  "maintenance": {
    "active": false,
    "start": null,
    "end": null,
    "message": null
  }
}

4가지 응답 상태

ok: 정상 진입. recommend_update: 부드러운 안내, 거부 가능. force_update: 차단, 업데이트 후 진입 가능. maintenance: 점검 중, 모든 사용자 차단.

호출 시점

앱 시작 시 1회 + 장시간 백그라운드 후 복귀 시. 게임 중간에 응답이 force_update로 바뀌어도 매치 끝까지는 진행 — 매치 종료 후 재시작 시 적용. 강제로 끊으면 사용자 분노.

응답 캐싱 — 신중하게

매 요청마다 호출하면 비용·지연. CDN 캐시 60초~5분 권장. 단 긴급 점검·강제 업데이트는 즉시 반영 필요: TTL 짧게 + 푸시로 클라가 강제 재조회 트리거. Firebase Remote Config 같은 Remote Config 도구의 fetch interval과 같은 메커니즘.

그레이스 윈도우 — 강제 업데이트 전 유예

"오늘 업데이트 안 하면 못 들어옴"은 사용자 분노 폭발. 며칠 전부터 안내 → 강제 전환의 점진적 패턴이 필요하다.

그레이스 길이 결정

보안 취약점: 48~72시간 (CVE 심각도에 따라). API 호환성 깨지는 변경: 30일. 법적 요구: 컴플라이언스 마감 기한. 일반적 cadence: 7~14일. 너무 짧으면 사용자 이탈, 너무 길면 구버전 운영 부담 큼.

안내 메시지 디자인

D-7부터: "곧 업데이트가 필요합니다 (XX월 XX일까지)". D-1: "내일까지 업데이트 안 하시면 게임 진입이 제한됩니다". D-day: "업데이트가 필요합니다. 스토어로 이동해 주세요". 왜 필요한지 한 줄: "보안 강화를 위한 필수 업데이트" 같은 이유 명시.

예외 — 즉시 차단

심각한 보안 사고 (계정 도용·결제 우회·실시간 치팅)은 그레이스 없이 즉시 force_update. 사용자 영향보다 사고 확산 방지가 우선. 이런 경우는 분기에 1번 이하여야 정상 — 자주 발생하면 평소 보안 운영에 문제.

점진적 강제 — 코호트 단위 롤아웃

강제 업데이트도 100% 즉시는 위험. 새 빌드에 새 버그가 있으면 강제 업데이트 받은 사용자 모두 영향. 코호트 단위로 점진 적용해야 한다.

코호트 정의

국가, OS 버전, 기기 종류, 사용자 ID hash, 게임 진행도 등으로 분기. 예: "한국 + iOS 17 이상 + UID hash 0~10%"를 1차 force update 대상. 24시간 모니터 → 안전 시 확대.

버전 게이트 API의 코호트 분기

API 응답에 cohort_id 받아 force_update 결정. 같은 클라 버전이라도 코호트별로 다른 응답 가능. Remote Config의 condition 또는 자체 코호트 시스템 활용.

강제 업데이트 자체의 메트릭

강제 업데이트 후 추적할 메트릭: update_completion_rate (실제 업데이트한 비율), user_drop_rate (안 돌아온 사용자 비율), store_failure_rate (스토어에서 다운로드 실패), support_ticket_volume (CS 문의 급증). 이상치 감지 시 자동으로 grace 연장 또는 force_update 해제.

⚠ 다중 강제 업데이트 금지
한 달에 강제 업데이트 2번 이상은 사용자 분노 임계 초과. 가능하면 강제 업데이트는 분기에 1~2번. "이번에 같이 강제하자"는 의도적 묶음이 답 — 보안 + 호환성 + 신기능을 한 빌드에 묶어 강제는 1번.

모바일 vs Steam — 강제 업데이트의 근본 차이

Steam은 본질적으로 강제 업데이트 메커니즘이 다르다. 모바일이 OS와 무관하게 앱이 자체 차단하는 반면, Steam은 클라이언트가 게임 시작 전 자동 업데이트.

크로스 플랫폼 게임의 함정

같은 게임의 모바일과 Steam이 다른 클라 버전 동시 운영 가능. 서버는 양쪽 호환 유지 필요 — Steam이 먼저 v1.5로 갔다고 모바일도 즉시 v1.5로 못 강제. 서버 API의 클라 버전 호환 매트릭스가 운영 중심.

Steam의 자동 업데이트 신뢰

Steam은 강제 업데이트가 사실상 자동이지만 100% 보장 X. "Only update this game when I launch it" 옵션을 사용자가 켜뒀을 수 있음. Steam도 게임 측 버전 게이트 API 권장 — 단 그레이스 짧게 (24시간).
