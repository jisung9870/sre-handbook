Steam & PC 퍼블리싱 생태계

PC 게임 퍼블리싱은 Steam이 사실상 표준 플랫폼이고, 그 위에 Epic Games Store·GOG·자체 런처·중국 런처가 보조 채널로 존재한다. 모바일과 운영 모델이 크게 달라 별도 학습이 필요하다. 본 가이드는 Steam에 집중한다.

PC 퍼블리싱 생태계 (2026-05 기준)

2026년 PC 시장 점유율 기준으로 Steam이 약 72%, Epic Games Store가 약 8%, 나머지가 GOG·자체 런처·중국 채널. Steam은 사실상의 표준이지만 글로벌·중국 시장 진입을 위해 멀티 채널 전략이 일반적이다.

왜 Steam이 표준인가

(1) 최대 동시 사용자 — 라이브 동접 수천만
(2) Steamworks SDK가 다른 어떤 플랫폼보다 풍부 (Workshop·Cloud·Achievements·Leaderboards·Friends·Rich Presence)
(3) 지역 결제·환불·세금 자동 처리
(4) Steam Deck·Proton으로 Linux까지 확장

왜 멀티 채널을 고려하는가

(1) Epic 12% 수수료가 매력적 (Steam 30% 대비 큰 차이)
(2) Steam에서 거절 또는 visibility 부족 시 대안
(3) 중국 시장은 Steam이 비공식 — 정식 출시는 현지 채널 필수
(4) 자체 런처는 anti-cheat·LiveOps 통제력 극대화

Steam 점유율의 함정

"Steam만 하면 된다"는 80%이다. 그러나 나머지 20%가 매출의 30% 이상을 차지하는 사례 多. 특히 한국·중국·동남아는 자체 런처·현지 채널이 강세.

SRE 관점

여러 채널을 운영하면 빌드 동기화·정산 분리·CS 채널 분리가 운영 부담. 채널이 늘면 매출은 늘지만 운영 복잡도는 채널 수의 제곱에 가깝게 늘어난다.

퍼블리셔 · 개발사 R&R

Steam 운영의 책임 분담은 모바일과 다르다. Steam이 대부분을 자동화해주지만 스토어 페이지·Workshop·CS·anti-cheat은 퍼블리셔의 적극적 운영이 필요하다.

모바일 vs PC 분기점

같은 게임이어도 모바일과 PC는 운영이 거의 다른 시스템. 표면적으로 비슷하지만 cadence·결제·anti-cheat이 다 다르다. (P1 가이드와 함께 보면 큰 그림이 명확.)

패치 cadence

모바일: 스토어 심사 24~48h. 긴급 패치도 시간 단위.
PC/Steam: SteamPipe 분 단위 즉시. branch 분리·롤백 자유.
같은 게임을 양 플랫폼 출시하면 cadence 동기화가 운영 핵심 과제 (P10).

결제 환경

모바일: 스토어 IAP 강제 (Apple 30%, Google 20%/10%)
Steam: 30/25/20% 구간 + 자체 환불 (2시간/14일)
크로스 플랫폼 게임은 양쪽 가격·재화 정합성이 필요.

Anti-cheat

모바일: SDK + 디바이스 무결성 (root/jailbreak 탐지)
PC/Steam: 커널 anti-cheat (VAC·EAC·BattlEye)
PC는 Anti-cheat 자체가 운영 부담이 더 큼.

UGC 정책

모바일: 보통 인게임 UGC만 (스토어 정책 영향).
Steam: Workshop이 강력한 UGC 채널. 게임 수명·매출 모두에 큰 영향.

● Store & Discovery

스토어 페이지 & Discovery 알고리즘

Steam에서 게임이 보이느냐는 곧 매출이다. Wishlist·리뷰 점수·visibility round가 알고리즘의 핵심 입력. 잘 만든 스토어 페이지 + 적극적 visibility 운영이 매출의 30~50%를 결정한다.

Store Page 구성 요소

스토어 페이지는 게임의 첫 인상이고 알고리즘의 입력이다. 각 요소마다 표준 규격·검수 가이드라인이 있다.

캡슐 이미지

Library·Search·Discovery 모두에서 노출. Main Capsule (616×353)·Small Capsule (231×87)·Vertical Capsule (374×448) 등 여러 크기. 각 위치의 픽셀 사이즈에 맞게 별도 디자인 권장.

헤더 · 스크린샷

Header capsule 460×215, 스크린샷 1920×1080 권장. 5장 이상 권장하며 첫 3장이 가장 중요. 텍스트 오버레이는 일부 영역에서 금지.

트레일러

Highlight 영상 1개 + 추가 영상 최대 5개. 자동 재생 정책 — 첫 30초 안에 핵심 게임플레이가 보여야 함. 장르별 변환율 데이터로 첫 5초 최적화가 표준.

설명 · 태그

Short description (300자 이내)·About this game·시스템 사양. 태그(Tag)는 알고리즘의 핵심 입력 — 사용자가 다는 popular tag가 추천 가중치에 영향. 의도된 태그가 안 달리면 visibility 손실.

★ TIP
스토어 페이지는 출시 30일 전까지는 stable 상태가 되어야 알고리즘이 평가를 시작한다. 출시 직전에 자료를 바꾸면 wishlist conversion 데이터가 흩어진다. 마케팅 자료는 6주 전부터 동결하는 게 표준.

Wishlist 메커니즘

Wishlist는 Steam의 가장 강력한 마케팅 자산. 출시 시점 wishlist 수가 첫 주 매출과 강한 상관이고, Wishlist velocity(증가율)가 알고리즘에 직접 영향.

Wishlist의 역할

(1) 출시 알림 — wishlist 사용자에게 자동 알림
(2) 세일 알림 — 가격 인하 시 자동 알림
(3) 알고리즘 가중치 — wishlist 가속도가 "Popular Upcoming"·visibility round에 영향
(4) 매출 예측 — 첫 주 매출 ≈ wishlist × conversion rate

Wishlist Velocity

Wishlist 절대 수보다 최근 30일간의 증가 가속도가 더 중요. 가속도가 있으면 "Popular Upcoming" + Discovery Queue에 노출 → 추가 wishlist → 가속도 유지의 선순환.

Wishlist Conversion

Wishlist 사용자가 출시 시점에 실제 구매로 전환되는 비율. 10~25%가 일반적(장르·가격·사전 마케팅에 따라). 사전에 wishlist 1만 = 출시 첫 주 매출 1~2.5천 카피 정도.

운영 — Wishlist 늘리기

(1) Coming Soon 페이지 일찍 공개
(2) Demo 제공 (Next Fest 활용)
(3) 트레일러·SNS·콘텐츠 크리에이터
(4) Cross-promotion (관련 게임의 스토어 페이지 광고 슬롯)
(5) Steam Festival·이벤트 참여

Discovery 알고리즘 (2026-05)

Steam Discovery는 사용자별 맞춤 추천 + 알고리즘 visibility round로 구성된다. 알고리즘은 비공개지만 Wishlist velocity, 리뷰 점수, 매출 가속도가 핵심 입력으로 알려져 있다.

Visibility Round

Steam이 신작·업데이트에 부여하는 노출 슬롯. 출시 visibility round가 가장 강력 — 출시 직후 며칠간 메인 페이지 노출. 업데이트 시 Update Visibility Round로 라이브러리·wishlist 사용자에게 노출.

리뷰 점수의 역할

Mixed(40%) 이상이면 알고리즘에 음의 영향이 거의 없다. Mostly Negative(40% 미만) 또는 Overwhelmingly Negative로 떨어지면 visibility가 급감. 출시 직후 첫 100~500리뷰가 결정적.

More Like This

비슷한 게임의 스토어 페이지 하단에 노출. 태그·플레이 패턴 기반. 경쟁 게임에 묻어가는 자연 트래픽 채널 — 태그가 정확해야 효과.

Discovery Queue

로그인 사용자에게 추천 게임을 큐 형태로 노출. 사용자가 "Skip" 또는 "Wishlist"하면 알고리즘 학습. 2026 업데이트로 hover 미리보기·infinite scroll 추가.

⚠ 현실
2025년 한 해 Steam에 출시된 17,889개 게임 중 절반이 10리뷰 미만, 약 2,100개는 리뷰 0건. visibility 받는 게 곧 생존. 알고리즘에 의존하지 말고 외부 마케팅·인플루언서·커뮤니티가 visibility의 절반은 만들어야 한다.

Demo · Steam Next Fest

Demo는 wishlist conversion rate를 1.5~3배 끌어올리는 가장 효과적 도구. Next Fest는 데모 게임만 모아 보여주는 연 3회 이벤트로 wishlist 폭발 기회다.

Steam Demo

풀 게임의 별도 App ID. 스토어 페이지에 "Demo 다운로드" 버튼이 노출됨. 데모 자체에 리뷰는 없지만, 본 게임 wishlist에 직결된다. 데모 플레이타임이 길면 wishlist 변환률↑.

Next Fest 자격

미발매 신작 + Demo 필수. 게임당 1회만 참여 가능 — 신중히 타이밍 선택. 재발매 게임은 자격 없음. 출시 임박한 게임은 Next Fest 참여가 사실상 의무.

2026 일정

2026-02-23 ~ 03-02 / 06-15 ~ 06-22 / 10-19 ~ 10-26 (각 7일). 행사 시작 6주 전까지 데모·이벤트 자료 등록.

Next Fest 운영

· 라이브 스트리밍 슬롯 (개발자가 직접 데모 플레이)
· 팬덤 리치 위해 SNS·포럼 사전 활동
· Wishlist velocity 폭발 시기이므로 트래픽 대응 + CS·서버 캐파시티 사전 점검

Early Access 운영

미완성 게임을 먼저 출시해 운영하면서 완성하는 모델. PUBG·Hades·Valheim 같은 큰 성공 사례가 많아 인디·중대형 모두에서 표준 옵션이 됐다.

EA의 장점

(1) 매출이 일찍 발생
(2) 커뮤니티 피드백 받아 게임 방향 조정
(3) Wishlist를 더 일찍 끌어모음
(4) 데모보다 더 강한 commitment 받기

EA의 위험

(1) "unfinished" 인상이 굳어질 수 있음
(2) 커뮤니티 대응에 운영 부담 증가
(3) 정식 출시 시점 visibility round를 놓칠 수 있음
(4) EA 종료 후 매출이 잘 안 늘어나는 함정

EA 환불

EA 게임의 14일 환불 카운터는 release date 기준이지만, 2026년 정책 변경: 정식 출시 후 EA 시점 playtime이 누적된다(과거에는 별도 카운트였으나 합쳐짐).

EA → 정식 출시 운영

· EA 가격에서 정식 가격으로 인상 (또는 동일 유지)
· "정식 출시 임박" 마케팅
· 콘텐츠·스토리 핵심 추가
· 출시 visibility round를 받아 마지막 매출 폭발

출시 후 visibility 관리

출시 후에도 visibility는 계속 관리해야 한다. 세일·이벤트·Update Visibility Round가 후반 매출의 큰 부분.

Steam 세일

여름·겨울·가을·봄 Steam Sale + 장르별 페스티벌 (Strategy Fest, RPG Fest 등). 세일 등록은 자체 신청 — 자동이 아님. 세일 가격은 직전 가격 대비 최소 10% 할인이어야 함.

Update Visibility Round

메이저 업데이트 출시 시 라이브러리·wishlist 사용자에게 알림. 너무 자주 사용하면 효과 감소(Steam이 자동으로 cooldown). 큰 콘텐츠·시즌·DLC 출시에 한 번씩.

DLC · 시즌 패스

DLC는 별도 App. 본 게임의 매출에도 영향 (visibility round + 세일 번들). 시즌 패스 vs DLC 분리 vs 무료 시즌 + 유료 cosmetic 모델 비교가 LiveOps 핵심 결정.

커뮤니티 활성화

스토어 페이지의 News·Update post·이벤트 announcement는 알고리즘에 일정 영향. 정기적인 dev log·로드맵 포스팅이 활성도 신호.

● Workshop · UGC

Workshop 운영

Steam Workshop은 사용자 제작 콘텐츠(UGC)의 호스팅·검색·구독 플랫폼. 게임 수명을 몇 배로 늘리는 강력한 도구지만 모더레이션·DMCA·어뷰징 대응이 만만치 않다.

Workshop 아키텍처

Workshop은 게임마다 활성화 여부를 결정. 활성화하면 Valve가 호스팅·검색·구독 인프라를 제공하고, 게임은 다운로드한 콘텐츠를 게임 내에서 로드하는 부분만 책임.

구성 요소

· Workshop Item: 사용자가 업로드한 콘텐츠 단위
· Subscription: 사용자가 item을 "구독"하면 자동 다운로드
· Tags · Categories: 게임마다 정의 (e.g. "Map", "Skin", "Mod")
· API: ISteamUGC 인터페이스로 게임이 query·download

호스팅 책임 분담

Valve: 파일 호스팅·CDN·검색 UI·구독 관리
퍼블리셔/개발사: 콘텐츠 형식 정의·게임 내 로드·모더레이션·DMCA 대응
Workshop은 게임이 아닌 플랫폼 차원에서 제공되는 인프라.

UGC 종류

· 단순 콘텐츠 (스킨·맵·캐릭터 모델)
· 게임 모드 (Workshop 게임 모드)
· DLC 모델 (Paid Workshop)
· 커뮤니티 콘텐츠 (스크린샷·아트·가이드)
각 종류마다 모더레이션·정책이 다름.

API 한도

각 Workshop item의 메타데이터·파일 사이즈 한도가 있음. 대용량 mod는 분할 또는 외부 호스팅. 다운로드 동시성·rate limit이 있어 인기 mod 출시 시 일시 다운로드 지연 가능.

Subscribe · 다운로드 흐름

사용자가 Workshop item을 구독하면 Steam 클라이언트가 자동 다운로드, 게임이 시작될 때 마운트한다.

구독 vs 다운로드

구독은 "관심 표시 + 자동 다운로드 동의". 사용자가 구독해도 게임 실행 전에는 게임이 모름. 게임이 ISteamUGC.GetSubscribedItems()로 query해서 로드.

Auto-update

Workshop item이 업데이트되면 Steam 클라가 자동 다운로드. 게임은 버전 체크 후 incompatible 시 사용자에게 안내(저장 파일 호환성 등).

오프라인 동작

오프라인 모드에서도 이미 다운로드된 mod는 사용 가능. 단, 새 구독·업데이트는 온라인 필요.

의존성 관리

Mod가 다른 mod에 의존하면 자동 다운로드 + 로드 순서 결정이 어려움. 게임이 의존성 그래프를 게임 내에서 처리해야 함.

모더레이션 — 자동 + GM

UGC가 많아지면 부적절·저작권 침해·악성 콘텐츠가 섞여 들어온다. 자동 필터 + 사용자 신고 + GM 검토의 3단 모더레이션이 표준.

자동 필터

· 텍스트 욕설·혐오 표현
· 이미지 NSFW·폭력 (CV 모델)
· 알려진 악성 파일 (해시 비교)
· 정책 키워드 (저작권 표시·외부 URL)

사용자 신고

Workshop 페이지의 "신고" 버튼. 카테고리: 부적절·스팸·저작권·기타. 같은 item에 신고가 누적되면 자동 hide + GM 큐 진입.

GM 검토 큐

자동 필터 통과 못한 콘텐츠 + 사용자 신고가 누적된 콘텐츠. GM이 맥락·게임 정책·법적 위험을 종합 판단. 결정: 게시·hide·삭제·BAN.

이의제기

크리에이터는 hide·삭제 결정에 이의제기 가능. 다른 GM이 재검토 — 최초 결정자 X. EU DSA·AI Act 준수.

저작권 · DMCA 처리

UGC의 가장 큰 법적 위험. Steam Workshop은 DMCA 대상이고 Valve는 신고 받으면 즉시 takedown. 2025년에는 가짜 DMCA 신고 폭주 사건이 있어 운영 부담이 컸다.

DMCA 표준 흐름

(1) 권리자가 Valve에 신고 → (2) Valve가 즉시 takedown + 크리에이터에 통지 → (3) 크리에이터가 counter-notice 가능 → (4) 권리자가 소송하지 않으면 복원.

퍼블리셔의 역할

Valve는 1차 처리, 그러나 게임 IP 권리자가 퍼블리셔/개발사이므로 자기 IP 침해 신고는 직접 모니터링·신고. 영화·캐릭터·음악 등 외부 IP 침해 mod도 자체 모니터링.

2025 false DMCA 사태

Stellaris·Left 4 Dead 2·Terraria 등에 대량의 가짜 DMCA 신고로 mod가 무더기 takedown. 가짜 신고에 대한 Steam 시스템 대응이 한계 있음을 드러냄. 모더레이션 시스템 점검 필요.

UGC 약관 — IP 권리

Workshop UGC의 저작권은 크리에이터 보유, Valve는 이용권을 가짐. 게임 회사가 UGC를 게임에 통합하거나 상품화하려면 별도 라이선스 필요.

⚠ PITFALL
Mod에 영화 캐릭터·라이선스 음악·다른 게임 모델을 그대로 쓰는 사례가 흔하다. 권리자가 신고하면 takedown — 모드 게시 가이드라인에 외부 IP 사용 금지를 명확히 표시해야 게임사가 책임에서 벗어남.

Paid Workshop · 수익 분배

일부 게임은 Workshop을 유료 채널로 운영한다. Dota 2 cosmetic이 대표적. 한때 Skyrim에 도입 시도했다가 커뮤니티 반발로 철회된 사례도 있어 정책 설계가 신중해야 한다.

표준 분배 (75/25)

일반 paid Workshop: Valve + 게임 개발사 75% / 크리에이터 25%. 게임마다 dev가 비율을 조정 가능 — 일부는 더 후하게, 일부는 다른 구조.

Dota 2 모델

크리에이터 ~25%. Cosmetic·아이템 셋·Treasure를 사용자가 게임 내에서 구매. Workshop 인기 작품을 dev가 picking해서 공식 in-game 아이템으로 채택하는 방식.

CS2 모델

Counter-Strike 2의 무기 스킨·케이스. 정확한 비율 비공개지만 커뮤니티 추정 ~10~25%. 경제 가치가 큰 만큼 어뷰징·시장 조작 리스크도 큼.

Skyrim 사례 (2015)

Bethesda가 paid mods 도입 → 4일 만에 철회. 커뮤니티가 "기존 무료 mod 문화 파괴"로 강하게 반발. 이후 Bethesda는 자체 Creation Club으로 우회.

★ NOTE
Paid Workshop은 (1) 기존 무료 문화와의 충돌 (2) 모더레이션·환불 부담 (3) 표절·도용 검증 필요라는 3중 부담이 있다. 처음부터 paid 모델로 시작하는 게임(Dota 2 cosmetic처럼)이 아니면 후발 도입이 어렵다.

UGC 어뷰징 — 가짜 평점 · copycat

UGC 채널은 어뷰징 표적. 가짜 평점, 인기 mod copycat, 번들·끼워팔기 등 다양한 패턴이 있어 모니터링이 필요하다.

가짜 평점

봇 또는 다중 계정으로 자기 mod에 평점·다운로드 부풀리기. 업로드 직후 비정상 패턴(짧은 시간 내 대량 +rating)이 시그널. 자동 탐지 + 사람 검증.

Copycat

인기 mod를 그대로 복사·이름만 바꿔서 재업로드. 검색 결과 잠식. 이미지 hash·파일 hash·메타데이터 비교로 탐지. 원작자 신고가 가장 큰 시그널.

번들 · 끼워팔기

여러 mod를 하나로 묶어 자기 mod에 trojan처럼 끼워 넣기. 사용자 신고 + 콘텐츠 분석으로 탐지.

악성 mod

크래시·악성코드·정보 탈취 mod. Workshop은 임의 코드를 실행할 수 없는 sandbox(Steam이 강제) — 하지만 게임이 자체 스크립팅을 허용하면 문제 발생. 게임 차원의 sandbox 설계가 필수.

Modding 정책

게임이 어디까지 수정을 허용할지를 명시한 정책. UGC 운영의 헌법.

허용 범위 명시

· 텍스처·모델·사운드 교체 — 보통 허용
· 게임 로직·룰 수정 — 정책별
· 다른 게임 콘텐츠 import — 보통 금지
· 코드 디컴파일 후 수정 — 보통 금지
모드 정책 페이지에 명확히 표기.

경쟁 영향

Multiplayer 게임은 mod로 경쟁 형평성이 깨질 수 있음 → mod 사용 시 ranked 매칭 불가, 또는 mod 사용 별도 표시.

NSFW 정책

Steam의 글로벌 정책 + 게임별 정책. 청소년이용가 게임에 NSFW mod는 게임 등급 위반 위험. NSFW filter + 별도 카테고리화가 표준.

치트성 mod

싱글 게임은 허용 가능. 멀티플레이는 명백한 치트 — anti-cheat 시스템과 충돌. mod loader가 anti-cheat 우회 시 자동 BAN 설계가 일반적.

● Steamworks SDK

Steamworks SDK 운영

Steamworks SDK는 게임이 Steam의 인증·저장·달성·친구·UGC를 사용하기 위한 인터페이스. 잘 통합하면 게임 가치가 크게 올라가지만, SDK 장애·서비스 다운에 대한 fallback 설계가 필수.

Auth · Steam ID · Encrypted Ticket

사용자가 Steam 계정으로 로그인했음을 게임 서버에 증명하는 메커니즘. Steam ID(64-bit) + ticket이 핵심.

Steam ID

모든 Steam 계정의 64-bit unique ID. 변하지 않는 절대 식별자. 게임 백엔드는 이걸 기준으로 계정·인벤토리·진행도 관리.

Encrypted App Ticket

클라이언트가 Steam에서 받아서 게임 서버로 전달. 서버가 Steam에 검증 요청 → 정상이면 진짜 사용자임이 증명. 30분 유효, 정기 재발급.

Auth Session Ticket

P2P 또는 게임 서버 인증용. 다른 사용자에게 자기 신원 증명. multiplayer 매칭에서 핵심.

Web API Key

게임 서버가 Steam Web API에 인증할 때 사용. 외부 노출 절대 금지 — 노출 시 즉시 회전. 환경 변수·시크릿 매니저 필수.

표준 인증 흐름
// 1. 클라가 Steam에서 ticket 받기
const ticket = SteamUser.GetAuthTicketForWebApi();

// 2. 게임 서버로 전송
await fetch('/auth', { headers: { 'X-Steam-Ticket': ticket } });

// 3. 서버가 Steam Web API에 검증
GET https://api.steampowered.com/ISteamUserAuth/AuthenticateUserTicket/v1/
    ?key={web_api_key}&appid=123456&ticket={ticket}

// 4. 응답에서 SteamID 추출 → 게임 세션 생성

Steam Cloud Save

사용자 진행도·설정을 Steam이 호스팅하는 클라우드에 저장. 디바이스 전환·재설치 후에도 자동 복원. 잘 설계하지 않으면 conflict·data loss로 CS 폭주.

한도 설정

App Admin에서 byte quota per user + 파일 수 limit 설정. 최대 100GB까지 가능하지만 보통 100MB~1GB 수준이 일반적. 각 게임이 자율 설정.

Conflict Resolution

여러 디바이스에서 다른 저장 → conflict. "keep local" / "use cloud" / "manual merge" 옵션을 사용자에게 보여주는 게 표준. 자동 선택은 데이터 손실 위험.

API

ISteamRemoteStorage로 read/write/delete. 동기화는 Steam 클라가 백그라운드 처리. 게임은 파일 시스템에 쓰는 것처럼 사용.

Cloud 비활성화

사용자가 게임별·전역으로 Cloud 비활성화 가능. Cloud OFF 사용자는 디바이스 간 동기화 안 됨 — 게임이 알고 있어야 함.

⚠ PITFALL
Cloud는 증분 백업이 아닌 단순 동기화. 게임이 저장 파일을 잘못 쓰면 Cloud에도 즉시 반영 → 사용자 진행도 영구 손실. 저장 파일은 atomic write + 다중 슬롯 + 자체 서버 백업까지 두어야 안전.

Achievements · Stats

달성 + 통계는 게임 retention에 직접 영향. Steam 프로필·활동 피드에 노출되어 사회적 가치도 있다.

Achievements

App Admin에서 미리 정의 + 클라/서버에서 unlock. 한 게임당 100개 권장(절대 한도는 없으나 너무 많으면 가치 감소). 다국어 텍스트 지원.

Stats

숫자 통계 (kill 수·플레이타임·점수). Achievement의 underlying counter로 자주 사용. Stats가 leaderboards·achievements를 trigger.

Server-side unlock

경쟁 게임은 클라이언트가 단독으로 unlock하면 치트 위험. 게임 서버가 검증 후 unlock 호출이 안전. ISteamGameServerStats 사용.

Global Stats

전체 플레이어 통계 (e.g. "전체 플레이어가 X마리 사냥"). 마케팅·이벤트에 활용. App Admin에서 publish 설정.

Leaderboards

전 세계·친구·지역별 점수 랭킹. 경쟁 게임의 핵심 retention 도구.

구성

App Admin에서 leaderboard 정의 — 이름·정렬 방향(asc/desc)·display type(num/time/ms). 게임마다 leaderboard 여러 개 가능.

Submit

UploadLeaderboardScore — keep best, force update, attach details(replay ID 등). 치트 방어를 위해 서버 검증 후 submit 권장.

Query

DownloadLeaderboardEntries — 전 세계·친구만·내 주변. UI는 게임이 직접 그림.

치트 대응

치트 점수가 leaderboard 상단을 점령하면 다른 사용자 의욕 상실. 이상치 자동 제거 + 정기 GM 검토 + replay 검증이 표준. Speedrun·점수 게임은 더 강력한 검증 필요.

Rich Presence · 친구

사용자가 지금 게임에서 무엇을 하는지 친구·프로필에 노출. 친구 가입·매칭·소셜 viral의 기반.

Rich Presence

"숲에서 사냥 중", "랭크 매치 진행 중" 같은 상태 텍스트. 친구 목록·프로필에 노출. 다국어 토큰 지원.

Friend List · Invite

친구 초대 (Steam Overlay에서). 게임이 connection string을 받아 매칭 처리. "같이 플레이"는 retention의 큰 동력.

Steam Overlay

게임 위에 떠 있는 Steam UI (Shift+Tab). 채팅·브라우저·스크린샷·가이드. Overlay 비활성화 시 친구·invite 등 큰 기능 손실 — overlay-friendly 설계가 중요.

Family Sharing 영향

Steam Family Sharing으로 부모 계정의 게임을 자녀가 플레이 가능. Achievement·Stats는 자녀 계정으로 분리 기록되지만 일부 멀티플레이는 제한.

SDK 장애 시 Fallback 패턴

Steam 서비스가 다운되면 매칭·인증·Cloud Save가 모두 영향. 큰 장애 시 게임이 어떻게 살아남느냐가 SRE 설계의 핵심.

인증 — degrade gracefully

Steam 인증 실패 시: (1) 캐시된 ticket으로 짧은 grace period (2) 오프라인 모드로 단독 플레이 (3) 결제·랭킹은 차단하되 캠페인은 진행 가능. "Steam 다운 = 게임도 다운"이 최악.

Cloud Save — local 우선

로컬 저장 항상 유지, Cloud는 보조. Cloud 다운 시 로컬로 계속 플레이 + 복원 시 sync. Cloud-only 설계는 큰 위험.

Workshop — local mod 활성

Workshop API 다운 시 이미 다운로드된 mod는 사용 가능. 새 mod 다운로드만 차단. 게임이 API 실패에 robust해야 함.

멀티플레이 — Steam P2P 외 추가

Steam Datagram Relay·Steam P2P가 있지만 자체 매칭 서버를 따로 두는 게 안전. 트랜잭션 게임은 Steam SDK + 자체 백엔드 dual-stack 표준.

♥ KEY
Steam SDK는 안정적이지만 100% 가용성은 아니다. "Steam 의존성을 어디까지 둘지"가 게임 가용성의 직접 입력. 핵심 게임플레이는 SDK 없이도 돌아가게 설계, 소셜·랭킹·결제만 SDK에 의존.

$10M 임계의 의미

대부분 게임이 절대 도달 못함. $10M 도달은 큰 성공의 신호. 이 구간에 들어가면 매출 1달러당 5센트가 더 들어옴.

실효 수수료 계산

매출이 $50M 이상 가면 실효 수수료가 25% 근처로 떨어짐. 매출이 클수록 평균 수수료가 낮음. 이 구조가 큰 게임에 유리.

Epic의 12%와 비교

$10M 미만 게임은 Steam 30% vs Epic 12% — Epic이 18%p 유리. 그러나 Steam 사용자 기반이 훨씬 커서 매출 자체가 더 많은 경우가 다수. 단순 수수료 비교만 보면 안 됨.

Region Pricing

Steam은 70+ 통화·100+ region별 가격을 별도 설정 가능. 권장가가 자동 환산되지만 게임마다 자율 조정 — 한국·중국·러시아·중남미는 권장가보다 저가가 일반적.

권장가 (Suggested Price)

Steam이 환율·지역 구매력 기반으로 자동 권장. 한국·일본·동남아·중남미·러시아는 권장가가 USD 대비 낮음. 권장가 그대로 사용해도 무방.

VPN 차익 거래 위험

저가 region에서 결제 → 고가 region에서 사용. Steam이 region locking을 자동 적용하는 경우(주로 러시아·아르헨티나)도 있고, 차이가 너무 크면 자동 region locking이 걸림.

가격 변경 정책

가격 인하는 자유, 인상은 정책 제약. 세일 직전·직후의 인상은 알림 의무. Steam은 30일 내 가격 변동 이력을 사용자에게 보여줌.

한국·일본 특수성

한국은 권장가가 USD 대비 ~75~85% 수준이 일반적. 일본은 90% 수준. 현지 인디 게임과의 가격 위치를 고려해 추가 조정.

Steam keys 배포

Steam key는 Valve가 무료로 발급해주는 게임 활성화 코드. 외부 판매·번들·번들 사이트에 활용. 정책을 어기면 키 발급 중단·플랫폼 차단.

키의 종류

· Customer Keys — 외부 판매용 (Humble Bundle, Fanatical 등)
· Developer Keys — 인플루언서·언론사용
· Beta Keys — 베타 테스터용
· Promotional Keys — 이벤트·콘테스트용

발급 정책

App당 1회 5,000 keys 발급 가능, 그 이상은 Valve 검토 필요. 30일 내 사용량의 10% 정도면 자동 추가 발급. 무제한 아님.

가격 동등성 (Price Parity)

외부에서 판매할 때 가격이 Steam 가격의 -10% 이상 낮으면 안 됨(영구 인하 금지). 단, 단기 세일은 허용. 이 정책 위반하면 키 발급 중단.

그레이 마켓

제3 키 사이트(G2A·Kinguin 등)에서 도난 카드로 결제된 키가 유통되는 경우. Valve가 chargeback 시 key 회수(이미 활성화됐어도). 게임사·사용자 모두 피해.

⚠ NOTE
Steam key는 Valve의 인프라(저장·다운로드·인증)를 무료로 사용하는 것. 너무 많이 외부 판매로 우회하면 Valve가 정책으로 막을 수 있다. 키 발급량의 절반 이하를 외부 판매로 두는 게 일반적 가이드라인.

Gifting · Family Sharing

Steam에는 게임을 다른 사용자와 공유하는 두 가지 메커니즘이 있다. 게임사 매출과 운영에 영향.

Gifting

다른 사용자에게 게임을 선물. 받는 사람의 라이브러리에 영구 추가. 환불 정책: 받는 사람이 받기 전이면 보낸 사람이 환불, 받은 후엔 받는 사람 기준.

Steam Family Sharing

최대 5명·10대 디바이스에서 가족 라이브러리 공유. 한 명이 게임을 살 때 다른 가족이 같이 플레이. 동시 플레이는 한 명만.

매출 영향

Family Sharing으로 매출이 줄 수 있지만, Steam 사용자 충성도·플랫폼 효과가 더 크다는 게 Valve의 입장. 게임사는 일반적으로 Family Sharing 비활성화 안 함.

비활성화 옵션

게임이 Family Sharing 비활성화 가능 (App Admin). 경쟁 멀티플레이 게임은 어뷰징 우려로 비활성화하는 경우 많음.

● Anti-cheat & Security

Anti-cheat · 보안

PC anti-cheat은 운영 부담이 크지만 멀티플레이 게임에는 필수. 본 가이드는 퍼블리셔 운영 관점(통합·이의제기·Proton)을 다루고, 알고리즘·기술 세부는 F4 (Anti-cheat & 게임 보안) 가이드 참조.

VAC · EAC · BattlEye · nProtect 비교

PC 게임의 주요 anti-cheat 4종. 운영 모드(kernel vs user)·시장 점유율·강점이 다르다. 하나만 쓰는 게임도 있고 여러 개를 layered로 쓰는 게임도 있다.

VAC의 특수 위치

Valve가 무료 제공 → 진입 장벽 낮음. 그러나 kernel 모드가 아니라 사용자 모드라 정교한 cheat에 약함. 캐주얼 게임·인디에 적합, 경쟁 강도 높은 게임은 EAC·BattlEye 추가가 일반적.

커널 모드의 trade-off

Kernel 모드는 cheat 탐지에 유리하지만 (1) 사용자 신뢰 문제 (2) 시스템 안정성 위험 (3) 다른 anti-cheat와 충돌. nProtect는 always-on으로 더 invasive해서 거부감 큼.

한국 시장 특수성

한국 MMORPG는 nProtect·Wellbia 사용이 표준. 매크로 어뷰징이 심하고 한국 anti-cheat이 한국식 매크로에 강한 점. 글로벌 진출 시 nProtect 호환성·Proton 이슈로 다른 솔루션 추가 검토.

Layered 접근

큰 멀티플레이 게임은 VAC + EAC/BattlEye + 자체 server-side detection의 multi-layer. 한 layer가 뚫려도 다른 layer가 잡음. 단, 통합 복잡도 + 사용자 부담 증가.

통합 패턴 (Kernel · User · Hybrid)

Anti-cheat을 게임에 어떻게 통합하느냐에 따라 사용자 경험·탐지 효과·시스템 안정성이 모두 달라진다.

Kernel mode (Ring 0)

드라이버로 동작. 모든 메모리·프로세스·디바이스에 접근 가능. 탐지 능력 최고지만 시스템 충돌·BSOD·OS 호환성 이슈 위험. 사용자 신뢰도 가장 낮음.

User mode (Ring 3)

일반 프로세스로 동작. 안전성 + 호환성 양호. 그러나 같은 권한의 cheat은 탐지 어려움. VAC가 대표.

Hybrid

핵심 탐지는 kernel, UI·네트워크는 user. EAC·BattlEye가 이 패턴. 게임 시작 시 kernel 모듈 로드 → 종료 시 unload로 always-on 부담 회피.

Always-on vs Game-only

nProtect·Vanguard는 항상 실행 (boot-time도 가능). EAC·BattlEye는 게임 시작 시에만. Always-on은 효과 ↑ 사용자 거부감 ↑↑.

⚠ PITFALL
Kernel anti-cheat은 OS 업데이트마다 호환성 깨질 수 있음. Windows 11 update·새 GPU 드라이버에 의해 BSOD 폭발한 사례가 있음. Anti-cheat 벤더의 vendor SLA가 운영 안정성에 직접 영향.

Server-side 검증

Client anti-cheat만 의존하면 우회 가능. 서버에서 게임 로직·통계·리플레이 검증이 마지막 방어선이다.

비정상 통계

· Headshot 비율 95%+ → 핵 의심
· 분당 처리량 비현실적
· 사망 횟수 0회 다수 매치
· 정밀 조준 분포가 사람과 다름
분포 분석 + ML 분류기로 자동 탐지.

리플레이 분석

의심 매치의 리플레이를 자동 분석 (또는 사람 검토). "믿을 수 없을 정도로 좋은 플레이"를 정량 측정. 타격 위치·반응 시간 분포가 사람과 다름.

Snapshot 검증

클라이언트가 보낸 위치·상태가 서버 모델과 일치하는지 검증. 위치·속도가 물리적으로 불가능하면 즉시 차단.

이상 거래 패턴

아이템 거래·매치메이킹·랭킹 등에서 비정상 패턴 (작업장·boost·매치 throwing) 탐지. 어뷰징 가이드(P8)와 연결.

BAN 정책 · 이의제기

자동 anti-cheat이 BAN을 결정하지만, EU DSA·AI Act는 사용자 이의제기 권리를 명시. 자동 BAN만 두면 위법 위험.

BAN 통보

· BAN 시 사용자에게 이메일 + 게임 내 메시지
· BAN 사유 카테고리 + 이의제기 절차 안내 필수
· "정확한 어떤 cheat을 썼다"는 detail은 cheat 우회 위험으로 비공개가 일반적

이의제기 절차

(1) 사용자가 폼 제출 → (2) 다른 GM 또는 별도 검토팀이 재검토 (최초 결정자 X) → (3) 결정 + 통보 → (4) 인용 시 BAN 해제 + 보상.

BAN 후 인벤토리

아이템·재화는 어떻게? 영구 BAN 시 일반적으로 회수 (작업장 인센티브 차단). 단, 실제 결제 아이템은 환불 또는 다른 계정 이전 옵션이 컴플라이언스에 안전.

VAC BAN의 특수성

VAC BAN은 Steam 계정 차원 — 라이브러리 전체 BAN이 아니라 해당 게임 멀티플레이 차단. 영구 + 이의제기 어려움. Valve 정책으로 사실상 항소 불가.

Steam Deck · Proton 호환성

Steam Deck은 Linux 기반(Steam OS) + Proton(Wine fork)으로 Windows 게임을 실행. Anti-cheat이 가장 큰 호환성 장애물 → 2026년 큰 진전.

EAC + Proton

Epic이 Proton 지원을 적극 발표. Steamworks 페이지에서 토글로 enable. recompilation·바이너리 업데이트 불필요. 가장 호환성 좋은 anti-cheat.

BattlEye + Proton

지원되지만 각 타이틀마다 manual config. Valve 또는 BattlEye 측에 연락해서 활성화. EAC보다 진입 장벽 약간 높음.

VAC

User mode라 Proton에서 별도 처리 없이 자동 동작. Steam Deck에서 가장 자연스럽게 작동.

nProtect · Vanguard

Linux/Proton 호환 거의 안 됨. Steam Deck Verified 인증을 받기 위해서는 다른 anti-cheat 추가가 필요한 경우 多. Helldivers 2가 nProtect 추가 도입 시 Linux 호환성 논란이 있었던 이유.

★ NOTE
Steam Deck Verified 마크는 사용자 검색·필터에 큰 영향. Anti-cheat 호환성이 Verified 인증의 가장 흔한 fail 사유. 글로벌 출시 시 Steam Deck 호환성을 anti-cheat 선택의 한 축으로 봐야 한다.
