● Time Complexity

시간복잡도 실무 예시

O(1)부터 O(n²)까지 — 인프라 운영에서 만나는 각 복잡도의 실체.

시간복잡도를 실무 예시와 함께 설명해 주세요.
시간복잡도는 입력 크기(N)에 따라 연산 횟수가 어떻게 증가하는지를 나타냅니다. O(1)은 해시맵 조회(입력 크기와 무관), O(log N)은 B-Tree 인덱스 검색(높이만큼), O(N)은 iptables 규칙 순회, O(N log N)은 정렬, O(N²)은 이중 루프(Nested Loop Join)입니다.
복잡도의미N=1000일 때인프라 실무 예시
O(1)상수1eBPF Map 조회, 해시맵 get, Redis GET
O(log N)로그10B+Tree 인덱스 검색, Skip List 조회
O(N)선형1,000iptables 규칙 순회, Seq Scan, 리스트 검색
O(N log N)선형로그10,000sort 명령어, Prometheus 시계열 정렬
O(N²)이차1,000,000Nested Loop Join (인덱스 없는 경우)

핵심 면접 사례 — iptables O(N) vs eBPF O(1):
kube-proxy의 iptables 모드에서 Service 수가 1000개이면, 매 패킷마다 수천 개의 iptables 규칙을 순차 평가(O(N))합니다. Cilium은 eBPF Hash Map에서 O(1) 조회로 Service→Pod 매핑을 찾습니다. Service 10,000개에서 이 차이가 수 ms의 latency 차이로 나타납니다.

Go에서:
map[string]int — O(1) 평균 조회. 해시 충돌 시 최악 O(N)이지만 Go가 내부적으로 잘 관리
[]int + 이진 탐색 — O(log N). sort.Search()
[]int + 선형 탐색 — O(N). 단순 for 루프
● HashMap

해시맵

O(1) 조회의 원리, 충돌 해결, Go map 내부 — 가장 많이 쓰이는 자료구조.

해시맵의 동작 원리와 충돌 해결 방법을 설명해 주세요.
해시맵은 키를 해시 함수로 변환하여 배열의 인덱스(버킷)를 결정하고, 해당 버킷에 값을 저장합니다. 서로 다른 키가 같은 버킷에 매핑되는 것이 해시 충돌이며, Chaining(링크드 리스트로 연결)과 Open Addressing(다른 빈 버킷 탐색)으로 해결합니다. Go의 map은 Chaining 방식을 사용합니다.
해시맵 동작: key "user:42" → hash("user:42") = 0x7F3A → 0x7F3A % 8 = 2 → bucket[2]에 value 저장 key "user:99" → hash("user:99") = 0x4E12 → 0x4E12 % 8 = 2 → 충돌! bucket[2]에 이미 데이터 있음 Chaining: bucket[2] → [user:42, val1] → [user:99, val2] Open Addressing: bucket[2] 사용 중 → bucket[3] 확인 → 비어있으면 저장
Go map 내부:
Go의 map은 해시 테이블 + 버킷 배열로 구현됩니다. 각 버킷은 8개의 키-값 쌍을 저장. 버킷이 가득 차면 overflow 버킷을 연결(chaining). Load factor가 6.5를 초과하면 자동 확장(rehash) — 이때 점진적 마이그레이션(incremental growth)으로 한 번에 전체 복사하지 않음.

Go map 주의사항:
• 동시 접근 안전하지 않음: 여러 goroutine이 동시에 읽기/쓰기하면 fatal error: concurrent map writes. sync.RWMutex 또는 sync.Map 사용 필수
• 순서 보장 없음: for k, v := range m의 순서가 실행마다 다름
• nil map에 쓰기하면 panic: var m map[string]int; m["a"] = 1 → panic. make(map[string]int)로 초기화 필요

// Go: 동시 접근 안전한 map 패턴 type SafeMap struct { mu sync.RWMutex m map[string]int } func (s *SafeMap) Get(key string) (int, bool) { s.mu.RLock() defer s.mu.RUnlock() v, ok := s.m[key] return v, ok } func (s *SafeMap) Set(key string, val int) { s.mu.Lock() defer s.mu.Unlock() s.m[key] = val }
인프라 매핑: eBPF Hash Map, Redis의 Hash 타입, Prometheus 라벨 셋 → 시계열 매핑 모두 해시맵 기반
● Tree Structures

트리 자료구조

BST, Red-Black Tree, 왜 DB는 B-Tree를 쓰는가 — 선택의 이유를 설명.

이진 탐색 트리(BST)와 레드블랙 트리의 차이, 그리고 DB가 B-Tree를 선택한 이유를 설명해 주세요.
BST는 왼쪽 자식 < 부모 < 오른쪽 자식 규칙의 트리로 O(log N) 검색이지만, 편향되면 O(N)으로 퇴화합니다. Red-Black Tree는 색상 규칙으로 균형을 유지하여 항상 O(log N)을 보장합니다. 하지만 DB는 B-Tree를 사용하는데, 이유는 하나의 노드에 수백 개의 키를 저장하여 트리 높이를 3~4로 유지하고, 디스크 I/O 횟수를 최소화하기 때문입니다.
트리노드당 키높이 (100만 행)디스크 I/O용도
BST1~20~20회이론적 기본
Red-Black1~20~20회메모리 내 (Go map, Java TreeMap, Linux CFS)
B-Tree수백~3~3회디스크 기반 (PostgreSQL 인덱스, 파일시스템)

핵심 인사이트: 디스크 I/O는 메모리 접근보다 10만 배 느립니다. Red-Black Tree는 메모리에서는 빠르지만 디스크에서 20번 I/O하면 치명적. B-Tree는 한 노드가 디스크 블록(4~8KB) 크기에 맞춰 설계되어 3번의 디스크 읽기로 완료합니다.

Go에서의 트리:
• map: 내부적으로 해시 테이블 (트리 아님)
• container/heap: 힙 트리 (우선순위 큐)
• BoltDB(etcd): B+Tree 구현
• Red-Black Tree: Go 표준에 없음. 직접 구현 또는 3rd party. Linux CFS의 vruntime 관리에 사용
● Queue & Stack

큐 & 스택

Kafka가 큐인 이유, K8s workqueue, BFS/DFS — 인프라에서의 큐/스택.

큐와 스택이 인프라에서 어떻게 사용되는지 설명해 주세요.
큐(FIFO)는 Kafka 파티션(메시지 순서 보장), K8s Controller의 workqueue(Reconcile 대상 순차 처리), 네트워크 패킷 버퍼에 사용됩니다. 스택(LIFO)은 함수 호출 스택, Terraform 의존성 해결(DFS 기반), undo 연산에 사용됩니다.
인프라에서의 큐:
• Kafka 파티션: append-only 로그 = 큐. Consumer가 offset으로 순서대로 읽음. 파티션 내 순서 보장
• K8s workqueue: Controller의 Reconcile 대상을 큐에 넣고 Worker가 순차 처리. RateLimiting Queue로 재시도 시 exponential backoff
• NIC Ring Buffer: 네트워크 패킷을 큐잉. 가득 차면 드롭
• BFS (Breadth-First Search): 큐를 사용. K8s에서 서비스 의존성을 탐색할 때 가까운 의존성부터 확인

인프라에서의 스택:
• 함수 호출 스택: goroutine 스택 (초기 2KB, 자동 확장)
• DFS (Depth-First Search): 스택을 사용. Terraform이 리소스 의존성을 해결할 때 깊이 우선 탐색
• Undo/Rollback: 변경 사항을 스택에 쌓고 역순으로 되돌리기

// Go: 간단한 큐 (slice 기반) type Queue[T any] struct { items []T } func (q *Queue[T]) Enqueue(item T) { q.items = append(q.items, item) } func (q *Queue[T]) Dequeue() T { item := q.items[0] q.items = q.items[1:] return item } // Go: channel을 큐로 사용 (더 Go-idiomatic) ch := make(chan string, 100) // buffered channel = bounded queue ch <- "task1" // enqueue task := <-ch // dequeue (blocking)
● Graph & DAG

그래프 & DAG

Argo Workflows가 DAG인 이유, 위상 정렬, 의존성 해결.

DAG(Directed Acyclic Graph)란 무엇이고, 인프라에서 어떻게 사용되나요?
DAG는 방향이 있고 순환이 없는 그래프입니다. "A를 먼저 하고 B를 한다"같은 의존성 관계를 표현합니다. 순환이 없으므로 위상 정렬(Topological Sort)로 실행 순서를 결정할 수 있습니다. Terraform, Argo Workflows, CI/CD 파이프라인, K8s Controller의 Owner Reference 체인이 모두 DAG입니다.
DAG 예시 (Terraform): VPC ──→ Subnet ──→ EKS Cluster ──→ Node Group │ │ └──→ Security Group ───┘ 위상 정렬 결과 (실행 순서): 1. VPC 2. Subnet, Security Group (병렬 가능) 3. EKS Cluster 4. Node Group
인프라에서의 DAG:
• Terraform: 리소스 의존성을 DAG로 구성. terraform graph로 시각화. 위상 정렬 순서로 생성, 역순으로 삭제
• Argo Workflows: DAG 타입 워크플로에서 step 간 의존성을 정의. 의존성 없는 step은 병렬 실행
• Jenkins Pipeline: stage 간 의존성이 DAG
• K8s Garbage Collection: Owner Reference가 DAG 구조. Deployment → ReplicaSet → Pod. 부모 삭제 시 GC가 DAG를 역순으로 자식 삭제
• Git: 커밋 히스토리가 DAG (병합 커밋이 여러 부모를 가짐)

위상 정렬 (Topological Sort):
DAG에서 의존성을 만족하는 실행 순서를 결정하는 알고리즘. O(V+E).
순환이 있으면 위상 정렬 불가능 → 순환 의존성 감지에 사용 (Terraform이 순환 참조를 에러로 보고하는 것)

// Go: 간단한 위상 정렬 (Kahn's Algorithm) func topoSort(graph map[string][]string) []string { inDegree := make(map[string]int) for node, deps := range graph { if _, ok := inDegree[node]; !ok { inDegree[node] = 0 } for _, dep := range deps { inDegree[dep]++ } } var queue, result []string for n, d := range inDegree { if d == 0 { queue = append(queue, n) } } for len(queue) > 0 { node := queue[0]; queue = queue[1:] result = append(result, node) for _, dep := range graph[node] { inDegree[dep]-- if inDegree[dep] == 0 { queue = append(queue, dep) } } } return result }
● Sorting

정렬 알고리즘

퀵소트 vs 머지소트 — 시간/공간 복잡도, 안정 정렬이 중요한 경우.

퀵소트와 머지소트를 비교하고, 안정 정렬(stable sort)이 중요한 경우를 설명해 주세요.
퀵소트는 평균 O(N log N), 최악 O(N²)이지만 in-place(추가 메모리 O(log N))라 실무에서 가장 많이 사용됩니다. 머지소트는 항상 O(N log N)이지만 O(N) 추가 메모리가 필요합니다. 안정 정렬은 같은 키의 원래 순서를 유지하는 것으로, DB 정렬이나 다중 기준 정렬에서 중요합니다. Go의 sort.Slice()는 비안정, sort.SliceStable()은 안정 정렬입니다.
알고리즘평균최악공간안정실무
퀵소트O(N log N)O(N²)O(log N)❌Go sort.Slice, C qsort
머지소트O(N log N)O(N log N)O(N)✅Go sort.SliceStable, Java Arrays.sort(Object)
팀소트O(N log N)O(N log N)O(N)✅Python sorted(), Java Arrays.sort
힙소트O(N log N)O(N log N)O(1)❌우선순위 큐 (K8s scheduler)

안정 정렬이 중요한 경우:
로그를 먼저 timestamp 순 정렬하고, 같은 timestamp 내에서 severity 순 정렬할 때. 안정 정렬이면 두 번째 정렬이 첫 번째 정렬 결과를 보존합니다. 비안정 정렬이면 같은 severity 내에서 timestamp 순서가 깨질 수 있습니다.

Go 코드:
// 비안정 정렬 (퀵소트 기반) sort.Slice(pods, func(i, j int) bool { return pods[i].CPU > pods[j].CPU // CPU 사용량 내림차순 }) // 안정 정렬 (머지소트 기반) sort.SliceStable(logs, func(i, j int) bool { return logs[i].Timestamp.Before(logs[j].Timestamp) }) // K8s scheduler에서: 우선순위 큐 (힙) 사용 // Pod을 우선순위별로 스케줄링할 때 heap.Interface 구현
면접 포인트: "왜 Go의 기본 sort.Slice()는 비안정인가?" → 퀵소트가 in-place(메모리 절약)이고 평균적으로 가장 빠르기 때문. 안정성이 필요하면 명시적으로 SliceStable()을 호출.
