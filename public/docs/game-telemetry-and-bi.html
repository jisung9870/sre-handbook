● Overview

게임 텔레메트리 & BI란?

플레이 행동·결제·시스템 이벤트를 수집하고 KPI로 변환해 운영·비즈니스 의사결정을 데이터 기반으로 만드는 인프라. 게임 운영의 거의 모든 결정(점검·이벤트·밸런싱·과금 설계)은 이 데이터 위에서 이루어진다.

텔레메트리란?

정의

게임 클라이언트와 서버에서 발생하는 모든 이벤트를 시계열로 수집하는 인프라. 플레이 행동(레벨업·아이템 획득), 시스템(에러·로딩 시간), 결제, 광고 등을 포함.

BI(Business Intelligence)와의 관계

텔레메트리는 raw 이벤트 수집 인프라, BI는 이를 KPI·대시보드·실험으로 가공하는 분석 레이어. 둘은 한 묶음으로 운영된다.

SRE의 역할

플랫폼 SRE는 수집 파이프라인의 가용성·정합성·비용을 책임진다. 분석 자체는 데이터팀·BI팀이 하지만, 데이터가 들어오지 않으면 모든 의사결정이 멈춘다.

왜 일반 웹 분석과 다른가

한 세션에 수백~수천 이벤트가 발생. 결제·인벤토리는 strict 정합성 필요. 가챠 확률은 법적 검증 대상. 일반 웹 분석보다 볼륨·정합성·규제 모두 강하다.

왜 BI가 게임에서 특히 중요한가

① 매출 직결

F2P(Free-to-Play) 모델은 소수 페이어가 매출의 대부분(상위 1%가 50% 이상)을 만든다. ARPPU·LTV·리텐션을 정확히 측정하지 않으면 광고비·이벤트 ROI 판단이 안 된다.

② LiveOps 사이클

매주 이벤트·점검·핫픽스가 도는데, 각 이벤트의 효과를 즉시 측정해야 다음 이벤트를 설계할 수 있다. 텔레메트리가 곧 LiveOps의 피드백 루프다.

③ 밸런싱

"이 캐릭터가 너무 강하다", "이 던전이 너무 어렵다" — 모두 텔레메트리로 검증. 사용자 감정과 데이터가 다를 때 데이터 우선이라는 원칙이 있을 정도.

④ 컴플라이언스

한국의 가챠 확률 공시(2024-03-22 시행)는 텔레메트리에서 실제 드롭률을 검증한다. 표시값과 실제값이 다르면 시정명령 대상.

데이터 흐름 큰 그림

게임 클라이언트에서 발생한 이벤트가 KPI 대시보드까지 가는 표준 흐름. 각 단계는 별개 시스템이고, 어디든 끊기면 데이터가 사라진다.

수집 파이프라인

클라이언트 SDK부터 데이터 웨어하우스까지의 5단계. 단계마다 SLO와 비용 통제 정책이 따로 정의되어야 한다.

① 클라이언트 SDK

핵심 책임

이벤트 발행 API 제공, 로컬 버퍼링, 배치 전송(N개 모이면 또는 N초 마다), 실패 시 재전송, 디바이스/세션 ID 관리, 타임존·타임스탬프 정확성.

설계 포인트

오프라인 큐(앱 종료 후 다시 시작 시 미전송 이벤트 보내기), 지수 백오프, payload 압축(gzip), 배터리·네트워크 절약 모드.

흔한 함정

클라이언트 시간이 틀린다(폰 시계 변경). 이벤트의 발생 시각은 클라가 보내고, 도달 시각은 서버가 기록해서 둘 다 저장해야 한다.

규제 대응

iOS는 App Tracking Transparency(ATT) 동의 없으면 IDFA 미수집. 광고 ID 없는 사용자도 게임 내 KPI는 측정 가능하도록 자체 anonymous ID 체계 필요.

② 수집 게이트웨이

핵심 책임

HTTP/gRPC 엔드포인트, 인증(SDK 키·게임 ID), throttle(악성 트래픽 차단), 페이로드 검증, 다음 단계(큐) 전달.

아키텍처

대부분 stateless. region별 endpoint를 두고 가까운 엣지로 보내야 latency·비용 둘 다 유리. CDN을 앞에 두기도 함.

흔한 패턴

완전 자체 구축(고비용/유연), Snowplow/Segment 같은 오픈소스/SaaS, 또는 클라우드 native(Cloud Run + Pub/Sub).

흔한 함정

게이트웨이가 다운되면 클라이언트는 모든 이벤트를 잃지 않고 로컬 버퍼에 쌓이게 해야 한다. 게이트웨이 복구 후 한꺼번에 몰리는 폭주(thundering herd) 대비 — 점진 재전송, jitter.

③~⑤ 큐 → ETL → 웨어하우스

버퍼링·변환·저장의 3단계. 하나라도 막히면 데이터가 손실되거나 지연된다.

③ 큐 (Kafka / Kinesis / Pub/Sub)

게이트웨이와 ETL 사이의 버퍼·역압 흡수. ETL이 다운되어도 큐에서 며칠은 보관 가능. partition 설계(게임 ID·이벤트 타입 기준)가 throughput을 결정.

④ ETL · 스트림 처리

큐에서 읽어 스키마 검증·변환·enrichment(예: 디바이스 ID → 사용자 ID 매핑) 후 웨어하우스에 적재. 도구: Flink, Spark Streaming, Dataflow, dbt(배치).

⑤ 데이터 웨어하우스 (BigQuery / Snowflake / ClickHouse)

장기 저장 + 분석 쿼리 엔진. BigQuery: serverless, 인프라 무관, Google 생태계. Snowflake: 멀티 클라우드, 2026년부터 interactive tables/warehouses로 low-latency 강화. ClickHouse: OLAP 특화, sub-second 응답에 1000+ QPS, 실시간 대시보드 + 분석 쿼리 동시 가능. 게임 분석은 케이스별로 선택.

♥ KEY
실시간 대시보드(분 단위)와 분석 쿼리(시간/일 단위)는 다른 패스로 처리하는 게 표준. 실시간은 큐에서 직접 읽어 ClickHouse나 in-memory store(Redis/Druid)에, 분석은 배치로 BigQuery/Snowflake에 넣는다.

비용 통제

텔레메트리는 게임 인프라 비용에서 가장 빨리 폭증하는 영역. 이벤트는 자유롭게 추가하다 보면 월 비용이 매년 N배가 된다.

전략설명주의
샘플링덜 중요한 이벤트는 1/10·1/100 샘플링 후 가중치 처리개별 사용자 추적이 필요한 이벤트는 샘플링 금지(결제·계정 변경 등)
TTL · 파티션raw 이벤트는 90일 보관, aggregate는 영구 보관법적 보관 의무 데이터(결제·로그인) 별도 정책
이벤트 거버넌스새 이벤트 추가 시 PR 리뷰, 페이로드 크기 제한, 카탈로그 등록거버넌스 없으면 6개월 내 이벤트 수가 5배 늘어남
스토리지 계층화최근 데이터는 hot 스토리지, 오래된 건 cold/object storage로 이전분석 쿼리 시 archive 데이터 접근은 별도 워크플로우
쿼리 비용 통제BigQuery 슬롯 / Snowflake credit 사용량 모니터링, 비싼 쿼리 알림분석가가 SELECT *로 매달 수천 달러를 태우는 사례 흔함
⚠ PITFALL
"일단 다 수집하고 나중에 보자"는 가장 비싼 패턴이다. 이벤트 추가는 쉽지만 제거는 어렵다. 처음부터 이벤트당 비용을 추산하고 거버넌스를 도입하는 게 6개월 후 비용 폭탄을 막는 유일한 방법.
● Event Schema Governance

이벤트 스키마 거버넌스

이벤트 카탈로그·명명 규칙·버저닝·검증을 통해 데이터 품질을 유지한다. 거버넌스 없는 텔레메트리는 6개월 후 누구도 분석 못 하는 데이터 늪이 된다.

이벤트 카탈로그 & 명명 규칙

카탈로그란?

모든 이벤트의 정의·필드·발생 조건·소유자를 한 곳에 정리한 문서/시스템. Confluence·GitHub repo·전용 도구(Atlan, Collibra) 등.

왜 필요한가

"이 이벤트가 정확히 언제 발생하는지" 모르면 분석 불가능. 카탈로그가 없으면 같은 이벤트를 여러 팀이 다른 의미로 사용하는 사고가 발생한다.

표준 명명 규칙 예시
# 권장: snake_case + 동사_명사 event_name: "item_purchased" # O event_name: "user_logged_in" # O event_name: "level_completed" # O # 비권장 event_name: "ItemBuy" # X (PascalCase, 동사 모호) event_name: "item" # X (너무 일반적) event_name: "purchase_v2_final" # X (버저닝은 별도 필드) # 권장 페이로드 구조 { "event_name": "item_purchased", "event_version": 2, "timestamp_client": "2026-05-03T12:34:56.789Z", "timestamp_server": "2026-05-03T12:34:57.123Z", "user_id": "u_xxx", "session_id": "s_xxx", "game_id": "game_a", "properties": { "item_id": "sword_001", "price_krw": 5500, "currency": "KRW", "payment_method": "google_play" } }
★ TIP
새 이벤트 추가는 PR 리뷰. 카탈로그 등록 없는 이벤트는 ETL에서 자동 reject하는 게이트도 흔하다. "이 이벤트가 누구 거예요?"라는 질문이 6개월 후에 안 나오게 하는 핵심.

버저닝 & 호환성

왜 버저닝이 필요한가

이벤트 페이로드는 시간이 지나며 변한다. 구버전 클라이언트는 여전히 옛 스키마로 보내는데, 분석은 신구 모두 처리해야 한다.

호환성 원칙

필드 추가는 OK(default 값 처리), 필드 삭제·타입 변경은 금지. 정말 변경이 필요하면 새 이벤트로 분리(item_purchased_v2). 모바일은 강제 업데이트가 어려워 호환성이 더 중요하다.

schema registry

중앙 schema registry(Confluent Schema Registry, AWS Glue Schema Registry)에 버전별 스키마를 등록하고 ETL에서 검증. JSON Schema, Avro, Protobuf 모두 가능.

ETL의 스키마 진화

웨어하우스 테이블의 스키마도 함께 진화. BigQuery·Snowflake는 컬럼 추가는 자동 지원, 타입 변경은 마이그레이션 필요.

클라/서버 이중 검증

결제·아이템 획득·전투 결과 같은 매출·정합성에 영향 있는 이벤트는 클라 단독 발행 금지. 서버 검증을 거친 이벤트만 신뢰.

클라 이벤트의심 가능
서버 검증인벤토리·결제·로그
서버 이벤트 발행신뢰 가능
텔레메트리 적재두 패스 분리
분석차이 = 어뷰징 신호
⚠ CRITICAL
클라 이벤트만 보고 매출·정합성 결정 금지. 매크로/봇/해킹된 클라이언트는 가짜 이벤트를 무한히 보낼 수 있다. "클라가 산 거"와 "서버가 청구한 거"의 차이는 anti-cheat 신호로도 활용된다.
● Game KPIs

게임 KPI · 도메인 용어

퍼블리싱 SRE도 알아야 할 핵심 KPI 정의와 공식. 다른 가이드(LiveOps, 결제, 런칭)에서 자주 등장하므로 여기서 한 번에 정리한다.

참여 (Engagement)

★ Engagement

DAU / MAU / Stickiness

유저가 얼마나 자주·꾸준히 플레이하는지의 가장 기본 지표. 모든 게임이 매일 추적한다.

DAU 정의
하루에 1회 이상 플레이한 고유 유저 수
MAU 정의
한 달 동안 1회 이상 플레이한 고유 유저 수
Stickiness
DAU / MAU — 0.2면 보통, 0.5+ 매우 높음
주의
"고유 유저"는 device_id냐 account_id냐 정의 통일 필요. 멀티 디바이스는 같은 사람이 중복 카운트될 수 있다.
★ Engagement

세션 수 · 세션 길이

유저가 한 번 켜서 얼마나 오래 플레이하는지, 하루에 몇 번 켜는지.

세션 정의
앱 부팅 ~ 백그라운드(보통 30초+) 또는 종료까지
평균 세션 길이
총 세션 시간 / 세션 수 — 게임 종류별로 차이 큼
DAU당 세션 수
일일 세션 수 / DAU — 캐주얼 게임은 5+, MMO는 1~2
활용
이벤트 효과 측정, 게임 피로도 파악, 서버 부하 예측

매출 (Revenue)

$ Revenue

ARPU (Average Revenue Per User)

전체 유저 1명당 평균 매출. 비교적 단순하지만 페이어와 비페이어를 섞어버려 해석에 주의.

공식
총 매출 / Active Users (보통 monthly: monthly revenue / MAU)
변형
ARPDAU = 일 매출 / DAU — 일별 추세 파악에 유용
한계
F2P 게임은 페이어 비율 1~5%. ARPU는 너무 낮게 보일 수 있음
활용
광고 ROI 비교, 게임 간 매출 효율 비교
$ Revenue

ARPPU (Average Revenue Per Paying User)

결제한 유저만 대상으로 한 평균 매출. F2P 게임에서 ARPU보다 훨씬 의미 있음.

공식
총 매출 / 결제 유저 수
관계
ARPU = ARPPU × 결제 전환율
의미
"한번 결제한 사람은 평균 얼마 쓰는가" — 페이어 가치 측정
주의
상위 1% 고래(whale)가 평균을 왜곡. 중앙값·구간별 분포도 함께 봐야 함
$ Revenue

LTV (Lifetime Value)

한 유저가 게임 전 생애 동안 가져다 줄 매출의 예측치. 광고비(UA cost)와의 비교로 ROI를 판단한다.

단순 공식
ARPU × 평균 수명(개월)
정밀 공식
코호트별 retention curve 적분 + 미래 예측 모델 (ML 기반)
DN-LTV
"가입 후 N일까지의 LTV" — D7-LTV, D30-LTV 등이 광고 입찰에 직접 사용
핵심 원칙
LTV > UA Cost가 성립해야 광고 ROI 양수. ATT 시대에 측정 정밀도가 떨어져 더 어려워짐

리텐션 & 코호트

유저가 가입 후 며칠 뒤에 다시 돌아오는지의 비율. 게임의 "재미" 척도이자 LTV의 가장 큰 변수.

지표정의업계 통상치 (모바일 게임)
D1 Retention가입 다음날 돌아오는 비율30~50% (좋음 50%+)
D7 Retention가입 7일 후 돌아오는 비율10~20% (좋음 20%+)
D30 Retention가입 30일 후 돌아오는 비율3~10% (좋음 10%+)
★ NOTE
위 통상치는 장르·타겟·플랫폼·년도에 따라 크게 다르다. 캐주얼 퍼즐은 D1 50%+가 흔하고 하드코어 RPG는 D1 30%만 되어도 잘 만든 편. 경쟁작 벤치마크가 아니라 자기 코호트 추이가 더 중요하다.

코호트(Cohort) 분석

가입일 기준으로 유저를 그룹화해 시간에 따라 추적. "2026-05-03 가입 코호트의 D7 retention은 25%" 같은 식. 신작 패치·이벤트 효과 측정의 표준.

리텐션 vs 처닝

리텐션의 반대는 처닝(churn). 둘은 동전의 양면. D30 리텐션 5%면 D30 churn은 95%. 어느 쪽 단어를 쓰든 같은 정보.

Funnel & Conversion

유저가 다단계 행동(가입 → 튜토리얼 완료 → 첫 결제)에서 각 단계 통과율을 추적. 게임의 어디서 유저를 잃는지 파악.

단계이벤트통과율 (예시)
0. 다운로드install100%
1. 첫 부팅app_first_open~75%
2. 튜토리얼 완료tutorial_completed~50%
3. 첫 세션 종료 (5분+)session_5min~40%
4. 첫 결제 (D7 이내)first_purchase~3%
♥ KEY
각 단계 통과율은 UA(광고 채널)·OS·국가별로 다르다. 전체 funnel만 보면 못 보는 패턴이 세그먼트별로 나타남. funnel 단계는 게임마다 다르게 정의해야 한다 (RPG는 "전직 완료"가 funnel에 들어갈 수 있음).

가챠 확률 표시 의무 (한국, 2024-03-22 시행)

한국에서 게임산업법 개정으로 확률형 아이템의 확률 정보 표시가 법적 의무가 되었다. 텔레메트리는 이를 검증·증빙하는 핵심 시스템.

대상

모든 확률형 아이템: 캡슐형(가챠), 강화형(효과/성능 변경), 합성형(아이템 조합으로 무작위 결과). 게임 본체·홈페이지·광고 모두에 표시 의무.

위반 시

표시하지 않거나 거짓 표시 시 문화체육관광부 시정 권고/명령. 게임물관리위원회 24명 모니터링단 운영. 2026년에는 위반에 관한 소송 특례도 시행 중.

SRE 관점 책임

① 텔레메트리에서 실제 드롭률을 검증(표시값 ±N% 이내인지). ② 가챠 결과는 서버 권위여야 함(클라 RNG 금지). ③ 결과 로그는 법적 보관 의무 — 별도 보관 정책 필요.

실무 체크리스트

대시보드에 가챠별 실제 vs 표시 확률 비교. 알림 임계치 설정. 신규 가챠 출시 시 표시 검수 워크플로우. CS팀과 사용자 문의 응대 매뉴얼 공유.

⚠ NOTE
이는 한국 한정 규제. 글로벌 출시 시에는 중국(2017년 도입), 일본(자율 규제)도 비슷한 정책이 있고 EU도 논의 중. 글로벌 텔레메트리는 region별 규제 매트릭스를 별도로 관리한다.
● Dashboards & Experimentation

대시보드 & A/B 테스트

수집한 데이터를 의사결정에 연결하는 마지막 단계. 운영 대시보드와 분석 대시보드는 분리 운영하고, A/B 테스트는 게임 도메인의 특수성을 인정해야 한다.

대시보드 분리 (운영 vs 분석)

운영 대시보드 (Real-time Ops)

SRE/온콜이 항상 보는 대시보드. 가용성·지연·결제 성공률·오류율. 분 단위 갱신, 알림 연동. 데이터는 큐 → 실시간 store(ClickHouse, Druid, Prometheus).

분석 대시보드 (BI · 일배치)

BI/PM/사업이 보는 대시보드. DAU/MAU/ARPU/리텐션/funnel. 일/주 단위 갱신, 더 깊은 SQL. 데이터는 웨어하우스(BigQuery/Snowflake/ClickHouse).

★ TIP
두 대시보드를 한 시스템에 욱여넣으면 둘 다 망가진다. 실시간은 latency·alert이 중요하고 분석은 깊은 쿼리·복잡한 join이 중요. 도구 선택과 SLO 모두 다르게 잡아야 한다.

A/B 테스트

새 기능·이벤트·UI를 일부 유저에게만 노출하고 KPI 차이를 측정. 게임은 LiveOps 사이클이 빨라 실험 인프라가 매우 중요.

플랫폼 종류

warehouse-native(GrowthBook, Statsig — 기존 웨어하우스 위에서 동작, 비용 효율적, 오픈소스), 통합형(Optimizely — enterprise, 고가), 자체 구축(대형 게임사). 2026년 트렌드는 warehouse-native.

구조

① 변형 정의 → ② 사용자에게 변형 노출 (해시 기반 분배) → ③ 노출 + 결과 이벤트 로깅 → ④ 통계 검증(p-value, CI) → ⑤ 결과 결정 (롤아웃/롤백).

게임 도메인 주의

① 밸런스 영향 실험은 위험 — 일부 유저만 OP 캐릭터 받으면 형평성 이슈. ② 결제·가챠 확률 실험은 법적 리스크. ③ 단기 KPI(D1 retention)와 장기 KPI(D30 retention, LTV)가 다르게 움직일 수 있다.

표본 크기

F2P는 결제 유저가 적어 결제 KPI 통계 검증에 큰 표본이 필요. ARPU의 5% 변화를 95% 신뢰도로 검증하려면 게임마다 다르지만 보통 수만~수십만 유저가 필요하다.

⚠ CRITICAL
가챠 확률·결제 가격·캐릭터 능력치를 임의로 A/B 테스트하면 한국 게임산업법(확률 표시 의무)·전자상거래법·약관법 모두에 저촉될 수 있다. 컴플라이언스 검토 없이 결제·확률 실험 금지.

ATT 시대 측정 — 더 어려워진 LTV

Apple iOS 14.5(2021-04)부터 시행된 ATT(App Tracking Transparency)로 IDFA 수집이 동의 기반이 됨. 게임 텔레메트리·UA 측정에 큰 영향.

현재 상황 (2026)

글로벌 ATT 동의율 ~50% (게임은 약간 높아 19%, 일반 앱 12% 기준 — 2025 통계). 동의율은 천천히 증가 중이나 여전히 절반 이상은 IDFA 미수집.

영향

광고 채널별 LTV·ROI 측정 정밀도 하락. 개별 사용자 retargeting 불가. 광고 입찰 기준이 흔들림. D7-LTV·D30-LTV의 신뢰도가 ATT 이전보다 낮아짐.

대안

① SKAdNetwork / AdAttributionKit(Apple 공식, aggregate 측정), ② probabilistic 모델, ③ server-side 측정(자체 SDK), ④ MMM(Media Mix Modeling), ⑤ incrementality testing(geo lift), ⑥ first-party 데이터 강화 (회원 가입·로그인 기반).

SRE 관점

광고 attribution 시스템(AppsFlyer, Adjust, Singular)과의 통합·정합성 모니터링이 점점 중요. SKAdNetwork 콜백 처리 인프라도 별도로 운영해야 함. "측정할 수 있는 것"과 "측정할 수 없는 것"의 경계가 매년 바뀐다.
