● Virtual Memory

가상 메모리

페이지 테이블, 페이지 폴트, TLB, 스왑 — 프로세스가 물리 메모리보다 큰 주소 공간을 사용할 수 있는 이유.

가상 메모리란 무엇이고, 왜 필요한가요?
프로세스에게 연속적이고 독립적인 주소 공간을 제공하는 추상화 계층입니다. 물리 메모리를 직접 접근하지 않고 가상 주소를 사용하여 프로세스 간 격리, 물리 메모리보다 큰 주소 공간 사용, 메모리 공유를 가능하게 합니다.
RTOS와의 차이: STM32 같은 MCU에서는 MPU(Memory Protection Unit)로 제한된 보호만 하거나 보호 없이 직접 물리 주소를 사용합니다. 리눅스는 MMU(Memory Management Unit) 하드웨어가 모든 메모리 접근을 가상→물리로 변환합니다.

핵심 동작:
1. 프로세스가 가상 주소 0x7fff1234에 접근
2. CPU의 MMU가 페이지 테이블을 조회하여 물리 주소로 변환
3. 해당 페이지가 물리 메모리에 없으면 → 페이지 폴트 발생
4. 커널이 디스크(스왑)에서 해당 페이지를 물리 메모리로 로드

K8s에서의 연결: 컨테이너도 프로세스이므로 가상 메모리를 사용합니다. container_memory_rss는 실제 물리 메모리에 매핑된 페이지 크기이고, container_memory_usage_bytes는 Page Cache를 포함합니다. 이 차이를 이해해야 OOMKill 원인을 진단할 수 있습니다.
페이지 테이블의 구조를 설명해 주세요. 왜 멀티레벨인가요?
가상 주소를 물리 주소로 변환하는 매핑 테이블입니다. 단일 레벨이면 64비트 주소 공간에서 테이블 크기가 수 PB이므로, 4~5단계 계층 구조(PGD→PUD→PMD→PTE)로 사용하지 않는 주소 범위의 테이블을 아예 생성하지 않아 메모리를 절약합니다.
x86-64 4-Level 페이지 테이블:
가상 주소 (48비트 유효): [PGD 9비트][PUD 9비트][PMD 9비트][PTE 9비트][Offset 12비트] PGD (Page Global Directory) → 512 엔트리 → PUD (Page Upper Directory) → 512 엔트리 → PMD (Page Middle Directory) → 512 엔트리 → PTE (Page Table Entry) → 512 엔트리 → 물리 페이지 페이지 크기: 4KB (2^12 = Offset 12비트) Huge Page: 2MB (PMD 레벨에서 직접 매핑) 또는 1GB (PUD 레벨)
Huge Pages가 중요한 이유: 일반 4KB 페이지로 수 GB 메모리를 매핑하면 PTE 엔트리가 수백만 개 필요하고 TLB 미스가 빈번합니다. 2MB Huge Page를 쓰면 엔트리 수가 512배 감소하여 TLB 히트율이 극적으로 개선됩니다. PostgreSQL, Redis 같은 대용량 메모리 앱에서 Huge Pages를 활성화하면 성능이 10~20% 향상될 수 있습니다.
페이지 폴트(Page Fault)의 종류와 성능 영향은?
Minor Fault는 물리 메모리에 페이지가 있지만 매핑만 없는 경우(수 μs), Major Fault는 디스크에서 페이지를 읽어와야 하는 경우(수 ms)입니다. Major Fault가 빈번하면 시스템이 극적으로 느려집니다.
Minor Page Fault: mmap()으로 파일을 매핑했지만 실제로 접근하지 않은 페이지를 처음 접근할 때. 또는 fork() 후 Copy-on-Write가 트리거될 때. 페이지가 이미 물리 메모리에 있으므로 PTE만 업데이트. ~1~10μs

Major Page Fault: 페이지가 디스크(스왑)에 있어서 디스크 I/O가 필요. HDD에서 ~5~10ms, SSD에서 ~0.1~1ms. 하지만 빈번하면 시스템 전체가 I/O wait으로 응답 불가.

K8s에서: container_memory_swap이 증가하면 major fault가 빈번하다는 뜻. 스왑은 K8s 노드에서 기본적으로 비활성화(kubelet --fail-swap-on=true)되므로, 메모리 부족 시 스왑 대신 OOMKill이 발생합니다.

모니터링:
ps -o min_flt,maj_flt -p PID — 프로세스별 fault 수
sar -B 1 — 시스템 전체 page fault 통계
Prometheus: node_vmstat_pgmajfault — major fault 급증 시 알림
TLB(Translation Lookaside Buffer)가 뭐고, 왜 성능에 중요한가요?
페이지 테이블 조회 결과를 캐시하는 CPU 내부의 고속 캐시입니다. 매 메모리 접근마다 4단계 페이지 테이블을 탐색하면 4번의 추가 메모리 접근이 필요하므로, TLB 히트 시 이를 건너뛰어 성능을 유지합니다. TLB 미스율이 높으면 모든 메모리 접근이 5배 느려집니다.
TLB 특성:
• 크기: 보통 64~1536 엔트리 (CPU마다 다름)
• 히트 시간: ~0.5~1ns (L1 캐시와 동급)
• 미스 시간: ~10~100ns (페이지 테이블 walk)
• 히트율: 일반적으로 >99% (지역성이 좋은 앱)

TLB Flush (Shootdown): 컨텍스트 스위칭 시 TLB 전체 또는 일부가 무효화됩니다. 이것이 컨텍스트 스위칭 비용의 상당 부분. ASID(Address Space ID)를 지원하는 CPU는 프로세스별 TLB 엔트리를 유지하여 flush를 최소화합니다.

Huge Pages와 TLB: 4KB 페이지로 1GB를 커버하려면 262,144 TLB 엔트리가 필요하지만, 2MB Huge Page로는 512 엔트리만 필요. TLB는 크기가 매우 제한적이므로 Huge Pages가 TLB 히트율을 극적으로 개선합니다.

RTOS 비교: STM32의 MPU는 리전 수가 8~16개로 매우 적고 TLB가 없습니다. 리눅스는 MMU+TLB가 하드웨어에 내장되어 자동으로 동작합니다.
스왑(Swap)이란? K8s에서 스왑을 비활성화하는 이유는?
물리 메모리가 부족할 때 사용 빈도가 낮은 페이지를 디스크로 옮기는 메커니즘입니다. K8s에서 스왑을 비활성화하는 이유는 kubelet의 메모리 관리(resource request/limit, OOMKill, eviction)가 정확한 물리 메모리 사용량에 의존하는데, 스왑이 활성화되면 실제 메모리 부족 시점을 정확히 판단할 수 없기 때문입니다.
스왑 동작:
1. 물리 메모리 부족 → kswapd(커널 스레드)가 페이지 회수 시작
2. Anonymous pages(힙, 스택)를 스왑 영역(디스크)으로 이동
3. File-backed pages(Page Cache)는 원본 파일이 있으므로 단순히 drop
4. vm.swappiness: 0(스왑 최소화)~100(적극 스왑). 기본 60

K8s 1.28+에서 스왑 지원 (alpha):
• NodeSwap feature gate로 제한적 스왑 허용
• LimitedSwap 모드: Burstable QoS Pod만 스왑 사용 허용
• 아직 프로덕션 미권장. 대부분의 환경에서 스왑 비활성화가 표준

모니터링:
free -h → Swap 사용량
vmstat 1 → si(swap in), so(swap out) 컬럼. 0이 아니면 메모리 부족
node_memory_SwapFree_bytes → Prometheus 메트릭
● Process & Thread

프로세스 vs 스레드

차이점, 컨텍스트 스위칭 비용, Go의 goroutine — 면접 최빈출 주제.

프로세스와 스레드의 차이를 설명해 주세요.
프로세스는 독립된 주소 공간, 파일 디스크립터, 시그널 핸들러를 가진 실행 단위입니다. 스레드는 같은 프로세스 내에서 코드, 데이터, 힙을 공유하면서 각자의 스택과 레지스터를 가진 실행 흐름입니다. 리눅스에서는 둘 다 task_struct로 표현되며, clone() syscall의 플래그로 공유 범위를 결정합니다.
항목프로세스스레드
주소 공간독립 (격리)공유 (같은 프로세스 내)
생성 비용높음 (페이지 테이블 복사)낮음 (스택만 할당)
통신IPC (pipe, socket, shm)공유 메모리 직접 접근
장애 격리한 프로세스 크래시 → 다른 프로세스 무관한 스레드 크래시 → 전체 프로세스 종료
Linux 내부clone(CLONE_NEWPID...)clone(CLONE_VM|CLONE_FS|CLONE_FILES)

리눅스의 특수성: 리눅스는 프로세스와 스레드를 구분하지 않습니다. 둘 다 task_struct이며 clone()의 플래그로 공유 범위만 다릅니다. CLONE_VM이면 주소 공간을 공유(=스레드), 없으면 복사(=프로세스).

RTOS 비교: FreeRTOS에서 "task"는 리눅스의 스레드에 가깝습니다. 모든 task가 같은 주소 공간을 공유하고 MPU로만 보호합니다. 리눅스의 프로세스 격리(MMU 기반)는 RTOS에는 없는 강력한 보안 경계입니다.

K8s 컨테이너: 컨테이너는 프로세스입니다 (namespace + cgroup으로 격리된). 같은 Pod의 컨테이너들은 Network Namespace를 공유하지만 PID Namespace는 분리됩니다.
Go의 goroutine은 스레드와 어떻게 다른가요?
goroutine은 Go 런타임이 관리하는 경량 "그린 스레드"입니다. OS 스레드 위에 M:N 멀티플렉싱(여러 goroutine을 적은 OS 스레드에 매핑)으로 동작합니다. 생성 비용이 ~2KB 스택으로 시작하여 OS 스레드(~1~8MB)보다 500배 이상 저렴하고, 수십만 개를 동시에 운영할 수 있습니다.
Go GMP 모델:
• G (Goroutine): 실행할 코드 + 스택. 초기 스택 2KB, 필요하면 자동 확장
• M (Machine): OS 스레드. GOMAXPROCS만큼 활성 (기본 = CPU 코어 수)
• P (Processor): 로컬 run queue를 가진 스케줄러 컨텍스트. M과 1:1 매핑

goroutine이 블록되면:
1. syscall (네트워크 I/O 등)로 goroutine이 블록
2. Go 런타임이 해당 M에서 goroutine을 분리
3. M은 다른 goroutine을 실행 (네트워크 I/O는 netpoller가 비동기 처리)
4. I/O 완료 → goroutine을 다시 run queue에 넣음

OpsGate/내부 도구에서: Go로 만드는 도구에서 HTTP 핸들러마다 goroutine이 생성됩니다. 동시 1000개 요청이 와도 1000개 goroutine이 ~2MB만 사용. OS 스레드로는 ~8GB가 필요했을 것.
컨텍스트 스위칭의 비용을 설명해 주세요. 프로세스 간 vs 스레드 간 차이는?
컨텍스트 스위칭은 CPU가 현재 실행 중인 작업의 상태(레지스터, PC, 스택 포인터)를 저장하고 다른 작업의 상태를 복원하는 과정입니다. 프로세스 간 전환은 주소 공간이 바뀌므로 TLB flush + 페이지 테이블 전환이 추가되어 ~5~10μs, 스레드 간 전환은 주소 공간이 같으므로 ~1~3μs로 더 가볍습니다.
비용 구성:
Direct Cost: 레지스터 저장/복원: ~0.5μs 커널 모드 전환: ~0.5μs TLB flush (프로세스 간): ~1~3μs 페이지 테이블 전환 (프로세스 간): ~0.5μs Indirect Cost (더 비쌈): CPU 캐시 오염: ~10~100μs (새 프로세스의 데이터가 캐시에 없음) TLB 미스 증가: ~5~50μs (워밍업 기간) Branch prediction 미스: ~1~5μs 총 비용: Direct 2~5μs + Indirect 10~100μs
K8s 관점: 노드에 너무 많은 Pod(프로세스)를 배치하면 컨텍스트 스위칭 오버헤드가 증가합니다. node_context_switches_total 메트릭이 초당 수만을 초과하면 CPU가 실제 작업보다 전환에 시간을 더 쓸 수 있습니다. 이것이 "noisy neighbor" 문제의 한 원인입니다.
● Scheduling

프로세스 스케줄링

FIFO, SJF, RR, CFS — CPU를 프로세스에 할당하는 알고리즘.

주요 스케줄링 알고리즘을 비교해 주세요.
FIFO(선입선출)는 가장 단순하지만 짧은 작업이 긴 작업 뒤에서 대기(convoy effect). SJF(최단 작업 우선)는 평균 대기 시간 최적이지만 실행 시간 예측 불가. RR(라운드 로빈)은 시간 할당량(time quantum)만큼 번갈아 실행하여 공정하지만 컨텍스트 스위칭 빈번. CFS(리눅스 기본)는 "가장 적게 실행된 프로세스 우선"으로 공정성을 보장합니다.
알고리즘원리장점단점실무 예시
FIFO먼저 온 순서단순, 오버헤드 0Convoy Effect배치 잡 큐
SJF짧은 것 먼저평균 대기 최소실행시간 예측 불가, 기아이론적
RR시간 할당량 순환공정, 응답 시간 양호스위칭 빈번RTOS 일반적
CFS가중 공정 스케줄링공정 + 우선순위 + 효율복잡Linux 기본

RTOS 경험 연결: FreeRTOS에서는 Priority-based Preemptive 스케줄링을 사용했을 것. 리눅스 CFS는 이와 다르게 "가상 런타임(vruntime)"이 가장 적은 프로세스를 다음에 실행하여 CPU 시간을 공정하게 분배합니다.
Linux CFS(Completely Fair Scheduler)의 동작 원리를 설명해 주세요.
CFS는 각 프로세스의 "가상 런타임(vruntime)"을 추적하고, vruntime이 가장 적은 프로세스를 다음에 실행합니다. vruntime은 실제 실행 시간을 nice 값(우선순위)의 가중치로 나눈 것이므로, 높은 우선순위 프로세스는 같은 실제 시간을 써도 vruntime이 느리게 증가하여 더 많은 CPU를 받습니다. Red-Black Tree로 vruntime을 관리하여 O(log N)으로 다음 프로세스를 선택합니다.
vruntime += 실제_실행_시간 × (기본_가중치 ÷ 프로세스_가중치) nice 0 (기본): 가중치 1024 → vruntime = 실제 시간 nice -20 (최고 우선순위): 가중치 88761 → vruntime 매우 느리게 증가 nice 19 (최저 우선순위): 가중치 15 → vruntime 빠르게 증가 → 높은 우선순위 프로세스는 vruntime이 느리게 증가 → CFS는 항상 vruntime이 가장 낮은 것을 선택 → 결과적으로 높은 우선순위가 더 많은 CPU를 받음
K8s에서 CFS와 CPU Limit:
K8s의 CPU limit은 CFS bandwidth throttling으로 구현됩니다. cpu.cfs_period_us(기본 100ms) 동안 cpu.cfs_quota_us만큼만 실행을 허용합니다.
CPU limit 200m → quota 20ms/100ms → 100ms 중 20ms만 실행, 나머지 80ms는 강제 대기(throttle).

이것이 CPU Throttling의 원인입니다. 앱이 순간적으로 200m 이상을 사용하려 해도 커널이 차단합니다. nr_throttled 카운터가 증가하고 latency가 스파이크합니다.
⚠ 면접 포인트
"K8s에서 CPU limit을 설정하지 않는 전략이 있다"는 점을 언급하면 인상적입니다. CPU request로 스케줄링 보장은 하되, limit을 없애서 throttling을 방지하는 것. container_cpu_cfs_throttled_periods_total 메트릭으로 throttling을 모니터링하는 것도 함께 설명하면 좋습니다.
● Synchronization

동기화

Mutex, Semaphore, Deadlock — 공유 리소스 접근을 제어하는 메커니즘.

Mutex와 Semaphore의 차이를 설명해 주세요.
Mutex는 "잠금 소유권"이 있는 이진 잠금으로, 잠근 스레드만 해제할 수 있습니다. Semaphore는 카운터 기반으로, 카운터가 0이면 대기하고 양수이면 진입합니다. Mutex는 "하나의 스레드만 접근"을 보장하고, Semaphore는 "N개까지 동시 접근"을 허용합니다.
항목MutexSemaphore
소유권있음 (잠근 스레드만 해제)없음 (아무 스레드나 signal 가능)
값0 또는 10 ~ N (카운팅)
용도상호 배제 (임계 영역)리소스 카운팅, 시그널링
Go에서sync.Mutexgolang.org/x/sync/semaphore

Go에서의 실무 패턴:
Go에서는 Mutex보다 채널(channel)을 선호합니다. "메모리를 공유하여 통신하지 말고, 통신하여 메모리를 공유하라(Don't communicate by sharing memory; share memory by communicating)"가 Go의 철학입니다.

하지만 Mutex가 필요한 경우: 단순한 카운터, 캐시 맵 보호 같은 경우 Mutex가 채널보다 단순하고 효율적입니다. OpsGate의 in-memory 상태 관리에서 sync.RWMutex(읽기 다수/쓰기 소수)를 사용하는 것이 적절합니다.

RTOS 비교: FreeRTOS의 xSemaphoreTake()/xSemaphoreGive()와 동일한 개념이지만, 리눅스는 futex(fast userspace mutex)로 구현하여 커널 진입 없이 잠금을 획득할 수 있어 더 효율적입니다.
Deadlock의 4가지 필요 조건과 해결 방법을 설명해 주세요.
Deadlock은 상호 배제(Mutual Exclusion), 점유 대기(Hold and Wait), 비선점(No Preemption), 순환 대기(Circular Wait) 4가지 조건이 모두 만족될 때 발생합니다. 하나라도 깨면 Deadlock을 방지할 수 있습니다. 실무에서 가장 흔한 방법은 "잠금 순서를 일정하게 유지(순환 대기 깨기)"입니다.
4가지 조건 (Coffman 조건):
1) 상호 배제: 리소스를 한 번에 하나만 사용 가능
2) 점유 대기: 리소스를 가진 채로 다른 리소스를 대기
3) 비선점: 강제로 리소스를 뺏을 수 없음
4) 순환 대기: A→B→C→A 순환 의존

실무에서의 Deadlock:
• DB 트랜잭션: 트랜잭션 A가 row 1 잠금 후 row 2 요청, 트랜잭션 B가 row 2 잠금 후 row 1 요청 → PostgreSQL이 deadlock 감지하여 한쪽 rollback
• Go: goroutine A가 ch1에 쓰면서 ch2 읽기 대기, goroutine B가 ch2에 쓰면서 ch1 읽기 대기 → Go 런타임이 "all goroutines are asleep" 감지
• K8s: PDB가 모든 Pod의 eviction을 막으면 drain이 영원히 대기 (일종의 deadlock)

해결 전략:
• 잠금 순서 강제: 항상 같은 순서로 잠금 획득 → 순환 대기 방지
• 타임아웃: 일정 시간 내 잠금 못 얻으면 포기 → 점유 대기 깨기
• Lock-free: CAS(Compare-And-Swap) 기반 구현 → 잠금 자체를 제거
● Memory Management

메모리 관리

힙/스택, malloc 동작, OOM Killer — 리눅스 메모리 관리의 실무 관점.

힙(Heap)과 스택(Stack)의 차이를 설명해 주세요.
스택은 함수 호출 시 자동 할당/해제되는 LIFO 구조의 메모리 영역으로, 크기가 고정(기본 8MB)이고 매우 빠릅니다(포인터 이동만). 힙은 프로그래머가 동적으로 할당/해제하는 영역으로, 크기 제한이 없지만 할당이 느리고(malloc이 커널과 상호작용) 메모리 누수와 단편화 위험이 있습니다.
프로세스 메모리 레이아웃 (높은 주소 → 낮은 주소): ┌─────────────────┐ 0xFFFF... │ Kernel Space │ (사용자 접근 불가) ├─────────────────┤ │ Stack ↓ │ 함수 호출마다 아래로 성장. 기본 8MB │ │ 지역 변수, 리턴 주소, 매개변수 ├─────────────────┤ │ ↕ 빈 공간 │ ├─────────────────┤ │ Heap ↑ │ malloc/new로 위로 성장 │ │ 동적 할당 데이터 ├─────────────────┤ │ BSS │ 초기화되지 않은 전역/정적 변수 (0으로) ├─────────────────┤ │ Data │ 초기화된 전역/정적 변수 ├─────────────────┤ │ Text (Code) │ 실행 코드 (읽기 전용) └─────────────────┘ 0x0000...
Go의 특수성: Go 컴파일러가 escape analysis로 변수가 함수 밖에서 참조되는지 분석합니다. 밖에서 참조되면 힙에, 아니면 스택에 할당합니다. go build -gcflags="-m"으로 어떤 변수가 힙으로 이스케이프되는지 확인할 수 있습니다. 힙 할당을 줄이면 GC 부하가 감소하여 latency가 개선됩니다.

RTOS 비교: STM32에서는 힙을 최소화하고 스택 크기를 task별로 정적 할당했을 것. 리눅스에서는 가상 메모리 덕분에 훨씬 자유롭지만, K8s 환경에서 cgroup memory limit이 물리적 제한을 부여합니다.
리눅스 OOM Killer의 동작 원리와 K8s에서의 영향을 설명해 주세요.
물리 메모리와 스왑이 모두 소진되면 커널의 OOM Killer가 oom_score가 가장 높은 프로세스를 강제 종료합니다. oom_score는 메모리 사용량이 크고 우선순위가 낮은 프로세스에 높게 부여됩니다. K8s에서는 cgroup 레벨에서 OOM이 발생하여 memory limit을 초과한 컨테이너를 OOMKill하고, Pod의 QoS 클래스(Guaranteed > Burstable > BestEffort)에 따라 종료 우선순위가 결정됩니다.
OOM 점수 계산:
oom_score = 메모리 사용 비율 기반 (0~1000)
oom_score_adj = 관리자가 조정 (-1000~1000). -1000이면 OOM Kill에서 제외

K8s QoS와 OOM 우선순위:
QoS 클래스조건oom_score_adjOOM 우선순위
Guaranteedrequest = limit (CPU & Memory 모두)-997가장 나중에 Kill
Burstablerequest < limit 또는 일부만 설정2~999 (사용량 비례)중간
BestEffortrequest/limit 없음1000가장 먼저 Kill

OOMKill 진단:
kubectl describe pod → OOMKilled 상태
kubectl get events → Killing 이벤트
dmesg | grep -i oom → 커널 OOM 로그 (어떤 프로세스가 얼마나 쓰고 있었는지)
Prometheus: kube_pod_container_status_last_terminated_reason{reason="OOMKilled"}

실전 대응: OOMKill이 반복되면 memory request를 VPA Off 모드의 추천값으로 조정합니다. 앱의 메모리 프로파일링(go tool pprof /debug/pprof/heap)으로 실제 누수인지 정상 사용인지 구분합니다.
● File System

파일 시스템

Inode, 파일 디스크립터, VFS — 리눅스 I/O의 기본 구조.

Inode와 파일 디스크립터(FD)의 차이를 설명해 주세요.
Inode는 파일의 메타데이터(크기, 권한, 데이터 블록 위치)를 저장하는 디스크 상의 구조입니다. 파일 디스크립터(FD)는 프로세스가 열린 파일에 접근하기 위한 정수 핸들로, 프로세스의 FD 테이블 → 시스템 오픈 파일 테이블 → Inode 순서로 연결됩니다. 파일 이름은 Inode에 저장되지 않고 디렉토리(Dentry)에 저장됩니다.
fd = open("/etc/hosts", O_RDONLY) 프로세스 FD 테이블 시스템 파일 테이블 Inode ┌───────┐ ┌───────────┐ ┌──────────┐ │ 0: stdin │ │ offset: 0 │ │ size: 412 │ │ 1: stdout│ │ flags: RD │─────────→│ uid: 0 │ │ 2: stderr│ │ refcount: 1│ │ blocks:.. │ │ 3: ──────┼───────────→│ │ │ │ └───────┘ └───────────┘ └──────────┘
K8s에서 FD 관련 문제:
• FD 소진: 프로세스별 ulimit -n (기본 1024). 커넥션 수가 많은 프록시(Envoy, Nginx)에서 FD 부족 에러 발생 → ulimit -n 65535 설정
• Inode 소진: 컨테이너 로그나 임시 파일이 많아지면 Inode가 먼저 소진되어 디스크 여유가 있어도 파일 생성 불가 → df -i로 확인
• 삭제된 파일이 공간을 차지: 프로세스가 파일을 열어둔 채 삭제하면, 프로세스가 종료되기 전까지 디스크 공간이 해제되지 않음 → lsof +D /var/log | grep deleted
VFS(Virtual File System)란 무엇이고, 왜 리눅스에서 중요한가요?
VFS는 리눅스 커널이 모든 파일시스템(ext4, XFS, NFS, procfs, sysfs 등)에 대해 동일한 인터페이스(open, read, write, close)를 제공하는 추상화 계층입니다. 앱은 실제 파일시스템이 무엇인지 몰라도 동일한 syscall로 접근할 수 있고, 리눅스의 "모든 것은 파일(everything is a file)" 철학을 가능하게 합니다.
Application → read() syscall → VFS (추상화 계층) → ext4? XFS? NFS? procfs? — VFS가 라우팅 → 실제 파일시스템 드라이버 → Block Layer → Device Driver → HW
"모든 것은 파일" 예시:
• /proc/[pid]/status: 프로세스 상태를 파일로 접근
• /sys/class/net/eth0/statistics: 네트워크 통계를 파일로 접근
• /dev/sda: 블록 디바이스를 파일로 접근
• Docker/K8s: /sys/fs/cgroup/으로 cgroup 제어를 파일로 접근

면접 포인트: "리눅스에서 모든 것은 파일이다"라는 원칙이 컨테이너 기술의 기반이 됩니다. cgroup의 CPU/메모리 제한은 /sys/fs/cgroup/cpu.max, /sys/fs/cgroup/memory.max 파일에 값을 쓰는 것으로 동작합니다. K8s kubelet이 컨테이너의 resource limit을 설정할 때 이 파일들을 사용합니다.
