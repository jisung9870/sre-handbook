● Foundation

SRE 핵심 기초 개념

Site Reliability Engineering의 철학적 기반과 서비스 신뢰성을 정량적으로 정의하는 핵심 프레임워크.

Core

SRE — Site Reliability Engineering

"서비스의 reliability를 software engineering 방식으로 운영하는 실무·역할·문화"
Google이 2003년부터 정립한 운영 철학. 핵심 사상은 "운영 문제를 소프트웨어 엔지니어링으로 풀자"는 것이다. 전통적 Ops가 수동 반복 작업(Toil)에 시간을 쓰는 반면, SRE는 자동화·코드화를 통해 시스템의 신뢰성을 엔지니어링적으로 확보한다.

SRE는 단순히 "모니터링을 잘 하는 사람"이 아니라, 서비스의 신뢰성 목표(SLO)를 정의하고, 그것을 달성하기 위한 시스템·프로세스·문화를 설계하는 역할이다.
핵심 원칙
1) 100% reliability는 잘못된 목표
2) Error Budget으로 혁신과 안정의 균형
3) Toil을 50% 이하로 유지
4) 모니터링이 아닌 Observability
전통 Ops vs SRE
Ops: 수동 배포, 티켓 기반 운영, 장애 시 당장 고치기
SRE: 자동 배포, SLO 기반 의사결정, 장애 후 Postmortem으로 재발 방지
Google SRE 핵심 도서
① Site Reliability Engineering (일명 SRE Book) — SRE 철학과 원칙의 바이블
② The Site Reliability Workbook — SLO 설정, 인시던트 대응 등 실무 가이드
③ Seeking SRE — 다양한 조직의 SRE 적용 사례
💡 핵심 한 줄
SRE는 "what does reliability mean for this service?"라는 질문에서 시작한다. 그 답이 SLI/SLO이고, 그것을 운영에 녹이는 전체 체계가 SRE다.
Core

Reliability vs Availability

둘 다 "서비스가 잘 되고 있나?"를 보지만, 바라보는 깊이가 다르다
Availability(가용성)는 "서비스가 떠 있는가?"에 대한 답이다. 단순히 서버가 응답하는지, HTTP 200을 반환하는지를 본다.

Reliability(신뢰성)는 한 단계 더 깊다. "서비스가 기대한 수준으로, 일관되게 동작하는가?"를 본다. 서버가 떠 있어도 응답이 5초 걸리거나, 결과가 틀리면 가용하다고 볼 수는 있어도 신뢰할 만하다고 보기는 어렵다.

비유하면: 수도꼭지에서 물이 나오면 available. 물이 깨끗하고, 적정 수압으로, 항상 같은 품질로 나오면 reliable.
Availability 공식
Uptime ÷ (Uptime + Downtime) × 100
또는 성공 요청 수 ÷ 전체 요청 수 × 100
Reliability가 포괄하는 것
Availability + Latency(응답 속도) + Correctness(정확한 결과) + Consistency(일관된 품질)

SLI → SLO → SLA 삼형제

이 세 개념은 측정 → 목표 → 약속의 계층 구조를 이룬다. SRE의 모든 의사결정의 출발점.

Measure

SLI — Service Level Indicator

"서비스 상태를 수치로 측정하는 지표"
SLI는 "이 서비스가 얼마나 잘 되고 있는지"를 0~100%로 나타내는 정량적 지표다. 감이 아닌 숫자로 서비스 건강을 판단하기 위해 존재한다. SLI를 잘 정의하는 것이 SRE의 첫걸음이자 가장 중요한 단계다.
대표 SLI 유형
Availability SLI: 성공 응답 비율
Latency SLI: P99 < 500ms 비율
Correctness SLI: 올바른 결과 비율
Throughput SLI: 처리량 달성 비율
좋은 SLI의 조건
1) 사용자 관점에서 측정 (서버 CPU가 아님)
2) 비율(ratio)로 표현 (0~100%)
3) 측정이 자동화 가능
4) 의미 있는 윈도우(30일 등)로 집계
SLI 공식 (일반)
SLI = Good Events ÷ Total Events × 100%
예: Latency SLI = (500ms 이내 응답 수) ÷ (전체 요청 수) × 100%
Target

SLO — Service Level Objective

"SLI에 대해 팀이 정한 목표치"
SLO는 SLI에 대한 목표 임계값이다. "우리 서비스의 latency SLI를 30일 동안 99.9% 이상 유지하겠다"가 SLO다. SLO가 있어야 비로소 "지금 상태가 좋은 건지 나쁜 건지" 판단할 수 있고, Error Budget을 계산할 수 있다.

SLO는 100%로 설정하면 안 된다. 100%는 달성 불가능하며, 달성하려면 모든 변경(배포)을 멈춰야 한다. SLO의 핵심은 "어느 정도의 실패는 허용한다"는 합의다.
SLO 예시
Availability ≥ 99.9% over 30d
Latency P99 ≤ 500ms, 99.5% over 30d
Error Rate ≤ 0.1% over 7d
SLO 설정 팁
1) 처음엔 현재 성능 기반으로 보수적 설정
2) 내부 SLO는 SLA보다 엄격하게
3) 서비스 등급별로 차등 (핵심 API vs 내부 도구)
4) 정기적으로 리뷰하고 조정
Contract

SLA — Service Level Agreement

"고객과의 공식 약속. 못 지키면 보상·패널티가 따른다"
SLA는 비즈니스·법적 계약이다. SLO가 팀 내부의 목표라면, SLA는 외부 고객과의 약속이다. SLA를 위반하면 크레딧 환불, 위약금 등이 발생할 수 있다.

SLA는 항상 SLO보다 느슨하게 설정한다. 내부적으로는 99.95%를 목표(SLO)로 하되, 고객에게는 99.9%를 약속(SLA)하는 식. 이렇게 해야 SLO를 살짝 못 맞춰도 SLA 위반은 피할 수 있다.
SLA 예시
AWS EC2: 월간 가용성 99.99% 미달 시 서비스 크레딧 제공
GCP: 월간 가용성 99.95% 미달 시 10~50% 크레딧
SLO → SLA 관계
SLA(느슨) ≤ SLO(엄격) ← SLI(측정)
내부 목표가 더 빡빡해야 외부 약속을 지킬 여유가 생긴다
▼ SLI → SLO → SLA 관계

Governance

Error Budget & Error Budget Policy

"SLO를 기준으로 '얼마나 실패를 허용할 수 있는가'를 나타내는 운영 여유분"
SLO가 99.9%라면, 0.1%가 Error Budget이다. 한 달 기준 약 43분 동안은 장애가 나도 "예산 내"인 것이다. 이 예산은 배포, 실험, 변경에 사용된다.

Error Budget은 SRE에서 가장 강력한 의사결정 도구다. 개발 속도(배포)와 운영 안정성 사이의 균형을 수치로 관리할 수 있게 해준다. Budget이 남아 있으면 적극적으로 배포하고, 소진되면 안정화에 집중한다.

Error Budget Policy는 "Budget을 얼마나 쓰면 어떤 행동을 할지"를 미리 정해두는 규칙이다. 이것이 없으면 Error Budget은 그냥 숫자에 불과하다.
Error Budget 공식
Error Budget = 1 − SLO Target
SLO 99.9% → Error Budget = 0.1%
= 30일 × 24h × 60m × 0.001 = 43.2분/월
Budget 소진 주체
계획된 변경: 배포, 마이그레이션, 실험
비계획 장애: 버그, 인프라 장애, 외부 의존성
→ 둘 다 같은 예산에서 차감됨
Error Budget Policy 예시
Budget > 50% 남음: 자유롭게 배포. 실험적 변경 OK
Budget 20~50%: 배포 전 추가 리뷰. Canary 필수
Budget < 20%: 긴급 픽스만 배포. 안정화 작업 우선
Budget 소진: 배포 동결(Deploy Freeze). 신뢰성 개선 집중
💡 핵심 인사이트
Error Budget의 진짜 가치는 "Dev팀과 Ops팀 사이의 갈등을 데이터로 해결"하는 것이다. "배포 해도 되나요?" → "Budget 확인해봐" 로 바뀐다. 감정이 아닌 숫자로 대화한다.
● Metrics & Signals

모니터링 지표와 시그널

서비스 상태를 빠르게 파악하고 장애를 정량적으로 추적하는 핵심 프레임워크.

Monitoring

Golden Signals (4 Golden Signals)

"Latency, Traffic, Errors, Saturation — 서비스 상태를 가장 빠르게 보는 4가지 신호"
Google SRE Book에서 제시한 모든 서비스에 적용 가능한 핵심 4대 모니터링 지표. 수백 개의 메트릭을 다 볼 필요 없이, 이 4가지만 잘 보면 대부분의 문제를 빠르게 감지할 수 있다.

① Latency (지연시간)

요청을 처리하는 데 걸리는 시간. 성공 요청과 실패 요청의 latency를 분리해서 봐야 한다. 실패가 빠르게 반환되면 평균 latency가 낮아져 문제를 숨길 수 있다.

histogram_quantile(0.99, rate(http_request_duration_seconds_bucket[5m]))

② Traffic (트래픽)

시스템에 들어오는 요청량. HTTP API라면 RPS(requests/sec), 스트리밍이면 동시 세션 수. 트래픽 변동은 다른 3개 시그널의 변화를 유발하는 원인 지표.

rate(http_requests_total[5m])

③ Errors (오류율)

실패한 요청의 비율. HTTP 5xx뿐 아니라 잘못된 결과를 반환하는 "silent error"도 포함해야 한다. 비즈니스 로직에서 HTTP 500으로 반환하는 경우 Error Rate SLI를 쓰기 어려울 수 있다.

rate(http_requests_total{code=~"5.."}[5m]) / rate(http_requests_total[5m])

④ Saturation (포화도)

시스템 리소스의 사용률. CPU, 메모리, 디스크 I/O, 네트워크 대역폭 등. 80%를 넘기면 성능 저하가 시작되는 경우가 많다. 가장 먼저 포화되는 리소스가 병목.

node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes

💡 USE vs RED vs Golden Signals
USE(Utilization, Saturation, Errors)는 인프라(노드) 관점. RED(Rate, Errors, Duration)는 서비스(요청) 관점. Golden Signals는 이 둘을 통합한 SRE 표준. 실무에서는 상황에 따라 섞어 쓴다.

MTTx — 장애 대응 시간 지표

Detection

MTTD — Mean Time To Detect

"장애가 발생한 시점부터 감지되기까지 걸린 평균 시간"
장애 타임라인의 첫 구간. MTTD가 길면 모든 후속 대응이 늦어진다. 모니터링 커버리지와 알림 품질이 MTTD를 결정한다. MTTD를 줄이는 것이 전체 복구 시간(MTTR)을 줄이는 가장 효과적인 방법이다.
공식
MTTD = Σ(감지 시간 − 장애 발생 시간) ÷ 장애 건수
줄이는 방법
SLO 기반 알림 설정, Burn Rate 알림, Synthetic Monitoring(Blackbox Exporter), 로그 기반 이상 탐지
Recovery

MTTR — Mean Time To Recovery

"장애를 복구하기까지 걸린 평균 시간"
이 문서에서는 장애 발생 시각부터 서비스 정상 복구까지의 전체 시간을 MTTR로 설명한다. 따라서 MTTR = MTTD + 진단 시간 + 수정 시간 + 검증 시간으로 볼 수 있다. 일부 조직은 감지 시각부터 별도로 재기도 하지만, 중요한 것은 팀 안에서 기준을 일관되게 유지하는 것이다. DORA 메트릭의 "Recovery Time"과 같은 축의 지표이며, SRE에서 가장 중시하는 운영 성과 지표 중 하나다.
공식
MTTR = Σ(복구 완료 시간 − 장애 발생 시간) ÷ 장애 건수
줄이는 방법
자동 롤백(ArgoCD), Runbook 정비, 온콜 체계 + 에스컬레이션 규칙, Observability 강화 (로그-트레이스 연결)
▼ 장애 타임라인 분해
장애 발생Root Cause

Performance

DORA Metrics

"Delivery 성과를 보는 4+1 프레임. Reliability 자체보다는 delivery + stability 관점"
Google DORA 팀이 정립한 소프트웨어 딜리버리 성과 지표. 속도(Velocity) 2개 + 안정성(Stability) 2개로 구성. SRE가 "서비스의 reliability"를 본다면, DORA는 "팀이 얼마나 빠르고 안전하게 변경을 전달하는가"를 본다.
⚡ 속도 (Velocity)
Deployment Frequency: 배포 빈도
Lead Time for Changes: 커밋→배포 시간
🛡 안정성 (Stability)
Change Failure Rate: 배포 실패 비율
Recovery Time: 장애 복구 시간 (= MTTR)
SRE와의 관계
DORA의 Change Failure Rate는 Error Budget 소진 속도와 직결. Recovery Time은 SRE의 MTTR과 동일. DORA 메트릭을 개선하면 자연스럽게 SRE 성과도 올라가는 상호 보완 관계.
● Operations

SRE 운영 실무

알림 설계부터 인시던트 대응, Postmortem, Toil 관리까지 — SRE가 일상적으로 수행하는 운영 체계.

Critical

Alerting — 알림 설계

"모든 metric에 반응하는 게 아니라, '사람을 깨워야 할 수준의 user impact'에만 반응"
SRE에서 알림은 모니터링보다 중요한 개념이다. 모니터링은 데이터를 수집하는 것이고, 알림은 "이 데이터 중 무엇이 행동을 요구하는가"를 결정하는 것이다.

나쁜 알림은 Alert Fatigue(알림 피로)를 유발한다. CPU 80% 같은 원인 기반(cause-based) 알림이 수십 개 울리면, 정작 중요한 알림을 놓치게 된다. SRE는 증상 기반(symptom-based) 알림을 지향한다.
좋은 알림의 조건
1) 즉각적 행동이 필요한 것만 (Actionable)
2) 사용자 영향 기반, 원인이 아님
3) 중복 알림 없음 (deduplicated)
4) 페이지 vs 티켓 vs 로그로 구분
알림 등급 체계
Page: 즉시 대응 (사용자 영향 중). 온콜 호출
Ticket: 비즈니스 시간 내 대응. 이슈 생성
Log: 기록만. 트렌드 분석용. 사람을 깨우지 않음
Burn Rate Alert (권장)
단순 임계치 대신 Error Budget 소진 속도로 알림. "현재 속도로 가면 N시간 후 Budget 소진" 방식. Multi-window(1h, 6h) Burn Rate를 조합하면 false positive를 대폭 줄일 수 있다.
Burn Rate = (현재 Error Rate) ÷ (SLO가 허용하는 Error Rate)
Burn Rate > 1이면 Budget이 예상보다 빨리 소진 중. > 14.4면 1시간 내 대응 필요(Page).

Observability — 관측 가능성

"Metric, Log, Trace로 시스템 내부 상태를 추론하고 디버깅 가능한 능력"
모니터링은 "미리 정해둔 것"을 감시하는 것. Observability는 "예상하지 못한 문제"도 찾아낼 수 있는 능력이다. 시스템이 observable하면 "왜 느려졌지?"라는 새로운 질문에도 답할 수 있다.

세 가지 필러(Pillar)는 독립적으로도 가치가 있지만, 서로 연결되었을 때 진가를 발휘한다. 느린 요청(Metric) → 해당 시간대 로그(Log) → 해당 요청의 트레이스(Trace)로 이어지는 drill-down이 가능해야 한다.
3 Pillars
Metrics: 수치 시계열. "지금 상태가 어떤가?"
Logs: 이벤트 기록. "정확히 무슨 일이 일어났나?"
Traces: 요청 경로 추적. "어디서 느려졌나?"
LGTM 스택
Loki (Logs) + Grafana (Visualization) + Tempo (Traces) + Mimir/Prometheus (Metrics). Alloy로 수집 통합.
Monitoring vs Observability
Monitoring: "CPU 80% 넘으면 알려줘" — 미리 정한 조건만 감시
Observability: "왜 특정 사용자만 느린지 트레이스로 추적하고, 해당 Pod의 로그를 확인하고, 메트릭 상관관계를 분석할 수 있다" — 새로운 질문에 대응 가능

Incident Management — 인시던트 관리

"장애를 탐지 → 분류 → 대응 → 복구 → 커뮤니케이션하는 운영 체계"
장애가 발생했을 때 "누가, 어떤 순서로, 무엇을" 해야 하는지가 명확해야 한다. 혼란 속에서 즉흥적으로 대응하면 복구 시간(MTTR)이 늘어나고, 커뮤니케이션 실패로 이차 피해가 발생한다.

인시던트 관리 체계는 역할 분리가 핵심. Incident Commander(지휘), Ops Lead(실행), Communication Lead(소통)을 분리하면 각자 맡은 일에 집중할 수 있다.
인시던트 대응 단계
1) Detect: 알림 수신, 장애 인지
2) Triage: 심각도 분류 (P1~P4)
3) Respond: 담당자 할당, 진단, 완화
4) Recover: 롤백 또는 핫픽스 배포
5) Communicate: 이해관계자 상태 공유
6) Follow-up: Postmortem 작성
심각도 분류
P1 (Critical): 서비스 전체 장애, 즉시 대응
P2 (Major): 주요 기능 장애, 1시간 내 대응
P3 (Minor): 부분 기능 영향, 비즈니스 시간 내
P4 (Low): 사용자 영향 미미, 정규 백로그
Learning

Postmortem — 장애 회고

"blame보다 learning. 타임라인, 원인, 영향, 재발 방지 액션을 남기는 회고"
Postmortem은 비난(blame)이 아닌 학습(learning)이 목적이다. "누가 실수했나"가 아니라 "시스템/프로세스에 어떤 약점이 있었나"를 찾는다. Google SRE에서는 Blameless Postmortem을 철칙으로 한다.

잘 쓴 Postmortem은 같은 장애의 재발을 방지하고, 팀 전체의 운영 역량을 높이는 지식 자산이 된다.
Postmortem 필수 항목
1) Summary: 한 줄 요약
2) Impact: 영향 범위와 시간
3) Timeline: 시간순 이벤트 기록
4) Root Cause: 근본 원인
5) Action Items: 재발 방지 조치 (담당자+기한)
Blameless 원칙
"사람은 실수한다"를 전제로 시작.
• 개인이 아닌 시스템을 개선
• "왜 그랬어?"가 아닌 "어떤 정보가 부족했나?"
• 모든 팀원이 솔직하게 말할 수 있는 환경
• 숨기면 더 큰 사고로 이어진다
Efficiency

Toil — 반복 운영 업무

"반복적이고, 수동적이며, 자동화 가능한 운영 업무. SRE에서 줄여야 할 대표 대상"
Toil은 "운영 작업" 전체가 아니다. 서비스를 운영하면서 발생하는 수동적이고, 반복적이며, 서비스 성장과 함께 비례 증가하는 작업을 말한다. Google SRE는 Toil을 전체 업무의 50% 이하로 유지하는 것을 목표로 한다.

Toil이 위험한 이유는 시간이 지날수록 늘어나기 때문이다. 서비스가 커지면 Toil도 비례해서 커진다. 자동화하지 않으면 결국 팀의 모든 시간이 Toil에 빠진다.
Toil의 5가지 특성
1) Manual: 사람이 직접 실행
2) Repetitive: 매번 같은 작업
3) Automatable: 자동화 가능
4) Reactive: 요청에 의한 수동 대응
5) No Lasting Value: 끝나면 가치 소멸
Toil 예시
• 수동 배포, 수동 롤백
• 인증서 갱신 수동 처리
• "이 로그 좀 봐주세요" 요청 대응
• 디스크 용량 정리 스크립트 수동 실행
• 반복적인 접근 권한 부여
Toil 줄이기
자동화: cert-manager, GitOps 자동 Sync, 자동 스케일링
셀프서비스: OpsGate 같은 내부 도구로 반복 요청을 사용자가 직접 처리
제거: 필요 없는 프로세스 자체를 없앰. "이 작업이 정말 필요한가?"

전체 연결 구조

SRE 개념들은 독립적이지 않다. 서로 맞물려 하나의 운영 체계를 이룬다.

SRE 개념 관계도

▼ 개념 간 인과 관계

SRE의 중심축: SLI → SLO → Error Budget

SLI로 서비스 상태를 측정하고, SLO로 목표를 세우며, Error Budget으로 "얼마나 실패해도 되는지"를 계산한다. Error Budget이 배포 속도와 안정성 사이의 균형추 역할을 한다.

Golden Signals → SLI 공급

Golden Signals(Latency, Traffic, Errors, Saturation)는 SLI의 원천 데이터다. Latency SLI는 Golden Signal의 Latency에서 나온다.

Observability → MTTD 단축

Metrics+Logs+Traces가 잘 구축되면 문제를 빠르게 감지하고 진단할 수 있다. MTTD가 줄면 MTTR도 줄어든다.

Alerting → Incident 트리거

Burn Rate 알림이 울리면 Incident Management 프로세스가 시작된다. 좋은 알림 = 빠른 MTTD = 짧은 MTTR.

Postmortem → Toil/자동화 개선

장애 회고에서 나온 Action Item이 자동화, 알림 개선, Runbook 정비로 이어진다. 이것이 다음 장애의 MTTD/MTTR을 줄인다.

DORA ↔ SRE 상호 보완

DORA의 CFR이 높으면 Error Budget이 빠르게 소진 → 배포 동결. DORA의 Recovery Time = MTTR. Delivery 성과와 Reliability는 같은 동전의 양면.
