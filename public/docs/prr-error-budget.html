● Error Budget 실전 산정

Error Budget — 실전 계산부터 정책까지

SLO에서 Error Budget을 산출하고, 소비를 추적하고, 정책으로 행동을 결정하는 전체 흐름.

Calculation

Error Budget 실전 산정

SLO 목표에서 "허용 가능한 에러 양"을 구체적 숫자로 계산하는 방법
Error Budget = 1 - SLO 목표. 이것이 전부다. 하지만 "이 숫자가 실제로 뭘 의미하는가?"를 시간, 요청 수, 분 단위로 변환해야 실무에서 쓸 수 있다.

시간 기반 Error Budget
계산:
SLO 99.9% / 30일 윈도우 Error Budget = 1 - 0.999 = 0.001 30일 = 30 × 24 × 60 = 43,200분 허용 다운타임 = 43,200 × 0.001 = 43.2분/월 즉, 한 달에 총 43분까지 서비스가 다운되어도 SLO 위반이 아니다.

실전 의미:
• 43분 = 배포 롤백 시간 2번 정도
• 한 번의 P1 인시던트가 30분이면 → 나머지 13분만 남음
• 두 번째 인시던트가 15분이면 → SLO 위반
요청 기반 Error Budget
계산:
SLO 99.9% / 30일 윈도우 / 월 1000만 요청 Error Budget = 1 - 0.999 = 0.001 허용 실패 요청 = 10,000,000 × 0.001 = 10,000 요청/월 즉, 한 달에 10,000개의 요청이 실패(5xx 또는 timeout)해도 SLO 위반이 아니다. 일 평균: 10,000 ÷ 30 ≈ 333 실패 요청/일 시간 평균: 333 ÷ 24 ≈ 14 실패 요청/시간

실전 의미:
• 시간당 14개 에러까지 허용 → 매우 적다!
• 배포 실패로 5분간 50% 에러 발생 → 분당 ~500 에러 × 5분 = 2,500 에러 소비
• 월 Budget의 25%를 한 번의 배포 실패로 소진
⚠ "우리 회사는 HTTP 500을 비즈니스 에러에도 사용한다"
HTTP 500으로 비즈니스 로직 에러를 반환하면 SLI 계산이 불가능하다. "유효한 요청 중 성공 비율"을 측정할 수 없기 때문이다. 이 경우 Latency SLI(P99 < 500ms)를 기본 SLO로 사용하는 전략이 현실적이다. SLO Engineering 문서의 SLO starter kit이 이 제약을 반영하여 설계됨.
Tracking

Error Budget 소비 추적

"지금 Error Budget을 얼마나 썼는가?" — 실시간 추적과 번다운 차트
소비율 계산
Error Budget 소비율 (%) = 실제 에러율 ÷ Error Budget × 100 예시: SLO 99.9%, 현재 에러율 0.05% Error Budget = 0.1% 소비율 = 0.05% ÷ 0.1% × 100 = 50% → 50% 소비됨, 50% 남음 남은 Budget (시간): 30일 윈도우, 43.2분 총 Budget 50% 소비 = 21.6분 사용 → 21.6분 남음

Rolling Window: "지난 30일"의 슬라이딩 윈도우. 매일 하루치가 추가되고 31일 전 하루치가 빠짐. 한 달 전 인시던트가 윈도우에서 빠지면 Budget이 자동 회복
Budget 번다운 차트
시각화: Y축이 남은 Budget %, X축이 시간. 우하향 직선이 이상적

패턴 분석:
• 완만한 하강: 건강. 작은 에러가 꾸준히 소비
• 급격한 절벽: 인시던트 발생. 한 번에 대량 소비
• 평탄: 에러 0. 좋지만 "너무 보수적인가?" 고민
• 0% 도달: SLO 위반. Error Budget 정책 발동

Grafana 대시보드:
SLO Engineering 문서의 Multi-window Burn Rate 알림과 결합하여:
• Burn Rate > 14.4× → 1시간 내 Budget 소진 예측 → P1 알림
• Burn Rate > 6× → 6시간 내 소진 → P2 알림
• Burn Rate > 1× → 정상 소비 중
• Burn Rate < 1× → Budget 회복 중
Policy

Error Budget 정책 — Budget 상태에 따른 행동 결정

"Error Budget이 X% 남았을 때 우리는 무엇을 하는가?" — 이것이 SRE의 핵심 거버넌스
Error Budget 정책이 없으면 SLO는 그냥 숫자다. Budget 상태에 따라 팀의 행동이 바뀌어야 SLO가 의미를 가진다. Google SRE에서 Error Budget 정책은 SRE팀과 개발팀의 계약이다.
4단계 Error Budget 정책
■ Budget > 75% 남음 — GREEN → 정상 배포. 새 기능 개발 우선. → Chaos 실험 적극 실행 → 공격적 변경(마이그레이션, 업그레이드) 가능 ■ Budget 50~75% 남음 — BLUE → 정상 배포. 하지만 높은 리스크 변경은 주의 → 배포 전 Canary 필수 → 인시던트 RCA 완료 확인 ■ Budget 25~50% 남음 — YELLOW → 신규 기능 배포 속도 감소 → 안정화 작업 우선 (성능 개선, 버그 수정) → 모든 배포에 rollback 계획 필수 → Chaos 실험 중단 ■ Budget < 25% 남음 — RED (Budget 거의 소진) → 기능 배포 동결 (change freeze) → 안정화 & 신뢰성 작업만 허용 → 인시던트가 추가 발생하면 SLO 위반 → 모든 변경에 SRE(본인) 승인 필수 → Budget이 50%까지 회복되면 YELLOW로 복귀
💡 Solo DevOps에서의 Error Budget 정책
"개발팀과의 계약"이 아니라 자기 자신과의 계약이다. Budget RED일 때 "이 기능 오늘 꼭 배포해야 하는데..."라는 압박에 "Error Budget 정책에 따라 안정화 먼저"라고 근거를 댈 수 있다. 정책을 문서화하고 경영진과 공유해두면 더 강력한 방어가 된다.
Implementation

Prometheus로 Error Budget 구현

PromQL로 Error Budget 잔량, 소비율, Burn Rate를 계산하는 실전 쿼리
Availability SLI/SLO/Budget
# SLI: 성공 요청 비율 (30일 윈도우) sli_availability = sum(rate(http_requests_total{ code!~"5.."}[30d])) / sum(rate(http_requests_total[30d])) # Error Budget 남은 비율 (%) # SLO 목표: 0.999 (99.9%) error_budget_remaining = 1 - ( (1 - sli_availability) / (1 - 0.999) ) # 결과 해석: # 1.0 = 100% 남음 (에러 0) # 0.5 = 50% 남음 # 0.0 = 소진 (SLO 위반) # -0.5 = 50% 초과 소진 (SLO 크게 위반)
Latency SLI/SLO/Budget
# SLI: P99 < 500ms인 요청 비율 (30일) sli_latency = sum(rate(http_request_duration_seconds_bucket{ le="0.5"}[30d])) / sum(rate(http_request_duration_seconds_count[30d])) # SLO 목표: 97% (요청의 97%가 500ms 이내) error_budget_latency = 1 - ( (1 - sli_latency) / (1 - 0.97) ) # 실전 제약: 회사가 500 에러를 비즈니스에 쓰면 # → Availability SLI가 무의미 # → Latency SLI를 기본 SLO로 사용 # → 이것이 SLO starter kit의 설계 이유
💡 Recording Rule로 비용 절감
30일 range vector 쿼리는 매번 실행하면 매우 비싸다. Recording Rule로 미리 집계해두면 Grafana 대시보드 렌더링 속도와 Prometheus 부하를 대폭 줄인다. SLO Engineering 문서의 Recording Rules 설정과 직접 연결.
● Production Readiness Review

PRR — 프로덕션 준비 검증

서비스를 프로덕션에 올리기 전 "이 서비스가 프로덕션에서 안전하게 운영될 수 있는가?"를 체계적으로 검증.

Concept

PRR이란

Google SRE의 Production Readiness Review — 서비스의 "운영 가능성"을 사전 평가
PRR은 "이 서비스가 프로덕션 장애를 견딜 수 있는가?"를 런칭 전에 평가하는 프로세스다. SLO가 정의되어 있는지, 모니터링이 충분한지, 장애 시 대응 계획이 있는지, 용량이 충분한지를 체계적으로 점검한다.

Solo DevOps에서도 새 서비스를 프로덕션에 올리기 전 PRR 체크리스트를 5분이라도 훑어보면 런칭 후 장애를 크게 줄일 수 있다.

● Change Management

변경 관리

변경 리스크 등급, 배포 윈도우, Freeze 정책 — 변경이 장애의 가장 큰 원인이다.

Risk

변경 리스크 등급

모든 변경은 같지 않다 — 리스크에 따라 검증 수준을 차등 적용
Google SRE의 연구에 따르면 전체 인시던트의 70%가 변경(배포, 설정 변경, 인프라 변경)에 의해 발생한다. 변경의 리스크를 분류하고, 리스크에 따라 검증 수준을 달리하면 인시던트를 대폭 줄일 수 있다.
등급설명예시검증 요건
Low영향 범위 좁고 롤백 쉬움앱 코드 배포 (Canary), ConfigMap 변경, 문서 업데이트CI 통과 + Canary. 자동 승인 가능
Medium영향 범위 넓거나 롤백에 주의 필요Helm 차트 변경, 새 서비스 런칭, DB 마이그레이션 (backward compatible)CI + Staging 검증 + PR 리뷰 + Canary
High전체 서비스 영향 또는 롤백 어려움K8s 업그레이드, CRD 변경, 네트워크 정책 변경, DB 스키마 (breaking)Staging 완전 검증 + SRE 승인 + 유지보수 윈도우 + Rollback 계획 문서화
Critical롤백 불가능한 변경클러스터 마이그레이션, 데이터 마이그레이션, 플랫폼 변경전체 테스트 + Blue-Green + 비상 연락망 + War Room 대기
Freeze

Change Freeze & 배포 윈도우

배포를 하면 안 되는 시간과 해야 하는 시간을 명확히
Freeze 정책
정기 Freeze:
• 금요일 오후~월요일 아침: 주말 배포 금지 (Solo DevOps면 더더욱)
• 매월 마지막 주: 월말 결산 기간 (금융/커머스)
• 연말 연휴: 12월 말 ~ 1월 초

비정기 Freeze:
• Error Budget RED (25% 미만) → 안정화 작업만 허용
• K8s 업그레이드 중 → 앱 배포 동결
• 대규모 이벤트/프로모션 기간

Freeze 예외:
• P1/P2 인시던트 핫픽스
• 보안 패치 (CVE Critical)
→ 예외에도 PR 리뷰 + Canary 필수
배포 윈도우
권장 배포 시간:
화~목 10:00~16:00 (KST)
→ 문제 발생 시 업무 시간 내 대응 가능
→ 월요일은 주말 축적 이슈 대응, 금요일은 주말 리스크 회피

High 리스크 변경:
유지보수 윈도우: 수요일 14:00~16:00
→ 사전 공지, War Room 대기, SLI 실시간 모니터링

자동화:
ArgoCD Sync Window로 배포 허용/금지 시간 제어

- kind: deny
schedule: "0 17 * * 5"
duration: 63h

● Dependency Management

의존성 관리와 SLO Budget 분배

서비스 의존성을 식별하고, 의존성의 SLO가 우리 SLO에 미치는 영향을 정량적으로 분석.

Mapping

의존성 매핑

Critical Path 위의 모든 의존성을 식별하고 HARD/SOFT로 분류
의존성 분류
HARD Dependency: 이것이 없으면 서비스 완전 불가능
예: DB(데이터 저장), Auth 서비스(인증), Redis(세션)
→ 반드시 이중화. Circuit Breaker로 장애 격리. Fallback 불가능 = 서비스 다운 인정

SOFT Dependency: 이것이 없어도 핵심 기능은 동작
예: 알림 서비스, 추천 엔진, 분석 서비스
→ Graceful Degradation. 해당 기능만 비활성화. 사용자에게 "일시적으로 이용 불가" 표시

INFRA Dependency: 모든 서비스가 공유하는 인프라
예: CoreDNS, NAT Gateway, Load Balancer
→ 인프라 이중화. 단일 장애점(SPOF) 제거
의존성 매핑 도구
수동 매핑: 아키텍처 다이어그램에서 화살표를 따라가며 목록 작성

자동 매핑:
• Kiali (Istio): Service Graph에서 자동 시각화
• Hubble (Cilium): Flow 데이터에서 서비스 간 통신 패턴 추출
• Tempo/분산 추적: Trace 데이터에서 의존성 체인 추적

결과물 예시:
api-server → [HARD] PostgreSQL → [HARD] EBS
api-server → [HARD] Redis → [HARD] ElastiCache
api-server → [SOFT] Payment API → [외부]
api-server → [INFRA] CoreDNS
api-server → [INFRA] NAT GW → [INFRA] Internet

Critical Path: 사용자 요청이 성공하기 위해 반드시 거쳐야 하는 HARD 의존성 체인

의존성 SLO Budget 분배

"우리 SLO 99.9%를 달성하려면, 각 의존성의 SLO는 얼마여야 하는가?"
서비스의 SLO는 의존성 SLO의 곱보다 높을 수 없다. 3개의 HARD 의존성이 각각 99.9%면, 최상의 경우 서비스 SLO는 99.9% × 99.9% × 99.9% = 99.7%이다. 우리 서비스의 에러가 0이라도 의존성 에러로 SLO를 달성하지 못할 수 있다.
의존성 SLO 곱 계산
서비스 Availability ≤ Π(의존성 Availability) 예시: api-server의 HARD 의존성 PostgreSQL: 99.95% Redis: 99.99% Payment API: 99.9% CoreDNS: 99.999% 최대 서비스 Availability: 0.9995 × 0.9999 × 0.999 × 0.99999 = 0.99839... ≈ 99.84% → 우리 코드가 완벽해도 SLO 99.9%는 불가능! → Payment API(99.9%)가 병목 해결: Payment API를 SOFT로 변경 → 장애 시 Fallback (결제 지연 처리) → HARD 의존성에서 제외 → 새 계산: 0.9995 × 0.9999 × 0.99999 ≈ 99.94% → SLO 99.9% 달성 가능
Budget 분배 전략
전체 Error Budget을 의존성별로 배분:

SLO 99.9% → Error Budget 0.1%
• 자체 에러 (코드 버그, 배포 실패): 0.04%
• PostgreSQL 장애: 0.03%
• Redis 장애: 0.01%
• 인프라 (CoreDNS, NAT GW): 0.01%
• 여유분 (예측 불가): 0.01%
• 합계: 0.10%

의미:
"PostgreSQL 장애로 인한 에러가 0.03%를 초과하면 DB 팀/AWS 지원에 에스컬레이션"
"자체 에러가 0.04%를 초과하면 배포 동결"

실전: 이 수준의 Budget 분배는 대규모 조직에서나 가능. Solo DevOps에서는 전체 Budget을 하나로 관리하되, 인시던트 Postmortem에서 "이 에러가 자체 원인인지 의존성 원인인지" 분류하는 것만으로도 충분
