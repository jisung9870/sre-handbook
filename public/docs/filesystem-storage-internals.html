● Virtual File System

VFS 계층 구조

Linux의 모든 파일시스템을 통합하는 추상화 계층 — 앱은 VFS만 보고, 실제 구현은 다양.

Architecture

VFS 계층 구조

Application → System Call → VFS → Filesystem → Block Layer → Device Driver → Hardware
VFS(Virtual File System)는 Linux 커널이 모든 파일시스템에 대해 동일한 인터페이스를 제공하는 추상화 계층이다. 앱이 open(), read(), write()를 호출하면 VFS가 실제 파일시스템(ext4, XFS, NFS, procfs 등)으로 요청을 라우팅한다.
I/O 경로
Application → write() system call → VFS (Virtual File System) → Filesystem (ext4/XFS) → Page Cache (buffered I/O) → Block Layer (blk-mq) → I/O Scheduler (mq-deadline/none) → Device Driver (nvme/scsi) → Hardware (SSD/HDD) Direct I/O: Page Cache를 건너뜀 App → VFS → Filesystem → Block Layer → Device
VFS 핵심 오브젝트
Superblock: 파일시스템의 메타데이터. 블록 크기, inode 수, 마운트 옵션
Inode: 파일의 메타데이터. 크기, 권한, 타임스탬프, 데이터 블록 위치
Dentry (Directory Entry): 파일 이름 → inode 매핑. 디렉토리 구조 표현
File: 열린 파일의 상태. 오프셋, 접근 모드

핵심: 파일 이름은 Dentry에, 파일 데이터 위치는 Inode에, 실제 데이터는 데이터 블록에 저장. "파일 이름을 바꾸면 데이터가 이동하지 않는다" — Dentry만 변경됨
Core

Inode & Dentry 캐시

파일 메타데이터 접근 성능의 핵심 — 디스크 읽기 없이 메모리에서 해결
Inode 구조
Inode 번호: 파일시스템 내 고유 식별자
포함 정보: 파일 크기, 소유자(UID/GID), 권한(mode), 타임스탬프(atime/mtime/ctime), 데이터 블록 포인터, 링크 수
포함하지 않는 것: 파일 이름! (Dentry가 담당)

ls -i filename — inode 번호 확인
stat filename — inode 상세 정보
df -i — inode 사용량 (inode 소진 시 파일 생성 불가!)

SRE 이슈: 작은 파일이 수백만 개(컨테이너 로그 등)면 inode가 먼저 소진되어 디스크 여유가 있어도 파일 생성 불가
Dentry Cache
경로 이름 → inode 매핑을 메모리에 캐시:
/var/log/nginx/access.log 접근 시:
→ / dentry → var dentry → log dentry → nginx dentry → access.log dentry
→ 각 dentry가 inode를 가리킴

캐시 히트 시: 디스크 읽기 없이 즉시 inode 접근
모니터링: /proc/slabinfo → dentry, inode_cache 크기

Negative Dentry: "이 파일은 존재하지 않는다"도 캐시. 없는 파일을 반복 조회할 때 디스크 접근 방지

SRE 관점: echo 2 > /proc/sys/vm/drop_caches로 dentry/inode 캐시를 강제 해제할 수 있지만, 프로덕션에서는 성능 급락을 유발하므로 피해야 함
● Filesystem Internals

ext4 & XFS 내부

저널링, Extent, Allocation — Linux 양대 파일시스템의 핵심 동작 원리.

Default

ext4 내부 동작

Linux 기본 파일시스템 — Extent, 저널링, delayed allocation이 핵심
핵심 특성
Extent 기반 할당: ext3의 간접 블록 포인터 대신, 연속 블록의 시작+길이로 표현. 큰 파일의 메타데이터 크기 대폭 감소. 순차 읽기 성능 향상

저널링 (JBD2): 메타데이터(기본) 또는 데이터+메타데이터를 저널에 먼저 기록하여 크래시 복구 보장
• journal: 데이터+메타 모두 저널 (가장 안전, 느림)
• ordered: 데이터를 먼저 쓰고 메타를 저널 (기본, 균형)
• writeback: 메타만 저널 (빠르지만 데이터 손실 가능)

Delayed Allocation: 블록 할당을 실제 쓰기 시까지 지연. 연속 블록 할당 확률 증가 → fragmentation 감소
한계 & 제한
최대 파일 크기: 16TB
최대 파일시스템: 1EB (1 Exabyte)
서브디렉토리 수: ~64,000 (HTree로 성능 개선됨)

단편화: 시간이 지나면 fragmentation 증가. e4defrag로 온라인 defrag 가능하지만 권장하지 않음 (SSD에서는 의미 없음)

SRE 튜닝:
noatime 마운트 옵션: 읽기 시 접근 시간 업데이트 안 함 → 불필요한 쓰기 제거. K8s 노드에 권장
discard 마운트 옵션: SSD TRIM 지원. fstrim.timer로 주기적 TRIM이 더 권장됨
Scale

XFS 내부 동작

대규모 파일과 높은 병렬 I/O에 최적화된 파일시스템 — RHEL/CentOS 기본
XFS의 차별점
Allocation Group (AG): 파일시스템을 여러 독립 영역으로 분할. 각 AG가 독립적으로 할당/관리 → 병렬 I/O 성능 극대화. 멀티스레드 워크로드에 최적

B+Tree Everywhere: 디렉토리, inode 할당, extent, free space 모두 B+Tree로 관리. 대규모 파일/디렉토리에서 안정적 성능

Online 기능:
• Online resize (확장만, 축소 불가)
• Online defrag (xfs_fsr)
• Online dump/restore

최대 파일 크기: 8EB
최대 파일시스템: 8EB
SRE 관점 ext4 vs XFS
ext4 선택:
• 소규모 파일시스템 (< 1TB)
• 일반 워크로드. 기본값으로 충분
• 파일시스템 축소가 필요한 경우
• Ubuntu 기본

XFS 선택:
• 대규모 파일시스템 (> 1TB)
• 높은 병렬 I/O (DB, 로그 서버)
• 대규모 파일 (백업, 미디어)
• RHEL/CentOS 기본

K8s PV: EBS gp3는 기본 ext4로 포맷됨. StorageClass.parameters.fsType: xfs로 XFS 지정 가능. DB 워크로드에 XFS 권장

● Page Cache & I/O Patterns

Page Cache, Direct I/O, Writeback

메모리와 디스크 사이의 캐시 계층 — DB와 파일 서버의 I/O 성능을 좌우하는 핵심.

Cache

Page Cache

디스크 데이터를 메모리 페이지에 캐시하여 반복 읽기를 메모리 속도로 제공
Linux는 사용 가능한 메모리를 최대한 Page Cache로 활용한다. 파일을 읽으면 해당 페이지가 캐시되고, 다음 읽기는 디스크 대신 메모리에서 즉시 제공된다. free 명령의 buff/cache가 이것이다.
동작
읽기:
1. 앱이 read() 호출
2. VFS가 Page Cache에 해당 페이지가 있는지 확인
3. Hit: 메모리에서 직접 반환 (마이크로초)
4. Miss: 디스크에서 읽어 Cache에 적재 후 반환 (밀리초)

쓰기 (Buffered):
1. 앱이 write() 호출
2. 데이터가 Page Cache에 기록 (dirty page로 마킹)
3. write() 즉시 반환 (앱 입장에서 빠름)
4. 커널이 나중에 dirty page를 디스크에 flush (writeback)

모니터링:
cat /proc/meminfo | grep -E "Cached|Dirty|Writeback"
K8s에서의 Page Cache
cgroup memory와의 상호작용:
Page Cache는 해당 cgroup(컨테이너)의 메모리 사용량에 포함된다.

문제: 컨테이너가 대량 파일을 읽으면 Page Cache가 cgroup 메모리를 채움 → memory limit 도달 → OOMKill
실제 앱 메모리가 아니라 캐시가 원인인데 OOM 발생!

지표:
container_memory_working_set_bytes — Page Cache 일부 포함
container_memory_rss — 실제 앱 메모리

해결:
• memory limit에 Page Cache 여유분 포함
• 대량 파일 I/O는 Direct I/O 사용 (Page Cache 우회)
• memory.high (soft limit)으로 조기 회수 유도
Bypass

Direct I/O — Page Cache 우회

"내가 캐시를 직접 관리하겠다" — DB가 O_DIRECT를 쓰는 이유
O_DIRECT 플래그로 파일을 열면 Page Cache를 경유하지 않고 디스크에 직접 읽기/쓰기한다. DB(PostgreSQL, MySQL)가 자체 버퍼 풀(shared_buffers)을 운영하므로 Page Cache는 이중 캐싱(double buffering)이 되어 메모리 낭비를 유발한다.
왜 DB는 Direct I/O를 쓰는가
PostgreSQL: 기본적으로 Buffered I/O 사용. shared_buffers가 자체 캐시 역할. Page Cache에도 같은 데이터가 캐시됨 → 메모리 2배 낭비 가능
→ PostgreSQL 16+: io_method = io_uring (Direct I/O + io_uring 조합 실험적 지원)

MySQL/InnoDB: innodb_flush_method = O_DIRECT 권장. InnoDB Buffer Pool이 자체 캐시. OS Page Cache 불필요

RocksDB (etcd, Loki): use_direct_io_for_flush_and_compaction = true. Compaction 중 대량 데이터를 읽고 쓰는데, Page Cache를 오염시키면 다른 앱의 캐시 히트율이 떨어짐
Direct I/O 제약
정렬 요구: 읽기/쓰기 버퍼, 오프셋, 크기가 블록 크기(보통 512B 또는 4KB)에 정렬되어야 함. 정렬 안 되면 에러

성능: Page Cache가 없으므로 모든 읽기가 디스크를 히트. 랜덤 읽기에서 Buffered I/O보다 느릴 수 있음. 자체 캐시가 있는 앱(DB)에서만 의미

K8s 영향: Direct I/O를 사용하는 Pod은 Page Cache를 소비하지 않으므로 container_memory_working_set_bytes가 실제 앱 메모리에 가까워짐. OOMKill 위험 감소
Flush

Writeback — Dirty Page 플러시

Page Cache의 dirty page가 실제 디스크에 기록되는 메커니즘
Writeback 트리거
1) 주기적: dirty_writeback_centisecs (기본 500 = 5초) 간격으로 flush 스레드가 dirty page를 디스크에 쓴다
2) Dirty ratio 초과: dirty_background_ratio (기본 10%) 초과 → 백그라운드 flush 시작
dirty_ratio (기본 20%) 초과 → 앱의 write()가 블록됨! (동기 writeback 강제)
3) fsync(): 앱이 명시적으로 호출. 해당 파일의 모든 dirty page를 즉시 디스크에
4) sync: 전체 dirty page flush. sync 명령

SRE 관점: dirty_ratio에 도달하면 write latency가 급증한다. etcd의 WAL fsync가 느려지면 리더 선출 실패. I/O 워크로드에서는 dirty ratio 모니터링 필수
튜닝
vm.dirty_background_ratio = 5 — 빠른 백그라운드 flush 시작
vm.dirty_ratio = 10 — 앱 블록 임계값 낮춤
vm.dirty_writeback_centisecs = 100 — flush 주기 단축 (1초)

DB 서버: dirty ratio를 낮게 설정하여 갑작스러운 flush storm 방지. 꾸준히 조금씩 flush가 더 예측 가능

etcd: WAL fsync 성능이 핵심. SSD에서도 dirty page가 많으면 fsync가 느려짐. dirty_background_ratio = 5 권장

모니터링:
node_memory_Dirty_bytes — 현재 dirty page 크기
node_memory_Writeback_bytes — 현재 writeback 중인 크기
급증하면 I/O 스파이크 예고
Scheduler

I/O Scheduler

디스크에 보낼 I/O 요청의 순서를 결정하는 커널 레이어
Linux I/O 스케줄러
none (noop): 스케줄링 없음. FIFO 큐. NVMe에 최적. NVMe 자체가 내부적으로 128K+ 큐를 가지고 있어 커널 스케줄링이 불필요

mq-deadline: 읽기/쓰기 별도 큐. deadline으로 starvation 방지. HDD/SATA SSD에 적합. K8s 노드 기본

kyber: 경량 스케줄러. latency 목표 기반으로 I/O 제한. SSD에 적합

bfq (Budget Fair Queueing): 프로세스별 공정한 I/O 대역폭 분배. 데스크톱/인터랙티브 환경

확인: cat /sys/block/sda/queue/scheduler
변경: echo none > /sys/block/nvme0n1/queue/scheduler
NVMe에서 none인 이유
HDD 시대: 디스크 헤드 이동이 가장 비싼 연산. 인접한 섹터의 요청을 모아서 처리(elevator)하면 헤드 이동 최소화 → 큰 성능 향상

SSD/NVMe: 기계적 헤드 없음. 랜덤 접근 성능이 순차와 거의 동일. 커널 스케줄러가 순서를 재배치해봐야 오버헤드만 추가

NVMe 특성:
• 내부 FTL(Flash Translation Layer)이 자체 스케줄링
• 64K 하드웨어 큐 × 64K depth = 40억 동시 I/O 가능
• 커널의 소프트웨어 스케줄러는 방해만 됨

EBS gp3: AWS 내부에서 네트워크를 통해 NVMe 프로토콜 사용. none 스케줄러 권장
● NVMe & Modern Storage

NVMe와 현대 스토리지

NVMe 내부, io_uring 연결, EBS/클라우드 디스크 — 스토리지 계층의 현재와 미래.

Architecture

NVMe 내부 아키텍처

PCIe 직결, 멀티큐, 64K 큐 깊이 — SSD의 성능을 완전히 활용하는 프로토콜
SATA/SAS는 HDD 시대에 설계된 프로토콜이다. 단일 큐, 32 큐 깊이가 SSD의 병렬성을 병목으로 만든다. NVMe(Non-Volatile Memory Express)는 SSD에 최적화되어 PCIe 버스를 직접 사용하고, 최대 64K 큐 × 64K depth의 병렬 I/O를 지원한다.
NVMe vs SATA 비교
SATA SSD:
• AHCI 프로토콜, 1 커맨드 큐, 32 depth
• 순차 읽기: ~550 MB/s (SATA 인터페이스 한계)
• IOPS: ~100K
• latency: ~100μs

NVMe SSD:
• NVMe 프로토콜, 64K 큐, 64K depth/큐
• 순차 읽기: ~3500 MB/s (PCIe Gen3) / ~7000+ (Gen4)
• IOPS: ~500K~1M
• latency: ~10~20μs

핵심: NVMe는 CPU 코어당 별도 Submission Queue + Completion Queue를 할당하여 Lock-free 병렬 I/O를 달성
NVMe Namespace
Namespace: NVMe 디바이스 내의 논리적 저장 단위. 하나의 NVMe 디바이스에 여러 namespace를 생성하여 별도 블록 디바이스로 사용

nvme list — NVMe 디바이스 목록
nvme smart-log /dev/nvme0 — SMART 상태 확인

SRE 관점:
• Wear Leveling: SSD 셀 수명 관리. percentage_used > 80%면 교체 계획
• Temperature: 과열 시 성능 throttling 발생
• Write Amplification Factor (WAF): 실제 쓰기/앱 쓰기. 높으면 SSD 수명 감소
Async

io_uring — NVMe의 성능을 최대로 끌어내는 I/O 인터페이스

커널과 유저스페이스 사이의 공유 링 버퍼로 syscall 오버헤드를 제거한 비동기 I/O
전통적인 read()/write()는 syscall마다 커널-유저 컨텍스트 스위칭이 발생한다. NVMe가 10μs 만에 I/O를 완료하는데, syscall 오버헤드가 수 μs이면 오버헤드가 I/O 자체보다 클 수 있다. io_uring은 공유 메모리 링 버퍼를 사용하여 syscall 없이 I/O를 제출/완료한다.
동작 원리
두 개의 Ring Buffer:
• Submission Queue (SQ): 앱이 I/O 요청을 넣는 큐
• Completion Queue (CQ): 커널이 완료 결과를 넣는 큐

두 큐가 커널-유저 공유 메모리에 있으므로:
1. 앱이 SQ에 요청을 넣음 (syscall 없음!)
2. 커널 폴링 스레드가 SQ에서 요청을 가져감
3. I/O 완료 → CQ에 결과 넣음
4. 앱이 CQ에서 결과를 읽음 (syscall 없음!)

SQPOLL 모드: 커널 스레드가 SQ를 지속 폴링. 완전히 syscall-free. CPU를 소비하지만 latency 극소화
인프라 적용
PostgreSQL 16+: io_method = io_uring (실험적). Direct I/O + io_uring 조합으로 대용량 WAL/checkpoint 성능 향상
RocksDB: io_uring 지원으로 compaction 속도 향상
liburing: io_uring의 Go/C 래퍼 라이브러리

Linux Kernel Deep Internals 문서에서 다룬 io_uring의 SQ/CQ 구조와 직접 연결되는 내용

Go에서: Go 런타임이 아직 io_uring을 네이티브 지원하지 않음. iceber/iouring-go 등 3rd party 라이브러리 존재. Go의 goroutine + epoll 모델이 대부분의 I/O에서 충분히 효율적이지만, 고빈도 디스크 I/O에서는 io_uring이 이점
Cloud

EBS / 클라우드 디스크 내부

네트워크로 연결된 블록 스토리지 — 로컬 디스크와 다른 성능 특성을 이해해야 한다
EBS 내부 동작
EBS는 네트워크 스토리지:
EC2 인스턴스 → EBS Nitro Card → 네트워크 → EBS 서버

물리적으로 같은 머신에 있지 않으므로:
• Latency: 로컬 NVMe ~10μs vs EBS gp3 ~1~3ms (100배 차이)
• IOPS 제한: gp3 기본 3000 IOPS. 프로비저닝으로 16000까지
• Throughput 제한: gp3 기본 125 MB/s. 최대 1000 MB/s
• Burst: gp3는 burst 없음 (일정한 baseline). gp2는 burst 모델

Instance Store (로컬 NVMe):
EC2에 물리적으로 연결된 NVMe. 초저 latency, 초고 IOPS. 하지만 인스턴스 종료 시 데이터 소실. 캐시, 임시 데이터용
SRE 운영 포인트
IOPS 제한 도달:
iostat -x 1 → %util 100% + await 급증
→ EBS IOPS 제한에 도달. 프로비저닝 IOPS 증가 또는 볼륨 타입 변경

EBS Throughput 제한:
대용량 순차 읽기/쓰기에서 125 MB/s가 병목될 수 있음
→ gp3 throughput을 250~1000 MB/s로 증가

EC2 인스턴스별 EBS 대역폭 제한:
인스턴스 자체에도 EBS 대역폭 제한이 있음
m5.xlarge: 최대 4750 Mbps ≈ 594 MB/s
여러 EBS 볼륨의 합이 이 제한을 초과하면 스로틀링

모니터링:
aws cloudwatch get-metric-statistics
VolumeReadOps, VolumeWriteOps, VolumeQueueLength
Prometheus: node_disk_* 메트릭과 CloudWatch 메트릭 교차 확인
💡 스토리지 성능 계층 정리
L1 Cache: ~1ns — CPU 내부
L3 Cache: ~10ns — CPU 공유
DRAM: ~100ns — Page Cache가 여기에
Local NVMe: ~10μs — Instance Store
EBS gp3: ~1ms — 네트워크 스토리지
S3: ~50~100ms — 오브젝트 스토리지

각 계층이 10~100배씩 느려진다. Page Cache 히트 여부가 성능을 100,000배 바꿀 수 있다.
