● SLI Design

SLI 선택과 설계

무엇을 측정할 것인가 — SLI는 SLO 전체 체계의 기초. 잘못된 SLI 위에 좋은 SLO는 없다.

Methodology

SLI 선택 방법론

"사용자가 체감하는 서비스 품질"을 수치화하는 지표를 선택하는 체계적 접근
SLI의 핵심 원칙: 사용자 관점에서 측정한다. 서버 CPU 사용률이나 메모리 사용량은 SLI가 아니다. 사용자는 "응답이 빠른가", "에러가 나지 않는가", "결과가 맞는가"만 관심 있다.

Google SRE Workbook은 서비스 유형별 SLI 메뉴를 제안한다. 모든 서비스에 같은 SLI를 적용하는 것이 아니라, 서비스 특성에 맞게 선택해야 한다.
SLI 선택 기준
1) 사용자 여정(User Journey) 분석: 사용자가 서비스를 어떻게 사용하는가?
2) Critical User Journey 식별: 가장 중요한 경로는 무엇인가?
3) SLI 후보 나열: 해당 경로에서 "좋은 경험"을 수치화할 수 있는 지표는?
4) 측정 가능성 검증: 현재 인프라에서 이 SLI를 자동 수집할 수 있는가?
5) 단순성 우선: SLI 수가 적을수록 좋다 (서비스당 1~3개)
서비스 유형별 SLI 메뉴
Request-driven (API 서버): Availability, Latency, Quality
Pipeline (데이터 처리): Freshness, Coverage, Correctness
Storage (DB, 오브젝트): Durability, Throughput, Latency
Streaming (이벤트): Throughput, Freshness, End-to-end Latency
SLI 공식 (ratio 형태)
SLI = (Good Events) ÷ (Total Valid Events) × 100%
Good Events: 사용자 기준으로 "정상"인 이벤트 수
Total Valid Events: 전체 이벤트에서 의도적 제외(health check 등)를 뺀 수
결과는 항상 0%~100% 범위. 이 ratio 형태가 SLO 설정과 Error Budget 계산을 가능하게 한다.

SLI 유형별 상세 설계

Availability SLI

정의: 전체 요청 중 성공(non-5xx) 응답 비율
공식: count(status≠5xx) / count(total)
주의: 비즈니스 로직이 5xx를 반환하는 경우 noise. 특정 에러 코드를 제외하거나, Latency SLI로 대체 고려
측정점: LB 레벨(가장 사용자에 가까움) 또는 Ingress 레벨

Latency SLI

정의: 전체 요청 중 threshold 이내 응답 비율
공식: count(latency ≤ 500ms) / count(total)
threshold 선택: P50(사용자 대다수 체감)과 P99(최악 사용자 경험) 두 개를 병행하는 것이 이상적
선택 이유: Error Rate SLI보다 noise가 적음. 비즈니스 에러 영향 없음

Freshness SLI (파이프라인)

정의: 데이터가 "최신"인 비율. 마지막 업데이트로부터 threshold 이내
공식: count(age ≤ 10min) / count(total records)
적용: ETL 파이프라인, 검색 인덱스, 캐시 갱신
측정: 타임스탬프 비교 또는 워터마크 기반

Correctness SLI

정의: 응답이 "올바른 결과"를 반환하는 비율
공식: count(correct response) / count(total)
측정 어려움: "올바른"의 기준이 비즈니스 로직에 따라 다름. Synthetic probing(알려진 입출력)으로 검증하는 방법이 일반적
적용: 결제, 추천, 검색 결과 품질

Specification

SLI Specification 작성법

SLI를 엄밀하게 정의하여 측정 모호성을 제거하는 명세 문서
SLI가 모호하면 "이 요청은 good인가 bad인가"에 대한 해석이 팀마다 달라진다. SLI Specification은 이 모호성을 제거하기 위해 측정점, good event 정의, 필터 조건, 집계 방식을 명확히 기술한다.
SLI Spec 예시: API 서버 Latency
서비스: api-server (production)
SLI 유형: Latency
측정점: Traefik IngressRoute (LB 레벨, 가장 사용자에 가까움)
Good Event 정의: HTTP response time ≤ 500ms AND status ≠ 5xx
Total Event 정의: 모든 HTTP 요청. 단, health check (/healthz), readiness probe (/readyz), Prometheus scrape (/metrics) 제외
집계 윈도우: 30일 rolling
데이터 소스: traefik_service_request_duration_seconds_bucket
비고: 비즈니스 로직이 HTTP 500을 반환하는 케이스가 있어 Error Rate SLI 대신 Latency SLI를 사용. HTTP 500 중 비즈니스 에러는 "good"으로 분류해도 무방하나, Latency로 측정하면 이 문제 자체를 회피함
💡 SLI 측정점 선택
사용자에 가까울수록 좋다. LB/Ingress > Application > 내부 메트릭 순으로 선호. 단, 너무 멀면(CDN 등) 제어할 수 없는 외부 요인이 포함됨. 팀이 제어 가능한 범위 내에서 가장 사용자에 가까운 지점이 최적.
● SLO Design

SLO 목표치 설계

어떤 수준을 목표로 할 것인가 — Error Budget 수학, 윈도우 전략, Composite SLO.

Target

SLO 목표치 결정 방법

처음에는 현재 성능 기반으로 보수적으로, 점진적으로 조정
SLO를 처음 설정할 때 가장 흔한 실수는 "99.99%로 하자"처럼 근거 없이 높은 목표를 잡는 것이다. 높은 SLO는 Error Budget을 극단적으로 줄여서 배포 한 번도 못 하는 상황을 만든다.

올바른 접근: 현재 서비스의 실제 성능을 먼저 측정하고, 그 수치를 기반으로 약간 느슨한 SLO를 설정. 운영하면서 점진적으로 조정한다.
단계별 접근
Step 1: SLI를 2~4주 수집하여 baseline 확인
Step 2: 현재 P50 성능을 SLO로 설정 (예: 현재 99.95%면 SLO는 99.9%)
Step 3: Error Budget이 합리적인지 확인 (배포 여유가 있는가?)
Step 4: 분기마다 SLO 리뷰. 필요 시 강화 또는 완화
Step 5: 내부 SLO가 안정되면 SLA 설정 (SLO보다 느슨하게)
서비스 등급별 SLO 가이드
Tier 1 (핵심): 결제, 인증 → 99.95%~99.99%
Tier 2 (중요): 메인 API, 검색 → 99.9%~99.95%
Tier 3 (일반): 관리도구, 내부 서비스 → 99%~99.9%
Tier 4 (실험): 베타, 테스트 → SLO 없음 또는 95%
모든 서비스에 같은 SLO를 적용하면 안 됨
Mathematics

Error Budget 수학

SLO 목표치가 허용하는 장애 시간과 실패 요청 수를 정확히 계산
Error Budget은 SLO가 허용하는 "실패의 총량"이다. 이것을 정확히 계산하면 "배포를 몇 번 더 할 수 있나", "이번 장애가 Budget의 몇 %를 소진했나"를 수치로 답할 수 있다.
▼ Error Budget 계산 테이블 (30일 윈도우)
SLO TargetError Budget월간 허용 다운타임월간 허용 실패 요청 (100만 req/일 기준)
99%1%7.3시간300,000건
99.5%0.5%3.65시간150,000건
99.9%0.1%43.8분30,000건
99.95%0.05%21.9분15,000건
99.99%0.01%4.38분3,000건
Error Budget (시간) = 30일 × 24h × 60m × (1 − SLO target)

Error Budget (요청) = 총 요청 수 × (1 − SLO target)

Budget 소진율 = (실제 bad events ÷ 허용 bad events) × 100%
⚠ 99.99% vs 99.9%의 차이
99.99%는 99.9%보다 10배 적은 Error Budget을 가진다. 월 4.38분 vs 43.8분. 9를 하나 추가하는 것은 운영 비용을 비선형적으로 증가시킨다. 반드시 비즈니스 가치와 대조하여 결정해야 한다.
Window

Rolling Window vs Calendar Window

SLO를 측정하는 시간 윈도우 전략의 선택
SLO 윈도우는 "어느 기간에 걸쳐 SLI를 평가하는가"를 결정한다. 선택에 따라 알림 민감도와 Error Budget 계산이 달라진다.
Rolling Window (권장)
항상 최근 N일을 윈도우로 사용. 예: "항상 최근 30일"
장점: 윈도우 경계에서 Budget이 갑자기 리셋되지 않음. 월초에 장애 나도 월말까지 영향
단점: 과거 장애가 윈도우에서 빠져나갈 때 Budget이 갑자기 회복
구현: Prometheus [30d] range로 자연스럽게 구현
Calendar Window
월 1일~월 말일 같은 고정 기간
장점: 리포팅과 회계 주기에 맞음. 월말에 Budget 리셋 = 새 시작
단점: 월 1일에 대형 장애 → 나머지 29일 동안 배포 동결. 월 30일 장애 → 다음 날 리셋으로 학습 기회 소실
사용처: SLA 보고, 경영진 리포트용
Advanced

Composite SLO

여러 SLI를 결합하여 하나의 종합적 서비스 품질 목표를 만드는 기법
하나의 서비스에 Availability SLI와 Latency SLI를 모두 적용하고 싶을 때, 두 가지 접근이 있다.
① 독립 SLO (권장)
각 SLI에 독립적인 SLO를 설정하고 각각 따로 알림
• Availability SLO: 99.9% over 30d
• Latency SLO: 97% over 30d (P99 ≤ 500ms)
장점: 단순, 어떤 SLI가 위반인지 명확
단점: 알림이 분산, 종합 건강 판단 어려움
② Intersection SLO
"Good event" = Availability good AND Latency good
Good = (status ≠ 5xx) AND (latency ≤ 500ms)
SLI = count(Good) / count(Total)
장점: 하나의 SLI/SLO로 종합 품질 표현
단점: 두 SLI가 동시에 나빠질 때 원인 분리 어려움
실무: SLO starter kit에서 사용 중인 방식과 유사
● Burn Rate Alerting

Multi-Window Burn Rate 알림

SLO 기반 알림의 핵심. 단순 threshold 대신 Error Budget 소진 속도로 알림하는 수학적 프레임워크.

Concept

Burn Rate란?

"현재 속도로 Error Budget을 소진하면, SLO 윈도우 대비 몇 배 빠르게 소진되는가"
Burn Rate = 1이면 SLO 윈도우(30일) 끝에 정확히 Budget이 0이 된다. Burn Rate = 10이면 3일 만에 Budget이 소진된다. Burn Rate가 높을수록 상황이 급박하다.

Burn Rate 알림의 장점: 짧은 장애는 무시하고, 지속적이고 심각한 품질 저하만 감지한다. 단발성 에러 spike에 울리지 않아 Alert Fatigue를 크게 줄인다.
공식
Burn Rate = (현재 Error Rate) ÷ (SLO가 허용하는 Error Rate)
SLO 99.9% → 허용 Error Rate = 0.1%
현재 Error Rate = 1% → Burn Rate = 1% ÷ 0.1% = 10x
= 30일 Budget이 3일 만에 소진되는 속도
Burn Rate → Budget 소진 기간
소진 기간 = SLO Window ÷ Burn Rate
Burn Rate 1x → 30일 (정상)
Burn Rate 2x → 15일
Burn Rate 10x → 3일
Burn Rate 14.4x → ~2일 (=50시간)
Burn Rate 36x → ~20시간
Architecture

Multi-Window, Multi-Burn-Rate 알림

장단기 윈도우를 조합하여 false positive를 제거하고 대응 시간을 최적화
단일 윈도우로 Burn Rate를 보면 문제가 있다. 긴 윈도우(6h)만 보면 이미 끝난 장애에도 계속 울리고, 짧은 윈도우(5m)만 보면 순간 spike에 울린다.

해결: 긴 윈도우(trigger)와 짧은 윈도우(confirmation)를 AND 조건으로 결합. 긴 윈도우가 Budget 소진 추세를 잡고, 짧은 윈도우가 "지금 이 순간에도 문제가 계속되고 있는가?"를 확인한다.
심각도Burn RateLong WindowShort WindowBudget 소진대응
Page (P1)14.4x1h5m2% in 1h즉시 대응. 온콜 호출
Page (P2)6x6h30m5% in 6h긴급 대응. 근무시간 내
Ticket3x1d2h10% in 1d이슈 생성. 1~2일 내 해결
Ticket1x3d6h10% in 3d백로그 추가. 스프린트 내 해결
💡 AND 조건의 핵심
burn_rate_1h > 14.4 AND burn_rate_5m > 14.4
1시간 윈도우가 높아도(과거 장애 잔여), 5분 윈도우가 정상이면 → 알림 안 울림 = 이미 복구됨. 이 AND 조건이 false positive를 90% 이상 줄인다.
Mathematics

14.4x, 6x, 3x, 1x의 수학적 근거

각 Burn Rate 값은 "N시간 내 Error Budget의 몇 %를 소진하는가"에서 역산
Google SRE Workbook의 Multi-Window 설계는 "어떤 시간 내에 Budget의 특정 비율을 소진하면 알림"이라는 원칙에서 수치를 도출한다.
도출 과정
원칙: 30일 SLO 윈도우에서, 특정 시간(detection window) 내에 Budget의 특정 비율(x%)이 소진되면 알림

Burn Rate = (Budget 소진 비율) × (SLO Window) ÷ (Detection Window) ● 1시간 내 2% 소진 → Page: Burn Rate = 0.02 × (30×24) ÷ 1 = 14.4 ● 6시간 내 5% 소진 → Page: Burn Rate = 0.05 × 720 ÷ 6 = 6.0 ● 1일 내 10% 소진 → Ticket: Burn Rate = 0.10 × 720 ÷ 24 = 3.0 ● 3일 내 10% 소진 → Ticket: Burn Rate = 0.10 × 720 ÷ 72 = 1.0

핵심: 14.4라는 "마법의 숫자"는 "1시간 만에 월간 Budget의 2%가 날아가는 속도". 2%가 작아 보이지만 1시간이면 감지하고 대응할 수 있어야 하는 수준.
● Prometheus Implementation

Prometheus로 구현하기

SLI Recording Rules, Burn Rate 계산, Alert Rules, Grafana 대시보드 설계까지 전체 구현.

Recording Rules

SLI Recording Rules

원본 메트릭에서 SLI ratio를 사전 계산하여 저장하는 recording rule
# ── SLI Recording Rules ── # Latency SLI: P99 ≤ 500ms 비율 (Traefik 기준) groups: - name: sli-recording rules: # Good events: 500ms 이내 응답 - record: sli:good_requests:rate5m expr: | sum(rate( traefik_service_request_duration_seconds_bucket{ le="0.5", code!~"5.." }[5m] )) by (service) # Total events: health check 등 제외 - record: sli:total_requests:rate5m expr: | sum(rate( traefik_service_request_duration_seconds_count{ service!~".*healthz.*|.*metrics.*" }[5m] )) by (service) # SLI ratio (0~1) - record: sli:ratio:rate5m expr: | sli:good_requests:rate5m / sli:total_requests:rate5m
Burn Rate

Burn Rate Recording Rules

다양한 윈도우(5m, 30m, 1h, 2h, 6h, 1d, 3d)에서 Error Ratio를 사전 계산
# ── Burn Rate를 위한 Error Ratio (다중 윈도우) ── # SLO Target = 0.999 (99.9%) - name: slo-burn-rates rules: # 각 윈도우별 error ratio = 1 - (good/total) - record: slo:error_ratio:rate5m expr: 1 - (sli:good_requests:rate5m / sli:total_requests:rate5m) - record: slo:error_ratio:rate30m expr: | 1 - ( sum(rate(traefik_service_request_duration_seconds_bucket{le="0.5",code!~"5.."}[30m])) / sum(rate(traefik_service_request_duration_seconds_count{service!~".*healthz.*"}[30m])) ) # 동일 패턴으로 1h, 2h, 6h, 1d, 3d 윈도우도 생성 # (간결화를 위해 생략, 동일 구조로 range만 변경) # Burn Rate = error_ratio / (1 - SLO target) - record: slo:burn_rate:1h expr: slo:error_ratio:rate1h / (1 - 0.999) - record: slo:burn_rate:6h expr: slo:error_ratio:rate6h / (1 - 0.999)

Grafana SLO 대시보드 설계

한눈에 SLO 상태를 파악할 수 있는 대시보드 구성 요소
SLO 대시보드는 "지금 괜찮은가?"에 즉시 답할 수 있어야 한다. 복잡한 그래프보다 핵심 수치를 크게 보여주는 것이 중요.
필수 패널
1) SLI 현재값 (Stat): 큰 숫자로 현재 SLI ratio. 색상으로 상태 표시 (초록/노랑/빨강)
2) Error Budget 잔여 (Gauge): 0~100% 게이지. 남은 Budget 비율
3) Error Budget 소진 추이 (Time Series): 30일간 Budget이 줄어드는 그래프. 기울기로 소진 속도 확인
4) Burn Rate 현재값 (Stat): 1h, 6h, 1d burn rate를 각각 표시
5) SLI 히스토리 (Time Series): SLI ratio의 시간별 변동. SLO 라인을 수평선으로 표시
유용한 추가 패널
6) 배포 마커: 그래프에 배포 이벤트를 annotation으로 표시. SLI 변동의 원인 파악
7) Top Bad Endpoints: 어떤 API 경로가 SLI를 떨어뜨리는지
8) Error Budget 소비 분석: 계획된 변경 vs 비계획 장애 비율
9) 서비스 비교: 여러 서비스의 SLI/Budget을 한눈에
Error Budget 잔여량 계산 PromQL
# Error Budget 잔여 비율 (0~1) 1 - ( (1 - avg_over_time(sli:ratio:rate5m[30d])) / (1 - 0.999) ) # 음수면 Budget 소진됨 # 예: 0.3 → Budget 30% 남음 # 예: -0.2 → Budget 20% 초과 소진됨

Governance

Error Budget Policy 상세 설계

"Budget을 얼마나 쓰면 어떤 행동을 할지" 미리 합의하는 운영 규칙
Error Budget Policy가 없으면 SLO는 예쁜 대시보드에 불과하다. Policy는 Budget 수준에 따라 구체적인 행동(배포 제한, 리뷰 강화, 동결)을 자동으로 트리거하는 합의된 규칙이다.

이 Policy는 Dev팀과 SRE/Ops팀이 함께 합의해야 한다. 한쪽만 결정하면 무시당한다.
Error Budget Policy 템플릿
🟢 Budget > 50% (건강):
• 자유롭게 배포. Feature 개발에 집중
• 실험적 변경(migration, refactoring) 허용
• SLO 리뷰: 분기 1회

🔵 Budget 25~50% (주의):
• 배포 전 추가 리뷰(peer review 필수)
• Canary 배포 비율 10% → 1시간 관찰 후 진행
• Toil 줄이기 작업 백로그에 추가
• SLO 리뷰: 월 1회

🟠 Budget 10~25% (경고):
• Feature 배포 중단. 버그 수정 + 안정화만 배포
• 모든 배포에 Canary + 자동 롤백 설정
• 주간 SLO 리뷰 미팅
• Reliability 개선 작업 우선순위 상향

🔴 Budget < 10% (위험):
• 배포 동결 (Deploy Freeze). 긴급 핫픽스만 예외
• 인시던트 레벨 대응: Incident Commander 지정
• 일일 SLO 리뷰
• Postmortem 작성 및 Action Item 최우선 처리

⛔ Budget 소진 (위반):
• 완전 동결. 모든 변경 중단
• Reliability sprint: 다음 스프린트 전체를 안정화에 투입
• 경영진 리포트 + 근본 원인 분석
• SLO 목표치 자체를 재검토 (너무 높지 않았는가?)

SLO 리뷰 프로세스

SLO는 한 번 설정하고 끝이 아니다. 정기적으로 점검하고 조정해야 살아있는 도구가 된다
분기 리뷰 체크리스트
□ SLI 정의가 여전히 사용자 경험을 대표하는가?
□ SLO 목표치가 적절한가? (너무 높거나 낮지 않은가)
□ Error Budget이 합리적으로 소진되고 있는가?
□ 알림이 정확한가? (false positive/negative 비율)
□ Budget Policy가 실제로 지켜지고 있는가?
□ 새로운 Critical User Journey가 추가되었는가?
□ 의존성 변경이 SLI에 영향을 주는가?
SLO 조정 기준
SLO 강화 (높이기):
• Budget을 거의 안 쓰고 남음 → 목표가 너무 낮음
• 고객 불만이 있는데 SLO 내 → SLI가 잘못됨

SLO 완화 (낮추기):
• Budget이 항상 부족 → 배포 동결이 일상화됨
• 비용 대비 효과가 없는 극단적 목표 (99.999%)
• 서비스 중요도가 낮아짐

흔한 실수와 안티패턴

❌ SLO = 100%

100% 신뢰성은 불가능하고, Error Budget = 0이므로 아무 변경도 못 한다. "최대한 높게"가 아니라 "비즈니스에 맞게"가 올바른 접근.

❌ 서버 메트릭을 SLI로

CPU 80%, 메모리 90%는 SLI가 아니다. 사용자는 CPU를 보지 않는다. 사용자가 체감하는 latency, error rate만 SLI.

❌ Budget Policy 없이 SLO만

SLO가 빨간색인데 아무 행동도 안 하면 대시보드 장식. Policy가 실제 행동(배포 제한 등)에 연결되어야 의미가 있다.

❌ 모든 서비스에 같은 SLO

결제 API와 내부 관리 도구가 같은 99.9%일 수 없다. Tier별 차등이 필수. 중요하지 않은 서비스에 높은 SLO = Budget 낭비.

⚠ SLO를 팀 성과 평가에 사용

SLO 위반이 팀 평가에 직결되면, 팀은 SLO를 낮추거나 SLI를 조작하려는 유인이 생긴다. SLO는 학습과 개선의 도구지 성과 평가 도구가 아니다. DORA 팀도 동일한 경고를 한다.
