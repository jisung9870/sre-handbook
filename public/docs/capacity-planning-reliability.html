● Queueing Theory

큐잉 이론과 Capacity 수학

Little's Law, Utilization-Latency 관계, Amdahl's Law — 시스템 용량을 수학적으로 이해하는 기반.

Fundamental

Little's Law

L = λ × W — 시스템의 동시 작업 수, 처리율, 체류 시간의 관계
큐잉 이론의 가장 기본 법칙이다. 어떤 안정 시스템에서든 성립하며, 분포 가정이 필요 없어 범용적으로 적용할 수 있다. Capacity Planning의 출발점이다.
L = λ × W
변수 정의
L (동시 작업 수): 시스템 안에 있는 요청/작업 수. = 필요한 리소스(스레드, 커넥션)
λ (처리율): 단위 시간당 완료되는 요청 수 (RPS, TPS)
W (체류 시간): 요청이 시스템에 머무는 시간 (지연 시간, response time)
실무 적용 예시
커넥션 풀 사이징:
λ=100 RPS, W=50ms(0.05s)
L = 100 × 0.05 = 5 (최소 커넥션)
여유분 2~3배 = 10~15 커넥션

Pod replica 수:
목표 1000 RPS, Pod당 200 RPS 처리 가능
최소 5 Pod. N+1 여유 = 6 Pod
역방향 활용
L과 λ를 알면 W를 예측: 현재 동시 커넥션 50, RPS 1000 → 평균 지연 시간 = 50/1000 = 50ms
L과 W를 알면 λ를 예측: 스레드 풀 20, 평균 지연 시간 100ms → 최대 처리율 = 20/0.1 = 200 RPS
핵심: 스레드 풀/커넥션 풀이 가득 찬 상태에서 지연 시간이 올라가면 처리율이 반드시 떨어진다. 이것이 과부하에서 시스템이 더 느려지는 수학적 이유.
Critical

Utilization vs Latency — 비선형 관계

사용률이 선형으로 올라가면 지연 시간은 기하급수적으로 올라간다
M/M/1 큐 모델에서 평균 응답 시간은 1/(1-ρ)에 비례한다(ρ = 사용률). 사용률 50%까지는 완만하지만, 80%를 넘으면 지연 시간이 급격히 치솟는다.

이것이 "CPU를 80% 이상 쓰면 안 된다"는 경험칙의 수학적 근거다. 또한 HPA가 CPU 80%가 아닌 60~70%를 타겟으로 해야 하는 이유다.
Response Time = Service Time × 1/(1 − ρ)

ρ=50% → 2× | ρ=80% → 5× | ρ=90% → 10× | ρ=99% → 100×
SRE 실무 임계값
Target utilization 60~70%: 일상적 부하에서 응답 시간이 안정적
경고 at 80%: 지연 시간 증가 시작. scale-out 준비
위험 at 90%+: 지연 시간이 비선형 증가. 이 시점에 스케일링이 완료되지 않으면 장애

K8s HPA targetCPUUtilizationPercentage를 70으로 설정하는 이유: 스케일링에 시간이 걸리므로(Pod 시작, readiness 통과) 80%에 도달하기 전에 트리거해야 한다. 70%에서 시작 → Pod 준비되면 50~60%로 안정화.
Scaling Limit

Amdahl's Law — 병렬화의 한계

시스템의 병렬화 불가능한 부분이 전체 성능 개선의 상한을 결정
아무리 코어/Pod를 늘려도, 직렬로 실행되는 부분(lock, 순차 처리, 공유 자원 접근)이 있으면 성능 향상에 한계가 있다. 코드의 5%가 직렬이면 아무리 코어를 늘려도 최대 20배 향상이 한계.
Speedup = 1 / (S + (1−S)/N)

S = 직렬 비율, N = 병렬 자원 수
S=5%, N=∞ → 최대 20배 | S=10%, N=∞ → 최대 10배
SRE 관점
Pod를 무한히 늘려도: 공유 DB가 직렬 병목이면 성능 향상 한계
DB 커넥션 풀: 모든 Pod이 같은 DB pool을 경쟁 → 직렬 지점
분산 Lock: Redis lock, DB row lock → 병렬화 한계
Ingress: 단일 LB가 병목이면 뒤에 Pod를 아무리 늘려도 의미 없음
대응 전략
직렬 부분을 줄이기: lock-free 자료구조, 비동기 처리, CQRS
sharding: DB를 분할하여 병렬화 가능 범위 확대
캐싱: 공유 자원 접근 빈도를 줄임
비동기 메시징: 동기 호출을 이벤트로 전환
● Load Testing

부하 테스트 방법론

테스트 유형, k6 시나리오 설계, 결과 해석 — "시스템이 얼마나 버티는가"에 대한 과학적 답.

Types

부하 테스트 유형

목적에 따라 다른 테스트 유형을 선택해야 한다
유형목적부하 패턴관측 지표
Baseline정상 상태 성능 기준선예상 평균 트래픽 유지P50/P99 latency, error rate, 리소스 사용률
Stress한계점 발견점진적 증가 → 시스템 한계까지어느 부하에서 SLO 위반이 시작되는가
Spike급격한 부하 대응 능력짧은 시간에 급증 → 정상 복귀복구 시간, HPA 반응 속도, error spike
Soak장시간 안정성 (메모리 누수 등)일정 부하를 수 시간~수 일 유지메모리 증가 추이, GC 패턴, 커넥션 누수
Breakpoint절대 최대 용량시스템 완전히 무너질 때까지 증가최대 RPS, 장애 발생 패턴
Tool

k6 시나리오 설계

Grafana k6로 실전 부하 테스트 시나리오를 구성하는 패턴
// k6 종합 시나리오: baseline → stress → spike → recovery import http from 'k6/http'; import { check, sleep } from 'k6'; export const options = { scenarios: { // ① Baseline: 평균 부하 5분 유지 baseline: { executor: 'constant-arrival-rate', rate: 100, // 100 RPS timeUnit: '1s', duration: '5m', preAllocatedVUs: 50, }, // ② Stress: 점진적 증가 stress: { executor: 'ramping-arrival-rate', startRate: 100, stages: [ { duration: '2m', target: 300 }, { duration: '5m', target: 300 }, // 300 RPS 유지 { duration: '2m', target: 500 }, { duration: '5m', target: 500 }, // 500 RPS 유지 { duration: '2m', target: 0 }, // ramp-down ], preAllocatedVUs: 200, }, // ③ Spike: 급증 후 즉시 복귀 spike: { executor: 'ramping-arrival-rate', startRate: 100, stages: [ { duration: '10s', target: 1000 }, // 10초만에 10배 { duration: '1m', target: 1000 }, { duration: '10s', target: 100 }, // 급감 ], startTime: '20m', // stress 후 시작 preAllocatedVUs: 500, }, }, thresholds: { 'http_req_duration': ['p(99)<500'], // SLO: P99 < 500ms 'http_req_failed': ['rate<0.01'], // 에러율 < 1% }, }; export default function () { const res = http.get('https://api.example.com/v1/items'); check(res, { 'status 200': (r) => r.status === 200, 'latency < 500ms': (r) => r.timings.duration < 500, }); }
💡 constant-arrival-rate vs constant-vus
constant-vus는 "N명의 사용자가 계속 요청"하는 모델. 서버가 느려지면 RPS가 떨어진다.
constant-arrival-rate는 "초당 N개의 요청을 보장"하는 모델. 서버가 느려져도 RPS를 유지 → 실제 과부하 시뮬레이션에 적합.
Analysis

결과 해석 가이드

부하 테스트 결과에서 무엇을 어떻게 읽어야 하는가
핵심 지표 해석
P50 vs P99 격차: 격차가 크면 tail latency 문제. GC, lock contention, slow query 의심
P99가 급등하는 RPS: 그것이 현재 용량의 한계. SLO와 대조
Error rate 증가 시점: error가 시작되는 RPS가 진짜 용량. latency 증가보다 뒤에 오면 graceful
Recovery 시간: spike 후 P99가 정상으로 돌아오는 데 걸리는 시간 = 시스템 탄력성
함께 관측할 것
K8s 메트릭: HPA 이벤트, Pod 수 변화, CPU/Memory 사용률
시스템 메트릭: CPU throttling (nr_throttled), 커넥션 풀 사용률, GC pause
인프라 메트릭: 노드 CPU, 네트워크 throughput, 디스크 IOPS
Grafana에서 k6 결과와 시스템 메트릭을 같은 시간축에 겹쳐 보는 것이 핵심
● Auto-Scaling

K8s 오토스케일링 심화

HPA, VPA, Karpenter — 워크로드와 인프라를 자동으로 확장하는 메커니즘과 의사결정 로직.

Workload

HPA (Horizontal Pod Autoscaler) 심화

replica 수를 자동 조정하는 HPA의 내부 알고리즘과 튜닝 포인트
HPA는 현재 메트릭 값과 목표 값의 비율로 desired replica 수를 계산한다. 단순해 보이지만 안정화 윈도우, 스케일링 정책, 메트릭 선택 등 튜닝 포인트가 많다.
스케일링 공식
desired = ceil(current_replicas × (current_metric / target_metric)) 예: 현재 3 Pod, CPU 사용률 90%, 타겟 70% desired = ceil(3 × (90/70)) = ceil(3.86) = 4
v2 핵심 설정
behavior.scaleUp.stabilizationWindowSeconds: 0 (즉시)
behavior.scaleDown.stabilizationWindowSeconds: 300 (5분 대기 → flapping 방지)
behavior.scaleUp.policies: maxPods, percent 기반 증가량 제한
behavior.scaleDown.policies: 한 번에 줄이는 최대량
Custom Metrics 스케일링
CPU만으로 스케일링하면 I/O-bound나 외부 의존성이 큰 워크로드를 잘못 판단할 수 있다. 비즈니스 메트릭 기반 스케일링이 더 정확하다:

RPS 기반: prometheus-adapter로 http_requests_per_second 메트릭을 HPA에 노출
Queue 길이: Kafka consumer lag, SQS message count → 처리해야 할 양 기반
응답 시간: P99 지연 시간이 임계값을 넘으면 스케일 아웃

여러 메트릭을 동시에 사용하면 HPA는 가장 많은 replica를 요구하는 메트릭을 채택한다.
Vertical

VPA (Vertical Pod Autoscaler)

Pod의 CPU/Memory request/limit을 자동 조정하는 수직 스케일러
VPA는 과거 사용 패턴을 분석하여 적절한 resource request 값을 추천한다. 너무 높은 request는 리소스 낭비, 너무 낮으면 throttling/OOM. VPA가 최적값을 찾아준다.
동작 모드
Off: 추천값만 제시, 적용 안 함 (가장 안전)
Initial: Pod 생성 시에만 적용
Auto: 실행 중인 Pod도 재시작하여 적용 (주의!)
권장: 처음에는 Off → 추천값 검토 → 수동 적용. 안정화 후 Auto 고려
HPA와의 공존
같은 메트릭으로 HPA+VPA 동시 사용 금지 (충돌)
안전한 조합:
• HPA: CPU 기반 스케일링
• VPA: Memory request 자동 조정 (Off/Initial 모드)
• 또는 HPA는 custom metric, VPA는 CPU/Memory
Infrastructure

Karpenter — 노드 레벨 오토스케일링

Pending Pod을 감지하여 최적의 노드를 즉시 프로비저닝하는 K8s 노드 스케일러
HPA가 Pod 수를 조정한다면, Karpenter는 그 Pod이 실행될 노드를 조정한다. Cluster Autoscaler와 달리 노드 그룹에 의존하지 않고, Pending Pod의 요구사항(CPU, Memory, GPU, topology)을 분석하여 최적의 인스턴스 타입을 직접 선택·프로비저닝한다.
핵심 동작
1) Pod 스케줄링 실패 (Pending) 감지
2) Pod의 resource request, nodeSelector, affinity, topology 분석
3) 가장 비용 효율적인 인스턴스 타입 선택
4) 노드 프로비저닝 (~60초 내)
5) 유휴 노드 감지 → 통합(consolidation) → 비용 절감
EKS 환경 설정 포인트
NodePool: 허용할 인스턴스 패밀리, 크기, AZ 정의
EC2NodeClass: AMI, 서브넷, 보안그룹, userData
consolidationPolicy: WhenEmptyOrUnderutilized (비용 최적화)
ttlSecondsAfterEmpty: 빈 노드 유지 시간 (기본 30초)
limits: 최대 CPU/Memory 총량으로 비용 상한 설정
Karpenter v1.0+ 에서 NodePool/EC2NodeClass CRD 사용

스케일링 안티패턴

❌ DB를 scale-out

DB 커넥션은 유한하다. Pod를 10배로 늘리면 DB 커넥션 풀이 폭발. 스케일링 전 DB 용량 확인 필수. PgBouncer 같은 connection pooler 도입 고려.

❌ CPU 100% 타겟

Utilization-Latency 비선형 관계로 80% 넘으면 latency 급등. HPA 타겟은 60~70%가 적정. 스케일링 지연(Pod 시작 시간)까지 고려.

❌ Flapping 무시

스케일업/다운이 반복(flapping)되면 Pod 재시작으로 오히려 불안정. scaleDown.stabilizationWindowSeconds를 충분히 (5~10분) 설정.

❌ Readiness 무시

새 Pod이 readiness probe를 통과하기 전에 트래픽을 받으면 에러 spike. readinessProbe + startupProbe를 반드시 설정.

Advanced

Canary 배포 심화 — 메트릭 기반 자동 판단

Canary 트래픽의 SLI를 기존 버전과 비교하여 자동으로 승격/롤백 결정
단순 Canary는 "10% 트래픽으로 보내고 수동으로 확인"이다. 메트릭 기반 Canary는 Canary와 Stable의 error rate, latency를 자동 비교하여 승격(promote) 또는 롤백(rollback)을 결정한다.
자동 판단 기준 예시
Success Rate: canary ≥ stable × 0.99 (1% 이내 차이)
P99 Latency: canary ≤ stable × 1.1 (10% 이내 증가)
Error Rate: canary - stable < 0.5%

모든 조건 충족 → 트래픽 비율 확대 (10→25→50→100%)
하나라도 실패 → 즉시 롤백 (0%)
구현 도구
Argo Rollouts: AnalysisTemplate + Prometheus 연동
successCondition: result[0] >= 0.99
failureLimit: 3
Flagger: Istio/Traefik/Nginx와 통합
ArgoCD + Argo Rollouts: GitOps + Progressive Delivery 조합
Argo Rollouts AnalysisTemplate 예시
apiVersion: argoproj.io/v1alpha1 kind: AnalysisTemplate metadata: name: canary-success-rate spec: metrics: - name: success-rate interval: 60s successCondition: "result[0] >= 0.99" failureLimit: 3 provider: prometheus: address: http://prometheus:9090 query: | sum(rate(http_requests_total{ status!~"5..", rollouts_pod_template_hash="{{args.canary-hash}}" }[5m])) / sum(rate(http_requests_total{ rollouts_pod_template_hash="{{args.canary-hash}}" }[5m]))
Strategy

Progressive Delivery

배포를 점진적으로 진행하면서 각 단계에서 메트릭을 검증하는 전체 프레임워크
Progressive Delivery = Canary + Feature Flag + 자동 분석의 조합이다. 코드 배포와 기능 릴리스를 분리하고, 각 단계에서 자동 검증을 통해 위험을 최소화한다.
단계
1) Deploy (코드 배포): 새 버전의 Pod을 배포하되 트래픽은 보내지 않음
2) Feature Flag: 기능을 활성화하되 내부 사용자 또는 특정 % 에게만
3) Canary: 트래픽의 5~10%를 새 버전으로. 메트릭 자동 분석
4) 점진적 확대: 분석 통과 시 25→50→100%
5) Full Rollout: 모든 트래픽이 새 버전으로
GitOps + Progressive Delivery
ArgoCD: Git에서 Rollout 리소스를 관리
Argo Rollouts: Canary/BlueGreen 전략 + AnalysisRun
Grafana Alerting: 배포 중 SLO 위반 시 자동 롤백 트리거

파이프라인: Git push → ArgoCD Sync → Rollout Canary → AnalysisRun (Prometheus 쿼리) → 자동 승격 또는 자동 롤백

프로덕션 준비 검증

Production Readiness Review, FMEA, Game Day — 서비스를 안전하게 프로덕션에 올리기 위한 체크 프로세스.

Checklist

Production Readiness Review (PRR)

새 서비스를 프로덕션에 배포하기 전, SRE 관점에서 검증하는 체계적 체크리스트
Google SRE에서 시작된 관행이다. 서비스가 프로덕션 환경에서 안정적으로 운영될 수 있는 최소 조건을 검증한다. PRR을 통과하지 못하면 프로덕션 배포를 보류한다.
관측 가능성
□ SLI/SLO가 정의되어 있는가
□ Metrics, Logs, Traces가 수집되는가
□ Grafana 대시보드가 준비되어 있는가
□ SLO 기반 알림(Burn Rate)이 설정되어 있는가
□ 알림 수신자(온콜)가 지정되어 있는가
안정성
□ Health check (liveness/readiness/startup probe) 구성
□ Graceful shutdown 구현 (SIGTERM 처리)
□ Resource request/limit 설정
□ PodDisruptionBudget 설정
□ HPA 설정 및 부하 테스트 통과
운영
□ 롤백 절차가 문서화되어 있는가
□ Runbook이 작성되어 있는가
□ 의존성 장애 시 행동(Circuit Breaker, fallback)이 정의되어 있는가
□ 데이터 백업/복구 절차가 있는가
□ 비밀 정보가 Secret/Vault로 관리되는가
보안
□ 컨테이너 이미지 취약점 스캔 통과
□ 최소 권한 원칙 (capabilities drop ALL)
□ NetworkPolicy 적용
□ seccomp 프로파일 적용
□ non-root 실행
Risk Analysis

FMEA — Failure Mode and Effects Analysis

"이 시스템에서 무엇이 어떻게 고장날 수 있는가?"를 체계적으로 분석
FMEA는 장애가 발생하기 전에 가능한 장애 모드를 식별하고, 각각의 영향과 확률, 감지 가능성을 평가하여 우선순위를 정하는 분석 기법이다. 제조업에서 시작되어 소프트웨어/SRE에 도입됨.
FMEA 테이블 컬럼
장애 모드: 무엇이 고장나는가? (DB 연결 실패, 디스크 풀, 인증서 만료...)
영향 (Severity 1~10): 사용자에게 미치는 영향. 10=전체 서비스 중단
발생 확률 (Occurrence 1~10): 얼마나 자주? 10=매주
감지 가능성 (Detection 1~10): 발생 시 얼마나 빨리 감지? 10=감지 불가
RPN = S × O × D: Risk Priority Number. 높을수록 우선 대응
FMEA 적용 예시
DB 연결 풀 소진: S=8, O=4, D=3 → RPN=96
인증서 만료: S=9, O=2, D=8 → RPN=144 ← 우선 대응!
디스크 풀: S=7, O=3, D=2 → RPN=42
DNS 장애: S=9, O=2, D=4 → RPN=72

RPN 높은 순서대로: 인증서 만료 → DB 풀 소진 → DNS → 디스크
Practice

Game Day — 장애 훈련

실제 장애를 의도적으로 발생시켜 대응 체계를 검증하고 약점을 발견하는 훈련
FMEA로 장애 모드를 식별했다면, Game Day에서 실제로 그 장애를 일으켜 본다. "DB가 죽으면 어떻게 되나?"를 상상하는 것과, 실제로 DB를 죽여보는 것의 차이는 크다.

Netflix의 Chaos Monkey에서 시작된 Chaos Engineering의 실무 적용 형태다.
Game Day 프로세스
1) 가설 수립: "DB 레플리카가 죽으면 read 트래픽이 primary로 failover되고, P99 latency가 200ms 이내로 유지될 것이다"
2) 범위 정의: 영향 범위, 롤백 계획, abort 조건
3) 실행: 장애 주입 (Pod kill, 네트워크 지연, 디스크 풀 등)
4) 관찰: SLI 변화, 알림 발생, 대응 시간 기록
5) 분석: 가설 검증, 발견된 약점 정리
6) 개선: Action Item 생성 → 백로그 추가
장애 주입 도구
K8s 네이티브:
• kubectl delete pod — 기본적인 Pod kill
• kubectl drain node — 노드 장애 시뮬레이션

전문 도구:
• Chaos Mesh: K8s 네이티브 Chaos Engineering. Pod, Network, I/O, Time, Kernel fault 주입
• Litmus: CNCF 프로젝트. 실험 카탈로그 제공
• tc (traffic control): 네트워크 지연/패킷 손실 주입
tc qdisc add dev eth0 root netem delay 100ms loss 5%
💡 Game Day를 Error Budget과 연결
Game Day에서 소비되는 Error Budget도 계산한다. "이 훈련으로 Budget의 2%를 썼다"를 정량화하면, Game Day 빈도와 범위를 Budget 내에서 관리할 수 있다. Budget이 부족하면 축소된 범위로 실행.
