● Overview

게임 퍼블리싱이란?

게임 개발사가 만든 게임을 시장에 출시·운영하는 모든 활동을 책임지는 사업 모델. SRE 관점에서 퍼블리싱은 곧 여러 게임이 공유하는 공통 플랫폼을 안정적으로 운영하는 일이다.

게임 퍼블리싱이란?

전통적 정의

개발사(Developer)는 게임을 만들고, 퍼블리셔(Publisher)는 게임을 시장에 내보내고 운영한다. 마케팅, 배포, 결제, 고객지원, 컴플라이언스 등을 담당.

현대적 형태

퍼블리셔는 자체 IP도 개발하면서 외부 IP를 동시에 퍼블리싱하는 듀얼 트랙이 일반적. 모바일·PC·콘솔을 모두 다루는 멀티 플랫폼 퍼블리싱이 표준이 되었다.

SRE의 역할

여러 게임이 공유하는 공통 플랫폼 (계정·결제·CDN·텔레메트리·CS)의 가용성·정합성·보안을 책임진다. 게임서버 자체는 게임팀(또는 개발사)이 운영하는 경우가 많다.

왜 SRE가 중요한가

플랫폼 장애는 모든 게임에 동시 영향을 준다. 결제 1시간 다운 → 매출 직접 손실, 계정 시스템 다운 → 모든 게임 로그인 불가. 일반 웹 서비스보다 폭발 반경이 크다.

개발사 vs 퍼블리셔의 일반적 R&R

계약마다 R&R은 다르지만, 다음이 가장 흔한 구조다. 퍼블리셔는 "게임 외 모든 것"을 맡는다고 이해하면 큰 그림이 잡힌다.

SRE 관점에서 본 퍼블리싱

퍼블리셔의 SRE는 일반 B2C 웹 서비스 SRE 일부 + 게임 도메인 특수성의 조합이다. 다음 4가지가 일반 웹 서비스와 다른 핵심 차이다.

① 폭발 반경이 N배

플랫폼 SDK 한 곳의 장애가 운영 중인 N개 게임 모두에 동시 영향. 개별 게임 한 곳의 장애와 격이 다르다.

② 트래픽이 매우 비균질

신작 런칭/이벤트는 평소 대비 10~100배 스파이크. 일반 웹 서비스의 black friday 수준 스파이크가 거의 매월 발생한다.

③ 데이터 정합성이 critical

결제·인벤토리·계정은 한 건 어그러져도 사용자 분노·CS 폭주·매출 손실. eventual consistency보다 strict transaction이 흔하다.

④ 외부 의존도 높음

스토어 (Apple/Google/Steam) 정책 변경, 결제 PG 장애, 외부 개발사 빌드 — 퍼블리셔 통제 밖의 장애 요인이 많다.

♥ KEY
"멀티 게임 공통 플랫폼"이 게임 퍼블리싱 SRE의 핵심 키워드다. 이 한 문장이 우선순위·아키텍처·사고 대응 모든 것을 관통한다. 플랫폼이 멈추면 라인업 전체가 멈춘다.
● Common Platform SDK

공통 플랫폼 SDK 구조

퍼블리셔는 라인업 전체에서 재사용되는 5대 영역의 플랫폼 SDK를 운영한다. 각 게임은 이 SDK를 통합해 공통 기능을 사용한다.

5대 영역

① 계정 · 인증 (Account & Auth)

통합 계정, 소셜 로그인 (LINE/Apple/Google/Facebook), Steamworks 인증, 토큰 발급, 디바이스 관리, 비밀번호 복구.

관련 가이드: P2 통합 계정 & 인증

② 결제 · IAP (Payment)

스토어 결제 (Apple/Google/Steam/원스토어), 영수증 검증, 환불, 청구 정합성, 미수령 아이템 복구.

관련 가이드: P3 결제 & IAP

③ 푸시 · 메시징 (Push)

FCM/APNs 통합, 인앱 메시지, 이메일, 캠페인 타겟팅, 다국어, A/B 테스트.

P5 LiveOps 안에서 다룸

④ 텔레메트리 · BI

이벤트 로그 수집, 스키마 거버넌스, KPI 집계 (DAU/MAU/ARPU/LTV/리텐션), 대시보드, A/B 테스트.

관련 가이드: P6 텔레메트리 & BI

⑤ 고객지원 (CS) · 운영

티켓 시스템, FAQ, 환불·문의 워크플로우, 어뷰징 탐지, 게임 마스터(GM) 도구.

관련 가이드: P8 CS · 어뷰징 · 컴플라이언스

+ 부가 영역

점검·공지 시스템 (P5), 패치·CDN (P4), 신작 런칭 인프라 (P7), Steam 통합 (P9). 5대 영역에 종속되거나 연결되는 보조 플랫폼들.

멀티 게임 환경에서의 재사용성

N개 게임이 같은 플랫폼 SDK를 쓴다는 건 SRE 관점에서 이중적 의미를 갖는다.

→ 좋은 점

한 번 잘 만들면 N번 재사용. 보안 패치 한 번에 N개 게임 보호. KPI·대시보드 표준화. 운영 효율이 게임 수에 비례하지 않고 sublinear로 늘어난다.

→ 위험

SDK 한 곳의 버그가 N개 게임에 동시 전파. 호환성 깨면 N개 게임의 클라이언트 빌드를 모두 새로 받아야 함. 플랫폼 변경은 항상 N배의 신중함이 필요하다.

♥ KEY
플랫폼 변경 시 표준 절차: (1) 호환성 매트릭스 정의 → (2) 단계적 롤아웃 (1게임 → 일부 → 전체) → (3) 게임별 카나리 → (4) 자동 롤백 트리거. 게임 한 곳에서 회귀가 보이면 즉시 차단.
● Self IP vs External IP

자체 IP vs 외부 IP 퍼블리싱

퍼블리셔는 자체 개발 IP와 외부 개발사 IP를 동시에 운영한다. 두 워크플로우는 SRE 관점에서 책임 분담·사고 대응·빌드 파이프라인이 크게 다르다.

R&R 매트릭스

동일한 운영 영역도 IP 유형에 따라 책임자가 달라진다. 외부 IP는 플랫폼 운영(퍼블리셔) + 게임 운영(외부 개발사)의 협업 모델이다.

영역자체 IP외부 IP 퍼블리싱
빌드 파이프라인사내 CI/CD 직접 통제개발사 빌드 수령 → 검수 → 등록
게임서버 인프라퍼블리셔가 운영개발사가 운영 (퍼블리셔는 모니터링·지원)
플랫폼 SDK 통합사내 통합 워크숍개발사에 SDK 가이드·테스트 환경 제공

영역별 비교

① 빌드 파이프라인

자체 IP는 사내 CI/CD에서 git push → 빌드 → 검수 → 배포가 한 흐름. 외부 IP는 개발사가 빌드 산출물을 전달해 주면 퍼블리셔가 검수(QA·악성코드·SDK 통합)하고 스토어 등록. 빌드 검수 단계가 추가된다.

② 인프라 책임

자체 IP는 게임서버까지 퍼블리셔가 운영(또는 그룹 계열사). 외부 IP는 게임서버는 개발사 인프라에서 돌리고, 퍼블리셔는 플랫폼 SDK·CDN·네트워크 게이트웨이 등 외곽만 책임. 모니터링은 양사 공유.

③ 패치·LiveOps

자체 IP는 점검·이벤트·핫픽스를 사내에서 결정. 외부 IP는 개발사와 패치 cadence 협의가 필수. 점검 윈도우, 시차, 인력 가용성을 모두 맞춰야 한다.

④ 사고 대응

자체 IP는 사내 온콜로 종결. 외부 IP는 공동 IR — 양사 온콜 동시 소집, 공유 워룸, 책임 영역 합의 후 분담. 시차가 있으면 더 복잡하다.

공동 IR (Incident Response) 흐름

외부 IP에서 사고가 났을 때의 표준 IR 흐름. 책임 영역이 모호한 초기에 시간을 잃지 않는 것이 관건이다.

● Mobile vs PC/Steam

모바일 vs PC/Steam 운영 차이

같은 게임이라도 모바일과 PC/Steam은 결제·환불·심사·anti-cheat·인증이 모두 다르다. 멀티 플랫폼 출시 게임은 양쪽을 모두 이해해야 한다.

패치 cadence의 근본적 차이

모바일

스토어 심사가 끼어 있어 긴급 패치도 수 시간~수 일 소요. 그래서 게임 데이터(아이템·이벤트·드롭률)는 서버 사이드에서 핫픽스하고, 클라이언트 패치는 minimum으로 줄이는 설계가 표준.

PC/Steam

Steam branch에 빌드를 push하면 분 단위로 즉시 배포. 다만 사용자는 게임을 종료한 다음 재시작해야 적용. 빌드 분리(beta/staging/live), 롤백, 차등 배포가 자유롭다.

♥ KEY
같은 게임을 모바일·PC 동시 운영할 때, 핫픽스 데이터는 서버에서, 클라이언트 패치는 플랫폼별 cadence를 인정해야 한다. 모바일에서 며칠 전 만든 패치를 Steam에선 오늘 같이 내보내는 식의 동기화 전략 필요. P10 크로스 플랫폼 운영에서 깊이 다룬다.
● SRE Scope

퍼블리셔 SRE의 5대 책임 영역

일반 SRE 책임에 게임 도메인 특수성을 더한 영역. 평소에 어디에 시간을 쓰는지가 곧 직무 정의다.

5대 책임 영역

① 가용성 · 지연 · 스파이크

플랫폼 SDK SLO (가용성·지연), 신작 런칭/이벤트 스파이크 처리, 캐파시티 플래닝, autoscaling. 일반 SRE 영역이지만 스파이크 빈도가 매우 높음.

→ P7 신작 런칭 캐파시티

② 데이터 정합성

결제↔아이템 지급 정합성, 인벤토리·진행도 백업, 계정 이전·복구. 한 건 어그러지면 사용자 분노 + 매출 손실 + CS 폭주.

→ P3 결제 / F5 게임 데이터

③ LiveOps 안정성

점검·공지·이벤트 스케줄, 핫픽스 배포, 게임 데이터 변경의 안전한 적용. 매주 반복되는 운영 사이클이라 자동화·체크리스트가 핵심.

→ P5 LiveOps

④ 보안 · 어뷰징

계정 탈취, 결제 사기, 매크로·봇, 어뷰징 탐지, anti-cheat 운영. 게임 도메인 고유 영역이 많음.

→ F4 anti-cheat / P8 어뷰징

⑤ 컴플라이언스

한국 게임 등급 분류 (게임물관리위원회·자체등급분류사업자), 청소년 보호 (게임시간 선택제), 약관·개인정보, 글로벌 규제 (GDPR, COPPA, 중국 판호 등). 법무팀과의 협업이 잦음.

→ P8 CS · 어뷰징 · 컴플라이언스

한국 컴플라이언스 핵심 (2026년 5월 기준)

한국 게임 운영 시 가장 자주 마주치는 컴플라이언스 항목. 정책은 변하므로 실제 적용 시 공식 자료 확인 필요.

게임 등급 분류

모바일·아케이드·시험용은 게임물관리위원회(GRAC), PC·콘솔은 게임콘텐츠등급분류위원회(GCRB)가 위탁받아 처리 (2025-11-01부터 청소년이용불가까지 위탁). 자체등급분류사업자(Apple/Google/Steam/닌텐도/카카오 등) 지정도 운영 중.

청소년 보호

강제적 셧다운제는 2022-01-01에 폐지되었다. 현재는 게임시간 선택제(게임산업법 §12조의3)로 일원화 — 만 18세 미만 본인 또는 법정대리인이 원하는 시간대 설정. 결제 한도(한도 초과 시 부모 동의 등) 별도 규제 존재.

개인정보 · 데이터 처리

개인정보보호법(PIPA) + 정보통신망법. 글로벌 운영 시 GDPR(EU), CCPA(미국 캘리포니아), 일본 APPI 등 동시 준수.

확률형 아이템 표시

2024년 3월 시행된 게임산업법 개정에 따라 확률형 아이템(가챠) 확률 정보 표시 의무가 법제화됨. 클라이언트·웹·광고에 모두 표기 필요.

⚠ NOTE
위 항목은 변경이 잦다. 셧다운제 폐지처럼 큰 변화도 종종 있고, 자체등급분류 위탁 범위도 매년 갱신된다. 출시 시점에 반드시 게임물관리위원회·문화체육관광부 공식 공지를 재확인.
