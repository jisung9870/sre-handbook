● On-Call System

온콜 체계 설계

로테이션, 에스컬레이션, 핸드오프, 건강 관리 — 지속 가능한 온콜 문화를 만드는 실무 가이드.

Architecture

온콜 체계 설계 원칙

누가, 언제, 어떤 순서로 장애에 대응하는가를 구조화하는 체계
온콜은 "알림이 울리면 정해진 사람이 대응한다"는 단순한 구조이지만, 잘못 설계하면 번아웃, 대응 지연, 지식 편중을 유발한다. Google SRE는 온콜을 엔지니어링 업무의 일부로 취급하며, 전체 업무의 25% 이하로 제한할 것을 권장한다.
핵심 설계 요소
Primary On-Call: 첫 번째 대응자. 알림 수신 후 5~15분 내 acknowledge
Secondary On-Call: Primary가 응답 못 할 경우 자동 에스컬레이션
Escalation Path: Primary → Secondary → Team Lead → Manager
에스컬레이션 타임아웃: P1: 15분, P2: 30분, P3: 2시간
Solo Engineer 환경
혼자 온콜을 감당할 때 (1인 DevOps):
• 자동 복구 최대화: 자동 롤백, 자동 재시작, self-healing
• Runbook 철저히 정비: 새벽에 반만 깨어서도 따라할 수 있게
• 알림 품질에 투자: false positive = 수면 방해 = 번아웃
• 비상 연락망: 최소 1명의 백업 (팀장 등)
• HolmesGPT 같은 AI 지원 인시던트 트리아지 도입 고려
알림 채널 설계
Page (즉시): PagerDuty/Opsgenie 전화+SMS. SLO Burn Rate Critical, 서비스 전체 장애
Urgent: Slack DM + 모바일 푸시. SLO Burn Rate High, 주요 기능 장애
Ticket: Slack 채널 + 이슈 자동 생성. Burn Rate Warning, 성능 저하
Log: Grafana Alerting → Slack 채널. 참고용. 사람을 깨우지 않음

핵심 원칙: Page로 올라오는 알림은 반드시 즉각 행동이 필요한 것만. 그렇지 않으면 Ticket으로 낮춰야 한다.
Schedule

로테이션 & 핸드오프

온콜 교대 주기, 인수인계 절차, 공정한 분배
로테이션 설계
주기: 1주 교대가 일반적. 2주는 부담, 3일은 컨텍스트 전환 비용
시작: 월요일~화요일 시작이 이상적 (주말에 교대하지 않도록)
시간대: Follow-the-Sun (글로벌 팀이면 시간대별 교대)
공정성: 주말/공휴일 온콜은 추가 보상 (대체 휴무, 수당)
최소 인원: 6~8명이 있어야 지속 가능한 로테이션 (1인 환경은 자동화로 보완)
핸드오프 체크리스트
교대 시 인수인계 사항:
□ 현재 진행 중인 인시던트 상태
□ 이번 주기 동안 발생한 주요 이벤트 요약
□ 주의해야 할 예정된 변경사항 (배포, 마이그레이션)
□ 알려진 이슈 (intermittent 장애 등)
□ 다음 온콜 담당자에게 특별 전달사항

형식: Slack 채널에 구조화된 핸드오프 메시지 작성
또는 Confluence/Notion에 온콜 일지 유지
Wellbeing

온콜 건강 관리

지속 가능한 온콜은 엔지니어의 번아웃을 방지하는 설계에서 시작
온콜이 고통스러우면 시스템에 문제가 있는 것이다. 알림이 너무 많거나, 자동화가 부족하거나, Runbook이 없는 것이 원인이다. Google SRE는 "온콜이 고통스럽다면, 그것은 개선할 시스템 문제의 목록"이라고 본다.
건강 지표
주당 page 수: 2건 이하가 이상적. 5건 이상이면 시스템 개선 필요
새벽 page 비율: 20% 이하. 높으면 알림 시간대 재설계 또는 자동 완화
Page당 대응 시간: 대부분 30분 이내 해결되어야. 1시간+ 이면 Runbook 개선
False positive 비율: 10% 이하. 높으면 알림 규칙 재검토 (Alert Fatigue)
온콜 후 번아웃 설문: 교대 후 간단한 피드백 수집
개선 전략
Toil 자동화: 같은 Runbook을 3번 이상 실행하면 자동화 대상
알림 품질 개선: Symptom-based + Burn Rate 전환
Noise 제거: Actionable하지 않은 알림은 Ticket/Log로 강등
온콜 레트로: 교대 후 "이번 주기에 가장 고통스러웠던 것"을 공유하고 개선 백로그에 추가
보상: 대체 휴무, 온콜 수당, 경험 공유 세션
⚠ 1인 DevOps 환경의 현실
현실적으로 Solo DevOps 환경에서 24/7 온콜은 지속 불가능하다. 자동 복구 + 알림 품질 + Runbook 자동화에 최대한 투자하고, 비즈니스 시간 외에는 Critical만 Page되도록 설계해야 한다. HolmesGPT 같은 AI 도구로 초기 트리아지를 자동화하면 새벽 대응 부담을 크게 줄일 수 있다.

인시던트 대응 체계

심각도 분류, 역할 분리, 대응 흐름, 커뮤니케이션 — 혼란 속에서 체계적으로 대응하는 프로세스.

Classification

심각도 분류 체계

장애의 영향 범위와 긴급도에 따라 대응 수준을 결정하는 표준 분류
심각도정의예시대응 시간에스컬레이션
P1 Critical서비스 전체 장애 또는 데이터 손실 위험전체 API 응답 불가, DB 데이터 손상, 보안 침해즉시 (5분 내 ACK)자동으로 TL + 매니저
P2 Major주요 기능 장애, 많은 사용자 영향결제 실패, 검색 불가, 특정 리전 장애15분 내 ACK30분 내 미해결 시 TL
P3 Minor부분 기능 영향, 우회 가능특정 API 느림, 일부 UI 깨짐, 비핵심 기능 오류비즈니스 시간 내다음 비즈니스일
P4 Low사용자 영향 미미로그 오류, 미사용 엔드포인트 장애, 경미한 UI 이슈정규 백로그스프린트 내
💡 심각도 판단 가이드
"사용자가 핵심 기능을 사용할 수 있는가?"가 P1/P2 구분의 핵심.
"우회 방법이 있는가?"가 P2/P3 구분의 핵심.
의심스러우면 높은 쪽으로 분류하고, 상황 파악 후 하향 조정하는 것이 안전하다.
Roles

인시던트 역할 분리

지휘, 실행, 소통을 분리하여 각자 맡은 일에 집중할 수 있는 구조
장애 상황에서 한 사람이 모든 것을 하면 병목이 되고 실수가 늘어난다. 역할을 분리하면 각자의 영역에 집중할 수 있다. 소규모 팀에서는 한 사람이 여러 역할을 겸하되, 어떤 역할을 맡고 있는지 명시적으로 선언하는 것이 중요하다.
핵심 역할
Incident Commander (IC): 전체 지휘. 의사결정, 우선순위 결정, 역할 할당. 직접 디버깅하지 않음
Ops Lead: 기술적 대응 실행. 진단, 완화, 롤백 등 실제 작업 수행
Communication Lead: 내부/외부 커뮤니케이션. 상태 업데이트, 이해관계자 알림
Scribe: 타임라인 기록. 모든 행동과 결정을 실시간 기록 (Postmortem의 기초)

소규모 팀: IC + Ops를 겸하되, "나는 지금 IC로서 결정한다" / "지금 Ops로서 작업한다"를 구분하여 말하면 의사결정 품질이 올라감
IC의 핵심 행동
선언: "내가 이 인시던트의 IC입니다"
크기 평가: 심각도 결정, 추가 인력 필요 여부 판단
정보 수집: "현재 상태가 뭔가?", "영향 범위는?", "마지막 변경은?"
지시: "A는 DB 확인, B는 최근 배포 롤백 시도"
주기적 업데이트: 10~15분마다 상태 요약
종료 선언: "서비스 복구 확인됨. 인시던트 종료. Postmortem은 48시간 내 작성"
Flow

인시던트 대응 흐름

알림 수신부터 복구 확인까지의 단계별 프로세스
▼ 인시던트 대응 파이프라인
Detect알림 수신
Acknowledge5~15분
Triage심각도 분류
Mobilize역할 할당
Diagnose원인 파악
Mitigate완화/복구
Resolve복구 확인

Detect → Acknowledge

알림 수신 → ACK 버튼 → 담당자가 보고 있음을 팀에 알림.
"내가 보고 있다"를 선언하는 것만으로도 중복 대응을 방지.

Triage → Mobilize

영향 범위 파악 → 심각도 결정 → 필요 시 추가 인력 호출.
P1이면 즉시 War Room(Slack 채널/화상회의) 개설.

Diagnose

최근 변경 확인(배포, 설정 변경) → 대시보드 확인 → 로그/트레이스 분석.
"마지막으로 무엇이 바뀌었는가?"가 80%의 장애 원인.

Mitigate → Resolve

완화 우선, 근본 원인 분석은 나중. 롤백, 트래픽 분산, 기능 비활성화 등으로 먼저 복구. 근본 원인은 Postmortem에서 분석.
복구 후: SLI 정상 확인 → 알림 해제 → 종료 선언.

⚠ 핵심 원칙: Mitigate First, RCA Later
장애 중에 "왜 이런 일이 일어났나?"를 파고들면 복구가 늦어진다. 먼저 서비스를 복구하고(롤백, 스케일 아웃, 기능 꺼기), 나중에 Postmortem에서 근본 원인을 분석한다. "지금 고치는 것"과 "다음에 방지하는 것"을 분리.
Communication

인시던트 커뮤니케이션

내부/외부 이해관계자에게 적시에 정확한 정보를 전달하는 체계
내부 커뮤니케이션
War Room: P1/P2 전용 Slack 채널 즉시 생성. #incident-YYYYMMDD-서비스명
업데이트 주기: P1은 10분, P2는 30분마다 상태 업데이트

상태 업데이트 템플릿:
• 현재 상태: 조사 중 / 완화 중 / 복구됨
• 영향: 어떤 서비스, 얼마나 많은 사용자
• 현재 조치: 무엇을 하고 있는가
• 다음 업데이트: 언제 다시 알려줄 것인가
외부 커뮤니케이션
상태 페이지: 사용자에게 공개하는 서비스 상태
• Investigating: 문제를 인지하고 조사 중
• Identified: 원인 파악됨, 해결 중
• Monitoring: 수정 적용됨, 안정화 관찰 중
• Resolved: 문제 해결됨

원칙: 기술 용어 없이 사용자가 이해할 수 있는 언어로. "etcd leader election이 실패하여..." ❌ → "서비스 접속에 일시적 장애가 발생했습니다" ✅

장애 회고 (Postmortem)

Blameless 원칙, 작성 템플릿, 리뷰 프로세스, 5 Whys — 장애에서 배우는 체계적 방법론.

Culture

Blameless Postmortem 원칙

"누가 실수했나"가 아니라 "시스템과 프로세스에 어떤 약점이 있었나"를 찾는 문화
Blameless는 "아무도 잘못하지 않았다"가 아니다. 사람은 실수한다는 것을 전제로, 그 실수가 장애로 이어진 시스템적 약점을 찾는 것이다. 비난이 있으면 사람들은 숨기고, 숨기면 같은 장애가 반복된다.
Blameless 핵심 규칙
1) 사람이 아닌 시스템/프로세스/도구를 개선 대상으로
2) "왜 그랬어?"가 아닌 "어떤 정보가 부족했나?"
3) 관련자가 솔직하게 말할 수 있는 안전한 환경
4) Postmortem 결과가 인사 평가와 분리
5) 숨기면 더 큰 사고로 이어진다는 공감대

"잘못"이 아니라 "기여 요인(contributing factor)"이라는 단어를 사용
Postmortem 작성 기준
반드시 작성:
• 사용자 영향이 있는 모든 P1/P2 인시던트
• SLO Error Budget을 1% 이상 소진한 장애
• 데이터 손실 또는 보안 사고

선택적 작성:
• P3이지만 학습 가치가 높은 근접 실패(near-miss)
• 같은 유형이 반복되는 장애

작성 기한: 인시던트 종료 후 48~72시간 내 초안, 1주 내 리뷰 완료

DB 측 원인 조사 14:50 - 특정 배치 쿼리가 table lock을 잡고 있음 발견 14:55 - 해당 쿼리 강제 kill 시도 15:00 - kill 실패 (lock 재획득). DB 연결 풀 강제 리셋 15:10 - 커넥션 풀 복구 시작, 점진적으로 요청 처리 15:30 - 에러율 1% 이하로 안정화 15:45 - SLI 정상 확인, 인시던트 종료 선언 4. Root Cause (근본 원인) 주간 배치 집계 쿼리가 인덱스 없이 full table scan을 실행. 대상 테이블에 이번 주 데이터 급증(평소 2배)으로 실행 시간이 평소 5분에서 45분으로 증가. 이 동안 table lock이 유지되어 다른 쿼리들의 커넥션이 대기 상태로 쌓이면서 풀 소진. 5. Contributing Factors (기여 요인) - 배치 쿼리에 인덱스 미적용 - 커넥션 풀 크기가 20으로 작았음 (기본값 그대로 사용) - 배치 쿼리가 서비스 DB와 같은 인스턴스에서 실행 - 커넥션 풀 사용률 모니터링/알림 없었음 6. What Went Well - Burn Rate 알림이 5분 내에 발동 - 온콜 엔지니어가 2분 내에 ACK - DB 모니터링 대시보드에서 원인 빠르게 특정 7. What Went Wrong - 커넥션 풀 사용률 알림이 없었음 (감지 지연) - 배치 쿼리 kill이 즉시 효과 없었음 (Runbook에 DB force kill 절차 없음) - 배치 쿼리가 서비스에 영향을 줄 수 있다는 인식 부족 8. Action Items [P1] 배치 쿼리 인덱스 추가 — 담당: DBA — 기한: 3/22 [P1] 배치 쿼리를 Read Replica로 분리 — 담당: Backend — 기한: 3/29 [P2] 커넥션 풀 크기를 50으로 증가 — 담당: DevOps — 기한: 3/22 [P2] 커넥션 풀 사용률 80% 알림 추가 — 담당: SRE — 기한: 3/25 [P3] Runbook에 DB lock 강제 해제 절차 추가 — 담당: SRE — 기한: 3/29
Process

Postmortem 리뷰 프로세스

작성 → 리뷰 → Action Item 추적 → 공유 — Postmortem을 살아있는 문서로 만드는 프로세스
리뷰 미팅 구조
참석자: 장애 관련자 + 학습 목적 참관자
시간: 30~60분

아젠다:
1. Timeline 함께 읽기 (5분)
2. Root Cause 검증: "이것이 진짜 근본 원인인가?" (10분)
3. Action Items 검토: 우선순위, 담당자, 기한 합의 (15분)
4. 다른 서비스에도 같은 위험이 있는가? (5분)
5. Blameless 환경이 유지되었는가? (5분)

결과: 승인된 Postmortem → 공유 채널에 게시 + Action Item을 이슈 트래커에 등록
Action Item 추적
이슈 트래커 연동: 모든 Action Item은 Jira/Linear 이슈로 생성
주간 추적: SRE 주간 미팅에서 미완료 Action Item 리뷰
완료 기준: "구현됨"이 아니라 "효과 확인됨"이 완료

P1 Action Item: 1주 내 완료
P2 Action Item: 스프린트 내 완료
P3 Action Item: 분기 내 완료

미완료 알림: 기한 초과 Action Item은 주간 리뷰에서 에스컬레이션
RCA

5 Whys & Root Cause Analysis

"왜?"를 반복하여 표면적 원인이 아닌 근본 원인을 찾는 기법
5 Whys는 단순하지만 강력하다. 핵심은 첫 번째 "왜?"에서 멈추지 않는 것. 대부분의 장애는 기술적 원인(직접 원인)과 프로세스/문화적 원인(근본 원인)이 겹쳐 있다.
5 Whys 예시
1. 왜 서비스가 다운됐나?
→ DB 커넥션 풀이 소진됨
2. 왜 커넥션 풀이 소진됐나?
→ 배치 쿼리가 45분간 커넥션을 점유
3. 왜 배치 쿼리가 45분 걸렸나?
→ 인덱스 없이 full table scan을 실행
4. 왜 인덱스가 없었나?
→ 배치 쿼리 추가 시 DB 리뷰 프로세스가 없었음
5. 왜 DB 리뷰 프로세스가 없었나?
→ 소규모 팀에서 개발과 DB 운영이 분리되지 않았고, 쿼리 성능 체크리스트가 없었음

근본 원인: DB 변경에 대한 리뷰 프로세스 부재
Action: 배치 쿼리 추가 시 EXPLAIN 필수 체크리스트 도입
5 Whys 주의사항
1) 하나의 답에 고정되지 않기: 각 "왜?"에 여러 답이 있을 수 있다. 트리 구조로 분기
2) "사람이 실수해서"에서 멈추지 않기: "왜 그 실수가 장애로 이어졌나?"를 계속 추적
3) 정확히 5번이 아님: 3번이면 충분할 수도, 7번이 필요할 수도. "더 이상 유용한 답이 안 나올 때"까지
4) Counterfactual 피하기: "만약 X가 있었다면 방지됐을 텐데"는 유용하지 않음. "X가 왜 없었나?"를 추적

Runbook 설계와 자동화

새벽 3시에 반만 깨어서도 따라할 수 있는 절차서 — 설계, 템플릿, 자동화 단계.

Design

Runbook 설계 원칙

모호함 없이, 복사-붙여넣기로 실행 가능한 수준의 운영 절차서
좋은 Runbook은 해당 시스템을 모르는 엔지니어도 따라할 수 있어야 한다. 온콜 담당자가 새벽에 깨어났을 때, 평소 다루지 않는 시스템에 대해서도 Runbook만 따라가면 복구할 수 있어야 한다.
좋은 Runbook의 조건
1) 결정 트리(Decision Tree): "이 값이 X이면 A를 하고, Y이면 B를 해라"
2) 복사 가능한 명령어: kubectl get pods -n production -l app=api 전체 명령어 제공
3) 예상 결과: 각 단계 후 무엇이 보여야 정상인지 명시
4) 롤백 절차: 각 단계의 역방향 동작이 무엇인지
5) 에스컬레이션 기준: 이 Runbook으로 해결 안 되면 누구를 호출하는지
6) 마지막 검증일: 오래된 Runbook은 위험. 분기마다 리뷰
Runbook을 작성해야 하는 알림
모든 Page/Ticket 알림에는 Runbook 링크가 있어야 한다.

Grafana Alerting에서 알림 메시지에 runbook_url 삽입:
annotations.runbook_url: "https://wiki/runbooks/api-high-latency"

알림이 울렸을 때 엔지니어가 첫 번째로 하는 행동이 "Runbook 클릭"이어야 한다. Runbook 없는 알림은 "울리기만 하고 어떻게 대응할지 모르는" 나쁜 알림.

Evolution

Runbook 자동화 단계

수동 Runbook → 반자동 → 완전 자동화로 진화시키는 단계별 접근
같은 Runbook을 3번 이상 실행하면 자동화 대상이다. 자동화는 한 번에 완전 자동화가 아니라, 단계적으로 진화시키는 것이 안전하다.
자동화 성숙도 단계
Level 0 — 수동: 문서에 절차만 기록. 사람이 직접 실행
Level 1 — 스크립트화: 각 단계를 셸 스크립트/Go CLI로 만들어 ./runbook-api-latency.sh diagnose 실행
Level 2 — 반자동: 알림 발생 시 자동으로 진단 결과를 Slack에 게시. 사람이 판단 후 복구 명령 실행
Level 3 — 자동 완화: 알림 발생 → 자동으로 스케일 아웃 또는 롤백 실행. 사람에게 결과만 통보
Level 4 — 자가 치유: 시스템이 스스로 이상을 감지하고 복구. HPA, PDB, liveness probe, ArgoCD auto-rollback이 이 수준

HolmesGPT: Level 2를 빠르게 달성하기 위한 도구. 알림 → 자동 컨텍스트 수집 → LLM이 원인 분석 → Slack 게시. 사람은 분석 결과를 보고 행동 결정

인시던트 메트릭과 장애 문화

MTTx 분해, 트렌드 분석, 건강한 장애 문화 — 인시던트에서 지속적으로 학습하는 체계.

Measurement

MTTx 분해 — 각 구간을 따로 측정해야 개선할 수 있다

MTTR을 한 덩어리로 보지 말고, MTTD + MTTE + MTTM + MTTV로 분해
이 문서에서는 MTTR을 장애 발생부터 복구 확인까지의 전체 시간으로 본다. MTTR이 2시간이라는 것만으로는 어디를 개선해야 하는지 알 수 없다. 감지가 느린 건지, 담당자 호출이 늦는 건지, 수정이 느린 건지, 검증이 느린 건지를 분해해야 실질적 개선이 가능하다. 약어 이름은 조직마다 조금씩 다를 수 있지만, 핵심은 각 구간을 따로 측정하는 것이다.
MTTR 분해 모델
MTTR = MTTD + MTTE + MTTM + MTTV MTTD (Detect) : 장애 발생 → 감지까지. 개선: 모니터링, Burn Rate 알림 MTTE (Engage) : 감지 → 담당자 대응 시작까지. 개선: 온콜 체계, 에스컬레이션 MTTM (Mitigate) : 대응 시작 → 완화/수정 적용까지. 개선: Runbook, 자동 롤백 MTTV (Verify) : 수정 적용 → 복구 확인까지. 개선: SLI 대시보드, 자동 검증

가장 큰 구간을 먼저 공략. 대부분의 경우 MTTD가 가장 긴데, 이는 "문제가 있는데 아무도 모르는 시간"이 가장 위험하기 때문이다. Burn Rate 알림이 MTTD를 줄이는 가장 효과적인 방법.
Analysis

인시던트 트렌드 분석

개별 장애가 아닌 패턴을 보면 구조적 약점이 보인다
추적해야 할 트렌드
월별 인시던트 수: 증가 추세면 시스템 복잡도 대비 안정성 투자 부족
심각도 분포: P1이 줄고 P3이 늘면 건강. P1이 꾸준하면 근본 문제 미해결
MTTR 추이: 개선되고 있는가? 정체 구간이 어디인가?
반복 장애 비율: 같은 원인 재발 비율. Postmortem Action Item이 실행되지 않으면 상승
서비스별 장애 빈도: 특정 서비스에 장애가 집중되면 해당 서비스 안정성 투자
분석 방법
인시던트 DB: 모든 인시던트를 스프레드시트 또는 전용 도구에 기록
최소 필드: 날짜, 심각도, 서비스, MTTD, MTTR, 원인 카테고리, Action Item 수

분기 리뷰: 인시던트 트렌드를 경영진에게 보고
• 이번 분기 인시던트 수 vs 지난 분기
• Top 3 원인 카테고리
• MTTR 개선 추이
• Error Budget 소진 상황
• 완료된 / 미완료 Action Item 비율
Culture

건강한 장애 문화 만들기

장애를 숨기는 문화 vs 장애에서 배우는 문화 — 선택이 아닌 생존 전략
건강한 문화의 신호
✅ 장애가 발생하면 자발적으로 보고
✅ Postmortem에서 솔직한 논의가 가능
✅ Near-miss(근접 실패)도 공유
✅ Action Item이 실제로 완료됨
✅ 온콜이 "당연한 업무"로 인정받고 보상됨
✅ 장애 후 개인을 비난하지 않음
✅ "이 알림은 왜 울렸나?"를 주기적으로 리뷰
건강하지 않은 문화의 신호
❌ 장애를 숨기거나 축소 보고
❌ "누구 때문이냐"를 찾는 분위기
❌ Postmortem이 형식적이거나 아예 안 씀
❌ Action Item이 영원히 미완료
❌ 같은 장애가 반복되는데 아무도 놀라지 않음
❌ 온콜이 벌칙처럼 취급됨
❌ 알림이 수백 개인데 "원래 그래"
문화를 만드는 실천법
1) Postmortem 공유 세션: 월 1회 전체 팀이 모여 주요 Postmortem 공유. 학습의 기회
2) Failure Friday: 금요일에 의도적으로 작은 장애를 일으키고 대응 연습 (Game Day 축소판)
3) Hero 방지: 한 사람이 모든 장애를 해결하는 구조 → 팀 전체의 역량 약화. 지식 공유 필수
4) 온콜 리뷰: 매주 "이번 주 온콜에서 가장 고통스러웠던 것"을 공유 → 시스템 개선 백로그화
5) Reliability Week: 분기 1회, 1주일간 feature 개발 멈추고 reliability 개선에만 집중하는 스프린트
