● Pipeline Design

파이프라인 설계 원칙

Stage 분해, DAG 구조, Fan-in/out, Matrix, Pipeline-as-Code — 빠르고 안전한 피드백 루프를 만드는 구조 설계.

Foundation

Stage 분해 원칙

commit → build → test → package → deploy — 각 단계의 책임과 종료 조건이 분명해야 한다
파이프라인은 "실패가 빠를수록 좋다(fail fast)"는 원칙으로 설계한다. 비싼 검증(E2E, 통합 테스트, 보안 스캔)은 싼 검증(lint, unit test) 뒤로 미룬다. 각 stage의 종료 조건(exit criteria)이 명확하지 않으면 "어디서 실패했는지" 추적이 어려워진다.
표준 Stage 구성
1. Validate (5초) — lint, format, commit-msg 2. Build (30초) — compile, type-check 3. Unit Test (1분) — 단위 테스트 + 커버리지 4. Package (1분) — Docker 이미지, 아티팩트 5. Integration (3분) — 컴포넌트 통합 테스트 6. E2E (5분) — 전체 시나리오 (PR에서는 선택) 7. Deploy (2분) — staging/canary 배포 8. Smoke (30초) — 배포 후 health check
핵심: 앞 단계 실패 시 뒤 단계 실행 안 함. 90% 이상의 실패는 앞 4단계에서 잡혀야 한다
Stage 종료 조건
각 stage는 "이 단계를 통과한다"가 무엇을 의미하는가를 명시:

build: 컴파일 성공 + 산출물 검증
unit-test: 모든 테스트 통과 + 커버리지 ≥ 80%
package: 이미지 빌드 + 서명 + SBOM 생성
integration: 외부 의존성과 연동 검증

안티패턴: "테스트가 일부 실패해도 일단 진행" — 한 번 허용하면 항상 깨진 상태로 굳는다
실행 모델: PR vs Merge vs Release
PR 파이프라인 (5분 이내): validate + build + unit + integration. E2E는 선택적
Merge 파이프라인 (10분 이내): 위 + E2E + 보안 스캔 + staging 배포 + smoke
Release 파이프라인 (수동 승인): 위 + canary → 점진적 prod 롤아웃 + 모니터링 대기

같은 파이프라인을 모든 트리거에 쓰지 말 것. 피드백 속도와 검증 깊이의 트레이드오프를 트리거별로 다르게 설정해야 개발자가 PR에서 5분 내에 결과를 받을 수 있다.
Topology

DAG vs Sequential — 의존성으로 모델링하라

stage → stage 직렬 실행은 단순하지만 느리다. 의존 그래프(DAG)는 병렬도를 극대화한다
전통적인 CI는 stage가 직렬로 흐른다(Jenkins의 기본 모델). 하지만 대부분의 stage는 서로 독립적이다 — lint와 type-check는 동시에 돌아도 된다. DAG로 표현하면 wall clock time이 줄어든다.

Sequential (전통)

build → test → deploy 순서로 직렬. 단순하지만 wall clock = sum(stages). lint(5s) + unit(60s) + e2e(180s) = 245s.

적합: 단순한 프로젝트, 의존 관계가 직선적인 경우

DAG (의존 그래프)

각 job이 needs:로 선행 의존만 명시. 독립 job은 자동 병렬. build → [unit, lint, sec-scan] → e2e → deploy: max path = ~110s.

적합: 복잡한 빌드, 다수의 독립 검증이 있는 경우

실전 패턴: 의존성 그래프 그리기

"이 job은 무엇이 끝나야 시작할 수 있나?" 만 답하면 된다. e2e는 package가 끝나야 시작 — unit 결과는 필요 없음. deploy는 e2e와 sec-scan 둘 다 통과해야 — fan-in.

GitHub Actions DAG 예시
jobs: build: runs-on: ubuntu-latest steps: [checkout, setup, build] unit: needs: build steps: [test --unit] lint: needs: build # build와 병렬 시작 가능 (build의 산출물만 필요) steps: [eslint] sec-scan: needs: build steps: [trivy, semgrep] e2e: needs: [unit, lint] # fan-in: 둘 다 통과해야 시작 steps: [cypress] deploy: needs: [e2e, sec-scan] if: github.ref == 'refs/heads/main' steps: [argocd sync]
WARN DAG의 함정: 너무 잘게 쪼개면 job 시작 오버헤드(러너 spin-up)가 누적되어 오히려 느려진다. 각 job은 최소 30초 이상의 작업을 가지도록 묶어라. GitHub Actions는 job당 평균 ~10초 startup 비용이 있다.
Pattern

Fan-out / Fan-in 패턴

하나의 빌드 결과를 N개의 검증으로 분기(fan-out)하고, 모두 통과 시 다음 단계로 합치(fan-in)는 구조
Fan-out은 병렬화로 벽시계 시간을 줄인다. 단위 테스트를 4개 shard로 나누면 4배 빨라진다(이상적). Fan-in은 합류 지점에서 "모든 분기가 성공"을 강제한다 — 한 분기라도 실패하면 다음 단계 진입을 막는다.
Fan-out: Test Sharding
jobs: test: strategy: matrix: shard: [1, 2, 3, 4] steps: - run: pytest --shard=${{ matrix.shard }}/4
4개 러너에 테스트를 분산. 주의: 테스트가 독립적이어야 함. 공유 DB/파일을 쓰는 테스트는 sharding 시 race 발생
Fan-in: 합류 게이트
deploy: needs: [test, lint, sec-scan, e2e] if: ${{ !contains(needs.*.result, 'failure') }} steps: - run: ./deploy.sh
4개 분기가 모두 성공해야 deploy 진입. needs.*.result로 모든 분기 상태 확인. 한 분기가 skip되면? contains(needs.*.result, 'skipped')로 따로 처리
실전: 다이아몬드 패턴
build / | \ unit lint sec \ | / e2e | deploy
장점: 의존성 명시 + 자동 병렬화 + 명시적 합류점
주의사항:
1. Fan-out 분기에 공유 자원(DB, port) 쓰지 말 것 — 각 분기가 isolated 환경 가져야 함
2. Fan-in 시 부분 성공 허용 여부 명확히 — 보안 스캔은 warning이어도 통과? 정책으로 명시
3. 분기 수 ≥ 8개면 비용 검토 — 러너 비용은 분기 수에 비례한다
Strategy

Matrix 빌드 — 다차원 검증

{OS, 언어 버전, DB 버전} 등 여러 차원을 곱해 모든 조합을 검증한다
라이브러리/CLI/오픈소스 도구는 여러 환경에서의 호환성이 핵심 가치다. Matrix 빌드는 cartesian product로 모든 조합을 자동 생성한다. 단, 조합 수가 폭발하므로 include/exclude로 핵심 조합만 골라야 한다.
Matrix with include/exclude
strategy: fail-fast: false # 한 조합 실패해도 다른 조합은 계속 matrix: os: [ubuntu-latest, macos-latest, windows-latest] node: ['18', '20', '22'] include: # 추가 조합 (메인 매트릭스 외) - os: ubuntu-latest node: '22' experimental: true # 실험 플래그 추가 exclude: # 제외할 조합 (지원 X) - os: windows-latest node: '18' # node 18 + windows는 빌드 깨짐
조합 수 관리
3 OS × 3 Node × 2 DB = 18 jobs. 각 job 5분이면 90 머신·분.

전략: PR에서는 핵심 조합 3-4개만 (Linux + 최신 Node + Postgres). nightly에서 전체 매트릭스 실행. fail-fast: false는 매트릭스 빌드의 기본값이어야 — 한 조합만 깨졌는데 전체 멈추면 정보 손실
Dynamic Matrix
변경된 패키지만 테스트하는 동적 매트릭스:
jobs: changes: outputs: pkgs: ${{ steps.f.outputs.pkgs }} steps: - id: f run: | PKGS=$(git diff --name-only HEAD^ \ | grep -E '^pkg/' \ | cut -d/ -f2 | sort -u \ | jq -R . | jq -sc .) echo "pkgs=$PKGS" >> $GITHUB_OUTPUT test: needs: changes strategy: matrix: pkg: ${{ fromJson(needs.changes.outputs.pkgs) }}
모노레포에서 변경된 패키지만 테스트해 CI 시간 단축
Principle

Pipeline-as-Code 원칙

파이프라인 정의도 코드다. 코드 리뷰, 버저닝, 테스트의 대상이 되어야 한다
UI에서 클릭으로 만든 Jenkins job, 외부에 저장된 CircleCI 설정 — 이런 구성은 "누가 언제 무엇을 바꿨는지" 추적이 안 된다. 파이프라인 정의(.github/workflows/*.yml, Jenkinsfile, .gitlab-ci.yml)는 반드시 애플리케이션 코드와 같은 저장소에서 같은 PR로 변경되어야 한다.
5가지 원칙
1. 코드와 같은 저장소 — 파이프라인 변경이 애플리케이션 변경과 함께 PR로 검토
2. 버저닝 — 모든 변경은 git history로 추적 가능
3. 테스트 가능 — act(GHA), gitlab-runner exec로 로컬 검증
4. 모듈화 — 반복되는 패턴은 reusable workflow / shared library
5. 리뷰 필수 — .github/workflows/*.yml 변경은 CODEOWNERS로 보안팀 리뷰 강제
위험 신호 — 안티패턴
UI 클릭 설정: Jenkins Freestyle, GitHub Actions UI에서 환경변수 직접 입력
외부 스크립트 의존: curl https://example.com/script.sh | bash — 변경 추적 불가
Inline secret: 파이프라인 YAML에 토큰 평문 삽입
여러 저장소에 분산된 정의: 한 서비스의 빌드 로직이 3곳에 흩어져 있음

해법: 모든 파이프라인 정의는 git 저장소 안에. 외부 스크립트는 git submodule이나 vendor된 형태로
파이프라인 자체의 테스트
파이프라인 yaml에 버그가 있으면 main branch가 빨갛게 된다. 다음으로 보호:
1. Schema validation — gh actions-validator, gitlab-ci-lint로 PR에서 syntax 검증
2. Dry-run — act -n으로 로컬 dry-run
3. Branch protection — .github/workflows/ 디렉토리는 별도 reviewer 필수 (CODEOWNERS)
4. Canary on feature branch — 파이프라인 변경은 feature branch에서 충분히 돌려보고 main 머지

"파이프라인 yaml 한 줄 고치고 머지" 같은 PR도 일반 코드와 동일한 검토 절차 적용
● GitHub Actions Deep Dive

GitHub Actions 심화 패턴

Reusable Workflow, Composite Action, Matrix 고급, OIDC 클라우드 인증, Self-hosted Runner — 대규모 조직에서 GHA를 안전·효율적으로 운영.

Reuse

Reusable Workflow vs Composite Action

코드 중복을 줄이는 두 가지 메커니즘 — 차이를 알아야 올바르게 선택할 수 있다
한 조직에서 수십 개 저장소가 같은 빌드/테스트/배포 패턴을 쓴다면, 매번 yaml을 복제할 수 없다. GHA는 두 가지 재사용 메커니즘을 제공한다.
특징Reusable WorkflowComposite Action
위치.github/workflows/*.ymlaction.yml in repo or path
호출 방식uses: org/repo/.github/workflows/x.yml@v1uses: org/repo/path@v1
실행 단위전체 job (러너 분리)step들의 묶음 (호출자 러너 공유)
입력inputs / secrets (named)inputs (named)
출력outputs (job-level)outputs (step-level)
제약최대 4단계 중첩shell command 위주
적합한 상황"빌드 → 테스트 → 배포" 같은 큰 흐름"setup + cache 복원" 같은 작은 step 묶음
Reusable Workflow 호출
# caller workflow jobs: deploy: uses: myorg/ci-templates/.github/workflows/deploy.yml@v2 with: env: production image: ghcr.io/myorg/app:${{ github.sha }} secrets: aws-role: ${{ secrets.AWS_PROD_ROLE }} # reusable workflow 정의 (myorg/ci-templates) on: workflow_call: inputs: env: { type: string, required: true } image: { type: string, required: true } secrets: aws-role: { required: true } outputs: url: value: ${{ jobs.deploy.outputs.url }}
TIP 버전 핀 @v2 또는 SHA @abc123...로 고정. @main은 절대 쓰지 말 것 — 외부 변경이 silent하게 파이프라인을 깨뜨린다. SHA pin은 보안적으로 가장 안전 (third-party action에 필수).
Action

Composite Action — 작은 step 묶음

setup + cache 복원 + 환경변수 설정 등 매번 반복되는 step 4-5개를 하나로 묶는 단위
Reusable Workflow가 "큰 흐름"이라면, Composite Action은 "자주 쓰는 step 묶음"이다. actions/setup-node 같은 marketplace action도 본질적으로 composite action이다.
.github/actions/setup-app/action.yml
name: 'Setup app environment' description: 'Node setup + pnpm cache + env restore' inputs: node-version: required: true default: '20' outputs: cache-hit: value: ${{ steps.cache.outputs.cache-hit }} runs: using: 'composite' steps: - uses: actions/setup-node@v4 with: node-version: ${{ inputs.node-version }} - id: cache uses: actions/cache@v4 with: path: ~/.pnpm-store key: pnpm-${{ hashFiles('pnpm-lock.yaml') }} - shell: bash run: corepack enable && pnpm install --frozen-lockfile
NOTE Composite action은 shell: 명시 필수 (Reusable Workflow와 다름). 호출자의 러너에서 실행되므로 별도 OS 지정 불가 — OS-specific 로직은 if: runner.os == 'Linux'로 분기.
Advanced

Matrix Strategy 고급

동적 매트릭스, fail-fast 제어, max-parallel, continue-on-error 조합
fail-fast 제어
기본값 true: 한 조합 실패 시 다른 조합 즉시 중단 — 비용 절약, 정보 손실
권장 false: 모든 조합 끝까지 실행 — 어떤 환경 조합에서 실패하는지 패턴 파악 가능

strategy: fail-fast: false max-parallel: 4 # 동시 실행 상한 matrix: { ... }
max-parallel: 러너 비용 통제. 24개 조합을 한 번에 돌리면 러너 부족할 수 있음
continue-on-error per matrix
"실험적" 조합은 실패해도 전체 실패로 안 치고 싶을 때:
strategy: matrix: node: ['20', '22'] include: - node: 'nightly' experimental: true jobs: test: continue-on-error: ${{ matrix.experimental == true }} steps: [...]
nightly Node에서 실패해도 PR 머지 가능. 알림은 별도로 받음
동적 매트릭스 — 변경 기반 sharding
전체 테스트를 항상 다 돌리지 말고, 변경된 영역만:
# Stage 1: 변경 감지 - id: changes uses: dorny/paths-filter@v3 with: filters: | backend: 'services/backend/**' frontend: 'apps/web/**' docs: 'docs/**' # Stage 2: 동적 매트릭스 생성 - id: matrix run: | SHARDS='[]' [ "${{ steps.changes.outputs.backend }}" = "true" ] && SHARDS=$(echo "$SHARDS" | jq '. + ["backend"]') [ "${{ steps.changes.outputs.frontend }}" = "true" ] && SHARDS=$(echo "$SHARDS" | jq '. + ["frontend"]') echo "matrix=$SHARDS" >> $GITHUB_OUTPUT # Stage 3: 매트릭스 실행 test: strategy: matrix: area: ${{ fromJson(needs.matrix.outputs.matrix) }}
모노레포에서 PR당 평균 CI 시간 60% 단축 사례 흔함
Security

OIDC — 클라우드 인증의 정도(正道)

정적 액세스 키를 secret에 박지 말 것. OIDC로 단기 토큰을 받아 써라
전통적 방식: AWS access key를 GitHub secret에 저장 → 매번 그 키로 인증. 문제는 (1) 키 유출 시 영구 권한 노출 (2) 키 회전 자동화 어려움 (3) 어떤 워크플로우가 무엇을 했는지 추적 어려움. OIDC는 이 모두를 해결한다.
OIDC 동작 흐름
1. GHA workflow 시작 → GitHub OIDC provider가 JWT 발급 2. JWT에 메타데이터 포함: - repo: org/repo - ref: refs/heads/main - actor: user-name - workflow: deploy.yml 3. AWS STS에 JWT 제시 → AssumeRoleWithWebIdentity 4. AWS는 IAM Role의 trust policy로 검증: - issuer: token.actions.githubusercontent.com - sub: repo:org/repo:ref:refs/heads/main 5. 검증 통과 → 단기 자격증명 발급 (1시간)
AWS 측 IAM Trust Policy
{ "Version": "2012-10-17", "Statement": [{ "Effect": "Allow", "Principal": { "Federated": "arn:aws:iam::ACCT:oidc-provider/token.actions.githubusercontent.com" }, "Action": "sts:AssumeRoleWithWebIdentity", "Condition": { "StringEquals": { "token.actions.githubusercontent.com:aud": "sts.amazonaws.com" }, "StringLike": { "token.actions.githubusercontent.com:sub": "repo:myorg/myrepo:ref:refs/heads/main" } } }] }
핵심: sub 조건으로 어떤 repo의 어떤 branch만 이 Role을 쓸 수 있는지 제한. PR에서는 다른 read-only Role을 쓰도록 분리
GHA workflow 측
permissions: id-token: write # OIDC 토큰 요청 권한 (필수) contents: read jobs: deploy: runs-on: ubuntu-latest steps: - uses: aws-actions/configure-aws-credentials@v4 with: role-to-assume: arn:aws:iam::ACCT:role/gha-deploy-prod aws-region: ap-northeast-2 - run: aws s3 sync ./dist s3://my-bucket/
주의: id-token: write 권한이 없으면 OIDC 토큰을 못 받음. 워크플로우 또는 job 레벨에서 명시 필요. 이걸로 정적 access key는 영원히 추방.
Runner

Self-hosted Runner — 보안과 성능의 트레이드오프

GitHub-hosted vs self-hosted — 언제 어떻게 쓸지의 기준

GitHub-hosted

매번 깨끗한 VM에서 실행. 장점: 격리, 관리 불필요. 단점: 큰 빌드/큰 이미지 캐시 활용 어려움, 비용(분당 과금), 네트워크 격리 환경 접근 불가.

Self-hosted

직접 운영하는 머신/k8s에서 실행. 장점: 캐시 재사용, 사내망 접근, 큰 사양 가능, 비용 통제. 단점: 보안 책임, 격리 깨질 위험, 운영 부담.

안전한 self-hosted 운영 5원칙

1. Public repo에 절대 사용 금지 — fork된 PR이 임의 코드 실행 가능
2. Ephemeral runner — 매 job마다 새 컨테이너/VM (ARC, runs-on cloud)
3. 네트워크 분리 — runner는 production 자원에 직접 접근 불가, 별도 deploy account 거침
4. Label로 격리 — runs-on: [self-hosted, prod-deploy]로 워크로드별 분리
5. 감사 로그 — runner의 모든 실행 stdout/stderr 보관 (90일)

ARC (Actions Runner Controller) — k8s 기반 ephemeral runner
apiVersion: actions.github.com/v1alpha1 kind: AutoscalingRunnerSet metadata: name: prod-deploy-runners spec: runnerScaleSetName: prod-deploy githubConfigUrl: https://github.com/myorg minRunners: 1 maxRunners: 10 template: spec: containers: - name: runner image: ghcr.io/actions/actions-runner:2.320.0 resources: limits: { cpu: "4", memory: "8Gi" } serviceAccountName: gha-prod-deploy # IRSA로 AWS 권한

Jenkins

Jenkins Declarative Pipeline 핵심

Scripted vs Declarative — 신규 프로젝트는 Declarative만 쓸 것
Jenkins는 두 가지 파이프라인 문법을 지원한다. Scripted는 Groovy 코드(자유도 ↑, 검증 어려움), Declarative는 구조화된 DSL(검증 용이, 표준화). 모든 신규 파이프라인은 Declarative로 작성하고, Scripted 영역이 필요하면 script {} 블록 안에 격리한다.
표준 Jenkinsfile 구조
pipeline { agent { label 'docker-builder' } options { timeout(time: 30, unit: 'MINUTES') disableConcurrentBuilds() buildDiscarder(logRotator(numToKeepStr: '30')) } environment { REGISTRY = 'ghcr.io/myorg' VERSION = "${env.BUILD_NUMBER}-${env.GIT_COMMIT.take(7)}" } stages { stage('Build') { steps { sh 'make build' } } stage('Test') { parallel { stage('Unit') { steps { sh 'make test-unit' } } stage('Integration') { steps { sh 'make test-int' } } stage('Lint') { steps { sh 'make lint' } } } } stage('Deploy') { when { branch 'main' } steps { sh './deploy.sh' } } } post { always { junit '**/test-results.xml' } failure { slackSend channel: '#ci', color: 'danger', message: "Build failed: ${env.BUILD_URL}" } } }
WARN Jenkins의 함정 — plugins 의존성 지옥. 플러그인 버전 불일치로 빌드가 silent하게 깨진다. 해법: Jenkins Configuration as Code(JCasC) + 플러그인 버전 매트릭스 고정 + 정기 업그레이드 윈도우.
Pattern

Jenkins Shared Library

조직 전체에서 재사용하는 파이프라인 로직을 별도 git repo로 분리
수십 개 서비스가 모두 비슷한 빌드/배포 패턴을 쓴다면, Jenkinsfile 복붙 대신 Shared Library를 쓴다. GHA의 Reusable Workflow와 같은 역할이다.
Shared Library 구조
my-jenkins-lib/ ├── vars/ │ ├── deployToK8s.groovy # @Library 호출자가 deployToK8s() 사용 가능 │ └── buildDockerImage.groovy ├── src/ │ └── com/myorg/ │ └── DeployHelper.groovy # 클래스 정의 └── resources/ └── templates/ └── deploy.yaml.tpl
vars/deployToK8s.groovy
def call(Map config) { assert config.image && config.namespace sh """ helm upgrade --install ${config.app} ./chart \\ --namespace ${config.namespace} \\ --set image.repository=${config.image} \\ --wait --timeout 5m """ }
호출 측: @Library('my-jenkins-lib@v3') _ 후 deployToK8s(image: 'foo', namespace: 'prod')
버저닝 & 보안
버전 핀: @Library('lib@v3.2')로 명시 — @main은 금지
Trusted Library: 관리자가 Jenkins UI에서 등록한 라이브러리만 sandbox 우회 가능. 일반 사용자가 @Library로 임의 repo 불러오면 보안 검사
변경 리뷰: Shared Library repo도 PR + 리뷰 필수. 한 번 버그 머지되면 모든 파이프라인이 깨진다
GitLab

GitLab CI 핵심 — rules / needs / DAG

GitLab은 stages 기반 직렬 + needs 기반 DAG의 하이브리드 모델
GitLab CI는 기본이 stages 직렬 모델이지만, needs:로 DAG 구성도 가능하다. rules:는 GHA의 if:에 해당하며, 이 둘을 능숙하게 다루는 게 GitLab CI의 핵심.
.gitlab-ci.yml — 실전 패턴
stages: [build, test, deploy] workflow: # 파이프라인 자체 실행 조건 rules: - if: $CI_PIPELINE_SOURCE == "merge_request_event" - if: $CI_COMMIT_BRANCH == "main" - when: never # 그 외에는 파이프라인 안 돔 .template-test: # YAML anchor 대용 — extends로 재사용 stage: test image: node:20 cache: key: $CI_COMMIT_REF_SLUG paths: [.pnpm-store/] unit-test: extends: .template-test script: pnpm test --unit artifacts: reports: junit: junit.xml # MR에 자동 통합 coverage_report: coverage_format: cobertura path: coverage.xml e2e: stage: test needs: [build] # DAG: build만 끝나면 시작 (다른 test 안 기다림) script: pnpm test:e2e rules: - if: $CI_PIPELINE_SOURCE == "merge_request_event" changes: ['src/**/*', 'tests/**/*'] - if: $CI_COMMIT_BRANCH == "main" deploy-prod: stage: deploy environment: production rules: - if: $CI_COMMIT_TAG =~ /^v\d+\.\d+\.\d+$/ when: manual # 수동 승인 필요
NOTE GitLab의 rules:는 only/except의 후계자. 신규 프로젝트는 rules:만 쓸 것. changes:로 변경된 파일에 따라 job 실행 결정 — 모노레포에서 강력하다.

Docker

Docker 레이어 캐싱 — 가장 흔하고 가장 잘못 쓰이는 도구

Dockerfile 순서가 잘못되면 캐시는 무용지물이다
Docker는 각 RUN/COPY/ADD 라인을 레이어로 캐싱한다. 한 레이어가 변경되면 그 이후 모든 레이어 재실행이다. 따라서 "자주 변경되는 것"은 뒤로 가야 한다.

❌ 나쁜 순서

COPY . .를 먼저 하고 RUN npm install. 코드 한 줄만 바뀌어도 npm install이 재실행됨 — 매 빌드 3분 추가.

FROM node:20 WORKDIR /app COPY . . RUN npm install RUN npm run build

✅ 좋은 순서

의존성 파일만 먼저 복사 → install → 그 다음 코드 복사. 코드 변경 시 install 레이어 캐시 적중.

FROM node:20 WORKDIR /app COPY package*.json ./ RUN npm ci COPY . . RUN npm run build
Multi-stage build — 최종 이미지 슬림화
# Stage 1: build — 큰 도구 포함 FROM node:20 AS build WORKDIR /app COPY package*.json ./ RUN npm ci COPY . . RUN npm run build && npm prune --production # Stage 2: runtime — 슬림 베이스 + 산출물만 FROM node:20-slim WORKDIR /app COPY --from=build /app/dist ./dist COPY --from=build /app/node_modules ./node_modules USER node CMD ["node", "dist/server.js"]
TIP --mount=type=cache(BuildKit)로 npm/apt 캐시를 레이어 밖에서 공유 — 이미지 크기 ↓ + 빌드 속도 ↑. RUN --mount=type=cache,target=/root/.npm npm ci처럼 사용.
BuildKit

BuildKit — 차세대 Docker 빌더

병렬 stage 빌드 + cache mount + secret mount + 외부 캐시 백엔드
BuildKit은 Docker 18.09부터 도입된 차세대 빌더. 레거시 builder 대비 (1) 독립 stage 병렬 빌드, (2) cache mount(레이어 안 가는 캐시), (3) secret mount(빌드 시 secret 사용 후 이미지에 안 남음), (4) registry 기반 외부 캐시 등을 제공한다.
Cache Mount — 의존성 캐시
# syntax=docker/dockerfile:1.6 FROM golang:1.22 AS build WORKDIR /app COPY go.* ./ RUN --mount=type=cache,target=/root/.cache/go-build \ --mount=type=cache,target=/go/pkg/mod \ go mod download COPY . . RUN --mount=type=cache,target=/root/.cache/go-build \ --mount=type=cache,target=/go/pkg/mod \ go build -o /bin/app ./cmd/app
Go module과 build cache가 레이어 외부에 보존. 매 빌드마다 90% 캐시 히트
Secret Mount — 안전한 빌드
RUN --mount=type=secret,id=npm_token \ NPM_TOKEN=$(cat /run/secrets/npm_token) \ npm install --legacy-peer-deps
호출:
DOCKER_BUILDKIT=1 docker build \ --secret id=npm_token,src=$HOME/.npm-token .
secret이 이미지 레이어에 절대 남지 않음. ENV로 넘기면 이미지 history에 남아 유출 위험
Registry 기반 외부 캐시 — CI에서 핵심
CI 러너는 매번 새 머신이라 로컬 레이어 캐시가 없다. Registry에 캐시를 push/pull 하면 다른 러너에서도 활용 가능:
docker buildx build \ --cache-from=type=registry,ref=ghcr.io/myorg/app:cache \ --cache-to=type=registry,ref=ghcr.io/myorg/app:cache,mode=max \ --tag ghcr.io/myorg/app:${SHA} \ --push .
mode=max: 모든 stage의 캐시 push (mode=min은 final stage만)
효과: 코드 변경이 적으면 90초 빌드가 15초로 단축
Distributed

분산 캐시 — sccache, Bazel Remote Cache

컴파일러 산출물을 팀 전체가 공유 — 같은 코드는 누구도 두 번 컴파일하지 않는다
대규모 C++/Rust 프로젝트는 한 번 빌드에 30분 걸리는 경우도 흔하다. 같은 입력 → 같은 출력이라는 결정성을 활용해, 컴파일러 호출 단위로 결과를 공유한다.
sccache — Rust/C++ 컴파일러 캐시
# 환경변수로 sccache를 컴파일러 wrapper로 사용 export RUSTC_WRAPPER=sccache export SCCACHE_BUCKET=my-build-cache export SCCACHE_REGION=ap-northeast-2 cargo build --release # S3에 캐시 자동 저장/조회
cargo의 incremental compilation과 별개로, 팀 전체가 같은 .o 파일을 공유. CI에서도 매번 클린 빌드 시 90% 캐시 적중
Bazel Remote Cache
Bazel은 빌드 그래프의 결정성을 보장하므로 캐시 친화적:
# .bazelrc build --remote_cache=grpcs://cache.myorg.io build --remote_executor=grpcs://exec.myorg.io # remote build도 가능 build --remote_upload_local_results=true
모노레포 빌드가 1시간 → 5분으로 단축되는 케이스 흔함. 비결정성 제거가 선결: 시간 stamp, /tmp 경로, 환경변수 의존 모두 격리
언제 분산 캐시를 도입할까
도입 가치 ↑:
· 컴파일 시간 ≥ 5분
· 팀 규모 ≥ 5명 (캐시 적중 빈도)
· CI 빌드 ≥ 일 50회

도입 가치 ↓:
· 인터프리터 언어 (Python, JS) — 컴파일이 가벼움
· 빌드 시간 ≤ 1분
· 비결정적 빌드 (캐시 무효화 잦음)

주의: 캐시 자체의 운영 비용(S3, Redis, gRPC 서버)도 고려. 작은 팀에는 BuildKit registry cache로 충분
Smart

Test Impact Analysis — 변경된 코드만 테스트

테스트 ↔ 코드 의존 그래프를 분석해, 변경된 코드와 무관한 테스트는 스킵
5만 개 테스트가 있는 프로젝트에서 PR마다 전체를 돌리면 30분 이상 걸린다. 변경된 파일이 영향을 미치는 테스트만 골라 돌리면 평균 5%만 실행해도 충분하다 — 95% 시간 절약.
파일 의존 기반 (간단)
Jest: jest --findRelatedTests $(git diff --name-only HEAD^)
Pytest: pytest-testmon 플러그인 — 코드 변경에 영향받는 테스트만
Bazel: bazel test $(bazel query 'rdeps(//..., set(...changed files...))')

의존 그래프를 빌드 시스템이 알면 정확. 모르면 import 분석으로 근사
코드 커버리지 기반 (정확)
아이디어: 이전 빌드에서 "테스트 X가 어떤 코드 라인을 실행했는지" 기록. 다음 빌드에서 변경된 라인이 X의 커버리지에 포함되면 X 실행

도구:
· Microsoft TIA (Azure DevOps)
· Datadog Test Visibility
· Launchable

주의: 학습 데이터(과거 커버리지)가 필요 → 도입 초기엔 효과 적음. 6개월 후 효과 본격화
실전 적용 — 점진적 도입
Phase 1: 모든 PR에서 전체 테스트. 단, 디렉토리/패키지 단위로 affected만 — pnpm --filter ...[origin/main] test
Phase 2: 모노레포 도구(Nx, Turborepo)의 affected 활용. 변경 패키지 + 그 의존 패키지
Phase 3: 파일 단위 TIA — Jest --findRelatedTests 또는 testmon
Phase 4: 머신러닝 기반 (Launchable) — flaky 우선순위, 실패 가능성 높은 테스트 먼저

안전장치: 매 nightly에 전체 테스트 1회 실행 — TIA가 놓친 영향 검증

파이프라인 운영

Flaky 테스트 대응, 품질 게이트, CI 관측성, 비용 최적화 — 파이프라인을 "제품"으로 운영하는 관점.

Reliability

Flaky Test — 신뢰의 가장 큰 적

"가끔 실패하는 테스트"가 한두 개만 있어도 파이프라인 전체에 대한 신뢰가 무너진다
Flaky test는 "같은 코드, 같은 테스트, 다른 결과"다. 원인은 (1) 비동기 race, (2) 시간/날짜 의존, (3) 외부 의존성, (4) 테스트 간 상태 누수 — 거의 항상 테스트 자체가 잘못된 것이다. 한두 개만 있어도 개발자는 "또 flaky겠지"하고 진짜 실패도 무시하게 된다.
탐지 — 자동화된 통계
매일 같은 SHA로 테스트 N회 재실행 → 결과 분포 기록:
test_name pass_rate last_30d test_login 100% test_payment_flow 87% <- FLAKY test_search 95% <- BORDERLINE test_logout 100%
도구:
· Buildkite Test Analytics
· Datadog Test Visibility
· GitHub Actions의 retry-on-failure stats
· 자체 구축: junit.xml 결과를 BigQuery에 적재 → 대시보드
대응 — 격리 우선
1. 즉시 격리 — flaky 발견 시 "skip"하지 말고 별도 job에 격리. 메인 파이프라인은 통과시키되, 격리 job 결과는 별도 추적
2. 책임자 지정 — 격리된 테스트마다 owner 자동 할당 (CODEOWNERS 기반)
3. 24시간 SLA — flaky로 등록된 테스트는 24시간 내 수정 또는 삭제. "계속 격리"는 죽음의 침대
4. retry는 최후 수단 — flaky 원인을 찾기 전 retry로 가리는 건 부채. 단, 정말 외부(네트워크) 원인이면 명시적 retry는 OK
근본 원인 패턴
대부분의 flaky는 다음 중 하나:

1. 시간 의존: setTimeout, sleep(100)로 동기화 — fake timer 또는 explicit waiting 사용
2. 순서 의존: 테스트 A가 만든 상태를 B가 기대 — 각 테스트는 setUp/tearDown으로 isolated 환경
3. 외부 의존: 실제 API 호출 — mock 또는 contract test
4. Race condition: 여러 비동기 작업의 완료 순서 — Promise.all + 명시적 ordering
5. 환경 변수 누수: 이전 테스트가 설정한 env가 남음 — beforeEach에서 reset

"이 테스트는 머신이 빠를 때만 실패"는 race condition이지 머신 문제 아니다. 마지막 카테고리 진단이 가장 어렵다.
Gate

품질 게이트 — 코드 머지 직전의 최후 검증

테스트 통과만으로는 부족하다. 커버리지, 정적 분석, 라이선스, 보안까지 자동 게이트
Quality Gate는 "이 PR이 머지될 자격이 있는가"를 자동 판정하는 룰셋이다. 사람의 코드 리뷰는 의도/설계를 보고, 게이트는 측정 가능한 품질을 본다.

Observability

CI 관측성 — 파이프라인도 모니터링 대상

파이프라인의 latency, success rate, queue time을 측정하지 않으면 점점 느려지는 걸 모른다
애플리케이션의 latency p95를 측정하듯, 파이프라인의 wall clock p95도 측정해야 한다. CI는 개발자 생산성에 직결되는 인프라이며, 한번 느려지면 사람이 느낄 즈음엔 이미 한참 악화된 후다.
측정해야 할 메트릭
1. PR 파이프라인 wall clock p50/p95 — 5분/10분 SLO
2. Stage별 duration 분포 — 어디가 병목인지
3. Success rate (per branch type) — main green ratio, PR 첫 시도 성공률
4. Queue time — 러너 대기 시간 (러너 부족 신호)
5. Retry rate — 재시도 빈도 (flaky 신호)
6. Cost per pipeline run — 비용 트렌드
DORA에 연결
CI 메트릭은 DORA 4 metrics에 직결:

Lead Time for Changes = code commit → prod deploy
· CI duration이 큰 비중
Deployment Frequency = 배포 빈도
· CI/CD 신뢰도가 높아야 자주 배포 가능
Change Failure Rate = 배포 후 실패율
· CI quality gate가 1차 방어선
MTTR = 복구 시간
· CI/CD가 빨라야 hotfix 배포도 빠름

CI 메트릭은 곧 DORA 메트릭의 leading indicator다
대시보드 — 무엇을 어떻게 보여줄까
일일 헬스 보드:
· main branch green ratio (24h, 7d)
· PR 파이프라인 p95 duration
· top 5 flaky tests
· top 5 slowest jobs

주간 트렌드:
· 파이프라인 duration 추이 (회귀 감지)
· 비용 추이
· quality gate 통과율 추이

대시보드는 개발자가 매일 보는 자리에 — Slack에 데일리 요약, 사무실 모니터에 표출. CI 헬스가 "안 보이는" 순간 악화한다
Cost

CI 비용 최적화

CI 비용은 조용히 자란다 — 한 번 검토하면 30~50% 절감하는 경우가 흔하다
팀이 커지고 빌드가 늘면 CI 비용은 선형~초선형으로 증가한다. GitHub Actions 같은 매니지드 서비스는 분당 과금이라, 월 수천 달러 단위가 금방 된다. 정기 점검으로 줄일 여지가 크다.
큰 절감 — 무엇을 안 돌릴 것인가
1. 중복 트리거 제거 — push와 pull_request 양쪽에 같은 워크플로우 트리거 금지. branch protection은 PR만 보면 충분
2. Skip on no-op — paths-ignore로 docs/README 변경 시 스킵
3. Cancel old runs — 같은 PR의 이전 실행은 자동 취소 (concurrency.cancel-in-progress: true)
4. Draft PR 스킵 — draft 상태에선 가벼운 검증만
5. Stale PR 정리 — 30일 이상 inactive PR 자동 close
중간 절감 — 어떻게 돌릴 것인가
1. 큰 러너 신중히 — 4 vCPU runner는 1 vCPU의 4배 비용. 정말 필요한 stage만
2. Self-hosted로 이전 — 빌드량 큰 워크로드는 자체 ECS/GKE에서 ARC로. 손익분기점 ≈ 월 50,000분
3. Spot instance — self-hosted runner를 spot으로 (interruption tolerant 워크로드만)
4. 러너 OS 선택 — macOS는 Linux의 10배 비용. 진짜 macOS 빌드만 macOS
5. Cache hit rate 모니터 — 50% 이하면 캐시 키 설계 재검토
정기 점검 체크리스트 (월 1회)
1. Top 10 가장 자주 도는 워크플로우 — 비용 차지 비중 / 정말 모두 필요한지
2. Top 10 가장 느린 job — 캐시 활용 여부 / 병렬화 여지
3. Cancellation rate — 5% 이상이면 같은 PR 다중 실행으로 낭비 중
4. Failure rate — 30% 이상이면 quality gate가 잘못됐거나 flaky
5. 비용 그래프 — 전월 대비 +20% 이상이면 원인 추적

행동 원칙: CI 비용을 명시적으로 보고하지 않으면 아무도 신경 안 쓴다. 매월 팀 회고에 CI 비용 요약 1페이지 포함하는 것만으로도 효과 있다.
