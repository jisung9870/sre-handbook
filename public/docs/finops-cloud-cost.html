● FinOps Fundamentals

FinOps 원칙과 프레임워크

클라우드 비용을 엔지니어링 메트릭으로 다루는 운영 모델.

Definition

FinOps란

Finance + DevOps — 클라우드 비용에 대한 엔지니어링 책임을 팀에 부여하는 운영 모델
FinOps는 "클라우드 비용을 기술 의사결정의 일부로 다루는 문화"다. 비용 절감만이 목적이 아니라, 비용 대비 가치(cost efficiency)를 최적화하는 것이 핵심이다. 저렴하게 운영하되 SLO를 유지하는 균형점을 찾는다.
FinOps 핵심 원칙
1) Teams own their cloud costs: 비용 가시성을 팀에 제공. 팀이 자기 비용을 인지하고 관리
2) Decisions are driven by business value: 비용 절감이 아니라 비용 대비 가치 최대화
3) Everyone takes ownership: 엔지니어, 매니저, 재무 모두가 참여
4) Reports are accessible and timely: 실시간 비용 대시보드. 월말 리포트가 아님
5) Centralized team drives FinOps: 중앙 팀이 도구/프로세스 제공, 실행은 각 팀
SRE와 FinOps의 연결
SLO ↔ 비용 트레이드오프:
99.99% SLO를 유지하면서 비용을 최소화하는 것이 목표

Error Budget ↔ Cost Budget:
SLO를 낮추면(99.9%→99.5%) 인프라 요구사항이 줄어 비용 절감
= 비용도 일종의 Budget으로 관리할 수 있다

Capacity Planning ↔ FinOps:
Little's Law로 최적 리소스를 계산하면 과잉 프로비저닝 방지 = 비용 절감
Framework

Inform → Optimize → Operate

FinOps Foundation의 3단계 반복 사이클
Inform (가시성 확보)
질문: "지금 얼마를 어디에 쓰고 있는가?"

• 비용 대시보드 구축 (Kubecost, AWS Cost Explorer)
• 태깅 표준 수립: 모든 리소스에 team, service, environment 태그
• K8s 리소스 사용률 vs request 대비 실사용 비율 시각화
• 주간/월간 비용 리포트 자동 생성
• 이상 비용 알림 설정 (전일 대비 20% 증가 시)
Optimize → Operate
Optimize: "어떻게 줄일 수 있는가?"
• 라이트사이징 (과잉 request 축소)
• Spot 인스턴스 도입
• RI/Savings Plan 구매
• 미사용 리소스 제거 (idle EBS, 비사용 EIP)
• 데이터 전송 경로 최적화

Operate: "지속적으로 관리하는가?"
• 자동 라이트사이징 (VPA Off 모드 → 분기 반영)
• 비용 예산 설정 & 알림
• 분기별 비용 리뷰 미팅
• 신규 서비스 PRR에 비용 예측 포함
Metric

Unit Economics — 비용 효율성 메트릭

"요청 1건 처리에 얼마가 드는가?"를 추적하여 비즈니스 성장과 비용의 관계를 이해
총 비용만 보면 서비스가 성장할수록 비용이 늘어나는 게 당연해 보인다. 단위당 비용(Unit Cost)을 추적하면 "효율적으로 성장하고 있는가?"를 판단할 수 있다.
주요 Unit Cost 메트릭
Cost per Request = 월간 인프라 비용 ÷ 월간 총 요청 수 Cost per User = 월간 인프라 비용 ÷ MAU Cost per Transaction = 결제 인프라 비용 ÷ 월간 거래 수 Cost per GB stored = 스토리지 비용 ÷ 저장 데이터 GB

트렌드 분석: Unit Cost가 월별로 감소하면 효율 개선. 증가하면 과잉 프로비저닝 또는 비효율 성장
알림: Cost per Request가 전월 대비 30% 이상 증가하면 조사 트리거
경영진 리포트: "총 비용은 20% 증가했지만 요청당 비용은 10% 감소했습니다" — 이런 스토리텔링이 가능
● Kubernetes Cost Optimization

K8s 리소스 비용 최적화

라이트사이징, Kubecost/OpenCost, 비용 할당 — K8s 비용의 60~80%를 차지하는 컴퓨팅 최적화.

Critical

리소스 라이트사이징

request/limit 과잉 설정이 비용 낭비의 최대 원인 — 실사용 대비 최적값 찾기
K8s에서 비용 = request 기준으로 과금된다 (Karpenter/CA가 request 합계로 노드를 프로비저닝). request를 실사용보다 3~5배 높게 잡으면 60~80%의 노드 리소스가 유휴 상태로 낭비된다.
진단: 과잉 프로비저닝 파악
CPU 효율:
sum(rate(container_cpu_usage_seconds_total[5m])) by (namespace, pod) / sum(kube_pod_container_resource_requests{resource="cpu"}) by (namespace, pod)
→ 30% 이하면 과잉. 70~80%가 이상적

Memory 효율:
sum(container_memory_working_set_bytes) by (namespace, pod) / sum(kube_pod_container_resource_requests{resource="memory"}) by (namespace, pod)
→ 50% 이하면 request 축소 여지 있음

클러스터 전체:
1 - (sum(kube_pod_container_resource_requests{resource="cpu"}) / sum(kube_node_status_allocatable{resource="cpu"}))
→ 할당되지 않은 CPU 비율. 이것이 "빈 공간"
최적화 방법
VPA Off 모드 활용:
VPA를 updateMode: "Off"로 배포 → 추천값만 수집
kubectl describe vpa my-vpa → Target 값 확인
분기마다 추천값을 수동으로 request에 반영

실전 가이드:
• CPU request: P95 사용량 × 1.2 (20% 여유)
• Memory request: P99 사용량 × 1.1 (10% 여유. OOM 방지)
• CPU limit: 설정하지 않거나 request의 2~3배 (throttling 방지)
• Memory limit: request와 같거나 1.5배 (OOM 방지)

절감 효과: request 50% 축소 → 노드 수 ~40% 감소 가능
⚠ 라이트사이징의 위험
너무 공격적으로 줄이면 CPU throttling(성능 저하)이나 OOM Kill(안정성 저하)이 발생한다. Utilization-Latency 비선형 관계를 기억하자. 리소스를 줄이면서 반드시 SLI(P99 latency)를 동시에 모니터링해야 한다. SLO를 유지하는 범위 내에서만 축소.

Allocation

비용 할당 (Chargeback / Showback)

팀/서비스별로 클라우드 비용을 분배하여 비용 의식을 높이는 체계
할당 전략
Showback: 비용을 보여주기만 (책임은 안 지움). 초기 도입에 적합
Chargeback: 비용을 팀 예산에서 차감. 강력한 인센티브

할당 기준:
• Namespace 기반: 가장 단순. namespace = team 매핑
• Label 기반: team: platform, service: api 라벨로 분류
• 공유 비용 분배: Control Plane, 모니터링 등 공통 비용은 요청 비율로 분배

리포트 주기: 주간 Slack 자동 게시 + 월간 상세 리포트
태깅 표준
K8s 라벨 (필수):
app.kubernetes.io/name
app.kubernetes.io/component
team
environment

AWS 태그 (필수):
Team, Service, Environment, CostCenter

모든 리소스에 default_tags 블록으로 기본 태그 적용

● Compute Cost

컴퓨팅 비용 최적화

Spot 인스턴스, Reserved Instance, Savings Plan, 인스턴스 선택 — 가장 큰 비용 항목 최적화.

Savings

Spot 인스턴스 전략

On-Demand 대비 60~90% 저렴. 단, 2분 전 알림으로 언제든 회수될 수 있다
Spot 인스턴스는 AWS의 여유 컴퓨팅 용량을 할인된 가격에 사용하는 것이다. 가격이 저렴하지만 수요 증가 시 2분 전 알림으로 회수(interruption)된다. K8s + Karpenter를 사용하면 Spot 회수를 graceful하게 처리할 수 있다.
Karpenter + Spot 설정
NodePool에서 Spot 우선 + On-Demand fallback:
spec.template.spec.requirements:
- key: karpenter.sh/capacity-type
operator: In
values: ["spot", "on-demand"]

Karpenter가 먼저 Spot을 시도하고, Spot 가용량 부족 시 On-Demand로 fallback

다양한 인스턴스 타입:
- key: node.kubernetes.io/instance-type
operator: In
values: ["m5.xlarge","m5a.xlarge","m6i.xlarge","c5.xlarge","c5a.xlarge"]

다양한 타입을 지정할수록 Spot 가용량 확보 확률 증가. 특정 타입 하나만 지정하면 Spot 확보 어려움
Spot 적합/부적합 워크로드
✅ 적합:
• Stateless 웹 서비스 (K8s가 자동 재스케줄링)
• 배치 처리, CI/CD 빌드 작업
• Dev/Staging 환경 전체
• HPA로 충분한 replica가 유지되는 서비스

❌ 부적합:
• Stateful 워크로드 (DB, Kafka, etcd)
• 단일 replica 서비스 (회수 시 다운타임)
• 장시간 실행 job (Spot 회수로 중단)

Spot 절감 예시:
m5.xlarge On-Demand: $0.192/hr
m5.xlarge Spot: ~$0.06/hr (약 70% 절감)
노드 20개 × 730시간 = $1,927/월 절감
💡 Spot 중단 대비
PodDisruptionBudget: minAvailable: 2로 동시에 모든 Pod이 죽지 않도록
Topology Spread: Spot 노드와 On-Demand 노드에 걸쳐 분산
Graceful Shutdown: SIGTERM 수신 시 진행 중 요청 완료 후 종료 (terminationGracePeriodSeconds)
AWS Node Termination Handler: Spot 중단 알림 수신 → Pod을 drain하여 안전 이동
Commitment

Reserved Instances & Savings Plans

1~3년 약정으로 On-Demand 대비 30~60% 할인받는 구매 모델
모델할인율유연성적합 대상
Standard RI~40~60%낮음 (인스턴스 패밀리/리전 고정)안정적인 baseline 워크로드, DB
Convertible RI~30~45%중간 (타입 변경 가능)장기 워크로드이지만 타입 변경 여지
Compute SP~30~50%높음 (패밀리/리전/OS 자유)다양한 인스턴스를 쓰는 환경 (K8s)
EC2 Instance SP~40~55%중간 (패밀리/리전 고정)특정 인스턴스에 집중된 환경
K8s 환경 구매 전략
Baseline 커버리지: 항상 필요한 최소 노드 수(예: 5대)를 Compute Savings Plan으로 약정. 이 위에 Spot으로 확장.

3계층 구조:
• Savings Plan (30~50%): baseline 커버 → 안정적 할인
• Spot (60~90%): 유동적 부하 흡수 → 최대 할인
• On-Demand (0%): Spot fallback → 안정성 보장

비율 예시: SP 40% + Spot 40% + On-Demand 20%
→ 평균 할인율 ~50%. On-Demand만 쓸 때 대비 절반 비용

주의: Karpenter가 인스턴스 타입을 유동적으로 선택하므로 Compute Savings Plan(타입 무관)이 가장 적합
Selection

인스턴스 타입 선택 전략

워크로드 특성에 맞는 인스턴스 패밀리 선택으로 비용 대비 성능 최적화
워크로드별 매핑
CPU-bound (API 서버): c7g (Graviton3), c6i
→ CPU 집약적. vCPU당 비용이 가장 낮음

범용 (대부분의 워크로드): m7g, m6i
→ CPU:Memory = 1:4 균형. K8s 범용 노드

Memory-bound (Redis, JVM): r7g, r6i
→ CPU:Memory = 1:8. 메모리 집약적

Graviton (ARM): *g 접미사 (m7g, c7g, r7g)
→ x86 대비 ~20% 저렴 + ~20% 성능 향상
→ Go, Node.js, Python 앱은 대부분 ARM 호환
→ Karpenter NodePool에서 kubernetes.io/arch: arm64 지정
노드 크기 전략
큰 노드 vs 작은 노드:

큰 노드 (m5.4xlarge):
장점: DaemonSet 오버헤드 비율 감소, bin packing 효율
단점: 단일 노드 장애 영향 큼

작은 노드 (m5.xlarge):
장점: 세밀한 스케일링, Spot 가용량 높음
단점: DaemonSet 비율 높음, 네트워크/ENI 제한

권장: 4~8 vCPU (xlarge~2xlarge)가 대부분의 K8s 환경에 최적
Karpenter가 Pod 요구사항에 맞는 최적 크기를 자동 선택하므로 여러 크기를 허용하는 것이 좋음
● Network Cost

네트워크 비용 최적화

NAT Gateway, Cross-AZ, 데이터 전송 — 보이지 않는 비용의 최대 주범.

Hidden Cost

NAT Gateway 비용 최적화

NAT Gateway의 데이터 처리 요금이 월 수천 달러에 달할 수 있다
NAT Gateway는 시간당 요금($0.045/hr ≈ $32/월) 외에 데이터 처리 요금 $0.045/GB가 있다. ECR 이미지 풀, S3 접근, 외부 API 호출이 모두 NAT를 경유하면 월 수백~수천 달러가 될 수 있다.
비용 분석
VPC Flow Log 분석:
NAT Gateway를 통과하는 트래픽의 목적지 분석
→ S3, ECR, CloudWatch가 대부분인 경우 많음

비용 시뮬레이션:
ECR 이미지 풀: 1GB/deploy × 10 deploys/day = 300GB/월
S3 접근: 100GB/월 (Loki 청크, 백업 등)
외부 API: 50GB/월
총: 450GB × $0.045 = $20.25/월 (데이터 처리만)
+ 시간 요금 $32 × AZ 수 = $96~$160/월
절감 방법
VPC Endpoint 필수 구성:
• com.amazonaws.region.s3 — Gateway (무료!)
• com.amazonaws.region.ecr.dkr — Interface
• com.amazonaws.region.ecr.api — Interface
• com.amazonaws.region.sts — Interface (IRSA)
• com.amazonaws.region.logs — Interface

S3 Gateway Endpoint만으로도 대폭 절감:
Loki/Tempo/Mimir가 S3에 데이터 저장 → Gateway Endpoint(무료)로 NAT 우회

Interface Endpoint 비용: ~$7.2/AZ/월 + $0.01/GB
NAT 비용보다 저렴한지 비교 후 결정
AZ

Cross-AZ 트래픽 비용

같은 리전이라도 AZ가 다르면 $0.01/GB 과금 — K8s에서 자주 발생
K8s Service는 기본적으로 모든 AZ의 Pod에 균등 분산한다. Pod A(AZ-a)가 Service B를 호출하면 B의 Pod이 AZ-b에 있을 수 있고, 이때 Cross-AZ 비용 $0.01/GB이 발생한다. 트래픽이 많으면 상당한 비용이 된다.
비용 추정
서비스 간 트래픽 50TB/월:
50% cross-AZ 확률 = 25TB cross-AZ
25TB × $0.01 × 2(양방향) = $500/월

모니터링:
VPC Flow Log에서 AZ 간 트래픽 분석
istio_tcp_sent_bytes_total 메트릭에서 source/destination 서비스의 AZ 비교
최적화 방법
Topology-Aware Routing (K8s 1.23+):
service.kubernetes.io/topology-mode: Auto
→ 같은 AZ의 Pod을 우선 라우팅. cross-AZ 트래픽 대폭 감소

주의: AZ별 Pod 수가 균등해야 효과적. 불균형하면 한 AZ에 트래픽 집중

Istio Locality Load Balancing:
DestinationRule의 localityLbSetting으로 같은 zone 우선

DB/Cache 같은 AZ:
RDS/ElastiCache와 앱 Pod을 같은 AZ에 배치
nodeSelector 또는 topology spread로 조정

인바운드는 무료, 아웃바운드에 과금 — 전송 경로를 최적화
비용 구조
Internet → AWS: 무료
AWS → Internet: $0.09/GB (첫 10TB), $0.085/GB (다음 40TB)
같은 AZ 내: 무료 (Private IP 사용 시)
AZ 간: $0.01/GB (각 방향)
리전 간: $0.02/GB
AWS → NHN Cloud (인터넷 경유): 아웃바운드 요금

CloudFront 활용:
정적 자산을 CloudFront로 서빙하면 AWS→Internet 전송 비용이 CloudFront 비용으로 대체 (보통 30~50% 절감)
절감 전략 요약
1) VPC Endpoint로 AWS 서비스 트래픽을 NAT에서 분리
2) S3 Gateway Endpoint (무료) 최우선 설정
3) ECR pull-through cache로 외부 레지스트리 이미지의 NAT 경유 제거
4) Topology-Aware Routing으로 cross-AZ 트래픽 최소화
5) CloudFront로 사용자 대면 트래픽의 전송 비용 절감
6) 로그/메트릭 전송량 자체를 줄이기 (불필요한 로그 드롭, 메트릭 샘플링)
7) 압축 (gzip/zstd)으로 전송 데이터 크기 축소

EBS 비용 최적화

EBS 볼륨 타입 선택, 미사용 볼륨 정리, IOPS/Throughput 최적화
볼륨 타입 선택
gp3 (범용): $0.08/GB/월. 3000 IOPS + 125MB/s baseline 포함. 대부분의 워크로드에 최적
gp2→gp3 마이그레이션: gp3가 gp2보다 20% 저렴하고 baseline 성능 우수. 무중단 변경 가능
aws ec2 modify-volume --volume-type gp3 --volume-id vol-xxx

io2 (고성능): IOPS 보장 필요 시. DB 전용
st1 (throughput): 시퀀셜 읽기 워크로드. 로그 스토리지
sc1 (cold): 거의 접근 안 하는 아카이브
낭비 제거
미사용 EBS 볼륨:
aws ec2 describe-volumes --filters Name=status,Values=available
→ available 상태 = 어떤 인스턴스에도 연결 안 됨 = 비용만 발생

과대 볼륨:
100GB PVC인데 실사용 10GB → 볼륨 축소 (새 PV 생성 → 데이터 복사 → PVC 전환)

스냅샷 정리:
오래된 EBS 스냅샷 삭제 ($0.05/GB/월)
aws ec2 describe-snapshots --owner-ids self --query 'Snapshots[?StartTime<=\`2024-01-01\`]'

자동화: 주간 CronJob으로 미사용 볼륨/스냅샷 감지 → Slack 알림

로그/메트릭 비용 관리

관측 데이터가 전체 인프라 비용의 20~30%를 차지할 수 있다
LGTM 스택을 자체 호스팅하더라도 스토리지 비용(S3), 컴퓨팅 비용(Ingester/Querier), 네트워크 비용(전송)이 발생한다. 관측 데이터 볼륨을 관리하지 않으면 인프라 비용을 잡아먹는다.
로그 비용 절감
Alloy에서 불필요한 로그 드롭:
• health check 로그 드롭 → 20~30% 볼륨 감소
• debug 레벨 로그 드롭 → 30~50% 감소
• 반복 로그 샘플링 (N건 중 1건만 저장)

구조화 로그 강제:
JSON 로그 → Loki 파싱 효율 증가, 저장 압축률 향상

보존 기간 차등:
• prod: 30일, staging: 7일, dev: 3일
• Loki limits_config.retention_period를 테넌트별 설정
메트릭 비용 절감
카디널리티 관리:
(Observability 문서에서 상세) 불필요한 라벨 제거, 메트릭 드롭

Scrape interval 조정:
• 핵심 서비스: 15초 (기본)
• 인프라 메트릭: 30초
• 비핵심: 60초
간격 2배 → 시계열 데이터 50% 감소

Recording Rule로 집계 후 원본 삭제:
원본 고카디널리티 메트릭은 7일만 보존
Recording Rule 결과는 1년 보존

비용 추정:
시계열 100만개 × 15초 간격 × 30일 보존
≈ 50~100GB 디스크 사용 (Prometheus TSDB)
