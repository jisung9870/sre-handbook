● Fundamentals

분산 시스템 이론 기초

네트워크는 신뢰할 수 없고, 장애는 반드시 발생하며, 완벽한 일관성과 가용성은 동시에 불가능하다.

Foundation

분산 컴퓨팅의 8가지 오류 (Fallacies)

Peter Deutsch / James Gosling — "개발자가 분산 시스템에서 처음에 가정하지만 틀린 것들"
이 8가지는 분산 시스템을 설계할 때 절대 가정해선 안 되는 것이다. 이 오류를 기반으로 코드를 짜면 프로덕션에서 반드시 문제가 된다.
8 Fallacies & SRE 영향
1) 네트워크는 신뢰할 수 있다 → 패킷 손실, 파티션 발생. 재시도, 타임아웃, 서킷 브레이커 필수
2) 지연시간은 0이다 → 원격 호출은 항상 지연. 비동기 설계, 캐시, batch 처리
3) 대역폭은 무한하다 → 대량 데이터 전송 시 병목. 압축, 페이지네이션, 스트리밍
4) 네트워크는 안전하다 → mTLS, NetworkPolicy, zero-trust
5) 토폴로지는 변하지 않는다 → K8s Pod IP는 매번 바뀜. Service Discovery 필수
6) 관리자는 한 명이다 → 멀티팀, 멀티클라우드. 표준화된 운영 체계
7) 전송 비용은 0이다 → 클라우드 네트워크 비용. cross-AZ/region 트래픽 최적화
8) 네트워크는 균일하다 → 이기종 환경, MTU 차이, NHN Cloud↔AWS 간 차이
Theorem

CAP Theorem

Consistency, Availability, Partition Tolerance — 네트워크 파티션이 생기면 C와 A 사이에서 선택이 필요하다
CAP은 "파티션이 발생했을 때"의 선택에 관한 것이다. 파티션이 없을 때는 C와 A를 모두 어느 정도 만족시킬 수 있다. 현실에서 네트워크 파티션은 반드시 발생하므로(Fallacy #1), 결국 CP와 AP 중 어디에 더 무게를 둘지 결정해야 한다.

실무에서 CAP는 이분법보다 스펙트럼에 가깝다. 완전한 C도, 완전한 A도 아닌 중간 지점(eventual consistency 등)을 선택하는 것이 일반적이다.
CP 시스템
파티션 시 일관성 유지, 일부 요청 거부
• etcd/ZooKeeper: 과반수 합의 필요. 소수 파티션은 read/write 불가
• PostgreSQL (sync replication): 레플리카 응답까지 대기
적합: 금융, 재고, 설정(etcd), 리더 선출
대가: 파티션 시 가용성 감소
AP 시스템
파티션 시 가용성 유지, 일관성 포기
• Cassandra, DynamoDB: 모든 노드가 읽기/쓰기 가능
• DNS: TTL 안에서는 최신이 아닌 데이터를 허용
• Redis (async replication): 쓰기 후 복제 전 장애 시 데이터 손실 가능
적합: 세션, 캐시, 소셜 피드, IoT 데이터
대가: conflict resolution 필요 (last-write-wins, vector clock 등)
K8s에서의 CAP
etcd는 CP 시스템이다. Raft consensus로 과반수 노드가 살아있어야 동작. 3노드 etcd에서 2노드가 죽으면 클러스터 전체가 read-only가 되고 새 Pod을 생성할 수 없다. 이것이 etcd가 반드시 홀수 노드(3, 5, 7)로 구성되어야 하는 이유이며, etcd 가용성이 K8s 전체 가용성의 상한선이 되는 이유다.
Algorithm

Consensus — Raft 알고리즘

분산 노드들이 하나의 값에 합의하는 알고리즘. etcd의 핵심 엔진
Consensus는 "여러 노드가 같은 데이터를 같은 순서로 가지고 있다"를 보장하는 알고리즘이다. Paxos가 이론적 기초이고, Raft는 이해하기 쉽게 재설계된 실용적 구현이다. etcd, Consul, CockroachDB 등이 Raft를 사용한다.
Raft 핵심 동작
Leader Election: 노드는 Follower, Candidate, Leader 중 하나. Leader가 죽으면 Candidate가 투표 요청 → 과반수 득표 시 새 Leader
Log Replication: 모든 쓰기는 Leader → 과반수 Follower에 복제 → commit 확인 → 클라이언트에 응답
Safety: commit된 로그 엔트리는 모든 미래 Leader에 포함 보장
SRE 관점: etcd 운영
홀수 노드 필수: quorum = N/2+1. 3→1 장애 허용, 5→2 허용
디스크 지연 시간에 민감: Raft heartbeat/election timeout이 디스크 fsync에 의존한다. SSD 필수, WAL_DIR을 별도 디스크로 두는 편이 좋다
모니터링: etcd_server_leader_changes_seen_total, etcd_disk_wal_fsync_duration_seconds, etcd_server_proposals_failed_total
백업: etcdctl snapshot save 정기 실행
Model

Consistency 모델 스펙트럼

Strong → Eventual — 일관성 강도에 따른 성능과 가용성 트레이드오프
모델보장대가예시
Linearizable모든 읽기가 가장 최근 쓰기를 반영높은 latency. 모든 노드 동기화 필요etcd, Spanner
Sequential모든 클라이언트가 같은 순서를 봄단일 리더 병목ZooKeeper
Causal인과관계가 있는 연산만 순서 보장구현 복잡도MongoDB (causal sessions)
Eventual충분한 시간 후 모든 노드가 같은 값일시적 불일치. conflict 가능Cassandra, DynamoDB, DNS
Read-your-writes자신이 쓴 값을 즉시 읽을 수 있음session affinity 필요대부분의 웹 앱에서 최소 요구
💡 실무 선택 기준
"이 데이터가 1초 동안 stale여도 괜찮은가?" → Eventual. "이 거래가 정확히 한 번만 처리되어야 하는가?" → Linearizable. 대부분의 서비스는 Read-your-writes + Eventual이면 충분하다.
● Failure Patterns

분산 시스템 장애 패턴

Cascading Failure, Retry Storm, Thundering Herd — 단일 장애가 시스템 전체를 무너뜨리는 메커니즘.

Critical

Cascading Failure (연쇄 장애)

하나의 컴포넌트 장애가 도미노처럼 전체 시스템으로 전파되는 현상
마이크로서비스에서 서비스 A가 죽으면 A를 호출하는 B가 timeout으로 느려지고, B를 호출하는 C도 느려지고, 결국 전체 시스템이 마비된다. 각 서비스가 독립적으로 보이지만, 의존성 체인을 통해 장애가 증폭된다.

핵심 메커니즘: 자원 고갈(resource exhaustion). 느려진 하위 서비스의 응답을 기다리면서 스레드/커넥션 풀이 소진되고, 새 요청도 처리 불가.
전파 경로
1) 서비스 C의 DB 느려짐
2) C의 응답 지연 → B가 C를 기다리며 커넥션 풀 소진
3) B도 느려짐 → A가 B를 기다리며 스레드 풀 소진
4) A가 요청 처리 불가 → 사용자에게 에러/타임아웃
5) 사용자가 새로고침 반복 → 트래픽 증가 → 악순환
방어 전략
Circuit Breaker: 실패율 임계값 초과 시 호출 차단
Timeout: 모든 외부 호출에 적절한 timeout 설정
Bulkhead: 의존성별 리소스 격리
Load Shedding: 과부하 시 의도적 요청 거부
Graceful Degradation: 핵심 기능만 유지, 부가 기능 OFF
Amplification

Retry Storm (재시도 폭풍)

장애 시 모든 클라이언트가 동시에 재시도하여 트래픽이 기하급수적으로 증폭
서비스가 일시적으로 에러를 반환하면, 모든 클라이언트가 동시에 재시도한다. 원래 1000 RPS → 에러 → 각 클라이언트 3회 재시도 → 3000 RPS. 회복 중인 서비스에 3배 부하가 가해져 영원히 복구 못 하는 상태에 빠진다.

다단계 아키텍처에서 각 레이어가 재시도하면 곱셈 효과: A→B→C, 각 3회 → 최대 27배 증폭.
방어 패턴
Exponential Backoff: 1초 → 2초 → 4초 → 8초... 간격 증가
Jitter 추가: 랜덤 지연을 더해 재시도 시점을 분산
delay = base × 2^attempt + random(0, base)
Max Retries: 최대 재시도 횟수 제한 (보통 3~5회)
Retry Budget: 전체 요청 중 재시도 비율을 10% 이하로 제한
K8s 환경에서의 retry
Envoy/Istio: Service Mesh에서 retry 정책을 중앙 관리
retryOn: 5xx,reset,connect-failure
numRetries: 3
retryBackOff: baseInterval: 100ms
핵심: 앱 레벨 retry + Mesh retry가 겹치면 증폭. 한 곳에서만 retry 하도록 조율
Burst

Thundering Herd (우르르 몰려가기)

동시에 많은 요청이 같은 리소스에 몰려 과부하를 일으키는 패턴
대표적 시나리오: 캐시 만료. 인기 키의 캐시가 동시에 만료되면, 수천 요청이 동시에 DB로 직행한다. DB가 부하를 감당 못 하면 전체 장애로 이어진다.

다른 시나리오: 서비스 재시작 후 콜드 스타트, 예약된 cron job 동시 실행, 대규모 이벤트(세일, 티켓팅) 시작 시점.
방어 패턴
Cache Stampede 방어:
• Lock (Singleflight): 캐시 미스 시 첫 요청만 DB 조회, 나머지는 결과 대기. Go의 singleflight 패키지
• Stale-while-revalidate: 만료된 캐시를 우선 반환하고 백그라운드에서 갱신
• Jittered Expiry: TTL에 랜덤 오프셋 추가. 동시 만료 방지
• Early Recomputation: 만료 N초 전에 미리 갱신
실무 적용
Redis TTL jitter: TTL = base + random(0, base×0.1)
10분 TTL → 10m~11m 사이 랜덤
K8s CronJob: 같은 시각에 여러 Job이 실행되면 노드 리소스 spike. startingDeadlineSeconds로 분산
Rolling restart: 모든 Pod이 동시에 재시작하지 않도록 maxUnavailable/maxSurge 조절
Ambiguity

Partial Failure (부분 장애)

요청이 성공했는지 실패했는지 알 수 없는 상태 — 분산 시스템의 가장 어려운 문제
단일 머신에서 함수 호출은 성공 또는 실패가 명확하다. 분산 시스템에서는 세 번째 상태가 존재한다: "모른다." 요청을 보냈는데 응답이 안 왔다. 서버가 처리 후 응답 전에 죽었을 수도, 네트워크가 응답을 삼켰을 수도 있다.

이 모호성을 해결하기 위해 Idempotency(멱등성)가 필수다.
시나리오
결제 API 호출 → timeout:
• 서버가 처리 안 했으면 → 재시도해야 함
• 서버가 처리했는데 응답만 실패 → 재시도하면 이중 결제
DB 쓰기 → 네트워크 끊김:
• commit 됐는지 안 됐는지 알 수 없음
• 재시도하면 중복 데이터 가능
해결 패턴
Idempotency Key: 클라이언트가 요청마다 고유 ID를 부여. 서버가 중복 처리 방지
At-least-once + Deduplication: 최소 1번 전달 보장 + 수신 측에서 중복 제거
Exactly-once (이론적): 실질적으로 at-least-once + idempotent 연산으로 구현
Saga 패턴: 분산 트랜잭션을 보상 트랜잭션 체인으로 대체
● Resilience Patterns

분산 시스템 방어 패턴

Circuit Breaker, Bulkhead, Backpressure, Load Shedding — 장애 전파를 막고 시스템을 보호하는 설계 패턴.

Protection

Circuit Breaker (회로 차단기)

하위 서비스의 실패율이 임계값을 넘으면 호출을 차단하여 연쇄 장애를 방지
전기 회로의 차단기처럼, 실패가 반복되면 회로를 열어 더 이상 요청을 보내지 않는다. 주기적으로 소량의 요청만 보내(half-open) 상태를 확인하고, 정상이면 다시 닫는다.
3가지 상태
Closed (정상): 요청을 정상 전달. 실패율 모니터링
Open (차단): 모든 요청 즉시 실패 반환 (fast fail). 하위 서비스 보호
Half-Open (시험): 소량 요청만 통과. 성공하면 Closed, 실패하면 Open 유지

트리거 조건 예: 최근 10초 내 50% 이상 실패, 최소 20건 이상 요청
구현 예시
Go: sony/gobreaker, afex/hystrix-go
Istio: DestinationRule의 outlierDetection
consecutiveErrors: 5
interval: 10s
baseEjectionTime: 30s
maxEjectionPercent: 50
Traefik: Circuit Breaker middleware
expression: "LatencyAtQuantileMS(50.0) > 100"
Isolation

Bulkhead (격벽) 패턴

의존성별로 리소스 풀을 격리하여 한 의존성의 장애가 전체에 영향을 주지 않도록
선박의 격벽처럼, 한 구획에 물이 들어와도 다른 구획은 안전한 구조. 서비스 A가 DB, Redis, 외부 API를 호출할 때, 각각의 커넥션 풀과 스레드 풀을 분리하면 하나가 느려져도 나머지는 정상 동작.
구현 방식
Thread Pool 격리: 의존성별 별도 스레드 풀. 한 풀이 소진되어도 다른 풀은 무사
Connection Pool 격리: DB pool, Redis pool, HTTP client pool 각각 독립
Semaphore 격리: 동시 호출 수 제한. 스레드보다 경량
K8s: 서비스별 별도 Deployment + 리소스 제한 (cgroup 격리)
설계 포인트
풀 크기 결정: Little's Law 기반
pool_size = throughput × avg_latency
100 RPS, 평균 50ms → 풀 크기 = 5 (최소). 여유분 2~3배 = 10~15
풀 소진 시: 즉시 에러를 반환하고 대기열을 길게 늘리지 않는다 → fast fail + circuit breaker 트리거
Flow Control

Backpressure (배압)

하류가 처리할 수 없는 속도로 상류가 데이터를 보내는 것을 제어하는 메커니즘
Backpressure는 "소비자가 생산자에게 속도를 줄여달라고 신호를 보내는 것"이다. 버퍼가 가득 차면 더 이상 받지 않고, 생산자가 속도를 조절하도록 한다.

Backpressure 없이 무한 버퍼링을 하면 메모리 고갈 → OOM으로 이어질 수 있다. 버퍼 없이 바로 드롭하면 데이터 손실이 생긴다. Backpressure는 이 둘 사이에서 시스템을 안정적으로 유지하는 장치다.
구현 전략
Bounded Queue: 큐 크기 제한. 가득 차면 생산자 block 또는 거부
Rate Limiting: 생산자의 요청 속도를 토큰 버킷/리키 버킷으로 제한
TCP Flow Control: TCP 수신 윈도우가 0이면 송신 중지 (프로토콜 레벨 backpressure)
Reactive Streams: request(n) 기반 pull model. 소비자가 처리 가능한 양만 요청
K8s/인프라에서의 backpressure
K8s HPA: 부하 증가 → replica 증가 (간접적 backpressure)
Kafka: consumer lag이 backpressure 신호. lag 증가 시 consumer 스케일 아웃
Ingress Rate Limiting: Traefik rateLimit middleware
gRPC: flow control window 기반 내장 backpressure
Overload

Load Shedding (부하 제거)

과부하 시 일부 요청을 의도적으로 거부하여 나머지 요청의 품질을 유지
모든 요청을 느리게 처리하는 것보다, 일부를 빠르게 거부하고 나머지를 정상 처리하는 것이 낫다. 100% 느린 것보다 80% 정상 + 20% 거부가 사용자 경험이 좋다.

Load Shedding은 의도적 품질 저하(Graceful Degradation)의 일종이다.
전략
Priority-based: 중요 요청 우선 처리, 덜 중요한 것 먼저 거부
LIFO (Latest-In First-Out): 최신 요청 우선. 오래된 요청은 클라이언트가 이미 타임아웃했을 가능성이 높음
Admission Control: 서버가 자신의 현재 부하를 측정, 처리 가능 수준 초과 시 503 반환
Deadline Propagation: 요청에 데드라인(클라이언트 타임아웃)을 포함. 데드라인이 지난 요청은 처리하지 않음
HTTP 응답 코드
429 Too Many Requests: Rate limit 초과. Retry-After 헤더 포함 권장
503 Service Unavailable: 서버 과부하. 일시적. Retry-After 포함
클라이언트는 이 코드를 받으면 exponential backoff + jitter로 재시도해야 함
Essential

Timeout 전략

모든 외부 호출에 timeout을 설정하는 것은 분산 시스템의 첫 번째 규칙
Timeout이 없는 외부 호출은 시한폭탄이다. 하위 서비스가 무한히 느려지면 호출자의 리소스(스레드, 커넥션)가 영원히 점유된다.
Timeout 유형
Connection Timeout: TCP 연결 수립까지 대기 (보통 1~5초)
Request/Read Timeout: 요청 전송 후 응답까지 대기 (서비스 SLO의 P99 기반)
Idle Timeout: 유휴 연결 유지 시간 (keepalive와 연관)
Total/Overall Timeout: retry 포함 전체 소요 시간 상한
Timeout Chain (핵심!)
상위 서비스의 timeout > 하위 서비스의 timeout 이어야 함
Client → API Gateway → Service A → Service B 10s 8s 5s 3s
B의 timeout이 A보다 길면, A가 먼저 timeout하고 B는 불필요한 작업을 계속함
WAF → LB → Traefik → Pod 경로에서도 동일 원칙 적용
⚠ 실무 사례
NHN Cloud 환경에서 WAF → LB → Traefik → Pod 경로의 timeout/keepalive 체인을 정렬하는 작업을 한 경험이 있을 것. 이것이 바로 Timeout Chain 원칙의 실무 적용이다. 각 계층의 idle timeout이 외부에서 내부로 갈수록 짧아져야 silent connection drop을 방지한다.
Safety

Idempotency (멱등성)

같은 요청을 여러 번 보내도 결과가 동일한 성질. 안전한 재시도의 전제 조건
Partial Failure 환경에서 재시도는 필수다. 하지만 재시도가 안전하려면 연산이 멱등해야 한다. "잔액에 1000원 추가"는 재시도하면 이중 적립이지만, "잔액을 5000원으로 설정"은 몇 번을 해도 같은 결과.
HTTP 메서드 멱등성
GET: 멱등 ✅ (읽기만)
PUT: 멱등 ✅ (전체 교체)
DELETE: 멱등 ✅ (없으면 404)
POST: 멱등 ❌ (생성). Idempotency Key로 보완
PATCH: 설계에 따라 다름
Idempotency Key 구현
클라이언트가 요청마다 UUID를 Idempotency-Key 헤더로 전달
서버가 이 키를 저장(Redis 등)하고, 동일 키의 재요청에 저장된 응답을 반환
키의 TTL: 클라이언트 재시도 윈도우보다 길게 (보통 24~48시간)
Stripe API가 대표적 구현 예시
● Distributed Observability

분산 추적과 관측

Distributed Tracing, Context Propagation, Sampling — 마이크로서비스에서 "요청이 어디서 느려졌는가"에 답하기.

Core

Distributed Tracing 내부 동작

요청이 여러 서비스를 거치는 전체 경로를 추적하고 시각화하는 기술
마이크로서비스에서 하나의 사용자 요청은 수십 개의 서비스를 거친다. 로그만으로는 "이 요청이 어디서 느려졌는지"를 추적할 수 없다. Distributed Tracing은 요청의 전체 여정을 하나의 Trace로 연결하여 병목을 시각화한다.
핵심 개념
Trace: 하나의 요청에 대한 전체 기록. 고유한 Trace ID로 식별
Span: Trace 내의 개별 작업 단위. 시작/종료 시간, 서비스명, 상태 코드 포함
Parent-Child: Span 간 호출 관계. A→B 호출 시 A가 parent, B가 child
Span Context: Trace ID + Span ID + 샘플링 플래그. 서비스 간 전파되는 메타데이터
LGTM 스택에서의 구현
수집: OTel SDK가 앱에서 Span 생성 → Alloy로 전송
저장: Grafana Tempo (object storage 백엔드)
시각화: Grafana에서 Trace 뷰
연결: Metric → Exemplar → Trace ID → Trace 뷰 drill-down
Loki 로그에도 Trace ID를 삽입하면 Log↔Trace 연결 가능
Mechanism

Context Propagation

Trace 컨텍스트를 서비스 간 전달하는 메커니즘 — 추적이 끊기지 않게
서비스 A가 B를 호출할 때, Trace ID와 Span ID를 HTTP 헤더(또는 gRPC metadata)에 실어 전달해야 한다. 이것이 없으면 각 서비스의 Span이 독립적으로 존재하여 하나의 Trace로 연결되지 않는다.
전파 형식
W3C Trace Context (표준):
traceparent: 00-{trace-id}-{span-id}-{flags}
tracestate: vendor=value

B3 (Zipkin 호환):
X-B3-TraceId, X-B3-SpanId, X-B3-ParentSpanId

Jaeger: uber-trace-id
전파 누락 원인
1) HTTP 클라이언트가 헤더를 전달하지 않음 → OTel SDK 미설정
2) 프록시/LB가 헤더를 제거 → Traefik forwaredHeaders 설정
3) 비동기 처리(큐, 이벤트)에서 컨텍스트 분실 → 메시지에 trace context 포함 필수
4) 서비스 A는 W3C, B는 B3 → 형식 불일치. OTel Collector에서 변환
Strategy

Sampling 전략

모든 요청을 추적하면 비용이 폭발한다. 무엇을, 얼마나 샘플링할 것인가
1000 RPS 서비스의 모든 요청을 100% 추적하면 저장 비용과 네트워크 오버헤드가 어마어마하다. 대부분의 프로덕션 환경에서는 1~10% 샘플링을 적용하되, 중요한 trace는 놓치지 않는 전략이 필요.
샘플링 유형
Head-based (확률): 요청 시작 시점에 랜덤으로 추적 여부 결정. 단순하지만 에러/slow trace를 놓칠 수 있음
Tail-based (결과): 요청 완료 후에 결정. 에러/slow 요청만 저장. Collector에서 전체 trace를 버퍼링해야 하므로 리소스 부담
Rate-limiting: 초당 최대 N개 trace만 수집
Priority-based: 결제=100%, 목록 조회=1% 등 경로별 차등
실무 권장
기본: Head-based 10% + error/slow trace는 100%
OTel SDK의 ParentBased(TraceIDRatioBased(0.1))
Tail-based 구성: Alloy 또는 OTel Collector에서 tail_sampling processor
정책: latency > 1s 또는 status = ERROR → always sample

비용 최적화: Tempo의 search API는 Parquet 기반으로 저비용 장기 저장 가능. sampling rate를 낮추되 저장 기간은 길게
💡 Exemplar로 Metrics↔Traces 연결
Prometheus Exemplar에 Trace ID를 첨부하면 metric graph에서 특정 spike 지점을 클릭 → 해당 시점의 slow trace로 바로 이동할 수 있다. LGTM 스택의 핵심 연결 고리.
