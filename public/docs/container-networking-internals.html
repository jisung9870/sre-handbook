● Network Primitives

컨테이너 네트워크 기본 구성 요소

Network Namespace, veth pair, Bridge, CNI — 컨테이너 네트워킹의 레고 블록.

Isolation

Network Namespace

각 Pod에 독립적인 네트워크 스택(인터페이스, 라우팅 테이블, iptables)을 제공하는 커널 기능
Linux의 Network Namespace는 프로세스에게 격리된 네트워크 환경을 제공한다. 각 namespace는 자신만의 네트워크 인터페이스, IP 주소, 라우팅 테이블, iptables 규칙, 소켓을 가진다. K8s Pod의 모든 컨테이너는 같은 Network Namespace를 공유한다. 이것이 같은 Pod의 컨테이너끼리 localhost로 통신할 수 있는 이유다.
핵심 동작
Pod 생성 시:
1. containerd가 새 Network Namespace 생성
2. Pause 컨테이너가 namespace를 점유 (namespace holder)
3. CNI 플러그인이 veth pair 생성, IP 할당
4. 앱 컨테이너가 같은 namespace에 진입

확인:
lsns -t net — 시스템의 네트워크 namespace 목록
ip netns list — named namespace (Docker/K8s는 unnamed)
nsenter -t [PID] -n ip addr — Pod namespace에서 네트워크 확인
Pause 컨테이너의 역할
왜 Pause 컨테이너가 필요한가?

Network Namespace는 마지막 프로세스가 종료되면 삭제된다. 앱 컨테이너가 재시작되면 namespace가 사라질 수 있다.

Pause 컨테이너(무한 sleep)가 namespace를 영구히 점유하여, 앱 컨테이너가 재시작되어도 같은 IP, 같은 네트워크 설정을 유지한다.

pause: 리소스 ~1MB. 거의 CPU/Memory 사용 없음
Connection

veth pair — 가상 이더넷 쌍

두 개의 Network Namespace를 연결하는 가상 네트워크 케이블
veth pair는 한쪽에 들어온 패킷이 반드시 다른 쪽으로 나가는 가상 네트워크 인터페이스 쌍이다. Pod의 eth0와 노드의 vethXXXX가 하나의 veth pair다.
구조
┌─ Pod Namespace ─┐ ┌─ Host Namespace ─┐ │ │ │ │ │ eth0 ◄──────────┼────┼── vethXXXX │ │ 10.244.1.5 │ │ (bridge에 연결) │ │ │ │ │ └──────────────────┘ └───────────────────┘

Pod 내: eth0으로 보임. Pod IP가 할당됨
Host에서: vethXXXX로 보임. Bridge 또는 라우팅 테이블에 연결

ip link show type veth — 노드의 모든 veth 확인
ip -n [netns] link show — Pod namespace의 인터페이스
Pod의 veth 찾기
Pod의 eth0 → 호스트의 veth 매핑:

1. Pod에서: cat /sys/class/net/eth0/iflink → 인덱스 번호
2. 호스트에서: ip link | grep [인덱스번호]: → vethXXXX

또는:
kubectl exec -it pod -- ip link show eth0
→ eth0@if7에서 7이 호스트의 인터페이스 인덱스
→ 호스트에서 ip link | grep "^7:"로 veth 확인

디버깅에 필수: 특정 Pod의 트래픽을 호스트에서 tcpdump로 캡처할 때 해당 veth를 지정해야 함
Switch

Linux Bridge — 가상 L2 스위치

같은 노드의 Pod들을 연결하는 소프트웨어 스위치
Linux Bridge는 가상 L2 스위치다. 여러 veth의 호스트 측 끝을 Bridge에 연결하면, Bridge가 MAC 주소를 학습하여 같은 노드의 Pod 간 L2 통신을 중계한다.

CNI에 따라 Bridge 사용 여부가 다르다. Flannel과 기본 bridge CNI는 Bridge를 사용하지만, Cilium과 AWS VPC CNI는 Bridge 없이 직접 라우팅한다.
Bridge 기반 CNI 구조
Pod A (10.244.1.5) Pod B (10.244.1.6) eth0 eth0 │ │ vethA vethB │ │ └───── cbr0 (bridge) ──────┘ 10.244.1.1/24 │ host routing table │ eth0 (node IP)

brctl show 또는 bridge link show — Bridge 상태
cbr0, cni0 — CNI가 생성하는 Bridge 이름
Bridge 없는 CNI
Cilium: veth의 호스트 측에 tc BPF를 직접 연결. Bridge 경유 없이 eBPF가 라우팅 결정

AWS VPC CNI: Pod에 ENI의 secondary IP를 직접 할당. VPC 라우팅 테이블이 직접 Pod IP를 라우팅. Bridge 불필요

장점: Bridge의 MAC learning/flooding 오버헤드 제거. 더 빠른 패킷 처리
단점: L2 브로드캐스트가 없으므로 일부 레거시 프로토콜(ARP-heavy) 호환 문제 가능
Spec

CNI 스펙 — kubelet이 네트워킹을 설정하는 방법

Pod 생성 → kubelet → CRI → CNI 플러그인 호출 → 네트워크 설정 완료
CNI 호출 흐름
1) kubelet이 Pod 생성 요청 수신
2) CRI(containerd)가 Pod의 Network Namespace 생성
3) kubelet이 CNI 바이너리를 실행 (/opt/cni/bin/)
4) CNI 플러그인이:
• veth pair 생성
• IP 할당 (IPAM 플러그인 호출)
• 라우팅 규칙 설정
• Bridge 또는 eBPF 프로그램 연결
5) 결과(IP, 인터페이스)를 kubelet에 반환
6) kubelet이 Pod status에 IP 기록
CNI 설정 파일
위치: /etc/cni/net.d/

예시 (bridge CNI):
{ "cniVersion": "1.0.0", "name": "k8s-pod-network", "type": "bridge", "bridge": "cni0", "ipam": { "type": "host-local", "subnet": "10.244.1.0/24" } }

CNI 체이닝: 여러 CNI 플러그인을 순서대로 실행 가능. 예: AWS VPC CNI(IPAM) + Cilium(정책/라우팅)

IPAM 플러그인:
• host-local: 노드별 서브넷에서 IP 할당
• calico-ipam: Calico의 IP Pool에서 할당
• AWS VPC CNI: ENI secondary IP에서 할당
● Packet Path Tracing

패킷 경로 추적

Pod→Pod, Pod→Service, Pod→External — K8s에서 패킷이 실제로 흐르는 경로.

Same Node

Pod → Pod (같은 노드)

같은 노드의 두 Pod 간 통신 — Bridge 또는 BPF를 거쳐 직접 전달
Bridge CNI 경로
Pod A eth0 → vethA → cbr0 (bridge) → MAC learning으로 vethB 찾기 → vethB → Pod B eth0 경로: 패킷이 호스트 네트워크 스택을 통과 → iptables/netfilter도 통과 → conntrack에 기록됨
Cilium eBPF 경로
Pod A eth0 → vethA → [tc BPF ingress] → BPF가 destination Pod의 veth를 직접 조회 → bpf_redirect() → vethB → [tc BPF ingress] → Pod B eth0 경로: Bridge 없음, iptables 없음, netfilter 없음 → 커널 네트워크 스택 대부분을 우회 → latency 대폭 감소
Cross Node

Pod → Pod (다른 노드)

Overlay(VXLAN) 또는 Native Routing에 따라 다른 경로
VXLAN Overlay 경로
Node 1 Node 2 Pod A → vethA → bridge/BPF → bridge/BPF → vethB → Pod B → host routing: 10.244.2.0/24 via vxlan0 → VXLAN 캡슐화 (외부: Node1→Node2 UDP 8472) → [원본 패킷] [UDP 8472][VXLAN Header] [Node1 IP → Node2 IP] → Node 2의 vxlan0에서 디캡슐화 → 내부 라우팅 → Pod B

VXLAN 오버헤드: 50바이트 추가 헤더. MTU를 1450으로 조정 필요 (기본 1500 - 50)
Native Routing 경로
Node 1 Node 2 Pod A → vethA → host routing → 10.244.2.0/24 via Node2 IP → 원본 패킷 그대로 (캡슐화 없음) → 물리 네트워크가 Pod CIDR를 인식해야 함 → Node 2의 라우팅 → Pod B AWS VPC CNI: VPC 라우팅 테이블이 Pod IP 인식 Calico BGP: BGP로 Pod CIDR 경로를 전파

장점: 캡슐화 없음 → MTU 감소 없음, latency 낮음, 네트워크 장비에서 Pod IP 직접 접근 가능
제약: 물리 네트워크(또는 클라우드 VPC)가 Pod CIDR 라우팅을 지원해야 함
Service

Pod → Service

가상 ClusterIP가 실제 Pod IP로 변환(DNAT)되는 과정
iptables 기반 (kube-proxy)
Pod A → ClusterIP:port (예: 10.96.1.100:80) → OUTPUT chain → KUBE-SERVICES chain → KUBE-SVC-XXXX (Service에 해당하는 chain) → iptables probability로 Pod 선택 33% → KUBE-SEP-AAA (10.244.1.5:8080) 33% → KUBE-SEP-BBB (10.244.2.3:8080) 33% → KUBE-SEP-CCC (10.244.3.7:8080) → DNAT: dst 10.96.1.100 → 10.244.2.3 → conntrack에 매핑 기록 → 라우팅 → Pod B

Service 1000개 × Pod 10개 = ~10,000 iptables 규칙
매 패킷마다 순차 평가 → O(n) 성능
eBPF 기반 (Cilium)
Pod A → ClusterIP:port → tc BPF ingress on vethA → BPF Map에서 Service IP:port 조회 (O(1)) → endpoint 목록에서 LB 알고리즘으로 선택 → DNAT: dst → 선택된 Pod IP → bpf_redirect() → 대상 veth 또는 routing → Pod B

iptables 체인 전체를 우회. conntrack도 BPF 자체 구현.
Service 수와 무관한 O(1) 성능
External

Pod → External (인터넷)

Pod에서 외부로 나갈 때 SNAT가 적용되는 과정
SNAT 과정
Pod A (10.244.1.5) → google.com:443 → vethA → host routing → POSTROUTING chain → KUBE-POSTROUTING → MASQUERADE: src 10.244.1.5 → Node IP → 노드의 eth0 → NAT Gateway (또는 직접 인터넷) → google.com 응답: google.com → Node IP → conntrack에서 매핑 확인 → 역 DNAT: dst Node IP → 10.244.1.5 → routing → Pod A

핵심: Pod IP는 VPC 내부에서만 유효. 외부에서는 Node IP(또는 NAT GW IP)로 보임
예외: AWS VPC CNI에서 Pod IP가 VPC IP면 → SNAT 없이 직접 라우팅 가능 (VPC 내부 리소스 접근 시)
conntrack 테이블
SNAT/DNAT된 연결의 매핑을 기록:
conntrack -L — 전체 conntrack 테이블
conntrack -L -d 10.244.1.5 — 특정 Pod으로의 연결

conntrack 테이블 크기:
net.netfilter.nf_conntrack_max — 최대 엔트리 수
기본 ~26만. 대규모 클러스터에서 부족할 수 있음
소진 시 새 연결 drop → Random 503 에러의 원인

모니터링:
node_nf_conntrack_entries / node_nf_conntrack_entries_limit
80% 이상이면 알림
● Overlay vs Native Routing

네트워크 모드 비교

VXLAN, Native Routing, AWS VPC CNI — 노드 간 Pod 트래픽 전달 방식의 트레이드오프.

Overlay

VXLAN Overlay

UDP 캡슐화로 물리 네트워크 위에 가상 L2 네트워크를 구축
동작 원리
VXLAN(Virtual Extensible LAN): 원본 L2 프레임을 UDP 패킷 안에 캡슐화

구조:
[Outer Ethernet][Outer IP][UDP:8472][VXLAN Hdr][Original Frame]

VNI (VXLAN Network Identifier): 16M개의 가상 네트워크를 식별하는 24비트 ID

VTEP: VXLAN Tunnel EndPoint. 각 노드에서 캡슐화/디캡슐화를 수행하는 인터페이스 (flannel.1, cilium_vxlan)

FDB (Forwarding Database): Pod MAC → 대상 노드 IP 매핑. ARP 대신 FDB로 대상 VTEP를 결정
장단점
장점:
• 물리 네트워크 변경 없음. 어떤 환경에서든 동작
• Pod CIDR이 물리 네트워크와 독립
• 멀티테넌트, 멀티클러스터에서 네트워크 격리

단점:
• 50바이트 오버헤드 → MTU 1450 (1500 - 50). jumbo frame 시 조정 필요
• 캡슐화/디캡슐화 CPU 오버헤드
• 외부 네트워크에서 Pod IP로 직접 접근 불가
• 트러블슈팅 시 패킷 캡처가 어려움 (캡슐화된 패킷은 외부에서 내용을 볼 수 없음)

사용: Flannel(기본), Calico(옵션), Cilium(옵션)
Native

Native Routing (Direct Routing)

캡슐화 없이 물리 네트워크가 직접 Pod CIDR을 라우팅
BGP 기반 (Calico, Cilium)
BGP(Border Gateway Protocol)로 각 노드의 Pod CIDR을 네트워크 라우터에 광고:

Node 1: "10.244.1.0/24는 나(Node 1)에게 보내라"
Node 2: "10.244.2.0/24는 나(Node 2)에게 보내라"

물리 라우터 또는 ToR 스위치가 이 경로를 학습 → Pod IP를 직접 라우팅

적합: 온프레미스 환경에서 네트워크 팀과 협업 가능할 때
Calico: BGP가 기본 모드. 가장 성숙한 BGP 구현
Cilium: BGP Control Plane 지원 (1.12+)
클라우드 라우팅 (AWS VPC CNI)
AWS VPC CNI: BGP 대신 VPC 라우팅 테이블이 직접 Pod IP를 라우팅
각 Pod에 ENI의 secondary IP(VPC CIDR 내)를 할당 → VPC가 이미 라우팅을 알고 있음

장점:
• 캡슐화 없음 → MTU 1500 유지, latency 최소
• AWS 보안 그룹을 Pod에 직접 적용 가능
• VPC 내 모든 리소스(RDS, ELB)와 Pod이 직접 통신

단점:
• ENI 수와 secondary IP 수에 의한 Pod 수 제한
• VPC CIDR의 IP를 소비 → IP 부족 위험
• Pod IP가 VPC에 노출 → 보안 고려 필요
AWS

AWS VPC CNI 내부 동작

ENI + Secondary IP로 Pod에 VPC IP를 직접 할당하는 AWS 네이티브 CNI
IP 할당 과정
1) CNI가 EC2에 추가 ENI를 attach
2) 각 ENI에 secondary private IP를 여러 개 할당
3) Pod 생성 시 secondary IP 중 하나를 Pod에 배정
4) veth pair 생성, Pod namespace에 IP 설정
5) 호스트 라우팅 테이블에 Pod IP → veth 추가

Warm Pool: WARM_IP_TARGET만큼 IP를 미리 할당해두어 Pod 시작 시 IP 할당 지연 방지

Prefix Delegation: /28 prefix(16 IP)를 한 번에 할당하여 ENI당 Pod 수 대폭 증가
ENI 제한과 영향
인스턴스 타입별 ENI/IP 제한:
m5.large: 3 ENI × 10 IP = 29 Pod
m5.xlarge: 4 ENI × 15 IP = 58 Pod
m5.2xlarge: 4 ENI × 15 IP = 58 Pod

Prefix Delegation 시:
m5.xlarge: 4 ENI × 4 prefix × 16 IP = 250+ Pod

조회:
kubectl describe node | grep -A5 "Allocatable"
aws ec2 describe-instance-types --instance-types m5.xlarge --query "InstanceTypes[].NetworkInfo"
● Network Debugging

K8s 네트워크 디버깅 실전

tcpdump, conntrack, DNS — 패킷 레벨에서 네트워크 문제를 진단하는 실전 기법.

Capture

K8s에서 tcpdump 사용법

Pod의 네트워크 트래픽을 캡처하는 3가지 방법
방법 1: Pod 내부 (exec)
kubectl exec -it pod-name -- tcpdump -i eth0 -nn port 8080

제약: Pod 이미지에 tcpdump가 있어야 함. Distroless 이미지에서는 불가

해결: netshoot 디버그 컨테이너 사용:
kubectl debug -it pod-name --image=nicolaka/netshoot --target=app-container -- tcpdump -i eth0 -nn port 8080

ephemeral container가 같은 network namespace를 공유하므로 eth0 캡처 가능
방법 2: 노드에서 veth 캡처
1) Pod의 veth 찾기 (위의 veth pair 섹션 참조)
2) 노드에서 해당 veth 캡처:
tcpdump -i vethXXXX -nn -w /tmp/capture.pcap

장점: Pod 이미지와 무관. 모든 Pod에 적용 가능
단점: 노드 접근 필요 (kubectl debug node 또는 SSH)

유용한 필터:
tcpdump -i vethXXXX -nn 'tcp port 5432' — DB 트래픽만
tcpdump -i vethXXXX -nn 'tcp[tcpflags] & (tcp-rst) != 0' — RST 패킷만 (연결 강제 종료)
Conntrack

conntrack 디버깅

DNAT/SNAT 매핑 확인, conntrack 테이블 소진 문제 진단
핵심 명령
conntrack -L — 전체 conntrack 테이블 출력
conntrack -L -s 10.244.1.5 — 특정 소스 IP 필터
conntrack -L -d 10.96.1.100 — Service ClusterIP로의 연결
conntrack -L -p tcp --dport 5432 — DB 커넥션 추적
conntrack -C — 현재 엔트리 수
conntrack -E — 실시간 이벤트 모니터링 (new/update/destroy)

DNAT 확인:
conntrack -L -d 10.96.1.100 | head
→ dst=10.96.1.100 → dst=10.244.2.3 매핑 확인
conntrack 테이블 소진 문제
증상: 간헐적 연결 실패, 503 에러, DNS 타임아웃
확인: dmesg | grep conntrack → "table full, dropping packet"

해결:
sysctl -w net.netfilter.nf_conntrack_max=524288
또는 /etc/sysctl.d/에 영구 설정

Cilium eBPF 모드:
netfilter conntrack을 사용하지 않으므로 이 문제가 근본적으로 발생하지 않음. BPF conntrack Map이 대체

모니터링:
node_nf_conntrack_entries / node_nf_conntrack_entries_limit
알림: 사용률 80% 초과 시
DNS

K8s DNS 문제 진단

ndots:5 문제, DNS 타임아웃, NXDOMAIN — K8s 환경의 DNS 특수 이슈
ndots:5 문제
K8s Pod의 /etc/resolv.conf:
nameserver 10.96.0.10
search production.svc.cluster.local svc.cluster.local cluster.local
options ndots:5

ndots:5의 의미: 도메인에 점(.)이 5개 미만이면 search 도메인을 먼저 시도
api.external.com (점 2개) → 먼저 api.external.com.production.svc.cluster.local 시도 → NXDOMAIN → api.external.com.svc.cluster.local → NXDOMAIN → 원본 시도

결과: 외부 도메인 조회 시 최대 4~5회 불필요한 DNS 쿼리가 먼저 발생. latency 추가

해결:
• FQDN에 trailing dot: api.external.com. (search 건너뜀)
• dnsConfig.options: [{name: ndots, value: "2"}] (Pod spec에서)
• CoreDNS autopath 플러그인 (서버 측에서 해결)
DNS 타임아웃 진단
Pod에서 테스트:
kubectl exec -it pod -- nslookup kubernetes.default
kubectl exec -it pod -- dig @10.96.0.10 api-server.production.svc.cluster.local

CoreDNS Pod 상태:
kubectl -n kube-system get pods -l k8s-app=kube-dns
kubectl -n kube-system logs -l k8s-app=kube-dns

흔한 원인:
• CoreDNS Pod이 Pending/CrashLoopBackOff
• CoreDNS에 HPA가 없어 부하 초과
• conntrack 테이블 소진 → DNS UDP 패킷 drop
• NetworkPolicy가 DNS egress를 차단 (kube-dns:53 허용 필수!)
• 노드의 /etc/resolv.conf 상속 문제
💡 네트워크 디버깅 순서
1) kubectl exec -- curl/wget → 앱 레벨 연결 테스트
2) kubectl exec -- nslookup → DNS 해석 확인
3) kubectl debug --image=netshoot -- tcpdump → 패킷 레벨 확인
4) 노드에서 conntrack -L → DNAT/SNAT 매핑 확인
5) iptables -t nat -L -n -v → kube-proxy 규칙 확인 (또는 cilium bpf ct list global)
6) ip route → 라우팅 테이블 확인
