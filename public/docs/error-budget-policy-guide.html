Error Budget은 숫자가 아니라 제품 속도와 안정성의 계약이다. 목표는 "배포 금지"가 아니라 "위험을 통제한 진전"이다.

● Strategy

Error Budget Policy의 목적

SLO를 숫자로만 두지 않고, 실제 배포·릴리즈·우선순위 조정에 연결하기 위한 운영 규칙이다. 핵심은 "장애를 벌주는 것"이 아니라 "리스크를 명시적으로 거래하는 것"이다.

Why

왜 Error Budget Policy가 필요한가

SLO 위반 여부를 실제 행동으로 연결하지 않으면 SLO는 리포트로만 남는다
많은 팀이 SLO 대시보드는 만들지만, Budget이 줄어들 때 무엇을 멈추고 무엇을 계속할지는 정하지 않는다. 그러면 숫자는 보이지만 의사결정은 사람 감에 의존하게 된다. Error Budget Policy는 이 모호성을 제거해 신규 기능, 리팩터링, 인프라 변경, 긴급 수정을 어떤 조건에서 허용할지 미리 정하는 장치다.
Policy가 없을 때
장애 직후에도 기능 배포를 강행하거나, 반대로 작은 문제에도 무기한 배포 동결이 발생한다.
결과: 갈등, 책임 공방, 회고 없는 반복.
Policy가 있을 때
"Budget 50% 초과 소진 시 고위험 변경 금지"처럼 사전에 합의된 룰로 빠르게 판단한다.
결과: 속도와 안정성의 균형이 예측 가능해진다.
핵심 목적
1. 배포 결정의 기준 통일
2. SLO 위반을 제품 우선순위에 반영
3. 장애 후 "더 조심하자" 같은 감정적 반응 대신 명시적 행동으로 전환
4. 속도와 안정성의 논쟁을 사람 간 논쟁이 아닌 정책 집행 문제로 바꾸기
Intent

Error Budget의 운영 의미

Budget은 "써도 되는 실패의 총량"이지 "절대 다 쓰면 안 되는 숫자"가 아니다
Error Budget은 고객에게 약속한 신뢰성 목표 안에서 변경과 학습을 허용하는 완충재다. 예산을 일부 소진하는 것은 자연스럽다. 중요한 것은 소진 속도와 남은 여유를 보고 리스크를 조정하는 것이다. Budget 0% 소진만을 목표로 하면 팀은 배포를 피하게 되고, 결국 기술 부채가 쌓여 더 큰 장애로 돌아온다.
▼ Error Budget을 해석하는 3가지 질문
질문좋은 해석나쁜 해석
지금 Budget을 얼마나 썼는가?현재 안정성 여유를 본다"누가 얼마나 망쳤나"를 따진다
소진 속도가 어떤가?최근 변경의 위험도를 판단한다한 번의 숫자만 보고 과잉반응한다
남은 Budget으로 무엇을 허용할 수 있나?릴리즈 전략을 조정한다무조건 배포를 멈추거나 무조건 진행한다
💡 좋은 정책의 기준
좋은 Error Budget Policy는 팀이 멈춰야 할 때를 알려주는 규칙이면서 동시에 언제 다시 움직여도 되는지를 알려주는 규칙이어야 한다. 정지 규칙만 있고 재개 규칙이 없으면 정책은 공포가 된다.
Principles

정책 설계 원칙

실무에서 정책이 살아남기 위한 최소 원칙

정책은 단순해야 한다

Guardrail 단계는 3~4단계면 충분하다. 복잡한 예외와 수식이 많아질수록 현장에서 사용되지 않는다.

사전 합의가 핵심이다

장애 이후에 임시로 만든 규칙은 신뢰를 잃기 쉽다. 제품/개발/운영 리더가 미리 서명한 정책이어야 한다.

행동으로 연결되어야 한다

"주의"가 아니라 "고위험 릴리즈 중단", "핫픽스만 허용", "주간 리뷰 강제"처럼 실행 가능한 문장으로 써야 한다.

회복 경로가 있어야 한다

Budget을 회복하면 어떤 조건에서 정상 배포 모드로 돌아가는지 반드시 정의해야 한다.

● Policy

정책 규칙과 Guardrail

정책은 "Budget을 얼마나 썼는가"와 "얼마나 빨리 태우고 있는가"를 함께 보고 행동을 결정해야 한다. 단순한 잔량만 보지 말고 소진 속도도 같이 판단한다.

Guardrails

권장 Guardrail 단계

Budget 상태에 따라 릴리즈와 변경 정책을 4단계로 구분
단계Budget 상태허용되는 변경추가 조치
Green소진율 < 25%, burn 안정정상 릴리즈, 실험, 리팩터링 허용주간 리뷰만 수행
Watch소진율 25~50% 또는 burn 증가고위험 변경 사전 리뷰 후 허용다음 배포 전에 rollback plan 검토
Control소진율 50~100% 또는 높은 burn핫픽스, 안정화 작업, low-risk 변경만 허용기능 개발보다 안정화 우선
FreezeBudget 초과 또는 다중 SLO 위반고객 영향 복구와 안전성 개선만 허용정책 예외 없이는 기능 배포 중단
소진율만 보지 말고 burn rate도 함께 사용한다. 예: 30일 Budget 기준 2시간 안에 10% 이상 태우는 패턴은 잔량이 남아 있어도 즉시 Control로 격상할 수 있다.
Release

배포와 변경에 대한 운영 규칙

정책이 실제 제품 속도에 미치는 방식을 명시적으로 적는다
릴리즈 정책은 "배포 가능/불가"의 이분법보다 변경 종류에 따라 다르게 허용하는 것이 현실적이다. DB schema, 네트워크 정책, 캐시 invalidate, 대규모 리팩터링은 위험도가 다르기 때문이다.
Green / Watch
허용: 신규 기능, 성능 개선, 리팩터링, 점진 배포
조건: canary/rollback/owner 명시
Watch 추가: 주요 변경은 peer review + 운영자 동의
Control / Freeze
허용: 핫픽스, 알림 개선, 용량 증설, 되돌리기 쉬운 안전 변경
금지: 대규모 마이그레이션, 실험성 기능, 다중 시스템 동시 변경
Freeze: 제품 기능 릴리즈는 VP/Head 예외 승인 없이는 금지
변경 위험도 분류 예시
Low-risk: 문구 수정, feature flag off, read replica 증설, 대시보드 추가
Medium-risk: 애플리케이션 배포, 라이브러리 업그레이드, 인스턴스 타입 조정
High-risk: DB schema 변경, 캐시 키 전략 변경, auth 흐름 변경, traffic routing 변경
⚠ 흔한 오해
Budget이 남아 있다고 해서 모든 고위험 변경이 허용되는 것은 아니다. Error Budget Policy는 위험 허용량을 보여주는 도구일 뿐이고, 개별 변경의 blast radius를 대신 판단해주지는 않는다.
Exceptions

예외 승인 정책

Freeze나 Control 상태에서도 꼭 진행해야 하는 변경을 위한 절차
실무에서는 법적 대응, 보안 패치, 대형 고객 계약, 인프라 수명 만료 같은 이유로 정책 예외가 필요하다. 중요한 것은 예외를 없애는 것이 아니라, 누가 어떤 위험을 인지한 채 승인했는지를 남기는 것이다.
Request변경 제안
Assess영향/대안
Approve책임자 서명
Mitigate완화장치 준비
Review결과 회고
예외 요청 시 필수 항목
목적: 왜 지금 해야 하는가
대안: 미룰 수 없는 이유는 무엇인가
리스크: 최악의 영향은 무엇인가
완화책: canary, kill switch, rollback, 담당자 대기 여부
승인 기준
보안/법적 필수: 높은 우선순위로 허용 가능
매출/고객 영향: 리스크와 사업가치를 함께 검토
순수 기능 출시: Freeze 상태에서는 기본적으로 불허
승인자: 서비스 오너 + 운영 책임자 + 필요 시 제품 리더
● Operations

일상 운영에서의 집행 방식

Error Budget Policy는 회의 문서가 아니라 운영 루틴이어야 한다. 측정, 알림, 리뷰, 배포 절차 안에 자연스럽게 들어가야 살아남는다.

Measurement

측정과 계산 원칙

정책 집행의 전제는 숫자에 대한 해석이 팀 내에서 동일해야 한다는 점이다
Error Budget Policy가 깨지는 가장 흔한 이유는 계산 자체보다 무엇을 bad event로 볼지, 어떤 윈도우를 쓸지, 어떤 서비스 단위로 볼지가 팀마다 다르기 때문이다. SLI Specification과 Policy는 반드시 같이 움직여야 한다.

Burn Rate

Burn Rate Alert와 정책 연계

알림은 "문제가 생겼다"를 알려주고, 정책은 "그래서 무엇을 할지"를 정한다
Burn rate alert는 단순 장애 탐지를 넘어서 정책 상태 전환 신호로 쓰는 것이 좋다. 예를 들어 빠른 burn alert가 울리면 즉시 신규 기능 배포를 멈추고 안정화 모드로 전환할 수 있다.
▼ 예시 연계 규칙
신호의미정책 반응
Fast burn alert짧은 시간에 Budget을 빠르게 소진즉시 Control 이상으로 격상, 현재 배포 중지
Slow burn alert눈에 띄지 않지만 지속적으로 새고 있음다음 주기까지 원인 파악과 개선 계획 필수
Repeated burn비슷한 유형의 반복 장애기능보다 구조 개선 우선, Freeze 가능
No alert but high spend큰 장애는 없지만 꾸준히 나쁨정책 리뷰 기준 재조정, SLI 정의 점검
⚠ 알림과 정책을 분리하면 생기는 문제
Burn rate alert는 울리는데 아무 행동도 바뀌지 않으면 팀은 곧 알림을 무시하게 된다. 알림은 반드시 배포 규칙, 에스컬레이션, 리뷰 강도와 연결되어야 한다.
Review

주간 운영 루틴

정책은 인시던트 때만 보는 문서가 아니라 주간 운영 리듬 안에 들어가야 한다

월요일: 상태 확인

현재 Guardrail 단계, 남은 Budget, 지난주 주요 소진 이벤트, 이번 주 예정된 고위험 변경을 함께 본다.

수요일: 변경 점검

릴리즈 전 high-risk 변경에 대해 canary, rollback, owner-on-call 준비 여부를 확인한다.

금요일: 회복 평가

Control/Freeze 상태였다면 정상 모드로 복귀 가능한지 판단하고, 다음 주 목표를 조정한다.

월말: 정책 재조정

Guardrail 기준이 너무 빡빡하거나 느슨하지 않은지, 팀이 실제로 따를 수 있는지 평가한다.

● Governance

역할, 책임, 그리고 회고

정책은 문장보다 운영 주체가 중요하다. 누가 상태를 선언하고, 누가 예외를 승인하고, 누가 정책이 현실과 맞는지 주기적으로 고칠지 명확해야 한다.

Ownership

역할과 책임 분리

정책 집행이 개인의 눈치가 되지 않도록 책임 주체를 분리한다
서비스 오너
현재 Guardrail 상태를 팀에 공유하고, 배포 지속 여부를 결정한다.
책임: 기능 우선순위 조정, 제품 커뮤니케이션.
운영 책임자 / SRE
Budget 계산, burn alert 품질, 예외 승인 시 완화장치 검토를 담당한다.
책임: 정책의 기술적 집행 가능성 보장.
제품 리더
Freeze 상태에서 매출/고객 영향 때문에 예외가 필요한 경우 사업적 근거를 제시한다.
엔지니어링 리더
장애 후 안정화 작업과 기능 작업 사이의 자원 배분을 최종 조정한다.
💡 역할 설계 팁
정책은 운영팀만의 문서가 되면 안 된다. 제품 리더도 서명하는 운영 규칙이어야 실제 배포 우선순위가 바뀐다.
Aftermath

장애 이후 정책 처리

큰 장애가 난 뒤에는 숫자 복구보다 운영 모드 전환이 더 중요하다
대형 장애 직후에는 모두가 "당분간 배포하지 말자"로 기울기 쉽다. 하지만 Freeze는 임시 상태여야 하며, 어떤 조건을 만족하면 Watch 또는 Green으로 되돌아갈지가 명확해야 한다.
장애 직후 24시간
SLO 영향 계산, 현재 Guardrail 단계 선언, 추가 배포 중지 여부 공지, 임시 완화장치 적용.
48~72시간
Postmortem 초안, 재발 방지 항목 선정, Freeze 해제 조건 합의.
Freeze 해제 예시 조건
1. 근본 원인 또는 우회 원인 파악됨
2. rollback/kill switch 경로 검증됨
3. 관련 알림/대시보드 보강 완료
4. 첫 1~2회 배포는 canary와 owner-on-call 조건으로 진행
Anti-patterns

Error Budget Policy의 대표적인 실패 패턴

정책이 살아남지 못하는 이유는 대부분 운영 현실과의 단절이다

징벌 도구로 쓰기

"이번 분기 Budget 다 썼으니 기능팀 잘못"처럼 쓰면 팀은 SLO 숫자를 숨기기 시작한다.

항상 Freeze 유지

해제 조건이 없으면 정책은 영구 비상상태가 된다. 결국 아무도 정책을 신뢰하지 않는다.

서비스 티어 무시

내부 도구와 결제 시스템에 같은 Guardrail을 적용하면 둘 다 망가진다. 티어별 차등이 필요하다.

정책과 배포 시스템 분리
