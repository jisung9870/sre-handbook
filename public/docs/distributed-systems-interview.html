● CAP Theorem

CAP 정리

Consistency, Availability, Partition Tolerance — 분산 시스템의 근본 트레이드오프.

CAP 정리를 설명하고, 왜 3개를 동시에 만족할 수 없는지 알려주세요.
CAP 정리는 분산 시스템에서 Consistency(모든 노드가 같은 데이터), Availability(모든 요청에 응답), Partition Tolerance(네트워크 분할에도 동작) 3가지를 동시에 만족할 수 없다는 정리입니다. 네트워크 파티션은 현실에서 반드시 발생하므로 P는 필수이고, C와 A 중 하나를 선택해야 합니다.
왜 동시에 불가능한가 (직관적 설명):
Node A와 Node B 사이 네트워크가 끊겼을 때(Partition):
• C를 선택하면: A와 B의 데이터가 동기화될 수 없으므로, 요청을 거부(에러 반환). → A 포기
• A를 선택하면: A와 B가 각자 응답하지만 데이터가 다를 수 있음. → C 포기
• P를 포기하면: 네트워크 파티션이 없다고 가정. → 현실적이지 않음

PACELC 확장: 파티션이 없을 때에도 Latency와 Consistency의 트레이드오프가 존재. "네트워크가 정상이어도 강한 일관성을 원하면 latency가 증가한다."
실제 시스템별 CAP 선택을 설명해 주세요.
etcd/ZooKeeper는 CP(일관성 우선 — 과반수 미달 시 쓰기 거부), Cassandra/DynamoDB는 AP(가용성 우선 — 파티션 시에도 응답, 나중에 동기화), MongoDB는 CP(primary 선출 실패 시 쓰기 불가), Kafka는 설정에 따라 CP 또는 AP(acks=all이면 CP, acks=1이면 AP 쪽)입니다.
시스템CAP파티션 시 동작실무 경험
etcdCP과반수 미달 → 쓰기 거부, 읽기만 가능K8s Control Plane. 3노드 중 2개 다운 → 클러스터 쓰기 불가
MongoDBCPprimary 선출 실패 → 쓰기 불가 (10~30초)ReplicaSet 과반수 투표 실패 시 장애 경험과 연결
Redis SentinelCPsentinel 과반수로 failover 결정failover 중 ~30초 쓰기 불가
Kafka설정 가변acks=all → CP (ISR 대기), acks=1 → AP 쪽Consumer Lag 폭증 시 acks 설정이 원인일 수 있음
CassandraAP파티션 시에도 응답. 나중에 repaireventual consistency + hints handoff

⚠ 면접 팁: "MongoDB 과반수 투표 실패" 경험
"MongoDB 3노드 ReplicaSet에서 2노드가 동시에 다운되어 과반수(2/3)를 충족하지 못해 primary 선출이 실패하고, 쓰기가 30초간 불가능했던 장애를 경험했습니다. 이것이 CP 시스템의 특성입니다. 가용성(A)을 포기하고 데이터 일관성(C)을 보존한 것이죠."
● Raft Consensus

Raft 합의 알고리즘

Leader Election, Log Replication — etcd/K8s의 Control Plane이 동작하는 원리.

Raft 알고리즘의 핵심 동작을 설명해 주세요.
Raft는 리더 기반 합의 알고리즘으로, 하나의 리더가 모든 쓰기를 받아 로그에 기록하고, 과반수 follower에 복제가 확인되면 커밋합니다. 리더가 죽으면 나머지 노드가 타임아웃 후 선거를 시작하여 새 리더를 선출합니다. etcd가 Raft를 사용하므로 K8s의 모든 상태 변경이 Raft로 합의됩니다.
3가지 역할: Leader(쓰기 처리, 복제), Follower(리더 복제 수신), Candidate(선거 후보)

Leader Election:
1. 모든 노드가 Follower로 시작 2. Leader의 heartbeat(AppendEntries RPC)를 기다림 3. election timeout(150~300ms 랜덤) 내 heartbeat 안 오면 → Candidate로 전환, term 증가, 자기 자신에 투표 4. 과반수 투표 받으면 → Leader 당선 5. 즉시 모든 Follower에 heartbeat 전송 핵심: election timeout을 랜덤화하여 동시 후보 난립(split vote) 방지
Log Replication:
1. Client → Leader: "key=foo, value=bar" 쓰기 요청 2. Leader: 자신의 로그에 추가 (uncommitted) 3. Leader → Followers: AppendEntries RPC로 로그 전파 4. Follower: 로그에 추가 후 ACK 반환 5. Leader: 과반수 ACK 확인 → 로그를 committed로 표시 6. Leader → Client: 쓰기 성공 응답 7. 다음 heartbeat에서 Followers에 commit index 전달
etcd 3노드의 의미: 3노드 → 과반수 = 2. 1노드 장애까지 허용. 5노드 → 과반수 = 3. 2노드 장애까지 허용. 짝수(4노드)는 3노드와 장애 허용 수가 같으므로(과반수 = 3) 비효율적. 항상 홀수를 권장.

K8s에서: kubectl apply → API Server → etcd Raft 합의 → 과반수 확인 → 저장 완료 → 응답. etcd가 느리면 모든 K8s 작업이 느려짐. etcd_disk_wal_fsync_duration_seconds P99 > 10ms면 SSD 성능 문제.
● Consistency Models

Consistency 모델

Strong, Eventual, Causal — 데이터 일관성 수준별 트레이드오프.

Strong Consistency와 Eventual Consistency의 차이를 설명하고, 각각 실무 예시를 들어주세요.
Strong Consistency는 쓰기가 완료되면 모든 후속 읽기에서 최신 값을 보장합니다(예: 은행 계좌 잔액). Eventual Consistency는 일정 시간 후 모든 복제본이 동일해지지만, 그 사이에 오래된 값을 읽을 수 있습니다(예: SNS 좋아요 수). Strong은 latency가 높고, Eventual은 가용성이 높습니다.
모델보장latency실무 예시
Strong읽기 = 최신 값 항상높음 (합의 대기)etcd, PostgreSQL(sync replication), ZooKeeper
Eventual"언젠가" 동기화낮음S3, DynamoDB(기본), DNS, Redis replica
Causal인과 관계 보존중간MongoDB(causal sessions), 채팅 메시지 순서
Read-your-writes내가 쓴 것은 즉시 읽음중간세션 기반 시스템, 사용자 프로필 수정

실무 패턴 — Read-after-Write 문제:
사용자가 프로필을 수정(쓰기 → primary) → 즉시 페이지 새로고침(읽기 → replica) → 아직 복제 안 됨 → 이전 데이터가 보임 → "수정 안 됐는데?!"
해결: 쓰기 직후 읽기를 primary로 라우팅(read-your-writes). 또는 쓰기 후 짧은 지연 후 표시.

K8s에서: etcd는 Strong Consistency. 하지만 kubectl이 API Server 캐시를 읽을 때는 약간의 지연이 있을 수 있음. --watch-cache=false로 캐시를 우회하면 항상 etcd에서 직접 읽지만 부하 증가.
● Failure Patterns

분산 시스템 장애 패턴

Split Brain, Cascading Failure, Thundering Herd — 실전 장애와 연결.

Split Brain이란 무엇이고, 어떻게 방지하나요?
네트워크 파티션으로 클러스터가 두 그룹으로 나뉘어 각각 자신이 리더라고 판단하는 상태입니다. 양쪽이 독립적으로 쓰기를 수락하면 데이터 불일치가 발생합니다. 과반수 투표(quorum)로 방지합니다: 3노드 중 과반수(2) 이상을 가진 그룹만 리더를 선출할 수 있고, 소수 그룹은 읽기만 가능합니다.
3노드 클러스터, 네트워크 파티션 발생: [Node A] ──── [Node B] | [Node C] 그룹 1 (2노드) | 그룹 2 (1노드) 과반수 확보 ✅ | 과반수 미달 ❌ → 리더 선출 가능 | → 리더 선출 불가 → 정상 읽기/쓰기 | → 읽기만 (또는 거부)
시스템별 Split Brain 방지:
• etcd (Raft): 과반수 미달 그룹의 리더가 자동 step down. term이 높은 리더만 유효
• Redis Sentinel: sentinel 과반수(quorum)가 동의해야 failover 실행. min-slaves-to-write로 고립된 primary의 쓰기 차단
• MongoDB: election에 과반수 투표 필요. arbiter 노드로 짝수 방지

Kafka Consumer Lag 폭증 장애 연결:
"Kafka Consumer Lag이 갑자기 폭증한 경험이 있습니다. 원인은 Consumer 그룹의 rebalancing이 네트워크 일시 불안정으로 반복 트리거되면서, 파티션 할당이 안정되지 못해 소비가 중단된 것이었습니다. session.timeout.ms와 heartbeat.interval.ms를 조정하여 해결했습니다."
Cascading Failure는 어떻게 발생하고 방지하나요?
하나의 서비스 장애가 의존하는 다른 서비스로 연쇄적으로 전파되는 현상입니다. 서비스 A가 느려지면 → B가 A를 기다리며 커넥션 풀 소진 → B도 느려짐 → C도 → 전체 다운. 방지: Timeout(빠르게 실패), Circuit Breaker(호출 자체를 차단), Retry Budget(재시도 비율 제한), Load Shedding(과부하 시 거부)으로 양성 피드백 루프를 끊어야 합니다.
양성 피드백 루프:
부하 증가 → latency 증가 → 동시 요청 증가(느린 요청이 쌓임) → 리소스 소비 증가 → 더 많은 latency → 더 많은 동시 요청 → ... 폭주

4가지 차단기:
• Timeout: 10초 이상 기다리지 않고 빠르게 실패 → 리소스 해방
• Circuit Breaker: 에러율 50% 초과 시 호출 자체를 차단 → 장애 서비스에 추가 부하 방지
• Retry Budget: 전체 요청의 10%까지만 재시도 → Retry Storm 방지
• Load Shedding: 동시 처리 수 초과 시 503 반환 → 처리 가능한 요청만 정상 처리

K8s에서: Istio DestinationRule의 outlierDetection(Pod 레벨 CB) + 앱 코드의 gobreaker(의존성 레벨 CB) 조합으로 다층 방어.
● Defense Patterns

멱등성 & Circuit Breaker

분산 환경에서 안전하게 재시도하고, 장애를 격리하는 패턴.

멱등성(Idempotency)이란 무엇이고, 왜 분산 시스템에서 중요한가요?
같은 요청을 여러 번 실행해도 결과가 한 번 실행한 것과 같은 성질입니다. 분산 시스템에서 네트워크 타임아웃 시 "요청이 처리되었는지 불확실"한 상황이 빈번하므로, 안전하게 재시도하려면 연산이 멱등해야 합니다. 예: "잔액을 100으로 설정"은 멱등하지만, "잔액에 100을 더하기"는 멱등하지 않습니다.
HTTP 메서드별 멱등성:
메서드멱등?안전?설명
GET✅✅여러 번 호출해도 서버 상태 불변
PUT✅❌"이 값으로 설정" — 여러 번 해도 같은 결과
DELETE✅❌이미 삭제된 것을 다시 삭제해도 OK
POST❌❌"주문 생성" — 여러 번 하면 여러 주문
PATCH❌ (보통)❌"잔액 +100" — 여러 번 하면 여러 번 증가

비멱등 연산을 멱등으로 만드는 법:
Idempotency Key 패턴: 클라이언트가 요청에 고유 ID(UUID)를 포함. 서버는 이 ID를 기록하고, 같은 ID가 오면 이전 결과를 반환.
POST /api/payments + Idempotency-Key: uuid-123
→ 첫 번째: 결제 처리 + uuid-123 기록
→ 재시도: uuid-123 이미 존재 → 이전 결과 반환 (중복 결제 방지)

Kafka exactly-once: Producer가 enable.idempotence=true + transactional.id로 중복 메시지 방지. Broker가 sequence number로 중복 감지.
Circuit Breaker 패턴의 동작을 설명하고, K8s에서 어떻게 구현하나요?
Circuit Breaker는 Closed(정상 통과) → Open(즉시 실패 반환) → Half-Open(탐색적 1개 요청) 3가지 상태로 동작합니다. 연속 실패가 임계값을 초과하면 Open으로 전환하여 장애 서비스 호출을 차단합니다. K8s에서는 Istio DestinationRule의 outlierDetection(Pod 레벨 ejection)과 앱 코드의 gobreaker(서비스 레벨)를 조합합니다.
Closed (정상) → 연속 5회 실패 또는 에러율 > 50% Open (차단 — 모든 호출 즉시 에러 반환) → 30초 대기 Half-Open (탐색 — 1개 요청만 통과) → 성공 → Closed 복귀 → 실패 → Open 재진입
Go 구현 (gobreaker):
cb := gobreaker.NewCircuitBreaker(gobreaker.Settings{
  MaxRequests: 1,
  Timeout: 30 * time.Second,
  ReadyToTrip: func(counts gobreaker.Counts) bool {
    return counts.ConsecutiveFailures > 5
  },
})
result, err := cb.Execute(func() (interface{}, error) {
  return httpClient.Do(req)
})

Istio (Mesh 레벨): outlierDetection으로 5xx 연속 5회인 Pod을 30초간 트래픽에서 제외(ejection). 앱 코드 변경 없이 YAML만으로 CB 적용.
