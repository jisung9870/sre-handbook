● B-Tree Index

B-Tree / B+Tree 인덱스

DB가 수억 행에서도 3~4번의 디스크 I/O로 데이터를 찾는 원리.

B-Tree와 B+Tree의 차이를 설명하고, 왜 DB가 B+Tree를 사용하는지 알려주세요.
B-Tree는 모든 노드에 키+데이터를 저장하고, B+Tree는 리프 노드에만 데이터를 저장하고 내부 노드는 키만 가집니다. B+Tree가 DB에 적합한 이유는: 내부 노드에 더 많은 키를 저장하여 트리 높이가 낮아지고(디스크 I/O 감소), 리프 노드끼리 링크드 리스트로 연결되어 범위 검색(range scan)이 효율적이기 때문입니다.
B+Tree 구조 (PostgreSQL 인덱스): [10 | 20 | 30] ← 내부 노드 (키만, 데이터 없음) / | \ [1,3,7] [12,15,18] [22,25,28] ← 리프 노드 (키+데이터 포인터) ↔ ↔ ↔ ← 리프 간 링크드 리스트 (range scan용) 높이 3~4: 수천만 행도 3~4번 디스크 읽기로 검색 노드 크기 = 페이지 크기 (8KB in PostgreSQL) 한 노드에 수백 개 키 → 높이가 매우 낮음
시간 복잡도: 검색/삽입/삭제 모두 O(log N). 하지만 실제 성능은 트리 높이(= 디스크 I/O 횟수)가 결정. 1억 행 → 높이 ~3~4 → 3~4번의 8KB 페이지 읽기로 완료.

인덱스 bloat 문제: PostgreSQL에서 DELETE/UPDATE가 빈번하면 dead tuple이 쌓여 인덱스가 비대화됨. 인덱스 크기가 커지면 캐시 효율이 떨어져 I/O 증가.
해결: REINDEX 또는 pg_repack (online rebuild, 락 없음)

실무 경험: shared_buffers를 128MB→4GB로 튜닝하여 캐시 히트율을 78%→95%로 개선한 경험과 연결하면 — B+Tree 인덱스의 내부 노드가 shared_buffers에 캐시되면 디스크 I/O 없이 메모리에서 검색. 캐시가 작으면 매번 디스크 읽기.
Hash 인덱스는 왜 B-Tree만큼 많이 안 쓰이나요?
Hash 인덱스는 등호(=) 검색에 O(1)이지만 범위 검색(BETWEEN, ORDER BY, LIKE 'prefix%')을 지원하지 않습니다. B-Tree는 O(log N)이지만 등호, 범위, 정렬, 부등호를 모두 지원합니다. 대부분의 쿼리가 범위 검색을 포함하므로 B-Tree가 범용적으로 더 유리합니다.
비교B-TreeHash
등호 (=)O(log N)O(1)
범위 (BETWEEN)✅ O(log N + M)❌
정렬 (ORDER BY)✅❌
부등호 (<, >)✅❌
LIKE 'prefix%'✅❌
WAL 지원✅✅ (PG 10+)

PostgreSQL에서 Hash 인덱스: PG 10부터 WAL 지원으로 프로덕션에서 사용 가능해졌지만, 여전히 등호 전용. UUID 등 등호 검색만 하는 컬럼에서는 B-Tree보다 약간 빠를 수 있음.
● Transaction Isolation

트랜잭션 격리 수준 & MVCC

동시성 제어의 핵심 — PostgreSQL이 Lock 없이 읽기를 허용하는 원리.

트랜잭션 격리 수준 4가지를 설명하고, 각 수준에서 발생하는 문제를 말해주세요.
Read Uncommitted(dirty read 허용), Read Committed(커밋된 것만 읽음 — PG 기본), Repeatable Read(트랜잭션 내 같은 쿼리 같은 결과), Serializable(완전 직렬화). 수준이 높을수록 안전하지만 동시성이 떨어집니다. PostgreSQL은 Read Committed가 기본이며 MVCC로 읽기에 Lock을 걸지 않아 높은 동시성을 달성합니다.
격리 수준Dirty ReadNon-repeatable ReadPhantom Read성능
Read Uncommitted발생발생발생최고
Read Committed방지발생발생높음 (PG 기본)
Repeatable Read방지방지PG에서 방지중간
Serializable방지방지방지낮음

각 문제의 의미:
• Dirty Read: 다른 트랜잭션이 아직 커밋하지 않은 데이터를 읽음. 롤백되면 읽은 데이터가 유효하지 않음
• Non-repeatable Read: 같은 쿼리를 두 번 실행했는데 결과가 다름 (사이에 다른 트랜잭션이 UPDATE+COMMIT)
• Phantom Read: 같은 쿼리를 두 번 실행했는데 행 수가 다름 (INSERT+COMMIT)
PostgreSQL의 MVCC(Multi-Version Concurrency Control)를 설명해 주세요.
MVCC는 데이터의 여러 버전을 동시에 유지하여, 읽기 트랜잭션이 쓰기 트랜잭션을 블록하지 않도록 하는 동시성 제어 기법입니다. PostgreSQL에서 UPDATE는 기존 행을 삭제 표시하고 새 버전을 삽입합니다. 각 트랜잭션은 자신의 스냅샷 시점에 커밋된 버전만 볼 수 있습니다.
PostgreSQL MVCC 동작: 1. INSERT: 행에 xmin = 현재 txid 기록 (이 트랜잭션이 생성) 2. UPDATE: 기존 행에 xmax = 현재 txid 기록 (삭제 표시) + 새 행에 xmin = 현재 txid 기록 (새 버전 생성) 3. DELETE: 행에 xmax = 현재 txid 기록 (삭제 표시) 읽기 트랜잭션(txid=100)이 볼 수 있는 행: xmin <= 100 AND (xmax IS NULL OR xmax > 100) = "내 스냅샷 전에 생성되고, 아직 삭제되지 않은 행" → 읽기에 Lock 없음! 쓰기와 읽기가 동시에 가능
VACUUM의 필요성: 삭제 표시된 옛 버전(dead tuple)은 어떤 트랜잭션도 더 이상 참조하지 않을 때 물리적으로 제거해야 함. VACUUM 또는 autovacuum이 이를 수행.

autovacuum이 느려지면: dead tuple이 쌓여 테이블/인덱스가 비대화(bloat) → 쿼리 성능 저하 → shared_buffers 효율 감소 → 캐시 히트율 하락. autovacuum 모니터링이 PG 운영의 핵심.
pg_stat_user_tables.n_dead_tup — 테이블별 dead tuple 수
pg_stat_user_tables.last_autovacuum — 마지막 autovacuum 시각
● WAL & Checkpoint

Write-Ahead Log

"데이터를 바꾸기 전에 먼저 로그에 기록한다" — 크래시 복구의 핵심.

WAL(Write-Ahead Log)의 동작 원리와 PostgreSQL에서의 역할을 설명해 주세요.
WAL은 데이터 페이지를 변경하기 전에 변경 내용을 먼저 순차적으로 로그 파일에 기록하는 메커니즘입니다. 크래시가 발생하면 WAL을 재생(replay)하여 마지막 일관된 상태로 복구합니다. PostgreSQL에서 WAL은 추가로 Streaming Replication(복제)과 PITR(Point-in-Time Recovery, 특정 시점 복구)에도 사용됩니다.
PostgreSQL 쓰기 경로: 1. UPDATE 실행 2. WAL 버퍼에 변경 내용 기록 3. 데이터 페이지를 shared_buffers에서 수정 (dirty page) 4. COMMIT → WAL 버퍼를 디스크에 fsync (WAL 영속화) ← 여기까지가 COMMIT 시간. 데이터 페이지는 아직 디스크에 안 씀! 5. 나중에 Checkpoint가 dirty page를 디스크에 flush 크래시 복구: 마지막 checkpoint 이후의 WAL을 재생 → 모든 커밋된 트랜잭션 복구 보장
핵심: WAL fsync가 쓰기 성능의 병목
COMMIT마다 WAL을 디스크에 fsync하므로 WAL 디스크의 쓰기 성능이 전체 TPS를 좌우. SSD 필수. etcd도 같은 원리 — etcd_disk_wal_fsync_duration_seconds P99 > 10ms면 리더 선출 문제.

Checkpoint: dirty page를 디스크에 일괄 기록하는 시점. checkpoint_timeout(기본 5분) 또는 max_wal_size(기본 1GB) 초과 시 트리거. Checkpoint 중 I/O 스파이크 → checkpoint_completion_target = 0.9로 분산 쓰기.

Aurora와의 차이: Aurora는 WAL 자체를 네트워크를 통해 6개 복제본(3 AZ × 2)에 직접 전송. 데이터 페이지의 checkpoint 개념이 다름. Aurora Serverless는 ACU(Aurora Capacity Unit) 단위로 컴퓨트를 스케일하고, 스토리지는 자동 확장. ACU 스케일링 시 수 초간 연결 끊김 가능 → max_connections과 무관하게 ACU가 커넥션 수를 제한할 수 있음.
💡 면접에서 shared_buffers 튜닝 경험을 말하는 법
"PostgreSQL의 shared_buffers를 기본 128MB에서 4GB(전체 RAM의 25%)로 조정하여, B+Tree 인덱스의 내부 노드가 메모리에 캐시되면서 캐시 히트율이 78%에서 95%로 개선되었습니다. 이는 WAL의 checkpoint 빈도를 줄이는 효과도 있어 쓰기 I/O가 감소했습니다."
→ B+Tree + shared_buffers + WAL + Checkpoint를 하나의 스토리로 연결
● Connection Pooling

커넥션 풀링

PgBouncer, HikariCP — DB 커넥션은 비싼 리소스. 풀링으로 재사용.

DB 커넥션 풀링이 왜 필요하고, PgBouncer는 어떻게 동작하나요?
PostgreSQL 커넥션 하나당 OS 프로세스 하나가 생성되어 ~10MB 메모리를 소비합니다. K8s에서 Pod이 수십 개이고 각각 커넥션 풀(20개)을 가지면 수백 개의 DB 커넥션이 열려 PostgreSQL의 max_connections(기본 100)을 초과합니다. PgBouncer는 앱과 DB 사이에서 커넥션을 공유(transaction pooling)하여, 실제 DB 커넥션 수를 앱 커넥션 수보다 훨씬 적게 유지합니다.
PgBouncer 없이: Pod A (pool=20) ─→ 20 connections Pod B (pool=20) ─→ 20 connections → PostgreSQL (max_connections=100) Pod C (pool=20) ─→ 20 connections ← 60개 사용. HPA로 5개 되면 100개 초과! PgBouncer 사용: Pod A (pool=20) ─┐ Pod B (pool=20) ─┼→ PgBouncer (max=50) → PostgreSQL (max=100) Pod C (pool=20) ─┘ Transaction Pooling: Pod D (pool=20) ─┘ 쿼리 실행 중에만 DB 커넥션 할당 Pod E (pool=20) ─┘ 쿼리 완료 → DB 커넥션 반환 → 다른 앱이 재사용
Pooling 모드:
• Session: 앱 세션 동안 DB 커넥션 고정. 풀링 효과 최소
• Transaction: 트랜잭션 단위로 DB 커넥션 할당/반환. 가장 많이 사용. prepared statement 주의
• Statement: 쿼리 단위로 할당/반환. 가장 공격적. 멀티 스테이트먼트 트랜잭션 불가

K8s에서 커넥션 문제 시나리오:
HPA 스케일 아웃 → 새 Pod 생성 → 각 Pod이 커넥션 풀 초기화 → Cold Start 시 커넥션 폭증 → DB max_connections 초과 → 기존 Pod도 커넥션 에러 → Cascading Failure
방어: PgBouncer 필수 + 앱의 pool 크기를 최소화 + PgBouncer의 max_client_conn과 default_pool_size 분리 설정
● Query Plan

EXPLAIN ANALYZE

느린 쿼리를 찾고 원인을 분석하는 핵심 도구.

EXPLAIN ANALYZE의 출력을 읽는 방법을 설명해 주세요.
EXPLAIN은 쿼리 실행 계획(예상)을 보여주고, EXPLAIN ANALYZE는 실제로 실행하여 실측 시간까지 보여줍니다. 핵심 포인트는 Seq Scan vs Index Scan, 예상 행 수 vs 실제 행 수의 차이, Nested Loop vs Hash Join 선택, 가장 비용이 높은 노드(bottleneck)입니다.
EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 42; -- 좋은 경우: Index Scan Index Scan using idx_orders_user_id on orders (cost=0.43..8.45 rows=5 width=120) (actual time=0.023..0.035 rows=5 loops=1) Index Cond: (user_id = 42) Planning Time: 0.1 ms Execution Time: 0.06 ms -- 나쁜 경우: Seq Scan (인덱스 없음) Seq Scan on orders (cost=0.00..25000.00 rows=5 width=120) (actual time=150.234..890.567 rows=5 loops=1) Filter: (user_id = 42) Rows Removed by Filter: 999995 Execution Time: 891 ms
읽는 법:
• cost=시작..끝: 예상 비용 (상대값). 숫자가 클수록 비쌈
• rows=N: 예상 반환 행 수. 실제와 크게 다르면 통계가 오래됨 → ANALYZE table_name
• actual time: 실측 시간 (ms). Execution Time이 최종 시간
• Rows Removed by Filter: 인덱스 없이 필터링한 행 수. 높으면 인덱스 필요

Scan 유형:
• Seq Scan: 전체 테이블 순차 읽기. 대량 데이터에서 느림. 작은 테이블이면 OK
• Index Scan: B+Tree 인덱스로 검색. 소수 행 조회에 빠름
• Index Only Scan: 인덱스만으로 결과 반환 (테이블 접근 불필요). 가장 빠름
• Bitmap Index Scan: 인덱스로 대상 페이지를 먼저 찾고, 페이지 단위로 읽기. 중간 범위 쿼리

면접 포인트: "옵티마이저가 Seq Scan을 선택하는 이유"를 설명할 수 있으면 좋습니다. 테이블의 20% 이상을 읽어야 하면 인덱스 사용보다 Seq Scan이 효율적일 수 있습니다. 인덱스는 랜덤 I/O, Seq Scan은 순차 I/O이고, 순차 I/O가 훨씬 빠르기 때문.
⚠ 느린 쿼리 진단 체크리스트
1) EXPLAIN ANALYZE 실행
2) Seq Scan + Rows Removed 많으면 → 인덱스 추가
3) 예상 rows vs 실제 rows 크게 다르면 → ANALYZE 실행 (통계 업데이트)
4) Nested Loop + 큰 테이블이면 → Hash Join이 나을 수 있음. 메모리 설정 확인 (work_mem)
5) Sort + Disk이면 → work_mem 증가 또는 인덱스로 정렬 회피
