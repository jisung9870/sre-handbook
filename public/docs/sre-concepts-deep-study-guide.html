● Foundation

SRE를 구성하는 최소 핵심 축

이 영역은 “SRE가 무엇을 목표로 하고, 무엇을 숫자로 정의하며, 그 숫자를 운영 의사결정에 어떻게 연결하는가”를 잡는 파트입니다. 용어를 따로 외우기보다 서로의 관계를 보는 것이 중요합니다.

Core

SRE — Site Reliability Engineering

운영을 사람의 감각이 아니라 engineering discipline으로 다루는 접근
SRE의 본질은 “서비스 신뢰성을 코드와 시스템 설계로 관리한다”입니다. 단순히 장애가 나면 대응하는 역할이 아니라, 장애가 덜 나도록 구조를 바꾸고, 장애가 나더라도 짧게 끝나도록 자동화와 기준을 넣는 역할입니다. 즉, 운영 업무를 엔지니어링 문제로 바꾸는 프레임입니다.
핵심 질문
우리 서비스에서 신뢰할 만하다는 것은 정확히 무엇인가?
왜 중요한가
이 질문에 답해야 SLI, SLO, Alert, Incident, Postmortem이 모두 한 방향으로 맞춰집니다.
자주 하는 오해
SRE = 모니터링 담당자, SRE = 플랫폼팀, SRE = 온콜만 하는 팀. 모두 일부만 맞고 전체는 아닙니다.
실무 예시
GitOps rollout, SLO starter kit, Burn Rate alert, 자동 rollback, runbook codification.
핵심 정리
SRE는 Reliability를 측정 가능하게 만들고, 그 수치를 바탕으로 변경 속도와 안정성의 균형을 운영하는 체계입니다.
핵심 포인트
“운영을 잘한다”는 표현은 모호합니다. SRE는 그 모호함을 SLI·SLO·Error Budget 같은 수치 구조로 바꿉니다.
Core

Reliability vs Availability

서비스가 떠 있는 것과, 기대한 품질로 지속적으로 동작하는 것은 다르다
Availability는 “응답 가능한가”에 가깝고, Reliability는 “기대한 수준으로 일관되게 동작하는가”에 가깝습니다. API가 200 OK를 반환해도 5초 걸리거나 잘못된 데이터를 주면 가용하다고 볼 수는 있어도 신뢰할 만하다고 보기는 어렵습니다.
Availability
보통 uptime, success rate, reachability로 표현됩니다.
Reliability
Availability + latency + correctness + consistency를 포괄합니다.
실무 중요성
SLO를 availability만으로 정의하면 “느리지만 살아있는 서비스”를 놓칠 수 있습니다.
자주 헷갈리는 점
“99.9% 가용성이면 곧 충분한 신뢰성이다”라는 말은 틀릴 수 있습니다.
Availability 9s 감각 잡기
LevelAvailability월 허용 다운타임해석
Three 9s99.9%약 43.8분일반 서비스의 흔한 시작점
Four 9s99.99%약 4.4분핵심 API, 결제, 인증 등
Five 9s99.999%약 26초비용과 복잡도가 급격히 증가

SLI → SLO → SLA는 같은 계층이 아니라 서로 다른 층위다

Measure

SLI — Service Level Indicator

서비스 품질을 수치화한 실제 측정값
SLI는 “서비스가 지금 어느 정도 수준인가”를 말하는 숫자입니다. 보통 good events / total events의 비율로 표현합니다. 좋은 SLI는 반드시 사용자 관점이어야 합니다.
예시
30일 동안 500ms 이내 응답한 요청 비율, 성공한 결제 비율, 정상 완료된 배치 비율
나쁜 예시
CPU usage, pod count 같은 내부 상태를 SLI로 착각하는 경우
기억할 공식
SLI = Good Events ÷ Total Events × 100%
Target

SLO — Service Level Objective

SLI에 대해 팀 내부가 합의한 목표치
SLO는 “어느 수준까지는 acceptable한가”를 정하는 기준입니다. SLO가 있어야 알림과 변경 통제가 의미를 갖습니다. 100%는 보통 잘못된 목표입니다. 100%를 요구하면 혁신을 멈추고 모든 변경을 금지해야 하기 때문입니다.
예시
30일 window에서 latency SLI 99.5% 이상, availability SLI 99.9% 이상
실무 포인트
처음에는 현재 성능보다 약간 도전적이되 달성 가능한 수준으로 시작하는 것이 좋습니다.
Contract

SLA — Service Level Agreement

고객 또는 외부 이해관계자와의 약속
SLA는 법적/상업적 의미를 가진 약속입니다. SLO가 내부 운전석이라면 SLA는 외부 대시보드입니다. 일반적으로 SLO가 SLA보다 엄격해야 여유가 생깁니다.
실무 감각
예: 내부 SLO 99.95, 외부 SLA 99.9
왜 분리하나
고객 약속은 보수적으로, 내부 운영 목표는 조금 더 엄격하게 가져가야 위반 위험을 줄일 수 있습니다.

Governance

Error Budget & Policy

신뢰성 목표를 배포·변경 의사결정에 연결하는 메커니즘
SLO가 99.9%라면 0.1%의 실패는 허용됩니다. 이 허용분이 Error Budget입니다. 중요한 점은 Budget이 단순 통계가 아니라 변경의 허용 범위라는 것입니다. Budget이 충분히 남으면 빠르게 배포해도 되고, 거의 소진되면 배포를 보수적으로 해야 합니다.
공식
Error Budget = 1 − SLO Target
30일 기준 예시
SLO 99.9% → 한 달에 약 43.2분의 실패 허용
왜 중요한가
개발팀의 배포 속도 요구와 운영팀의 안정 요구를 감정이 아니라 숫자로 조정할 수 있습니다.
Policy 예시
Budget 50% 이상 남음: normal deploy
20% 이하: review 강화
소진: freeze + reliability work
주의
Budget이 없는데 SLO만 두면 “좋은 숫자”는 생기지만 “행동 규칙”은 생기지 않습니다. SLO와 Budget Policy는 같이 가야 합니다.
● Metrics & Signals

지표는 많아도, 먼저 봐야 할 것은 제한적입니다

이 영역은 “무엇을 봐야 문제가 빨리 드러나는가”를 다룹니다. 모든 메트릭을 다 보는 것이 아니라, 사용자의 경험과 운영 속도에 직접 연결되는 핵심 신호를 먼저 잡는 것이 목적입니다.

Signals

Golden Signals

Latency, Traffic, Errors, Saturation — 서비스를 가장 빠르게 이해하는 4개의 질문
Golden Signals는 “어떤 서비스든 우선적으로 봐야 하는 최소 공통 시그널”입니다. 수백 개 메트릭보다 먼저 이 4개가 중요합니다.

Latency

사용자가 체감하는 속도. 평균보다 P95/P99가 중요합니다. 성공 요청과 실패 요청의 latency를 분리해서 봐야 합니다.

Traffic

요청량. RPS, QPS, jobs/sec, messages/sec 등. 다른 지표 변동의 원인이 되는 경우가 많습니다.

Errors

실패율. HTTP 5xx뿐 아니라 business failure, timeout, wrong result도 고려해야 합니다.

Saturation

리소스 병목 신호. CPU, memory, queue depth, connection pool exhaustion, disk I/O wait 등.

학습 팁
Golden Signals는 “메트릭 이름”이 아니라 “질문 틀”입니다. 서비스마다 실제 구현 지표는 달라집니다.
Detection

MTTD

Mean Time To Detect
문제가 발생한 시점부터 팀이 그것을 실제로 인지하기까지 걸린 평균 시간입니다. 알림 품질과 관측 커버리지가 MTTD를 좌우합니다.
MTTD = Σ(감지 시각 − 장애 발생 시각) ÷ 장애 건수
MTTD가 길면 복구를 아무리 잘해도 전체 대응은 느립니다.
Recovery

MTTR

Mean Time To Recovery
이 문서에서는 장애 발생 시각부터 진단·완화·복구·검증을 거쳐 서비스가 정상 수준으로 돌아오기까지를 MTTR로 설명합니다. 일부 조직은 감지 시각부터 측정하기도 하지만, 핵심은 기준을 팀 안에서 일관되게 유지하는 것입니다.
MTTR = Σ(복구 완료 시각 − 장애 발생 시각) ÷ 장애 건수
Runbook, rollback, observability, ownership clarity가 직접적인 개선 수단입니다.

Performance

DORA Metrics

딜리버리의 속도와 안정성을 동시에 보는 운영 성과 프레임
DORA는 SRE 자체는 아니지만 SRE와 매우 잘 맞물립니다. SRE가 서비스 품질을 정의한다면, DORA는 팀이 변경을 얼마나 빠르고 안전하게 배포하는지 보여줍니다.
Velocity
Deployment Frequency, Lead Time for Changes
Stability
Change Failure Rate, Recovery Time (MTTR)
SRE와 연결
CFR는 Error Budget 소진 속도와 연결되고, Recovery Time은 MTTR과 사실상 같은 축입니다.
실무 포인트
고속 배포와 안정성은 반드시 trade-off인 것만은 아닙니다. 작은 변경·자동화·관측이 있으면 둘 다 좋아질 수 있습니다.

자주 헷갈리는 비교를 한 번에 정리

비교왼쪽오른쪽핵심 차이
Availability vs Reliability살아있음기대한 품질로 동작Reliability가 더 넓은 개념
SLI vs SLO측정값목표값하나는 fact, 하나는 target
SLO vs SLA내부 기준외부 약속SLA가 보통 더 느슨함
MTTD vs MTTR문제 인지문제 복구MTTD는 앞 구간, MTTR은 전체 복구 구간
Golden Signals vs DORA서비스 상태배포 성과운영 상태 vs 전달 성과
Monitoring vs Observability미리 정한 항목 감시새로운 질문에 답할 능력후자가 더 탐색적
● Operations

SRE는 결국 운영 체계로 완성됩니다

수치만 있다고 SRE가 되지는 않습니다. 알림이 잘 울리고, 장애가 잘 다뤄지고, 회고가 시스템 개선으로 이어질 때 비로소 살아 있는 운영 체계가 됩니다.

Action

Alerting

모든 신호를 사람에게 보내는 것이 아니라, 행동이 필요한 신호만 선별하는 설계
좋은 알림은 팀이 바로 행동할 수 있어야 합니다. 나쁜 알림은 정보만 많고 행동이 불분명합니다. SRE에서 좋은 알림의 핵심은 사용자 영향 기반, 중복 최소화, 명확한 액션입니다.
좋은 알림
SLO burn rate high, checkout success rate drop, API p99 severe regression
나쁜 알림
CPU 80%, pod restarted once, disk 70% — 맥락 없이 울리면 피로만 유발
Burn Rate
현재 속도로 error budget이 얼마나 빨리 닳는지 보는 지표. 단순 threshold보다 더 운영 친화적입니다.
운영 원칙
Page는 정말 깨워야 할 때만, 나머지는 ticket 또는 dashboard로 흡수합니다.
Explore

Observability

예상한 문제뿐 아니라 예상하지 못한 문제에도 질문할 수 있는 능력
Monitoring이 “정해진 조건 감시”라면, Observability는 “처음 보는 문제에도 이유를 추적할 수 있는 체계”입니다. Metrics, Logs, Traces가 따로 존재하는 것이 아니라 서로 drill-down되어야 효과가 큽니다.
Metrics
현재 상태와 추세. 무엇이 나빠졌는지 빨리 파악
Logs
무슨 이벤트가 일어났는지 구체적 서술
Traces
한 요청이 시스템을 어떻게 통과했는지 경로 제공
실무 목표
느린 요청 발견 → 관련 로그 → 관련 trace → root cause로 이어지는 흐름을 짧게 만드는 것
Respond

Incident Management

문제 발생 시 혼란을 줄이고 복구를 빠르게 만드는 운영 규율
장애가 났을 때 중요한 것은 “누가 리드하는가, 상태를 누가 공유하는가, 기술 조치는 누가 하는가”가 명확한 것입니다. 즉흥 대응은 사람을 지치게 하고 MTTR을 늘립니다.
단계
Detect → Triage → Assign → Mitigate → Recover → Communicate → Review
핵심 역할
Incident Commander, Ops Lead, Communication Lead
심각도
P1 전체 장애, P2 주요 기능 영향, P3 부분 영향, P4 경미한 이슈
실무 포인트
기술 문제 해결과 상태 공유를 분리해야 현장이 덜 흔들립니다.
Learn

Postmortem

장애를 지식 자산으로 바꾸는 과정
Postmortem의 목적은 책임자 찾기가 아니라 시스템적 재발 방지입니다. 잘 쓴 postmortem은 다음 장애에서 MTTD와 MTTR을 동시에 줄이는 자산이 됩니다.
필수 요소
Summary, Impact, Timeline, Root Cause, Contributing Factors, Action Items
Blameless 원칙
왜 개인이 실수했는지보다, 왜 그 실수가 서비스 영향으로 이어졌는지를 봅니다.
좋은 Action Item
담당자, 마감일, 성공 조건이 명확해야 합니다.
자주 망하는 이유
문서만 쓰고 실제 추적을 안 해서 운영 개선으로 연결되지 않음
Scale

Toil

반복적이고 수동적이며 자동화 가능하지만 지금은 사람이 하고 있는 운영 일
Toil은 단순히 귀찮은 일이 아닙니다. 서비스 규모가 커질수록 선형 또는 그 이상으로 증가하는 운영 부하입니다. Toil을 방치하면 팀의 시간이 reliability engineering이 아니라 반복 작업에 잠식됩니다.
예시
수동 배포, 반복적인 권한 부여, 수동 인증서 교체, 반복 로그 조사, 동일 장애 수동 조치
판별 기준
manual / repetitive / automatable / tactical / no enduring value
왜 줄이나
Toil은 시간을 먹지만 시스템 수준의 개선을 남기지 않습니다.
대응 방식
automation, self-service platform, better defaults, removal of unnecessary process
● Practice

개념을 연결해서 봐야 실무에 들어옵니다

이 파트는 앞서 본 개념들을 분리된 용어가 아니라 운영 시스템으로 엮어 보는 영역입니다. 학습용으로 반복해서 보기 좋게 관계도, 적용 순서, 실무 매핑을 넣었습니다.
