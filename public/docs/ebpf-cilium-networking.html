● eBPF Internals

eBPF 프로그래밍 모델

커널을 수정하지 않고 커널 내부에서 안전하게 코드를 실행하는 혁명적 기술.

Architecture

eBPF 아키텍처

유저스페이스에서 컴파일 → 커널 Verifier 검증 → JIT 컴파일 → Hook point에서 실행
eBPF(extended Berkeley Packet Filter)는 커널 모듈을 작성하지 않고도 커널 내부에서 코드를 실행할 수 있게 하는 기술이다. 네트워킹, 보안, 트레이싱, 관측에 혁명을 일으키고 있으며, Cilium, Falco, Tetragon, bpftrace 등의 기반 기술이다.

핵심은 안전성이다. eBPF 프로그램은 커널의 Verifier가 실행 전에 검증하여 커널 크래시, 무한 루프, 메모리 접근 위반을 방지한다.
실행 흐름
1. C / Rust로 eBPF 프로그램 작성 2. LLVM/Clang으로 BPF 바이트코드 컴파일 3. bpf() syscall로 커널에 로드 4. Verifier가 안전성 검증 → 무한 루프? 범위 초과 메모리 접근? 금지! 5. JIT 컴파일러가 네이티브 코드로 변환 6. Hook point에 연결 (kprobe, tracepoint, XDP, tc 등) 7. 이벤트 발생 시 eBPF 프로그램 실행 8. Maps를 통해 유저스페이스와 데이터 교환
Verifier 제약
루프: bounded loop만 허용 (5.3+). 무한 루프 불가
스택: 최대 512바이트. 큰 데이터는 Map에 저장
명령어 수: 최대 100만 (5.2+, 이전은 4096)
메모리 접근: 모든 포인터 접근을 정적 분석. NULL 체크 강제
헬퍼 함수: 승인된 커널 함수만 호출 가능. 임의 커널 함수 호출 불가

결과: eBPF 프로그램은 커널을 크래시시킬 수 없다. 커널 모듈 대비 극적으로 안전

Maps & Helper Functions

Maps = eBPF 프로그램과 유저스페이스 간 데이터 공유 채널. Helpers = 커널 기능 호출 인터페이스
Map 유형
BPF_MAP_TYPE_HASH: 키-값 해시맵. 가장 범용적. 커넥션 추적, 카운터
BPF_MAP_TYPE_ARRAY: 인덱스 기반 배열. 고정 크기, O(1) 접근
BPF_MAP_TYPE_LRU_HASH: LRU 해시. 크기 제한 + 오래된 엔트리 자동 삭제. 커넥션 캐시에 유용
BPF_MAP_TYPE_RINGBUF: 링 버퍼. 이벤트 스트리밍. perf_event 대체
BPF_MAP_TYPE_PERCPU_HASH: CPU별 해시맵. Lock-free 카운터
BPF_MAP_TYPE_LPM_TRIE: Longest Prefix Match. IP 기반 라우팅 테이블

Map은 커널-유저 공유 메모리. eBPF 프로그램이 Map에 쓰고, 유저스페이스 앱이 Map에서 읽는다.
주요 Helper Functions
네트워크:
bpf_redirect() — 패킷을 다른 인터페이스로 리다이렉트
bpf_skb_load_bytes() — 패킷 데이터 읽기
bpf_fib_lookup() — FIB(라우팅) 테이블 조회
bpf_sk_lookup_tcp() — 소켓 조회

트레이싱:
bpf_probe_read_kernel() — 커널 메모리 읽기
bpf_get_current_pid_tgid() — 현재 PID
bpf_get_current_comm() — 프로세스 이름
bpf_ktime_get_ns() — 나노초 타임스탬프

Map 조작:
bpf_map_lookup_elem() / bpf_map_update_elem() / bpf_map_delete_elem()
● XDP & tc BPF

XDP와 tc BPF

NIC 드라이버 레벨(XDP)과 네트워크 스택(tc)에서 패킷을 처리하는 eBPF 데이터플레인.

Fast Path

XDP — eXpress Data Path

NIC 드라이버에서 패킷을 처리하는 가장 빠른 프로그래밍 가능 데이터 경로
XDP는 패킷이 커널 네트워크 스택에 진입하기 전, NIC 드라이버 레벨에서 eBPF 프로그램을 실행한다. sk_buff 할당, netfilter 통과 등 모든 오버헤드를 건너뛰어 초당 수백만~수천만 패킷을 처리할 수 있다.
XDP 동작 모드
Native (드라이버): NIC 드라이버에 직접 연결. 최고 성능. 드라이버 지원 필요 (ixgbe, i40e, mlx5, virtio-net 등)
Generic (커널): 드라이버 미지원 시 커널 레벨에서 에뮬레이션. 성능은 tc BPF와 유사. 테스트/개발용
Offload (NIC HW): NIC 하드웨어에서 eBPF 실행. Netronome SmartNIC 등. CPU 사용 0

XDP 액션 (반환값):
XDP_PASS — 패킷을 정상 네트워크 스택으로 전달
XDP_DROP — 패킷 즉시 드롭. DDoS 방어에 최적
XDP_TX — 같은 NIC으로 패킷 반송 (헤더 수정 후)
XDP_REDIRECT — 다른 인터페이스 또는 AF_XDP 소켓으로 전달
XDP 활용 사례
DDoS Mitigation: 악성 패킷을 XDP_DROP으로 즉시 차단. CPU 부하 최소. Facebook은 초당 수천만 패킷을 XDP로 필터링
L4 Load Balancer: Facebook Katran, Cloudflare Unimog. XDP에서 IPIP/GRE 캡슐화 후 백엔드로 리다이렉트
Monitoring: 패킷 샘플링 후 분석 프로그램으로 전달

Cilium에서: --bpf-lb-acceleration=native 설정 시 NodePort/LoadBalancer 트래픽을 XDP에서 처리하여 kube-proxy 대비 5~10배 성능 향상
Dataplane

tc BPF — Traffic Control에서의 eBPF

sk_buff 레벨에서 패킷을 가공하는 주요 Cilium 데이터플레인 경로
tc(Traffic Control)의 ingress/egress 훅에 eBPF 프로그램을 연결한다. XDP보다 늦게 실행되지만 sk_buff에 접근할 수 있어 L3/L4/L7 정보를 더 쉽게 처리할 수 있다. Cilium의 주요 데이터플레인이 tc BPF에서 동작한다.
XDP vs tc BPF 비교
XDP:
• 실행 시점: NIC 드라이버 (네트워크 스택 전)
• 접근 데이터: raw 패킷 (xdp_md)
• 성능: 최고 (sk_buff 할당 없음)
• 제약: L2/L3 처리만. L4+ 파싱 직접 구현 필요

tc BPF:
• 실행 시점: 네트워크 스택 진입 후 (ingress/egress)
• 접근 데이터: sk_buff (L3/L4 정보 파싱됨)
• 성능: XDP보다 낮지만 iptables보다 훨씬 높음
• 장점: 더 풍부한 컨텍스트, 양방향(ingress+egress)

Cilium: 대부분의 로직을 tc BPF에서 처리하고, NodePort/LB 가속만 XDP 사용
패킷 처리 위치 비교
NIC HW → [XDP] → sk_buff 할당 → [tc ingress] → netfilter/conntrack → routing → [tc egress] → NIC TX iptables: netfilter에서 처리 (중간) tc BPF: sk_buff 직후 처리 (netfilter 전) XDP: sk_buff 전에 처리 (가장 빠름)
● Cilium CNI

Cilium 아키텍처와 핵심 기능

eBPF 기반 K8s CNI — kube-proxy 대체, L3/L4/L7 정책, WireGuard 암호화.

Architecture

Cilium 아키텍처

각 노드의 Cilium Agent가 eBPF 프로그램을 관리하여 네트워킹, 보안, 관측을 커널 레벨에서 처리
Cilium은 iptables/kube-proxy를 완전히 대체하고, 모든 패킷 처리를 eBPF로 수행한다. 사이드카 프록시 없이도 L7 정책 적용, mTLS, 네트워크 관측이 가능한 것이 핵심 차별점이다.
핵심 컴포넌트
Cilium Agent (DaemonSet): 각 노드에서 실행. K8s API 감시 → eBPF 프로그램 생성·업데이트·로드
Cilium Operator (Deployment): 클러스터 레벨 작업. IPAM, CiliumNode 관리
eBPF Programs: 각 Pod의 veth에 tc BPF 프로그램 연결. 패킷 라우팅, 정책 적용, LB, NAT 수행
eBPF Maps: 정책 맵, 커넥션 트래킹 맵, 서비스 맵, endpoint 맵 등을 커널 내에 유지
Hubble (선택): eBPF에서 수집한 Flow 데이터를 시각화하는 관측 계층
데이터 경로
Pod A → veth → [tc BPF: policy check + NAT] → (같은 노드면) → Pod B veth → [tc BPF] → (다른 노드면) → VXLAN/native routing → 대상 노드 → [tc BPF] → Pod B kube-proxy 없음: Service ClusterIP → BPF Maps에서 직접 DNAT iptables 체인 전체를 건너뜀
kube-proxy

kube-proxy 완전 대체

iptables/IPVS 규칙 수천 개를 eBPF Map 조회 한 번으로 대체
kube-proxy는 Service마다 iptables 규칙을 생성한다. Service 1000개 × 엔드포인트 10개 = 수만 개의 iptables 규칙. 패킷마다 이 규칙을 순차 평가하므로 Service가 많아질수록 O(n) 성능 저하가 발생한다.

Cilium은 Service→Endpoint 매핑을 eBPF Hash Map에 저장하여 O(1) 조회로 대체한다.
설정
Cilium Helm values:
kubeProxyReplacement: true
k8sServiceHost: "API_SERVER_IP"
k8sServicePort: "6443"

kube-proxy DaemonSet 삭제:
kubectl -n kube-system delete ds kube-proxy
kubectl -n kube-system delete cm kube-proxy

XDP 가속 (선택):
loadBalancer.acceleration: native
→ NodePort/LoadBalancer 트래픽을 XDP에서 처리
iptables 대비 이점
성능: Service 수에 무관한 O(1) 조회. 대규모 클러스터에서 극적 차이
연결 추적: eBPF conntrack이 netfilter conntrack 대체. 더 효율적인 메모리 사용
Session Affinity: BPF Map으로 클라이언트→백엔드 매핑 유지
DSR (Direct Server Return): 응답 패킷이 LB를 경유하지 않고 클라이언트에 직접 전달. latency 감소
Maglev: 일관된 해싱으로 백엔드 변경 시 최소한의 연결만 재분배

주의: kube-proxy 대체 모드에서는 externalTrafficPolicy: Local 동작이 Cilium 방식으로 처리됨. NHN Cloud LB와의 호환성 사전 검증 필요
Policy

Cilium NetworkPolicy — L3/L4/L7 + DNS-aware

K8s NetworkPolicy의 상위 호환. DNS 이름 기반, HTTP 경로 기반 정책까지 지원
CiliumNetworkPolicy (L3/L4)
apiVersion: cilium.io/v2 kind: CiliumNetworkPolicy spec: endpointSelector: matchLabels: { app: api-server } ingress: - fromEndpoints: - matchLabels: { app: frontend } toPorts: - ports: - { port: "8080", protocol: TCP } egress: - toFQDNs: - matchName: "api.external.com" toPorts: - { ports: [{ port: "443" }] }

DNS-aware egress: FQDN 기반 정책. IP가 변해도 자동 추적
L7 정책 (HTTP-aware)
toPorts: - ports: - { port: "8080", protocol: TCP } rules: http: - method: GET path: "/api/v1/public.*" - method: POST path: "/api/v1/orders" headers: - "X-Auth-Token"

HTTP 메서드 + 경로 기반 접근 제어. Istio AuthorizationPolicy와 유사하지만 사이드카 없이 커널 레벨에서 처리. L7 정책 적용 시 Cilium이 Envoy 프록시를 투명하게 삽입
Encryption

WireGuard 노드 간 암호화

사이드카 없이 노드 간 트래픽을 커널 레벨에서 암호화 — mTLS의 경량 대안
Cilium은 WireGuard(커널 내장 VPN 프로토콜)를 사용하여 노드 간 모든 Pod 트래픽을 투명하게 암호화한다. IPsec보다 빠르고, 사이드카 mTLS보다 오버헤드가 훨씬 적다.
설정
Helm values:
encryption.enabled: true
encryption.type: wireguard

활성화하면 각 노드에 WireGuard 인터페이스(cilium_wg0)가 생성되고, 노드 간 키 교환이 자동으로 이루어진다. 앱 변경 없음.

확인:
cilium encrypt status
cilium node list — WireGuard 공개키 확인
WireGuard vs IPsec vs mTLS
WireGuard: 커널 레벨, ~2~5% throughput 오버헤드. 키 관리 자동. 가장 간단
IPsec: 커널 레벨, ~10~20% 오버헤드. 설정 복잡. 레거시 호환
mTLS (Istio/Linkerd): 유저스페이스, ~5~10% + 사이드카 메모리. Pod 신원 기반 인증

차이점: WireGuard는 노드 간 암호화 (같은 노드의 Pod 간은 미암호화). mTLS는 Pod 간 암호화 (모든 경로).
조합: Cilium WireGuard(노드 간) + Cilium L7 정책(접근 제어) = Service Mesh 없이 상당 수준의 보안 달성
💡 Solo DevOps 환경에서의 선택
현재 cert-manager로 수동 mTLS를 구현하고 있다면, Cilium WireGuard가 가장 낮은 운영 부담으로 네트워크 암호화를 달성하는 방법이다. 앱 코드 변경 없이 Helm 값 하나로 활성화. Service Mesh가 필요한 L7 기능(트래픽 분할, JWT 검증 등)이 아니라면 충분할 수 있다.
● Hubble Observability

Hubble — 네트워크 관측

eBPF에서 직접 수집하는 L3/L4/L7 Flow 데이터. 사이드카 없는 네트워크 관측.

Observability

Hubble 네트워크 관측

모든 패킷 흐름을 eBPF에서 캡처하여 서비스 간 통신을 시각화
핵심 기능
Flow 관측: 모든 네트워크 연결의 source, destination, port, protocol, verdict(allowed/denied) 기록
Service Map: 서비스 간 의존성을 실시간 그래프로 시각화 (Kiali와 유사)
Network Policy 감사: 어떤 정책이 트래픽을 허용/차단했는지 추적
DNS 관측: 모든 DNS 쿼리/응답을 기록. 어떤 Pod이 어떤 도메인을 조회하는지

CLI:
hubble observe --namespace production
hubble observe --from-pod prod/api-server --to-pod prod/db
hubble observe --verdict DROPPED — 차단된 트래픽만

UI: Hubble UI (웹 대시보드)로 Service Map 시각화
Hubble 설정
Helm values:
hubble.enabled: true
hubble.relay.enabled: true
hubble.ui.enabled: true
hubble.metrics.enabled:
- dns:query;ignoreAAAA
- drop
- tcp
- flow
- icmp
- httpV2:exemplars=true;labelsContext=source_ip,source_namespace,source_workload,destination_ip,destination_namespace,destination_workload

Prometheus 연동:
Hubble 메트릭을 Prometheus로 수집 → Grafana 대시보드
LGTM 스택과 자연스럽게 통합
Metrics

Hubble Flow 메트릭

eBPF에서 직접 수집되는 L3/L4/L7 메트릭 — Istio 메시 메트릭의 경량 대안
