HTTP/2 내부 동작

멀티플렉싱, 바이너리 프레이밍, HPACK, Flow Control — HTTP/1.1의 근본적 한계를 해결.

Core

멀티플렉싱 & 바이너리 프레이밍

하나의 TCP 연결에서 여러 요청/응답을 동시에 주고받는 핵심 혁신
HTTP/1.1에서는 하나의 TCP 연결에서 한 번에 하나의 요청-응답만 처리할 수 있었다(Head-of-Line Blocking). 6개 병렬 연결을 열어 우회했지만 근본적 해결이 아니었다. HTTP/2는 단일 TCP 연결에서 여러 Stream을 동시에 주고받는다.
핵심 개념
Connection: 하나의 TCP 연결. 클라이언트-서버 간 1개면 충분
Stream: Connection 내의 양방향 데이터 흐름. 요청-응답 쌍 하나가 1 Stream
Frame: HTTP/2의 최소 전송 단위. 바이너리 형식
• HEADERS frame: 요청/응답 헤더 (HPACK 압축)
• DATA frame: 바디 데이터
• SETTINGS, PING, GOAWAY, WINDOW_UPDATE 등 제어 프레임

바이너리 프레이밍: HTTP/1.1은 텍스트 기반이라 파싱 비용이 크고 모호성 있음. HTTP/2는 고정 포맷 바이너리 → 파싱 빠름, 에러 적음
멀티플렉싱 동작
TCP Connection ├── Stream 1: GET /api/users (요청 전송 중) ├── Stream 3: GET /api/orders (응답 수신 중) ├── Stream 5: POST /api/payment (요청 완료, 응답 대기) └── Stream 7: GET /api/items (응답 수신 완료) 각 Stream의 Frame이 인터리브되어 전송: [S1:HEADERS][S3:DATA][S1:DATA][S5:HEADERS][S3:DATA]...

Server Push: 서버가 클라이언트 요청 없이 리소스를 미리 보낼 수 있음. 하지만 실무에서 활용도 낮아 HTTP/3에서 사실상 폐기됨
Efficiency

HPACK & Flow Control

헤더 압축으로 대역폭 절감, Flow Control로 스트림별 흐름 제어
HPACK 헤더 압축
문제: HTTP/1.1에서 매 요청마다 Cookie, User-Agent 등 수백 바이트의 동일 헤더를 반복 전송

HPACK 해결:
• Static Table: 자주 쓰는 헤더 61개를 인덱스로 (예: :method GET = index 2)
• Dynamic Table: 연결 중 사용된 헤더를 동적으로 테이블에 추가. 이후 인덱스로만 참조
• Huffman Encoding: 문자열을 Huffman 코드로 압축

효과: 헤더 크기 85~95% 감소. API 서비스에서 체감 큰 대역폭 절감
Flow Control
스트림별 흐름 제어:
• 수신측이 WINDOW_UPDATE frame으로 수신 가능 크기를 알림
• 송신측은 윈도우 크기 내에서만 DATA frame 전송
• Connection 레벨 + Stream 레벨 이중 제어

SRE 관점: Flow Control이 너무 작으면 throughput 저하. Envoy/Nginx에서 initial_stream_window_size, initial_connection_window_size 튜닝 가능

기본값: 65535 bytes (64KB). 고대역폭 환경에서는 1MB+ 권장
Problem

TCP Head-of-Line Blocking 문제

HTTP/2가 해결하지 못한 근본적 문제 — HTTP/3(QUIC)이 등장한 이유
HTTP/2는 HTTP 레벨의 HOL Blocking은 해결했지만, TCP 레벨의 HOL Blocking은 해결하지 못했다. 하나의 TCP 세그먼트가 손실되면 해당 연결의 모든 스트림이 재전송을 기다리며 대기한다. 무선 환경(패킷 손실 1~5%)에서 이 문제가 심각하다.
TCP vs QUIC HOL Blocking
HTTP/2 over TCP: Stream 1: [pkt1][pkt2] ← pkt1 손실! Stream 2: [pkt3][pkt4] ← pkt3 정상인데 pkt1 재전송 대기! Stream 3: [pkt5] ← 역시 대기! → 모든 Stream이 하나의 패킷 손실에 막힘 HTTP/3 over QUIC: Stream 1: [pkt1][pkt2] ← pkt1 손실 → Stream 1만 대기 Stream 2: [pkt3][pkt4] ← 정상 처리 계속! Stream 3: [pkt5] ← 정상 처리 계속! → 손실된 Stream만 영향. 나머지는 독립적 진행
● HTTP/3 & QUIC

HTTP/3 & QUIC 프로토콜

UDP 기반 전송 프로토콜. TCP의 HOL Blocking을 근본적으로 해결.

Architecture

QUIC 아키텍처

UDP 위에 구축된 전송 프로토콜. TLS 1.3 통합, 스트림별 독립 흐름 제어
QUIC(Quick UDP Internet Connections)는 Google이 개발하고 IETF가 표준화한 UDP 기반 전송 프로토콜이다. TCP + TLS의 기능을 하나로 통합하면서 TCP의 근본적 한계(HOL Blocking, 연결 설정 지연)를 해결한다.
QUIC 핵심 특성
1) UDP 기반: 커널 TCP 스택을 우회. 유저스페이스에서 구현 가능 → OS 업데이트 없이 프로토콜 개선
2) TLS 1.3 통합: 전송 핸드셰이크 + 암호화 핸드셰이크를 합쳐 1-RTT. 재연결 시 0-RTT
3) 스트림 독립: 각 스트림이 독립적 흐름 제어. 패킷 손실이 해당 스트림만 영향
4) Connection Migration: 연결 ID 기반. IP가 바뀌어도(Wi-Fi→LTE) 연결 유지
5) 개선된 Loss Detection: 패킷 번호가 단조 증가하여 재전송 모호성 제거
연결 설정 비교
TCP + TLS 1.2:
TCP 3-way (1 RTT) + TLS (2 RTT) = 3 RTT

TCP + TLS 1.3:
TCP 3-way (1 RTT) + TLS (1 RTT) = 2 RTT

QUIC (첫 연결):
QUIC + TLS 1.3 통합 = 1 RTT

QUIC (재연결):
0-RTT — 이전 세션 키로 즉시 데이터 전송 = 0 RTT

모바일 환경(RTT 50~100ms)에서 체감 100~300ms 개선
Benefits

HTTP/3의 SRE 관점 장점

네트워크 불안정 환경에서의 성능, 연결 마이그레이션, 빠른 프로토콜 진화
패킷 손실 환경
TCP (HTTP/2): 패킷 손실 2%에서 성능이 급격히 저하. 모든 스트림이 재전송 대기
QUIC (HTTP/3): 패킷 손실 2%에서도 영향받지 않는 스트림은 정상 처리

실측: 손실 1%에서 HTTP/3가 HTTP/2 대비 P99 latency 30~50% 개선
모바일 앱, 글로벌 서비스에서 특히 효과적
Connection Migration
시나리오: 모바일 사용자가 Wi-Fi에서 LTE로 전환

TCP: IP 변경 → 기존 TCP 연결 끊김 → 새 연결 + TLS 핸드셰이크 → 2~3 RTT 손실
QUIC: Connection ID 기반으로 연결 유지 → 끊김 없이 계속 통신

SRE 영향: 모바일 앱의 에러율과 재연결 비용 대폭 감소. SLI에서 네트워크 전환 관련 에러가 줄어듦
Adoption

HTTP/3 도입 현황과 고려사항

CDN/브라우저는 이미 기본 지원. 서버/프록시 지원은 성장 중
지원 현황
CDN: Cloudflare, AWS CloudFront, Google Cloud CDN — 기본 활성
브라우저: Chrome, Firefox, Safari, Edge — 모두 지원
서버: Nginx(실험적), Caddy(기본), LiteSpeed(기본), Go net/http(1.24 실험적)
프록시: Envoy(진행 중), Traefik(진행 중), HAProxy(미지원)

K8s 환경: 아직 Ingress/Gateway 레벨에서 HTTP/3 지원이 제한적. CDN → HTTP/3 → LB → HTTP/2 → 내부 서비스 구조가 현실적
도입 시 주의
방화벽: 많은 기업 방화벽이 UDP 443을 차단. QUIC fallback → TCP 필수
DDoS: UDP 기반이라 반사 공격에 취약할 수 있음. QUIC은 자체 대응(Initial packet 검증) 내장
모니터링: TCP 기반 모니터링 도구가 QUIC 트래픽을 인식 못 할 수 있음. Wireshark 4.0+에서 QUIC 디코딩 지원
Alt-Svc 헤더: 서버가 Alt-Svc: h3=":443" 헤더로 HTTP/3 지원을 알림. 클라이언트가 다음 요청부터 QUIC 시도
● gRPC Internals

gRPC 내부 동작

HTTP/2 기반 RPC 프레임워크. Protobuf, 스트리밍, LB 문제 — 마이크로서비스 통신의 핵심.

Architecture

gRPC 내부 아키텍처

HTTP/2 + Protocol Buffers — 바이너리 직렬화 + 멀티플렉싱의 조합
gRPC는 Google이 개발한 HTTP/2 기반 고성능 RPC 프레임워크다. REST(JSON over HTTP/1.1) 대비 직렬화 크기 3~10배 축소, latency 2~5배 개선이 가능하다. Go, Java, Python, C++ 등 다언어 지원으로 마이크로서비스 간 통신의 표준이 되고 있다.
핵심 구성
Protocol Buffers (protobuf): 인터페이스 정의 언어(IDL) + 바이너리 직렬화 포맷
.proto 파일에서 서비스/메시지 정의 → 각 언어 코드 자동 생성

HTTP/2 전송:
• 각 RPC 호출 = 하나의 HTTP/2 Stream
• 멀티플렉싱으로 단일 연결에서 수천 RPC 동시 처리
• 헤더 압축(HPACK)으로 메타데이터 오버헤드 최소화

gRPC 상태 코드: HTTP 상태 코드와 별도. OK, CANCELLED, DEADLINE_EXCEEDED, UNAVAILABLE, RESOURCE_EXHAUSTED 등. grpc-status 헤더로 전달
REST vs gRPC 비교
직렬화: JSON ~200 bytes vs protobuf ~50 bytes (같은 데이터)
파싱: JSON 텍스트 파싱 vs protobuf 바이너리 디코딩 (10배 빠름)
타입 안전성: JSON은 런타임 에러 vs protobuf는 컴파일 타임 검증
스트리밍: REST는 요청-응답만 vs gRPC는 양방향 스트리밍
코드 생성: REST는 수동 클라이언트 vs gRPC는 자동 생성

REST가 나은 경우: 브라우저 직접 호출, 사람이 읽을 수 있는 API, 간단한 CRUD
gRPC가 나은 경우: 서비스 간 내부 통신, 고성능 요구, 스트리밍 필요
Streaming

gRPC 스트리밍 4패턴

Unary, Server Streaming, Client Streaming, Bidirectional — 실시간 데이터 전달
패턴설명사용 예
Unary1 요청 → 1 응답. REST와 동일일반 API 호출, 데이터 조회
Server Streaming1 요청 → N 응답 스트림로그 테일링, 실시간 주가, 검색 결과 스트리밍
Client StreamingN 요청 스트림 → 1 응답파일 업로드, 센서 데이터 수집, 배치 쓰기
BidirectionalN 요청 ↔ N 응답 동시 스트림채팅, 실시간 협업, 게임 상태 동기화
💡 gRPC Deadline/Timeout
gRPC는 Deadline propagation을 내장 지원한다. 클라이언트가 deadline을 설정하면 전체 호출 체인(A→B→C)에 자동 전파되어, deadline이 지난 요청은 중간에 취소된다. Distributed Systems 문서의 Timeout Chain 원칙이 프로토콜 레벨에서 구현된 것.
Critical

gRPC 로드밸런싱 문제

HTTP/2 단일 연결 → L4 LB가 한 서버에만 트래픽 몰리는 문제
gRPC는 HTTP/2의 단일 연결에서 모든 요청을 멀티플렉싱한다. L4 로드밸런서(K8s Service 기본)는 연결 레벨에서 분산하므로, 한 클라이언트의 모든 gRPC 호출이 같은 서버에만 전달되는 문제가 발생한다.
문제 시나리오
Client → K8s Service (L4 ClusterIP) → Pod A ↗ 모든 RPC가 같은 TCP 연결을 사용 ↗ L4 LB는 연결 시작 시 한 번만 Pod 선택 ↗ Pod A에 100% 트래픽, Pod B/C는 0% HTTP/1.1은 요청마다 새 연결 → L4로도 잘 분산 HTTP/2(gRPC)는 연결 재사용 → L4 분산 불가능
해결 방법
1) L7 로드밸런싱 (권장):
Envoy, Traefik, Nginx가 RPC 레벨에서 분산
K8s: Headless Service + 클라이언트 사이드 LB
또는 Istio/Linkerd가 L7 LB 자동 처리

2) 클라이언트 사이드 LB:
gRPC 클라이언트가 여러 서버 주소를 알고 직접 분산
Go: google.golang.org/grpc/balancer/roundrobin
K8s Headless Service로 Pod IP 목록 획득

3) Lookaside LB (외부 LB):
별도 LB 서비스가 서버 목록과 가중치를 클라이언트에 제공
gRPC xDS 프로토콜로 Envoy와 통합

Traefik Gateway API: GRPCRoute CRD로 gRPC 트래픽을 L7에서 라우팅 가능
⚠ K8s 기본 Service로 gRPC를 쓰면
ClusterIP Service는 L4(iptables/IPVS) 로드밸런싱이다. gRPC 클라이언트가 연결을 장시간 유지하면 새 Pod이 추가되어도 트래픽을 받지 못한다. 반드시 L7 프록시 또는 클라이언트 사이드 LB를 사용해야 한다. 이것이 gRPC + K8s에서 가장 흔한 실수.

프록시 gRPC/HTTP/2 설정

Traefik, Envoy에서 gRPC 트래픽을 올바르게 프록시하는 설정
Traefik gRPC 설정
IngressRoute (H2C — TLS 없이):

• Traefik이 TLS를 종료하고 백엔드에 H2C(HTTP/2 Cleartext)로 전달
• 백엔드가 gRPC를 H2C로 리스닝해야 함
• keepalive 설정: serversTransport.forwardingTimeouts.idleConnTimeout

grpc-health-probe 바이너리를 이미지에 포함해야 했음. 이제 네이티브 지원
