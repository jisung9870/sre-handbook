● Chaos Engineering Principles

Chaos Engineering 원칙

의도적으로 장애를 주입하여 시스템의 약점을 사전에 발견하는 학문.

Definition

Chaos Engineering이란

"프로덕션 환경에서 시스템의 신뢰성에 대한 확신을 구축하기 위해 통제된 실험을 수행하는 학문"
Chaos Engineering은 "테스트"가 아니다. 테스트는 "이 기능이 올바르게 동작하는가?"를 확인한다. Chaos Engineering은 "이 시스템이 예상치 못한 상황에서도 견디는가?"에 대한 확신을 구축하는 실험이다. Netflix가 2011년 Chaos Monkey로 시작하여 업계 표준이 되었다.
핵심 원칙 (Netflix)
1) Steady State에 대한 가설 수립:
"정상 상태에서 에러율은 0.1% 미만이다"

2) 실제 이벤트를 반영한 변수 도입:
서버 다운, 네트워크 지연, 디스크 가득 참 — 실제 발생 가능한 장애

3) 프로덕션에서 실험:
staging과 프로덕션은 다르다. 프로덕션에서 해야 진짜 약점을 발견

4) 자동화하여 지속적으로:
한 번의 실험이 아니라, CI/CD처럼 지속적으로 실행

5) Blast Radius 최소화:
실험이 실제 장애로 번지지 않도록 범위를 제한하고 중단 기준을 설정
Chaos ≠ 무작위 파괴
잘못된 이해: "서버를 마구 죽이는 것"
올바른 이해: "가설을 세우고, 통제된 환경에서 검증하는 과학적 실험"

Chaos Engineering의 가치:
• 장애 대응 능력 사전 검증
• Runbook이 실제로 동작하는지 확인
• 모니터링/알림이 장애를 감지하는지 검증
• 팀의 인시던트 대응 근육 훈련
• "우리 시스템은 이 장애를 견딜 수 있다"는 확신

SLO와의 연결: Error Budget이 충분할 때 Chaos 실험을 실행. Error Budget이 부족하면 실험보다 안정화에 집중
Process

Chaos 실험 프로세스

가설 → 실험 설계 → 실행 → 관찰 → 학습 — 과학적 방법론
5단계 프로세스
1. 가설 수립 "api-server Pod 1개가 죽어도 P99 latency는 500ms 이하를 유지할 것이다" 2. 실험 설계 - 장애 유형: Pod kill (1/3 replica) - 대상: production api-server Deployment - 지속 시간: 5분 - Blast radius: api-server만. DB/Redis 영향 없음 - Abort 기준: 에러율 > 5% 또는 P99 > 2s 3. 실행 - SLI 대시보드를 모니터링하면서 장애 주입 - Chaos Mesh PodChaos 적용 4. 관찰 - P99 latency 변화 기록 - HPA 반응 시간 기록 - 알림이 발생했는가? 올바른 채널로 왔는가? 5. 학습 & 개선 - 가설이 맞았는가? → Yes: 시스템이 견딤. 확신 증가 - 가설이 틀렸는가? → No: 약점 발견! → 개선 작업 → 재실험
Hypothesis

Steady State 가설

"정상 상태"를 SLI 기반으로 정량적으로 정의 — 실험의 성공/실패 기준
좋은 가설 vs 나쁜 가설
나쁜 가설:
"Pod이 죽어도 시스템이 잘 동작할 것이다"
→ "잘 동작"의 기준이 없음. 검증 불가

좋은 가설:
"api-server 3 replicas 중 1개가 강제 종료되어도, P99 latency는 500ms 이하를 유지하고, 에러율은 0.5%를 초과하지 않을 것이다. HPA가 60초 이내에 3 replicas를 복구할 것이다."
→ 구체적 SLI 기반. 측정 가능. 성공/실패 명확

Steady State 지표:
• Availability SLI (에러율)
• Latency SLI (P99)
• Throughput (RPS)
• 복구 시간 (MTTR)
가설 템플릿
[장애 유형]이 발생해도, [SLI 메트릭]은 [임계값] 이하를 유지하고, [복구 메커니즘]이 [시간] 이내에 동작할 것이다. 예시: "Redis primary 노드가 다운되어도, API 에러율은 1%를 초과하지 않고, Sentinel failover가 30초 이내에 완료될 것이다." "NAT Gateway 하나가 장애가 발생해도, outbound 트래픽은 다른 AZ의 NAT GW를 통해 30초 이내에 자동 failover될 것이다."
● Game Day

Game Day 설계와 실행

팀 전체가 참여하는 통제된 장애 시뮬레이션 — Chaos를 조직적으로 실행하는 방법.

Planning

Game Day 설계

시나리오 선택, 참가자, 일정, Abort 기준 — 사전 준비가 90%
구성 요소
시나리오: 어떤 장애를 시뮬레이션할 것인가? Postmortem에서 나온 장애 패턴이 가장 좋은 소스

참가자:
• Game Master: 장애를 주입하고 실험 진행 (보통 SRE)
• Incident Commander (연습): 인시던트 대응 연습
• 관찰자: SLI 대시보드, 알림, 팀 반응을 기록

일정:
• 사전 브리핑: 15분 (시나리오, 역할, Abort 기준)
• 실험 실행: 30~60분
• 디브리핑: 30분 (관찰 결과, 학습, 개선 사항)

Abort 기준:
"에러율 > 10% 또는 P99 > 5초가 3분 이상 지속되면 즉시 중단"
Solo DevOps에서의 Game Day
혼자서도 Game Day를 할 수 있다:

Mini Game Day (30분):
1. 시나리오 하나 선택
2. SLI 대시보드 열기
3. 장애 주입 (kubectl delete pod --random)
4. 관찰: 알림 오는가? HPA 반응하는가? 서비스 복구되는가?
5. 결과 기록 (Notion/Wiki)

월 1회 "자가 Game Day":
• 매월 첫 번째 수요일 오후
• 이전 달의 인시던트에서 시나리오 추출
• 혼자 하더라도 기록은 반드시

가장 간단한 첫 번째 실험:
kubectl delete pod -l app=api-server --field-selector=status.phase=Running
→ Pod 하나를 죽이고 복구를 관찰

● K8s Chaos Tools

K8s Chaos 도구

Chaos Mesh, Litmus — K8s 네이티브 장애 주입 도구와 실전 패턴.

Tool

Chaos Mesh (CNCF)

K8s 네이티브 Chaos Engineering 플랫폼 — CRD로 장애를 선언적으로 정의
지원 장애 유형
PodChaos: Pod kill, Pod failure, Container kill
NetworkChaos: 지연, 패킷 손실, 대역폭 제한, 파티션
IOChaos: 디스크 I/O 지연, I/O 에러 주입
StressChaos: CPU/Memory 스트레스 (리소스 압박)
DNSChaos: DNS 실패, 랜덤 IP 응답
TimeChaos: 시간 왜곡 (시계 밀기/당기기)
HTTPChaos: HTTP 응답 변조 (지연, 에러 코드)
KernelChaos: 커널 레벨 장애 주입

배포:
helm install chaos-mesh chaos-mesh/chaos-mesh -n chaos-mesh --create-namespace
실전 예시
# 네트워크 지연 주입 (200ms) apiVersion: chaos-mesh.org/v1alpha1 kind: NetworkChaos spec: action: delay mode: all selector: namespaces: [production] labelSelectors: app: api-server delay: latency: "200ms" jitter: "50ms" duration: "5m" scheduler: cron: "@every 24h"

5분간 지연 주입 후 자동 복구. 일 1회 자동 실행 가능
Alternative

Litmus (CNCF)

ChaosHub 기반 실험 카탈로그 — 사전 정의된 수백 개의 장애 시나리오
Litmus vs Chaos Mesh
Litmus:
• ChaosHub에서 사전 정의된 실험 선택 → 바로 실행
• 실험 결과를 Pass/Fail로 자동 판정
• CI/CD 파이프라인 통합에 강점
• GameDay 워크플로 기능 내장

Chaos Mesh:
• CRD 기반 더 세밀한 제어
• Dashboard UI 직관적
• Istio Fault Injection과 유사한 사용 경험
• 설정이 더 단순

권장: 소규모 팀에서는 Chaos Mesh가 더 가볍고 직관적. 대규모 팀에서 표준화된 실험 카탈로그가 필요하면 Litmus
Istio Fault Injection과의 비교
Istio Fault Injection (Service Mesh 문서 참조):
• VirtualService에서 HTTP 지연/에러 주입
• L7 레벨 (HTTP 요청 단위)
• Mesh가 있어야 사용 가능

Chaos Mesh:
• OS/커널 레벨 장애 주입 (네트워크, 디스크, CPU)
• L3/L4 레벨 (TCP 패킷 단위)
• Mesh 없이도 사용 가능

조합:
Istio: HTTP 응답 에러/지연 (앱 레벨 탄력성 테스트)
Chaos Mesh: 인프라 장애 (노드/네트워크/디스크 레벨 테스트)
두 도구가 다른 계층을 커버하므로 보완적으로 사용

● Pre-mortem

Pre-mortem & Failure Mode Analysis

장애가 발생하기 전에 "만약에" 연습 — Postmortem의 선행 버전.

Concept

Pre-mortem이란

"이 프로젝트/서비스가 실패했다고 가정하자. 왜 실패했을까?" — 인지 편향을 깨는 기법
인간은 낙관 편향을 가진다. "우리 시스템은 잘 동작할 것이다." Pre-mortem은 이 편향을 의도적으로 뒤집어, "이미 실패했다"를 전제로 원인을 역추적한다. Gary Klein의 연구에 의하면 Pre-mortem을 수행한 팀은 실패 가능성을 30% 더 많이 식별한다.
실행 방법
1) "6개월 후, 이 서비스가 P1 장애를 겪었다고 상상합시다"
2) 각자 5분간 "왜 실패했을까?" 목록 작성 (개인 브레인스토밍)
3) 돌아가며 공유. 중복 제거하지 않음 (같은 원인을 여러 명이 꼽으면 더 위험한 것)
4) 위험도 × 가능성으로 우선순위 매기기
5) 상위 3~5개에 대한 예방 조치(mitigation) 결정

Solo DevOps: 혼자서도 가능. 서비스 아키텍처 다이어그램을 보면서 각 컴포넌트가 "어떻게 실패할 수 있는가?" 목록 작성
적용 시점
새 서비스 런칭 전: PRR(Production Readiness Review)의 일부로 수행
대규모 변경 전: 마이그레이션, 클러스터 업그레이드, 아키텍처 변경 전
정기적: 분기 1회, 핵심 서비스에 대해
인시던트 후: Postmortem에서 나온 "다른 서비스에서도 비슷한 일이 일어날 수 있는가?" 확인

결과물: 위험 목록 + 각 위험의 예방 조치 + 담당자 + 기한 → Jira/Linear 티켓으로 추적
Analysis

Failure Mode Analysis

서비스의 모든 의존성을 나열하고, 각각이 실패할 때의 영향을 분석
의존성 매핑
api-server 의존성: [HARD] PostgreSQL — 없으면 전체 서비스 불가 [HARD] Redis — 세션 스토어. 없으면 로그인 불가 [SOFT] Payment API — 없어도 조회는 가능 [SOFT] Notification — 없어도 핵심 기능 동작 [INFRA] CoreDNS — 없으면 모든 연결 실패 [INFRA] NAT Gateway — 없으면 외부 통신 불가

HARD: 없으면 서비스 불가능. Circuit Breaker + Fallback 필수
SOFT: 없어도 핵심 기능은 동작. Graceful Degradation 설계
INFRA: 인프라 레벨 의존성. 이중화/Multi-AZ로 방어
각 Failure Mode의 질문
각 의존성에 대해:

1) 어떻게 실패할 수 있는가?
DB: 커넥션 풀 고갈, 슬로 쿼리, 디스크 가득 참, 복제 지연

2) 실패하면 우리 서비스에 어떤 영향?
전체 다운? 일부 기능 불가? 느려짐?

3) 어떻게 감지하는가?
메트릭 알림? 헬스체크? 사용자 보고?

4) 어떻게 완화하는가?
Retry? Circuit Breaker? Fallback? Graceful Degradation?

5) 복구 계획은?
자동 복구? Runbook? 수동 개입?

이 분석을 서비스별로 문서화하면 PRR 문서가 되고, Runbook의 기반이 된다
● Chaos Maturity

Chaos Engineering 성숙도

현재 어디에 있고 다음 단계는 무엇인가 — Solo DevOps에서 시작하는 현실적 로드맵.

Model

Chaos Engineering 성숙도 모델

Ad-hoc → Manual → Automated → Advanced → Culture — 5단계 진화
5단계
Level 0 — 없음: Chaos 실험 없음. 장애를 사후에만 대응. 대부분의 팀이 여기.

Level 1 — Ad-hoc: 가끔 수동으로 Pod 삭제 테스트. 기록 없음. 비체계적.

Level 2 — 수동 Game Day: 분기 1회 Game Day 실행. 시나리오 문서화, 결과 기록. 대부분의 조직이 여기를 목표로

Level 3 — 자동화: Chaos Mesh/Litmus로 정기적 자동 실험. CI/CD 통합. SLI 기반 자동 판정.

Level 4 — 문화: 모든 신규 서비스의 PRR에 Chaos 실험 포함. 팀 전체가 Chaos를 일상으로. Netflix 수준.
Practical

Solo DevOps에서 Chaos 시작하기

혼자서도 할 수 있는 현실적인 첫 걸음
Phase 1: 관찰 (지금 바로)
도구 설치 없이 시작:

1) 핵심 서비스의 의존성 맵 그리기 (Failure Mode Analysis)
2) 각 의존성의 "어떻게 실패하는가?" 목록 작성
3) 현재 모니터링/알림이 이 실패를 감지하는지 확인
4) Runbook이 이 실패를 커버하는지 확인

이것만으로도 가치가 크다. "모니터링이 감지하지 못하는 장애"를 발견하게 됨
Phase 2~3: 실험 & 자동화
Phase 2 — kubectl 기반 수동 실험:
• kubectl delete pod → Pod 자가 치유 확인
• kubectl drain node → PDB 동작 확인
• kubectl scale --replicas=1 → 단일 replica 장애 취약점
• 매월 30분 "Mini Game Day"

Phase 3 — Chaos Mesh 도입:
• staging에 Chaos Mesh 설치
• NetworkChaos로 서비스 간 지연 주입
• PodChaos로 정기적 Pod kill 자동화
• Grafana Annotation으로 실험 시점 표시
• 결과를 SLI 대시보드와 함께 분석

목표: Level 2 (분기 Game Day + 기록) → 이것만 해도 상위 10% 조직
💡 Chaos와 Error Budget의 관계
Error Budget이 충분할 때: Chaos 실험을 적극적으로 실행. 시스템의 약점을 찾아 개선
Error Budget이 부족할 때: Chaos 실험을 중단하고 안정화에 집중. 알려진 약점을 먼저 수정
→ Error Budget 정책 문서에 "Error Budget > 50%일 때만 Chaos 실험 허용" 같은 규칙을 명시하면 체계적
