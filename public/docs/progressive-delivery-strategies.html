● Deployment Strategies

배포 전략 — 기본기

Recreate, Rolling, Blue/Green, Canary — 각 전략의 트레이드오프와 DB 마이그레이션 같은 까다로운 케이스 처리법.

Compare

4가지 기본 전략 — 무엇을 언제 쓸까

downtime, 리소스 비용, 롤백 속도, 사용자 영향 — 4개 축으로 비교
"어떤 배포 전략을 쓸까"는 리스크 허용도와 비용의 함수다. 사용자 0명인 사내 도구는 Recreate로 충분하고, 결제 시스템은 Canary가 필수다. 한 조직에서도 서비스마다 다른 전략을 쓰는 게 정상이다.
전략Downtime리소스 (peak)롤백 속도부분 노출적합한 상황
Recreate있음 (수십 초~분)1x재배포 필요X (전부 또는 무)개발 환경, stateful 단일 인스턴스
Rolling없음 (이상적)1.25x ~ 1.5x새 rolling 필요 (느림)X (점진 교체지만 모두 같은 코드)일반 API 서비스 — 가장 흔한 선택
Blue/Green없음2x (전체 복제)즉시 (트래픽 스위치)X큰 변경, 즉시 롤백 필요한 미션 크리티컬
Canary없음1.05x ~ 1.5x즉시 (canary 제거)O (1% → 5% → 25% ...)대규모 사용자, 실험적 변경, 결제 시스템

리소스 1.5x의 의미

Rolling 1.5x = 기존 100 pod + 새 50 pod 동시 운영(maxSurge=50%) 시점이 있음. 핵심 비용 고려: peak 시점에 클러스터 capacity가 충분한가. CA(cluster autoscaler) 응답 시간 (수 분)을 고려.

Canary가 만능인가?

아니다. 다음 경우 Canary 부적합: (1) 사용자 수 ≤ 100 — 통계적 분석 불가, (2) DB 스키마 변경 동반, (3) 세션 sticky 필수 — canary 전환 시 사용자가 두 버전 사이 왔다 갔다.

Default

Rolling Update — k8s의 기본값을 안전하게

maxSurge / maxUnavailable / readinessProbe — 3가지 노브를 정확히 이해하라
k8s Deployment의 기본 전략. 동작은 단순해 보이지만 readinessProbe 설정이 잘못되면 트래픽이 안 받는 pod에 요청이 흘러간다. Rolling 사고의 90%는 probe 설정 실수다.
표준 Rolling 설정
apiVersion: apps/v1 kind: Deployment spec: replicas: 10 strategy: type: RollingUpdate rollingUpdate: maxSurge: 25% # 최대 12.5개까지 동시 존재 (10 + 2.5) maxUnavailable: 0 # 0이 안전 — capacity 줄지 않음 minReadySeconds: 30 # Ready 후 30초간 더 관찰 (warm-up) progressDeadlineSeconds: 600 # 10분 안에 안 끝나면 실패로 마킹 template: spec: containers: - name: app readinessProbe: httpGet: { path: /ready, port: 8080 } initialDelaySeconds: 10 periodSeconds: 5 failureThreshold: 3 lifecycle: preStop: exec: { command: ["sleep", "15"] } # graceful shutdown terminationGracePeriodSeconds: 60
readinessProbe — Live 상태 분리
liveness: "이 pod 죽었나? (재시작 필요)"
readiness: "이 pod 트래픽 받을 준비 됐나?"

두 개를 같은 endpoint로 쓰면 안 된다. /health 하나로 통합하면, 한순간 DB 끊김에 모든 pod가 다 unready가 되어 503 폭발.

분리 원칙:
· liveness: 프로세스 자체 헬스 (deadlock 등)
· readiness: 의존성 + 큐 길이 + warm-up 완료
PreStop & graceful shutdown
Pod 종료 시 순서:
1. SIGTERM → 컨테이너에 전송 2. preStop hook 실행 (병렬: endpoint 제거) 3. terminationGracePeriodSeconds 카운트다운 4. SIGKILL (강제 종료)
endpoint 제거는 비동기다. SIGTERM 받자마자 종료하면 아직 트래픽 받는 중. preStop sleep 15로 endpoint 제거 propagate 시간 확보. graceful shutdown은 sleep 후에야 시작
실전 사고 — Rolling이 다운타임을 만드는 패턴
1. readinessProbe 빠짐 — Pod가 시작 즉시 Ready. warm-up 안 끝났는데 트래픽 받음 → 첫 요청 타임아웃
2. PDB 미설정 — node drain과 deployment가 겹쳐 가용 pod ≤ 1
3. maxUnavailable: 25% — capacity 75%로 떨어진 순간 부하 spike → 503
4. preStop 없음 — endpoint 제거 전 pod 종료 → 인 flight 요청 끊김
5. liveness가 너무 공격적 — DB 잠시 느려지면 모든 pod kill, cascading restart

방어: maxUnavailable: 0 + preStop + 분리된 readiness/liveness + PDB
B/G

Blue/Green — 즉시 롤백의 대가

전체 복제 환경을 만들고 트래픽을 한 번에 스위치 — 비용은 2x, 안전성은 최고
Blue(현 운영) 옆에 Green(신 버전)을 완전 배포 → 검증 후 LB/Service에서 selector 변경으로 트래픽 전환. 롤백은 selector 되돌리기 한 번으로 즉시. 비용은 2x이지만, 큰 변경이나 복구 시간이 비즈니스 손실로 직결되는 경우엔 정당화된다.
k8s Service selector swap (Argo Rollouts BlueGreen)
apiVersion: argoproj.io/v1alpha1 kind: Rollout metadata: name: payment-api spec: replicas: 10 strategy: blueGreen: activeService: payment-api-active # prod 트래픽 받는 service previewService: payment-api-preview # 검증용 service autoPromotionEnabled: false # 수동 승인 필수 scaleDownDelaySeconds: 600 # 전환 후 10분간 old 유지 (롤백용) prePromotionAnalysis: # 전환 전 자동 검증 templates: - templateName: smoke-test postPromotionAnalysis: # 전환 후 모니터링 templates: - templateName: error-rate-check
Blue/Green이 적합한 경우
1. 큰 아키텍처 변경 — 새 frame, 새 ORM, 새 언어 — Rolling으로 두 버전이 동시 운영하면 호환성 문제
2. 미션 크리티컬 — 결제, 인증, 핵심 거래 — 5분 이상 다운타임 = 중대한 손실
3. 빠른 전후 비교 필요 — preview env에서 충분한 검증 후 cutover
4. 외부 통합 테스트 — preview env로 외부 파트너가 테스트 가능
B/G의 함정
1. DB 공유 문제 — Blue와 Green이 같은 DB 쓰면 스키마 변경 시 양쪽 호환 필수 (다음 섹션 참고)
2. 캐시 cold start — 새 환경은 캐시가 비어 있음 → 전환 직후 latency spike. preview에서 trial 트래픽으로 warm-up
3. 세션 — sticky 의존 — sticky session 쓰면 전환 후 사용자 세션 끊김
4. 비용 — 2x 리소스. 작은 서비스 다수보다 큰 서비스 하나에 적합
Hard

DB 마이그레이션과 무중단 배포의 충돌

스키마를 바꾸면서 어떻게 두 버전(old/new)이 같은 DB를 쓰게 할 것인가
Rolling이든 Canary든 잠시 두 버전이 공존한다. 즉 한 DB가 두 코드 버전을 동시에 만족해야 한다. 컬럼 삭제, 컬럼 이름 변경 같은 변경은 한 번에 하면 안 된다. 항상 Expand → Migrate → Contract 3단계로 분리한다.
컬럼 이름 변경 — `user_email` → `email`
# 한 번에 하면 안 되는 변경: # ALTER TABLE users RENAME COLUMN user_email TO email; # → 배포 중간 시점에 old 코드가 user_email을 읽으면 에러 ─── Phase 1: EXPAND (deploy v2) ─────────────────────────────── ALTER TABLE users ADD COLUMN email VARCHAR(255); # 트리거로 양쪽 동기화 CREATE OR REPLACE FUNCTION sync_email() RETURNS trigger AS $$ BEGIN NEW.email := COALESCE(NEW.email, NEW.user_email); NEW.user_email := COALESCE(NEW.user_email, NEW.email); RETURN NEW; END; $$ LANGUAGE plpgsql; # 코드 v2: 양쪽 모두 읽고, 양쪽 모두 씀 ─── Phase 2: MIGRATE (백필) ────────────────────────────────── UPDATE users SET email = user_email WHERE email IS NULL; # 코드 v3: 새 컬럼만 읽고/씀 ─── Phase 3: CONTRACT (deploy v4) ───────────────────────────── ALTER TABLE users DROP COLUMN user_email; DROP TRIGGER sync_email;
대규모 백필 주의사항
1억 row 테이블 UPDATE는 락을 잡고 replica lag을 만든다. 배치 백필:
DO $$ DECLARE batch_size INT := 5000; affected INT; BEGIN LOOP UPDATE users SET email = user_email WHERE email IS NULL AND id IN ( SELECT id FROM users WHERE email IS NULL LIMIT batch_size ); GET DIAGNOSTICS affected = ROW_COUNT; EXIT WHEN affected = 0; PERFORM pg_sleep(0.1); -- replica lag 회피 COMMIT; END LOOP; END $$;
위험한 변경 패턴
1. NOT NULL + DEFAULT 추가 (큰 테이블) — 전체 락 위험. Postgres 11+ ADD COLUMN ... DEFAULT는 metadata-only, 그 이전 버전은 위험
2. 인덱스 추가 — 항상 CREATE INDEX CONCURRENTLY (Postgres). 그렇지 않으면 write lock
3. ENUM 변경 — Postgres ENUM은 ALTER 어려움. lookup 테이블이나 CHECK constraint 권장
4. 외래키 추가 — NOT VALID로 추가 후 VALIDATE CONSTRAINT로 비차단 검증
자동화 — 마이그레이션 도구
도구 선택:
· Flyway — SQL 파일 버전 관리, 단순
· Liquibase — XML/YAML, 롤백 지원
· Atlas (ariga) — declarative schema, dev/prod diff
· sqlx migrate (Rust), knex (Node) — 언어별 도구

운영 원칙:
1. 마이그레이션은 app과 별도 job으로 — 배포와 동시 X
2. idempotent 보장 — 중간 실패해도 다시 돌릴 수 있음
3. 롤백 스크립트 필수 — DROP COLUMN의 역은 ADD COLUMN 후 백필
4. 리허설 — staging에서 prod 데이터 크기로 timing 측정
● Canary Release

Canary 심화 — 점진적 위험 노출

Canary의 본질은 "전체 영향 전에 작은 모집단에서 새 코드의 안전성을 측정"하는 것. 트래픽 분기 메커니즘과 자동 분석이 핵심.

Anatomy

Canary 배포의 구조

"가장 작은 모집단부터 점진 확대" — 각 단계에서 검증 후 다음 단계로
Canary는 "작은 비율 → 측정 → 다음 비율"의 반복이다. 단계 수, 비율, 단계별 시간, 측정 메트릭, 자동/수동 승인 — 이 5가지가 모든 canary 시스템의 공통 노브다.
표준 단계 예시
Step 1: 1% canary, 10분 측정 Step 2: 5% canary, 10분 측정 Step 3: 25% canary, 15분 측정 Step 4: 50% canary, 15분 측정 Step 5: 100% (promotion)
총 60분의 점진적 노출. 단계마다 메트릭 게이트 통과해야 다음 단계 진입. 한 단계라도 실패면 즉시 abort + 롤백
단계 설계 원칙
처음은 작게: 1% — 통계적으로 의미 있으면서 영향 최소
비율 증가는 비선형: 1, 5, 25, 50 — 초기엔 보수적, 후반엔 빠르게
단계 시간은 진단 가능 시간: 캐시 warm-up(5분) + 측정 윈도우(10분) 이상
최소 트래픽 임계: 1%가 RPS 10도 안 되면 통계 무의미 — 절대값 floor 설정
측정 메트릭 — 무엇을 볼까
SLI 기반 4개 + α:
1. Error rate — HTTP 5xx, RPC 실패 (canary vs baseline)
2. Latency p95/p99 — 회귀 시 즉시 abort
3. Saturation — CPU/memory 비정상 상승
4. Traffic — canary로 트래픽이 정상적으로 흐르는가

비즈니스 메트릭:
· 결제 성공률
· 로그인 성공률
· 핵심 funnel 통과율

주의: canary와 baseline은 같은 시간대, 같은 사용자 분포여야 비교 의미. 1%만 받으면 RPS 작아 노이즈 큼 — Mann-Whitney U test 같은 통계 검정 필요
Routing

트래픽 분기 메커니즘

replica 비율 기반 vs Service Mesh 기반 — 정밀도와 복잡도의 트레이드오프
"1% 트래픽을 canary로" 라는 단순한 명세 뒤에는 다양한 구현이 있다. replica 비율(가장 단순), L7 weighted routing(정밀, mesh 필요), header 기반(특정 사용자 그룹만) — 정밀도가 높을수록 인프라 요구사항도 높다.
방식정밀도요구사항장점단점
Replica 비율~replica 단위k8s만단순, 의존성 없음1% canary 못 함 (replica 100개 필요), 부하 분산이 비율과 어긋날 수 있음
Mesh weighted (Istio/Linkerd)임의 % (정밀)Service Mesh정밀, replica 수 무관mesh 운영 부담
Ingress weighted (NGINX/Traefik)임의 %특정 ingress controllermesh 없이 가능edge에서만 분기 (서비스 간 호출은 분기 X)
Header 기반 (sticky)사용자 단위mesh 또는 app 로직특정 코호트만, sticky실제 비율 측정 어려움
Feature Flag 기반사용자/기능 단위flag 시스템코드 레벨 제어, A/B 가능app 코드에 logic 침투
Istio VirtualService로 가중치 분기
apiVersion: networking.istio.io/v1 kind: VirtualService metadata: name: payment-api spec: hosts: [payment-api] http: - match: # 내부 직원은 무조건 canary로 (early validation) - headers: x-employee: { exact: "true" } route: - destination: { host: payment-api, subset: canary } - route: # 일반 트래픽: 95% stable, 5% canary - destination: { host: payment-api, subset: stable } weight: 95 - destination: { host: payment-api, subset: canary } weight: 5
WARN Sticky session이 필요한 서비스(쇼핑카트, multi-step form)는 canary 시 사용자 단위 sticky가 필수. 그렇지 않으면 사용자가 페이지 간 이동 중 두 버전을 오가며 카트가 깨진다. x-canary-cohort 같은 hash-based 헤더로 사용자 → 버전 매핑 고정.
Analysis

자동 분석 (Automated Analysis)

사람이 매번 메트릭을 보고 판단하면 canary는 의미 없다 — 자동화가 본질
Canary의 가치는 "자동화된 게이트"에 있다. 사람이 메트릭 대시보드를 30분간 응시하다가 "OK, 다음 단계"를 누르는 거라면 그건 canary가 아니라 "느린 배포"다. 자동 분석은 매 단계마다 측정 → 통계 검정 → 결정을 무인으로 수행한다.
분석 방법론
1. 임계치 기반 (Threshold)
· "canary error rate < 0.5%이면 통과"
· 단순, 해석 쉬움
· baseline 변동에 약함

2. 차이 비교 (Diff)
· "canary error rate가 baseline 대비 +0.1% 이내"
· 시간대 변동 흡수
· 트래픽 작으면 노이즈 큼

3. 통계 검정 (Statistical)
· Mann-Whitney U, Kolmogorov-Smirnov
· 두 분포가 같다는 가설 검증
· 정확하나 데이터 양 필요
메트릭 소스
Prometheus — 가장 흔함:
sum(rate(http_requests_total{ app="payment", version="canary", status=~"5.." }[5m])) / sum(rate(http_requests_total{ app="payment", version="canary" }[5m]))
canary 라벨로 분리해 query. Argo Rollouts, Flagger 모두 PromQL을 직접 받음

다른 소스: Datadog, New Relic, Stackdriver — Argo Rollouts에 provider plugin
실전 — Argo AnalysisTemplate
apiVersion: argoproj.io/v1alpha1 kind: AnalysisTemplate metadata: name: error-rate spec: args: - name: service-name metrics: - name: error-rate interval: 1m successCondition: result[0] <= 0.005 failureLimit: 3 # 3회 연속 실패 시 abort provider: prometheus: address: http://prometheus.monitoring:9090 query: | sum(rate(http_requests_total{ service="{{args.service-name}}", version="canary", status=~"5.." }[1m])) / sum(rate(http_requests_total{ service="{{args.service-name}}", version="canary" }[1m]))
핵심 노브:
· interval: 측정 주기
· successCondition: 통과 조건
· failureLimit: 연속 실패 허용
· inconclusiveLimit: 데이터 부족 시
Failure

Abort & Rollback — 실패의 우아한 처리

canary 실패는 흔하다. 빠르고 깔끔하게 되돌리는 메커니즘이 더 중요하다
완벽한 canary 시스템은 실패가 정상이라고 가정한다. canary 10번 중 1-2번 실패는 정상 — 그게 canary의 목적이니까. 핵심은 실패 시 사용자 영향 최소화다.
Abort 흐름
1. AnalysisRun 실패 감지 2. canary pod로의 트래픽 0%로 즉시 변경 3. canary replicaset 0으로 scale down (선택) 4. Slack/PagerDuty 알림 5. Rollout 상태: Degraded 6. 운영자 확인 → 원인 분석 → 재배포
주의: canary pod를 즉시 죽이지 말 것 — 디버깅 자료 유지. 트래픽만 차단하고 pod는 유지하면 로그/profile 분석 가능
자동 vs 수동 abort
자동 abort 적합:
· 5xx error rate 급등 (명확한 신호)
· latency p99 2배 이상
· readiness 실패 다수

수동 검토 적합:
· 비즈니스 메트릭 미세 변동 (전환율 -1%)
· 사용자 피드백 기반
· 통계 유의성 애매한 경우

혼합: 명확한 SRE 메트릭은 자동, 비즈니스 메트릭은 알림만 + 수동 결정
Forward fix vs Rollback — 어느 쪽?
롤백 우선 (default): 알려진 안전한 버전으로 즉시 복귀. 원인 분석은 그 후

Forward fix 적합한 경우:
· 데이터 마이그레이션이 이미 진행 — 롤백이 더 위험
· 보안 패치 — 이전 버전이 더 위험
· 외부 API 호환성 변경 — 이전 버전이 외부와 불일치

원칙: "정말 forward fix가 필요한가?" 의심부터. 90% 케이스는 롤백이 답. forward fix는 새 변경을 또 prod에 푸는 거라 또 다른 사고 위험

롤백 자체의 함정: "롤백은 안전"이라는 가정이 항상 맞지는 않음. DB 스키마 변경 후 롤백 시 old 코드가 새 스키마를 못 다룰 수 있음 → expand/contract 설계가 필수

Argo Rollouts — k8s native progressive delivery

Rollout CRD, AnalysisTemplate, Experiment, Service Mesh 통합 — Kubernetes 환경에서 가장 널리 쓰이는 progressive delivery 도구.

CRD

Rollout CRD — Deployment의 확장

Argo Rollouts는 Deployment를 대체하는 Rollout CRD를 제공. 기존 Deployment YAML과 거의 호환
Argo Rollouts는 Deployment의 strategy.type을 RollingUpdate/Recreate에서 Canary/BlueGreen으로 확장한다. spec 구조는 거의 동일해 마이그레이션이 쉽다 — kind: Deployment를 kind: Rollout로 바꾸고 strategy만 추가하면 된다.
Canary Rollout 표준 정의
apiVersion: argoproj.io/v1alpha1 kind: Rollout metadata: name: payment-api spec: replicas: 10 strategy: canary: canaryService: payment-api-canary # canary 트래픽용 service stableService: payment-api-stable # stable 트래픽용 service trafficRouting: istio: virtualService: name: payment-api-vs routes: [primary] steps: - setWeight: 5 - pause: { duration: 10m } - analysis: templates: - templateName: error-rate args: - name: service-name value: payment-api - setWeight: 25 - pause: { duration: 10m } - analysis: templates: [{ templateName: error-rate, args: [...]}] - setWeight: 50 - pause: {} # 무기한 — 수동 promotion scaleDownDelaySeconds: 300 # promotion 후 5분간 old 유지 selector: { matchLabels: { app: payment-api } } template: # Deployment.spec.template과 동일 spec: { containers: [...] }
step 종류
setWeight: 트래픽 비율 변경
pause: 일정 시간 또는 무기한 대기
analysis: AnalysisTemplate 실행
experiment: 단기간 실험 (별도 pod)
setCanaryScale: replica 수 직접 제어

실전 패턴: setWeight + pause + analysis 3종 세트의 반복
CLI & UI
kubectl plugin:
kubectl argo rollouts get rollout payment-api kubectl argo rollouts promote payment-api # 다음 step 진행 kubectl argo rollouts abort payment-api # 현재 진행 중단 kubectl argo rollouts retry payment-api # 실패한 rollout 재시도 kubectl argo rollouts pause payment-api # 일시정지
UI: kubectl argo rollouts dashboard → localhost:3100. progress 시각화, manual promote 버튼
Template

AnalysisTemplate — 재사용 가능한 측정 정의

"error rate < 0.5%" 같은 게이트를 한 번 정의하고 모든 rollout에서 재사용
조직 표준 게이트(error rate, latency, saturation)는 AnalysisTemplate으로 한 번 정의해 모든 rollout에서 참조한다. 서비스별로 임계치만 args로 다르게 주면 된다.
조직 표준 AnalysisTemplate 세트
# 1) Error rate 게이트 apiVersion: argoproj.io/v1alpha1 kind: ClusterAnalysisTemplate # 모든 namespace에서 참조 가능 metadata: { name: error-rate } spec: args: - name: service - name: threshold value: "0.005" metrics: - name: error-rate interval: 1m count: 10 # 10번 측정 successCondition: "result[0] <= {{args.threshold}}" failureLimit: 3 # 3번 실패하면 abort provider: prometheus: address: http://prometheus.monitoring:9090 query: | sum(rate(http_requests_total{service="{{args.service}}",version="canary",status=~"5.."}[2m])) / sum(rate(http_requests_total{service="{{args.service}}",version="canary"}[2m])) # 2) Latency p99 게이트 # 3) Saturation 게이트 # ... (조직 공용)
TIP ClusterAnalysisTemplate으로 정의하면 모든 namespace의 rollout이 참조 가능. 플랫폼 팀이 표준 게이트 셋트를 관리하고, 서비스 팀은 args만 채워 사용 — 일관성과 유지보수 둘 다 잡힘.
Experiment

Experiment — 일회성 실험 단위

canary와 별개로 "실험용 pod"를 띄워 짧게 검증 — 본 트래픽에 영향 없이 가설 검증
Canary는 실제 사용자 트래픽을 통해 검증한다. Experiment는 별도 pod를 띄워 (1) 합성 트래픽 또는 (2) 미러된 트래픽을 받아 짧게 측정한다. 본 사용자 영향이 0이라 더 공격적인 실험이 가능하다.
Experiment 정의
apiVersion: argoproj.io/v1alpha1 kind: Experiment metadata: { name: payment-perf-test } spec: duration: 15m templates: # 실험에 띄울 pod 종류 - name: baseline replicas: 2 spec: containers: - image: payment:v1.0 - name: experimental replicas: 2 spec: containers: - image: payment:v1.1-experimental analyses: # 두 그룹 비교 분석 - name: latency-comparison templateName: compare-latency args: - name: baseline value: baseline - name: experimental value: experimental

A/B

A/B Testing 모드

트래픽 비율이 아닌 헤더/쿠키 매칭으로 사용자 코호트 단위 분기
Canary는 "임의의 5% 트래픽"이지만, A/B test는 "특정 사용자 그룹"에 새 버전 노출 — sticky하고, 비즈니스 메트릭 측정에 적합. Flagger는 같은 Canary CRD에 match 조건을 추가해 A/B 모드로 전환할 수 있다.
A/B Testing 모드
spec: analysis: interval: 1m threshold: 10 iterations: 10 # 10회 측정 후 promotion match: # 트래픽 매칭 조건 - headers: x-canary: exact: "insider" # 내부 직원만 새 버전 - headers: cookie: regex: "^(.*?;)?(canary=true)(;.*)?$" metrics: - name: conversion-rate thresholdRange: { min: 95 } # 비즈니스 메트릭 interval: 1m
NOTE A/B는 "새 기능이 비즈니스 KPI에 어떤 영향?"을 묻는 도구. canary가 "안전한가?"라면 A/B는 "더 좋은가?". 두 개를 혼동하지 말 것 — 보통 둘 다 필요하다 (canary로 안전 확보, A/B로 효과 측정).

Feature Flags — Deploy ≠ Release

코드 배포와 기능 노출을 분리하는 도구. Dark launch, Kill switch, percentage rollout, A/B test 모두 같은 메커니즘 위에서 동작.

Types

플래그의 4가지 유형

모든 플래그가 같은 게 아니다 — 수명과 목적이 다르면 관리 방식도 달라야 한다
유형수명목적제거 시점
Release toggle주~수개월점진적 노출 (1%→100%)100% 도달 후 즉시
Experiment toggle주A/B 테스트실험 종료 + 결정 후
Ops toggle (Kill switch)영구장애 시 기능 차단제거하지 않음 (유지)
Permission toggle영구사용자별 기능 권한제거하지 않음 (정책)
유형 구분의 중요성
"수명이 짧은" 플래그(Release/Experiment)와 "영구" 플래그(Ops/Permission)를 같은 시스템에서 관리하면 혼란 발생.

안티패턴: 1년 전 만든 release toggle이 아직 코드에 남아 있음 → 죽은 코드 양산

해법: 플래그마다 유형 + 만료일 기록. release toggle은 만료 30일 전부터 알림
Targeting Rule
단순 on/off 외에 다양한 조건 매칭:
{ "key": "new-checkout-flow", "rules": [ { "if": "user.country == 'KR'", "value": true }, { "if": "user.tier == 'enterprise'", "value": true }, { "if": "user.id % 100 < 5", "value": true }, // 5% { "default": "value": false } ] }
SDK가 사용자 컨텍스트와 룰을 매칭해 평가
Pattern

Dark Launch — 코드는 배포, 기능은 비공개

Trunk-based development의 핵심 — feature branch 없이 main에 머지하되 사용자에겐 안 보임
Long-lived feature branch는 머지 충돌 지옥과 통합 지연을 유발한다. Dark launch는 미완성 기능도 main에 머지하되 if (flag.isOn("new-feature")) {...}로 사용자에겐 가려진 채로 prod에 배포하는 패턴이다.
실전 예시
// 사용자가 보는 코드 if (await flags.isOn('new-checkout-v2', { user })) { return renderCheckoutV2(); } else { return renderCheckoutV1(); } // 백엔드도 동일하게 if (await flags.isOn('use-new-payment-api', { user })) { return await newPaymentApi.charge(amount); } else { return await legacyPaymentApi.charge(amount); }
flag off 상태로 main에 머지 → CI 통과 → prod 배포 → 충분히 검증 후 단계적으로 flag on
Dark launch의 단계
1. Build (off) — 코드 작성 + main 머지. 사용자에게 안 보임
2. Internal (employees) — 직원에게만 on. dogfooding
3. Beta (1-5%) — 소수 사용자. 메트릭 관찰
4. Gradual (5→25→50→100%) — 점진적 확대
5. Cleanup — 100% 도달 후 if 분기 + flag 모두 제거. 죽은 legacy 코드 삭제

각 단계에서 멈추거나 되돌릴 수 있음 — 배포가 아닌 "릴리즈"의 통제권 확보
Backend dark launch — Shadow Traffic
"새 알고리즘을 prod에 도입하기 전에 실제 트래픽으로 검증"하고 싶을 때:
// 사용자에게는 old 결과 반환, 새 알고리즘은 background에서 비교용 실행 const oldResult = await oldAlgorithm(input); if (await flags.isOn('shadow-new-algo', { user })) { // 비동기로 새 알고리즘 실행, 결과 비교 + 메트릭 기록 newAlgorithm(input).then(newResult => { metrics.record('algo_diff', { old: oldResult, new: newResult }); if (!equals(oldResult, newResult)) logger.warn(...); }).catch(err => metrics.record('algo_error')); } return oldResult; // 사용자는 항상 old 결과
새 알고리즘이 사용자에게 영향 없이 실제 부하/데이터로 검증됨. 차이 0이 되면 cutover
Ops

Kill Switch — 장애 시 차단 스위치

새 기능에만 쓰는 게 아니다. 기존 핵심 기능도 비상시 차단할 수 있어야 한다
결제, 추천, 알림 같은 핵심 기능에 "이 기능을 잠시 비활성화"할 수 있는 ops toggle을 미리 박아두자. 장애 시 코드 변경 + 배포 없이 1초 만에 영향 차단 가능 — MTTR이 분 단위에서 초 단위로 단축.
Kill switch 패턴
// 외부 추천 서비스 호출 — 가끔 죽으면 전체 페이지 응답 느려짐 async function getRecommendations(user) { if (await flags.isOn('recommendations-killswitch')) { metrics.increment('recommendations.disabled'); return []; // 빈 결과로 graceful degradation } try { return await recommendationService.fetch(user); } catch (err) { metrics.increment('recommendations.error'); return []; } }
어디에 kill switch를 박을까
외부 의존: 다른 서비스/외부 API 호출 — 그 의존이 죽으면 격리
비싼 연산: ML inference, 큰 쿼리 — 부하 spike 시 차단
최근 변경: 신규 기능은 6개월간 kill switch 유지 → 안정화 확인 후 제거 검토
운영 위험: 결제, 인증 같은 자체 — 더 큰 영향 막기 위한 마지막 카드

핵심 코드 경로마다 1개씩 박는 게 운영 부담 만드는 게 아니라, 운영 자유도를 높인다
Kill switch의 함정
1. Flag 시스템 자체 의존 — flag service가 죽으면 kill switch 못 씀. SDK fallback default를 안전한 값으로 (true/false 명확)
2. 캐시 stale — flag 변경이 즉시 반영 안 됨. SDK polling 주기 ≤ 30초
3. 문서화 부족 — runbook에 "어떤 상황에 어떤 kill switch" 명시. 새벽 3시에 어떤 스위치를 누를지 모르면 의미 없음
4. 정기 점검 안 함 — 6개월에 한 번 staging에서 모든 kill switch 토글해서 동작 검증
Debt

플래그 부채 관리

플래그는 부채다 — 만들 때 즐겁고, 안 갚으면 코드를 갉아먹는다
Feature flag는 강력하지만 코드 분기를 영구화한다. 100개 release flag가 코드에 남아 있으면 모든 함수가 if-else 지옥. 1년 후 어떤 플래그가 살아있는지 아무도 모른다 — 새 사람 입사 시 "이 코드 뭐예요?"의 90%는 죽은 플래그.
방지 — 만들 때부터 청산 계획
1. 만료일 필수 — 플래그 생성 시 expiration 입력. release toggle은 기본 30일
2. Owner 명시 — CODEOWNERS 매핑. 만료 시 owner에게 자동 PR
3. Linter — flag-cleanup-bot: 100% rollout된 flag를 코드에서 자동 제거 PR 생성
4. 분기당 정리 데이 — 분기마다 reading day 1일 = 죽은 flag 제거

"나중에 정리하자"는 영원히 정리 안 한다. 청산 자동화가 핵심
Detection — 죽은 플래그 찾기
지표:
· 100% rollout 상태로 30일 이상 — 그냥 코드에 박아도 됨
· 0% rollout 상태로 90일 이상 — 안 쓰는 코드, 그냥 삭제
· 마지막 evaluation으로부터 60일 — 코드에서 호출 안 됨

SDK 운영 — Flag eval의 안전성
앱 내에서 매 요청 flag eval은 hot path다. 잘못 쓰면 latency 증가 또는 SPOF.

1. 로컬 평가 (streaming) — SDK가 모든 flag 정의를 메모리에 보유. 외부 호출 없이 평가. LaunchDarkly/Unleash 모두 지원
2. Bootstrap fallback — flag service 첫 연결 실패 시 fallback 값 (env var, 디스크 cache)
3. Default 안전성 — 모든 isOn() 호출에 default 값 명시. flag 시스템 죽어도 안전한 동작:
  flags.isOn('experimental-feature', false) ← 죽으면 off (안전)
  flags.isOn('killswitch-payment', false) ← 죽으면 off = 정상 동작 (안전)
4. Telemetry — flag eval 횟수, 결과 분포 메트릭. anomaly 감지에 유용
