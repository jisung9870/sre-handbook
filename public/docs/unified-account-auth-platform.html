● Overview

통합 계정 & 인증 플랫폼

모든 게임의 진입점이자 공통 플랫폼 SDK의 첫 번째 모듈. 가용성·정합성·보안 어느 하나가 무너져도 라인업 전체가 영향을 받는 가장 critical한 시스템.

계정 플랫폼이란?

정의

유저가 게임에 접속하기 위한 identity 발급·인증·권한 관리·세션 발행을 담당하는 백엔드. 단일 게임의 회원가입과 달리, 퍼블리셔는 여러 게임이 공유하는 통합 계정 플랫폼을 운영.

핵심 기능

회원 가입·로그인, 소셜 SSO 연동, 토큰 발급, 세션 관리, 디바이스 관리, 계정 복구·삭제, 멀티 디바이스 정책, 2FA·OTP, 감사 로그.

관련 시스템

결제 시스템(영수증을 계정에 귀속), 인벤토리·진행도(계정 단위), CS 백엔드(계정 식별), 텔레메트리(user_id), 어뷰징 탐지(다계정·계정 공유).

실패 시 영향

로그인 다운 → 모든 게임 동시 접속 불가. 결제 영수증의 사용자 식별 실패 → 매출 정합성 깨짐. 토큰 검증 다운 → 모든 인증 API 거부. 가장 폭발 반경이 큰 플랫폼.

왜 통합 계정인가

① 멀티 게임 사용자 경험

같은 퍼블리셔의 게임을 옮겨다닐 때 한 번만 로그인. 친구·길드·실적이 게임 간 연결되는 가능성도 열림.

② 크로스 플랫폼

모바일에서 시작한 진행을 PC(Steam)에서 이어가기. 플랫폼 계정(Steam/Apple ID)과 통합 계정의 매핑이 있어야 가능. 자세한 운영은 P10 크로스 플랫폼.

③ 보안·컴플라이언스 일원화

비밀번호 정책·2FA·계정 복구·개인정보 보유 정책을 한 곳에서 관리. 게임마다 따로 만들면 보안 수준 편차가 생김.

④ 어뷰징·CS 효율

다계정·환불사기·매크로 탐지가 전 게임 통합으로 가능. CS도 한 계정으로 여러 게임 문의를 처리.

SRE 관점의 책임

가용성

SLO를 매우 높게(예: 99.95%+). 로그인 다운 = 라인업 전체 다운. region별 multi-AZ 필수, DB 장애에도 토큰 검증은 살아있어야 함(read replica·캐시).

정합성

같은 user_id가 두 디바이스에서 동시 로그인하면 정책에 따라 처리(허용/이전 세션 무효화). 분산 환경에서의 race condition 처리가 핵심.

보안

credential stuffing(타사 유출 비밀번호 대입), phishing, 토큰 탈취. 인증 시스템은 항상 표적이라 평소에 보안 운영을 게을리하면 안 된다.

지연

로그인은 게임 진입의 첫 화면. 3초 이상 걸리면 사용자가 떠난다. region별 latency 목표(국내 100ms·해외 500ms 등) 별도 관리.

LINE Login v2.1

OAuth 2.0 + OIDC 기반. 주요 scope: openid(필수, JWT ID Token 받기), profile(이름·프로필 이미지), email. ID Token은 JWT 형식. 한국·일본·동남아 라인업에 가장 자주 통합.

Sign in with Apple

OIDC 기반. email 비공개 옵션(Apple이 relay 이메일 발급)을 지원해 사용자 프라이버시 강함. 자체 sub identifier는 앱별로 다름(같은 유저가 다른 앱에선 다른 sub).

Google Sign-In / Play Games Services

모바일은 Google Play Games Services로 통합 (도전과제·랭킹·세이브 무료). 일반 Google Sign-In은 OIDC 표준이라 PC 웹 통합에 자주 사용.

공통 운영 포인트

① IdP 장애 시 fallback(게스트로라도 진입 가능?), ② IdP rate limit 모니터링, ③ scope 최소 원칙(쓰지 않는 scope는 요청하지 말 것 — 사용자 동의율 떨어짐).

플랫폼 인증 — Steamworks Auth Ticket

Steam에서 게임을 실행할 때, Steam이 발급한 ticket으로 사용자 검증. PC 게임에서 가장 흔한 인증.

Steamworks Auth Ticket — Web API 검증 흐름
# 1. 클라이언트 (게임 안)에서 ticket 요청 ISteamUser::GetAuthTicketForWebApi("identity_string") # identity_string = ticket 수신자 식별자 (서버 ID 등) # 콜백 GetTicketForWebApiResponse_t 대기 후 hex 인코딩 # 2. 클라이언트가 자체 게임 서버로 ticket 전송 POST /auth/steam { "ticket": "abcdef0123...", "identity": "identity_string" } # 3. 서버가 Steam Web API에 검증 요청 (publisher API key 필요) GET https://partner.steam-api.com/ISteamUserAuth/AuthenticateUserTicket/v1/ ?key="PUBLISHER_API_KEY" &appid=123456 &ticket="abcdef0123..." &identity="identity_string" # 4. 응답: 64-bit SteamID + ownership 정보 { "response": { "params": { "result": "OK", "steamid": "7656119xxxxxxxxxx", "ownersteamid": "7656119xxxxxxxxxx" }}}
⚠ CRITICAL
① Web API 검증은 반드시 서버에서. 클라이언트가 직접 호출 절대 금지(API key 노출). ② identity 파라미터를 명시해 ticket이 다른 서버에서 재사용되지 않게 막을 것. ③ ticket은 짧은 수명이므로 발급 직후 검증 → 자체 세션 토큰 발급으로 전환.

Apple "Sign in with Apple" 정책 (2024년 1월 변경)

이전 (2019~2023)

Google·Facebook 등 소셜 로그인을 쓰면 Sign in with Apple도 의무 제공. 위반 시 앱 심사 거절.

현재 (2024-01~)

의무가 완화됨. 소셜 로그인 사용 시 "동등한 프라이버시 보호 옵션" 1개 제공이면 OK. 조건: (1) 이름·이메일만 수집, (2) 이메일 비공개 가능, (3) 광고용 추적 동의 없이 안 함.

면제 케이스

① 자체 회원 시스템만 사용, ② 특정 third-party 서비스의 클라이언트(예: Gmail 앱).

실무 권장

여전히 Sign in with Apple은 추가하는 게 안전. 사용자 프라이버시 친화·iOS 사용자 신뢰·Apple 심사 리스크 회피. 다른 옵션이 조건을 만족하는지 매번 검토 비용 < SiwA 한 번 통합.

● Tokens & Sessions

토큰 & 세션 관리

인증이 끝난 뒤 발급되는 토큰의 구조·수명·갱신·무효화. OAuth 2.1(2026 표준)을 기준으로 운영.

토큰 구조

Access Token

API 호출 시 Authorization 헤더에 포함. 짧은 수명(보통 15분~1시간). 탈취 시 피해 한정 위해 짧을수록 좋음.

Refresh Token

Access Token이 만료되면 갱신 받기 위한 토큰. 긴 수명(수일~수개월). 클라에 안전하게 저장(iOS Keychain, Android Keystore).

JWT vs Opaque

JWT: 토큰 안에 사용자 정보 + 서명. 검증을 stateless로 가능 → 빠름. 단점: 한 번 발급되면 만료 전 무효화 어려움. Opaque: 의미 없는 random string. 검증 시 매번 토큰 store 조회. 즉시 무효화 가능. 게임은 보통 access는 JWT, refresh는 opaque.

ID Token

OIDC에서 사용. 사용자 정보가 담긴 JWT. 인증 직후 한 번만 검증하고 폐기. API 호출용이 아님(Access Token과 혼동 주의).

OAuth 2.1 표준 (2026 권장)

OAuth 2.0의 보안 베스트 프랙티스(RFC 9700)를 표준화한 새 버전. 2026년 시점에서는 OAuth 2.1이 사실상 표준. 신규 시스템은 처음부터 2.1로 설계.

① PKCE 의무

OAuth 2.0에선 public client만 PKCE 권장이었으나, 2.1에선 모든 클라이언트(public·confidential)에 의무. authorization code 가로채기 공격(intercept)을 차단.

② Refresh Token Rotation

Refresh Token 사용 시마다 새 토큰 발급 + 이전 토큰 무효화. 탈취된 토큰을 attacker가 재사용하면 서버가 "replay" 감지 → 전체 세션 종료.

③ Sender-constrained Tokens

토큰을 발급받은 클라이언트만 사용 가능하게 묶음(DPoP, mTLS). public client의 refresh token은 rotation 또는 sender-constrained 둘 중 하나는 필수.

④ Implicit Grant 폐지

OAuth 2.0의 implicit grant flow는 2.1에서 삭제. 모든 클라이언트는 Authorization Code + PKCE를 사용해야 함. SPA·모바일 앱 모두 동일.

★ NOTE
이미 운영 중인 시스템도 OAuth 2.1 권장 사항으로 점진 마이그레이션 권장. PKCE·refresh token rotation은 비교적 적은 변경으로 적용 가능. 2026 시점에서 PKCE 없는 신규 OAuth 구현은 보안 결함으로 간주된다.

● Account Operations

계정 운영

가입 이후의 라이프사이클 — 연결·해제, 디바이스 관리, 복구, 삭제. CS 문의의 큰 비중을 차지하므로 셀프서비스 자동화가 중요.

계정 연결 (linking) · 해제 (unlinking)

유저가 게스트로 시작 후 소셜 SSO를 추가로 연결하거나, 다른 SSO를 추가하는 흐름. 가장 사고가 많이 나는 영역 — 진행도가 사라지거나 두 계정이 합쳐지는 일이 발생한다.

게스트 → 소셜 연결

로그인 상태에서 소셜 SSO 인증 → 토큰 받기 → 같은 user_id에 social_id 매핑 추가. 해당 social_id가 다른 user_id에 이미 연결되어 있으면 충돌 — 정책에 따라 처리(거부 / 진행도 병합 / 사용자 선택).

여러 SSO 동시 연결

한 계정에 LINE + Apple + Google 모두 연결 가능. 어느 한 SSO로 로그인해도 같은 계정 진입. 연결된 SSO 중 하나 이상은 항상 남아있어야 함(전부 해제하면 로그인 불가능).

계정 병합 (account merge)

두 계정을 하나로 합치는 것. 매우 위험 — 결제 내역·인벤토리·진행도 충돌 처리, 환불·CS 영향 큼. 보통 자동화하지 않고 CS 수동 처리.

SRE 관점

모든 link/unlink는 감사 로그 필수(누가·언제·어떤 SSO를 연결/해제). 분산 race condition 대비 — 같은 social_id를 두 user에 동시에 연결하는 시도는 트랜잭션·unique constraint로 차단.

디바이스 관리

디바이스 등록

로그인 시 디바이스 핑거프린트(OS·모델·고유 ID·IP) 저장. 같은 계정의 디바이스 목록을 사용자가 마이페이지에서 볼 수 있게 함.

디바이스 차단

탈취 의심 시 사용자가 특정 디바이스를 원격 차단(refresh token 무효화). CS 신청 또는 셀프서비스로 처리.

새 디바이스 알림

새 디바이스에서 로그인 시 기존 등록 디바이스에 알림 발송(이메일·푸시). 본인이 아니면 즉시 비밀번호 변경 + 디바이스 차단 유도.

비밀번호 없는 게스트 보호

게스트 계정은 디바이스 ID에 묶여 있어 디바이스 분실 = 진행도 분실. 일정 시점에 소셜 연결 강제 권유(레벨 N 도달 시·결제 후 등).

계정 복구

비밀번호 복구

이메일·전화번호로 OTP·재설정 링크 발송. 링크는 짧은 수명(15~30분), 1회용. 비밀번호 변경 시 모든 세션 강제 로그아웃.

소셜 SSO 복구

SSO 자체 복구 흐름(LINE·Apple·Google 비밀번호 찾기) — 퍼블리셔는 개입 못함. 백업 SSO 추가 권유가 유일한 사전 대응.

분실 계정 복구 (CS)

모든 인증 수단을 잃은 경우 — CS가 본인 확인(결제 영수증, 캐릭터 정보, 가입 시점 등)으로 신원 확인 후 복구. 매크로·사칭 시도가 많은 영역이라 신중한 매뉴얼 필요.

SRE 책임

OTP·복구 링크 발송 인프라 가용성, rate limit(같은 계정 OTP 발송 1분에 N회), 감사 로그(누가 언제 어떤 복구를 시도).

계정 삭제 (PIPA / GDPR)

한국 PIPA·EU GDPR 모두 사용자의 삭제 요청권을 인정. 게임 운영에서 가장 까다로운 부분 — 결제·세무 보관 의무와 충돌하기도 한다.

한국 PIPA

사용 목적 달성 시 즉시 삭제 의무. 보유 기간은 가입 시 명시 + 동의 받아야 함. 보유 사유 없이 보관하거나 만료 후 미삭제 시 과태료. 게임 결제·과세 관련 데이터는 별도 보관 의무(전자상거래법 5년 등)로 예외.

EU GDPR

"Right to be forgotten" — 사용자가 삭제 요청 시 기본적으로 삭제. 단, 법적 의무·다른 사용자 권리 보호 등 예외 사유는 인정. 30일 이내 처리 권고.

게임 도메인 특수성

인벤토리·결제 내역·길드 기여 등 다른 유저와 얽힌 데이터는 단순 삭제가 어려움. 보통 식별자만 익명화(anonymize)하고 게임 데이터는 유지.

실무 처리

① 삭제 요청 접수 → CS 검토 → 본인 확인 → ② 식별자 즉시 익명화 → ③ 결제·세무 데이터는 법적 보관 기간만큼 별도 보관 → ④ 기간 만료 시 자동 영구 삭제. 유예 기간(7~30일) 내 취소 가능은 흔한 정책.

⚠ NOTE
"삭제했다"는 사용자 안내와 실제 처리 사이의 정합성에 주의. 사용자에게 "삭제 완료"라고 통보했는데 백업·로그·텔레메트리에 그대로 남아있으면 컴플라이언스 위반. 삭제 범위와 보관 예외를 사용자에게 명확히 안내해야 함.
● Security

보안

인증 시스템은 항상 표적. credential stuffing·phishing·토큰 탈취 같은 흔한 공격을 방어하고, 어뷰징 패턴을 탐지·차단한다.

위협 매트릭스

Credential Stuffing

타사에서 유출된 비밀번호 목록으로 대규모 로그인 시도. 가장 흔한 공격. 사용자가 비밀번호 재사용을 멈추지 않는 한 계속 발생.

Phishing

가짜 로그인 페이지·고객센터로 유도해 비밀번호·OTP 탈취. 게임 도메인은 가짜 이벤트·아이템 지급 미끼가 많음.

토큰 탈취

로컬 저장된 refresh token이나 JWT를 디바이스 해킹으로 탈취. 다른 기기에서 그대로 사용. OAuth 2.1 sender-constrained / rotation으로 방어.

계정 공유 · 다계정

한 계정을 여러 사람이 공유(rmt 거래)하거나, 한 사람이 여러 계정 운영(어뷰징·이벤트 보상 다중 수령). 게임 도메인 고유 위협.

계정 탈취 → 결제 사기

탈취된 계정으로 IAP 결제 후 아이템·캐시를 다른 계정으로 이전. 결제 chargeback 시 손해는 게임사.

OTP 가로채기 (SIM swap)

이동통신사에서 SIM 재발급받아 SMS OTP 가로채기. SMS OTP는 점점 신뢰 낮아지는 추세. TOTP·앱 푸시 인증·passkey가 대안.

감사 (Audit) & 의심 활동 탐지

감사 로그 항목

로그인 (성공·실패·IP·디바이스), 비밀번호 변경, 2FA 등록·해제, SSO 연결·해제, 디바이스 추가·삭제, 계정 삭제 신청. 변경 가능한 모든 보안 액션은 기록.

보관 기간

최근 30~90일은 hot store(즉시 조회), 1~3년은 cold store(분석·법적 대응). 텔레메트리 인프라(P6)와 같이 운영하기도 함.

탐지 룰

예: "1분에 같은 IP에서 100계정 로그인 시도", "다른 국가에서 5분 안에 두 번 로그인", "정상 시간대 외(예: 새벽 3시) 비밀번호 변경 후 즉시 결제". 룰 + ML 혼합.

대응 흐름

의심 활동 감지 → 사용자에게 알림 → 의심 시 디바이스 강제 로그아웃 + 비밀번호 재설정 요구. 심각하면 계정 일시 잠금 + CS 본인 확인.

★ TIP
감사 로그는 사용자가 직접 볼 수 있게 마이페이지에서 노출하는 게 효과적. "로그인 활동"·"디바이스 활동" 화면이 있으면 사용자가 자기 계정 이상을 빨리 발견한다 (Google·Apple 모두 제공).
