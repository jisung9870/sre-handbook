● Kernel Memory Internals

커널 메모리 할당기와 회수 메커니즘

Buddy System, SLAB/SLUB, Page Reclaim, PSI, THP — Virtual Memory 아래 계층의 물리 메모리 관리.

Allocator

Buddy System (Page Allocator)

물리 메모리를 2^n 페이지(order) 단위로 할당하는 커널의 기본 페이지 할당기
커널이 물리 메모리를 관리하는 최하위 할당기. 메모리를 2^0(4KB), 2^1(8KB), ... 2^10(4MB) 크기의 블록(order)으로 관리한다. 요청된 크기에 맞는 블록이 없으면 큰 블록을 반으로 쪼개서(split) 제공하고, 해제 시 인접한 같은 크기 블록과 합쳐서(merge/coalesce) 큰 블록을 복원한다.

External fragmentation이 핵심 문제다. 시스템이 오래 운행되면 메모리가 조각나서 큰 order 할당이 실패할 수 있다. 이는 huge page 할당, 네트워크 버퍼(고order SKB) 등에 영향을 준다.
확인 명령
cat /proc/buddyinfo
각 zone별 order 0~10의 free 블록 수를 보여줌
order 3+ 블록이 0이면 fragmentation 심각
cat /proc/pagetypeinfo — 이동 가능/불가 분류
SRE 영향
고order 할당 실패: huge page 할당 불가, 네트워크 성능 저하
Memory Compaction: 커널이 페이지를 재배치해서 연속 공간 확보. CPU 비용 발생
echo 1 > /proc/sys/vm/compact_memory 수동 트리거
vmstat -m — slab 할당 통계
Buddy Split/Merge 시각화
Order 3 (32KB) 할당 요청 → Order 3 free 없음 → Order 4 (64KB) 블록을 split → 32KB + 32KB → 하나를 반환, 하나를 Order 3 free list에 추가 해제 시: 인접 buddy가 free이면 merge → Order 3 + Order 3 = Order 4 복원 → 재귀적으로 상위 order까지 merge 시도

SLAB / SLUB Allocator

커널 오브젝트를 위한 고속 캐시 할당기. Buddy System 위에서 동작
Buddy System은 페이지(4KB) 단위로만 할당한다. 하지만 커널 오브젝트(task_struct ~6KB, inode ~1KB, sk_buff 등)는 대부분 페이지보다 작다. SLAB/SLUB 할당기는 Buddy에서 받은 페이지를 동일 크기 오브젝트 단위로 쪼개서 캐싱하고 재사용한다.

현대 Linux는 SLUB을 기본으로 사용한다(SLAB은 6.5에서 제거됨). SLUB은 per-CPU 캐시, NUMA 인식, lockless fast path로 할당 성능을 극대화한다.
주요 Slab 캐시
task_struct — 프로세스/스레드 디스크립터
inode_cache — VFS inode 객체
dentry — 디렉토리 엔트리 캐시
skbuff_head_cache — 네트워크 패킷 버퍼
kmalloc-* — 범용 커널 메모리 할당
확인 & 모니터링
slabtop -s c — 캐시 크기 순 정렬
cat /proc/slabinfo — 상세 슬랩 정보
cat /proc/meminfo | grep Slab
SReclaimable: 회수 가능 슬랩 (dentry/inode 캐시)
SUnreclaim: 회수 불가 (커널 자료구조)
SRE 관점: Slab 메모리 누수
특정 슬랩 캐시가 지속적으로 성장하면 커널 메모리 누수 의심. dentry 캐시가 수 GB로 커지는 것은 정상(파일시스템 캐시)이지만, task_struct나 커스텀 슬랩이 비정상 성장하면 커널 버그.
echo 2 > /proc/sys/vm/drop_caches — dentry/inode 캐시만 해제 (프로덕션에서 신중히)
Prometheus: node_memory_SReclaimable_bytes / node_memory_SUnreclaim_bytes 모니터링
Architecture

Memory Zones

물리 메모리를 하드웨어 제약에 따라 영역으로 분류하는 체계
모든 물리 메모리가 동일하게 사용될 수 있는 것은 아니다. 구형 DMA 컨트롤러는 16MB 이하만, 32비트 시스템은 약 896MB까지만 직접 매핑할 수 있다. 이런 하드웨어 제약을 반영하여 커널은 메모리를 zone으로 분류한다.
Zone 종류 (x86_64)
ZONE_DMA: 0~16MB. ISA DMA용. 현대 서버에서 거의 사용 안 됨
ZONE_DMA32: 0~4GB. 32비트 DMA 디바이스용
ZONE_NORMAL: 4GB~전체. 일반 메모리. 대부분의 할당이 여기서
ZONE_MOVABLE: 핫플러그/compaction 가능한 이동식 페이지 전용
확인
cat /proc/zoneinfo — zone별 상세 (free, min, low, high 워터마크)
cat /proc/buddyinfo — zone별 free 블록
워터마크: min 아래면 direct reclaim + OOM 위험
low 아래면 kswapd 기동
high 이상이면 정상
Critical Path

Page Reclaim 심화

메모리 압박 시 커널이 페이지를 회수하는 메커니즘 — kswapd, direct reclaim, compaction
메모리가 부족해지면 커널은 사용 중인 페이지를 회수하여 free 메모리를 확보해야 한다. 회수 방법은 ① Page Cache(clean page)를 버리기 ② Anonymous page를 swap으로 보내기 ③ Slab 캐시 shrink가 있다.

이 과정이 성능에 미치는 영향이 크다. 특히 direct reclaim은 메모리 할당을 요청한 프로세스가 직접 회수를 수행하므로 응답 지연 급증의 주범이 된다.
회수 경로
kswapd (백그라운드): free가 low 워터마크 아래로 떨어지면 커널 스레드가 자동 기동. 비동기로 페이지 회수
Direct Reclaim (동기): free가 min 워터마크에 도달하면 할당 요청 프로세스가 직접 회수. 이 순간 응답 지연이 급격히 늘 수 있다
OOM Killer: 모든 회수 시도 실패 시 최후 수단
LRU Lists
커널은 페이지를 Active/Inactive × Anonymous/File 4개 LRU 리스트로 관리:
Active Anon: 최근 접근된 프로세스 메모리
Inactive Anon: 오래된 프로세스 메모리 → swap 후보
Active File: 최근 접근된 Page Cache
Inactive File: 오래된 Page Cache → 회수 1순위
cat /proc/meminfo | grep -E 'Active|Inactive'
Memory Compaction
페이지를 물리적으로 재배치하여 연속된 free 블록을 만드는 과정. Buddy System의 fragmentation 문제를 해결하기 위해 필요하다.
Movable page(LRU, Page Cache)는 이동 가능하지만, Unmovable page(커널 자료구조)는 이동 불가.
Compaction은 CPU를 소모하고 latency를 유발할 수 있다. cat /proc/vmstat | grep compact로 발생 빈도 확인.
K8s 영향: 노드에서 대량 Pod 생성/삭제 반복 시 fragmentation → compaction 빈번 → 노드 전체 latency 증가 가능
Observability

PSI — Pressure Stall Information

CPU, 메모리, I/O 리소스 압박을 정량적으로 측정하는 커널 메트릭 (Linux 4.20+)
기존 메트릭(load average, free memory 등)은 "리소스 부족이 실제로 워크로드를 지연시키고 있는가?"에 직접 답하지 못한다. PSI는 "태스크가 리소스 부족으로 실행을 못 하고 대기한 시간의 비율"을 직접 측정한다.

some은 일부 태스크가 대기 중인 시간, full은 모든 태스크가 대기 중(= 생산적 작업 0%)인 시간의 비율이다.
확인 방법
cat /proc/pressure/cpu
cat /proc/pressure/memory
cat /proc/pressure/io
출력 예: some avg10=2.50 avg60=1.30 avg300=0.80 total=123456
cgroup별: /sys/fs/cgroup/<path>/cpu.pressure
SRE 활용
memory some > 10%: Page Cache thrashing 시작
memory full > 5%: direct reclaim 빈번, 심각한 메모리 부족
io some > 20%: 디스크 I/O 대기 발생
Prometheus: node_pressure_memory_stalled_seconds_total
K8s Pod eviction: cgroup PSI를 기반으로 eviction 결정 가능 (KEP-2837)
💡 PSI vs Load Average
Load Average는 running + uninterruptible sleep 태스크 수로, 원인(CPU? I/O?)을 구분 못 한다. PSI는 CPU/Memory/IO를 분리하여 정확히 어떤 리소스가 병목인지 알려준다.
Performance

Transparent Huge Pages (THP)

4KB 대신 2MB 페이지를 자동으로 사용하여 TLB 미스를 줄이는 메커니즘
TLB 엔트리 수는 제한적이다(보통 수백~수천 개). 4KB 페이지로는 TLB가 커버하는 메모리 범위가 좁아 TLB miss → Page Table Walk가 빈번하다. THP는 2MB(또는 1GB) 페이지를 사용하여 하나의 TLB 엔트리로 더 넓은 범위를 커버한다.

그러나 THP는 양날의 검이다. 커널이 2MB 연속 메모리를 확보하기 위해 compaction을 트리거하고, 이 과정에서 latency spike가 발생한다. 특히 Redis, MongoDB, DB 워크로드에서 문제가 심각하다.
상태 확인 & 제어
cat /sys/kernel/mm/transparent_hugepage/enabled
값: always | madvise | never
cat /sys/kernel/mm/transparent_hugepage/defrag
값: always | defer | defer+madvise | madvise | never
워크로드별 권장
Redis / MongoDB: never (공식 문서에서 명시적 비활성 권고)
PostgreSQL / MySQL: never 또는 madvise
JVM (대용량 힙): madvise + -XX:+UseTransparentHugePages
범용 서버: madvise (앱이 명시적으로 요청할 때만)
HPC/ML: always (큰 메모리 블록 접근 패턴)
THP 문제 진단
grep -i huge /proc/meminfo — AnonHugePages, HugePages_Total 확인
cat /proc/vmstat | grep thp — thp_fault_alloc, thp_collapse_alloc, thp_fault_fallback(실패)
perf stat -e dTLB-load-misses,dTLB-store-misses ./app — TLB miss 측정
thp_fault_fallback가 높고 latency spike가 동반되면 THP compaction이 원인

I/O 서브시스템 내부 동작

blk-mq, io_uring, writeback 심화, NVMe — Block Layer 아래까지의 I/O 경로.

Architecture

blk-mq (Multi-Queue Block Layer)

NVMe 시대에 맞게 재설계된 멀티큐 블록 I/O 계층
기존 블록 레이어는 단일 요청 큐를 사용하여 모든 I/O가 하나의 lock을 경쟁했다. NVMe 디바이스는 수십만 IOPS를 제공하지만 단일 큐로는 이를 활용할 수 없었다.

blk-mq는 소프트웨어 큐(CPU당 하나) → 하드웨어 큐(디바이스 큐 수에 매핑) 2계층 구조로, lock contention을 제거하고 CPU별로 I/O를 병렬 처리한다.
구조
Software Staging Queue: per-CPU. 요청을 모아 배치 제출
Hardware Dispatch Queue: 디바이스의 실제 큐에 1:1 매핑
매핑: CPU N → SW Queue N → HW Queue (N % hw_queue_count)
cat /sys/block/nvme0n1/queue/nr_requests
SRE 영향
I/O 스케줄러 선택: blk-mq에서 NVMe는 none이 최적 (디바이스가 자체 스케줄링)
I/O 우선순위: ionice -c 1 -n 0 ./app (realtime 클래스)
Queue depth: nr_requests 값이 높으면 throughput↑ latency↑
iostat -xz 1의 aqu-sz(avg queue size)로 포화도 확인
Next-Gen

io_uring

차세대 비동기 I/O 인터페이스. Syscall 오버헤드를 제거하는 공유 링 버퍼 기반
전통적 비동기 I/O(aio)는 syscall 오버헤드가 크고 기능이 제한적이었다. io_uring(Linux 5.1+)은 유저-커널 공유 메모리 링 버퍼를 사용하여 syscall 없이 I/O 요청을 제출하고 완료를 확인할 수 있다.

핵심은 두 개의 링: Submission Queue (SQ)에 요청을 넣고, Completion Queue (CQ)에서 결과를 읽는다. 커널과 유저스페이스가 같은 메모리를 공유하므로 데이터 복사가 없다.
동작 구조
User Space Kernel Space ┌──────────┐ ┌──────────┐ │ SQ Ring │───────→│ I/O 처리 │ │ (제출) │ mmap │ │ └──────────┘ └────┬─────┘ │ ┌──────────┐ ┌────▼─────┐ │ CQ Ring │←───────│ 완료 기록 │ │ (완료) │ mmap │ │ └──────────┘ └──────────┘
성능 특성
Syscall 제거: SQPOLL 모드 시 커널 폴링 스레드가 SQ 감시. 유저는 syscall 없이 SQ에 쓰기만
Batching: 여러 I/O를 하나의 submit으로 제출
Fixed buffers: 미리 등록된 버퍼로 매 I/O의 mapping 비용 제거
지원 연산: read, write, fsync, poll, accept, connect, send, recv 등 거의 모든 I/O
SRE 관점
활용처: 고성능 스토리지 엔진(RocksDB), 웹서버(Nginx 개발 중), DB, 네트워크 프록시
보안 이슈: io_uring의 광범위한 커널 접근이 보안 취약점 경로가 됨. Google은 프로덕션에서 io_uring을 비활성화. 컨테이너 환경에서 seccomp으로 차단하는 경우 많음
cat /proc/config.gz | zgrep IO_URING — 커널 빌드 여부 확인

Writeback 심화 — cgroup writeback & I/O 우선순위

컨테이너 환경에서의 writeback 격리와 blkio 컨트롤러
이전 문서에서 dirty page writeback의 기본을 다뤘다면, 여기선 컨테이너 간 I/O 간섭 문제를 다룬다. 한 Pod의 대량 쓰기가 다른 Pod의 I/O latency를 높이는 Noisy Neighbor 문제는 cgroup writeback과 blkio로 완화할 수 있다.
cgroup v2 writeback
cgroups v2에서는 dirty page가 어느 cgroup에 속하는지 추적하여 cgroup별로 writeback을 제어할 수 있다.
memory.current에 dirty page 포함
cgroup의 memory.high 도달 시 해당 cgroup의 dirty page만 먼저 writeback → 격리 효과
blkio 컨트롤러
io.max — 디바이스별 BPS/IOPS 하드 리밋
io.weight — 비례 분배 (1~10000, 기본 100)
io.latency — 타겟 latency. 초과 시 해당 cgroup의 I/O를 throttle
K8s에서 직접 지원하지 않지만, init container 또는 device plugin으로 설정 가능

Hardware

NVMe 내부 구조

NVMe 프로토콜의 커맨드 큐, 인터럽트 처리, polling 모드
NVMe는 AHCI(SATA)를 대체하는 PCIe 네이티브 스토리지 프로토콜이다. AHCI의 단일 큐(depth 32) 한계를 넘어 최대 65535개 큐, 큐당 64K 엔트리를 지원한다. 이것이 수백만 IOPS가 가능한 이유다.
핵심 구조
Submission Queue (SQ): 호스트→디바이스 커맨드 제출
Completion Queue (CQ): 디바이스→호스트 완료 통지
SQ/CQ 쌍을 CPU 코어별로 생성 → lock-free 병렬 I/O
Admin Queue(설정용) + I/O Queue(데이터용) 분리
Polling Mode (io_poll)
인터럽트 대신 CPU가 CQ를 직접 폴링하여 완료 감지
장점: 인터럽트 latency(수 μs) 제거, ultra-low latency 달성
단점: CPU 사이클 소모 (유휴 시에도 폴링)
mount -o io_poll /dev/nvme0n1p1 /data
io_uring의 IORING_SETUP_IOPOLL과 결합
SRE 관점: NVMe 모니터링
nvme smart-log /dev/nvme0n1 — 온도, 수명(percentage_used), 에러 카운트
nvme error-log /dev/nvme0n1 — 에러 로그
SMART 경고 알림: percentage_used > 80%, critical_warning ≠ 0, media_errors 증가 추이
iostat -xz 1 — await(평균 I/O latency)가 NVMe에서 0.1ms 이하가 정상. 1ms 이상이면 문제 의심
● Interrupt & Softirq

인터럽트 처리 내부 동작

Hardware IRQ, softirq, NAPI, IRQ affinity, RPS/RFS/XPS — 네트워크 성능의 숨은 병목 계층.

Architecture

IRQ 처리 모델: Top-Half / Bottom-Half

하드웨어 인터럽트를 최소한으로 처리하고, 나머지를 지연 실행하는 2단계 구조
디바이스(NIC, 디스크 등)가 이벤트를 알리면 CPU는 현재 작업을 중단하고 인터럽트 핸들러를 실행한다. 이 핸들러가 오래 걸리면 다른 인터럽트도 처리 못 하고 시스템이 멈춘다.

해결: Top-Half(하드웨어 IRQ)에서는 최소한의 작업(ACK, 상태 저장)만 하고, 나머지는 Bottom-Half(softirq, tasklet, workqueue)로 지연 처리한다.
Top-Half (Hard IRQ)
• 인터럽트 비활성 상태에서 실행 (preemption 불가)
• 최소한의 작업만: 디바이스 ACK, 데이터 DMA 확인
• softirq 스케줄링 (raise_softirq(NET_RX_SOFTIRQ))
• 수 μs 이내 완료 필수
/proc/interrupts — CPU별 인터럽트 카운트
Bottom-Half (Softirq)
• 인터럽트 활성 상태에서 실행 (다른 인터럽트 처리 가능)
• 네트워크 패킷 처리, 블록 I/O 완료 등 실제 작업 수행
• 우선순위: HI_SOFTIRQ > TIMER > NET_TX > NET_RX > BLOCK > ...
• budget 소진 시 ksoftirqd 커널 스레드로 위임
cat /proc/softirqs — softirq 유형별 카운트
ksoftirqd
softirq가 한 번에 처리할 수 있는 양(netdev_budget=300, time_squeeze)을 초과하면, ksoftirqd/N 커널 스레드에서 이어서 처리한다. top에서 ksoftirqd가 높은 CPU를 사용하면 네트워크 또는 블록 I/O 처리량이 인터럽트 처리 능력을 초과한 것.
cat /proc/net/softnet_stat — 2번째 열(time_squeeze)이 증가하면 budget 부족. net.core.netdev_budget 증가로 대응.
Network

NAPI (New API) — 네트워크 패킷 고속 처리

인터럽트와 폴링을 혼합하여 고트래픽에서 인터럽트 폭풍을 방지
패킷이 한 개 올 때마다 인터럽트를 발생시키면, 고트래픽에서 초당 수십만 인터럽트가 발생하여 CPU가 인터럽트 처리에만 시간을 소모한다(livelock). NAPI는 첫 패킷에만 인터럽트를 발생시키고, 이후는 폴링 모드로 전환하여 연속된 패킷을 batch 처리한다.
동작 흐름
1) 첫 패킷 도착 → 하드웨어 인터럽트
2) Top-half에서 NAPI 스케줄링, 인터럽트 비활성
3) softirq에서 poll 함수 호출, ring buffer에서 패킷 batch 수거
4) budget(기본 64개) 소진하거나 ring 빌 때까지 반복
5) 처리 완료 → 인터럽트 재활성, 다음 패킷 대기
고트래픽: 2→5가 반복되어 순수 폴링에 가까워짐
Interrupt Coalescing
NIC 하드웨어 레벨에서 여러 패킷을 모아 인터럽트를 줄이는 기법:
ethtool -c eth0 — 현재 coalescing 설정
ethtool -C eth0 rx-usecs 50 rx-frames 64
latency ↔ throughput 트레이드오프:
• rx-usecs 낮으면: 즉시 인터럽트, 낮은 latency
• rx-usecs 높으면: 모아서 인터럽트, 높은 throughput
Tuning

IRQ Affinity & Balancing

인터럽트를 어느 CPU에서 처리할지 제어하여 성능을 최적화
기본적으로 irqbalance 데몬이 인터럽트를 CPU에 분산한다. 하지만 고성능 환경에서는 수동으로 NIC 인터럽트를 특정 CPU에 바인딩하여 cache locality를 높이는 것이 효과적이다.
확인 & 설정
cat /proc/interrupts — 인터럽트별 CPU 카운트
cat /proc/irq/<N>/smp_affinity — CPU 비트마스크
echo 2 > /proc/irq/<N>/smp_affinity — CPU 1에 바인딩
cat /proc/irq/<N>/smp_affinity_list — 읽기 쉬운 형식
systemctl stop irqbalance — 수동 관리 시 비활성
전략
RSS(Receive-Side Scaling): 멀티큐 NIC의 각 큐를 다른 CPU에 바인딩
NIC 큐 수 = CPU 코어 수로 맞추는 것이 이상적
ethtool -l eth0 — 큐 수 확인
ethtool -L eth0 combined 8 — 8개 큐 설정
각 큐의 IRQ를 다른 CPU에 배치하면 패킷 처리가 완전 병렬화
Software Scaling

RPS / RFS / XPS

소프트웨어 레벨에서 네트워크 처리를 멀티코어로 분산하는 기법
RSS가 하드웨어(NIC) 레벨의 멀티큐 분산이라면, RPS/RFS/XPS는 소프트웨어(커널) 레벨의 분산이다. 단일 큐 NIC나 가상환경(virtio)에서 유용하다.
기법방향원리설정
RPS
(Receive Packet Steering)		RX (수신)softirq에서 패킷의 해시를 계산, 다른 CPU의 backlog에 전달. RSS의 소프트웨어 버전echo ff > /sys/class/net/eth0/queues/rx-0/rps_cpus
RFS
(Receive Flow Steering)		RX (수신)패킷을 해당 소켓을 처리하는 애플리케이션의 CPU로 전달. Cache locality 극대화echo 32768 > /proc/sys/net/core/rps_sock_flow_entries
echo 2048 > /sys/.../rps_flow_cnt
XPS
(Transmit Packet Steering)		TX (송신)송신 시 CPU→TX 큐 매핑을 최적화. 같은 CPU의 패킷을 같은 큐로 보내 lock 감소echo 1 > /sys/class/net/eth0/queues/tx-0/xps_cpus
⚠ 가상환경(K8s 노드) 주의
클라우드 VM의 virtio-net은 보통 큐 수가 제한적(1~4개). RPS를 활성화하면 단일 큐의 소프트 인터럽트 처리를 멀티코어로 분산하여 네트워크 성능이 크게 향상될 수 있다. EKS/NHN Cloud VM에서 network intensive Pod 운영 시 고려.
● FD & epoll Internals

File Descriptor와 I/O 멀티플렉싱

FD 테이블 3계층, epoll 내부 자료구조, select/poll/epoll 비교, Unix Domain Socket.

Structure

File Descriptor 테이블 3계층 구조

프로세스별 FD → 시스템 File Table → inode — "모든 것은 파일"의 실체
FD(0, 1, 2, 3...)는 단순한 정수다. 이 정수가 실제 파일/소켓/파이프에 연결되는 과정에 3개의 테이블이 관여한다. 이 구조를 이해해야 FD 누수, 리밋 설정, fork 시 FD 상속 등을 정확히 이해할 수 있다.
3계층 구조
1. Process FD Table (per-process)
→ FD 번호 → file 구조체 포인터 배열
2. System Open File Table (global)
→ file 구조체: 현재 offset, 접근 모드, f_op(연산 함수 포인터)
3. inode Table (global)
→ 파일의 실체: 크기, 권한, 디스크 위치, 타입
리밋 설정
프로세스별: ulimit -n (기본 1024, 부족!)
/etc/security/limits.conf 또는 systemd LimitNOFILE=
시스템 전체: fs.file-max (sysctl)
cat /proc/sys/fs/file-nr → 사용중 / 0 / 최대
K8s: Pod spec에서 ulimit 직접 설정 불가. 노드 레벨 조정 필요
fork() 시 FD 상속
fork() 시 부모의 FD 테이블이 복제된다. 하지만 File Table 엔트리는 공유된다. 즉, 부모와 자식이 같은 파일 offset을 공유.
O_CLOEXEC 플래그: exec() 시 자동으로 FD를 닫음. 보안상 항상 설정 권장.
FD 누수: close()하지 않은 FD가 누적 → EMFILE(Too many open files) 에러. ls /proc/[pid]/fd | wc -l로 현재 FD 수 확인
Core

epoll 내부 동작

Red-Black Tree + Ready List로 O(1) 이벤트 감지를 달성하는 I/O 멀티플렉싱
epoll은 Linux의 고성능 I/O 이벤트 통지 메커니즘이다. Nginx, Node.js, Go 런타임, Redis 등 거의 모든 고동시성 서버가 epoll 기반이다.

핵심 자료구조: Red-Black Tree(관심 FD 관리)와 Ready List(이벤트 발생한 FD 목록). FD에 이벤트가 발생하면 커널이 callback으로 Ready List에 추가하므로, 전체 FD를 스캔할 필요가 없다.
API
epoll_create1() — epoll 인스턴스 생성 (RB-tree + ready list 초기화)
epoll_ctl(ADD/MOD/DEL) — FD를 RB-tree에 등록/수정/삭제. 각 FD에 callback 등록
epoll_wait() — ready list에서 이벤트 발생 FD를 꺼냄. 없으면 sleep
Level vs Edge Triggered
Level-Triggered (LT, 기본): 데이터가 있는 한 반복 통지. 안전하지만 epoll_wait 호출 빈번
Edge-Triggered (ET): 상태 변경 시에만 통지. 놓치면 영원히 통지 안 됨. non-blocking I/O 필수
Nginx: ET 모드 사용 (성능 최적화)
Go runtime: LT 모드 (안전성 우선)
내부 동작 상세
epoll_ctl(ADD, fd=5, EPOLLIN): → RB-tree에 fd=5 노드 삽입 (O(log n)) → fd=5의 wait queue에 callback 등록 fd=5에 데이터 도착 (NIC → socket buffer): → 커널이 fd=5의 callback 호출 → callback이 fd=5를 ready list에 추가 (O(1)) epoll_wait(): → ready list에서 이벤트를 복사하여 반환 (O(ready count)) → ready list가 비어있으면 sleep

핵심: 10만 FD를 모니터링해도 이벤트가 10개면 10개만 처리. O(active events) 성능.
Comparison

I/O 멀티플렉싱 성능 비교

항목selectpollepoll

IPC

Unix Domain Socket & Zero-Copy

같은 호스트 내 프로세스 간 고속 통신과 zero-copy 데이터 전달
Unix Domain Socket(UDS)은 같은 머신의 프로세스 간 통신에서 TCP보다 훨씬 빠르다. TCP는 전체 네트워크 스택(IP, TCP, checksum, segmentation)을 통과하지만, UDS는 커널 내 메모리 복사만으로 데이터를 전달한다.

Docker/containerd 소켓(/var/run/docker.sock), kubelet→CRI 통신 등 K8s 인프라에서 핵심적으로 사용된다.
UDS vs TCP loopback 차이
UDS: socket layer → 커널 메모리 복사 → 대상 소켓 버퍼. Netfilter, TCP 스택 모두 우회
TCP 127.0.0.1: socket → TCP → IP → loopback → IP → TCP → socket. 전체 스택 통과
UDS가 1.5~3배 빠르고 CPU 사용 50% 적음
Latency: UDS ~2μs vs TCP loopback ~8μs
Zero-Copy 기법들
splice(): 파이프를 매개로 FD 간 데이터 전달. user↔kernel 복사 제거
sendfile(): 파일→소켓 직접 전송. 웹서버 정적 파일 서빙에 최적
MSG_ZEROCOPY: TCP/UDP send에서 커널 복사 제거 (4.14+)
io_uring: 공유 메모리 링으로 syscall + 복사 모두 제거
mmap: 파일을 가상 주소에 매핑하여 복사 없이 접근
● Security Internals

커널 보안 메커니즘

Capabilities, seccomp-BPF, MAC, Kernel Hardening — 컨테이너 보안과 커널 하드닝의 기반.

Privilege

Linux Capabilities

root 권한을 세분화하여 최소 권한 원칙을 적용하는 POSIX 모델
전통적으로 Linux는 root(UID 0) = 모든 권한, non-root = 제한된 권한의 이분법이었다. Capabilities는 root의 권한을 약 40개의 세분화된 capability로 분해한다. 프로세스에 필요한 capability만 부여하면 보안 위험을 최소화할 수 있다.
핵심 Capabilities
CAP_NET_ADMIN — 네트워크 설정 변경 (iptables 등)
CAP_NET_BIND_SERVICE — 1024 이하 포트 바인딩
CAP_NET_RAW — raw 소켓 (ping, tcpdump)
CAP_SYS_ADMIN — mount, cgroups 등. 사실상 root 수준
CAP_SYS_PTRACE — 다른 프로세스 디버깅
CAP_DAC_OVERRIDE — 파일 권한 무시
K8s securityContext
securityContext.capabilities.drop: ["ALL"]
securityContext.capabilities.add: ["NET_BIND_SERVICE"]
Best Practice: 모든 capability를 drop하고 필요한 것만 add
Docker 기본: 14개 capability 부여 (CAP_CHOWN, CAP_SETUID 등)
getpcaps [pid] — 프로세스의 현재 capability 확인
Syscall Filter

seccomp-BPF

프로세스가 호출할 수 있는 system call을 BPF 프로그램으로 필터링
Capabilities가 "어떤 종류의 작업"을 제어한다면, seccomp은 "어떤 syscall을 호출할 수 있는가"를 제어한다. 불필요한 syscall을 차단하면 커널 취약점의 공격 표면(attack surface)을 크게 줄일 수 있다.
동작 모드
Strict: read, write, _exit, sigreturn만 허용 (너무 제한적)
Filter (BPF): BPF 프로그램으로 세밀한 필터링. 각 syscall에 대해 ALLOW, KILL, ERRNO, TRACE 등 결정
필터는 fork/clone 시 자식에 상속
컨테이너 런타임(containerd, CRI-O)이 기본 seccomp 프로파일 적용
K8s 적용
securityContext.seccompProfile.type: RuntimeDefault
→ 컨테이너 런타임의 기본 프로파일 적용
type: Localhost → 커스텀 프로파일 (JSON)
차단되는 대표 syscall: mount, reboot, kexec_load, bpf(일부), io_uring_setup
grep Seccomp /proc/[pid]/status — 현재 모드 확인
MAC

AppArmor / SELinux

DAC(파일 퍼미션) 위에 추가되는 Mandatory Access Control 계층
일반 파일 퍼미션(DAC)은 소유자가 권한을 결정한다. MAC은 시스템 정책이 강제하여 root라도 정책이 금지하면 접근 불가. 컨테이너 탈출 공격 방어에 중요.
AppArmor
• 경로(path) 기반 접근 제어. 프로파일이 파일 경로별 규칙 정의
• Ubuntu/Debian 기본. K8s 컨테이너에 프로파일 적용 가능
• aa-status — 로드된 프로파일 확인
• aa-logprof — 위반 로그 기반 프로파일 자동 생성
• enforce(차단) / complain(로그만) 모드
SELinux
• 레이블(label) 기반 접근 제어. 모든 객체에 보안 컨텍스트 부여
• RHEL/CentOS 기본. 더 강력하지만 더 복잡
• getenforce — 현재 모드 (Enforcing/Permissive/Disabled)
• semanage, restorecon으로 정책 관리
• K8s: seLinuxOptions로 Pod 레이블 설정 가능
Hardening

Kernel Hardening

메모리 공격, 권한 상승, 정보 유출을 방어하는 커널 보호 메커니즘
메커니즘보호 대상원리확인/설정

● Kernel Tracing & Profiling

커널 추적과 프로파일링 심화

ftrace, kprobes, perf, bpftrace — 커널 내부 동작을 관찰하는 도구의 원리와 고급 활용.

Kernel Built-in

ftrace — 커널 내장 함수 추적기

커널 빌드 시 포함되는 추적 프레임워크. tracefs를 통해 제어
ftrace는 추가 도구 설치 없이 커널 함수 호출을 추적할 수 있는 내장 프레임워크다. /sys/kernel/debug/tracing/(또는 /sys/kernel/tracing/)에서 파일 읽기/쓰기로 제어한다. 가장 기본적이면서도 강력한 커널 디버깅 도구.
주요 Tracer
function: 커널 함수 호출 추적
function_graph: 함수 진입/종료 + 실행 시간 (콜 트리 시각화)
nop: 추적 비활성 (이벤트만 사용할 때)
사용 가능: cat available_tracers
핵심 사용법
cd /sys/kernel/tracing
echo function_graph > current_tracer
echo tcp_v4_connect > set_graph_function
cat trace
echo 0 > tracing_on (중지)
echo > trace (버퍼 클리어)
trace-cmd (편의 도구)
trace-cmd record -p function_graph -g tcp_sendmsg -F curl example.com
trace-cmd report
ftrace의 tracefs 조작을 CLI로 감싸서 편리하게 사용. flamegraph 생성도 가능.
Dynamic

kprobes / uprobes

커널 함수(kprobe) 또는 유저 함수(uprobe)에 동적으로 프로브를 삽입
kprobe: 거의 모든 커널 함수의 진입(entry)이나 반환(return)에 동적으로 훅을 걸 수 있다. 커널을 재컴파일하거나 재부팅할 필요 없이, 실행 중인 시스템에서 함수 인자, 반환값, 실행 시간을 관찰할 수 있다.

uprobe: 유저스페이스 바이너리의 함수에 프로브를 거는 것. Go, Python, Java 앱의 함수 호출을 커널 레벨에서 추적 가능.
kprobe 예시
bpftrace -e 'kprobe:tcp_retransmit_skb { printf("retrans: %s pid=%d\n", comm, pid); }'
→ TCP 재전송 발생 시 프로세스 정보 출력

bpftrace -e 'kretprobe:vfs_read /retval > 1048576/ { printf("large read: %s %d bytes\n", comm, retval); }'
→ 1MB 이상 파일 읽기 감지
uprobe 예시
bpftrace -e 'uprobe:/usr/bin/bash:readline { printf("bash cmd: pid=%d\n", pid); }'
→ bash에서 명령 입력 시마다 추적

Go 앱: uprobe:/path/to/binary:"main.handleRequest"
USDT(User Statically-Defined Tracing): 앱이 미리 정의한 트레이스포인트
Hardware

perf 심화 — PMU 하드웨어 카운터

CPU Performance Monitoring Unit을 활용한 마이크로아키텍처 레벨 성능 분석
perf는 소프트웨어 이벤트(context switch, page fault)뿐 아니라 하드웨어 이벤트(CPU 사이클, 캐시 미스, 브랜치 예측 실패)를 측정할 수 있다. CPU의 PMU(Performance Monitoring Unit) 레지스터를 직접 읽어 커널/앱 코드의 마이크로아키텍처 병목을 찾는다.
핵심 명령
perf stat ./app — 전체 실행의 HW 카운터 요약
perf record -g -F 99 -p PID -- sleep 30 — 99Hz 샘플링, 콜스택 포함
perf report — 핫스팟 함수 분석
perf script | ./stackcollapse-perf.pl | ./flamegraph.pl > out.svg
→ Flame Graph 생성 (Brendan Gregg)
주요 HW 이벤트
cycles — CPU 사이클 수
instructions — 실행된 명령 수. IPC = instructions/cycles
cache-misses / cache-references — L1/L2/LLC 캐시 미스율
branch-misses — 분기 예측 실패 (파이프라인 스톨)
dTLB-load-misses — TLB 미스 (THP 효과 측정에 유용)
perf stat -e cycles,instructions,cache-misses ./app
Flame Graph 해석
X축: 스택 프레임의 알파벳 순서 (시간 순서 아님!)
Y축: 콜 스택 깊이. 위로 갈수록 호출된 함수
너비: 해당 함수가 CPU를 점유한 비율. 넓으면 핫스팟
색상: 의미 없음 (구분용). 단, differential flame graph에서는 빨강=증가, 파랑=감소
Off-CPU flame graph: CPU를 기다리는(sleep, I/O 대기) 시간 분석. latency 원인 추적에 핵심
eBPF

bpftrace 심화

프로브 타입별 활용 패턴, 맵, 히스토그램, 스택 트레이스
bpftrace는 awk-like 문법으로 eBPF 프로그램을 작성하는 고급 추적 도구다. 원라이너로 커널/앱의 거의 모든 동작을 관찰할 수 있다.
▼ 실무 bpftrace 레시피
# ① 프로세스별 syscall 빈도 bpftrace -e 'tracepoint:raw_syscalls:sys_enter { @[comm] = count(); }' # ② 블록 I/O latency 히스토그램 (μs) bpftrace -e 'tracepoint:block:block_rq_issue { @start[args->dev, args->sector] = nsecs; } tracepoint:block:block_rq_complete /@start[args->dev, args->sector]/ { @usecs = hist((nsecs - @start[args->dev, args->sector]) / 1000); delete(@start[args->dev, args->sector]); }' # ③ TCP 연결 설정 latency (connect → established) bpftrace -e 'kprobe:tcp_v4_connect { @start[tid] = nsecs; } kretprobe:tcp_v4_connect /@start[tid]/ { @ms = hist((nsecs - @start[tid]) / 1000000); delete(@start[tid]); }' # ④ 특정 PID의 파일 읽기 크기 분포 bpftrace -e 'tracepoint:syscalls:sys_exit_read /pid == 1234 && args->ret > 0/ { @bytes = hist(args->ret); }' # ⑤ Page Fault 발생 위치 스택 트레이스 bpftrace -e 'software:page-faults:1 { @[ustack, comm] = count(); }' # ⑥ CPU runqueue length 프로파일링 bpftrace -e 'profile:hz:99 { @runqlen = lhist(curtask->se.cfs_rq->nr_running, 0, 100, 1); }'
Always-On

Continuous Profiling

CPU/메모리 프로파일링을 상시 수집하여 과거 시점의 성능을 소급 분석
기존 프로파일링은 "문제가 발생한 후" 도구를 붙여서 재현해야 했다. Continuous Profiling은 항상 낮은 오버헤드로 프로파일링 데이터를 수집하여, 과거 시점의 CPU/메모리 사용 패턴을 flame graph로 볼 수 있다.

Observability의 제4 필러(Metrics, Logs, Traces + Profiles)로 불리기도 한다.
도구
Pyroscope: Grafana Labs 인수. LGTM 스택과 통합. Go/Python/Java/eBPF 지원
Parca: Polar Signals. eBPF 기반 zero-instrumentation. Kubernetes 네이티브
Google Cloud Profiler: GKE 환경 기본 지원
데이터 수집: eBPF(perf_event_open) 또는 언어 런타임 프로파일러
SRE 활용
인시던트 사후 분석: "2시간 전 CPU spike 때 어떤 함수가 핫스팟이었나?" 소급 확인
리소스 최적화: 어떤 함수가 전체 CPU의 몇 %를 차지하는지 상시 파악
회귀 감지: 배포 전후 flame graph diff로 성능 회귀 자동 감지
오버헤드: 보통 CPU 1~3% 미만. 프로덕션 상시 수집 가능
💡 LGTM + Pyroscope = Observability 4 Pillars
Loki(Logs) + Grafana(Viz) + Tempo(Traces) + Mimir/Prometheus(Metrics) + Pyroscope(Profiles). Grafana에서 metric spike → trace → profile drill-down이 가능해져 "왜 느린가"에 대한 완전한 답을 얻을 수 있다.
