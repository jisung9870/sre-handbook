● Terraform Deep Dive

Terraform 내부 동작과 설계 패턴

State 관리, Module 설계, import/moved, Provider 내부, 프로젝트 구조 — 대규모 인프라를 안전하게 코드로 관리.

Core

State 관리

terraform.tfstate — Terraform이 "현재 인프라 상태"를 기억하는 유일한 방법
State 파일은 Terraform이 실제 인프라와 코드의 매핑을 저장하는 데이터베이스다. terraform plan은 코드와 state를 비교하여 변경사항을 계산한다. State 손상이나 분실은 인프라 전체를 잃는 것과 같다.
Remote State 필수
로컬 state 금지. 팀 환경에서는 state 충돌과 유실 위험이 크다.

S3 + DynamoDB (AWS):
backend "s3" { bucket = "my-tf-state" key = "prod/vpc/terraform.tfstate" region = "ap-northeast-2" dynamodb_table = "tf-lock" encrypt = true }
Locking: DynamoDB로 동시 실행 방지. 같은 state에 두 사람이 apply하면 하나가 lock 대기
Versioning: S3 버킷 버저닝 활성화. state 손상 시 이전 버전 복구
State 운영 명령
terraform state list # 관리 중인 리소스 목록 terraform state show aws_vpc.main # 리소스 상세 terraform state mv # 리소스 주소 변경 (리팩토링 시) terraform state rm # state에서 제거 (실제 인프라 유지) terraform state pull # remote state 다운로드 terraform state push # state 강제 업로드 (위험!)
State 분리 원칙:
1. 환경별: prod / staging / dev 각각 별도 state 2. 계층별: network / compute / app 분리 3. 폭발 반경 최소화: state 하나가 관리하는 리소스는 100개 이하 권장
State 사고 대응
Lock 해제 안 될 때: terraform force-unlock LOCK_ID (DynamoDB에서 확인)
State 손상: S3 버저닝에서 이전 state 복원 → terraform state push
State와 실제 인프라 불일치: terraform refresh(deprecated) → terraform apply -refresh-only
State 마이그레이션: backend 변경 시 terraform init -migrate-state

⚠ 절대 하면 안 되는 것: state 파일 수동 편집, state push 없이 다른 backend로 교체, 여러 워크스페이스에서 같은 인프라 관리
Design

Module 설계 패턴

재사용 가능하고 테스트 가능한 Terraform 모듈을 만드는 원칙
좋은 모듈의 조건
1. 단일 책임: 하나의 논리적 리소스 그룹만 관리 예: VPC 모듈, EKS 모듈 2. 명확한 인터페이스: variables.tf(입력), outputs.tf(출력)이 깔끔 3. 합리적 기본값: 대부분의 변수에 default 제공, 필수 입력 최소화 4. 문서화: 변수 description, README.md, examples/ 디렉토리 유지 5. 버전 고정: required_providers에 버전 제약 명시
모듈 구조
modules/ └── eks-cluster/ ├── main.tf # 리소스 정의 ├── variables.tf # 입력 변수 ├── outputs.tf # 출력 값 ├── versions.tf # provider/terraform 버전 ├── locals.tf # 로컬 계산 값 ├── data.tf # data sources ├── README.md # 사용법 문서 └── examples/ └── basic/ └── main.tf # 사용 예시
Composition 패턴
모듈 간 의존성은 output → variable로 연결:
module "vpc" { source = "./modules/vpc" } module "eks" { source = "./modules/eks" vpc_id = module.vpc.vpc_id subnet_ids = module.vpc.private_subnet_ids }
Root Module: modules를 조합하여 하나의 환경을 구성하는 최상위 모듈
anti-pattern: 모듈 안에서 다른 모듈을 호출하는 깊은 중첩은 테스트와 디버깅이 어려워진다. 가능하면 root에서 flat하게 조합
Migration

import / moved / removed

기존 인프라를 Terraform 관리로 가져오고, 리소스 주소를 안전하게 변경하는 기법
import (1.5+ HCL import)
기존 리소스를 state로 가져오기:

CLI 방식 (기존):
terraform import aws_vpc.main vpc-12345
HCL import 블록 (1.5+, 권장):
import { to = aws_vpc.main id = "vpc-12345" }
terraform plan -generate-config-out=generated.tf
리소스 HCL을 자동 생성한 뒤 수동으로 다듬어 apply한다.

주의: import는 state에만 추가된다. HCL 코드는 별도로 작성하거나 `generate-config`를 활용해야 한다.
moved / removed (1.1+/1.7+)
moved (리팩토링): 리소스 주소 변경 시 destroy+create 방지
moved { from = aws_instance.web to = module.compute.aws_instance.web }
plan에서 move로 표시되며 실제 인프라 변경은 없다.

removed (1.7+): state에서 제거하되 실제 리소스 유지
removed { from = aws_instance.legacy lifecycle { destroy = false } }
terraform state rm의 선언적 대안이다.
Internals

Provider 내부 동작

Terraform이 AWS/NHN/K8s API를 호출하는 메커니즘
Provider는 Terraform과 실제 인프라 API 사이의 번역기다. Provider가 각 리소스의 CRUD 로직을 구현하고, Terraform Core가 state와 비교하여 필요한 API 호출을 결정한다.
동작 흐름
terraform plan: 1. State 파일 읽기 2. Provider에게 각 리소스의 현재 상태 요청 (Read) 3. HCL 코드와 비교하여 diff 계산 4. 변경 계획 출력 terraform apply: 1. Plan 확인 2. 의존성 그래프 순서로 Provider API 호출 Create / Update / Delete 3. 결과를 State에 기록
Provider 버전 관리
버전 고정 필수:
required_providers { aws = { source = "hashicorp/aws", version = "~> 5.0" } kubernetes = { source = "hashicorp/kubernetes", version = "~> 2.25" } }
~> 5.0 : 5.x 허용, 6.0 불허 = exact : 정확한 버전, 안전하지만 업데이트 번거로움 >= min : 최소 버전만 보장, 예측 불가 업데이트 위험
Lock file: .terraform.lock.hcl은 반드시 Git에 커밋해 팀원 간 동일 provider 버전을 보장한다.
Structure

프로젝트 구조 패턴

환경별/계층별 분리로 폭발 반경을 최소화하는 디렉토리 구조
# 환경별 × 계층별 분리 (권장 구조) terraform/ ├── modules/ # 재사용 모듈 │ ├── vpc/ │ ├── eks/ │ ├── rds/ │ └── monitoring/ ├── environments/ │ ├── prod/ │ │ ├── network/ # VPC, Subnet, NAT GW │ │ │ ├── main.tf │ │ │ ├── backend.tf # s3 backend (prod/network) │ │ │ └── terraform.tfvars │ │ ├── eks/ # EKS Cluster, Node Groups │ │ ├── data/ # RDS, Redis │ │ └── monitoring/ # LGTM Stack infra │ ├── staging/ │ │ ├── network/ │ │ ├── eks/ │ │ └── ... │ └── dev/ └── .terraform.lock.hcl # Git에 커밋!
💡 State 분리 전략
각 environments/{env}/{layer}/가 독립된 state를 가진다. prod/network를 변경해도 prod/eks의 state에 영향 없음. 계층 간 참조는 terraform_remote_state data source 또는 SSM Parameter Store로.
● ArgoCD Deep Dive

ArgoCD 심화

App-of-Apps, ApplicationSet, Sync 전략, Secret 관리, 멀티클러스터 — 선언적 GitOps의 핵심 도구.

Pattern

App-of-Apps 패턴

하나의 "루트 앱"이 다른 ArgoCD Application들을 관리하는 부트스트랩 패턴
클러스터에 수십 개의 Application을 수동으로 등록하는 대신, 루트 Application 하나를 등록하면 그 안에 정의된 하위 Application들이 자동으로 생성된다. 클러스터 부트스트랩의 표준 패턴.
구조
gitops-repo/ ├── apps/ # 루트 앱이 가리키는 디렉토리 │ ├── traefik.yaml # Application: traefik │ ├── prometheus.yaml # Application: prometheus │ ├── loki.yaml # Application: loki │ ├── api-server.yaml # Application: api-server │ └── ... ├── charts/ # 커스텀 Helm 차트 │ ├── api-server/ │ └── ... └── values/ # 환경별 values ├── prod/ └── staging/
루트 앱 정의
apiVersion: argoproj.io/v1alpha1 kind: Application metadata: name: root-app namespace: argocd spec: project: default source: repoURL: https://github.com/org/gitops path: apps targetRevision: main destination: server: https://kubernetes.default.svc namespace: argocd syncPolicy: automated: prune: true selfHeal: true
⚠ App-of-Apps vs ApplicationSet
App-of-Apps: 각 Application을 YAML 파일로 직접 정의. 유연하지만 반복적
ApplicationSet: 템플릿으로 여러 Application을 자동 생성. DRY하지만 복잡도 증가
CRD 관리를 ApplicationSet에서 개별 Application으로 마이그레이션한 경험이 있다면, 각각의 트레이드오프를 이미 체감했을 것
Generator

ApplicationSet

Generator 기반으로 여러 Application을 템플릿에서 자동 생성
주요 Generator
List: 정적 목록에서 Application 생성. 가장 단순
Git Directory: Git 리포의 디렉토리 구조에서 자동 발견
Git File: 특정 config 파일에서 파라미터 읽기
Cluster: 등록된 클러스터마다 Application 생성
Matrix: 두 Generator를 조합 (클러스터 × 서비스)
Merge: 여러 Generator 결과를 병합

Cluster + Git Matrix:
3개 클러스터 × 5개 서비스 = 15개 Application 자동 생성
주의사항
1. 삭제 정책 ApplicationSet이 삭제되면 생성된 모든 Application도 삭제된다. preserveResourcesOnDeletion: true 로 방어 2. CRD 관리 문제 ApplicationSet으로 CRD 포함 차트를 관리하면 Sync 순서 문제로 CRD 삭제 -> 의존 리소스 전체 삭제 위험이 있다. CRD는 별도 Application으로 분리 관리하는 편이 안전 3. 고유 Application 필요 시 템플릿에 맞지 않는 특수 설정이 필요하면 App-of-Apps로 직접 정의하는 편이 깔끔
Strategy

Sync 전략

Sync Wave, Hook, Prune, Self-Heal — 배포 순서와 자동 동기화 제어
Sync Wave (순서 제어)
annotation으로 리소스 적용 순서 지정:
argocd.argoproj.io/sync-wave: "-1" # 먼저 적용 argocd.argoproj.io/sync-wave: "0" # 기본 (동시) argocd.argoproj.io/sync-wave: "1" # 나중에 적용
활용 예시:
Wave -1: Namespace, CRD 생성 Wave 0 : ConfigMap, Secret, PVC Wave 1 : Deployment, Service Wave 2 : Ingress, HTTPRoute Wave 3 : Post-deploy Job (DB migration)
Sync Hook
Sync 단계에 Hook으로 Job/Pod 실행:
argocd.argoproj.io/hook: PreSync # Sync 전 (DB migration) argocd.argoproj.io/hook: Sync # Sync 중 argocd.argoproj.io/hook: PostSync # Sync 후 (smoke test) argocd.argoproj.io/hook: SyncFail # Sync 실패 시 (알림) argocd.argoproj.io/hook-delete-policy: HookSucceeded
Hook 성공 후 자동 삭제되고, 실패 시에는 남겨 두어 디버깅할 수 있다.
자동 Sync 정책
automated.prune: Git에서 삭제된 리소스를 클러스터에서도 삭제. true 권장이지만 초기에는 false로 시작
automated.selfHeal: 누군가 kubectl로 직접 변경해도 Git 상태로 되돌림. GitOps 핵심 원칙
automated.allowEmpty: Git이 비어있을 때 모든 리소스 삭제 허용. 절대 true로 설정하지 말 것
retry: Sync 실패 시 자동 재시도. limit: 5, backoff.duration: 5s, backoff.maxDuration: 3m

ignoreDifferences: 특정 필드의 diff를 무시 (kubectl annotate 등으로 추가된 annotation). jqPathExpressions으로 세밀 제어
Security

Secret 관리

Git에 Secret을 직접 커밋하지 않고 GitOps로 관리하는 방법
GitOps의 가장 큰 도전: "모든 것을 Git에"라는 원칙과 "Secret을 Git에 넣지 마라"는 원칙의 충돌.
접근법 비교
Sealed Secrets: kubeseal로 암호화 → Git에 커밋 → 클러스터에서 복호화
장점: 단순, ArgoCD 친화적
단점: 키 관리, 키 로테이션 수동

External Secrets Operator: AWS SM, Vault 등에서 Secret 동기화
장점: 외부 시크릿 관리자 활용, 자동 로테이션
단점: 외부 의존성 추가

SOPS (Mozilla): 파일 단위 암호화. age/PGP/KMS 키 사용
장점: Git diff 가능 (일부 필드만 암호화)
단점: 키 관리 복잡
권장 구성
프로덕션 권장: External Secrets Operator
ExternalSecret spec.data[].remoteRef.key: "prod/api-server/db-password"
ArgoCD가 Sync하고, ESO가 AWS Secrets Manager에서 값을 가져와 K8s Secret을 생성한다.

장점:
1. Secret 값은 Git에 없음 2. ExternalSecret 정의는 Git에 있음 3. AWS Secrets Manager에서 자동 로테이션 가능 4. 여러 클러스터에서 같은 시크릿 소스 공유 가능
Multi

멀티클러스터 ArgoCD

하나의 ArgoCD에서 여러 K8s 클러스터를 관리하는 패턴
허브-스포크 모델
관리 클러스터: ArgoCD가 설치된 허브
워크로드 클러스터: ArgoCD가 배포하는 대상
argocd cluster add eks-prod --name eks-prod argocd cluster add nhn-prod --name nhn-prod
Application에서 대상 지정:
spec.destination.server: https://eks-prod-api.example.com # 또는 spec.destination.name: eks-prod
AWS EKS + NHN NKS 구성
ArgoCD 위치: EKS(주) 클러스터에 설치
NHN NKS 등록: kubeconfig 기반으로 클러스터 추가

환경별 values 분기:
# apps/api-server.yaml source.helm.valueFiles: - values/common.yaml - values/{{ .cluster }}/values.yaml
ApplicationSet + Cluster Generator: 등록된 모든 클러스터에 동일 앱을 자동 배포하고, 클러스터별 values만 다르게 가져간다.

Design

차트 설계 원칙

유지보수 가능하고, 환경별로 유연하게 커스텀할 수 있는 차트 구조
차트 구조
charts/api-server/ ├── Chart.yaml # 차트 메타데이터 ├── values.yaml # 기본 values ├── templates/ │ ├── _helpers.tpl # 템플릿 함수 │ ├── deployment.yaml │ ├── service.yaml │ ├── hpa.yaml │ ├── pdb.yaml │ ├── serviceaccount.yaml │ ├── configmap.yaml │ ├── ingress.yaml # or httproute.yaml │ └── tests/ │ └── test-connection.yaml ├── ci/ # CI용 values │ └── test-values.yaml └── README.md
설계 원칙
1. 모든 것을 values로 제어 가능하게 이미지, 리소스, replica, 환경변수 등 하드코딩 금지 2. 조건부 렌더링 {{- if .Values.hpa.enabled }} 로 선택적 리소스 생성 3. 공통 라벨/어노테이션 _helpers.tpl 에 정의해 모든 리소스에 일관 적용 4. 리소스 네이밍 {{ include "chart.fullname" . }} 로 Release별 고유 이름 생성 5. 의존성 최소화 subchart보다 독립적으로 배포 가능한 구조 선호
Values

Values 구조화 전략

기본값, 환경별 오버라이드, 시크릿 분리 — 다층 values 관리
Values 계층
1. values.yaml 차트 내장 기본값. 대부분의 환경에서 동작 2. values-common.yaml 조직 공통 설정. 이미지 레지스트리, 라벨 정책 등 3. values-{env}.yaml 환경별 오버라이드. replica 수, 리소스 크기, 도메인 4. values-secret.yaml 민감 정보. Git에 커밋하지 않고 External Secrets로 대체
ArgoCD에서 다중 values 적용:
spec.source.helm.valueFiles: - values.yaml - values/common.yaml - values/prod.yaml
뒤의 파일이 앞의 값을 오버라이드한다.
Values 설계 패턴
# values.yaml — 좋은 구조 예시 replicaCount: 2 image: repository: registry.example.com/api tag: "" # CI에서 주입 pullPolicy: IfNotPresent resources: requests: { cpu: 100m, memory: 128Mi } limits: { cpu: 500m, memory: 512Mi } hpa: enabled: true minReplicas: 2 maxReplicas: 10 targetCPU: 70 ingress: enabled: true className: traefik hosts: [{ host: api.example.com }]
Testing

Helm 테스트 전략

lint, template, unittest, integration — 차트 품질을 보장하는 다단계 테스트
테스트 레벨

● GitOps Operations

GitOps 운영 패턴

GitOps 원칙, Drift Detection, 환경별 전략 — Git을 Single Source of Truth으로 운영하는 실무.

Principles

GitOps 4대 원칙

OpenGitOps 프로젝트가 정의한 GitOps의 핵심 원칙
4대 원칙
1. Declarative 원하는 상태(desired state)를 선언적으로 기술 예: Kubernetes YAML, Helm values, Terraform HCL 2. Versioned & Immutable 원하는 상태를 Git에 버전 관리 누가, 언제, 왜 바꿨는지 커밋으로 추적 가능 3. Pulled Automatically 에이전트(ArgoCD)가 Git에서 상태를 자동으로 가져와 적용 Push가 아닌 Pull 모델이며, CI가 kubectl apply 하는 것은 GitOps가 아님 4. Continuously Reconciled 실제 상태가 원하는 상태와 다르면 자동으로 수정 Drift를 감지하고 되돌림
💡 Push vs Pull 모델
Push (전통 CI/CD): Jenkins가 kubectl apply 실행. CI에 클러스터 credential 필요. 보안 위험
Pull (GitOps): ArgoCD가 클러스터 안에서 Git을 polling. CI는 Git에만 push. 클러스터 credential이 외부에 노출되지 않음
Detection

Drift Detection & Self-Heal

실제 상태가 Git과 달라졌을 때 감지하고 자동 복구하는 메커니즘
Drift 발생 원인
1. kubectl 직접 수정 kubectl edit deployment 가 가장 흔한 원인 2. 다른 컨트롤러 HPA가 replica 변경, VPA가 resource 변경 3. Admission Webhook 리소스 생성 시 webhook이 필드 추가 4. K8s 기본값 명시하지 않은 필드에 K8s가 기본값 삽입
Drift를 허용해야 하는 경우: HPA의 replica 수, VPA의 resource 값은 ignoreDifferences로 제외한다.
ArgoCD Self-Heal 설정
syncPolicy.automated.selfHeal: true argocd_app_info{sync_status="OutOfSync"} # drift 감지 argocd_app_sync_total{phase="Failed"} # self-heal 실패 trigger: on-sync-status-unknown trigger: on-health-degraded
3분마다 상태를 비교하고 diff를 발견하면 자동으로 Sync한다.

문화: "kubectl로 직접 변경하면 ArgoCD가 되돌린다"를 팀 전체가 이해해야 한다. 긴급 상황에서도 Git에 커밋 → Sync가 원칙이다.
Strategy

환경별 관리 전략

단일 브랜치 vs 멀티 브랜치 vs 디렉토리 분리 — GitOps 리포 구조 설계
디렉토리 분리 (권장)
단일 브랜치(main) + 디렉토리로 환경 분리:
gitops/ ├── apps/ │ └── api-server.yaml # dest별로 cluster 지정 ├── base/ # 공통 manifests ├── overlays/ │ ├── dev/ │ ├── staging/ │ └── prod/ # Kustomize overlay └── values/ ├── dev/ ├── staging/ └── prod/ # Helm values
장점: 한 PR에서 모든 환경 변경 확인 가능. 브랜치 머지 지옥 없음
ArgoCD: targetRevision은 모두 main. path만 다르게
멀티 브랜치 (비권장)
브랜치별 환경: main→prod, develop→staging, feature→dev

단점:
• cherry-pick / merge 지옥
• 환경 간 차이가 브랜치에 숨겨져 파악 어려움
• PR 리뷰가 환경별로 분산

유일한 장점: 환경별 접근 제어 (branch protection). 하지만 ArgoCD Project RBAC으로 대체 가능

결론: 새로 시작한다면 단일 브랜치 + 디렉토리 분리가 가장 깔끔

CI/CD 파이프라인 패턴

Jenkins 고급 패턴, 빌드 캐싱, 아티팩트 관리, GitOps 연동 — 효율적인 딜리버리 파이프라인.

Jenkins

Jenkins 고급 패턴

Shared Library, 에이전트 관리, 병렬 실행, 보안 — 대규모 Jenkins 운영
Shared Library
Jenkinsfile의 중복을 제거하는 공통 함수 라이브러리:
vars/ ├── buildDocker.groovy # def call(Map config) { ... } ├── deployHelm.groovy ├── notifySlack.groovy └── runTests.groovy src/org/company/ └── Pipeline.groovy # 클래스 기반 복잡 로직
@Library('shared-lib') _ buildDocker(image: 'api-server', tag: env.GIT_COMMIT)
장점: DRY, 일관된 파이프라인, 중앙 관리 가능
버전 고정: @Library('shared-lib@v2.0')
K8s Agent
Jenkins 에이전트를 K8s Pod으로 동적 실행:
kubernetes { yaml """ containers: - name: docker image: docker:dind - name: helm image: alpine/helm """ }
장점: 빌드 시에만 리소스를 쓰고, 격리된 환경에서 다양한 도구 컨테이너를 함께 올릴 수 있다.
빌드 캐시:
volumes: [persistentVolumeClaim(claimName: 'build-cache')]
PVC를 마운트해 Go mod cache, npm cache 등을 보존한다.
Patterns

CI/CD + GitOps 연동 파이프라인

CI는 빌드와 이미지 푸시만, 배포는 ArgoCD가 — 역할 분리
GitOps 환경에서 CI는 이미지를 빌드하여 레지스트리에 push하고, GitOps 리포의 이미지 태그를 업데이트하는 것까지만 담당한다. 실제 배포(kubectl apply)는 하지 않는다.
전체 파이프라인
1. 개발자가 app-repo에 코드 Push 2. Jenkins가 PR 트리거 (또는 merge 트리거) 3. CI 파이프라인: a. 코드 Checkout b. 린트 + 단위 테스트 c. Docker 이미지 빌드 (commit SHA 태그) d. 이미지 취약점 스캔 (Trivy) e. 이미지 Push → Container Registry f. GitOps 리포의 values.yaml에서 image.tag 업데이트 g. GitOps 리포에 커밋 + Push 4. ArgoCD가 GitOps 리포 변경 감지 5. ArgoCD가 새 이미지로 배포 (Sync) 6. ArgoCD Notification → Slack 알림
핵심: CI(Jenkins)는 Git에 커밋하는 것까지만 한다. kubectl이나 helm install을 직접 실행하지 않는다. 배포 권한을 CI에서 분리하여 보안 향상 + 감사 추적 용이.

빌드 캐싱 전략

Docker layer cache: --cache-from으로 이전 이미지 레이어 재사용
Go: GOMODCACHE PVC 마운트 → 의존성 다운로드 스킵
npm: .npm 캐시 PVC 마운트
Multi-stage build: 빌드 스테이지와 실행 스테이지 분리 → 최종 이미지 최소화

이미지 태그 전략

Commit SHA: api-server:a1b2c3d. 불변. 어떤 코드인지 추적 가능
latest 금지: 어떤 버전인지 알 수 없음. 롤백 불가
Semantic (릴리스): v1.2.3. 주요 릴리스용
Branch+SHA: main-a1b2c3d. 브랜치 식별 포함
