PostgreSQL SRE 운영

커넥션 관리, Replication, VACUUM, 쿼리 성능 진단, 모니터링 — DB 장애의 80%를 커버하는 핵심.

Connection

커넥션 관리

max_connections, 커넥션 폭발, 풀링 — DB 장애의 가장 흔한 원인
PostgreSQL은 프로세스 기반 모델이다. 클라이언트 커넥션마다 별도의 백엔드 프로세스를 fork한다. 각 프로세스는 ~10MB 메모리를 사용하므로, 커넥션 수가 많으면 메모리 폭증과 context switch 오버헤드가 발생한다.

K8s 환경에서 Pod 수 × Pod당 커넥션 풀 크기가 max_connections를 초과하면 모든 새 연결이 거부된다. 이것이 Cascading Failure의 시작점이 된다.
핵심 파라미터
max_connections — 기본 100. K8s 환경에서 200~500 정도. 무작정 높이면 메모리 부족
superuser_reserved_connections — 기본 3. 관리자 긴급 접근용

메모리 계산:
shared_buffers + (max_connections × work_mem)
max_connections=500, work_mem=4MB → 커넥션만으로 2GB 필요

진단:
SELECT count(*) FROM pg_stat_activity;
SELECT state, count(*) FROM pg_stat_activity GROUP BY state;
K8s에서의 커넥션 관리
문제: Pod 10개 × 풀 크기 20 = 200 커넥션. HPA가 30개로 스케일 → 600. max_connections=200이면 즉시 장애

해결 1 — PgBouncer: 커넥션 풀러를 DB 앞에 배치. Pod→PgBouncer(수천)→PostgreSQL(수십). Transaction 모드 권장
해결 2 — 풀 크기 계산: max_connections ÷ 최대 Pod 수 = Pod당 풀 크기
해결 3 — 앱 레벨 풀링: Go의 sql.DB의 SetMaxOpenConns를 적절히 설정

PgBouncer 모드:
• Session: 커넥션 1:1 매핑. 풀링 효과 없음
• Transaction: 트랜잭션 단위 공유. 가장 효율적
• Statement: 쿼리 단위. prepared statement 불가
HA

Replication & HA

Streaming Replication, Replication Lag, Failover — 고가용성 PostgreSQL 운영
Streaming Replication
비동기 (기본): Primary가 WAL을 보내고 Replica의 확인을 기다리지 않는다. 빠르지만 장애 시 데이터 손실 가능성이 있다
동기: Replica가 WAL 수신을 확인할 때까지 commit을 대기한다. 데이터 손실 가능성은 줄지만 latency가 증가한다
synchronous_commit = on/off/remote_write/remote_apply

Read Replica 활용:
• 읽기 트래픽 분산: 리포트, 분석 쿼리를 Replica로
• 백업: Replica에서 pg_dump 실행 (Primary 부하 회피)
• Cascading: Primary→Replica1→Replica2 체인 가능
Replication Lag
측정:
SELECT now() - pg_last_xact_replay_timestamp() AS lag;
SELECT * FROM pg_stat_replication; (Primary에서)

lag 원인:
• Replica의 디스크 I/O가 느림 (SSD 미사용 등)
• 네트워크 대역폭 부족
• Replica에서 장시간 실행되는 쿼리가 recovery 적용을 막음
• Primary의 대량 쓰기 급증

알림: pg_replication_lag_seconds > 30 → 경고
SLO 영향: lag 동안 Replica에서 최신이 아닌 데이터를 읽을 수 있다
Maintenance

VACUUM & Table Bloat

MVCC의 대가 — dead tuple을 정리하지 않으면 테이블이 무한히 커진다
PostgreSQL의 MVCC(Multi-Version Concurrency Control)는 UPDATE/DELETE 시 이전 row를 즉시 삭제하지 않고 "dead tuple"로 남긴다. 다른 트랜잭션이 참조할 수 있기 때문이다. VACUUM은 이 dead tuple을 정리하여 공간을 재사용 가능하게 하는 프로세스다.

VACUUM이 제대로 작동하지 않으면 table bloat(테이블 비대화)이 발생하여 쿼리 성능이 저하되고, 디스크가 고갈되며, 최악의 경우 Transaction ID Wraparound로 DB가 읽기 전용이 된다.
VACUUM 유형
VACUUM (일반): dead tuple 정리, 공간을 재사용 가능하게. OS에 반환은 안 함
VACUUM FULL: 테이블 재작성. 공간을 OS에 반환. 배타 잠금 필요 → 프로덕션에서 위험
autovacuum: 자동 실행. autovacuum_vacuum_threshold + autovacuum_vacuum_scale_factor × 행 수 = 트리거 기준
ANALYZE: 통계 정보 갱신. 쿼리 플래너가 최적 실행 계획을 세우는 데 필수
autovacuum 튜닝
기본값이 대규모 테이블에 부적절:
autovacuum_vacuum_scale_factor = 0.2 → 1억 행 테이블은 2천만 행이 죽어야 실행
해결: 테이블별 설정
ALTER TABLE big_table SET (autovacuum_vacuum_scale_factor = 0.01, autovacuum_vacuum_threshold = 5000);

autovacuum worker 수:
autovacuum_max_workers = 5 (기본 3)
I/O 제한:
autovacuum_vacuum_cost_limit = 1000 (기본 200. 너무 낮으면 vacuum이 느림)
Transaction ID Wraparound 방지
PostgreSQL의 XID는 32비트 (약 42억). 모든 트랜잭션에 순번이 부여되고, 오래된 XID의 행은 "미래"로 보여 invisible이 된다. VACUUM이 오래된 XID를 frozen으로 마킹하여 이 문제를 방지한다.
위험 신호: age(datfrozenxid)가 2억에 접근하면 DB가 자동으로 읽기 전용 모드에 진입하여 데이터 보호.
모니터링: SELECT datname, age(datfrozenxid) FROM pg_database;
알림: age > 150,000,000 → 긴급 VACUUM 필요
Performance

쿼리 성능 진단

pg_stat_statements, EXPLAIN ANALYZE, 느린 쿼리 추적
pg_stat_statements
필수 확장. 모든 쿼리의 실행 통계를 수집
SELECT query, calls, mean_exec_time, rows, shared_blks_hit, shared_blks_read FROM pg_stat_statements ORDER BY mean_exec_time DESC LIMIT 10;

핵심 지표:
• mean_exec_time: 평균 실행 시간
• calls: 호출 횟수. 느린 쿼리 × 높은 calls = 최우선 최적화
• shared_blks_read: 디스크 읽기. 높으면 인덱스 필요
• blk_read_time: I/O 대기 시간 (track_io_timing = on 필요)
EXPLAIN ANALYZE
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT) SELECT ...;

주의할 노드:
• Seq Scan on large table → 인덱스 필요
• Nested Loop with large outer → JOIN 순서 확인
• Sort → sort_mem 부족하면 디스크 정렬
• actual rows ≫ plan rows → 통계 부정확. ANALYZE 필요

느린 쿼리 로깅:
log_min_duration_statement = 1000 (1초 이상 쿼리 로그)
log_statement = 'none' (모든 쿼리 로그는 성능 저하)

Memory

Redis 메모리 관리

maxmemory, eviction policy, 메모리 단편화 — Redis 장애의 핵심 원인
Redis는 모든 데이터를 메모리에 보관한다. 메모리가 가득 차면 eviction policy에 따라 키를 삭제하거나, 새 쓰기를 거부한다. maxmemory를 설정하지 않으면 시스템 메모리를 모두 소진하여 OOM Kill될 수 있다.
maxmemory & Eviction
maxmemory 4gb — 반드시 설정. 시스템 메모리의 70~80%

Eviction Policy:
noeviction — 쓰기 거부 (기본값. 데이터 손실 방지)
allkeys-lru — 모든 키 중 LRU 삭제 (캐시 용도 최적)
volatile-lru — TTL이 있는 키만 LRU 삭제
allkeys-lfu — 사용 빈도 기반 삭제 (4.0+)

캐시: allkeys-lru 또는 allkeys-lfu
세션 저장소: volatile-ttl (TTL 짧은 것부터 삭제)
영속 데이터: noeviction (데이터 손실 불가)
메모리 단편화
mem_fragmentation_ratio = used_memory_rss / used_memory
• > 1.5: 심한 단편화. 메모리 낭비
• < 1.0: 스왑 사용 중. 매우 위험
• 1.0~1.5: 정상

원인: 크기가 다양한 키의 생성/삭제 반복
해결:
• activedefrag yes (4.0+, 온라인 defrag)
• Redis 재시작 (가장 확실)
• jemalloc 튜닝 (MALLOC_CONF)

INFO memory로 전체 메모리 상태 확인

⚠ RDB fork() 문제

HA

HA 아키텍처: Sentinel vs Cluster

자동 장애 감지와 장애 조치를 위한 두 가지 접근
Redis Sentinel
용도: Master-Replica 구성의 자동 장애 조치
• Sentinel 프로세스 3+ 가 Master를 감시
• Master 장애 감지 → Replica를 새 Master로 승격
• 클라이언트에게 새 Master 주소 통지

장점: 단순. 데이터 파티셔닝 불필요
단점: 단일 Master가 쓰기 병목. 수평 확장 불가
적합: 소~중규모, 단일 인스턴스로 충분한 경우
Redis Cluster
용도: 데이터를 여러 노드에 분산 (16384 hash slots)
• 각 Master가 slot의 일부를 담당
• 각 Master에 1+ Replica
• Master 장애 시 해당 slot의 Replica가 승격

장점: 수평 확장. 쓰기 분산. 최대 1000 노드
단점: 멀티키 연산 제한 (같은 slot만), 운영 복잡도 높음
적합: 대규모 데이터, 높은 쓰기 처리량 필요 시
Migration

Valkey 전환

Redis의 라이선스 변경(SSPL) → Linux Foundation 산하 Valkey (BSD 라이선스)로의 전환
Redis가 7.4부터 SSPL/RSALv2 듀얼 라이선스로 전환하면서, 클라우드 서비스 제공자와 오픈소스 사용자에게 제약이 생겼다. Valkey는 Redis 7.2.4에서 fork된 BSD 라이선스 프로젝트로, Linux Foundation이 관리한다.
Valkey 장점
라이선스: BSD 3-Clause (진정한 오픈소스)
거버넌스: Linux Foundation (벤더 중립)
호환성: Redis 7.2 프로토콜 완전 호환
성능: io-threads 개선, 메모리 효율 향상
후원: AWS, Google, Oracle, Ericsson 등

전환 근거: BSD 라이선스로 법적 리스크 제거, 낮은 메모리 사용, 커뮤니티 주도 개발
전환 방법
Drop-in replacement에 가깝다: 설정과 프로토콜 호환성이 높아 바이너리 교체만으로 전환 가능한 경우가 많다
• Redis conf 파일 그대로 사용 가능
• 클라이언트 라이브러리 변경 불필요
• RDB/AOF 파일 호환

K8s 전환:
1. Helm chart에서 이미지를 valkey/valkey:8로 변경
2. ConfigMap의 설정 파일은 그대로 유지
3. Rolling update로 Pod 교체
4. 클라이언트 연결 문자열 변경 없음
● Kafka for SRE

Kafka 운영 핵심

파티션 설계, Consumer Lag, ISR, 장애 대응 — 이벤트 스트리밍 플랫폼의 SRE 관점.

Architecture

Kafka 아키텍처 핵심

Topic → Partition → Broker — 분산 로그 기반 메시지 시스템
Kafka는 "분산 커밋 로그"다. 메시지를 순서대로 추가(append-only)하고, Consumer가 offset을 관리하여 원하는 위치부터 읽는다. 파티션 단위로 병렬 처리하고, Replica로 내구성을 보장한다.
핵심 개념
Topic: 메시지의 논리적 카테고리. 여러 Partition으로 분할
Partition: 순서 보장 단위. 병렬 처리의 단위. Consumer Group 내 파티션:Consumer = 1:1
Broker: Kafka 서버. 여러 Partition을 호스팅
Replica: 파티션의 복제본. Leader + Follower(ISR)
ISR (In-Sync Replicas): Leader와 동기화된 Replica 집합. ISR에서 빠지면 데이터 손실 위험
Consumer Group: 파티션을 나눠 읽는 Consumer 집합
파티션 설계
파티션 수 = 최대 병렬 Consumer 수
파티션 3개 → Consumer 최대 3개 동시 처리

너무 적으면: 병렬성 부족
너무 많으면: Leader 선출 시간 증가, 메모리/파일 핸들 증가, rebalancing 느림

가이드:
• 목표 throughput ÷ Consumer당 처리량 = 최소 파티션 수
• 보통 6~12개로 시작, 필요 시 증가 (감소 불가!)
• 순서 보장이 필요한 키(user_id 등)는 같은 파티션으로 라우팅
Operations

Kafka 운영 핵심 포인트

Consumer Lag, ISR 관리, 보존 정책, 모니터링 — 안정적 운영을 위한 핵심
Consumer Lag
정의: (Latest Offset - Consumer Offset). 처리되지 않은 메시지 수

Lag 원인:
• Consumer 처리 속도 < Producer 속도
• Consumer 재시작/rebalancing 중 처리 중단
• 느린 외부 시스템 호출 (DB 쓰기 등)

대응:
• Consumer 인스턴스 추가 (파티션 수까지)
• 배치 크기 증가 (max.poll.records)
• 처리 로직 최적화

알림: kafka_consumer_group_lag > 10000 → 경고
Prometheus: kafka_exporter 또는 burrow
ISR & 내구성
min.insync.replicas: 쓰기 시 최소 동기화 Replica 수
min.insync.replicas=2 + acks=all → 2개 Replica에 기록되어야 commit

ISR Shrink: Follower가 뒤처지면 ISR에서 제거됨
원인: 느린 디스크, 네트워크 지연, GC pause
알림: kafka_server_replica_manager_isrshrinksrate > 0

Under-Replicated Partitions:
kafka.server:type=ReplicaManager,name=UnderReplicatedPartitions
0이어야 정상. > 0이면 데이터 손실 위험

RPO & RTO 설계

Recovery Point Objective(얼마나 잃을 수 있는가) × Recovery Time Objective(얼마나 빨리 복구할 수 있는가)
RPO는 "장애 시 최대 몇 분/시간의 데이터 손실을 허용하는가"이고, RTO는 "장애 후 최대 몇 분/시간 내에 서비스를 복구해야 하는가"이다. 이 두 값이 백업 빈도, 복제 방식, DR 아키텍처, 비용 수준을 결정한다.
RPO/RTO 설계
RPO = 0 → 동기 복제 필수 (비용 최고) RPO = 1분 → 비동기 복제 + 빈번한 WAL 아카이브 RPO = 1시간 → 시간별 스냅샷 RPO = 24시간 → 일별 백업 RTO = 분 단위 → Hot standby (항상 실행 중인 Replica) RTO = 시간 단위 → Warm standby (스냅샷에서 복원) RTO = 일 단위 → Cold backup (테이프/S3에서 복원)
서비스 등급별 가이드
Tier 1 (결제 DB): RPO=0, RTO=5분. 동기 복제 + 자동 장애 조치
Tier 2 (서비스 DB): RPO=1분, RTO=30분. 비동기 복제 + 자동 장애 조치
Tier 3 (로그/분석): RPO=1시간, RTO=4시간. 시간별 스냅샷
Tier 4 (아카이브): RPO=24시간, RTO=24시간. 일별 백업

비용 원칙: RPO/RTO를 절반으로 줄이면 비용은 대체로 2배 가까이 증가
Strategy

백업 전략

논리 백업 vs 물리 백업, PITR, 백업 검증 — 백업이 없으면 데이터도 없다
PostgreSQL 백업
논리 백업 (pg_dump):
pg_dump -Fc -h localhost -U postgres mydb > backup.dump
장점: 버전 간 이식성, 선택적 복원
단점: 대규모 DB에서는 느리고, 일관성 확보를 위해 부하가 커질 수 있음

물리 백업 (pg_basebackup):
pg_basebackup -h localhost -D /backup/base -Fp -Xs -P
장점: 빠름, 전체 클러스터 복원
단점: 같은 메이저 버전에서만 복원

PITR (Point-In-Time Recovery):
기본 백업 + WAL 아카이브로 특정 시점까지 복원
recovery_target_time = '2025-03-20 14:00:00'
RPO를 최소화하는 가장 강력한 방법
백업 검증 — 가장 중요
"검증하지 않은 백업은 백업이 아니다."

자동 검증 CronJob:
1. 백업 파일에서 별도 인스턴스로 복원
2. 기본 데이터 정합성 쿼리 실행
3. 결과를 메트릭으로 Prometheus에 push
4. 실패 시 알림

검증 빈도:
• Tier 1: 매일
• Tier 2: 주 1회
• Tier 3/4: 월 1회

DR 훈련: 분기 1회 백업에서 전체 복원 연습. 실제 RTO 측정
DR

Disaster Recovery 설계

리전/AZ 장애에서 서비스를 유지하는 아키텍처 패턴
전략RPORTO구성비용
Active-Active00 (무중단)양쪽 리전에서 동시 서비스. 데이터 동기 복제최고 (2배+)
Active-Passive (Hot)~0분 단위대기 리전에 실시간 복제. DNS 전환으로 장애 조치높음 (1.5배)
Active-Passive (Warm)분~시간시간 단위대기 리전에 주기적 복제. 스케일업 후 서비스중간
Backup-Restore (Cold)시간~일시간~일백업만 외부 저장. 장애 시 새 인프라에 복원최저
💡 멀티클라우드 DR 전략
AWS EKS(주) + NHN Cloud NKS(DR) 구성에서:
• DB: PostgreSQL Streaming Replication (AWS → NHN) 또는 WAL 아카이브를 S3 → NHN Object Storage로 복제
• 앱: ArgoCD App-of-Apps로 두 클러스터에 동일 배포. DNS 전환으로 트래픽 이동
• Stateless 서비스: 양쪽 모두 Active-Active 가능
• Stateful 서비스: 데이터 복제가 핵심. cross-cloud 네트워크 지연 시간을 고려해야 함

Connection Pooling 아키텍처

PgBouncer, ProxySQL — DB 커넥션 폭발을 방지하는 인프라 계층
PgBouncer (PostgreSQL)
배포 위치:
• DB 사이드카: DB Pod 옆에 배치. 가장 단순
• 독립 Deployment: 별도 Pod으로 배포. DB와 분리된 스케일링
• 앱 사이드카: 각 앱 Pod 옆에 배치. 로컬 연결로 latency 최소화

Transaction 모드 설정:
pool_mode = transaction
max_client_conn = 1000
default_pool_size = 20
min_pool_size = 5
reserve_pool_size = 5
server_lifetime = 3600
사이징 가이드
DB측 max_connections:
default_pool_size × DB 수 + reserve + 관리용
예: 20 × 2 + 5 + 3 = 48

Little's Law 적용:
목표 1000 TPS, 평균 쿼리 시간 5ms
필요 커넥션 = 1000 × 0.005 = 5 (이론치)
여유 3배 = 15 → default_pool_size = 15
