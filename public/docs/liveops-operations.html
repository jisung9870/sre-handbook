LiveOps
= Continuous Game Service

● Overview

LiveOps 운영

출시 후 게임이 살아있는 동안 매주·매일 반복되는 운영 활동의 총칭. 점검·공지·이벤트·핫픽스·콘텐츠 업데이트를 안정적으로 굴리는 일이 SRE의 핵심 일상 업무다.

LiveOps란?

정의

Live Operations의 약자. 출시된 게임을 지속적으로 운영하며 콘텐츠·이벤트·밸런스를 갱신하는 모든 활동. 일회성이 아니라 매주/매일 반복되는 운영 사이클.

왜 중요한가

F2P 게임은 출시 자체보다 출시 후 운영이 매출의 90%를 만든다. LiveOps 품질이 게임 수명을 결정. 안정적 운영 = 신뢰 = 결제 = 매출.

주요 활동

점검·핫픽스, 공지·배너, 이벤트(시즌·기간 한정), 가챠·상품 변경, 보상 지급, 패치 노트 발행, A/B 테스트 (P6 참조), 사고 대응.

SRE의 위치

운영팀(GM)·기획·QA·CS 모두와 매주 협업. LiveOps의 안전망이자 자동화 책임자. 잘못된 데이터가 라이브에 적용되면 즉시 롤백할 수 있어야 함.

운영 사이클 (일·주·월)

주기활동담당
일일대시보드 모니터링, 핫픽스 데이터 변경, CS 문의 대응, 어뷰징 검토SRE·운영·CS
주간 (정기 점검)주간 점검(보통 1~3시간), 콘텐츠 업데이트, 이벤트 시작/종료, 패치 노트전 팀
월간시즌 종료·시작, 메이저 이벤트, 가챠 신상 출시, KPI 리뷰전 팀
분기/연간대형 업데이트(신캐릭터·신지역), 주요 시스템 변경, 1주년 등 마일스톤전 팀
★ NOTE
장르마다 cadence가 다르다. PC MMO·MOBA는 1~2주 단위 패치(예: League of Legends는 14일 cadence, 2026년 24개 패치 예정), 모바일 RPG는 주간 점검 + 월간 신챕터, 캐주얼은 주간 이벤트 중심. 자기 게임의 패턴을 표준화하고 자동화하는 게 핵심.

SRE 책임 (LiveOps 관점)

① 점검 인프라

점검 페이지·점검 모드 토글·세션 드레인. 모든 게임이 같은 패턴으로 점검할 수 있게 표준화.

② 공지·이벤트 시스템

공지 CMS, 인앱 메시지·푸시 채널, 이벤트 스케줄러 운영. 운영팀이 코드 없이 작업할 수 있게 도구화.

③ 변경의 안전망

Feature Flag·remote config·롤백 자동화. 잘못된 변경이 라이브에 가도 즉시 되돌릴 수 있는 인프라.

④ 외부 협업

외부 개발사 빌드 검수·인수 테스트·점검 동기화. 시차·언어·도구 차이를 메우는 워크플로우.

● Maintenance

점검 운영

게임 서비스를 일시 중단하고 변경을 적용하는 가장 흔한 LiveOps 활동. 사용자 영향과 변경 안정성의 균형이 핵심.

점검 유형

정기 점검 (Scheduled)

매주 같은 요일·시간(예: 매주 화요일 오전 6~9시). 사용자가 익숙해 영향 최소. 콘텐츠 업데이트·게임 데이터 변경의 기본 슬롯.

긴급 점검 (Emergency)

예상 못한 사고·치명 버그·결제 오류 등. 사전 공지가 짧거나 없음. 사용자 신뢰에 큰 타격이라 최후의 수단.

부분 점검 (Partial)

특정 서비스만 (예: 매치메이킹·길드전·결제) 점검. 게임 본체는 계속 플레이 가능. 사용자 영향 적음.

무점검 운영 (Zero Downtime)

롤링 배포·feature flag로 점검 없이 변경. 인프라가 받쳐줘야 가능. 모든 변경이 무점검은 어려움(DB 스키마 변경 등).

점검 페이지 시스템

역할

점검 중 게임 부팅 시 사용자에게 보여주는 페이지. 왜·언제까지·뭐가 바뀌는지 명확히. 점검 모드 토글로 자동 전환.

다국어 지원

글로벌 출시 게임은 region별 다국어 필수. 한국어·영어·일본어·중국어(번체/간체) 등. 운영팀이 코드 없이 입력 가능한 CMS.

외부 도구

Atlassian Statuspage, OneUptime, BetterStack 등 도입 가능. 게임은 자체 CMS도 흔함 (게임 클라가 직접 호출).

점검 페이지 베스트 프랙티스

① 명확한 시간(KST 시간대 명시), ② 무엇이 바뀌는지 1줄 요약, ③ 보상 안내(있으면), ④ 진행 중 업데이트, ⑤ 종료 시 명확한 복구 안내. 침묵은 가장 큰 적.

점검 윈도우 설계

점검 시간 선택은 사용자 영향을 최소화하면서 작업 안정성을 확보하는 트레이드오프.

요소고려 사항
시간대해당 region의 사용자 수가 가장 적은 시간 (한국은 보통 새벽 5~9시). 글로벌 동시 출시 게임은 region별 점검 필요
요일주말 회피(놀러 오는 사용자), 금요일·일요일 회피(주말 직전·당일 사고 시 대응 인력 부족)
예상 시간"1시간 점검"이라 공지하고 30분 일찍 끝나는 게 좋음. 지연은 사용자 신뢰를 크게 깎음
마진예상 시간의 1.5~2배를 윈도우로. 작업 실패 시 롤백·재시도 시간까지 확보
사전 공지최소 24시간 전, 큰 변경은 일주일+ 전. 시작 직전 한 번 더 알림
복구 검증점검 종료 전 핵심 시나리오(로그인·결제·매치) 자동 검증. 통과 후에 점검 페이지 내림
⚠ PITFALL
"점검 시간 30분 연장" 공지는 사용자 분노의 가장 큰 원인 중 하나. 차라리 처음부터 충분한 윈도우를 잡고 일찍 끝내는 게 신뢰에 좋음. 지연 시에도 침묵하지 말고 정직하게 새 ETA 안내가 원칙.

부분 점검 · 세션 드레인

전체 점검을 피하고 영향을 줄이는 기법. 인프라가 받쳐줘야 가능.

세션 드레인

새 사용자 접속을 막고, 기존 접속 세션은 자연 종료될 때까지 대기. 롤링 배포·서버 교체의 표준 패턴. 보통 5~30분 소요.

기능별 토글

매치메이킹·결제·길드전 같은 개별 기능을 끄고 점검. 게임 본체는 계속 플레이 가능. 해당 기능의 의존성(예: 결제는 인벤토리·계정에 의존)을 잘 끊어야 함.

region별 분할 점검

한국 점검 → 일본 점검 → 글로벌 점검 식으로 region별 시차 활용. 인력 분산도 가능. 글로벌 게임에 흔함.

주의

부분 점검 시에도 사용자에게 정확히 무엇이 막혔는지 안내. "결제만 점검 중"이라고 명확히 표시하지 않으면 사용자는 전체 다운으로 인식.

● Announcements & Events

공지 & 이벤트

사용자에게 새 콘텐츠·이벤트·점검을 알리는 채널과 시스템. 보상 지급 자동화로 운영팀 부담 최소화.

공지 채널

채널특성활용
인게임 공지로그인 후 팝업·공지 게시판사용자가 가장 많이 보는 채널. 핵심 정보
점검 페이지로그인 전, 점검 시 표시점검 안내 전용
인앱 메시지·배너특정 화면·시간에 표시이벤트 알림, 결제 유도, 신규 콘텐츠
푸시 알림 (FCM/APNs)앱 외부에서 수신. 옵트아웃 가능중요 알림, 이벤트 시작, 보상 도착
이메일가장 격식 있는 채널약관 변경·휴면 안내·계정 보안
홈페이지 / 카페 / 디스코드외부 채널, 영구 게시패치 노트, 자세한 안내, 커뮤니티 응대
SNS (X, 페이스북)커뮤니케이션 + 마케팅실시간 사고 안내·이벤트 홍보
♥ KEY
중요 공지는 여러 채널 동시 발송. 사용자마다 보는 채널이 다르고, 한 채널이 다운돼도 다른 채널로 도달. 단, 메시지는 일관되게 — 채널마다 다른 내용이면 신뢰 잃음.

이벤트 스케줄러

역할

이벤트 시작·종료 시각·보상·조건을 사전 등록. 시간 도달 시 자동 활성화·종료. 운영팀이 코드 없이 등록 가능한 백오피스.

구성 요소

이벤트 정의(이름·기간·조건), 보상 정의(아이템·캐시·등급), 노출 대상(전체/세그먼트/개인), 진행률 추적(랭킹·누적), UI 리소스(배너·아이콘).

주의 사항

① 시간대(KST·UTC) 명확히, ② 시간이 지나도 자동 종료 안 되는 버그 (자주 발생), ③ 종료 직전 마지막 1분 트래픽 폭증 대비, ④ 시계 동기화 (서버 시간 기준).

SRE 모니터링

예약된 이벤트 누락 알림(스케줄러 다운 시), 이벤트 시작/종료 실패 알림, 보상 미지급 모니터링.

● Hotfix & Game Data

핫픽스 & 게임 데이터

긴급 변경을 라이브에 안전하게 적용하는 4가지 메커니즘 — 클라이언트 패치·서버 핫픽스·게임 데이터·Feature Flag.

데이터 기반 변경 — 게임 데이터(테이블)

왜 데이터로?

모바일은 스토어 심사가 끼어 클라 패치는 며칠. 그래서 아이템·드롭률·이벤트·캐릭터 능력치 등 자주 바뀌는 값은 서버에서 데이터로 내려주는 설계가 표준.

변경 워크플로우

기획 작성 → 검수 PR → 스테이징 검증 → 라이브 적용 → 클라이언트 자동 갱신(다음 부팅 또는 polling). 모든 변경은 버전 관리 + 감사 로그.

위험

데이터 변경의 영향 범위를 정확히 파악하기 어려움. 한 값 변경이 다른 시스템(밸런스·매출)에 연쇄 영향. 스테이징·QA·소수 사용자 카나리 단계 필수.

SRE 인프라

데이터 적용 자동화, 검증 도구, 즉시 롤백 버튼, 변경 알림(슬랙·이메일). 운영팀이 SRE 없이도 안전하게 작업할 수 있게 도구화가 SRE의 일.

Feature Flag · Remote Config

코드 배포 없이 기능을 켜고 끄는 메커니즘. 신기능 점진 노출, 긴급 차단, A/B 테스트(P6)에 모두 사용.

LaunchDarkly

업계 표준 enterprise 플랫폼. Progressive Rollout(점진 비율 노출)·Guarded Rollout(메트릭 모니터링 + 회귀 시 자동 일시정지). 모바일 변경 즉시 반영.

GrowthBook

오픈소스, warehouse-native. 기존 데이터 웨어하우스 위에서 동작 → 비용 효율적. 게임·핀테크 등 컴플라이언스 강한 도메인에 적합. P6에서도 다룸.

Firebase Remote Config

Google 무료 (Firebase 생태계). AI 기반 personalization(예: 게임 난이도 자동 조정). CrazyLabs·Halfbrick 등 게임 사례 다수.

AWS AppConfig

AWS 생태계 통합. 배포 없이 config 변경 + 검증 + 모니터링. Pay-as-you-go (config 조회마다 과금). 큰 트래픽 시 비용 통제 주의.

♥ KEY
"이 기능은 토글 가능하게 만들었나?"가 SRE의 새 기능 검토 체크리스트 1번. Feature Flag 없는 신기능은 사고 시 핫픽스 배포에 의존해야 함. 토글 가능 = 사고 시 즉시 차단 가능.

롤백 정책

롤백 결정 기준

신규 기능 배포 후 임계 메트릭(에러율·결제 성공률·KPI) 모니터링. 임계 초과 시 자동 알림 → 1차 운영자 결정 → 즉시 롤백.

롤백 vs 핫픽스

"앞으로 가는 핫픽스" vs "뒤로 가는 롤백" 선택. 영향 범위·신뢰도·시간으로 판단. 의심스러우면 일단 롤백 → 분석 → 다시 정상 배포가 안전.

데이터 변경 롤백

이전 버전의 데이터 set을 별도 보관 → 즉시 재적용. 변경 후 N분 동안은 이전 버전을 hot 상태로 유지.

롤백 불가능 케이스

DB 스키마 변경·데이터 마이그레이션 후. 이런 변경은 backward-compatible하게 설계해 롤백 가능성 확보.

R&R 명확화

계약 시점에 LiveOps R&R 명문화 (P1 가이드 참조). "긴급 핫픽스는 누가 결정?" 같은 질문이 매번 나오면 안 됨.

공유 채널

슬랙/디스코드 공동 채널, 공유 일정표(Notion·Confluence), 공동 점검 캘린더. 시차로 인한 정보 누락 방지.

빌드 검수 · 인수 테스트

인수 기준

SDK 통합 확인(계정·결제·텔레메트리), 자동 회귀 테스트, 보안 스캔(악성코드·의존성 취약점), 성능 회귀 테스트. 기준 미달이면 거부.

스테이징 환경 제공

외부 개발사가 빌드 검증할 수 있는 퍼블리셔 측 스테이징(통합 계정·결제·텔레메트리 다 연결된 환경). 운영 환경과 분리.

긴급 핫픽스 fast track

치명 버그 발생 시 검수 단계 일부 단축. 최소 보안 + 통합 확인은 절대 스킵 안 함. 핫픽스 후 정밀 검수는 이후에.

피드백 루프

검수 결과를 개발사에 명확히 전달. 자주 발생하는 회귀 패턴은 사전 체크리스트로 표준화 → 검수 시간 단축.

시차 동기화

한국 퍼블리셔 + 일본·동남아·구미 개발사 협업 시 시차가 큰 변수. "한국 점검 시각이 개발사 새벽"인 상황이 자주 발생.

점검 시각 협상

점검은 사용자 시각 기준. 외부 개발사 인력이 새벽이라도 정기 점검은 한국 시간 기준으로 가는 경우가 흔함. 긴급 핫픽스는 양사 인력이 동시에 가능한 시간으로 협상.

온콜 분담

퍼블리셔 영역(플랫폼 SDK·결제·계정)은 퍼블리셔 온콜, 게임서버 영역은 개발사 온콜. 사고 발생 시 즉시 양사 워룸 가동.

자동화 우선

시차로 인한 인력 한계는 자동화로 해결. 점검 시작·종료, 보상 지급, 모니터링 알림 모두 자동화. 사람 개입 최소화가 시차 협업의 핵심.

비동기 커뮤니케이션

실시간 회의 줄이고 문서·티켓·녹화 영상으로 비동기 처리. "내가 자는 동안 일이 진행될 수 있게"가 글로벌 협업의 원칙.
