● Layer Models

OSI 7계층 vs TCP/IP 4계층

각 계층 역할과 실무에서 문제가 생기는 지점.

OSI 7계층과 TCP/IP 4계층을 비교하고, 실무에서는 어떤 모델을 쓰나요?
OSI는 교육/참조 모델이고, 실무에서는 TCP/IP 4계층(Link→Internet→Transport→Application)을 사용합니다. OSI의 5~7계층(Session/Presentation/Application)은 TCP/IP에서 하나의 Application 계층으로 통합됩니다. 장애 진단 시 "L3 문제(라우팅)인가? L4 문제(포트/방화벽)인가? L7 문제(앱 로직)인가?"로 계층을 나눠 접근합니다.
OSITCP/IP프로토콜K8s 연결
L7 ApplicationApplicationHTTP, gRPC, DNSIngress, Gateway API, Istio VirtualService
L6 PresentationTLS, SSLcert-manager, mTLS
L5 SessionWebSocket, gRPC stream서비스 메시 세션 관리
L4 TransportTransportTCP, UDPService (ClusterIP), kube-proxy, NodePort
L3 NetworkInternetIP, ICMPPod CIDR, CNI, NetworkPolicy
L2 Data LinkLinkEthernet, ARPveth, bridge, VXLAN
L1 Physical전기 신호NIC, 케이블
실무에서 네트워크 문제가 가장 자주 발생하는 계층은?
L7(앱) > L4(방화벽/포트) > L3(라우팅) 순서로 빈번합니다. L7에서는 CORS, TLS 인증서 만료, HTTP 타임아웃이 흔하고, L4에서는 SecurityGroup 미설정, 포트 미개방이 흔하고, L3에서는 라우팅 테이블 오류, VPN CIDR 충돌이 발생합니다.
실무 경험 연결:
• L7 — NHN NKS CORS 403: kube-proxy의 SNAT가 LB 소스 IP를 변환 → Traefik Proxy Protocol trust가 실패 → CORS 검증 실패. externalTrafficPolicy: Local로 해결. 이 사례는 L4(SNAT)와 L7(CORS)이 결합된 문제
• L4 — SecurityGroup: 새 서비스 배포 시 포트를 SG에 추가하지 않아 타임아웃. 에러 메시지가 없어 진단이 어려움 → telnet host port로 L4 연결 확인
• L3 — VPN CIDR 충돌: AWS VPC CIDR과 NHN Cloud 대역이 겹쳐 VPN 라우팅 실패

진단 순서:
ping host → L3 확인 (ICMP)
telnet host port → L4 확인 (TCP 연결)
curl -v https://host → L7 확인 (HTTP 응답)
● TCP Deep Dive

TCP 핵심

3-way/4-way handshake, 흐름제어, 혼잡제어 — SRE가 반드시 알아야 하는 TCP 내부.

TCP 3-way handshake와 4-way handshake를 설명해 주세요. TIME_WAIT은 왜 존재하나요?
3-way handshake(SYN→SYN-ACK→ACK)로 연결을 설정하고, 4-way handshake(FIN→ACK→FIN→ACK)로 연결을 종료합니다. TIME_WAIT(기본 60초)는 연결 종료를 먼저 시작한 쪽에서 발생하며, 네트워크에 남아있을 수 있는 지연된 패킷이 새 연결에 영향을 주지 않도록 보장합니다.
3-Way Handshake (연결 설정): Client → SYN (seq=x) → Server Client ← SYN-ACK (seq=y, ack=x+1) ← Server Client → ACK (ack=y+1) → Server → 1 RTT (SYN + SYN-ACK) 후 데이터 전송 시작 4-Way Handshake (연결 종료): Client → FIN (seq=x) → Server Client ← ACK (ack=x+1) ← Server (Server 아직 보낼 데이터 있을 수 있음) Client ← FIN (seq=y) ← Server (Server도 종료 준비 완료) Client → ACK (ack=y+1) → Server Client: TIME_WAIT 상태 (60초) → CLOSED
TIME_WAIT이 문제가 되는 경우:
고빈도 HTTP 연결 (매 요청마다 연결/종료)에서 TIME_WAIT 소켓이 수만 개 쌓이면 소스 포트(ephemeral port, 32768~60999) 고갈. 새 연결을 만들 수 없게 됨.

해결:
• net.ipv4.tcp_tw_reuse = 1 — TIME_WAIT 소켓을 새 연결에 재사용 (안전)
• HTTP keep-alive로 연결 재사용 → TIME_WAIT 자체를 줄임
• K8s에서: Traefik/Nginx의 upstream keepalive 설정으로 백엔드 커넥션 재사용

SYN Flood 공격: SYN만 대량 전송하고 SYN-ACK에 응답하지 않아 서버의 SYN Queue를 소진시키는 DDoS. net.ipv4.tcp_syncookies = 1로 방어. SYN Queue 없이 SYN-ACK에 쿠키를 담아 상태 없이 검증.
TCP 흐름제어(Flow Control)와 혼잡제어(Congestion Control)의 차이를 설명해 주세요.
흐름제어는 "수신자가 처리할 수 있는 속도로 보내기"(수신자 보호), 혼잡제어는 "네트워크가 감당할 수 있는 속도로 보내기"(네트워크 보호)입니다. 흐름제어는 TCP 윈도우(rwnd)로, 혼잡제어는 혼잡 윈도우(cwnd)로 구현되며, 실제 전송 속도는 min(rwnd, cwnd)로 결정됩니다.
흐름제어 — Sliding Window:
수신자가 Window Size 헤더로 "나는 X바이트까지 받을 수 있다"를 알림. 송신자는 이 범위 내에서만 ACK 없이 전송. Window Size = 0이면 전송 중단 (수신 버퍼 가득참).

혼잡제어 — CWND 알고리즘:
Slow Start: cwnd = 1 MSS → ACK마다 cwnd × 2 (지수 증가) → cwnd가 ssthresh에 도달하면 Congestion Avoidance로 전환 Congestion Avoidance: ACK마다 cwnd += 1 MSS (선형 증가) → 패킷 손실 감지 시: ssthresh = cwnd / 2 cwnd = 1 (Slow Start로 복귀) — TCP Tahoe cwnd = cwnd / 2 (Fast Recovery) — TCP Reno/CUBIC
BBR (Bottleneck Bandwidth and RTT): Google이 개발한 혼잡제어 알고리즘. 패킷 손실이 아닌 대역폭과 RTT를 측정하여 최적 전송 속도를 결정. 패킷 손실이 잦은 환경(무선, 장거리)에서 CUBIC 대비 2~25배 처리량 향상. sysctl net.ipv4.tcp_congestion_control=bbr

실무 연결: Traefik/Nginx의 keepalive timeout 체인 설정에서 언급한 것처럼, Nginx upstream keepalive < LB idle timeout < Traefik idleTimeout 순서가 중요합니다. 이 체인이 어긋나면 한쪽에서 연결이 끊겨 TCP RST가 발생합니다.
● HTTP Versions

HTTP/1.1 vs 2 vs 3

keep-alive, 멀티플렉싱, QUIC — 프로토콜 진화와 K8s 환경에서의 영향.

HTTP/1.1, HTTP/2, HTTP/3의 핵심 차이를 설명해 주세요.
HTTP/1.1은 텍스트 기반, 연결당 하나의 요청-응답(HOL Blocking), keep-alive로 연결 재사용. HTTP/2는 바이너리 프레이밍, 단일 연결에서 멀티플렉싱(여러 요청 동시), HPACK 헤더 압축. 하지만 TCP HOL Blocking은 미해결. HTTP/3는 UDP 기반 QUIC으로 TCP를 대체하여 스트림별 독립 전송, 0-RTT 재연결, 연결 마이그레이션을 지원합니다.

멀티플렉싱❌ (6 병렬 연결)✅ (단일 연결)✅ + 스트림 독립
헤더 압축없음HPACKQPACK

K8s에서기본gRPC, Traefik H2CCDN에서 종단

gRPC와 HTTP/2 LB 문제 (실무 핵심):
gRPC는 HTTP/2 위에서 동작합니다. HTTP/2는 단일 TCP 연결에서 모든 요청을 멀티플렉싱하므로, L4 LB(K8s ClusterIP)는 연결 수준에서만 분산합니다. 한 클라이언트의 모든 gRPC 호출이 같은 서버로 가게 됩니다. 해결: Headless Service + 클라이언트 사이드 LB, 또는 L7 프록시(Traefik GRPCRoute), 또는 Istio/Linkerd가 L7 LB를 자동 처리.
● DNS & TLS

DNS 동작 & TLS Handshake

이름 해석과 암호화 — 모든 네트워크 통신의 시작점.

DNS 질의의 전체 과정을 설명해 주세요. Recursive와 Iterative의 차이는?
Recursive 질의는 클라이언트가 DNS 리졸버에 "최종 답을 달라"고 요청하는 것이고, Iterative 질의는 리졸버가 루트→TLD→권한 서버를 단계별로 직접 질의하며 최종 답을 찾는 것입니다. 일반적으로 클라이언트→로컬 리졸버는 Recursive, 리졸버→각 DNS 서버는 Iterative로 동작합니다.
브라우저가 api.example.com 접속: 1. 로컬 DNS 캐시 확인 (/etc/hosts, 브라우저 캐시) 2. 없으면 → 로컬 DNS 리졸버(ISP 또는 8.8.8.8)에 Recursive 질의 3. 리졸버 캐시 확인 4. 없으면 → Root DNS (.에 대해) Iterative 질의 "com.의 NS는 a.gtld-servers.net이다" 5. → TLD DNS (com.에 대해) Iterative 질의 "example.com.의 NS는 ns1.example.com이다" 6. → 권한 DNS (example.com.에 대해) Iterative 질의 "api.example.com = 1.2.3.4 (TTL 300)" 7. 리졸버가 결과를 캐시 (TTL 300초간) 8. 클라이언트에 응답
K8s DNS (CoreDNS):
Pod의 /etc/resolv.conf:
nameserver 10.96.0.10 (CoreDNS ClusterIP)
search prod.svc.cluster.local svc.cluster.local cluster.local
options ndots:5

ndots:5 문제: api.external.com(점 2개 < 5)이면 search 도메인을 먼저 시도 → 4번의 불필요한 DNS 쿼리 → 외부 도메인 해석 지연. 해결: FQDN 끝에 . 추가 또는 ndots: 2로 변경

DNS Cache Poisoning: 공격자가 가짜 DNS 응답을 리졸버 캐시에 주입. DNSSEC(서명 검증)으로 방어. 클라우드 환경에서는 Route 53/Cloud DNS가 기본적으로 방어.
TLS 1.3 Handshake 과정을 설명해 주세요. mTLS는 무엇인가요?
TLS 1.3은 1-RTT로 핸드셰이크를 완료합니다. ClientHello(지원 암호화 스위트 + 키 공유)→ServerHello(선택된 스위트 + 키 공유 + 인증서)→완료. mTLS(mutual TLS)는 서버뿐 아니라 클라이언트도 인증서를 제시하여 양방향 인증을 하는 것으로, Istio가 서비스 간 통신에 SPIFFE 기반 mTLS를 자동 적용합니다.
TLS 1.3 Handshake (1-RTT): Client → ServerHello: - 지원 암호화 스위트 목록 - Key Share (ECDHE 공개키) - SNI (Server Name Indication — 어떤 도메인에 접속하는지) Server → Client: - 선택된 암호화 스위트 - Key Share (ECDHE 공개키) - 인증서 + 인증서 체인 - CertificateVerify (서버 개인키로 서명) - Finished → 대칭키 유도 완료, 암호화 통신 시작 → TLS 1.2 대비 1 RTT 절감 (1.2는 2 RTT)
SNI (Server Name Indication): 하나의 IP에 여러 도메인을 호스팅할 때, 클라이언트가 "어떤 도메인에 접속하는지"를 알려줘야 서버가 올바른 인증서를 제시할 수 있음. SNI는 평문이므로 방화벽/프록시가 도메인을 볼 수 있음. ESNI/ECH가 이를 암호화하는 표준.

mTLS in K8s:
• Istio: istiod CA가 각 사이드카에 SPIFFE 인증서를 발급. PeerAuthentication STRICT 모드로 모든 서비스 간 통신을 mTLS 강제
• cert-manager + Traefik: Istio 없이도 cert-manager가 인증서를 발급하고 Traefik이 mTLS를 수행할 수 있지만, 서비스별 설정이 필요하여 운영 부담이 큼
• Cilium WireGuard: 커널 레벨에서 노드 간 암호화. mTLS보다 가볍지만 Pod 레벨이 아닌 노드 레벨 암호화
● Subnetting

서브넷 & CIDR 계산

CIDR 표기법, VPC 설계 시 서브넷 분할 — 면접에서 빠르게 계산하는 법.

CIDR 표기법을 설명하고, /16, /24, /28의 차이를 말해 주세요.
CIDR(Classless Inter-Domain Routing)은 IP 주소 뒤에 슬래시와 서브넷 마스크 비트 수를 표기하는 방법입니다. /16은 65,536개 IP(대규모 VPC), /24는 256개 IP(일반 서브넷), /28은 16개 IP(소규모 서브넷). 슬래시 뒤 숫자가 커질수록 네트워크 범위가 좁아지고, 32에서 빼면 호스트 비트 수가 됩니다.
CIDR서브넷 마스크IP 수사용 가능VPC 설계 용도
/16255.255.0.065,53665,531 (AWS 5개 예약)VPC 전체 (10.0.0.0/16)
/20255.255.240.04,0964,091큰 서브넷 (EKS 노드)
/24255.255.255.0256251일반 서브넷
/28255.255.255.2401611소규모 (NAT GW, LB)

빠른 계산법: IP 수 = 2^(32-CIDR)
/24 → 2^8 = 256개
/20 → 2^12 = 4,096개
/28 → 2^4 = 16개

VPC 설계 실전 (EKS + NHN NKS 멀티클라우드):
AWS VPC: 10.0.0.0/16 (65,536 IP) ├── Public Subnet: 10.0.0.0/24 × 3 AZ (LB, NAT GW) ├── Private Subnet: 10.0.10.0/20 × 3 AZ (EKS 노드) └── DB Subnet: 10.0.100.0/24 × 3 AZ (RDS) NHN Cloud: 172.16.0.0/16 ├── Subnet: 172.16.0.0/20 × 2 AZ (NKS) ⚠ VPN 연결 시 CIDR 겹침 방지! 10.0.0.0/16 ↔ 172.16.0.0/16 = 겹치지 않음 ✅
EKS Pod CIDR: VPC CNI는 VPC CIDR에서 Pod IP를 할당합니다. /20 서브넷(4,091 IP)이면 Pod 수 제한이 생길 수 있어 Prefix Delegation으로 확장하거나, 큰 서브넷(/16)을 사용합니다.
● NAT in K8s

NAT/SNAT/DNAT

K8s에서 kube-proxy가 하는 것, iptables 규칙, Proxy Protocol — 실무 경험 연결.

SNAT, DNAT의 차이와 K8s에서의 동작을 설명해 주세요.
SNAT(Source NAT)는 패킷의 소스 IP를 변경하는 것이고, DNAT(Destination NAT)는 목적지 IP를 변경하는 것입니다. K8s에서 kube-proxy는 Service ClusterIP→Pod IP로 DNAT를 수행하고, Pod→외부 통신 시 Node IP로 SNAT(MASQUERADE)를 수행합니다.
K8s Service DNAT: Client → Service ClusterIP:80 (10.96.1.100:80) → kube-proxy iptables: DNAT → Pod IP:8080 (10.244.1.5:8080) → conntrack에 매핑 기록 → 응답: Pod → Client (conntrack이 역변환) Pod→External SNAT: Pod (10.244.1.5) → google.com:443 → POSTROUTING: MASQUERADE → Node IP (192.168.1.10) → NAT Gateway → Internet → 응답: Node IP → conntrack 역변환 → Pod
NHN NKS CORS 403 해결 사례 (핵심 면접 소재):
문제: NHN NKS에서 kube-proxy SNAT가 LB의 소스 IP를 노드 IP로 변환 → Traefik이 Proxy Protocol로 받은 클라이언트 IP와 실제 소스 IP가 불일치 → Proxy Protocol trust 실패 → CORS 403

근본 원인: externalTrafficPolicy: Cluster(기본)일 때, 다른 노드의 Pod으로 트래픽이 라우팅되면서 추가 SNAT이 발생

해결: externalTrafficPolicy: Local 설정 → LB가 해당 노드의 Pod에만 직접 전달 → 추가 SNAT 없음 → 소스 IP 보존 → Proxy Protocol trust 성공

Proxy Protocol이란: LB가 클라이언트의 원본 IP를 보존하여 백엔드에 전달하는 프로토콜. TCP 연결 시작 시 PROXY TCP4 client_ip client_port server_ip server_port\r\n 헤더를 추가. Traefik에서 trustedIPs를 VPC CIDR로 설정하여 신뢰할 수 있는 소스에서만 Proxy Protocol을 수용합니다.
💡 면접에서 이 사례를 설명하는 법
"NHN Cloud NKS 환경에서 간헐적으로 CORS 403 에러가 발생하는 장애를 해결한 경험이 있습니다. 원인은 kube-proxy의 SNAT가 LB의 소스 IP를 변환하여 Traefik의 Proxy Protocol trust가 실패하는 것이었습니다. externalTrafficPolicy를 Local로 변경하여 SNAT를 방지하고 소스 IP를 보존하는 방식으로 해결했습니다."
→ L3/L4 수준의 이해 + 실전 디버깅 능력 + K8s 네트워킹 지식을 한 번에 보여줄 수 있는 강력한 사례
