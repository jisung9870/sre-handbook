● Static Application Security Testing

SAST / SCA — 코드와 의존성 정적 분석

코드 작성 단계에서 보안 결함을 찾는 도구. SAST(자체 코드), SCA(의존성), 그리고 false positive 노이즈를 다루는 운영 기법.

Principle

Shift-Left — 보안을 왼쪽으로 옮긴다

"prod 직전 보안 검토"의 한계 — 결함은 빨리 찾을수록 싸다
전통적 보안은 "개발 다 끝난 다음 보안팀이 검토"였다. 이 시점에서 발견된 결함은 (1) 수정 비용이 100배 증가, (2) 일정 압박으로 무시되거나 (3) 부분 패치로 끝난다. Shift-left는 같은 검사를 IDE/PR/CI 단계로 끌어와서 결함이 만들어지자마자 잡는다.
단계비용 (상대)도구 예시피드백 시간
IDE (개발 중)1xSemgrep CLI, Snyk IDE plugin실시간 (밀리초)
Pre-commit hook2xgitleaks, semgrep커밋 시 (초)
PR (CI)5xSemgrep, Snyk, Trivy5-10분
Merge / staging20xSAST + SCA + DAST30분-수시간
Production100xWAF, runtime monitoring실시간 ~ 일/주
Shift-left 5가지 통제점
1. SAST — 자체 코드의 패턴 기반 결함 (SQLi, XSS, hardcoded credential)
2. SCA — 의존성의 알려진 CVE
3. Secret scanning — 코드/커밋에 시크릿 평문
4. IaC scanning — Terraform/k8s 매니페스트의 잘못된 설정
5. Container scanning — 베이스 이미지의 OS-level CVE

각 통제점이 다른 결함을 잡음 — 모두 필요. 하나라도 빠지면 그 영역은 prod에서 발견됨
"Shift-left 절대주의"의 함정
모든 검사를 PR로 옮기면 PR 시간이 30분 → 90분이 됨. 개발자 생산성 직격.

균형:
· 빠른 검사 (lint, secret scan, basic SAST) — PR
· 중간 검사 (full SAST, SCA) — merge to main 트리거
· 느린 검사 (DAST, fuzzing, full container scan) — nightly

"shift-left"는 모든 걸 PR로 옮기는 게 아니라 가장 빠른 적합한 단계로 옮긴다는 뜻
SAST

SAST 도구 — 패턴으로 결함 잡기

코드를 실행하지 않고 결함 패턴을 찾는다 — 빠르지만 false positive 잦음
SAST는 코드의 AST(Abstract Syntax Tree)를 분석해 위험 패턴을 찾는다. 장점: 100% 코드 커버리지, 빠름. 단점: 데이터 흐름 추적 한계, false positive 30~50% — 좋은 룰셋과 triage가 핵심.
도구특징적합한 상황
SemgrepYAML 룰, 빠름, OSS커스텀 룰 작성 쉬움 — 조직별 패턴
SonarQube품질 + 보안 통합, 큰 룰셋코드 품질도 통합 관리
CodeQL (GitHub)강력한 데이터 흐름 분석GitHub 생태계, 복잡한 vuln
Snyk CodeML 기반 false positive 감소SaaS 통합 운영
Bandit (Python), gosec (Go), Brakeman (Rails)언어별 OSS단일 언어 프로젝트, 가벼움
Semgrep 커스텀 룰 — 조직별 패턴
# .semgrep/rules/dangerous-eval.yml rules: - id: dangerous-eval message: "eval()은 RCE 위험. JSON.parse나 명시적 파서 사용" severity: ERROR languages: [javascript, typescript] pattern-either: - pattern: eval(...) - pattern: new Function(...) paths: exclude: ['**/test/**'] # 테스트는 제외 - id: hardcoded-jwt-secret message: "JWT secret은 env/secret manager에서 읽어야 함" severity: ERROR pattern: jwt.sign($PAYLOAD, "...", ...) - id: deprecated-crypto message: "MD5/SHA1 암호화 용도 금지 — SHA-256 이상" severity: WARNING pattern-either: - pattern: createHash("md5") - pattern: createHash("sha1")
TIP Semgrep registry에 OWASP Top 10, CWE 매핑된 룰셋 무료 공개. semgrep --config=p/owasp-top-ten으로 즉시 시작 가능. 그 위에 조직 특화 룰 (위처럼) 추가해 나가는 게 실전 패턴.
SCA

SCA — 의존성의 CVE 추적

자체 코드보다 외부 라이브러리에서 결함이 더 많이 나온다 — Log4Shell, OpenSSL Heartbleed
현대 프로젝트의 코드 95%는 외부 의존성이다. SCA(Software Composition Analysis)는 의존성 트리를 분석해 알려진 CVE와 매칭한다. SCA가 SAST보다 ROI 높은 경우가 많다 — Log4Shell처럼 한 번에 산업 전체를 위협하는 결함은 거의 항상 의존성에서 나온다.
주요 도구
Trivy (OSS) — 컨테이너 + 파일시스템 + git repo
Snyk — SaaS, fix PR 자동 생성
Dependabot (GitHub) — GitHub 통합, 자동 업데이트 PR
Renovate — Dependabot 대체, 더 강력한 룰
OWASP Dependency-Check — Java 중심 OSS
Grype — Anchore의 CLI, SBOM 입력 지원
실전: PR에서 SCA 차단
# GitHub Action — Trivy로 의존성 스캔 - uses: aquasecurity/trivy-action@v0.20.0 with: scan-type: fs scan-ref: . severity: HIGH,CRITICAL ignore-unfixed: true # fix 없는 CVE는 스킵 (자체 노력 한계) exit-code: 1 # 발견 시 PR 차단 format: sarif output: trivy.sarif - uses: github/codeql-action/upload-sarif@v3 with: { sarif_file: trivy.sarif } # GitHub Security 탭에 표시
핵심 옵션: severity로 차단 임계, ignore-unfixed로 노이즈 감소
자동 업데이트 — Dependabot vs Renovate
Dependabot (GitHub 통합):
· 설정 파일 단순
· GitHub 네이티브
· 그룹/스케줄 제한적

Renovate (Mend):
· 강력한 그룹/스케줄 (예: "patch는 자동 머지, minor는 PR")
· dashboard으로 일괄 관리
· 모노레포 지원 우수
· 더 많은 매니저 지원 (Helm, Terraform 등)

실전 권장:
· 작은 프로젝트 → Dependabot
· 큰 모노레포 → Renovate
· 둘 다 자동 머지(patch만)와 매주 머지 데이를 통한 minor 처리 룰 정의
· "auto merge"는 CI 통과 + 변경 분량 임계 이하만
Noise

Triage — false positive와 노이즈 관리

SAST/SCA의 가장 큰 적은 false positive — 알람 피로로 진짜 결함도 무시당한다
10번 알람 중 8번이 잘못된 알람이면 사람들은 9번째도 무시한다. 이게 alert fatigue이고, 보안 도구 도입 후 자주 일어난다. 처음부터 "모든 결함 0개"를 목표로 하지 말고, 새로 발생한 결함을 막는 것부터 시작하라.
3단계 점진 도입
Phase 1: Baseline 측정 (1-2주)
현재 코드 스캔 → 결함 N개 발견. 각각의 우선순위/영향 평가. 이 N개는 "기존 부채"로 분류 (당장 안 막음)

Phase 2: New만 차단 (1-3개월)
PR이 baseline 대비 새 결함을 추가하면 차단. 기존 N개는 점진 정리. 개발자는 "내 PR이 새 문제 안 만들면 OK"

Phase 3: 전체 차단 (점진)
baseline N개를 분기마다 일정 수 줄임. 0이 되면 모든 결함 차단
Suppression — 의도된 예외
false positive나 비즈니스 결정으로 무시해야 할 결함:
# Semgrep // nosemgrep: dangerous-eval -- BLOG 작성자만 접근, sanitized const result = eval(post.expr); # Trivy trivyignores: - id: CVE-2023-XXXX expires: 2026-12-31 reason: | 이 CVE는 Linux kernel 5.x만 영향. 우리는 6.5 사용
원칙:
· suppression마다 이유와 만료일 명시
· 만료된 suppression은 자동 검토 PR
· suppression 추가는 보안팀 리뷰 필수 (CODEOWNERS)
SARIF — 통합된 결과 형식
SARIF는 SAST/SCA/DAST 결과의 표준 JSON 포맷.

장점:
· 도구 바뀌어도 같은 워크플로우
· GitHub Security tab, GitLab, Sonar 모두 SARIF 입력
· 결과 deduplication (다른 도구가 같은 결함 보고)

실전:
1. 모든 보안 도구가 SARIF 출력하도록 설정
2. CI에서 모은 SARIF를 GitHub upload-sarif로 업로드
3. 결과를 코드 위에서 inline annotation으로 검토
4. PR에 자동 코멘트 (github/codeql-action)

"결과를 어디서 보지?" 문제를 해결한 게 SARIF의 본질. 도구별 dashboard 흩어짐 ↓
● Secret Management

시크릿 관리 — 절대 코드에 박지 말 것

시크릿 스캐닝, Vault, External Secrets Operator, 회전 전략 — secret leak은 가장 흔하고 가장 큰 사고.

Detect

시크릿 스캐닝 — 사전 차단과 사후 탐지

"git push 했더니 토큰이 같이 나갔다" — 가장 흔한 보안 사고. 다층 방어 필수
시크릿이 git history에 한 번 들어가면 그 시크릿은 영원히 노출된 것으로 간주해야 한다. force-push로 제거해도 fork, 캐시, 봇이 이미 본 후일 수 있다. 따라서 (1) 들어가기 전에 막고, (2) 들어가도 빨리 발견하고, (3) 즉시 회전하는 다층 방어가 필요하다.
3단계 방어
1. Pre-commit hook — 로컬에서 차단:
# .pre-commit-config.yaml - repo: https://github.com/gitleaks/gitleaks rev: v8.22.0 hooks: - id: gitleaks
2. PR CI 검사 — 이중 안전망:
- uses: gitleaks/gitleaks-action@v2 env: { GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }} }
3. Push protection — 서버 측:
· GitHub의 push protection (Advanced Security)
· 알려진 패턴 매칭 시 push 자체 거부
도구
gitleaks — OSS, 빠름, 가장 많이 쓰임
truffleHog — entropy 기반, 진짜 secret 검증 (API 호출)
GitHub Secret Scanning — Advanced Security, partner와 자동 알림
detect-secrets (Yelp) — Python
Trivy — 컨테이너 이미지 안 secret도 스캔

truffleHog의 차별점: 패턴만 매칭하지 않고 실제 API에 인증 시도해서 "살아있는 secret"인지 검증 — false positive ↓
사고 발생 시 대응 — 회전 우선
"git history에서 지웠으니 OK"는 잘못된 가정. 노출된 것으로 간주하고:

1. 즉시 회전 (가장 중요)
· 노출된 secret 즉시 폐기 + 새 secret 발급
· git history 정리는 그 후에

2. 영향 분석
· 그 secret이 무엇에 접근했나? (CloudTrail, audit log)
· 비정상 접근 흔적이 있나?

3. git history 정리 (선택)
· git filter-repo로 history 재작성
· 하지만 이미 노출은 끝난 일 — clone, fork, 봇이 본 후

4. 사후 분석
· 어떻게 들어갔나? hook 빠진 사람? CI 검사 빠진 PR?
· 같은 패턴 재발 막을 통제 추가

회전이 어렵다는 게 secret leak 사고의 본질 — 회전 가능한 구조를 평소에 만들어 두는 게 진짜 방어
Vault

HashiCorp Vault — 중앙화된 secret store

정적 secret 저장 + 동적 secret 발급 + 회전 자동화의 표준 도구
Vault는 단순한 "secret 저장소"가 아니다. (1) 정적 secret KV 저장, (2) 동적 자격증명 발급(DB user, AWS STS), (3) 자동 회전, (4) audit log를 통합한 보안 인프라다. 잘 도입하면 "secret 회전"이 사람의 일에서 시스템의 일이 된다.
정적 vs 동적 secret
정적 secret (KV):
vault kv put secret/payment/stripe \ api_key=sk_live_xxx
API 키, OAuth client secret 등 외부에서 받은 값. 회전 어려움 — 외부 서비스 의존

동적 secret:
vault read database/creds/payment-app # username: v-payment-app-xK3... # password: A5p9... # lease_id: database/creds/payment-app/abc # lease_duration: 1h
요청 시 새 DB 사용자 생성 → 1시간 후 자동 폐기. 회전 자체가 무의미 — 매번 새로 받음
애플리케이션 통합
옵션 1: Vault Agent — sidecar로 secret 자동 주입
template { source = "/etc/vault/db.tpl" destination = "/secrets/db.env" command = "kill -HUP $(pidof app)" # 갱신 시 reload }
secret 갱신 시 파일이 자동 업데이트, 앱 reload

옵션 2: Direct API — 앱에서 SDK로 호출
· 주기적 lease 갱신 필요
· 앱 코드에 Vault 의존

옵션 3: External Secrets (k8s) — 다음 섹션 참고
k8s 인증 (Auth Method)
Pod가 Vault에 어떻게 인증할까? k8s service account를 활용:
# Vault에 k8s auth 활성화 vault auth enable kubernetes vault write auth/kubernetes/config \ kubernetes_host="https://kubernetes.default.svc" # Pod의 SA 토큰 → Vault Role 매핑 vault write auth/kubernetes/role/payment \ bound_service_account_names=payment \ bound_service_account_namespaces=prod \ policies=payment-read \ ttl=1h
Pod의 SA 토큰을 Vault에 제시 → Vault가 k8s API로 검증 → 정책 기반 secret 접근 허용. 정적 자격증명 zero
k8s

External Secrets Operator — k8s native secret 동기화

Vault/AWS Secrets Manager/GCP Secret Manager의 secret을 k8s Secret으로 자동 동기화
k8s Secret은 base64 인코딩일 뿐 — 평문과 다를 바 없다. 그렇다고 모든 앱이 Vault SDK를 쓰게 만들 수도 없다. External Secrets Operator (ESO)는 외부 secret store에서 값을 가져와 k8s Secret을 자동 생성·갱신해 준다. 앱은 일반 Secret을 envFrom으로 쓰면 됨.
ExternalSecret 정의
# 1. SecretStore — 백엔드 정의 (한 번만) apiVersion: external-secrets.io/v1beta1 kind: SecretStore metadata: { name: vault-backend, namespace: prod } spec: provider: vault: server: https://vault.internal:8200 path: secret version: v2 auth: kubernetes: mountPath: kubernetes role: payment serviceAccountRef: { name: payment } # 2. ExternalSecret — secret 매핑 apiVersion: external-secrets.io/v1beta1 kind: ExternalSecret metadata: { name: payment-secrets } spec: refreshInterval: 5m # 5분마다 동기화 secretStoreRef: name: vault-backend kind: SecretStore target: name: payment-secrets # 생성될 k8s Secret 이름 creationPolicy: Owner data: - secretKey: STRIPE_API_KEY remoteRef: { key: payment/stripe, property: api_key } - secretKey: DB_PASSWORD remoteRef: { key: payment/db, property: password }
지원 백엔드
· HashiCorp Vault
· AWS Secrets Manager
· AWS Parameter Store
· GCP Secret Manager
· Azure Key Vault
· 1Password Connect
· Kubernetes (다른 클러스터)
· ... 30+ providers

한 클러스터에서 여러 백엔드 동시 사용 가능. 점진적 마이그레이션에 유용
갱신 동작 — Pod restart 트리거
ExternalSecret이 갱신되면 k8s Secret이 변경. 그러나 Pod는 자동 재시작 안 함(Secret을 envFrom으로 쓰는 경우).

해법:
1. stakater/Reloader 설치 — Secret 변경 감지 시 Deployment annotation 트리거 → rollout
2. 또는 Secret을 volume mount → 파일 변경 시 앱이 reload
3. Vault Agent와 비슷하게 sidecar로 SIGHUP

"secret 회전이 자동 반영되는가?" 검증 필수 — 6개월에 한 번 staging에서 회전 시뮬레이션
Rotation

회전 전략 — "회전 가능한 구조"

90일 회전 정책이 있어도 회전 시 앱이 깨지면 정책은 죽은 글자
회전 사고의 90%는 "회전했더니 앱이 깨짐"이다. 원인은 (1) 새 secret이 즉시 반영 안 됨, (2) 한 secret이 N개 앱에서 쓰이는데 타이밍 어긋남, (3) 외부 third-party가 옛 secret 캐시. 회전 가능한 구조를 평소에 갖춰야 회전이 일상이 된다.
Dual-secret 패턴 (overlap)
한 시점에 두 secret이 동시에 유효하게:
T+0: secret_v1만 유효 T+0: secret_v2 발급, 둘 다 유효 T+0: 앱 설정 업데이트 — secret_v2 사용 T+1h: 모든 앱이 v2 사용 확인 T+2h: secret_v1 폐기
이 전환 윈도우 동안엔 어느 쪽 secret으로 와도 OK. 외부 API key, JWT signing key 같은 곳에 적용
Auto rotation 가능 vs 불가능
완전 자동 가능:
· DB password (Vault dynamic credential)
· AWS IAM access key
· TLS 인증서 (cert-manager + Let's Encrypt)
· k8s SA 토큰 (자동 만료)

반자동 (코드 변경 없이 회전 가능, 트리거는 수동):
· OAuth client secret
· webhook secret

수동만 가능 (외부 의존):
· 외부 SaaS API key (Stripe, SendGrid 등)
· Partner 통합 자격증명

"수동만" 카테고리도 dual-secret 지원하면 무중단 가능
회전 SLA & 검증
회전 주기 결정:
· DB credential: 1-24시간 (dynamic)
· 외부 API key: 90일
· TLS: 자동 (cert-manager)
· SSH 키: 사용자 1년
· root password: 사고 시에만 (그 외엔 사용 X)

"우리는 회전 정책 있다" ≠ "회전 가능". 검증:
1. 분기마다 staging에서 강제 회전 시뮬레이션
2. 모든 자동 회전 secret의 마지막 회전 시각 모니터링 — 정체 알람
3. 회전 실패 시 페이지
4. runbook: "이 secret 회전하려면 어떤 절차?"

회전 메커니즘 자체가 "비상시 처음 시도"가 되면 위험. 평시에 검증된 메커니즘만 사고 대응에 의존할 수 있다

OPA

OPA / Conftest — 조직 정책 코드화

조직만의 룰을 Rego로 작성 — "주석 없는 prod 리소스 금지", "특정 region만 허용" 등
Checkov 등의 표준 룰 외에, 조직 고유 룰이 있다 — "모든 prod 리소스에 cost_center 태그 필수", "DB는 ap-northeast-2만 허용". OPA(Open Policy Agent)의 Rego 언어로 이런 정책을 코드화하고, conftest로 IaC 파일에 적용한다.
정책 예시 — 필수 태그 검증
# policy/required_tags.rego package main required_tags := ["owner", "cost_center", "environment"] deny[msg] { resource := input.resource_changes[_] resource.type == "aws_s3_bucket" not resource.change.after.tags msg := sprintf("S3 bucket '%s'에 tags 없음", [resource.address]) } deny[msg] { resource := input.resource_changes[_] startswith(resource.type, "aws_") tag := required_tags[_] not resource.change.after.tags[tag] msg := sprintf( "%s에 필수 태그 '%s' 누락", [resource.address, tag] ) } deny[msg] { resource := input.resource_changes[_] resource.type == "aws_db_instance" resource.change.after.region != "ap-northeast-2" msg := "DB는 ap-northeast-2 region만 허용" }
CI 통합
# 1. Terraform plan을 JSON으로 terraform plan -out=tf.plan terraform show -json tf.plan > plan.json # 2. Conftest로 정책 검증 conftest test plan.json --policy ./policy/ # 결과: # FAIL - plan.json - main - DB는 ap-northeast-2 region만 허용 # 1 test, 0 passed, 1 failure
terraform apply 전에 실행 → 위반 시 차단. main branch protection에 status check로 강제
Test 가능한 정책
정책도 코드 — 테스트가 필요:
# policy/required_tags_test.rego package main test_missing_tag { deny[_] with input as { "resource_changes": [{ "address": "aws_s3_bucket.foo", "type": "aws_s3_bucket", "change": { "after": {} } }] } } # 실행: opa test policy/
정책 추가 시 테스트 필수. CI가 정책 자체의 회귀도 잡음

CI에서 차단을 못 했다면 클러스터에서 다시 차단 — 다층 방어

SBOM

SBOM — Software Bill of Materials

"이 이미지에 정확히 무엇이 들어있나?" — 새 CVE 발생 시 영향 범위 파악의 기초
Log4Shell이 터졌을 때 가장 먼저 한 질문은 "우리 이미지 중 어떤 게 log4j 쓰지?"다. 각 이미지를 다시 빌드해서 확인한다면 너무 느리다. 빌드 시점에 "이 이미지의 모든 구성 요소"를 SBOM 파일로 만들어 두면, CVE 발생 시 grep만 하면 된다.
SBOM 형식 — SPDX vs CycloneDX
SPDX (Linux Foundation): 라이선스 컴플라이언스 강점, 큰 생태계
CycloneDX (OWASP): 보안 강점, 더 풍부한 메타데이터

둘 다 JSON 직렬화 가능. 도구 대부분 양쪽 출력. 실전 권장: SPDX (생태계가 더 큼) + CycloneDX (보안 도구 통합)

한 이미지의 SBOM 예시:
{ "bomFormat": "CycloneDX", "components": [ { "type": "library", "name": "log4j-core", "version": "2.15.0", "purl": "pkg:maven/..." }, { "type": "operating-system", "name": "alpine", "version": "3.19" } ] }
생성 — Syft / docker buildx
Syft (Anchore):
syft ghcr.io/myorg/app:v1.2 \ -o spdx-json > sbom.spdx.json # 컨테이너 외에 git repo, dir도 가능 syft dir:./src
docker buildx — 빌드 시 자동 생성:
docker buildx build \ --sbom=true --provenance=true \ -t ghcr.io/myorg/app:v1.2 . # Pull 시 별도로: docker buildx imagetools inspect \ ghcr.io/myorg/app:v1.2 --format='{{json .SBOM}}'
SBOM이 이미지와 함께 registry에 attestation으로 저장됨
활용 — CVE 영향 분석
새 CVE 발생 시 모든 이미지의 SBOM을 한곳에 모아 query:
# 모든 prod 이미지의 SBOM을 BigQuery/ClickHouse에 적재 # 그 후 SQL 한 번으로 영향 분석: SELECT image, version FROM image_components WHERE component_name = 'log4j-core' AND component_version < '2.17.0'; # 결과: 영향받는 이미지 12개, 4개 서비스
도구:
· Grype — SBOM에 대해 CVE 매칭
· Dependency-Track — SBOM 중앙 저장 + 알림
· Trivy — SBOM 입력 지원 (trivy sbom)

"SBOM 만든다"가 끝이 아니라 "SBOM을 활용 가능한 곳에 모아둔다"가 핵심
Sign

이미지 서명 — Sigstore / Cosign

"이 이미지가 정말 우리가 빌드한 것인가?" — registry 침해, MITM 방지
전통적 GPG 서명은 키 관리 부담이 커서 거의 안 쓰였다. Sigstore (Cosign)는 OIDC 기반 keyless signing으로 키 관리 없이 서명 가능 — CI에서 OIDC 토큰으로 인증 → 단기 인증서로 서명 → Rekor (transparency log)에 기록. 키가 유출될 위험이 없다.
GitHub Actions에서 keyless signing
jobs: build: permissions: id-token: write # OIDC 토큰 (cosign이 사용) packages: write # registry push steps: - uses: docker/build-push-action@v5 id: build with: push: true tags: ghcr.io/myorg/app:${{ github.sha }} - uses: sigstore/cosign-installer@v3 - name: Sign image (keyless) env: { COSIGN_EXPERIMENTAL: "true" } run: | cosign sign --yes \ ghcr.io/myorg/app@${{ steps.build.outputs.digest }} - name: Attach SBOM as attestation run: | cosign attest --yes \ --predicate sbom.spdx.json \ --type spdxjson \ ghcr.io/myorg/app@${{ steps.build.outputs.digest }}
검증 — Pull 측
cosign verify \ --certificate-identity-regexp "^https://github.com/myorg/.*" \ --certificate-oidc-issuer "https://token.actions.githubusercontent.com" \ ghcr.io/myorg/app:v1.2
"이 이미지가 myorg의 GitHub Actions에서 서명됐는지" 검증. 서명자(certificate-identity)와 발급자(oidc-issuer)를 모두 매칭

Rekor (transparency log):
· 모든 서명이 공개 로그에 기록
· 키 유출돼도 추가 위조 즉시 감지
· rekor-cli search로 조회
왜 keyless가 더 안전한가
전통 키:
· 키를 어딘가 저장 (HSM, Vault)
· 회전 어려움
· 유출 시 모든 과거 서명 무효화 검토

Keyless (Cosign):
· 매 서명마다 단기 인증서 (5-10분)
· OIDC 인증 결과를 인증서에 embed
· 키 자체가 없음 → 유출할 게 없음
· Transparency log로 모든 서명 audit 가능

키 회전, HSM 운영, 서명 키 access 통제가 사라짐. OIDC 인증을 신뢰하는 모델
SLSA

SLSA Framework — 빌드 무결성 레벨

SLSA는 "공급망 보안 성숙도" 레벨 정의 — Build, Source, Dependencies 트랙
SLSA (Supply-chain Levels for Software Artifacts)는 Google이 제안한 공급망 보안 프레임워크다. 빌드 프로세스의 무결성을 0~3 (or L1~L4) 레벨로 정의. "우리는 SLSA L3 준수"는 "빌드 산출물이 어떤 코드/의존성에서 어떤 환경에서 만들어졌는지를 위변조 불가능하게 증명할 수 있다"는 뜻.
레벨요구사항달성 방법
SLSA L1provenance(빌드 메타데이터) 생성빌드 시 어떤 commit/branch/builder 기록
SLSA L2호스팅된 빌드 + 서명된 provenanceGitHub Actions 등 + cosign 서명
SLSA L3격리된 빌드 + 위변조 불가 provenanceSLSA generator + ephemeral runner
SLSA L4 (deprecated)2명 리뷰 + 재현 가능 빌드(현재 spec에선 L3가 최고)
SLSA L3 — slsa-github-generator
# Reusable workflow가 격리 환경에서 빌드 + provenance 생성 jobs: build: permissions: id-token: write contents: read actions: read uses: slsa-framework/slsa-github-generator/.github/workflows/builder_docker-based_slsa3.yml@v2.0.0 with: image: ghcr.io/myorg/app:${{ github.sha }} builder-image: docker.io/library/docker:25 config-path: .slsa/build.yml registry-username: ${{ github.actor }} secrets: registry-password: ${{ secrets.GITHUB_TOKEN }}
NOTE SLSA는 점진 도입이 자연스럽다. L1(provenance만 생성) → L2(서명 추가) → L3(격리 builder). 처음부터 L3 시도하면 큰 변경. 일단 모든 빌드에 provenance 생성하는 것만으로도 큰 가치.
Verify

Admission에서 서명/SBOM 검증

"서명되지 않은 이미지 거부", "특정 builder가 만든 이미지만 허용"
서명을 했는데 검증 안 하면 의미 없다. k8s admission controller에서 모든 이미지를 검증해 서명 안 된 이미지 자체를 거부한다.
Kyverno — Cosign 서명 검증
apiVersion: kyverno.io/v1 kind: ClusterPolicy metadata: { name: verify-images } spec: validationFailureAction: Enforce webhookTimeoutSeconds: 30 rules: - name: check-image-signature match: any: - resources: { kinds: [Pod] } verifyImages: - imageReferences: ["ghcr.io/myorg/*"] attestors: - entries: - keyless: subject: "https://github.com/myorg/*" issuer: "https://token.actions.githubusercontent.com" rekor: url: https://rekor.sigstore.dev
WARN 처음 도입 시 Audit로 시작. 외부 이미지(nginx, busybox)를 그대로 쓰는 곳이 있으면 차단. 외부 이미지는 별도 화이트리스트 또는 자체 registry mirror로.
● DAST & Runtime

DAST 및 런타임 보안

SAST가 못 잡는 결함을 잡는 동적 분석, runtime 위협 탐지(Falco), NetworkPolicy로 lateral movement 차단.

DAST

DAST — 실행 중인 앱을 공격

SAST는 코드 패턴, DAST는 실제 동작. 둘은 보완 — 한쪽만으로는 부족하다
DAST(Dynamic Application Security Testing)는 staging 환경에 배포된 앱에 실제 공격 시도를 보내 결함을 찾는다. SAST가 못 잡는 결함: 인증 우회, 비즈니스 로직, 환경 설정 결함, 외부 의존성과의 interaction.
주요 도구
OWASP ZAP — 무료, GUI + CLI + API. baseline scan을 CI에 통합 가능
Burp Suite — 상용, 펜테스트 표준
Nuclei — YAML 템플릿 기반, 빠른 CVE 스캔
StackHawk / Detectify / Acunetix — SaaS

실전 추천:
· OWASP ZAP baseline → CI (5분, low noise)
· OWASP ZAP full → nightly
· Nuclei → CI에서 CVE 빠른 매칭
CI 통합 — ZAP baseline
# GitHub Actions - name: Deploy to staging run: ./deploy.sh staging - name: ZAP Baseline Scan uses: zaproxy/action-baseline@v0.13.0 with: target: 'https://staging.myapp.io' cmd_options: '-a -j -m 5' # 인증 정보, JSON, 5분 limit rules_file_name: '.zap/rules.tsv' fail_action: true
baseline scan: passive — 트래픽 보고 분석. 실제 공격 X. 5-10분 소요. 노이즈 적어 CI 적합
full scan: active — 실제 공격 시도. 30분-수시간. nightly에만
DAST의 한계와 보완
한계:
· 인증된 영역 스캔 어려움 (login flow 자동화 필요)
· 비즈니스 로직 결함 못 잡음
· 큰 앱은 시간 ↑
· false positive 적지만 결함 자체 적음

보완:
· API contract 기반 fuzzing — OpenAPI spec → schemathesis로 자동 fuzz
· 인증 자동화 — ZAP authentication script로 로그인 후 스캔
· 주기적 펜테스트 — 자동 도구가 못 잡는 비즈니스 결함은 사람만 가능. 분기마다 외부 펜테스트

"DAST가 통과 = 안전"이 아니다. SAST + SCA + DAST + 펜테스트 4종 세트가 보완 관계
Runtime

런타임 위협 탐지 — Falco

CI 모든 게이트를 통과해도 prod에서 이상 행동은 발생할 수 있다 — eBPF로 syscall 감시
Falco는 런타임 syscall을 감시해 의심 행동을 탐지한다. eBPF 기반이라 성능 오버헤드 작고 컨테이너 내부까지 가시화. 룰셋이 풍부 — "컨테이너에서 셸 실행", "passwd 파일 read", "외부 IP로 curl" 같은 패턴 즉시 탐지.
기본 룰 예시
- rule: Terminal shell in container desc: A shell was used as the entrypoint condition: > spawned_process and container and shell_procs and proc.tty != 0 and container_entrypoint output: > Shell spawned in container (user=%user.name container=%container.name shell=%proc.name parent=%proc.pname) priority: NOTICE tags: [container, shell, mitre_execution]
이런 룰이 수백 개 기본 제공. CIS, MITRE ATT&CK 매핑
알람 → 대응 자동화
1. Falcosidekick — Falco 알람을 다양한 sink로
· Slack, PagerDuty
· OpenTelemetry
· webhook → Lambda

2. 자동 격리 (Falco Talon):
rule: container-shell-spawn action: kubernetes.kill-pod condition: priority >= NOTICE
의심 행동 감지 시 pod 자동 종료 — "containment" 자동화. 주의: false positive로 정상 pod 죽이면 더 큰 사고. audit 모드 충분히 검증 후 enforce
대안과 보완 — 다른 도구들

Network

NetworkPolicy — Lateral Movement 차단

한 pod이 침해되어도 다른 pod에 갈 수 없게 — zero trust networking
k8s 기본은 "모든 pod이 모든 pod과 통신 가능"이다. 즉 한 컨테이너 침해 = 클러스터 내 모든 곳 접근 가능. NetworkPolicy로 명시적 허용된 경로만 열어야 lateral movement(공격자 측면 이동)가 막힌다.
기본 deny + 명시적 allow 패턴
# 1) 모든 ingress/egress 기본 deny (namespace 단위) apiVersion: networking.k8s.io/v1 kind: NetworkPolicy metadata: { name: default-deny, namespace: prod } spec: podSelector: {} policyTypes: [Ingress, Egress] # 2) payment app은 정해진 곳만 허용 apiVersion: networking.k8s.io/v1 kind: NetworkPolicy metadata: { name: payment-policy, namespace: prod } spec: podSelector: { matchLabels: { app: payment } } ingress: - from: # API gateway에서만 - podSelector: { matchLabels: { app: gateway } } ports: [{ protocol: TCP, port: 8080 }] egress: - to: # DB만 접근 - podSelector: { matchLabels: { app: postgres } } ports: [{ protocol: TCP, port: 5432 }] - to: # DNS - namespaceSelector: { matchLabels: { name: kube-system } } podSelector: { matchLabels: { k8s-app: kube-dns } } ports: [{ protocol: UDP, port: 53 }] - to: # 외부 Stripe API (CIDR allowlist) - ipBlock: cidr: "54.187.0.0/16" ports: [{ protocol: TCP, port: 443 }]
CNI 지원 — 일부 기능은 CNI마다 다름
k8s NetworkPolicy 표준은 L3/L4까지만. L7 (HTTP path, gRPC method)은 CNI 확장:

· Calico: GlobalNetworkPolicy, L7 (Calico Cloud)
· Cilium: CiliumNetworkPolicy — L7 강력 (HTTP, gRPC, DNS, Kafka)
· Antrea: ClusterNetworkPolicy

표준 NetworkPolicy로 시작 → 필요 시 CNI 확장 사용. 주의: CNI 종속 코드는 마이그레이션 비용
실전 도입 단계
Phase 1: 가시화 (1-2주)
· cilium hubble 또는 service mesh로 현재 트래픽 흐름 시각화
· "어떤 pod이 어떤 pod과 통신하는가" 확보

Phase 2: namespace 격리 (2-4주)
· namespace 간 통신 제한 (prod ↔ staging 막음)
· 같은 namespace 내는 일단 허용

Phase 3: pod 단위 정책 (점진)
· 핵심 서비스부터 (payment, auth)
· 차차 모든 서비스로 확대

주의: 처음부터 deny-all 하면 클러스터 정지. 점진적 + audit 단계 필수
