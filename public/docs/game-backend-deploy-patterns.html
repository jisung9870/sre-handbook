● Overview

게임 백엔드 배포 — 일반 HTTP 서비스와 다른 세계

게임 서버는 긴 세션, stateful, 실시간 통신이 본질. 일반 HTTP 백엔드의 rolling/blue-green이 그대로 안 통하고, 매치 한복판에서 사용자를 끊지 않으려면 세션 인지 배포가 필수. 2026-05 기준

왜 일반 백엔드 배포 패턴이 부족한가

① 긴 세션

HTTP 요청: 100ms~수 초. 게임 매치: 10분~수 시간. 매치 한복판에 pod 종료 = 진행 중인 모든 사용자 끊김. 일반 rolling update의 maxUnavailable 모델로 안 됨.

② Stateful 인스턴스

일반 백엔드: 임의의 인스턴스가 임의의 요청 처리. 게임: 특정 인스턴스가 특정 매치 소유 — 다른 인스턴스로 못 옮김 (메모리 안의 게임 상태). LoadBalancer round-robin 안 통함.

③ 실시간 통신

UDP/TCP 직접 연결, 30Hz tick. readiness probe로 트래픽 차단해도 이미 연결된 사용자엔 무영향. 일반 LB drain 패턴 부적합.

④ 클라/서 강결합

HTTP API는 보통 클라가 새로고침 가능. 게임은 클라 빌드와 서버 빌드가 프로토콜 호환이어야. 클라 빌드 cadence가 서버보다 느려 — 서버는 구 프로토콜과 호환 의무.

★ 핵심 원칙
"세션이 끝날 때까지 인스턴스 살려둔다" — drain pattern. 진행 중인 매치는 끝까지, 새 매치는 신버전으로. 인스턴스 종료는 활성 세션 0이 된 후. 이 한 가지만 지켜도 게임 서버 배포의 90% 사고 방지.

게임 서버 유형 — 각자 다른 배포 전략

Dedicated Server (DS)

매치 한 개에 인스턴스 한 개. FPS / MOBA / 배틀 로얄. 매치 종료 → 인스턴스 종료 또는 재사용. 가장 stateful, drain pattern 핵심.

Persistent World

MMORPG / Survival. 한 인스턴스에 수십~수백 명, 24/7 운영. drain 시간 너무 길어 (사용자가 자연 떠날 때까지) — 점검 시간 활용 + 인스턴스별 마이그레이션.

Lobby / Matchmaker

매치 시작 전 사용자가 머무는 곳. 짧은 stateful. lobby drain은 비교적 쉬움 (사용자가 매치 시작하면 lobby 떠남).

일반 백엔드 (HTTP API)

인증, 결제, 친구 목록, 인벤토리, 리더보드. 전형적 stateless HTTP — 일반 rolling update 그대로 적용. Argo Rollouts, 그냥 k8s Deployment OK.

♥ 한 게임에 4가지 모두 존재
현대 게임은 보통 4가지 백엔드 모두 운영. 각자 다른 배포 전략이 필요. "게임 서버 배포 전략"이라고 하나로 묶으면 안 됨 — DS는 drain pattern, persistent world는 점검 마이그레이션, lobby는 짧은 drain, HTTP API는 일반 rolling.

● Agones

Agones — k8s 위의 게임 서버 운영 표준

Google + Ubisoft가 시작한 OSS 프로젝트. k8s를 게임 서버 호스팅 플랫폼으로 변환. GameServer CRD, Fleet, Allocation으로 게임 특화 워크로드 관리.

GameServer 상태 머신

Agones GameServer는 5가지 주요 상태로 lifecycle 관리. 일반 Pod의 Pending/Running/Terminating과 다른 game-specific state.

Scheduled

Pod 생성 중

→

Ready

매치 받을 준비

→

Allocated

매치 진행 중

→

Shutdown

매치 종료, Pod 삭제

Ready

매치 할당 가능 풀(pool)에 있음. SDK.Ready() 호출로 진입. 헬스체크 통과 + 게임 로딩 완료 후. matchmaker가 이 상태의 GameServer를 선택해 사용자 매칭.

Allocated

매치 진행 중. matchmaker의 Allocation 요청으로 Ready → Allocated 전환. 핵심: Allocated GameServer는 Fleet update로 자동 삭제 안 됨 — drain pattern의 본질.

Reserved

일정 시간 동안 삭제 보호. matchmaker가 Allocate하기 전에 잠시 holding하는 용도. 기간 만료 시 Ready로 자동 복귀.

Shutdown

매치 종료. SDK.Shutdown() 호출로 Agones에 알림 → Agones가 Pod 삭제. 이 상태가 되어야 Fleet의 진짜 종료.

Fleet & Allocation — 풀 운영

Fleet은 GameServer 풀의 운영 단위 — Deployment에 해당. replicas로 풀 크기 관리, GameServerAllocation으로 매치메이커가 GameServer 할당.

Fleet 정의
apiVersion: agones.dev/v1
kind: Fleet
metadata: { name: shooter-game }
spec:
  replicas: 100                           # Ready 풀 크기
  strategy:
    type: RollingUpdate                  # Allocated는 보호됨
    rollingUpdate:
      maxSurge: "25%"
      maxUnavailable: "25%"             # Ready 상태에만 적용
  template:
    spec:
      ports:
      - name: default
        portPolicy: Dynamic              # Agones가 포트 자동 할당
        containerPort: 7654
      template:
        spec:
          containers:
          - name: shooter
            image: ghcr.io/myorg/shooter:v2.3.1
            resources:
              requests: { cpu: "500m", memory: "512Mi" }
              limits: { cpu: "1000m", memory: "1Gi" }

Allocation 흐름

(1) matchmaker가 GameServerAllocation CRD 생성. (2) Agones가 Ready 상태 GameServer 1개 선택 → Allocated로 전환. (3) GameServer의 IP/port 반환. (4) 클라들이 그 주소로 직접 연결.

Buffer 정책

BufferPolicy로 항상 N개의 Ready 유지 — 매치 시작 지연 방지. 사용자가 매칭 누른 시점에 Ready GameServer가 부족하면 새로 띄울 시간 (~수 초~분) 동안 대기. Buffer 충분히 크게.

세션 인지 롤링 업데이트

Agones의 핵심 기능. Fleet update 시 Allocated GameServer는 그대로 유지, Ready GameServer만 신버전으로 교체. 진행 중인 매치 영향 zero.

업데이트 흐름

(1) Fleet의 spec.template 변경 (새 image 태그). (2) Agones가 Ready 상태 GameServer만 점진적으로 신버전으로 교체. (3) Allocated GameServer는 매치 종료 후 SDK.Shutdown() 호출 → 자연 삭제. (4) 모든 Ready가 신버전이 되고 모든 구버전 Allocated가 종료될 때 업데이트 완료.

긴 매치의 함정

구버전 Allocated가 매치 1시간 이상 지속하면 업데이트 wall clock도 1시간 이상. 두 버전 동시 운영이 매치 시간 동안 지속. 클라/서 호환성 + 메트릭 분리 필수.

Lobby 같은 긴 운영 — 라벨 트리거 패턴

Lobby GameServer 같이 매치 종료 trigger가 자연적으로 안 오는 경우. Fleet update 시 Allocated GameServer에 라벨 추가(Agones 1.27+) → 게임 코드가 그 라벨 감지 → graceful shutdown 시작 (신규 사용자 차단, 기존은 자연 떠남, 마지막에 SDK.Shutdown).

★ Argo Rollouts와의 차이
일반 Argo Rollouts는 readiness/liveness 기반. Agones는 game-specific Allocated 상태를 알기 때문에 매치를 보호. Argo Rollouts를 게임 서버에 그냥 쓰면 매치 진행 중 pod이 종료되어 사용자 끊김 — 이게 game-specific 도구가 필요한 이유.

SDK 통합 — 게임 코드에서 호출

Agones SDK는 게임 코드에서 직접 호출. SDK.Ready(), SDK.Allocate(), SDK.Shutdown(), SDK.Health() 4가지가 핵심. 다양한 언어 client (C++, C#, Go, Rust, Unity, Unreal).

표준 게임 서버 lifecycle (의사 코드)
async function main() {
  const sdk = new AgonesSDK();
  await sdk.connect();

  // 1. Health ping을 5초마다 (필수 — Agones가 죽었다고 판정 막기)
  setInterval(() => sdk.health(), 5000);

  // 2. 게임 로딩, 맵 로드 등 준비 작업
  await loadGameContent();
  await setupNetworking();

  // 3. Ready — matchmaker가 할당 가능
  await sdk.ready();

  // 4. matchmaker로부터 Allocation 받음 (외부 트리거)
  await waitForAllocation(sdk);

  // 5. 매치 진행
  await runMatch();

  // 6. Shutdown — Agones에 매치 종료 알림
  await sdk.shutdown();
  process.exit(0);
}

Health ping 실패

Agones는 일정 시간 health ping 안 오면 GameServer를 Unhealthy로 마크 → Pod 삭제. 5초 간격 + grace period 30초 권장. 게임 로직이 한 프레임 무거워도 ping 별도 thread로 보장.

Reserve 패턴

matchmaker가 매치 만들기 위해 N개 player 모으는 동안, GameServer를 미리 holding. SDK.Reserve(60s) → 60초간 다른 매치에 할당 안 됨. 매치 만들기 실패하면 자동으로 Ready 복귀.

● AWS GameLift

AWS GameLift — 매니지드 게임 서버

AWS의 매니지드 게임 서버 호스팅. 3가지 모델(Servers, FleetIQ, Anywhere) + FlexMatch 매치메이킹. 23 region + 10 Local Zone(33 location)으로 글로벌 커버리지가 강점.

GameLift 3가지 모델

GameLift는 단일 제품이 아니라 3가지 다른 운영 모델의 묶음. 게임 규모와 인프라 통제 욕구에 따라 선택.

모델인프라가격 모델적합 상황
GameLift ServersAWS 매니지드 EC2EC2 instance hour + GameLift 마진완전 매니지드 원할 때
GameLift FleetIQ본인 EC2 + GameLift가 관리EC2 instance hour만 (GameLift 무료)비용 최적화 + AWS 운영
GameLift Anywhere (2023+)자체 인프라 (온프레/타 클라우드)game session count + connection minutesBYO 인프라 + AWS 매치메이킹

FlexMatch — 공통 매치메이커

3 모델 모두 FlexMatch 통합. JSON으로 매칭 룰 정의, latency-based + skill-based 매칭. 큰 강점이지만 AWS 종속.

글로벌 region

23 region + 10 Local Zone (33 location, 2026 Q1 기준). 아시아 (서울, 도쿄, 싱가포르, 인도, 홍콩 등) 충분 한국 게임의 KR/JP/SEA 운영에 적합.

FleetIQ + Spot — 비용 최적화

FleetIQ의 가장 큰 강점. Spot instance + 자동 회복으로 EC2 비용 90% 절감 가능. 게임 서버는 매치당 인스턴스 분리되어 spot interruption tolerance 높음.

Spot Interruption 회복

FleetIQ는 spot 중단율 모니터링 → 중단 빈도 낮은 instance type 자동 선택. 중단 받아도 GameLift가 새 인스턴스에 자동 reschedule.

매치 영향 최소화

Spot 중단 알림 받으면 (보통 2분 전) → 새 매치 받지 않기 + 진행 중 매치는 강제 종료. 사용자 경험: 갑자기 매치 종료 — 하지만 빈도 낮으면 (월 0.5% 이하) 허용 가능.

가격 절감 시나리오

예: c5.xlarge On-Demand $0.17/h × 100 instances × 720h = $12,240/월. Spot 평균 $0.05/h = $3,600/월. 70% 절감. 단, AAA 인기 게임은 spot price 변동성 + 가용성 위험 — fallback to on-demand 정책.

GameLift Anywhere — BYO 인프라

2023년 출시. "AWS 매니지드 매치메이킹 + 본인 인프라" 모델. 온프레미스, 타 클라우드, 자체 데이터센터에서 게임 서버 운영하면서 GameLift의 매칭/오케스트레이션은 활용.

적합 상황

(1) 이미 자체 인프라 있음 (한국 IDC, 자체 베어메탈), (2) 멀티 클라우드 정책, (3) AWS region에 없는 지역 (한국 IDC 직접), (4) 매치메이킹 자체 만들기 부담스러움.

가격 모델

EC2 비용 X. 매치 세션 수 + connection minute로 과금. 자체 인프라 비용 + GameLift 매니지먼트 비용 별도.

BYO + AWS 매치메이커 패턴

한국 시장 게임에서 흔한 시나리오: KR 사용자는 한국 IDC 게임 서버, JP/SEA는 AWS region. FlexMatch는 글로벌 통합, 인스턴스 위치만 본인 + AWS 혼합. region별 latency-based 라우팅 자동.

Region 점진 롤아웃

글로벌 게임의 배포는 region별로 단계적. 한 region에서 검증 후 다음 region. 사고 발생 시 그 region만 영향, 다른 region 사용자 보호.

Region 단계 결정

(1) 운영팀 timezone에 가까운 region 먼저 — 사고 시 즉시 대응. (2) 사용자 규모 작은 region 먼저 — 영향 최소. (3) 큰 region(NA, EU)는 마지막. 한국 게임 + 한국 운영팀이면 KR 먼저 자연스러움.

Region 메트릭 분리

대시보드는 region별 분리 필수. "글로벌 평균 crash rate"는 의미 없음 — KR만 5% 폭증해도 글로벌 평균은 정상. region × game version 매트릭스로 각 셀의 메트릭 모니터.

● Matchmaking

매치메이킹 시스템 배포

매치메이커는 게임 서버와 별도의 백엔드 컴포넌트. Open Match (OSS, k8s 위) 또는 FlexMatch (AWS)가 대표 선택. 자체 매치메이커 구축도 흔하나 운영 부담 큼.

Open Match — Google + Riot 시작 OSS

Kubernetes 위에서 동작하는 OSS 매치메이커. 2019 Google Cloud + Riot Games + Unity 협업으로 시작. 매치 로직 자체는 게임팀이 작성, Open Match는 인프라.

Assignment 반환

3가지 컴포넌트

FrontendService: 클라이언트가 매치 ticket 만들고 상태 조회. BackendService: Director가 매치 결과 받음. Director: 게임팀이 작성하는 자체 코드 — 매치 알고리즘 + GameServer 할당 로직.

인프라 — Redis + gRPC

State storage는 Redis. 컴포넌트 간 통신은 gRPC. k8s 위에 설치, autoscaling 표준. Helm chart 제공 — k8s 운영 역량 있으면 시작 쉬움.

커스텀 매치 함수

"latency 50ms 이내 + skill 차이 100 이내 + 5명 모임"같은 룰을 게임팀이 Go/Python/등으로 작성. Open Match는 그 함수에 후보 ticket을 전달, 매치 결과 받음. 룰 변경 시 함수만 재배포.

FlexMatch — AWS GameLift 통합

AWS의 매치메이커. JSON으로 룰 정의, latency-based + skill-based + custom attribute. GameLift Servers/FleetIQ/Anywhere 모두 통합.

FlexMatch 매치메이킹 룰 (JSON)
{
  "name": "BR_5v5",
  "ruleLanguageVersion": "1.0",
  "playerAttributes": [
    { "name": "skill", "type": "number", "default": 100 }
  ],
  "teams": [
    { "name": "red", "minPlayers": 5, "maxPlayers": 5 },
    { "name": "blue", "minPlayers": 5, "maxPlayers": 5 }
  ],
  "rules": [
    {
      "name": "FairTeamSkill",
      "description": "평균 skill 차이 100 이내",
      "type": "distance",
      "measurements": ["avg(teams[red].players.attributes[skill])"],
      "referenceValue": "avg(teams[blue].players.attributes[skill])",
      "maxDistance": 100
    }
  ],
  "expansions": [
    {
      "target": "rules[FairTeamSkill].maxDistance",
      "steps": [
        { "waitTimeSeconds": 15, "value": 200 },
        { "waitTimeSeconds": 30, "value": 500 }
      ]
    }
  ]
}

Expansion — 시간에 따라 완화

매칭 대기 시간 길어지면 룰 자동 완화. 15초 후 skill 차이 200 허용, 30초 후 500. 사용자가 무한정 기다리지 않게.

장점

JSON으로 정의 → 코드 변경 없이 룰 갱신 가능. AWS 매니지드 — 인프라 운영 부담 X. GameLift와 통합 — Allocation 자동.

매치메이커 배포 — 무중단의 본질

매치메이커는 게임 서버와 다른 배포 패턴. 일반 stateless HTTP에 가까움 — 매치 만드는 동안 일시 중단되어도 사용자는 다시 시도하면 OK. 단, 새 매치 만들기 자체가 멈추면 게임 서버는 텅 비고 사용자는 매칭 실패.

매치메이커 배포 = HTTP API 배포

일반 rolling update 적용 가능. readinessProbe + maxUnavailable: 0으로 가용성 유지. 매치 ticket은 short-lived (수 초~분) — drain 시간 짧음.

룰 변경 — 컨피그 푸시

매치 룰은 코드가 아닌 컨피그로 관리. JSON/YAML이 Remote Config 또는 ConfigMap에 저장. 룰 변경은 컨피그 푸시 — 매치메이커 재배포 불필요.

스파이크 처리

이벤트 시작·점검 종료·신작 런칭 시 매치메이커 호출 spike. HPA + 사전 warmup으로 대응. Pod 시작 시간이 길면(JVM 등) 미리 capacity 키워두기 — 사후 대응으로는 늦음.

호환성 매트릭스 & 운영

클라/서버 버전이 항상 동기화되지 않는 게 현실. 모바일은 스토어 심사로 며칠 지연, Steam은 즉시 — 같은 시점에 클라 N개 / 서버 1개의 호환을 운영해야 한다.

클/서 호환성 매트릭스

"클 v1 + 서 v1, 클 v2 + 서 v2"라는 단순 매칭은 비현실적. 구버전 클라이언트도 신버전 서버에 접속 가능해야 사용자 영향 최소.

클라 버전 → 서버 버전v1.0 (구)v1.1 (현재)v1.2 (신)
v1.0 (구) 클라OKOK제한 OK
v1.1 (현재) 클라제한OKOK
v1.2 (신) 클라제한제한 OKOK

구 클라 + 신 서버 → OK

가장 흔한 상황 (모바일 스토어 심사 지연 등). 서버는 구 프로토콜과 호환 의무. 신 기능은 클라가 모르므로 서버에서 자동 비활성화 (graceful degradation).

신 클라 + 구 서버 → 제한 OK

Steam이 빨리 신 클라 받았는데 서버는 region별 점진 배포로 일부 region 구 서버. 클라가 서버 버전 감지 후 신 기능 자체 비활성화. UI에서 "이 region은 점검 중" 안내.

호환성 윈도우

모든 버전을 영원히 지원할 순 없다. 운영 정책: 최근 3개 클라 버전 + 최근 1개 서버 버전. 그 이전 클라는 강제 업데이트 트리거. 호환성 매트릭스가 너무 커지면 코드 복잡도 ↑ — 관리 가능 범위 명시.

프로토콜 버저닝 — 깨지는 변경 vs 호환

클라/서 통신 프로토콜의 변경 패턴. Backward-compatible 변경(새 필드 추가 등)과 breaking 변경(필드 제거·타입 변경)을 구분해 관리.

Backward-compatible 변경 (대부분)

· 새 RPC 메서드 추가
· 메시지에 새 optional 필드 추가
· 새 enum 값 추가 (기본 케이스 fallback)
· 프로토콜 옵션 변경
구 클라가 신 서버에 정상 통신 가능. protobuf의 기본 설계 원칙.

Breaking 변경 (드물게)

· 필드 타입 변경 (int32 → int64)
· 필드 제거 후 ID 재사용
· RPC signature 변경
· 의미 변경 (같은 필드 다른 의미)
서버 버전을 새 메이저로 + 호환 윈도우 동안 두 버전 동시 운영.

protobuf의 함정 — required vs optional

proto3는 모든 필드 optional. 빠진 필드는 기본값. 매우 backward-compatible 친화. proto2의 required는 절대 사용 X — 한 번 박으면 영원히 호환성 파괴 위험. 모든 신규 프로젝트는 proto3.

Capacity-aware 배포

게임 서버 fleet의 크기는 시간대별로 크게 다름. 배포 시점이 peak time이면 capacity 부족. 일반 백엔드보다 더 강한 시간대 의존성.

피크 시간 회피

한국 게임의 평일 피크: 저녁 8~11시. 이 시간에 배포 금지 정책. 새벽 4~6시에 배포 + 점검 가능. 핫픽스만 피크 중 가능.

배포 자체의 capacity 영향

Rolling 시 maxSurge로 +25% capacity, maxUnavailable -25%. 피크 시간에 -25%면 매치메이킹 대기열 폭증. 사전에 capacity 1.5x로 키워놓고 배포 (HPA min replicas 일시 상향).

게임 서버 빌드 시간

Pod 시작 시간이 길면(컨테이너 이미지 큰 경우, GB 단위) capacity 회복도 느림. 이미지 슬림화 + multi-stage 빌드 + 이미지 prepull(노드에 미리 받아두기). 배포 시 신 이미지 prepull로 시작 시간 ↓.

게임 서버 Canary — 일부 매치만 신버전

일반 HTTP의 traffic-weighted canary는 게임 서버에 안 통함. 매치 단위로 분기해야. "이 매치는 v2 서버로, 저 매치는 v1로".

매치 단위 라우팅

matchmaker가 GameServer 선택 시 label 기반 분기. Agones GameServerAllocation의 selector로 version=v2 5%, version=v1 95%. 사용자가 어느 버전에 가는지 매칭 시점 결정.

코호트 sticky

같은 사용자가 여러 매치 사이에서 다른 버전 받으면 혼란. user ID hash로 결정적 라우팅 — 한 번 v2 받은 사용자는 계속 v2. 사용자 단위 코호트 분리.

메트릭 격리

v1 / v2 메트릭 분리 — crash rate, match completion rate, latency. 버전 라벨로 Prometheus 시계열 분리. baseline 대비 회귀 자동 감지 + 자동 abort. Argo Rollouts의 AnalysisTemplate 패턴 차용 가능.
