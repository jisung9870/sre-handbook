● RBAC Design

RBAC 설계와 운영

Role, ClusterRole, Binding, ServiceAccount — K8s 접근 제어의 핵심 체계.

Model

RBAC 동작 모델

누가(Subject) → 무엇을(Resource) → 어떻게(Verb) 할 수 있는가를 정의
K8s RBAC은 화이트리스트 기반이다. 명시적으로 허용하지 않은 모든 것은 거부된다. Role(권한 정의)과 Binding(권한 부여)의 조합으로 접근을 제어한다.
핵심 리소스
Role: 네임스페이스 내 권한 정의
ClusterRole: 클러스터 전체 또는 non-namespaced 리소스 권한
RoleBinding: Subject에게 Role 부여 (네임스페이스 범위)
ClusterRoleBinding: Subject에게 ClusterRole 부여 (전체 클러스터)

Subject 유형:
• User: 사람 (인증서 CN 기반)
• Group: 사용자 그룹
• ServiceAccount: Pod/앱이 사용하는 자동 계정
Verb (API 동작)
읽기: get, list, watch
쓰기: create, update, patch, delete
기타: deletecollection, impersonate

예시 Rule:
rules: - apiGroups: [""] resources: ["pods", "pods/log"] verbs: ["get", "list", "watch"]
apiGroups: [""] # core API (v1) apiGroups: ["apps"] # apps/v1 apiGroups: ["*"] # 모든 API 그룹 (위험!)
Design

역할 설계 패턴

최소 권한 원칙 기반 — 역할별로 필요한 권한만 정확히 부여
역할범위권한대상
cluster-adminClusterRole전체 리소스 전체 verb인프라 관리자 (최소 인원)
namespace-adminRole네임스페이스 내 모든 리소스 CRUD팀 리드, 시니어 개발자
developerRoleDeployment, Pod, ConfigMap 읽기. Pod exec/logs 가능개발자 (일반)
viewerRole모든 리소스 읽기 전용 (get, list, watch)QA, 모니터링, 감사
ci-deployerRoleDeployment, Service, ConfigMap CRUD. Secret 생성만CI/CD 서비스 계정
monitoringClusterRolePod, Node, Endpoint 읽기. 메트릭 엔드포인트 접근Prometheus ServiceAccount
⚠ 위험한 권한 조합
secrets: [get, list] — 모든 시크릿 읽기 가능. 민감 데이터 노출
pods/exec — Pod 내부에서 임의 명령 실행. 컨테이너 탈출 가능
*: [*] — cluster-admin과 동등. 절대 일반 사용자에게 부여하지 말 것
escalate, bind — 자신의 권한을 스스로 확장할 수 있는 위험한 verb
Identity

ServiceAccount 보안

Pod의 K8s API 접근 권한을 제어하는 서비스 계정 — 토큰 자동 마운트 주의
기본 동작과 위험
모든 Pod에 default ServiceAccount 토큰이 자동 마운트된다. 이 토큰으로 K8s API를 호출할 수 있다. 대부분의 앱은 K8s API를 호출할 필요가 없으므로 불필요한 공격 표면이 된다.

방어:
automountServiceAccountToken: false
Pod spec 또는 ServiceAccount에 설정하고, K8s API 접근이 필요 없는 모든 Pod에 적용한다.
IRSA (EKS)
IAM Roles for Service Accounts: K8s ServiceAccount에 AWS IAM Role을 매핑해, Pod이 AWS 서비스에 접근할 때 노드 IAM Role 대신 Pod별 세분화된 권한을 부여한다.
eks.amazonaws.com/role-arn: arn:aws:iam::ACCOUNT:role/pod-role
해당 Pod만 특정 S3 버킷에 접근할 수 있다.

없으면: 노드의 IAM Role을 모든 Pod이 공유해 한 Pod 탈출 시 노드 권한 전체를 획득할 수 있다.
EKS Pod Identity (신규): IRSA보다 설정이 단순하고 OIDC 없이 직접 매핑한다.
Audit

RBAC 감사

현재 RBAC 설정의 과잉 권한을 발견하고 개선하는 방법
감사 도구
kubectl auth can-i:
kubectl auth can-i create pods --as=system:serviceaccount:prod:api-sa kubectl auth can-i --list --as=user@example.com
rbac-lookup (kubectl plugin):
kubectl rbac-lookup --kind=serviceaccount
모든 ServiceAccount의 권한을 한눈에 볼 수 있다.

rakkess (kubectl plugin):
kubectl access-matrix --as=system:serviceaccount:prod:default
리소스×verb 매트릭스를 출력한다.
감사 체크리스트
□ cluster-admin ClusterRoleBinding에 바인딩된 Subject 수 확인 □ default ServiceAccount에 추가 권한이 있는지 확인 -> 전용 SA 생성 □ secrets 리소스에 get/list 권한이 있는 Role 확인 □ pods/exec 권한이 필요 없는 Subject에 부여되어 있는지 확인 □ * 와일드카드 verb/resource를 사용하는 Role 확인 □ 사용하지 않는 RoleBinding 정리
자동화: CI에서 OPA/Kyverno로 RBAC 정책을 검증하고, 과잉 권한이 감지되면 PR을 차단한다.
● Pod Security Standards

Pod Security Standards & SecurityContext

PSA 3단계, 네임스페이스별 적용, SecurityContext 종합 — 컨테이너 격리의 표준.

Standards

Pod Security Standards 3단계

Privileged → Baseline → Restricted — 보안 수준별 허용/금지 사항
PSP(PodSecurityPolicy)는 1.25에서 제거되었다. 대체로 Pod Security Admission (PSA)이 도입되었으며, 3단계 보안 프로파일(Pod Security Standards)을 네임스페이스 레이블로 적용한다.
레벨설명주요 제약대상
Privileged제한 없음. 모든 것 허용없음kube-system, 인프라 워크로드
Baseline알려진 위험한 설정만 차단hostNetwork/hostPID/hostIPC 금지, privileged 금지, 위험 capabilities 금지, hostPath 제한대부분의 앱 워크로드
Restricted최대 보안. 현재 보안 모범 사례Baseline + root 실행 금지, seccomp 필수, capabilities drop ALL, readOnlyRootFilesystem 권장보안 민감 워크로드, 신규 앱
Implementation

PSA 적용 전략

네임스페이스 레이블로 PSS를 enforce/audit/warn 모드로 적용
적용 모드
enforce: 위반하는 Pod 생성을 거부
audit: 위반을 audit log에 기록하고 Pod 생성은 허용
warn: 위반 시 경고 메시지를 표시하고 Pod 생성은 허용

점진적 적용 전략:
1단계: warn: restricted # 경고만, 현황 파악 2단계: audit: restricted # 로그 기록 추가 3단계: enforce: baseline # 위험한 것만 차단 4단계: enforce: restricted # 최대 보안, 준비된 후
네임스페이스 레이블
# 프로덕션 네임스페이스: baseline 강제 + restricted 감사 apiVersion: v1 kind: Namespace metadata: name: production labels: pod-security.kubernetes.io/enforce: baseline pod-security.kubernetes.io/enforce-version: latest pod-security.kubernetes.io/audit: restricted pod-security.kubernetes.io/warn: restricted

kube-system은 privileged로 유지 (CNI, kube-proxy 등 필요)
Hardening

SecurityContext 종합 가이드

Pod과 Container 레벨에서 적용할 수 있는 모든 보안 설정
# ── 최소 권한 SecurityContext (Restricted PSS 호환) ── spec: securityContext: # Pod 레벨 runAsNonRoot: true # root 실행 금지 runAsUser: 1000 # UID 1000으로 실행 runAsGroup: 1000 # GID 1000 fsGroup: 1000 # 볼륨 파일의 GID seccompProfile: type: RuntimeDefault # seccomp 프로파일 containers: - name: app securityContext: # Container 레벨 allowPrivilegeEscalation: false # 권한 상승 금지 readOnlyRootFilesystem: true # 루트 FS 읽기 전용 capabilities: drop: ["ALL"] # 모든 capability 제거 # add: ["NET_BIND_SERVICE"] # 필요한 것만 추가 volumeMounts: - name: tmp mountPath: /tmp # 쓰기 필요 시 emptyDir 마운트 volumes: - name: tmp emptyDir: {}
💡 readOnlyRootFilesystem 적용 시
앱이 /tmp, /var/run 등에 쓰기가 필요하면 해당 경로에 emptyDir을 마운트하여 쓰기 가능하게 한다. 루트 파일시스템 자체는 읽기 전용으로 유지하면서 필요한 경로만 쓰기 허용.
● Supply Chain Security

소프트웨어 공급망 보안

이미지 스캐닝, 서명, SBOM, Admission 게이트 — 빌드부터 배포까지 신뢰의 체인.

Scanning

컨테이너 이미지 취약점 스캐닝

빌드 시점과 런타임 시점에서 이미지의 알려진 취약점을 탐지
CI 파이프라인 스캐닝
Trivy (Aqua Security): 가장 널리 쓰이고 빠른 OSS 도구다.
trivy image --severity HIGH,CRITICAL myapp:latest trivy image --exit-code 1 --severity CRITICAL myapp:latest
CRITICAL 취약점이 발견되면 CI를 실패시킨다.

Grype (Anchore): SBOM 기반 스캐닝
grype myapp:latest --fail-on high
CI 통합:
1. 빌드 후 자동 스캔 2. CRITICAL / HIGH 발견 시 이미지 push 차단 3. 결과를 Slack / 이슈 트래커에 자동 보고
런타임 스캐닝
이미 배포된 이미지도 새 CVE가 발견될 수 있다.

Trivy Operator: 클러스터 내 모든 이미지를 주기적으로 스캔한다.
VulnerabilityReport kubectl get vulnerabilityreports -A
정책:
• 신규 CVE 발견 -> Slack 알림 • CRITICAL -> 24시간 내 패치 또는 mitigation • HIGH -> 스프린트 내 패치 • 기본 이미지(Alpine, Distroless) 주기적 업데이트
Distroless 이미지: 공격 표면을 최소화한다. 셸과 패키지 매니저가 없다.
Trust

이미지 서명 & 검증

이미지가 신뢰할 수 있는 소스에서 빌드되었음을 암호학적으로 보장
이미지 스캐닝은 "알려진 취약점이 있는가?"에 답한다. 이미지 서명은 "이 이미지가 정말 우리 CI에서 빌드된 것인가?"에 답한다. 악의적으로 변조된 이미지 배포를 방지하는 마지막 방어선.
Cosign (Sigstore)
서명:
cosign sign --key cosign.key myregistry/app:sha256-abc
Keyless 서명 (OIDC 기반):
cosign sign myregistry/app:sha256-abc
GitHub Actions의 OIDC 토큰으로 서명하므로 키 관리가 필요 없다.

검증:
cosign verify --key cosign.pub myregistry/app:sha256-abc
CI 통합:
빌드 -> 스캔 -> 서명 -> Push
서명되지 않은 이미지는 클러스터 진입을 차단한다.
Notation (CNCF)
OCI 표준 기반 이미지 서명: AWS Signer, Azure Key Vault 같은 클라우드 KMS와 통합한다.
notation sign myregistry/app:v1 notation verify myregistry/app:v1
서명을 OCI artifact로 레지스트리에 함께 저장하므로 별도 서명 저장소가 필요 없다.

Cosign vs Notation:
• Cosign : 커뮤니티 표준, Sigstore 에코시스템, keyless 지원 • Notation : OCI 표준, 클라우드 KMS 통합 강점 • 둘 다 CNCF 프로젝트
Inventory

SBOM (Software Bill of Materials)

이미지에 포함된 모든 패키지와 라이브러리의 목록 — 취약점 추적의 기반
SBOM은 "이 이미지 안에 정확히 무엇이 들어있는가?"의 명세서다. 새로운 CVE가 발표되면 SBOM을 검색하여 영향받는 이미지를 즉시 식별할 수 있다. Log4Shell(CVE-2021-44228) 사태에서 SBOM이 있었다면 영향 범위를 몇 분 만에 파악할 수 있었다.
SBOM 생성
Syft (Anchore):
syft myapp:latest -o spdx-json > sbom.spdx.json syft myapp:latest -o cyclonedx-json > sbom.cdx.json
Trivy:
trivy image --format spdx-json myapp:latest > sbom.json
형식:
• SPDX : Linux Foundation 표준 • CycloneDX : OWASP 표준
CI 통합: 빌드 시 SBOM을 생성해 이미지와 함께 레지스트리에 저장한다.
cosign attach sbom --sbom sbom.spdx.json myregistry/app:v1
SBOM 활용
취약점 매칭:
grype sbom:sbom.spdx.json
SBOM을 기반으로 취약점을 검색하므로 이미지를 다시 풀 필요가 없다.

활용 시나리오:
• 라이선스 감사 모든 패키지의 라이선스를 추출해 컴플라이언스를 검증 • 사고 대응 "log4j 2.14를 사용하는 이미지가 어디에 있나?" -> SBOM DB 검색 -> 즉시 목록 확보 -> 패치 우선순위 결정 • 규정 준수 미국 Executive Order 14028처럼 SBOM 제출을 요구하는 환경 대응
Enforcement

Admission 게이트

서명되지 않거나 취약한 이미지의 클러스터 배포를 차단하는 최후 방어선
정책 도구
Kyverno YAML 기반 정책 엔진, 학습 곡선이 낮음 ClusterPolicy 로 이미지 레지스트리 화이트리스트, 서명 검증 등을 정의 OPA / Gatekeeper Rego 언어 기반, 강력하지만 복잡도는 높음 ConstraintTemplate + Constraint 로 정책 정의 Sigstore Policy Controller Cosign 서명 검증 전용 서명되지 않은 이미지 배포를 자동 차단
필수 정책 목록
□ 허용된 레지스트리에서만 이미지 pull image: "registry.example.com/*" 만 허용 □ latest 태그 금지, 불변 태그(SHA / commit) 강제 □ 이미지 서명 검증 필수 (Cosign) □ privileged: true 금지 (kube-system 제외) □ hostNetwork / hostPID 금지 □ resource requests / limits 필수 □ runAsNonRoot: true 강제 □ app, owner, version 라벨 필수
● Secret Management

Secret 관리 심화

K8s Secret의 한계, Vault/External Secrets, 자동 로테이션 — 민감 데이터를 안전하게.

Risk

K8s Secret의 위험

K8s Secret은 기본적으로 안전하지 않다 — base64는 암호화가 아니다
기본 Secret의 문제
1. Base64 인코딩 != 암호화 누구나 base64 -d 로 복호화 가능 2. etcd에 평문 저장 etcd 접근이 가능하면 모든 Secret 노출 3. RBAC 미설정 secrets: [get, list] 권한이 있으면 전체 Secret 읽기 가능 4. 환경변수 노출 env 로 주입된 Secret은 /proc/[pid]/environ 에서 읽을 수 있음 5. 로그 유출 앱이 Secret 값을 로그에 출력할 수 있음
기본 방어
etcd 암호화:
EncryptionConfiguration KMS envelope encryption
RBAC 최소화: Secret 접근 권한을 필요한 ServiceAccount에만 부여하고, resourceNames로 특정 Secret만 허용한다.

Volume 마운트 선호: env 대신 volumeMounts로 파일 주입하면 /proc/environ 노출을 줄이고 핫 리로드도 가능하다.
Platform

HashiCorp Vault & External Secrets Operator

전문 시크릿 관리 플랫폼을 K8s와 연동하는 두 가지 접근
Vault Agent Injector
Vault가 Pod에 사이드카를 주입하여 Secret 전달:
vault.hashicorp.com/agent-inject: "true" vault.hashicorp.com/agent-inject-secret-db: "secret/data/prod/db" vault.hashicorp.com/role: "api-server"
Init container와 Sidecar가 Vault에서 Secret을 가져와 /vault/secrets/db에 파일로 마운트한다.

장점: 앱 코드 변경 없음, 동적 Secret 지원, 자동 갱신
단점: Vault 인프라 운영 부담, 사이드카 리소스 오버헤드
External Secrets Operator (ESO)
외부 시크릿 소스에서 K8s Secret을 자동 동기화:

지원 백엔드: AWS Secrets Manager, AWS SSM, Vault, GCP SM, Azure Key Vault 등
ExternalSecret spec.secretStoreRef.name: aws-sm spec.data[].remoteRef.key: prod/db-password spec.refreshInterval: 1h
ESO가 AWS SM에서 값을 가져와 K8s Secret을 생성하고 갱신한다.

GitOps 호환: ExternalSecret YAML은 값 없이 참조만 Git에 커밋하고, ArgoCD가 Sync하면 ESO가 실제 Secret을 생성한다.

추천 이유: Vault 운영 부담 없이 AWS SM/SSM을 직접 활용할 수 있어 소규모 팀에 적합하다.
Rotation

Secret 자동 로테이션

주기적으로 비밀 값을 교체하여 유출 시 피해를 최소화
로테이션 전략
DB 비밀번호 AWS SM 자동 로테이션 -> Lambda가 RDS 비밀번호 변경 -> ESO가 K8s Secret 갱신 -> 앱이 재연결 API 키 듀얼 키 전략: key-v1 / key-v2 동시 유효 새 키 배포 후 이전 키 폐기 TLS 인증서 cert-manager가 자동 갱신 (만료 30일 전) Traefik / Nginx가 파일 변경 감지 후 자동 리로드 권장 주기 DB 비밀번호: 90일 API 키: 180일 TLS: 자동 (cert-manager) 서비스 토큰: 24시간 이하

Detection

런타임 위협 탐지

실행 중인 컨테이너의 비정상 행동을 실시간으로 감지하는 도구
Falco (CNCF)
eBPF 기반 런타임 보안 모니터링:
시스템 콜을 관찰하여 비정상 행동 감지

기본 규칙 예시:
• 컨테이너에서 셸(bash/sh) 실행 감지
• /etc/shadow, /etc/passwd 읽기 감지
• 컨테이너 내 패키지 관리자(apt, yum) 실행
• 예상치 못한 네트워크 연결 시도
• 중요 파일 시스템 변경

배포: DaemonSet으로 각 노드에 설치
출력: Slack, Loki, 이벤트 시스템으로 알림

falco -r /etc/falco/falco_rules.yaml
Tetragon (Cilium)
eBPF 기반 프로세스/네트워크 관측 + 차단:

Falco가 감지만 한다면, Tetragon은 감지 + 차단(enforcement) 가능

TracingPolicy CRD:
• 특정 syscall 호출 시 프로세스 kill
• 네트워크 연결을 특정 CIDR로 제한
• 파일 접근 패턴 감시

장점:
• 커널 레벨 관측 (컨테이너 탈출 시도도 감지)
• Cilium CNI와 통합 (NetworkPolicy + Runtime)
• 낮은 오버헤드 (eBPF 네이티브)
Network

NetworkPolicy 종합 전략

Default Deny → 필요한 통신만 허용 — Zero Trust 네트워크의 K8s 구현
단계별 적용
Phase 1 — 가시성:
Cilium Hubble 또는 Calico Flow Logs로 현재 트래픽 패턴 파악. "어떤 Pod이 어디와 통신하는가?"

Phase 2 — Default Deny:
각 네임스페이스에 default deny ingress + egress 정책 적용

Phase 3 — 허용 정책 추가:
파악된 트래픽 패턴에 따라 필요한 통신만 명시적 허용
• 앱 → DB: 5432 허용
• 앱 → Redis: 6379 허용
• 앱 → 외부 API: egress 허용
• DNS: kube-system의 CoreDNS UDP 53 egress 허용 (필수!)

Phase 4 — 모니터링:
차단된 트래픽 로그 모니터링. 정상 트래픽이 차단되면 정책 추가
DNS Egress 허용 (필수)
Default deny egress 적용 시 DNS도 차단됨!
CoreDNS로의 egress를 반드시 허용:
kind: NetworkPolicy spec: podSelector: {} # 모든 Pod policyTypes: [Egress] egress: - to: - namespaceSelector: matchLabels: kubernetes.io/metadata.name: kube-system podSelector: matchLabels: k8s-app: kube-dns ports: - {protocol: UDP, port: 53} - {protocol: TCP, port: 53}
Hardening

클러스터 하드닝 체크리스트

CIS Benchmark 기반 — K8s 클러스터의 보안을 체계적으로 강화

API Server

□ Anonymous auth 비활성화
□ --insecure-port=0 (이미 기본)
□ Audit logging 활성화
□ RBAC 활성화 (기본)
□ APF로 API 우선순위 설정
□ etcd at-rest 암호화

노드

□ kubelet 인증 필수 (--anonymous-auth=false)
□ --read-only-port=0
□ --protect-kernel-defaults=true
□ OS 보안 패치 자동 적용
□ SSH 접근 최소화 (kubectl debug node 사용)

워크로드

□ PSS Baseline 이상 적용
□ automountServiceAccountToken: false 기본
□ Resource requests/limits 필수
□ 이미지 태그 불변 (SHA 또는 서명)
□ Distroless/Minimal 이미지 사용

네트워크

□ NetworkPolicy default deny
□ API Server 접근을 VPN/Bastion으로 제한
□ 서비스 간 mTLS (가능하면)
□ Ingress에 WAF 적용
□ 불필요한 NodePort 비활성화
