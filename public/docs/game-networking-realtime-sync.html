● Overview

게임 네트워킹 — 일반 백엔드와 본질이 다르다

웹 백엔드 SRE 경험만으로 게임 트래픽을 운영하면 곧 한계에 부딪힌다. UDP·jitter·NAT·시간 되감기 같은 게임 도메인 개념을 모르면 게임사가 던지는 질문에 답할 수 없다. 이 문서는 그 빈 공간을 채운다. (2026-05 기준)

일반 백엔드 네트워킹과 뭐가 다른가

Latency가 절대적

웹은 1초 응답이 평범하지만 FPS는 100ms도 큼. 한 자릿수 ms 차이가 게임플레이를 결정. 평균보다 jitter와 p99 tail이 더 중요.

UDP가 기본

웹은 TCP/HTTP가 표준이지만 게임은 UDP가 기본. 패킷 손실보다 지연이 더 큰 적이라 "다시 보내지 말고 차라리 버리기"가 합리적인 경우가 많음.

Stateful 연결 장기 유지

웹은 request/response 단발성, 게임은 30분~수 시간 동안 한 연결 유지. 중간에 끊기면 매치 손실.

대량 동시 양방향

한 서버가 100명에게 1초 30회씩 다른 데이터 송신. 일반 API와 트래픽 패턴 자체가 다름.

NAT 적대적

웹은 client→server 단방향이라 NAT 무관, P2P/Listen 게임은 NAT traversal이 운영 핵심 문제.

치팅이 일상

패킷·메모리 변조·매크로·인스펙션. 패킷 암호화·서명·검증이 모든 단계에 들어가야 함.

TCP가 게임에 안 되는 이유 — Head-of-Line Blocking

TCP는 신뢰성을 위해 "순서대로 도착"을 강제한다. 한 패킷이 늦게 도착하면 그 뒤 모든 패킷이 대기. 게임에서 "300ms 전 위치"는 가치가 없는데 그 패킷 때문에 "현재 위치"가 막힘.

SRE가 추적할 4대 네트워크 메트릭

RTT (Round-Trip Time)

패킷 왕복 시간. 평균보다 p50·p95·p99 분포가 중요. 같은 평균 50ms라도 p99가 50인지 300인지가 게임 품질을 결정.

Jitter (RTT 변동)

RTT 표준편차. 일정한 100ms는 보간으로 흡수 가능, 들쭉날쭉한 50~200ms는 rubberbanding 유발.

Packet Loss

UDP 송수신 손실율. 게임은 1~3% 정도까지는 보간으로 흡수, 5% 이상은 즉시 게임 품질 저하.

Throughput / Bandwidth

송수신 대역폭. 모바일은 인당 사용량이 비용·UX에 직결. CCU 늘면 전체 대역폭 추적.

★ 이 4가지를 어디서 측정하나
(1) 서버 측 — 각 클라 연결 메타데이터 (2) 클라 측 — 게임 내 디버그 오버레이 + 텔레메트리 (3) 중간 경로 — MTR·smokeping. 셋이 다 있어야 "서버는 정상인데 사용자만 lag" 같은 케이스 디버깅 가능.
● Transport

전송 프로토콜 깊이

QUIC & WebTransport — 차세대 표준 후보

QUIC는 UDP 위에 동작하는 transport 프로토콜로 HTTP/3의 기반. 2026년 기준 web traffic의 40%+ QUIC, Meta는 75%. 게임 적용은 아직 초기지만 cloud gaming·로비 영역에서 채택 진행.

QUIC 장점

독립 stream multiplex (HOL 없음), 내장 TLS 1.3, 0-RTT 핸드셰이크, 연결 ID로 IP 변경에도 연결 유지 (모바일 핸드오프).

QUIC 단점

UDP 차단 환경에서 막힘, kernel bypass라 CPU 부하 큼, 일부 ISP 우회 라우팅. 인게임 high-frequency 트래픽엔 아직 무거움.

WebTransport

QUIC 기반 브라우저 API. reliable byte stream + unreliable datagram 모두 지원. WebSocket의 차세대 후보지만 2026 초 기준 브라우저 production 지원 제한적.

게임 적용 사례

Cloud gaming (Stadia 후속·GeForce NOW 등) 일부, Roblox·Fortnite 같은 대형 사례는 공개 자료 상 미확인. 차세대 후보로 모니터링 가치 있음.

★ 언제 도입?
당장 인게임에 QUIC 도입은 risky. (1) 로비·매칭 API: HTTP/3로 전환 가치 있음 (2) WebTransport: 브라우저 게임에서 WebSocket 대안으로 실험 (3) 인게임 UDP: 여전히 ENet·KCP·GNS·custom UDP가 현실적.

WebRTC DataChannel — HTML5/브라우저 게임의 유일한 UDP-like 옵션

브라우저는 보안상 raw UDP를 못 쓴다. WebRTC DataChannel이 사실상 유일한 UDP-like 솔루션. 내부적으로는 SCTP over DTLS over UDP.

특성

P2P 가능 (STUN), reliable/unreliable 모드 선택 가능, 자동 NAT traversal (STUN → TURN 폴백), 내장 암호화.

Unity WebGL 통합

Unity WebRTC 패키지 + .jslib 브리지로 JS WebRTC 호출. coherence 같은 솔루션이 drop-in UDP transport 제공.

주의: 기본 reliable

WebRTC DataChannel 기본 모드는 reliable+ordered. 게임용으로는 unordered+max-retransmits=0 같은 unreliable 모드 명시 설정 필수.

인프라 요구

STUN 서버 (NAT 발견)·TURN 서버 (relay 폴백) 운영 필수. coturn 같은 오픈소스 사용. TURN은 대역폭 비용 큼.

채널 다중화·우선순위

한 연결로 여러 종류 데이터를 보낸다. "위치 업데이트"는 unreliable·high-frequency, "채팅"은 reliable·ordered, "결과 리포트"는 reliable·once. 채널 분리가 표준.

ENet 채널

최대 255 채널 지원, 채널별 reliable/unreliable, ordered/sequenced 설정. 위치 = 채널 0 (unreliable), 채팅 = 채널 1 (reliable ordered).

QUIC streams

독립 stream 다수, stream별 우선순위. HOL blocking 없음 — stream 1 손실이 stream 2 안 막음. QUIC가 게임에 유리한 핵심 이유.

GNS lanes

Valve GNS는 "lane" 개념으로 채널 우선순위 지원. 중요 메시지가 덜 중요한 메시지에 막히지 않음.

Custom 우선순위 큐

송신 큐를 priority queue로 만들고 매 tick마다 budget 안에서 우선순위순으로 전송. 대역폭 적응 + 우선순위 둘 다 확보.

● Packet Design

패킷 설계 & 직렬화

한 패킷에 1바이트라도 줄이는 게 의미있다. 100명 × 30Hz × 30분 = 540만 패킷. 1바이트 절약 = 5MB 절약. 모바일은 사용자 데이터 비용까지 영향.

Delta encoding — 변한 것만 보낸다

매 tick 전체 월드 상태를 보내는 대신 "직전 스냅샷 대비 차이"만 송신. Quake III·Source 엔진 표준. 변화 적은 캐주얼 게임에선 90%+ 압축률.

Snapshot 송신 패턴
Frame 100: baseline = full snapshot (4 KB)
Frame 101: delta vs frame 100 (300 B) ← 5명만 움직임
Frame 102: delta vs frame 101 (250 B)
Frame 103: delta vs frame 100 (520 B) ← 클라가 ack한 마지막 frame 기준
                  # 클라가 102 ack 못 하면 100 기준으로 계속
Frame 200: 새 baseline
★ 중요
Delta 기준 frame은 "클라가 받았다고 확인한 가장 최근 frame". 클라 ack 메커니즘 필수. 이 ack가 없으면 baseline drift → 데이터 깨짐.

Bit packing — 바이트 경계 무시

"플레이어 ID는 0~63 → 6비트면 충분", "체력은 0~100 → 7비트", "방향은 0~7 → 3비트". 표준 8비트 단위가 아니라 비트 단위로 packing.

Bit packing 예시 (16비트로 4 필드)
player_id  : 6 bits  (0~63)
health     : 7 bits  (0~100)
crouching  : 1 bit
direction  : 3 bits  (0~7)
─────────────────────────────
total      : 17 bits → 3 byte
naive (4 byte 각각): 4 bytes (33% 절약)

대규모 패킷에선 누적 효과 큼.
gafferongames "Reading and Writing Packets" 참조.

Quantization — 위치·회전을 더 적은 비트로

float32 (4 바이트) 좌표를 그대로 보내면 낭비. 월드 크기 1km × 정밀도 1cm = 100,000 단계 = 17 비트면 충분. 4바이트 → 2.1바이트.

Position

월드 bounds + 정밀도 결정 → bit 수 계산. (max - min) / precision = 단계 수, log2 = 비트 수.

Rotation

Quaternion 4값 중 가장 큰 1개는 다른 3개로 추론 → 3개 + 인덱스 2비트만 송신. "Smallest Three" 패턴.

Velocity

실제 게임 속도 범위 (예: -50~50 m/s) + 정밀도 0.1 → 적은 비트.

주의

Quantize 시 정확도 손실 → 클라/서버 간 desync 위험. lockstep 게임은 fixed-point 사용으로 결정성 보장.

패킷 우선순위 & 채널

채널 분리

위치 (unreliable, high-freq), 채팅 (reliable ordered), 인벤토리 (reliable once). 채널별 reliability·ordering 분리.

우선순위 큐

송신 budget 초과 시 낮은 우선순위 패킷 drop or 다음 tick 미루기. AOI 결과 객체 우선순위 (가까운 객체 먼저).

대역폭 cap per client

클라별 bandwidth 한도 (예: 256 kbps) — 모바일 사용자에 맞춰 더 적은 객체 송신. 동적 조정.

밤처리·고우선 합리화

이벤트 (적 처치·아이템 획득) = 즉시 + reliable, 평화 시 위치 업데이트는 빈도 낮춰 송신.

압축 (LZ4 · zstd) — 언제 쓰나

실시간 인게임 패킷은 보통 압축 안 함 (CPU·latency 부하 > 절약 효과). 큰 메시지 (월드 초기 동기화·인벤토리 dump)는 압축이 의미.

LZ4

매우 빠름, 압축률 중간. 100~500 MB/s. 인게임 대형 메시지에 적합.

zstd

속도-압축률 균형 우수, level 조정 가능. asset·매치 결과 dump에 표준.

brotli

압축률 최고지만 느림. 정적 asset (CDN) 적합, 인게임 비추.

주의

UDP 작은 패킷 (~100 B) 압축은 보통 손해 — 헤더가 압축률 못 이김. 1KB 이상에서 효과.

● NAT & Connection

NAT traversal & 연결

P2P·Listen 게임은 NAT 통과가 핵심 운영 문제. 대칭 NAT 비율, hole punching 성공률, TURN 폴백 비용 — SRE가 모니터링해야 한다.

STUN · TURN · ICE — WebRTC 기본 3종

STUN (Session Traversal)

"내 외부 IP·port 뭔지 알려줘" 서버. 클라가 STUN 서버에 패킷 보내면 보이는 외부 endpoint 회신. NAT 종류 발견에 사용.

TURN (Traversal Using Relays)

STUN 실패 시 모든 트래픽 중계. 양 클라가 TURN 서버 거쳐 통신. 100% 성공률, 단 대역폭 비용 풀로 발생.

ICE (Interactive Connectivity)

STUN+TURN을 종합해 최선 경로 선택 framework. WebRTC 표준. host(LAN) → STUN(direct) → TURN(relay) 순으로 시도.

운영 도구

오픈소스: coturn (가장 표준), Pion (Go WebRTC stack). 매니지드: Twilio·Xirsys 같은 SaaS.

★ TURN 비용 함정
TURN은 대역폭이 게임 전체 트래픽 전부 통과. 100명 × 30 kbps × 30분 = 5.4 GB / 매치. 매월 수만 매치면 비용 급등. SRE는 "TURN 폴백률" 메트릭으로 비용 추적 필수.

Photon Cloud relay 폴백 모델

기본 동작

Photon Realtime은 client → Photon region cluster → other clients. 클라이언트 간 직접 연결 없이 모두 cloud relay 경유.

비용 모델

CCU 기반 과금. Photon이 모든 트래픽 통과하므로 대역폭 비용 모두 Photon 측 부담 (사용자가 사용량 분 만큼 plan 결제).

장점

NAT 무관, 자동 region 선택, 인프라 불필요. 모바일 협동 게임 빠른 출시에 표준.

한계

대규모 (CCU 수만~수십만)에서 비용 폭증, custom 로직 한계, vendor lock-in. 대형 게임은 자체 dedicated 서버로 이전.

모바일 CGNAT (Carrier-Grade NAT) — 가장 큰 함정

모바일 캐리어가 IPv4 부족으로 다수 가입자를 한 공인 IP 뒤에 묶음. 동일 공인 IP의 다른 가입자가 같은 source port를 동시에 쓸 수 없으므로 symmetric NAT가 됨.

증상

UDP hole punching 실패율 증가, P2P 매치 못 만들기, 같은 캐리어 사용자도 P2P 안 됨.

대응 1

TURN/Relay 폴백 필수. 모바일 P2P는 폴백률 50%+ 가정.

대응 2

P2P 포기, dedicated 서버 모델 채택. 모바일 협동 게임은 사실상 cloud relay (Photon·자체)가 표준.

대응 3

IPv6 사용 — 모바일 캐리어가 IPv6 enable한 경우 NAT 우회 가능. 한국·일본 캐리어 IPv6 보급률 확인.

● Lag Compensation

Lag Compensation 깊이

"내가 분명 맞췄는데 안 맞아요" — 모든 PvP 게임의 영원한 클레임. 서버가 시간을 되감아 hit detection하는 lag compensation의 동작 원리와 트레이드오프.

Favor the Shooter vs Favor the Dodger — 영원한 트레이드오프

Favor the Shooter

"내 화면에서 맞췄으면 맞는다". A가 본 시점 기준으로 hit 검증. CS·Apex 표준. Latency 높은 사람도 자기 화면에선 정확.

Favor the Dodger

"내 화면에서 안 맞았으면 안 맞는다". B 입장에선 이미 코너 뒤에 숨었는데 죽는 일 없음. 단 A 입장에선 "분명 맞췄는데" 클레임 폭발.

Hybrid

버퍼 시간 제한 (예: 250ms 이상 lag 있는 사람의 hit은 무효). "어느 정도 lag까지만 보호".

운영 영향

SRE가 "왜 nerf 됐냐" 하는 게 아니라, 정책 결정 시 latency 분포 데이터 제공. p99 RTT가 어떻게 변화 중인지 정책 회의에 가져갈 책임.

Interpolation delay 튜닝

클라가 두 스냅샷 사이를 보간하려면 최소 1 스냅샷 간격 (예: 33ms @ 30Hz) 지연이 필요. 이걸 늘리면 jitter 흡수 능력 ↑ but latency ↑.

짧은 interp delay (1 tick)

가장 낮은 latency, jitter 한 번에 끊김 발생.

긴 interp delay (3 tick)

jitter 흡수 우수, 100ms 추가 지연.

적응형

네트워크 jitter에 맞춰 동적 조정. 안정적이면 짧게, 불안정하면 늘리기.

설정 가능 노출

경쟁 FPS는 사용자에게 cl_interp 같은 설정 노출. 기본값 + power-user 노출 패턴.

Backwards Reconciliation — 클라이언트 prediction 보정

F1에서 client prediction 개념 다룸. 여기선 보정 과정 깊이. 클라가 100ms 동안 예측 실행 → 서버 응답 도착 → 차이 발견 → 100ms 전 시점부터 재시뮬레이션.

의사 코드
on input(input):
    predicted_state = simulate(current_state, input)
    apply(predicted_state)              # 즉시 화면 반영
    pending_inputs.append((seq, input))
    send_to_server(seq, input)

on server_state(seq_ack, server_state):
    # 서버가 ack한 input부터 다시 시뮬레이션
    state = server_state
    for input in pending_inputs.from(seq_ack + 1):
        state = simulate(state, input)
    apply(state)                        # 보정된 위치
    pending_inputs.remove_up_to(seq_ack)
★ Reconciliation glitch
예측이 크게 틀리면 사용자가 "순간 이동"처럼 느낌 (rubberbanding). 부드러운 보정 (interpolation between predicted ↔ corrected)이 일반적. 단, 정확도 vs 부드러움 트레이드오프.

모바일 Jitter 대응 — 셀 핸드오프 충격

증상

5G ↔ Wi-Fi 전환, 셀 타워 핸드오프 시 100~500ms RTT spike + 일시적 손실. 보통 1~3초 안에 복구.

대응 1: Adaptive interp delay

네트워크 불안정 감지 시 interp delay 자동 증가. jitter 흡수 우선.

대응 2: Jitter buffer

받은 패킷을 작은 큐에 잠시 보관 후 일정 페이스로 처리. 음성·게임 모두 표준.

대응 3: Connection migration

QUIC connection ID 같은 메커니즘으로 IP 변경 시 연결 유지. 모바일 재접속 UX 핵심.

Anti-lag 어뷰징 — Lag switching

공격 패턴

일부러 송신 끊거나 지연시켜 favor-the-shooter 모델 악용. "사라진 듯 보였다가 갑자기 hit" 패턴.

탐지

플레이어 RTT 패턴 이상 (의도적 spike), packet 송신 gap 분석. 상습 어뷰저 자동 탐지 후 매칭 풀 분리.

완화

Buffer 시간 cap (예: 250ms 이상 무효), high-latency 플레이어 매칭 분리, 서버 측 hit 보정 한계 설정.

SRE 역할

RTT 분포·패턴 텔레메트리 제공. anti-cheat 팀이 분석 가능한 형태로 데이터 노출.

● Security & QoS

보안 & QoS

UDP는 본질적으로 spoofing·flooding·replay에 약하다. 게임 트래픽 보호의 핵심 기술과 운영 패턴을 정리한다.

DTLS · GameNetworkingSockets 암호화

DTLS 1.3

TLS의 UDP 변형. WebRTC 표준. UDP 위에 양방향 암호화·인증 채널. 핸드셰이크 latency 있음 (1-2 RTT).

GNS (Valve)

AES-GCM-256 per-packet 암호화, Curve25519 키교환·서명. DTLS 미사용·자체 프로토콜이지만 DTLS 수준 보안 제공.

QUIC built-in

QUIC는 TLS 1.3 내장. 별도 암호화 레이어 불필요. 0-RTT resumption 가능.

CPU 영향

패킷당 암호화는 CPU 부하 (특히 모바일 클라). AES-NI 같은 하드웨어 가속 활용 필수.

Replay Attack 방어

공격자가 정상 패킷 (예: "보스 처치 보상") 캡처 후 반복 송신 → 보상 무한 획득. UDP는 connectionless이라 특히 취약.

시퀀스 번호

각 패킷에 단조 증가 sequence. 서버는 받은 sequence 추적, 중복·역순 거부. Sliding window 기법.

Nonce + 타임스탬프

각 패킷에 nonce 포함, 서버는 일정 시간 윈도우 안 nonce 추적. 만료된 패킷 거부.

HMAC 서명

패킷 내용 + nonce 해시 — 변조 시 검증 실패. 키는 세션 수립 시 교환.

서버 측 idempotency

중요 액션 (보상·결제)은 idempotent key로 한 번만 처리. 재전송 안전.

Packet Flooding · 입력 검증

Rate limiting per source

IP·세션별 패킷 처리 한도. 초과 시 drop or kick. SDR/Photon 같은 relay는 자체 rate limit 적용.

입력 sanity check

"5초만에 10000 칸 이동" 같은 비정상 입력 거부. 게임 로직 한계 안 입력만 수용.

크기 제한

UDP 패킷 max size 강제 (보통 MTU 1200B 안). 비정상 크기 패킷 즉시 drop.

Connection 검증

handshake 완료된 connection만 게임 로직 진입. unconnected source는 핸드셰이크 단계에서 차단.

Congestion Control — UDP 위 자체 구현

TCP는 CUBIC·BBR 같은 검증된 알고리즘 내장. UDP는 게임이 직접 구현해야 함. 안 하면 네트워크 혼잡 시 패킷 폭격으로 손실 폭증.

방식원리게임 적용
BBRBandwidth + RTT 모델 — 패킷 손실 안 기다림Linux 6.19에서 jitter 20-30% 감소. 게임 트래픽에 우수
CUBIC전통 loss-based — 손실 발생 시 cwnd 감소일반 TCP 표준. 게임에 적합 ↓
QUIC built-inQUIC 자체 congestion control (CUBIC/BBR 선택)QUIC 사용 시 자동
KCP customFastResend·NoCwnd 모드 — 게임 특화 aggressive중국 게임·고손실 환경
GNS 내장Valve 자체 알고리즘 — bandwidth·RTT 기반Steam 게임에 그대로 사용
★ 게임 특이성
게임은 "지연 vs 손실"에서 지연이 더 큰 적. TCP 표준 congestion control은 손실 회피 우선이라 게임에 부적합. KCP·GNS 같은 게임 전용 알고리즘이 더 aggressive 하게 송신.

대역폭 적응 — 네트워크 변화에 동적 대응

Per-client bandwidth budget

클라별 송신 budget (예: 256 kbps) 설정. 모바일 사용자에는 더 적게.

객체 우선순위 down-sampling

대역폭 부족 시 가까운 객체는 30Hz, 먼 객체는 10Hz로 차등.

Adaptive quantization

대역폭 부족 시 위치 정밀도 낮추기 (cm → 10cm 단위).

피드백 loop

클라가 받은 대역폭·손실 보고 → 서버가 송신 rate 조정. 1~2초 주기 적응.

진단 도구 — MTR · iperf · tcpdump · pcap

MTR

traceroute + ping 통합. hop별 loss·latency 보여줌. ISP 라우팅 문제 진단 1순위.

iperf3

두 노드 간 대역폭·jitter·loss 측정. UDP·TCP 모두. region별 백본 품질 검증.

tcpdump / wireshark

패킷 캡처·분석. 게임 프로토콜 디버깅 — 어떤 패킷이 손실되는지 직접 확인.

nload / iftop

실시간 인터페이스 대역폭 모니터링. 노드별 트래픽 spike 즉시 확인.

ss / netstat

현재 connection 상태. send/recv queue 크기, 오래 걸리는 connection 확인.

RIPE Atlas

전 세계 분산 측정 노드. "다른 region에서 우리 서버 어떻게 보이나" 외부 시점 측정.

ISP 라우팅 트러블슈팅 — "특정 ISP만 lag" 케이스

증상

특정 ISP·지역 사용자만 RTT 폭증·손실. 같은 region 다른 ISP는 정상.

1단계: MTR

해당 사용자 또는 RIPE Atlas로 MTR. 어느 hop에서 loss 시작? 보통 ISP transit 또는 IXP.

2단계: BGP route

BGPView로 AS path 확인. 비효율 경로 (예: KR ISP → US backbone → KR server) 발견.

3단계: 우회

(1) Anycast/CDN 활용 (2) 다른 region 옵션 제공 (3) ISP에 peering 요청 (4) SDR 같은 사설 백본 도입.

Troubleshooting Playbook — "Lag 클레임 들어왔을 때"

텔레메트리지역·ISP·플랫폼별 RTT
서버 healthy?tick·send queue
국지적?특정 region·ISP만
진단MTR·BGP·RIPE
완화region 추가·peering·SDR
사용자 통보알려진 이슈 페이지

Step 1: 범위 결정

특정 사용자 1명? 한 ISP? 한 region? 글로벌? 텔레메트리로 즉시 판단. 단일 사용자면 클라 환경, 글로벌이면 서버 측.

Step 2: 서버 자체 검증

tick budget·send queue·CPU. 문제 없으면 네트워크 경로 의심.

Step 3: 외부 측정

RIPE Atlas 같은 도구로 외부에서 우리 서버 ping 측정. 우리 모니터링 시스템과 다른 시점.

Step 4: 완화 + 통보

긴급은 region 추가·SDR fallback. 사용자에게 알려진 이슈 페이지·인게임 공지.
