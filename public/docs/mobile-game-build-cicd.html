● Overview

모바일 게임 빌드의 특수성

모바일 게임 빌드는 일반 모바일 앱과 다르고, 일반 모바일 앱 빌드는 웹 빌드와 다르다. 큰 산출물·플랫폼별 사이닝·스토어 정책·엔진 빌드 시간이 모든 의사결정의 제약이다. 2026-05 기준

왜 일반 앱 빌드와 다른가

① 산출물 크기

일반 앱: 수십 MB. 모바일 게임: 수백 MB ~ 수 GB. AAA급 게임은 압축 후에도 4~10GB. 빌드 시간·CI 비용·아티팩트 저장 비용이 비례 증가.

② 빌드 시간

일반 앱(Swift/Kotlin): 1~5분. 모바일 게임(Unity/Unreal): 30분 ~ 2시간. IL2CPP·셰이더 컴파일·에셋 import가 주범. 캐싱·증분 빌드가 필수.

③ 플랫폼별 사이닝

iOS: Apple Developer 인증서 + provisioning profile + (App Store) notarization. Android: Play App Signing + upload key. 잘못 사이닝하면 스토어 거부 — 빌드 자동화의 가장 흔한 실패 지점.

④ 에셋 파이프라인

코드뿐 아니라 에셋(텍스처·모델·사운드)도 빌드 산출물의 일부. Unity는 Library/, Unreal은 Cooked/에 변환된 에셋. 일반 앱에서 보지 못하는 빌드 단계.

★ 운영 함의
모바일 게임 CI는 일반 모바일 앱 CI 도구만으로는 부족한 경우가 많다. Bitrise/Codemagic은 모바일 앱에 특화, EAS Build는 RN/Expo 특화. 큰 게임 프로젝트는 Unity Build Automation 또는 자체 GitHub Actions + 큰 self-hosted runner 조합이 일반적.

모바일 게임 CI 도구 비교 (2026-05)

모바일 빌드 가능한 도구는 다양하지만, 게임 빌드에 적합한 건 일부. macOS 러너(iOS 빌드 필수)와 큰 RAM/CPU(Unity/Unreal 빌드)가 핵심 요건.

표준 파이프라인 단계

모바일 게임 빌드 파이프라인은 일반 앱 대비 단계가 많다. 각 단계가 5~20분 — 단계 분리 + 캐싱이 wall clock time을 결정.

Checkout~30s
Asset Import~5min
Compile/Cook~20min
Sign~1min
Test~10min
Submit~5min

① Asset Import (Unity)

Unity는 첫 빌드 시 Library/ 디렉토리 생성 — 텍스처 압축, 셰이더 컴파일, 메시 변환. 한 번 만들면 캐싱 가능. CI 러너 간 공유하면 5분 → 30초.

② Compile / Cook

가장 긴 단계. Unity IL2CPP는 C# → C++ → 네이티브 코드. Unreal "cooking"은 에셋 → 플랫폼 native 형식. 증분 빌드 + 분산 컴파일로 줄임.

③ Sign

iOS: provisioning profile + 인증서로 서명. Android: AAB sign. 실패 시 build 자체 실패. CI 시크릿 관리 + match 패턴(Fastlane).

④ Test

유닛 테스트 + 디바이스 팜에서 실행 테스트. Firebase Test Lab(Android), Xcode 시뮬레이터(iOS). 모든 디바이스 매트릭스 돌리면 시간 폭발 — 핵심 디바이스만.

⑤ Submit

App Store Connect API / Play Developer API로 업로드. 심사 트리거는 별도 — 자동으로 release 안 함, internal/alpha track으로 우선. 심사 거부 시 자동 실패 처리.

● iOS Build

iOS 빌드 — 인증서 지옥에서 살아남기

iOS 빌드의 90% 사고는 인증서 / 프로파일 / 사이닝 관련. macOS 러너 + Xcode + 인증서 자동화가 핵심. 2026-04-28부터 iOS 26 SDK + Xcode 26 빌드 의무.

iOS 빌드 필수 요소

① Apple Developer Program

$99/년 (개인) 또는 $299/년 (Enterprise). App Store 배포는 일반 Program. Enterprise는 사내 배포 전용으로 App Store 사용 불가.

② Xcode + macOS

iOS 빌드는 macOS 필수. 2026-04-28부터 Xcode 26 + iOS 26 SDK로 빌드 의무. CI 러너의 Xcode/macOS 버전을 정기 업데이트.

③ 인증서 (Certificates)

Development: 개발 디바이스 설치용. Distribution: App Store / Ad Hoc / Enterprise. Distribution 인증서는 1개만 두는 게 권장 (분실 시 모든 빌드 영향).

④ Provisioning Profile

인증서 + Bundle ID + Device(개발용) + Capability를 묶음. 만료 1년. App Store 배포용 profile은 Bundle ID당 1개. 자동 갱신 도구 필수.

⚠ 인증서 분실의 비용
Distribution 인증서를 분실하면 새 인증서를 만들어야 하고, 그 인증서로 만든 새 빌드만 App Store에 업로드 가능. 이전 빌드 hotfix가 어려워진다. 인증서 .p12 파일을 안전한 곳에 백업 + 사이닝 파워는 최소 인원만.

인증서 / 프로파일 관리 — 자동화 패턴

모든 개발자 머신과 CI에 인증서를 수동 설치하는 건 운영 지옥. git 저장소에 암호화된 인증서를 두고 자동 동기화하는 패턴이 표준 (Fastlane match).

전통 방식 — 수동

Apple Developer Portal에서 인증서/프로파일 생성 → 다운로드 → 각 머신에 설치. 인원이 늘면 동기화 불가능. 한 사람이 인증서 갱신하면 다른 사람들 빌드 깨짐.

현대 방식 — match (Fastlane)

인증서/프로파일을 git private repo에 암호화 저장. fastlane match으로 동기화. 새 사람도 match development 한 번이면 환경 셋업 완료.

현대 방식 + ASC API — 더 진보

App Store Connect API로 인증서/프로파일을 코드로 생성·갱신. fastlane match는 ASC API를 backed로 사용하므로 (1) 인증서 만료 자동 갱신, (2) 새 device 자동 등록, (3) 사람의 Apple ID 인증 불필요.

match — git 기반 인증서 동기화
# Matchfile
git_url("git@github.com:myorg/ios-certificates.git")
storage_mode("git")
type("appstore")
app_identifier(["com.example.game"])
api_key_path("./asc-api-key.json")

# 새 머신 / CI에서 한 번만 실행
# 1) 인증서 + 프로파일 다운로드 (없으면 자동 생성)
# 2) keychain에 import
# 3) Xcode가 자동 인식
$ fastlane match appstore --readonly
♥ 보안 — match 암호
match는 git repo에 인증서를 암호화 저장한다. MATCH_PASSWORD 환경 변수를 시크릿으로 관리. 이 password가 유출되면 인증서 통째로 노출. CI는 OIDC + Vault에서 받아 쓰고, 사람은 1Password 같은 manager로.

Xcode Cloud — Apple 공식 CI

Apple이 직접 운영하는 CI/CD. Xcode 안에서 설정, GitHub/Bitbucket/GitLab 연동, TestFlight 자동 배포. 인증서/프로파일 자동 관리가 가장 큰 장점 — match 같은 도구 불필요.

장점

(1) 인증서/프로파일 완전 자동, (2) Xcode 통합 (push 시 자동 빌드), (3) TestFlight 자동 업로드, (4) Apple 인프라 최적화 — 빠른 macOS 러너.

한계

(1) iOS / macOS / tvOS / watchOS만, Android X, (2) 컴퓨트 시간 비싸짐 빠름 (1000h $399.99 = 시간당 $0.40), (3) Unity/Unreal 빌드 어려움 — 보통 별도 단계로 빌드 후 Xcode 프로젝트만 Xcode Cloud로.

게임 적합도

네이티브 iOS 게임(Swift/Objective-C)이라면 좋은 선택. Unity/Unreal 게임은 적합도 낮음 — 엔진 빌드는 다른 CI(Unity Build Automation, self-hosted runner)에서, 마지막 archive/sign/upload만 Xcode Cloud로 분리하는 패턴은 가능.

TestFlight — 베타 배포 표준

App Store 출시 전 베타 테스터에게 배포. 내부(Internal) + 외부(External) 두 그룹. 내부는 즉시, 외부는 짧은 검토(보통 24시간 이내).

Internal Testing

App Store Connect 팀 멤버 최대 100명. Apple 검토 없이 즉시 배포. 일일 빌드를 자동으로 internal로 push하는 게 표준 워크플로우.

External Testing

외부 테스터 최대 10,000명. 첫 빌드는 Beta App Review 필수 (보통 24시간 이내, 내용 변경 없으면 이후 빌드는 fast-track).

운영 패턴 — daily / nightly / canary

Daily build: main branch push → 자동 internal 배포. 개발팀이 즉시 검증.
Weekly external: 안정화된 빌드만 external group에. 외부 테스터 피드백 수집.
Pre-release canary: App Store 직전 빌드를 100% external에 우선 배포 → 1주 검증 → 문제 없으면 prod release.

App Store Connect API — 자동화의 핵심

Apple ID + 2FA로 사람이 매번 로그인하는 시대는 끝났다. App Store Connect API Key는 토큰 기반 인증으로 완전 자동화 가능. CI에서 사람 개입 없이 빌드 → 업로드 → release까지.

API Key 구성 요소

(1) Key ID: 10자 알파벳/숫자, (2) Issuer ID: UUID 포맷, (3) Private Key: .p8 파일 — 한 번만 다운로드 가능, 분실 시 재발급. CI 시크릿에 안전 저장.

최소 권한 원칙

API Key는 role 단위 권한. Developer role로 notarytool, fastlane 인증 가능. Admin은 절대 금지 — 노출 시 영향 막대. 키마다 한 가지 용도로.

실전 — 자동화로 할 수 있는 것

(1) 빌드 업로드: xcrun altool 또는 fastlane pilot. (2) TestFlight 그룹 관리: 테스터 추가/제거, 빌드 알림. (3) App Store metadata: 설명, 스크린샷 업데이트. (4) Release 트리거: 검토 완료된 빌드 자동 release. (5) Notarization: macOS 게임 사전 검증.

ASC API key를 fastlane에서 사용
# asc-api-key.json (CI secret으로 관리)
{
  "key_id": "ABC1234XYZ",
  "issuer_id": "57246542-96fe-1a63-...",
  "key": "-----BEGIN PRIVATE KEY-----\\n...",
  "is_key_content_base64": false
}

# Fastfile
lane :beta do
  app_store_connect_api_key(
    key_filepath: "./asc-api-key.json",
    in_house: false
  )
  build_app(scheme: "GameApp")
  upload_to_testflight
end
● Android Build

Android 빌드 — Play App Signing 시대

Android 빌드는 iOS 대비 단순한 면이 있지만, 2021년 이후 모든 신규 앱이 Play App Signing 의무가 되며 keystore 관리 모델이 바뀌었다. AAB 형식, 다양한 track, Play Developer API.

Android 빌드 필수 요소

① Android Studio + Gradle

Gradle이 빌드 시스템. build.gradle(또는 build.gradle.kts)에 의존성·signing·flavor 정의. 큰 프로젝트는 빌드 5~30분.

② Keystore (Upload Key)

Play App Signing 등록 후에는 upload key로 서명 → Play가 receive → Play의 app signing key로 재서명. upload key 분실 시 Google support 통해 reset 가능 (이전엔 분실 = 앱 영원히 업데이트 불가).

③ Java/Kotlin → DEX

Java/Kotlin 코드 → bytecode → D8/R8 컴파일러로 DEX. R8은 ProGuard 후신 — minify + obfuscate + optimize. 게임에선 Unity/Unreal이 자체 빌드 도구 사용해 이 단계 hidden.

④ Multiple ABI

Android는 arm64-v8a, armeabi-v7a, x86_64 등 여러 CPU 아키텍처. 각 ABI마다 네이티브 라이브러리 다름. AAB로 빌드 시 Play가 디바이스별로 자동 분배.

Play App Signing — 필수 가입

신규 앱은 Play App Signing 가입 의무 (2021년 이후). 기존 앱도 점진 의무화. Google이 app signing key를 안전하게 관리, 개발자는 upload key만 사용.

흐름

(1) 개발자 → upload key로 AAB 서명 → Play 업로드. (2) Play가 upload signature 검증. (3) Play가 자체 app signing key로 재서명. (4) 사용자에게 Play app signing key로 서명된 APK 배포.

Upload Key 요구사항

RSA 2048-bit 이상. keytool로 생성. 만료 25년 권장 (앱 수명보다 길게). 분실 시 Google에 reset 요청 가능 — 이전엔 절대 불가능했음.

장점 + 함정

장점: upload key 분실 가능, 키 회전 가능, Play가 디바이스별 최적화 APK 생성 (size ↓). 함정: "사용자가 받는 APK는 개발자 keystore 시그니처가 아니다" — 다른 스토어(Galaxy Store, 화웨이 AppGallery, 자체 APK)에 같은 빌드를 못 올림. 멀티 스토어 운영 시 빌드 파이프라인 분기 필요.

⚠ Galaxy Store / 화웨이 / Steam
한국 시장은 Google Play 외에도 Galaxy Store(삼성), One Store(KT/SKT/LG)가 있다. Play App Signing은 Google Play 전용 — 다른 스토어용은 자체 keystore로 별도 빌드. 빌드 파이프라인이 N개 스토어 × M개 ABI 조합으로 늘어남.

AAB (Android App Bundle) vs APK

2021년 8월 이후 Play Store 신규 앱은 AAB 의무. APK 시대는 끝났다. AAB는 빌드 산출물이고, Play가 디바이스별로 최적화된 APK를 동적 생성.

AAB의 본질

"한 번 빌드, 디바이스별 최적화 APK 생성". 사용자 디바이스의 ABI / 화면 해상도 / 언어 / DPI에 맞는 리소스만 다운로드 → APK 사이즈 평균 15~50% 감소.

Dynamic Feature Modules

한 앱을 모듈로 쪼개 필요할 때 다운로드. 게임에서는 잘 안 씀 — 게임은 보통 한 덩어리. 단, 큰 게임의 language pack이나 고해상도 에셋은 dynamic feature로 분리 가능.

Play Asset Delivery

큰 에셋(게임 데이터)을 install-time / fast-follow / on-demand로 분리. 150MB APK 한도 우회. 게임에서 가장 자주 쓰는 AAB 기능.

APK는 언제?

다른 스토어 배포(Galaxy Store, 직접 배포): APK 필요. AAB → APK 변환은 bundletool로. 단 Play App Signing 적용된 AAB는 그 키로 서명되므로 다른 스토어용은 별도 keystore로 빌드.

Play Console Tracks — 단계별 출시

Play Console은 4개 release track을 제공. internal → closed → open → production의 단계적 출시 패턴.

Track대상심사적합 단계
Internal Testing최대 100명 (이메일)없음 (몇 분)일일 빌드, 사내 QA
Closed Testing (Alpha)이메일 리스트 / Google 그룹1차 심사 (수 시간 ~ 1일)외부 베타 테스터
Open Testing (Beta)opt-in 누구나심사 완료 (1일~)대규모 베타 (소프트 런칭)
Production모든 사용자완전 심사정식 배포

Staged Rollout

Production track도 100% 즉시 배포 안 함. 1% → 5% → 20% → 50% → 100% 점진. 각 단계에서 crash rate 모니터 → 이상 시 halt + rollback. Play Console UI에서 staged rollout 설정 가능.

Halt vs Rollback

Halt: 새 버전 분배 중단 (이미 받은 사용자는 그대로). Rollback: 이전 버전 신규 분배 — 단, 이미 받은 사용자는 새 버전 유지 (downgrade는 데이터 호환 문제). 새 신규 다운로드만 이전 버전.

Play Developer API — 자동화

Apple과 마찬가지로 자동 업로드/release/track 관리 가능. Service account JSON key로 인증, REST API 또는 fastlane supply으로 사용.

Play Developer API — 자동 internal track 업로드
# Fastfile (supply 액션)
lane :beta_android do
  gradle(task: "bundle", build_type: "Release")

  upload_to_play_store(
    track: "internal",                # internal | alpha | beta | production
    aab: "app/build/outputs/bundle/release/app.aab",
    json_key: "./play-service-account.json",
    skip_upload_apk: true,           # AAB 우선
    release_status: "completed"          # draft | inProgress | completed | halted
  )
end
♥ Service Account 보안
Play Developer API 인증의 service account JSON key는 Apple ASC API key와 동일 수준의 시크릿. 분실 시 즉시 Google Cloud Console에서 disable + 새 key 발급. CI 시크릿에 저장, Vault/Secret Manager 권장.
● Fastlane

Fastlane — 모바일 빌드 자동화의 사실상 표준

2014년부터 운영, 모바일 CI/CD의 사실상 표준. 인증서 관리(match), 빌드(gym/gradle), 업로드(pilot/supply), 메타데이터 관리(deliver/produce). Fastlane 없이 모바일 자동화는 어렵다.

Fastlane 핵심 — lane, action, plugin

Lane

실행 단위. fastlane beta로 호출. lane은 action들의 시퀀스. 일반적으로 dev / beta / release 같은 단계별 lane 정의.

Action

한 작업. build_app, upload_to_testflight, match 등 200+ built-in. 자체 action 정의 가능 (Ruby).

Plugin

community-contributed action. fastlane add_plugin firebase_app_distribution으로 추가. Firebase, AWS, Slack 등 외부 통합 대부분 plugin으로 제공.

★ Fastlane 2026 상태
Fastlane은 여전히 활발히 유지보수 중. App Store Connect API를 backend로 사용하는 방향으로 점진 전환 — 사람의 Apple ID + 2FA 의존 줄임. 단 일부 메타데이터/스크린샷 업로드는 ASC API 미지원으로 여전히 spaceship(legacy iTunes Connect API) 사용.

match — 인증서/프로파일 git 동기화

Fastlane의 가장 강력한 도구. 인증서·프로파일을 git private repo에 암호화 저장 → 모든 머신에서 동일 환경. 새 개발자 온보딩 5초.

1. Generate

match가 ASC API로 cert/profile 생성

→

2. Encrypt

OpenSSL로 암호화

→

3. Push

git private repo 저장

→

4. Sync

다른 머신에서 pull + decrypt

Storage 옵션

git: 가장 일반적, private repo. S3: AWS 환경, 더 빠름. Google Cloud Storage: GCP 환경. 어느 backend든 MATCH_PASSWORD로 암호화.

readonly 모드 in CI

CI는 match --readonly로 실행. 새 인증서 생성 안 함, 기존 것만 다운로드. 실수로 CI가 새 인증서 만드는 사고 방지. 인증서 갱신은 사람 머신에서만.

Lane 패턴 — 표준 구성

실전에서 많이 쓰는 lane 구조. dev → beta → release의 단계별 lane + 플랫폼별 분리.

Fastfile — 표준 패턴
platform :ios do
  before_all do
    setup_ci                              # CI 환경 자동 감지 + keychain 설정
    app_store_connect_api_key(
      key_filepath: ENV["ASC_API_KEY_PATH"]
    )
  end

  desc "개발용 빌드 (사내 배포)"
  lane :dev do
    match(type: "adhoc", readonly: true)
    build_app(scheme: "Game-Dev", export_method: "ad-hoc")
    firebase_app_distribution(
      app: "1:...",
      groups: "qa-team"
    )
  end

  desc "TestFlight internal 배포"
  lane :beta do
    match(type: "appstore", readonly: true)
    build_app(scheme: "Game", export_method: "app-store")
    upload_to_testflight(
      skip_waiting_for_build_processing: true,
      groups: ["Internal QA"]
    )
  end

  desc "App Store 제출"
  lane :release do
    match(type: "appstore", readonly: true)
    build_app(scheme: "Game")
    deliver(
      submit_for_review: true,
      automatic_release: false,            # 수동 release
      force: true
    )
  end
end

platform :android do
  lane :beta do
    gradle(task: "bundle", build_type: "Release")
    upload_to_play_store(track: "internal", aab: "app/build/.../app.aab")
  end
end

CI 시크릿 관리

Fastlane은 시크릿을 많이 다룬다 — match password, ASC API key, Play service account, Firebase token. CI에서 이걸 어떻게 안전하게 주입할지가 보안 핵심.

관리해야 할 시크릿

MATCH_PASSWORD, FASTLANE_PASSWORD (legacy), ASC_API_KEY_*, PLAY_SERVICE_ACCOUNT_JSON, FIREBASE_TOKEN, SLACK_WEBHOOK. 각각 다른 권한 범위로.

저장 위치

(1) GitHub Secrets / GitLab CI variables: 가장 단순, repo 단위. (2) HashiCorp Vault: 큰 조직, 다양한 환경. (3) AWS Secrets Manager + OIDC: 정적 키 zero. CI가 OIDC로 인증.

회전 — 정기 / 사고 시

정기 회전: 분기마다 ASC API key, Play service account, match password 회전. 사고 시: 시크릿 leak 의심 시 즉시 무효화. ASC API key는 Apple Dev Portal에서 즉시 revoke. 회전 가능한 구조는 평소 검증 — 분기 회전 자동화로 검증 겸 운영.

Unity / Unreal 모바일 빌드

엔진 빌드는 일반 모바일 앱과 별도의 세계. IL2CPP 컴파일, 에셋 import/cooking, Addressables/Pak 패키징. 빌드 시간 30분~2시간이 일반적, 캐싱이 곧 생산성.

Unity 빌드 파이프라인

Unity의 모바일 빌드는 7단계 — Editor 시작 → Library 빌드 → Asset import → Script compile → IL2CPP 변환 → 네이티브 컴파일 → archive. 각 단계가 캐싱 대상.

Editor 자동화

UI 없이 빌드 — Unity -batchmode -nographics -executeMethod BuildScript.BuildAndroid. BuildScript.cs에 빌드 로직. Unity 라이센스 활성화가 CI 첫 단계 — Personal/Plus/Pro/Enterprise.

Library 캐싱

가장 큰 빌드 시간 단축 효과. Library/ 디렉토리를 CI 러너 간 공유. GitHub Actions actions/cache로 ~1GB 캐시. 매 빌드 5분 → 30초.

Unity Build Automation (구 Cloud Build)

Unity 공식 CI — Unity DevOps의 일부. 2026-03-01 가격 정책 변경: 25GB 무료, Windows 200분/Mac 100분/Linux 100분 무료, 추가 $0.14/GB. asset import 캐싱이 자동이라 self-managed 대비 빌드 시간 절반. Unity 게임은 1순위 후보.

IL2CPP 최적화 — 빌드 시간의 30%

IL2CPP(Intermediate Language to C++)는 Unity의 AOT 컴파일러. C# IL → C++ → 플랫폼 native 코드. 런타임 성능 좋지만 빌드 시간 길고 메모리 많이 씀.

Code Generation 옵션

Player Settings에 Faster runtime vs Faster (smaller) builds. 후자는 빌드 시간 + 사이즈 최적화, 런타임 약간 느림. Dev / staging 빌드는 후자, prod는 전자 권장.

Storage — SSD 필수

IL2CPP는 대량 read/write. HDD 사용 시 빌드 시간 2~5배. CI 러너는 NVMe SSD 권장. Mac mini M2/M3은 기본 SSD 충분.

증분 빌드

같은 빌드 디렉토리에 다시 빌드하면 IL2CPP가 변경된 부분만 재컴파일. CI에서는 보통 새 디렉토리 → 증분 안 됨. 캐싱 디렉토리 보존하면 증분 가능 — 다만 stale 위험 있어 정기적 clean build 필요.

Mono vs IL2CPP

iOS는 IL2CPP 강제. Android는 선택 가능 — Mono는 빌드 빠르나 런타임 느리고 64-bit 제한. 2026 기준 Android도 IL2CPP 표준이며 Mono는 빌드/디버그 환경 한정.

Addressables 빌드 — 에셋 분리

큰 에셋(텍스처·모델·사운드)은 빌드에 박지 말고 별도 번들로 빌드 → CDN에 업로드. 클라는 첫 실행 시 다운로드 또는 on-demand. 패치 사이클이 클라이언트 사이클과 분리되는 핵심 도구.

빌드 흐름

(1) Build Content: Addressable group → bundle 생성. (2) Catalog 생성: catalog.json (어떤 키가 어떤 번들에). (3) CDN 업로드: bundle + catalog. (4) Player Build: 클라 빌드는 catalog URL만 박고 빌드.

CI에서

Addressable build를 일반 빌드와 분리. Addressable이 변경된 PR만 별도 step에서 빌드 → CDN 업로드. 클라이언트 빌드는 catalog URL만 참조 (Remote Config로 catalog 버전 관리).

운영 — catalog 버전 관리

여러 catalog 버전이 동시 운영 가능 (구 클라가 구 catalog 사용 중). 강제 업데이트로 catalog 통합 또는 호환성 유지 둘 중 선택. 보통 메이저 패치는 새 catalog, 마이너는 기존 catalog에 변경 추가.

Unreal Build Pipeline

Unreal의 모바일 빌드는 Unity와 비슷한 구조 — 다만 더 복잡. Cooking + Packaging + Staging + Archiving 4단계 + UnrealBuildTool. AAA급 게임은 빌드 1~2시간이 일반.

UnrealBuildTool (UBT)

C++ 코드 빌드 매니저. 모듈 단위 컴파일, 헤더 의존성 관리. 분산 컴파일(Incredibuild, FASTBuild) 지원 — 큰 팀에선 빌드 시간 1/4로.

Cooking

에셋을 플랫폼 native 형식으로 변환. 텍스처 압축, 메시 최적화. iOS/Android 각각 별도 cooking — 같은 에셋이 플랫폼별 다른 결과.

Pak / IO Store

Cooked 에셋을 Pak 파일로 패키징. UE 5.x는 IO Store로 진화 — 더 효율적, 에셋 의존성 메타데이터 보유. delta 패치에 유리.

BuildGraph

Unreal 공식 빌드 자동화 도구. XML로 빌드 단계 정의, 분산 실행 가능. RunUAT BuildGraph -script=path/to/Game.xml. 큰 프로젝트 표준.

빌드 캐싱 — 시간 단축의 핵심

엔진 빌드 30분~2시간을 2분으로 줄이는 가장 큰 lever. 캐싱 안 한 CI는 게임 프로젝트에서 비용·생산성 둘 다 잡아먹는다.

캐시 대상

Unity: Library/, Library/PackageCache/, Builds/ (증분). Unreal: Intermediate/, DerivedDataCache/, Saved/. 플러그인: Library/PackageCache/ 또는 Plugins/.

캐시 키 설계

${{ hashFiles('**/*.cs', 'ProjectSettings/**') }} — 코드/설정 변경 시에만 invalidate. 파일이 정말 영향을 미치는 것만 hash에 포함. Library/만 캐싱하면 90% hit rate 가능.

분산 컴파일

Unreal: Incredibuild ($commercial), FASTBuild (OSS), Unreal Horde. C++ 컴파일을 여러 머신에 분산 → 빌드 시간 1/4. Unity: 분산 컴파일 도구는 적음, 대신 캐싱이 핵심.

Self-hosted runner의 이점

큰 게임 프로젝트는 self-hosted runner의 로컬 캐시가 hosted runner의 네트워크 캐시보다 빠름. Mac Studio/PC 한 대에 SSD 1TB + 8GB+ RAM이 일반적.
