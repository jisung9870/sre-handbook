● Overview

크로스 플랫폼 운영의 본질

한 게임을 모바일·PC·콘솔에서 동시에 굴리면 매출은 1.5~2배가 되지만 운영 부담은 플랫폼 수의 제곱에 가깝게 늘어난다. 본 가이드는 "동시 운영"의 갭(gap)을 메우는 SRE 패턴에 집중한다. 각 플랫폼 자체 깊이는 P1·P3·P4·P9 가이드 참조.

Cross Play / Cross Save / Cross Progression — 정의·구분

세 용어는 자주 혼용되지만 운영에서는 각각 다른 시스템·정책·KPI를 갖는다. 시작 시점에 어디까지 지원할지 명확히 정해야 한다.

Cross Play

다른 플랫폼 사용자와 실시간으로 같이 플레이. PC↔모바일 또는 PC↔콘솔 매칭. 매칭·input fairness·anti-cheat이 핵심 과제.

Cross Save

저장 파일·진행도를 플랫폼 간 동기화. 모바일에서 시작 → PC에서 이어 플레이. 충돌 처리·sync 빈도·인벤토리 정합이 핵심.

Cross Progression

레벨·도전과제·소장 아이템·결제 재화를 플랫폼 간 통합. Cross Save의 상위 개념 — 단순 저장이 아니라 계정 단위 진행도 통합. 결제 재화 정합성이 가장 큰 과제.

★ TIP
"Cross Play만 + 진행도 분리" 구조도 흔하다(콘솔 정책 때문). 가장 야심찬 모델은 "Cross Play + Cross Progression" — Fortnite, Warframe, Marvel Rivals가 이 모델. 운영 복잡도가 가장 높지만 사용자 만족도도 가장 높다.

시작 전 결정 5가지

크로스 플랫폼 게임을 출시하기 전에 운영 모델을 결정하는 5가지 핵심 질문. 한번 결정하면 나중에 바꾸기 어려운 항목들이다.

① Cross Play를 할 것인가

같이 플레이 가능 vs 분리. 매칭·anti-cheat·input fairness 모두에 영향. PvE 게임은 자유, PvP는 신중. 한 번 cross play로 시작하면 떼어내기 어려움.

② 진행도 통합 vs 분리

Cross Progression vs 플랫폼별 분리. 결제 재화 정합·환불·세금 처리가 통합 시 매우 복잡. 분리하면 사용자 만족도 ↓.

③ 통합 계정 시스템

자체 계정 ID vs 플랫폼 ID 그대로. 자체 계정 ID는 cross progression의 전제. 사용자가 가입을 한 번 더 해야 하는 부담.

④ 동시 출시 vs 시차 출시

모바일·PC·콘솔 동시 vs 순차 출시. 동시 출시는 마케팅 시너지 ↑ 운영 부담 ↑↑. 시차는 안전하지만 매출 곡선이 분산.

⑤ 콘솔 포함 여부

모바일+PC만 vs 콘솔 추가. 콘솔은 플랫폼 인증·테스트·정책 추가 부담이 크고 첫 타이틀은 더 어려움. 그러나 일부 장르(액션 RPG·격투·협력 게임)는 콘솔 매출이 큰 비중. 매출 모델·장르 적합도로 판단.

SRE 운영 부담

크로스 플랫폼 운영이 SRE에게 추가하는 5가지 부담. 단순히 "여러 플랫폼 모니터링"이 아니라 운영 본질이 달라진다.

① 데이터 정합성 ×N

플랫폼마다 결제·인증·진행도 데이터가 분리되어 들어옴. 이걸 single source of truth로 통합하는 게 핵심. 한 곳이라도 어긋나면 사용자가 불일치를 발견.

② 빌드 동기화

같은 코드베이스에서 N개 빌드 — 플랫폼별 SDK·anti-cheat·UI 차이. 한 빌드의 회귀가 다른 빌드를 막을 수 있음.

③ 점검 윈도우 시차

모바일은 심사 일정에 묶이고 PC는 즉시 — 같은 시간에 점검을 시작·종료해도 사용자 영향 범위가 다름. 글로벌 region 시차까지 더해지면 점검 계획이 매우 복잡.

④ CS · 컴플라이언스

플랫폼 × 지역 매트릭스 — 각 셀마다 다른 정책·답변 시나리오. CS 인력·법무 자문이 모두 늘어남.

⑤ 인시던트 R&R

한 플랫폼에서 사고가 나면 그 플랫폼 IR + 다른 플랫폼 영향 평가 + 통합 시스템 영향의 3중 작업. 한 사고가 N개 플랫폼에 도미노로 번지는 사례 빈번.

● Account & Auth

통합 계정 & 인증

크로스 플랫폼의 출발점. 사용자가 모바일·PC·콘솔 어디서든 같은 계정으로 들어갈 수 있어야 cross save·cross progression이 성립한다. 플랫폼별 인증 메커니즘을 단일 게임 계정 ID로 매핑하는 게 핵심 설계.

플랫폼별 인증 메커니즘

각 플랫폼이 사용자를 인증하는 방식이 다르다. 게임은 모든 플랫폼의 ticket·token·OAuth를 받아 자체 시스템과 매핑해야 한다.

계정 매핑 모델

자체 게임 계정 ID와 N개 플랫폼 ID를 어떻게 매핑하느냐가 운영의 기반. 잘못 설계하면 cross progression이 영원히 안 됨.

모델 A — 1:1 (단순)

각 플랫폼 사용자가 별도 게임 계정. cross progression 불가. 운영은 단순하지만 사용자 만족도 ↓. 모바일 단독 게임이 PC로 확장할 때 임시로 쓰는 모델.

모델 B — 1:N (통합)

1개 게임 계정 ↔ N개 플랫폼 ID. 사용자가 플랫폼별로 로그인하면 그 게임 계정에 매핑. cross progression의 표준 모델. 운영 복잡도 ↑.

모델 C — 자체 ID 우선

사용자가 게임 자체 계정(이메일·비밀번호)으로 가입 + 플랫폼 ID는 보조 매핑. 가입 부담은 있지만 통제 가장 좋음. Riot ID, EA Account, Activision ID가 이 모델.

모델 D — 플랫폼 ID 우선

사용자가 플랫폼 ID로 자동 가입(별도 계정 X). 첫 플랫폼이 마스터 ID. 추가 플랫폼은 매핑. 가입 부담은 적지만 첫 플랫폼이 사라지면 복잡.

표준 1:N 매핑 스키마 (모델 B/C)
// game_account 테이블 (자체 계정)
{
  "game_account_id": "uuid-...",           // single source of truth
  "created_at": "2026-05-03",
  "primary_platform": "steam"            // 첫 가입 플랫폼
}

// platform_links 테이블 (플랫폼 매핑)
{
  "game_account_id": "uuid-...",
  "platform": "steam | apple | google | psn | xbox | nintendo",
  "platform_id": "<벤더 식별자>",
  "linked_at": "2026-05-04",
  "unlinked_at": null
}

계정 복구 · 이전 · 이중 가입 방지

크로스 플랫폼에서 계정 분실·이전·이중 가입은 가장 흔한 CS 사례. 표준 워크플로우 + 안전 장치가 필요하다.

본인 확인 다중 채널

(1) 이메일 (2) 휴대폰 (3) 결제 카드 (4) 디바이스 핑거프린트 (5) 게임 내 진행도 정보. 2개 이상 일치해야 복구 진행.

이중 가입 방지

한 플랫폼 ID가 N개 게임 계정에 매핑되면 안 됨. 플랫폼 ID는 unique 제약. 사용자가 실수로 두 번째 가입 시도하면 자동으로 기존 계정으로 안내.

플랫폼 ID 변경

드물지만 발생 — 사용자가 Apple ID 바꾸거나 PSN ID 변경. 옛 ID와 새 ID를 매핑해 같은 사용자임을 인식. Apple은 user identifier가 안 변하지만 일부 케이스에서 변경.

데이터 보존

한 플랫폼 ID 해제(unlink) 시 데이터를 즉시 삭제하지 말 것. 30일 grace period 두고 그 안에 사용자가 재연결 가능하게. GDPR 잊혀질 권리는 별도 처리.

Apple · Google 로그인 정책 (2026-05)

두 플랫폼은 자체 로그인 옵션을 강제하거나 권고하는 정책이 있다. 정책이 종종 바뀌므로 출시·갱신 시점에 재확인 필요.

Sign in with Apple — 가이드라인 4.8 (2024-01 완화)

외부 소셜 SSO를 1차로 쓰는 앱은 Apple 또는 "동등 옵션"(이메일·이름만 수집·private email 가능·추적 안 함)을 제공해야 함. Apple 강제 의무는 완화됨. 자체 계정·교육·기업·정부 ID 앱은 예외.

Google Play Games Services v2

v1에서 v2로 마이그레이션 마감 2026-05. 그 이후 v1 게임은 publish/update 불가. Google Sign-In이 PGS와 분리되면서 dependency 정리 필요.

이메일 private relay

Sign in with Apple은 random relay 이메일을 줌. 진짜 이메일이 아니라 Apple이 중계하는 alias. 게임이 이걸 보내는 메일은 사용자에게 잘 도달하지만, relay 이메일을 다른 플랫폼 인증과 매칭하지 말 것.

다중 SSO 운영

한 사용자가 LINE·Apple·Google 3개 SSO를 모두 가질 수 있음. 게임 계정 1개에 3개 SSO를 매핑하되 같은 SSO를 2번째 사용 시 자동 매핑. 새 게임 계정 생성하지 말 것.

한국 본인인증 통합

한국 게임은 본인인증(휴대폰·아이핀·금융인증)이 사실상 표준이다. 청소년이용불가·결제·게임시간 선택제 모두 본인인증 기반. 크로스 플랫폼 시점에는 한 번만 받아도 모든 플랫폼에 적용되어야 한다.

본인인증 옵션

· 휴대폰 인증 (가장 일반적, KISA·통신사 연동)
· 아이핀 (NICE·SCI 등)
· 금융인증 (PASS 앱·공동인증서)
· I-PIN (정부 통합 인증)

저장 데이터

본인인증 결과는 CI/DI (연계정보·중복가입확인정보)로 저장. 실명·주민번호는 즉시 폐기. CI는 같은 사람을 여러 사이트에서 식별 가능.

통합 적용

한 게임 계정에 본인인증 1회 → 모바일·PC 양쪽에서 같이 적용. 게임 계정 ID 단위로 본인인증 상태 관리. 단, 본인인증 유효기간(1년 등)이 지나면 재인증 필요.

자녀 결제 동의

청소년 결제는 법정대리인 동의 필요. 한 번 동의 → 모든 플랫폼 적용. 동의 취소는 모든 플랫폼에서 결제 차단.

★ NOTE
한국 본인인증은 외부 PG·인증사를 통한 별도 비용. 1건당 수십원 단위 — DAU가 크면 무시 못할 비용. 본인인증을 매번 묻지 말고 게임 계정 단위로 한 번만 받기가 비용·UX 모두 유리.

계정 통합 · 분리 운영

출시 후에 계정 통합·분리 요청이 종종 들어온다. 정책·도구를 미리 갖춰놓지 않으면 1건 처리에 GM 수 시간이 든다.

통합 (Merge)

실수로 두 계정 만든 사용자가 통합 요청. 두 계정의 진행도·재화·결제 이력을 한 계정으로 합침. 결제 합치는 게 가장 어려움 — 환불·세금·카드사 분쟁 위험.

분리 (Split)

한 계정을 두 사람이 공유하다가 분리 요청. 거의 들어주지 않음 — 사후 분리는 거래·진행도 추적 불가능. 약관에 "계정 1인 1개" 명시.

플랫폼 매핑 변경

한 플랫폼 ID를 다른 게임 계정에 옮김. 어뷰징 위험 ↑(작업장이 매번 새 계정으로 매핑) — 일정 cooldown + 매니저 승인이 표준.

사망·상속 처리

사용자가 사망한 경우 가족이 계정 인계 요청. 법적 자료(사망 진단서·가족 관계 증명) + 매니저 승인. 결제 이력은 환불 또는 보존.

재화 단위 통합

"보석 100개"는 어느 플랫폼에서 사도 100개. 가격은 다를 수 있어도 재화 양은 동일. 가격 차이를 사용자에게 노출할지는 정책 결정.

플랫폼 segregated 재화

일부 게임은 플랫폼별 분리 재화(예: "iOS 보석"·"Steam 보석"). 환불·정합이 단순하지만 사용자 불만 ↑. 콘솔 정책상 강제되는 경우 있음(Sony 정책 등).

가격 차이 처리

같은 "보석 100개"라도 Apple 30% / Steam 30~20% / 자체 결제 0%라 게임사가 받는 금액이 다르다. 사용자에게는 결제 가격이 다르고. 이 차이를 어떻게 보여주고 처리할지가 정책.

전략 1 — 동일 가격

모든 플랫폼에 같은 가격. 사용자에게 단순. 게임사는 자체 결제로 마진 ↑. 외부 결제 안내가 정책에 막혀있어 마진 차이를 사용자에게 잘 안 보임.

전략 2 — 자체 채널 보너스

자체 결제로 사면 "보석 +20% 보너스" 등 추가 혜택. 게임사 마진을 사용자와 분배해 자체 결제 유도. 합법적 + 사용자에게 매력적.

전략 3 — 자체 채널 저가

자체 결제 가격을 낮게 (예: $9.99 vs $7.99). 사용자가 자체 결제로 옮겨감. 모바일 스토어 정책 위반 위험(가격 차이로 IAP 회피 유도가 강하면).

region별 차이

한국·중국·러시아처럼 권장가가 낮은 region은 절대 가격이 다름. VPN 어뷰징·차익 거래 방지를 위한 region locking 검토.

환불 정합성

환불은 정합성 깨지기 가장 쉬운 지점. 사용자가 모바일에서 환불 받았는데 PC에서 이미 재화를 썼다면 어떻게 할지의 정책이 필요.

환불 처리 케이스

· 구매 직후 환불: 재화 미사용 → 그대로 회수, 단순
· 일부 사용 후 환불: 사용한 재화는 "마이너스"로 → 사용자 인벤토리가 음수가 될 위험
· 전부 사용 후 환불: 가장 어려움 — 게임 진행도가 이미 영향 받음

음수 인벤토리 방지

환불 시 인벤토리에서 차감해서 음수 → 다음 결제까지 차감 미루기 또는 다음 게임 플레이 시 재화 0으로 normalize. 두 옵션 모두 사용자 알림 필수.

환불 어뷰징

매번 결제 → 사용 → 환불의 반복 패턴. 환불 빈도·총액 임계 + 자동 차단. P3 결제 가이드와 P8 어뷰징 가이드에 깊이.

플랫폼별 환불 정책 차이

· Apple/Google: 스토어가 환불 결정, 게임사는 알림 받음
· Steam: 2시간/14일 자동, 게임사 동의 X
· 자체 결제: 게임사가 환불 정책 결정
각 플랫폼 환불을 통합 인벤토리에 반영하는 webhook이 필수.

미수령 · 중복 지급 방지

결제는 됐는데 재화가 안 들어갔다 / 한 번 결제에 두 번 들어갔다는 클레임이 CS의 큰 부분. 표준 패턴으로 자동 처리해야 한다.

Idempotency (멱등성)

같은 결제 ID를 두 번 받아도 한 번만 지급. 결제 ID를 unique key로 인벤토리 트랜잭션 테이블에 저장. 중복 지급 자체가 시스템적으로 불가능해야 함.

미수령 자동 복구

결제 영수증은 있는데 인벤토리 트랜잭션이 없으면 자동 복구. 게임 시작 시 미수령 영수증 자동 query. Apple StoreKit·Google Billing·Steam 모두 영수증 query API 제공.

네트워크 단절 처리

결제 직후 네트워크 단절 시 클라이언트는 결제 결과를 모름. 다음 접속 시 영수증 검증 → 재화 지급. 사용자가 다시 클릭해도 idempotency로 중복 X.

플랫폼 webhook

· Apple App Store Server Notification
· Google Real-time developer notifications
· Steam server-to-server callbacks
결제·환불·구독 변경 모두 webhook으로 받아 통합 인벤토리에 즉시 반영.

외부 결제 (Apple v Epic 이후)

2024-2026년 사이 모바일 외부 결제 정책이 크게 변동. 미국·EU·한국이 각자 다른 규제 체계라 region별 운영이 필요하다. (2026-05 기준)

미국 — 변동 중

2025-04 Epic v Apple 1심: 외부 결제 링크 허용 + Apple 수수료 X. 2025-12 항소법원 부분 reverse — Apple이 "합리적 수수료" 부과 가능. 정확한 비율은 1심 재검토 진행 중. 운영자는 모니터 필수.

EU — DMA 적용

2024-03 DMA 시행 후 외부 결제·alternative payment provider 허용. 2026-01부터 Core Technology Fee 폐지 + 5% commission으로 변경. 게임 스토어 진입 가능.

한국 — 통신사업법 개정

2021-09 통신사업법 개정 후 Apple/Google이 한국에서 사전 승인 PSP 사용 가능. StoreKit External Purchase Entitlement (KR) 신청 필요.

실무 운영

각 region별로 다른 결제 흐름·수수료를 코드에서 분기. "한국·EU·미국·일반"의 4분기 코드 path가 표준. 정책이 자주 바뀌므로 feature flag로 region별 토글.

⚠ NOTE
외부 결제는 매출 중요한 레버이지만 정책이 매월 바뀌는 영역. 법무·외부 자문·Apple/Google 공식 발표를 정기 모니터. P3 결제 가이드에 더 깊이.

EU DMA · 사이드로딩

EU Digital Markets Act는 Apple·Google에 대한 가장 큰 규제 변화. 2024-03 시행, 2026년 본격 영향. 사이드로딩·alternative app stores·외부 결제 모두 게임 운영에 영향.

"가장 느린 플랫폼이 기준"

크로스 플레이 게임은 모든 플랫폼이 같은 버전이어야 매칭 가능. 가장 느린 플랫폼(Switch·Apple)에 맞춰 출시일 정함. 다른 플랫폼은 그날까지 대기.

핫픽스 동기화

긴급 패치도 마찬가지. Steam에 즉시 배포해도 모바일은 심사 대기. 그 사이 Steam 사용자만 새 버전 → 매칭 깨짐. 핫픽스는 서버 사이드만 활용 권장.

역동기 (lag) 처리

한 플랫폼만 출시 지연되면 다른 플랫폼도 그 기간 동안 새 콘텐츠를 막아야 함(매칭 깨지기 때문). Server-side feature flag로 통합 제어.

예외 — 분리 운영

크로스 플레이가 없으면 각 플랫폼 자율 cadence 가능. 모바일은 매주, PC는 격주 식으로 분리 운영하는 게임도 있음.

단일 코드베이스 · 다중 빌드

같은 코드에서 플랫폼별 빌드가 나오는 게 표준 (Unity·Unreal 모두 다중 플랫폼 빌드 지원). 그러나 플랫폼별 SDK·anti-cheat·UI 차이가 빌드 분기점.

플랫폼별 SDK 분기

· iOS: StoreKit·GameKit·Apple Push
· Android: Play Billing·Play Games·FCM
· Steam: Steamworks SDK
· PSN/Xbox: 플랫폼 SDK
Conditional compile (#if PLATFORM_IOS)로 분기.

UI 차이

· 모바일: 터치 + 작은 화면
· PC: 마우스/키보드 + 큰 화면 + 윈도우 크기 가변
· 콘솔: 게임패드 + TV 뷰
UI 시스템이 플랫폼별 layout을 자동 선택해야 함.

Anti-cheat 분기

· 모바일: SDK + 디바이스 무결성
· Steam: VAC 또는 EAC/BattlEye
· 콘솔: 플랫폼 자체
각 빌드에 다른 anti-cheat 모듈을 통합. P9 가이드 참조.

CI/CD 파이프라인

한 git push → N개 빌드 자동 생성. 빌드 시간이 N배이지만 일관성·동기화가 중요. 전용 빌드 머신·캐시·병렬 처리 필수.

Feature Flag · Kill Switch

플랫폼별 cadence 갭과 동기 출시 요구를 동시에 만족시키는 핵심 도구. 모든 새 기능은 flag 뒤에 두는 게 표준.

Feature Flag 패턴

· Build에 코드는 포함되지만 server flag로 비활성
· 모든 플랫폼에 새 빌드 출시 후 server에서 동시 활성
· 한 플랫폼만 문제 시 그 플랫폼만 비활성

Kill Switch

장애 시 즉시 기능 끄기. 새 빌드 없이 server에서 분 단위 비활성. 결제·매칭·신규 콘텐츠 등에 모두 적용.

플랫폼별 분기 flag

같은 기능이라도 플랫폼별로 다른 timing. { "newFeature": { "ios": false, "android": true, "steam": true } } 같은 매트릭스. 서버에서 사용자 플랫폼별 다른 응답.

Region별 flag

한국 EU 미국에 다른 정책(외부 결제·확률 표시 등). region × platform 매트릭스로 flag 운영. 코드 복잡도 ↑이지만 컴플라이언스 안전.

호환성 매트릭스

크로스 플레이·cross save가 깨지지 않으려면 플랫폼·버전 간 호환성을 명확히 정의해야 한다.

저장 파일 호환성

같은 게임 계정의 저장 파일을 모든 플랫폼이 읽을 수 있어야 함. schema 버전 관리 필수. 새 schema는 backward-compatible 또는 자동 migration.

네트워크 프로토콜 호환

크로스 플레이는 모든 플랫폼이 같은 protocol. 버전 N+1을 출시할 때 N과 호환되어야 함. 한 플랫폼이 N에 멈춰 있어도 N+1 사용자와 매칭 가능.

Asset 호환

플랫폼별로 텍스처 압축 포맷이 다름 (ASTC·ETC2·BC7 등). 같은 콘텐츠를 다른 포맷으로 빌드. 게임 로직은 호환.

호환성 매트릭스 표

버전 × 플랫폼 매트릭스를 명시적으로 관리. "1.5.x ↔ 1.6.x 매칭 가능, 1.7.x는 1.6.x 매칭 차단" 식. 변경 시 모든 플랫폼 영향 사전 평가.

점검 윈도우 · 시차

크로스 플랫폼 게임의 점검은 단순 "한 시간 다운"이 아니다. 플랫폼별·region별 시차를 동시에 고려해야 한다.

점검 시작 정책

· 모든 플랫폼 동시 점검 — 정합성 보장
· 플랫폼별 순차 — 위험 분산
· Region별 순차 — 시차 활용
크로스 플레이 게임은 동시 점검이 표준(매칭 깨지지 않게).

시간대 선택

한국·일본 새벽 (KST 02:00~04:00)이 일반적. 그러나 북미·EU는 그 시간이 일과 시간. 글로벌 게임은 시간대 trade-off가 큼.

점검 중 사용자 메시지

각 플랫폼 클라이언트가 "점검 중" 화면 표시. 플랫폼별로 텍스트·UI가 약간 다를 수 있음. 점검 종료 시점·새 빌드 다운로드 안내 일관.

점검 종료 동시성

점검 끝나는 순간 모든 플랫폼이 동시 트래픽 폭발. P4 가이드의 prefetch + multi-CDN이 직결. 모바일·PC·콘솔 다운로드 트래픽이 한 시점에 몰림.

● Matching & Anti-cheat

매칭 & Anti-cheat

크로스 플레이의 가장 어려운 두 영역. 플랫폼 간 input 차이로 fairness가 깨지고, 플랫폼별 anti-cheat이 다르면 cheat 침투 경로가 늘어난다. 잘 설계하지 않으면 사용자가 떠난다.

크로스 플레이 매칭

같은 매치에 PC·모바일·콘솔 사용자가 섞이는 시점부터 매칭 알고리즘이 복잡해진다. 표준 패턴 4가지.

패턴 A — Full Cross

모든 플랫폼이 한 풀에서 매칭. 가장 단순·풀 사이즈 최대. input fairness가 깨지면 큰 사용자 불만. PvE·협력 게임에 적합.

패턴 B — Input-based 분리

마우스+키보드 vs 컨트롤러 vs 터치 별 풀 분리. Fortnite 모델. PC 사용자가 콘솔 컨트롤러 사용 시 컨트롤러 풀로. 옵트인·아웃 정책.

패턴 C — Platform 분리

모바일 ↔ PC ↔ 콘솔 별 풀. cross play 없는 모델. 각 풀이 작아지지만 fairness 보장.

패턴 D — Cross with opt-in

기본은 같은 플랫폼끼리, 사용자가 "친구와 같이 플레이" 옵션 시에만 cross. fairness 우선 + 친구 매칭 가능. PvP 게임에 흔함.

★ NOTE
Fortnite의 매칭 정책은 잘 알려진 사례: PC와 파티 → PC 풀 강제. 콘솔 사용자가 PC 친구와 매칭 시 PC 풀에 들어가서 어려운 상대를 만남. 사용자에게 이를 명확히 표시하는 게 중요.

Input Fairness — 마키 vs 패드 vs 터치

PC 마우스·키보드의 정밀도, 콘솔 패드의 aim assist, 모바일 터치의 한계 — 각각이 매우 다르다. PvP 게임에서 input fairness는 매출과 직결.

입력정밀도반응 속도aim assist표준 처리
PC 마우스+키보드최상최상없음가장 빠른 풀
콘솔 게임패드중상aim assist 강함aim assist 보정
모바일 터치낮음중강한 aim assist분리 풀 또는 강한 보정

Aim Assist 보정

콘솔·모바일은 자동 조준 강도를 게임이 조정. PC 풀에 들어가면 aim assist 강도를 PC 매칭 풀에 맞게 자동 조정. 단, "PC + aim assist"가 너무 강하면 PC 사용자 불만.

입력 감지

플레이어가 어떤 입력을 쓰는지 게임이 감지 — 마지막 입력 디바이스 기준. 콘솔에서 마우스 사용 → "PC급" 입력으로 인식(USB 허브 등으로 콘솔에 마우스 연결).

매칭 풀 동적 전환

입력 디바이스가 매치 중에 바뀌면 다음 매치부터 다른 풀로. 매치 도중 전환은 어뷰징 위험.

FPS 제한

모바일·콘솔은 30~60 fps, PC는 144+ fps 가능. 같은 매치에 fps 차이가 크면 PC가 유리. 게임 로직을 fps에 독립적으로 설계.

플랫폼별 anti-cheat 통합

크로스 플레이 게임은 모든 플랫폼에 anti-cheat이 있어야 한다. 한 곳이 약하면 cheater가 그 플랫폼으로 들어와 매치를 망친다.

플랫폼별 anti-cheat

· 모바일: 디바이스 무결성 SDK (root/jailbreak·앱 변조)
· PC: VAC·EAC·BattlEye·nProtect (P9 참조)
· 콘솔: 플랫폼 자체 anti-cheat (폐쇄 환경)
같은 게임이 모든 플랫폼에서 보호되어야 함.

Server-side 통합 검증

플랫폼별 client anti-cheat 외에 서버에서 통합 행동 분석. 비정상 통계·패턴은 플랫폼 무관하게 탐지. P9 anti-cheat 가이드의 server-side 검증 참조.

BAN 정책 통합

한 플랫폼에서 cheat 적발 → 모든 플랫폼에서 BAN(통합 게임 계정). 단, "이 플랫폼에서만" cheat한 사용자가 다른 플랫폼에서는 정상이라면 정책 신중. EU DSA 이의제기 권리 적용.

가장 약한 고리

크로스 플레이의 anti-cheat은 가장 약한 플랫폼만큼만 강함. 모바일이 약하면 모바일 cheater가 PC를 이김. 모든 플랫폼 anti-cheat을 평행하게 강화.

콘솔 폐쇄 vs PC 개방 환경

콘솔은 OS·하드웨어가 통제된 폐쇄 환경, PC는 열린 환경. anti-cheat·정책·운영 패턴이 근본적으로 다르다.

콘솔 — 폐쇄 환경의 장점

· OS·드라이버·메모리 전부 통제됨
· 외부 프로그램 거의 실행 불가
· cheat 만들기 매우 어려움
· PC 대비 cheat 비율 1/100 수준

PC — 개방 환경의 부담

· 누구나 메모리 접근·DLL 인젝션 가능
· 가상 머신·hypervisor cheat
· DMA cheat (외부 디바이스로 메모리 읽기)
· anti-cheat이 끝없는 군비경쟁

크로스 플레이 부작용

콘솔 사용자가 "PC cheater 때문에 매치 망쳤다"고 분노. 일부 게임은 콘솔 사용자에게 "cross play OFF" 옵션 제공.

Steam Deck

Linux 기반 PC지만 콘솔에 가까운 폐쇄성. anti-cheat 호환성이 가장 큰 변수(P9 참조). EAC·BattlEye Proton 지원으로 점차 호환성 ↑.

친구 · 초대 cross-platform

크로스 플레이의 큰 동력은 친구. 모바일 사용자가 PC 친구를 초대하고, 콘솔 사용자가 모두에게 보이는 친구 시스템이 retention의 핵심.

게임 자체 친구 시스템

플랫폼별 친구(Steam Friends·PSN Friends)와 별도로 게임 계정 단위 친구 시스템. game_account_id 기반. 플랫폼 무관하게 친구.

플랫폼 친구 import

처음 가입 시 플랫폼 친구 중 같은 게임 사용자 자동 추천. "PC 친구가 모바일에서도 게임 중"임을 알려줌. retention 큰 효과.

Cross-platform invite

친구 초대는 게임 계정 단위로 가능해야 함. 모바일에서 보낸 초대를 PC 친구가 받음. 플랫폼 SDK invite + 게임 자체 invite를 모두 지원.

친구 onboarding 마찰

친구 추가가 너무 복잡하면 사용자 떠남. 이메일·QR·짧은 코드로 빠른 추가 옵션. 콘솔 정책이 외부 친구 추가 막는 경우는 우회 어려움.

● Operations Integration

운영 통합

크로스 플랫폼의 마지막 도전 — CS·정지·이의제기·KPI·컴플라이언스·사고 대응을 통합 운영. 잘못 분리하면 한 사용자가 두 번 답하고, 한 사고가 N번 처리된다.

CS 권한 분리

사용자가 어느 플랫폼에서 문의를 하든 응대해야 한다. 그러나 환불·계정 권한은 플랫폼별로 다름.

CS 응대 흐름

(1) 어느 플랫폼에서 문의 → 통합 티켓 시스템
(2) 게임 계정 ID 확인 (모든 플랫폼 ID 매핑된)
(3) 이슈 유형별로 권한 라우팅
(4) 결제는 플랫폼별 환불 흐름

"플랫폼 vs 게임사" 분리

사용자에게 명확히 안내 — "환불은 Apple App Store에 신청하세요". 게임사가 직접 환불 못하는 케이스 多. 단, 게임 내 보상은 게임사가 처리 가능.

크로스 플랫폼 CS 일관성

같은 이슈가 모바일·PC에서 다르게 처리되면 사용자 분노. 표준 답변·보상·정책을 플랫폼 무관하게 통일. P8 CS 가이드 참조.

티켓 시스템 통합

모바일 인앱 SDK·PC 웹폼·콘솔 메시지가 모두 한 시스템(Zendesk 등)으로 들어옴. 플랫폼별로 다른 GM 큐는 비추 — 사용자별로 통합.

정지 · 이의제기

한 사용자가 cheat·어뷰징으로 적발됐을 때 모든 플랫폼에서 정지할지, 한 플랫폼만 정지할지가 정책.

통합 정지 (표준)

게임 계정 단위 BAN → 모든 플랫폼 ID에서 정지. 새 플랫폼으로 우회 못함. 단, EU DSA 이의제기 권리 보장 필요.

플랫폼 BAN 별도 처리

· VAC BAN(Steam) — Valve 자동 BAN, 게임사 통제 X
· Apple/Google 계정 정지 — 스토어 정책
· 게임사 BAN과 별개 — 한 곳 BAN되어도 다른 곳은 정상

이의제기 — 통합 워크플로우

사용자가 어느 플랫폼에서 이의제기해도 같은 시스템. 다른 GM이 재검토 → 결정 → 통보. P8 어뷰징·CS 가이드 참조.

인벤토리 처리

BAN 시 인벤토리는? 일반적으로 회수 (모든 플랫폼에서). 단, 결제 아이템은 환불 또는 다른 계정 이전 옵션이 컴플라이언스에 안전. 사망·상속 케이스는 별도.

KPI · 텔레메트리 통합

"전체 DAU 100만"보다 "모바일 70만 + PC 25만 + 콘솔 5만"이 훨씬 유용. 그러나 "한 사용자가 모바일·PC 모두 쓰면 중복 계산?" 같은 정의 문제가 있다.

플랫폼별 vs 게임 단위

· 플랫폼별 DAU: 운영 KPI·플랫폼 협상에 사용
· 게임 단위 DAU: game_account_id 기준, 중복 제거. 진짜 사용자 수
두 지표를 모두 추적해야 함.

매출 분리·통합

플랫폼별 매출 (Apple·Google·Steam·자체)을 분리해서 보고. 그 위에 게임 단위 ARPU도 계산. 한 사용자가 모바일+PC에서 결제하면 두 매출 모두 그 사용자에게 귀속.

크로스 플랫폼 LTV

한 사용자의 lifetime value를 모든 플랫폼 합산. "PC에서 무료로 시작 → 모바일에서 결제" 패턴이 있어 분리 분석으로는 missed.

이벤트 스키마 통합

모든 플랫폼이 같은 텔레메트리 스키마로 이벤트 보냄. "플랫폼 = 차원"으로 한 이벤트 안에 포함. P6 텔레메트리 가이드 참조.

컴플라이언스 매트릭스 (region × platform)

크로스 플랫폼의 컴플라이언스는 (region 수) × (플랫폼 수) 셀의 매트릭스. 각 셀에 다른 정책이 적용된다.

매트릭스 운영

각 셀별로 (1) 적용 법령 (2) 시스템 요구사항 (3) UI 차이를 명시. 출시 전 체크리스트로 모든 셀 검증. 셀이 늘면 빠진 곳이 생기기 쉬움.

변동 모니터

법령은 자주 바뀜 — EU AI Act 2026-08, 한국 게임시간 선택제 폐지 가능성, 미국 외부 결제 항소심 등. 분기별 컴플라이언스 리뷰로 매트릭스 갱신.

플랫폼 전이 위험

iOS 정책 변화는 Steam 운영에도 영향 (예: 동일 가격 vs 차등 가격 정책 결정). 한 플랫폼 변경이 다른 플랫폼 운영에 미치는 영향 사전 분석.

법무 자문 분리

한국·EU·미국·일본·중국 각각 현지 로펌 1곳 이상. 한 곳에서 모든 region을 보면 누락 위험. 분기 정기 미팅으로 각 region 변동 점검.

사고 대응 R&R

한 플랫폼에서 사고가 나도 다른 플랫폼·통합 인벤토리에 영향. 사고 대응이 한 플랫폼에 그치지 않는다.

통합 IR 트리거

(1) 어느 플랫폼에서 알림 → 즉시 통합 채널 전파
(2) 다른 플랫폼 영향 평가
(3) 공통 시스템(통합 인벤토리·인증) 영향 평가
(4) 필요 시 모든 플랫폼 점검 윈도우 동시 시작

플랫폼 간 도미노

한 플랫폼 결제 시스템 장애가 통합 인벤토리에 부정합 → 다른 플랫폼 사용자도 영향. "한 플랫폼 사고"가 곧 "전 플랫폼 사고".

R&R 명확화

· 모바일 SDK 장애 → 모바일 SRE + 플랫폼 SDK팀
· Steam 장애 → PC 운영팀 + Valve 협의
· 통합 인벤토리 장애 → 백엔드팀 (가장 자주)
· 플랫폼 정책 변경 → 컴플라이언스팀
매트릭스를 사고 대응 매뉴얼에 명시.

외부 IR 조정

외부 IP 게임은 개발사 + 퍼블리셔 공동 IR. 플랫폼 IR도 추가 — Apple·Google·Valve와도 협의가 필요한 큰 사고 시. 시차·언어가 IR 부담을 키움.

♥ KEY
크로스 플랫폼 게임의 IR은 "한 사고 = N개 IR 동시 진행"이 자주 발생. 작은 사고도 도미노로 커지므로 첫 5분 안에 영향 범위를 정확히 평가하는 게 핵심. 각 플랫폼 SRE 온콜이 통합 채널에서 함께 일하는 게 표준.
