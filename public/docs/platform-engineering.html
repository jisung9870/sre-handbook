Platform Engineering이란

개발자가 인프라를 몰라도 제품을 빠르게 배포할 수 있게 만드는 — DevOps/SRE의 다음 진화.

Definition

Platform Engineering 정의

"개발 팀의 인지 부하를 줄이고 셀프서비스를 제공하는 내부 플랫폼을 설계·운영하는 분야"
DevOps가 "개발자도 운영을 해야 한다(You build it, you run it)"를 주장했지만, 현실에서 개발자에게 K8s, Terraform, 모니터링, 보안을 모두 알아야 한다고 강요하는 것은 인지 부하(Cognitive Load) 과부하를 일으킨다.

Platform Engineering은 복잡성을 추상화한 Self-Service 플랫폼을 제공하여, 개발자가 인프라 세부사항 없이도 안전하고 빠르게 배포할 수 있게 한다. "도로를 포장하되 울타리를 치지 마라(Pave the road, don't build fences)"가 핵심 철학.
핵심 가치
1) 인지 부하 감소: 개발자가 K8s YAML을 몰라도 배포 가능
2) Self-Service: 티켓/요청 없이 개발자가 직접 인프라 프로비저닝
3) Golden Path: 모범 사례를 기본으로 제공하되 강제하지 않음
4) 추상화: 복잡성은 플랫폼이 흡수, 개발자에게는 단순한 인터페이스
5) 프로덕트 사고: 내부 플랫폼을 "제품"으로 다루고, 개발자를 "고객"으로 대응
실무 예시
Before (티켓 기반):
"DB가 필요합니다" → DevOps에 Jira 티켓 → 3일 대기 → DevOps가 Terraform 실행 → 개발자에게 연결 정보 전달

After (Self-Service):
개발자가 플랫폼 UI에서 "PostgreSQL 생성" 클릭 → 서비스 이름, 크기, 환경 선택 → 5분 후 자동 프로비저닝 + 연결 정보 자동 주입

Solo DevOps 관점: 자동화가 곧 Self-Service. "나 대신 자동화가 응답"하는 것이 Solo 환경에서의 Platform Engineering

Maturity

플랫폼 성숙도 모델

현재 어디에 있고, 다음 단계는 무엇인가
5단계 성숙도
Level 0 — Ad-hoc: 자동화 없음. 모든 것을 수동으로, 담당자에게 요청. 티켓 기반 운영

Level 1 — Scripts: 반복 작업을 셸 스크립트/Ansible로 자동화. 하지만 여전히 "스크립트를 아는 사람"이 실행. binbox CLI collection이 이 수준

Level 2 — Self-Service Basics: Terraform/ArgoCD로 선언적 인프라. GitOps 기반 배포. 개발자가 PR로 변경 요청 가능. 현재 환경의 대부분이 여기

Level 3 — Internal Platform: 통합된 포탈/CLI에서 서비스 생성, 환경 프로비저닝, 모니터링 조회가 가능. 템플릿 기반. OpsGate가 이 방향

Level 4 — Product Platform: 플랫폼이 제품으로 운영됨. 내부 SLO, 로드맵, 사용자 피드백, 지속적 개선. 대규모 조직의 목표
● Internal Developer Platform

IDP 설계

개발자에게 인프라를 Self-Service로 제공하는 내부 플랫폼의 설계와 도구.

Architecture

IDP 아키텍처 설계

IDP = 개발자 포탈(UI) + 추상화 계층(API) + 인프라 오케스트레이션(실행)
IDP는 단일 도구가 아니라 여러 도구의 조합이다. 중요한 것은 도구 선택이 아니라 개발자에게 제공하는 인터페이스의 단순성이다.
3계층 구조
Layer 1 — Developer Portal (UI):
서비스 카탈로그, 문서, 대시보드, Self-Service UI
도구: Backstage, Port, Cortex

Layer 2 — Abstraction (API):
개발자의 "서비스 생성" 요청을 인프라 리소스로 번역
도구: Crossplane, Terraform, ArgoCD ApplicationSet

Layer 3 — Infrastructure (실행):
실제 리소스 프로비저닝
도구: Terraform, Helm, K8s Operator, 클라우드 API
Solo DevOps의 IDP
풀 스택 IDP는 부담. 소규모 환경에서는 이미 가지고 있는 도구를 IDP처럼 조합:

포탈: Git 리포의 README + Grafana 대시보드
추상화: Helm 차트 + ArgoCD App-of-Apps
Self-Service: PR 기반 변경 (개발자가 values.yaml 수정 → PR → ArgoCD Sync)
자동화: Jenkins Shared Library + OpsGate CLI

핵심: Backstage를 도입하지 않아도 PR 기반 Self-Service + 잘 설계된 Helm 차트만으로 Level 2 플랫폼이 가능
Portal

Backstage (Spotify)

CNCF 프로젝트. 서비스 카탈로그, 템플릿, 플러그인 기반 개발자 포탈
Backstage는 서비스 카탈로그(누가 뭘 소유하는가) + 소프트웨어 템플릿(새 서비스 생성) + TechDocs(문서 통합) + 플러그인(K8s, CI/CD, 모니터링 연동)을 하나의 UI에 통합한 개발자 포탈이다.
핵심 기능
Software Catalog: 모든 서비스/라이브러리/인프라를 등록. 소유자, 의존성, SLO, 문서 링크를 한곳에
catalog-info.yaml을 각 리포에 추가하면 자동 등록

Software Templates: 새 서비스를 템플릿에서 생성. 리포 생성 + CI 설정 + Helm 차트 + ArgoCD 등록을 한 번에

TechDocs: MkDocs 기반. 코드 옆에 문서를 두고 Backstage에서 통합 조회

Plugins: K8s 상태, Grafana 대시보드, PagerDuty 온콜, ArgoCD Sync 상태, GitHub Actions 등을 플러그인으로 포탈에 통합
도입 판단
도입 추천:
• 서비스 15개+, 개발팀 3팀+
• "이 서비스는 누가 담당이지?" 질문이 자주 나옴
• 새 서비스 온보딩에 1주일+
• 문서가 Confluence, Wiki, README에 분산

도입 보류:
• 서비스 10개 미만, 1~2팀
• 팀원이 모든 서비스를 알고 있음
• Backstage 운영 자체가 추가 부담

대안: Git 리포의 잘 구조화된 README + Grafana 대시보드 + ArgoCD UI로도 카탈로그 역할 가능. "Backstage 없는 Platform Engineering"은 충분히 가능
Self-Service

Self-Service 인프라 패턴

개발자가 직접 인프라를 프로비저닝하되, 가드레일 안에서 안전하게
PR 기반 Self-Service (가장 실용적)
개발자가 GitOps 리포에 PR을 올려 인프라를 요청:

1) 개발자가 apps/my-new-service.yaml 추가 (템플릿에서 복사)
2) values 파일에서 도메인, replica, 리소스 설정
3) PR 생성 → CI에서 자동 검증 (lint, kubeconform, policy check)
4) DevOps/SRE가 리뷰 → Approve
5) Merge → ArgoCD가 자동 배포

장점: 추가 도구 불필요. Git이 감사 로그. 리뷰로 가드레일
가드레일: OPA/Kyverno가 PR에서 정책 위반 자동 감지
Crossplane (고급)
K8s CRD로 클라우드 리소스를 선언적으로 관리:

개발자가 Database CRD를 생성하면 Crossplane이 실제 RDS를 프로비저닝:
kind: Database
spec: { engine: postgresql, size: small, environment: prod }

→ Crossplane Composition이 이를 aws_db_instance + SecurityGroup + Secret으로 변환

장점: K8s 네이티브. Terraform 없이 클라우드 리소스 관리
복잡도: Composition 설계가 까다로움. Provider 유지보수 부담
적합: 이미 K8s 중심 환경이고, Terraform을 K8s에 통합하고 싶을 때
● Developer Experience

Developer Experience (DX)

Golden Path, 서비스 템플릿, DX 메트릭 — 개발자가 "인프라 때문에 막히지 않는" 환경.

Practice

Golden Path — 포장된 길

모범 사례를 기본값으로 제공하되, 벗어나는 것도 허용하는 "추천 경로"
"Golden Path ≠ Golden Gate(강제)". 이 차이가 핵심이다. Golden Path는 "이렇게 하면 가장 쉽고 안전해요"라는 추천 경로이지, "이렇게 안 하면 안 돼요"가 아니다. 개발자가 필요하면 벗어날 수 있되, 벗어나는 비용을 스스로 감수하는 구조.
Golden Path 예시
새 Go 서비스 생성:
1. 서비스 템플릿에서 리포 생성 → 표준 디렉토리 구조, Dockerfile, Helm 차트, CI 파이프라인이 이미 포함
2. make dev → 로컬 개발 환경 자동 구성
3. PR 생성 → 자동 lint, test, 이미지 빌드
4. Merge → ArgoCD가 staging에 자동 배포
5. Canary 검증 → production 승격

개발자는 "코드 작성 → PR → 배포됨"만 경험. 나머지는 Golden Path가 처리
Golden Path의 범위
언어/프레임워크: Go (표준) → 템플릿, CI 파이프라인, 모니터링이 자동
데이터 저장소: PostgreSQL (표준) → PgBouncer, 모니터링, 백업이 포함
메시징: Kafka (표준) → 파티션 설계, 모니터링 가이드 포함
모니터링: Prometheus 메트릭 노출 (표준) → Grafana 대시보드 자동 생성
배포: Helm + ArgoCD (표준) → 표준 차트 구조

벗어나야 할 때: Python ML 서비스, 특수한 프로토콜 → 기본 템플릿에서 확장하되 추가 설정은 개발팀이 책임
Scaffold

서비스 템플릿 설계

새 서비스를 5분 만에 시작할 수 있는 스캐폴딩
# 서비스 템플릿 구조 (Go API 서버) template-go-api/ ├── cmd/ │ └── server/ │ └── main.go # 서버 진입점 ├── internal/ │ ├── handler/ # HTTP 핸들러 │ ├── service/ # 비즈니스 로직 │ └── repository/ # 데이터 접근 ├── deploy/ │ ├── helm/ # Helm 차트 (표준 구조) │ │ ├── Chart.yaml │ │ ├── values.yaml # 기본값 (라이트사이징됨) │ │ └── templates/ │ ├── argocd/ │ │ └── application.yaml # ArgoCD Application │ └── k8s/ │ └── networkpolicy.yaml # 기본 NetworkPolicy ├── Dockerfile # Multi-stage, distroless ├── Makefile # dev, test, build, deploy ├── Jenkinsfile # CI 파이프라인 (Shared Library) ├── .golangci.yml # 린터 설정 ├── monitoring/ │ ├── dashboards/ │ │ └── service.json # Grafana 대시보드 │ ├── alerts/ │ │ └── slo-alerts.yaml # SLO Burn Rate 알림 │ └── runbook.md # 기본 Runbook └── docs/ ├── README.md # 서비스 개요 └── API.md # API 문서
💡 템플릿에 기본 포함되어야 할 것
Observability: Prometheus 메트릭 노출, 구조화 로그, trace context 전파 — 코드에 이미 포함
Security: non-root Dockerfile, capabilities drop ALL, readOnlyRootFilesystem — Helm 기본값
Resilience: health check 엔드포인트, graceful shutdown, PDB — 코드 + Helm
CI/CD: Jenkinsfile, ArgoCD Application — 복사해서 바로 사용 가능
Measurement

DX 메트릭 — 플랫폼의 성공을 어떻게 측정하는가

개발자 생산성과 만족도를 정량적으로 측정하는 지표
핵심 DX 메트릭
온보딩 시간: 새 개발자가 첫 PR을 머지하기까지 걸리는 시간. 목표: 1일 이내
서비스 생성 시간: 새 서비스를 생성~프로덕션 배포까지. 목표: 1시간 이내
배포 리드타임: 코드 커밋 → 프로덕션 배포. DORA Lead Time for Changes
인프라 티켓 수: 개발자가 DevOps에 요청하는 티켓 수. 감소 추세가 목표
Self-Service 비율: 전체 인프라 변경 중 개발자가 직접 수행한 비율
개발자 만족도: 분기별 설문. "인프라 때문에 업무가 막힌 적이 있는가?"
SPACE 프레임워크
Google/GitHub 연구 기반 개발자 생산성 프레임워크:

Satisfaction: 개발자 만족도 설문
Performance: 코드 품질, 서비스 안정성 (SLO)
Activity: 커밋 수, PR 수, 배포 빈도 (주의: 단독 사용 금지)
Communication: 코드 리뷰 응답 시간, 문서 활용도
Efficiency: 빌드 시간, CI 대기 시간, 환경 프로비저닝 시간

주의: Activity 메트릭(커밋 수 등)만으로 생산성을 측정하면 Goodhart's Law("측정 대상이 된 지표는 좋은 지표가 되지 못한다")에 빠진다. 여러 차원을 함께 봐야 한다
● Toil Automation

Toil 측정과 자동화

반복적이고 수동적이며 자동화 가능한 운영 작업을 체계적으로 제거하는 방법론.

Definition

Toil 정의와 측정

Google SRE의 Toil 정의 — "수동적, 반복적, 자동화 가능, 전술적, 서비스 성장에 비례"
모든 운영 작업이 Toil은 아니다. Toil의 5가지 특성을 모두 만족해야 한다. 전략적 판단이 필요한 작업(아키텍처 설계, SLO 리뷰)은 Toil이 아니다. Google SRE는 엔지니어 업무의 50% 이상이 Toil이면 위험하다고 본다.
Toil의 5가지 특성
1) 수동적(Manual): 사람이 직접 실행해야 함
2) 반복적(Repetitive): 같은 작업을 여러 번 수행
3) 자동화 가능(Automatable): 기계가 대신할 수 있음
4) 전술적(Tactical): 전략적 가치 없음. 현상 유지
5) 서비스 성장에 비례(O(n) with service growth): 서비스가 늘면 작업도 비례 증가

Toil 예시: 인증서 수동 갱신, 수동 스케일링, 로그 파일 수동 정리, 반복적인 배포 절차, 수동 DR 테스트
측정 방법
시간 추적: 1~2주간 업무 분류 기록
• Engineering (프로젝트, 설계, 코딩): __시간
• Toil (반복 운영 작업): __시간
• Overhead (미팅, 이메일): __시간

Toil 비율: Toil 시간 ÷ 총 업무 시간
• < 30%: 건강
• 30~50%: 주의. 자동화 투자 필요
• > 50%: 위험. 개선 작업이 불가능한 수준

Solo DevOps 현실: Toil 비율이 높기 쉬움. 자동화에 투자할 시간이 Toil에 잡아먹히는 악순환. 의식적으로 20%의 시간을 자동화에 투자해야 탈출
Prioritization

자동화 우선순위 결정

모든 Toil을 한 번에 없앨 수 없다 — 투자 대비 효과가 가장 큰 것부터
우선순위 매트릭스
우선순위 = 빈도 × 소요 시간 × 위험도 ÷ 자동화 난이도 ① 주 5회 × 30분 × 위험 높음 ÷ 쉬움 = 최우선 ② 월 1회 × 2시간 × 위험 낮음 ÷ 어려움 = 후순위

빈도: 하루에 여러 번? 주 1회? 월 1회?
소요 시간: 한 번 실행에 얼마나 걸리는가?
위험도: 수동 실행 시 실수 확률과 영향
자동화 난이도: 구현에 얼마나 걸리는가?
Quick Wins 목록
1시간 이내 자동화 가능 + 높은 빈도:
• 인증서 갱신 → cert-manager (이미 완료)
• 로그 파일 정리 → logrotate / Loki retention
• DNS 레코드 추가 → ExternalDNS
• 미사용 리소스 알림 → CronJob + kubectl

1~2일 작업:
• 새 서비스 온보딩 → 서비스 템플릿
• DB 백업 검증 → CronJob 자동화
• Runbook 실행 → 반자동 스크립트 (Level 1~2)

1~2주 프로젝트:
• Self-Service 인프라 프로비저닝
• 통합 운영 대시보드 (OpsGate 확장)
• HolmesGPT 인시던트 트리아지 자동화
Platform

운영 플랫폼 설계 (OpsGate 방향)

반복적인 운영 작업을 통합하는 내부 운영 플랫폼 — Go로 구축
OpsGate 같은 운영 플랫폼은 Toil을 코드로 옮기는 허브다. 개별 스크립트가 흩어져 있는 대신, 하나의 도구에서 운영 작업을 실행하고 기록한다.
운영 플랫폼 핵심 기능
1) 작업 실행: kubectl, terraform, helm 작업을 래핑한 안전한 CLI/API
2) 감사 로그: 누가, 언제, 무엇을 실행했는지 기록
3) 권한 제어: RBAC으로 실행 가능한 작업 제한
4) Runbook 연동: 알림 → 자동으로 관련 Runbook 작업 제안
5) 실행 결과 알림: Slack으로 실행 결과 자동 게시
설계 원칙 (Hexagonal Architecture)
Core Domain: 작업 정의, 실행, 기록 로직
Ports (인터페이스):
• CLI (cobra) — 터미널에서 직접 실행
• HTTP API (gin/echo) — 웹 UI에서 실행
• Webhook — Grafana Alerting에서 자동 트리거

Adapters (구현):
• K8s client-go — kubectl 작업
• AWS SDK — 클라우드 리소스 조작
• Slack API — 알림 전송
• PostgreSQL — 감사 로그 저장

OpsGate의 hexagonal architecture 설계가 이 구조와 일치
● Solo DevOps → Scale

Solo DevOps에서 팀으로 확장

혼자서 모든 것을 하는 환경에서 조직이 성장할 때 어떻게 전환하는가.

Strategy

Solo DevOps 생존 전략

1인이 전체 인프라를 운영할 때 번아웃 없이 가치를 극대화하는 전략
우선순위 원칙
1) 자동화 > 수동 대응: 같은 작업을 3번 하면 자동화. "지금 30분 vs 자동화에 2시간 → 앞으로 매주 30분 절약"이면 자동화가 이김

2) 문서화 > 속도: 혼자만 아는 지식 = 버스 팩터 1. Runbook, 아키텍처 문서, 의사결정 기록(ADR). 누군가 합류했을 때 인수인계 시간을 최소화

3) 구매 > 구축: 직접 만들 수 있어도 관리형 서비스가 TCO에서 유리한 경우 많음. Solo에서 Kafka를 직접 운영하는 것보다 MSK/Confluent Cloud가 낫다

4) SLO 기반 우선순위: SLO에 영향이 없는 "개선"은 후순위. Error Budget이 충분하면 안정화보다 자동화에 투자
시간 배분
이상적 배분:
• 50%: 엔지니어링 (자동화, 도구 개선, 아키텍처)
• 25%: 운영 (온콜 대응, 배포 지원, 요청 처리)
• 15%: 학습 (SRE 학습, 새 기술 평가)
• 10%: 문서화 (Runbook, ADR, 위키)

현실: 운영이 50%+ 차지하는 경우가 많음
탈출법: 매주 금요일 오후를 "자동화 시간"으로 고정. 이 시간에는 운영 요청을 받지 않음 (Slack 상태 변경). 이 습관이 장기적으로 Toil 비율을 줄임
Growth

팀 성장 로드맵

Solo → 2~3인 → Platform Team — 조직이 성장할 때의 전환 전략
Phase 1: Solo (현재)
모든 것을 혼자 담당.
• 핵심: 자동화로 Toil 최소화
• 위험: 버스 팩터 1, 번아웃
• 대비: 문서화, IaC, GitOps로 "나 없이도 돌아가는" 시스템
• 채용 어필: "이 정도 규모를 혼자 운영할 수 있는 자동화 체계를 만들었다"
Phase 2~3: 팀 성장
Phase 2 — 2~3인 팀:
• 역할 분리 시작: 인프라/앱 배포 또는 EKS/NHN Cloud
• 온콜 로테이션 가능 (번아웃 방지)
• 코드 리뷰 상호 수행
• 티켓 기반 → PR 기반 Self-Service 전환

Phase 3 — Platform Team (5인+):
• Platform Engineer + SRE + DevOps 역할 분화
• IDP 도입 (Backstage 등)
• 개발팀에 인프라 권한 위임 (Self-Service)
• 플랫폼 SLO 설정 ("배포 파이프라인의 SLO")
• 개발팀을 고객으로, 플랫폼을 제품으로
Mindset

Platform as a Product

내부 플랫폼을 "제품"으로 다루는 사고방식 — 고객은 개발자, 제품은 플랫폼
프로덕트 사고 적용
고객 인터뷰: 개발자에게 "가장 불편한 점"을 정기적으로 물어보기. 설문 또는 1:1
로드맵: 플랫폼 개선사항을 분기별 로드맵으로 관리. 개발팀에 공유
릴리스 노트: 플랫폼 변경사항을 공지. "이번 주 변경: HPA 기본 타겟을 70%로 조정"
독싱(Dogfooding): 플랫폼 팀도 자기 플랫폼을 사용. "먹어보지 않은 음식을 팔지 마라"
피드백 루프: Slack 채널 #platform-feedback. 개발자의 불만이 개선 백로그가 됨
플랫폼 SLO
플랫폼 자체에도 SLO를 설정:

CI 파이프라인 SLO:
• 빌드 시간 P95 < 10분
• 파이프라인 성공률 > 95%

배포 SLO:
• Merge → Production 배포 < 30분
• 배포 성공률 > 99%

인프라 프로비저닝 SLO:
• 요청 → 완료 < 1시간 (Self-Service)
• 요청 → 완료 < 1일 (티켓 기반)

모니터링: DORA 메트릭과 동일한 프레임워크로 플랫폼 성과 추적
💡 Solo DevOps에서 Platform Engineer로의 진화
현재 하는 일(자동화, IaC, GitOps, 모니터링)은 이미 Platform Engineering의 핵심이다. 차이는 관점의 전환: "인프라를 관리한다" → "개발자에게 플랫폼을 제공한다". 이 관점 전환이 Resume에서도 차별화 포인트가 되고, SRE/Platform Engineer 포지션에서도 핵심 역량으로 평가받는다.
