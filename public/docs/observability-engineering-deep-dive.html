Architecture

OpenTelemetry 아키텍처

CNCF의 관측 데이터 수집 표준 — Metrics, Logs, Traces를 하나의 프레임워크로
OTel은 계측과 수집·전달 계층을 담당한다. 저장/시각화/알림은 백엔드(Prometheus, Loki, Tempo 등)의 역할이다. 핵심 가치는 벤더 종속 없이 관측 데이터를 생성·수집·전달할 수 있다는 데 있다.

구성 요소: API(인터페이스 정의) → SDK(구현체, 앱에 내장) → Collector(중간 수집·변환·전달) → 백엔드(저장·분석).
데이터 파이프라인
Application (OTel SDK / Auto-instrumentation) → OTLP (gRPC/HTTP) → Alloy / OTel Collector → Receivers → Processors → Exporters → Prometheus (Metrics) → Loki (Logs) → Tempo (Traces)
OTLP (OpenTelemetry Protocol)
OTel의 표준 전송 프로토콜
gRPC: 고성능. 기본 포트 4317
HTTP/protobuf: 프록시 친화적. 기본 포트 4318
Metrics, Logs, Traces 모두 동일 프로토콜로 전송
압축, 배치 전송, 재시도 내장
OTel Semantic Conventions
라벨/속성 이름을 표준화하는 규약. 모든 OTel 구현체가 동일한 이름을 사용하면 대시보드와 알림을 벤더 간에 재사용할 수 있다.
예시: http.request.method, http.response.status_code, server.address, service.name, deployment.environment
K8s 리소스 속성: k8s.pod.name, k8s.namespace.name, k8s.node.name
Instrumentation

SDK & Auto-instrumentation

앱에 계측 코드를 삽입하는 두 가지 방법 — 수동(SDK) vs 자동(Agent)
수동 (SDK)
Go: go.opentelemetry.io/otel 패키지 직접 사용
Span 생성, 속성 추가, 에러 기록을 코드로 명시
장점: 비즈니스 로직에 맞는 정교한 계측
단점: 코드 변경 필요, 유지보수 비용

Go에서의 최소 설정:
TracerProvider + OTLP Exporter 초기화
HTTP middleware로 자동 Span 생성
gRPC interceptor로 RPC 추적
자동 (Auto-instrumentation)
Java: javaagent로 바이트코드 조작. 코드 변경 없음
Python: opentelemetry-instrument CLI
Node.js: @opentelemetry/auto-instrumentations-node
K8s: OTel Operator의 Instrumentation CRD로 Pod에 자동 주입

장점: 코드 수정 없이 HTTP, DB, gRPC 자동 추적
단점: 비즈니스 로직 추적 불가, 오버헤드

Go는 자동 계측 제한적 (컴파일 언어 특성). SDK 직접 사용 권장
Collector

Grafana Alloy

OTel Collector + Prometheus Agent + Loki Agent를 통합한 Grafana의 관측 데이터 수집기
Alloy는 Metrics, Logs, Traces 수집을 하나의 에이전트로 통합한다. 기존에 Prometheus 계열 scrape, Promtail 로그 수집, OTel Collector 트레이스 전달로 분리되어 있던 파이프라인을 Alloy 하나로 묶어 운영 복잡도를 낮출 수 있다.
핵심 특징
River 언어: HCL 유사 설정 파일. 파이프라인을 컴포넌트 연결로 표현
컴포넌트 기반: source → process → export 체인
핫 리로드: 설정 변경 시 재시작 없이 반영
K8s Service Discovery: Pod, Service, Node 자동 감지
relabeling: Prometheus 호환 relabel_config 지원
파이프라인 구성 예시
Metrics: prometheus.scrape → prometheus.relabel → prometheus.remote_write (Mimir/Prometheus)
Logs: loki.source.kubernetes → loki.process (파싱, 필터) → loki.write (Loki)
Traces: otelcol.receiver.otlp → otelcol.processor.batch → otelcol.exporter.otlp (Tempo)

모든 파이프라인이 하나의 Alloy 인스턴스에서 동작
DaemonSet vs Deployment
DaemonSet (노드 에이전트): 각 노드에 1개. 로그 수집(파일 tail), node_exporter 대체, host-level 메트릭에 사용
Deployment (중앙 수집기): OTLP 수신, 원격 scrape, 중앙 집중 처리에 사용
권장 구성: DaemonSet(로그+노드 메트릭) + Deployment(OTLP 트레이스 수신+서비스 scrape)

Storage

TSDB (Time Series Database) 내부

Prometheus의 로컬 스토리지 엔진. Block 기반 + WAL + Compaction
Prometheus TSDB는 시계열 데이터를 2시간 블록 단위로 관리한다. 최근 데이터는 WAL(Write-Ahead Log)에 먼저 기록되고, 2시간마다 블록으로 압축(compaction)된다.
블록 구조
Head Block (메모리): 현재 수집 중인 데이터. WAL에 동시 기록
Persistent Block (디스크): 2시간 단위로 flush된 불변 블록
각 블록: chunks/(시계열 데이터) + index(레이블 인덱스) + meta.json

Compaction: 작은 블록을 큰 블록으로 병합. 디스크 효율성 향상
최대 블록 크기 = retention 기간의 10% 또는 31일
WAL (Write-Ahead Log)
수집된 샘플은 먼저 WAL에 쓰이고 나중에 블록으로 변환
WAL 덕분에 Prometheus 재시작 시 데이터 손실 최소화

WAL 구조: 128MB 세그먼트 파일로 분할
data/wal/ 디렉토리에 저장
Checkpoint: 주기적으로 WAL을 정리하여 디스크 사용 제한

⚠ WAL 디스크가 느리면 scrape 지연 발생. SSD 필수
디스크 사이징
디스크 = 수집 속도 × 샘플 크기 × retention 기간 수집 속도 = 시계열 수 × scrape 빈도(1/interval) 샘플 크기 ≈ 1~2 bytes (압축 후) 예: 100,000 시계열, 15초 interval, 15일 retention = 100,000 × (1/15) × 1.5 × 86400 × 15 ≈ 86GB

prometheus_tsdb_storage_blocks_bytes — 현재 블록 디스크 사용량
prometheus_tsdb_head_series — 현재 활성 시계열 수 (카디널리티 직접 지표)

Scrape & Federation

Prometheus의 Pull 모델과 대규모 환경에서의 데이터 수집 전략
Scrape 내부 동작
Pull 모델: Prometheus가 타겟에 HTTP GET → /metrics 엔드포인트에서 텍스트 파싱
Service Discovery: K8s API로 Pod/Service/Node를 자동 감지

Relabeling: scrape 전에 라벨을 변환/필터
__meta_kubernetes_pod_label_* → 커스텀 라벨로 변환
action: drop / keep으로 타겟 필터링

Stale Markers: 타겟이 사라지면 stale 마커 삽입 → 시계열 종료 표시
Federation & Remote Write
Federation (계층형): 하위 Prometheus가 수집 → 상위가 /federate 엔드포인트에서 선별 수집
장점: 단순. 단점: 데이터 선별 필요, 지연

Remote Write (권장): Prometheus가 실시간으로 외부 스토리지에 push
→ Mimir, Thanos Receive, Cortex 등으로 전송
장점: 전체 데이터 전송, 중앙 쿼리 가능

Alloy remote_write: Alloy가 Prometheus Agent 모드로 scrape → remote_write. 로컬 TSDB 불필요
Optimization

Recording Rules 설계 패턴

복잡한 PromQL을 사전 계산하여 쿼리 성능과 대시보드 응답 시간을 개선
Recording Rule은 자주 사용되는 복잡한 쿼리를 주기적으로 계산하여 새 시계열로 저장하는 메커니즘이다. 대시보드가 로드될 때마다 무거운 쿼리를 실행하는 대신, 미리 계산된 결과를 읽기만 하면 된다.
네이밍 컨벤션
Google SRE 권장 형식:
level:metric_name:operations level = 집계 수준 (namespace, job, cluster) metric = 원본 메트릭 이름 operations = 적용된 연산 (rate, sum, ratio) 예시: namespace:http_requests_total:rate5m job:sli_latency:ratio_rate30d cluster:node_cpu:utilization
계층적 집계 패턴
Level 1 (raw → rate):
instance:http_requests_total:rate5m = rate(http_requests_total[5m])

Level 2 (rate → sum by job):
job:http_requests:rate5m = sum by(job)(instance:http_requests_total:rate5m)

Level 3 (ratio / SLI):
job:sli:ratio_rate5m = good / total

이렇게 계층화하면 상위 level이 하위 level을 참조하므로 중복 계산 제거 + 쿼리 단순화

Architecture

Loki 아키텍처 & 인덱싱

"Like Prometheus, but for logs" — 라벨로 인덱싱, 로그 내용은 인덱싱하지 않음
Loki의 핵심 설계 철학: 로그 내용을 인덱싱하지 않는다. Elasticsearch는 모든 단어를 인덱싱하여 빠른 풀텍스트 검색을 제공하지만, 인덱스 크기와 비용이 크다. Loki는 라벨(메타데이터)만 인덱싱하고 로그 내용은 압축 저장만 한다. 쿼리 시 라벨로 스트림을 선택한 뒤, 해당 스트림 내에서 grep처럼 필터한다.
핵심 컴포넌트
Distributor: 수신 → 해시링으로 Ingester에 분배
Ingester: 메모리에 로그 청크 축적 → 주기적으로 Object Storage에 flush
Querier: 쿼리 실행. Ingester(최근) + Storage(과거) 모두 검색
Compactor: 인덱스 최적화, retention 적용
Query Frontend: 쿼리 분할, 캐싱, 큐잉
스토리지 구조
Index: 라벨 → 청크 매핑. BoltDB-shipper 또는 TSDB(권장)
Chunks: 압축된 로그 데이터. Object Storage(S3, GCS, MinIO)에 저장

TSDB 인덱스 (Loki 2.8+): Prometheus TSDB와 유사한 구조. BoltDB 대비 성능 대폭 향상
schema_config에서 tsdb 스키마 사용 권장
Critical

라벨 전략 — Loki 성능의 핵심

라벨이 Loki의 인덱스이므로, 라벨 설계가 쿼리 성능과 비용을 결정
Loki에서 라벨 = 스트림. 라벨 조합이 하나의 스트림을 정의하고, 각 스트림은 별도의 청크로 관리된다. 라벨이 많으면 스트림이 많아지고 → 인덱스가 커지고 → Ingester 메모리 사용 증가 → 성능 저하.
✅ 좋은 라벨 (낮은 카디널리티)
namespace — K8s 네임스페이스 (수십 개)
pod — Pod 이름 (수백 개, 관리 가능)
container — 컨테이너 이름
level — info, warn, error (3~5개)
job — 서비스명

원칙: Prometheus에서 라벨로 쓸 만한 것만 Loki에서도 라벨로 사용
❌ 나쁜 라벨 (높은 카디널리티)
trace_id — ❌ 수백만 고유값. 스트림 폭발
user_id — ❌ 사용자 수만큼 스트림
request_id — ❌ 요청 수만큼 스트림
ip_address — ❌ IP 수만큼 스트림
timestamp — ❌ 절대 라벨로 쓰지 않음

이런 값은 로그 내용에 포함하고, LogQL |= / | json으로 필터링
스트림 수 모니터링
loki_ingester_streams_created_total — 생성된 스트림 총 수
loki_distributor_lines_received_total — 수신 로그 라인 수

경고 기준: Ingester당 활성 스트림 10만 개 이상이면 메모리 부족 위험
sum(loki_ingester_memory_streams) 모니터링 필수

스트림 수 = ∏(각 라벨의 고유값 수)
namespace(10) × pod(100) × level(3) = 3,000 스트림 — 건강
namespace(10) × pod(100) × user_id(10000) = 10,000,000 — ❌ 폭발
Query

LogQL 고급 패턴

Log Stream Selector → Pipeline → Metric Query — LogQL의 3단계 구조
# ① 기본: 라벨 선택 + 텍스트 필터 {namespace="production", level="error"} |= "timeout" # ② JSON 파싱 후 필드 필터 {job="api-server"} | json | status_code >= 500 | line_format "{{.method}} {{.path}} {{.status_code}}" # ③ 에러율 메트릭 (rate) sum(rate({job="api-server"} | json | status_code >= 500 [5m])) / sum(rate({job="api-server"} [5m])) # ④ Latency P99 from logs (logfmt 파싱) quantile_over_time(0.99, {job="api-server"} | logfmt | unwrap duration_ms [5m] ) by (path) # ⑤ 특정 trace_id의 로그 (Trace↔Log 연결) {namespace="production"} |= "abc123def456" # ⑥ Top 10 느린 API 경로 topk(10, avg_over_time( {job="api-server"} | json | unwrap duration_ms [1h] ) by (path) )
💡 LogQL 성능 팁
1) 라벨 선택자를 가능한 좁게 (namespace, job 필수 지정)
2) 시간 범위를 최소한으로 (1h > 24h > 7d)
3) |= 텍스트 필터를 JSON 파싱 전에 배치 (조기 필터링)
4) line_format으로 불필요한 필드 제거 → 네트워크 전송량 감소

Tempo & Distributed Tracing 운영

Architecture

Tempo 아키텍처

트레이스 전용 Object Storage 백엔드. 인덱스 없이 Trace ID로 직접 조회
Tempo는 Loki와 유사하게 인덱스를 최소화한다. Trace ID를 기반으로 블룸 필터와 Object Storage에서 직접 조회한다. 이 덕분에 저비용 장기 저장이 가능하다.
컴포넌트
Distributor: OTLP/Zipkin/Jaeger 수신 → 해시링으로 분배
Ingester: 트레이스를 메모리에 버퍼링 → WAL → 블록으로 flush
Compactor: 블록 병합, 블룸 필터 생성
Querier: Trace ID 기반 조회. Ingester(최근) + Storage(과거)
Query Frontend: 검색 쿼리 분할, 캐싱

저장: Parquet 형식. S3/GCS/MinIO. 압축률 높고 쿼리 효율적
TraceQL (Tempo 2.0+)
SQL-like 쿼리 언어로 트레이스 검색:
{resource.service.name="api-server" && span.http.status_code>=500}
{duration > 1s && span.db.system="postgresql"}

Structural: 특정 구조의 span 조합 검색
{span.http.method="GET"} >> {span.db.statement=~"SELECT.*"}
→ GET 요청 아래에 SELECT 쿼리가 있는 트레이스

샘플링 전략
Head-based (기본): SDK에서 확률적 결정
Alloy/OTel Collector의 probabilistic_sampler: rate 0.1 = 10%

Tail-based (권장): Alloy의 tail_sampling processor
정책 예시:
• latency > 1s → always sample
• status = ERROR → always sample
• 나머지 → 5% 확률

비용 임팩트: 10% 샘플링 → 10% 스토리지. 에러/느린 요청은 100% 수집하면 디버깅에 충분

설정: OTel SDK에서 Exemplar 활성화 → Prometheus가 수신 → Grafana에서 Exemplar + Tempo 데이터소스 연결

Grafana 고급 패턴

대시보드 설계 패턴, Alerting 아키텍처, 시그널 간 연결 전략.

Design

대시보드 설계 패턴

USE/RED/Golden Signals 기반 대시보드 계층 구조
4-Level 대시보드 계층
L0 — Overview: 전체 클러스터/서비스 건강. SLO 상태, Error Budget 게이지. 경영진/온콜이 첫 번째로 보는 뷰
L1 — Service: 서비스별 Golden Signals(Latency, Traffic, Errors, Saturation). 온콜이 장애 서비스 특정
L2 — Component: DB, Redis, Kafka 등 개별 컴포넌트 상세. 인프라 팀이 병목 진단
L3 — Debug: 특정 Pod/Node의 상세 메트릭, 로그 패널. 장애 진단 시 사용
대시보드 규칙
변수(Variables) 활용: namespace, service, pod를 드롭다운으로. 하나의 대시보드로 여러 서비스 커버
Annotation: 배포 이벤트를 그래프에 수직선으로 표시. "이 spike는 배포 직후에 발생했나?"
링크: 대시보드 간 drill-down 링크. Overview → Service → Component
패널 수: 대시보드당 15~20개 이하. 많으면 로딩 느리고 정보 과부하
새로고침: 실시간 모니터링은 30초, 일반은 5분, 리포트용은 수동
Alerting

Grafana Alerting 아키텍처

Alert Rule → Notification Policy → Contact Point — Grafana 내장 알림 체계
Grafana Alerting은 Alertmanager를 내장하고 있어 별도 Alertmanager 없이도 알림 라우팅, 그룹핑, 억제(silencing)가 가능하다. Grafana Alerting Webhook을 트리거로 사용하는 방식은 Alertmanager 의존성을 제거한다.
3계층 구조
Alert Rules: PromQL/LogQL 기반 조건 정의. 평가 주기(evaluation interval) 설정. 폴더별 그룹핑

Notification Policies: 라벨 매칭으로 알림 라우팅
• severity=page → PagerDuty Contact Point
• severity=ticket → Slack + Jira
• severity=log → Slack only
그룹핑, 반복 주기, 음소거 설정

Contact Points: 알림 전송 대상. Slack, PagerDuty, Webhook, Email 등
Webhook으로 커스텀 시스템 연동 가능
Webhook 기반 인시던트 트리거
Alertmanager 없이: Grafana Alerting → Webhook Contact Point → 커스텀 서비스

활용:
• Webhook → Incident 기록 서비스 (타임스탬프 저장)
• Webhook → HolmesGPT (AI 트리아지 트리거)
• Webhook → Slack Bot (구조화된 인시던트 메시지)
• Webhook → OpsGate (자동 Runbook 실행)

resolved 상태도 Webhook으로 전달 → 인시던트 종료 자동 기록 → MTTR 자동 계산
Connection

시그널 간 연결 — Metrics ↔ Logs ↔ Traces

LGTM 스택의 진짜 가치는 3개 시그널이 서로 연결될 때 발현
개별 시그널만으로는 "무엇이 느린가?"(Metrics), "무엇이 일어났나?"(Logs), "어디서 느린가?"(Traces)에 각각 답할 수 있다. 연결되면 "왜 느린가?"에 한 번에 답할 수 있다.
연결 매트릭스
Metrics → Traces (Exemplar):
Prometheus 메트릭에 Trace ID 첨부. Grafana에서 spike 지점 클릭 → 해당 Trace 뷰

Traces → Logs (Trace ID):
Tempo Trace 뷰에서 "Logs for this span" 클릭 → Loki에서 해당 Trace ID의 로그 자동 조회
설정: Tempo 데이터소스에서 tracesToLogs 연결 설정

Logs → Traces (Derived Fields):
Loki 로그에서 trace_id 필드를 감지 → 클릭하면 Tempo로 이동
설정: Loki 데이터소스에서 Derived Fields + regex로 trace_id 추출

Metrics → Logs (같은 라벨):
Grafana에서 메트릭 그래프 → "Explore Logs" 버튼 → 같은 namespace/job의 Loki 로그 자동 조회
라벨이 Prometheus와 Loki에서 동일해야 연결 가능 (Alloy의 장점)
💡 완전한 drill-down 시나리오
SLO 대시보드에서 P99 latency spike 발견 → Exemplar 클릭 → Tempo에서 느린 Trace 확인 → 특정 Span에서 DB 쿼리가 3초 걸림 → "Logs for this span" 클릭 → Loki에서 해당 시점 slow query 로그 확인 → 근본 원인: 인덱스 없는 쿼리. 이 전체 과정이 Grafana 안에서 클릭 3~4번으로 완료.
● Cardinality Management

메트릭 카디널리티 관리

카디널리티 폭발의 원인, 진단, 관리 전략 — Prometheus와 Loki의 최대 운영 리스크.

Problem

카디널리티 폭발이란?

라벨 조합의 곱이 시계열 수를 결정. 하나의 라벨이 카디널리티를 수십만 배로 증폭
카디널리티 = 고유한 시계열 수. 시계열은 메트릭 이름 + 라벨 조합으로 정의된다. 라벨 하나를 추가할 때마다 시계열 수가 곱셈으로 증가한다.
폭발 예시
http_requests_total{method, status, path}
method: 5개 × status: 10개 × path: 100개 = 5,000 시계열 — 건강

http_requests_total{method, status, path, user_id}
5 × 10 × 100 × 100,000 users = 5억 시계열 — 💀 폭발

user_id 라벨 하나가 시계열을 10만 배 증가시킴. Prometheus 메모리 폭발 → OOM Kill → 모니터링 전체 장애
위험한 라벨 유형
❌ 절대 라벨로 쓰면 안 되는 것:
• user_id, session_id, request_id
• IP address, email
• 전체 URL path (/users/12345)
• 자유 텍스트 (에러 메시지 전체)
• UUID, 해시값

✅ 안전한 라벨:
• method (GET/POST 등 ~5개)
• status_code 그룹 (2xx, 3xx, 4xx, 5xx)
• service, namespace, pod
• 정규화된 path 패턴 (/users/{id})
Solution

카디널리티 관리 전략

탐지 → 제거 → 예방의 3단계 접근
탐지
TSDB Status API:
curl prometheus:9090/api/v1/status/tsdb
→ seriesCountByMetricName (메트릭별 시계열 수)
→ labelValueCountByLabelName (라벨별 고유값 수)

핵심 메트릭:
prometheus_tsdb_head_series — 현재 활성 시계열 수
prometheus_tsdb_head_chunks — 메모리 청크 수
scrape_series_added — 새로 추가된 시계열

Grafana Mimir: Cardinality Analysis 대시보드 내장
제거 & 예방
① Relabeling으로 제거:
scrape 단계에서 고카디널리티 라벨 drop
metric_relabel_configs에서 action: labeldrop

② Recording Rules로 집계:
상세 시계열 → recording rule로 집계 저장 → 원본 drop

③ 앱 코드 수정:
메트릭 생성 시 고카디널리티 라벨 자체를 넣지 않음
URL path를 정규화: /users/123 → /users/:id

④ 리밋 설정:
sample_limit: scrape당 최대 샘플 수
target_limit: scrape 대상 최대 수
Mimir: max_series_per_user 테넌트별 제한

⑤ 코드 리뷰: 메트릭 추가 PR에 카디널리티 리뷰 포함
⚠ 카디널리티 폭발 = 모니터링 장애 = 모든 것의 장애
Prometheus가 OOM으로 죽으면 알림이 울리지 않고, 대시보드가 작동하지 않는다. 모니터링 시스템의 장애는 다른 모든 장애의 MTTD를 급격히 악화시킨다. 카디널리티 관리는 "성능 최적화"가 아니라 "모니터링 시스템의 가용성 보장"이다.
