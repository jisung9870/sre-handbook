● Performance Methodologies

성능 분석 방법론

USE, RED, TSA — 체계적으로 성능 병목을 찾는 프레임워크. Brendan Gregg의 Systems Performance 핵심.

Resource

USE Method — 리소스 관점 성능 분석

Utilization, Saturation, Errors — 모든 리소스에 대해 세 가지를 확인하면 병목을 찾을 수 있다
Brendan Gregg가 제안한 USE Method는 모든 하드웨어 리소스(CPU, Memory, Disk, Network)에 대해 Utilization(사용률), Saturation(포화도), Errors(에러)를 확인하는 체계적 접근이다. "어디가 병목인가?"에 대한 가장 빠른 답을 제공한다.
리소스Utilization (사용률)Saturation (포화)Errors (에러)
CPUmpstat -P ALL 1
node_cpu_seconds_total		vmstat 1 → r 컬럼 (run queue)
node_load1 > CPU 코어 수		perf stat → machine_clears
dmesg → MCE
Memoryfree -h → used/total
node_memory_MemAvailable		vmstat 1 → si/so (swap in/out)
node_vmstat_pgmajfault		dmesg | grep oom
OOMKill 카운터

Networksar -n DEV 1 → rxKB/txKB

netstat → backlog overflow		ip -s link → errors, drops
ethtool -S → rx_dropped
💡 USE Method 적용 순서
1) CPU → 2) Memory → 3) Disk → 4) Network 순서로 확인. 가장 먼저 포화(Saturation)된 리소스가 병목.
K8s에서: kubectl top node로 노드별 CPU/Memory → kubectl top pod --containers로 Pod별 확인 → 노드 접근하여 USE 체크리스트 실행
Service

RED Method — 서비스 관점 성능 분석

Rate, Errors, Duration — 서비스의 건강 상태를 측정하는 3가지 메트릭
USE가 리소스(인프라) 관점이라면, RED는 서비스(앱) 관점이다. Tom Wilkie가 제안했으며 Google의 Four Golden Signals(Latency, Traffic, Errors, Saturation)의 단순화 버전이다.
RED 3요소
Rate: 초당 요청 수 (RPS). 서비스가 얼마나 바쁜가
sum(rate(http_requests_total[5m])) by (service)

Errors: 초당 실패 요청 수 또는 에러율
sum(rate(http_requests_total{code=~"5.."}[5m])) / sum(rate(http_requests_total[5m]))

Duration: 요청 처리 시간. P50, P99 히스토그램
histogram_quantile(0.99, sum(rate(http_request_duration_seconds_bucket[5m])) by (le))
USE vs RED 선택
USE: "노드/서버가 왜 느린가?" — 인프라 병목 진단
→ CPU 과부하? 메모리 부족? 디스크 I/O 포화?

RED: "서비스가 왜 느린가?" — 서비스 건강 상태 진단
→ 에러율 증가? latency 증가? 트래픽 급증?

실전 흐름:
1. RED로 어떤 서비스가 문제인지 식별
2. 해당 서비스가 실행 중인 노드에서 USE로 리소스 병목 진단
3. 병목 리소스를 확인 후 Flame Graph/profiling으로 근본 원인 추적

SLI/SLO와 연결: RED의 Errors = Availability SLI, Duration = Latency SLI
Thread

TSA Method — Thread State Analysis

스레드가 시간을 어디서 보내는가 — On-CPU, Off-CPU(대기), Sleep
스레드 상태 분해
스레드의 전체 시간 = On-CPU + Off-CPU On-CPU: 실제로 CPU에서 실행 중 → CPU-bound인가? → CPU Flame Graph로 확인 Off-CPU: CPU를 기다리며 대기 중 = Runnable (CPU 큐에서 대기 — CPU 부족) + Blocked (I/O, Lock, Sleep 등) = Disk I/O wait + Network I/O wait + Lock contention + Sleep (timer, condition variable)

핵심 통찰: P99 latency가 높지만 CPU 사용률이 낮다면 → Off-CPU 시간이 범인. I/O 대기, Lock 대기, 네트워크 대기 중 하나
진단 도구
On-CPU 분석:
perf record -g -p PID sleep 30
→ CPU Flame Graph 생성

Off-CPU 분석:
bpftrace -e 'tracepoint:sched:sched_switch { ... }'
→ Off-CPU Flame Graph 생성. 어디서 블록되었는지

Go 앱:
/debug/pprof/goroutine — 고루틴 상태 (running/waiting)
/debug/pprof/block — Lock contention 프로파일
/debug/pprof/mutex — Mutex contention

K8s에서: kubectl exec -it pod -- curl localhost:6060/debug/pprof/goroutine?debug=1
Analysis

워크로드 분석 (Workload Characterization)

"어떤 종류의 부하가 시스템에 가해지고 있는가?"를 체계적으로 파악
질문 체크리스트
Who: 어떤 클라이언트/서비스가 부하를 생성하는가?
What: 어떤 요청 패턴? 읽기/쓰기 비율? 크기 분포?
When: 부하가 집중되는 시간대? 주기적 패턴?
How much: 초당 요청 수? 동시 연결 수? 대역폭?
Distribution: 요청 크기/지속 시간의 분포? 평균 vs P99?

결과: "API 서버는 평균 500 RPS, 읽기 80%/쓰기 20%, 요청 크기 P99 = 2KB, 평일 10~18시 피크, 주말 20% 수준"
→ 이 프로파일이 Capacity Planning과 성능 최적화의 기반
Latency 분포 분석
평균이 아닌 분포를 봐야 한다:

히스토그램: Prometheus histogram_quantile
P50 = 10ms, P99 = 200ms, P99.9 = 2s
→ 평균(30ms)만 보면 문제 없어 보이지만, P99.9에서 200배 느린 꼬리가 존재

Bimodal 분포: 캐시 히트(2ms) vs 캐시 미스(200ms)가 혼재 → 평균 100ms. 하지만 실체는 두 개의 모드

Coordinated Omission: 벤치마크 도구가 느린 응답 중에 새 요청을 보내지 않아 latency를 과소 측정하는 문제. Gil Tene의 발견. wrk2, k6가 이를 보정
● Linux Performance Tools

Linux 성능 분석 도구

perf, BPF tools, strace, ss, sar — 시스템 성능을 깊이 파고드는 도구.

Core Tool

perf 심화

Linux Performance Counters — CPU 마이크로아키텍처 수준의 성능 분석
핵심 명령
perf stat: CPU 카운터 기반 통계
perf stat -p PID sleep 10
→ instructions, cycles, cache-misses, branch-misses
→ IPC (Instructions Per Cycle): 1.0 이상이면 양호. < 0.5면 메모리 바운드

perf record + report:
perf record -g -p PID sleep 30
perf report
→ 어떤 함수에서 CPU 시간을 가장 많이 소비하는가

perf top: 실시간 CPU 핫스팟
perf top -p PID
→ top처럼 실시간으로 CPU를 많이 쓰는 함수 목록
PMU (Performance Monitoring Unit)
CPU 하드웨어 카운터:
• instructions: 실행된 명령어 수
• cycles: CPU 사이클 수
• cache-references: 캐시 참조
• cache-misses: 캐시 미스 → 메모리 접근 필요
• branch-misses: 분기 예측 실패
• context-switches: 컨텍스트 스위칭 수

Cache Miss Rate > 10%: 데이터가 캐시에 안 올라옴. 자료구조의 메모리 레이아웃 문제
Branch Miss > 5%: 조건 분기가 많은 코드. 루프 언롤링, 분기 제거 필요
BPF

BPF 성능 도구 (bpftrace / BCC)

eBPF 기반 동적 트레이싱 — perf보다 유연한 커널/앱 분석
bpftrace 원라이너
프로세스별 syscall 카운트:
bpftrace -e 'tracepoint:raw_syscalls:sys_enter { @[comm] = count(); }'

TCP 재전송 추적:
bpftrace -e 'kprobe:tcp_retransmit_skb { @[kstack] = count(); }'

Block I/O latency 히스토그램:
bpftrace -e 'kprobe:blk_account_io_done { @us = hist(nsecs - @start[arg0]); }'

파일 열기 추적:
bpftrace -e 'tracepoint:syscalls:sys_enter_openat { printf("%s %s\n", comm, str(args->filename)); }'

DNS 조회 지연 측정:
bpftrace -e 'uprobe:/lib/x86_64-linux-gnu/libc.so.6:getaddrinfo { @start[tid] = nsecs; } uretprobe:/lib/x86_64-linux-gnu/libc.so.6:getaddrinfo { @ms = hist((nsecs - @start[tid])/1000000); delete(@start[tid]); }'
BCC tools (사전 구축된 도구)
execsnoop: 새 프로세스 실행 추적
opensnoop: 파일 열기 추적
biolatency: Block I/O latency 히스토그램
biosnoop: 개별 Block I/O 요청 추적
tcplife: TCP 연결 수명 추적
tcpretrans: TCP 재전송 추적
runqlat: CPU run queue latency (얼마나 CPU를 기다렸는가)
profile: CPU 프로파일링 → Flame Graph
offcputime: Off-CPU 시간 분석 → Off-CPU Flame Graph

apt install bpfcc-tools 또는 dnf install bcc-tools
Classic

strace / ss / sar

전통적이지만 여전히 강력한 성능 진단 도구
strace — syscall 추적
strace -f -e trace=network -p PID — 네트워크 syscall만
strace -f -e trace=file -p PID — 파일 syscall만
strace -c -p PID — syscall별 통계 요약
strace -T -e read,write -p PID — 각 syscall 소요 시간

⚠ 주의: strace는 ptrace 기반이라 성능 오버헤드가 크다 (2~100배 느려질 수 있음). 프로덕션에서는 bpftrace/BCC를 권장

K8s: kubectl debug -it pod -- strace -p 1 -e trace=network
ss & sar
ss — 소켓 통계 (netstat 대체):
ss -s — TCP/UDP 소켓 요약
ss -tnp — TCP 연결 목록 + PID
ss -tnp state time-wait | wc -l — TIME_WAIT 수
ss -tnp state established dst :5432 — DB 연결 수

sar — 히스토리컬 시스템 통계:
sar -u 1 5 — CPU 사용률 5초간
sar -r 1 5 — 메모리 사용률
sar -d 1 5 — 디스크 I/O
sar -n DEV 1 5 — 네트워크 인터페이스 통계
sar -n TCP 1 5 — TCP 통계 (active/passive/retrans)
과거 데이터: sar -u -f /var/log/sa/sa15 (15일 데이터)
● Flame Graphs

Flame Graph 분석

CPU, Memory, Off-CPU Flame Graph — 성능 병목을 시각적으로 식별하는 가장 강력한 도구.

Reading

Flame Graph 읽기 & 해석

X축은 알파벳 순서(시간순 아님!). Y축은 콜 스택 깊이. 너비가 해당 함수의 CPU 점유 비율
Brendan Gregg가 창시한 Flame Graph는 스택 트레이스를 시각화한 것이다. 가장 흔한 오해: X축은 시간이 아니다. 같은 함수 호출을 합쳐서 알파벳 순으로 정렬한 것이다. 넓은 막대 = 많은 시간을 차지하는 함수.
읽는 법
Y축 (아래→위): 콜 스택. 아래가 호출자, 위가 피호출자
X축: 알파벳 순으로 정렬된 스택 프레임. 좌우 위치에 의미 없음
너비: 해당 함수가 전체 샘플에서 차지하는 비율. 넓을수록 많은 시간 소비
색상: 보통 랜덤. 특별한 의미 없음 (일부 도구는 언어별 색상)

핫스팟 찾기:
→ 꼭대기(top of stack)에서 넓은 "플래토(고원)"를 찾는다
→ 이 함수가 실제로 CPU를 소비하는 곳
→ 아래쪽이 넓으면: 많은 함수를 호출하지만 본인은 CPU 적게 사용
생성 방법
Linux perf:
perf record -g -p PID sleep 30
perf script | stackcollapse-perf.pl | flamegraph.pl > flame.svg

Go pprof:
go tool pprof -http=:8080 http://localhost:6060/debug/pprof/profile?seconds=30
→ 브라우저에서 Flame Graph 자동 생성

bpftrace:
bpftrace -e 'profile:hz:99 { @[kstack] = count(); }' > out.txt
stackcollapse-bpftrace.sh out.txt | flamegraph.pl > flame.svg

Pyroscope: 프로덕션에서 지속적으로 수집 → 시간대별 Flame Graph
Types

Flame Graph 유형

CPU, Off-CPU, Memory, Differential — 각각 다른 병목을 보여준다
유형별 용도
CPU Flame Graph: "어떤 함수가 CPU를 많이 쓰는가?" CPU-bound 문제 진단. 가장 기본적

Off-CPU Flame Graph: "어떤 함수에서 블록되어 기다리는가?" I/O wait, Lock contention 진단. CPU 사용률은 낮은데 latency가 높을 때

Memory Flame Graph: "어떤 함수가 메모리를 많이 할당하는가?" 메모리 누수, 과도한 allocation 진단. Go: /debug/pprof/heap

Differential Flame Graph: 두 시점의 Flame Graph를 비교. "어떤 함수가 더 많이 실행되게 됐는가?" 성능 퇴화(regression) 진단
Off-CPU Flame Graph 핵심
가장 과소평가되는 Flame Graph. "서비스가 느린데 CPU 사용률이 30%뿐이에요" → Off-CPU 문제일 확률 높음

흔한 Off-CPU 원인:
• Disk I/O wait: 느린 디스크 읽기/쓰기
• Network I/O wait: 느린 DB 쿼리, 외부 API 호출
• Lock contention: Mutex/RWLock 경합
• Sleep/Timer: 불필요한 sleep 호출
• Page Fault: 메모리 매핑된 파일의 major fault

생성: offcputime-bpfcc -p PID 30 > off.txt
flamegraph.pl --color=io < off.txt > offcpu.svg
● Continuous Profiling

프로덕션 프로파일링

Pyroscope, Parca, Go pprof — 프로덕션 환경에서 상시 프로파일링하여 성능 변화를 추적.

Always-On

Continuous Profiling이란

메트릭이 "무엇이 문제"를 알려주면, 프로파일링은 "코드의 어디가 문제"를 알려준다
전통 프로파일링은 "문제가 발생했을 때" 수동으로 실행한다. Continuous Profiling은 프로덕션에서 항상 프로파일을 수집하여, 과거 시점의 성능 분포를 언제든 조회할 수 있다. Metric → Trace → Profile로 drill-down하여 코드 라인까지 근본 원인을 추적한다.
도구
Pyroscope (Grafana): LGTM 스택과 네이티브 통합. Grafana에서 Flame Graph 조회
Pull/Push 모드. Go, Java, Python, Ruby 지원
Alloy에서 pyroscope.scrape로 수집

Parca (Polar Signals): eBPF 기반 zero-instrumentation 프로파일링. 언어 무관

오버헤드: ~2~5% CPU. 프로덕션에 상시 적용 가능한 수준

LGTM + P (Profiling) = LGTMP:
Metric(Prometheus) → 이상 감지
Trace(Tempo) → 느린 요청 추적
Profile(Pyroscope) → 코드 핫스팟 식별
→ 완전한 관측 파이프라인
활용 시나리오
성능 퇴화 감지:
배포 전후의 CPU 프로파일을 비교 (Differential Flame Graph). "이 배포 후 parseJSON 함수가 3배 느려졌다"

리소스 최적화:
"이 Pod이 CPU 500m을 사용하는데, 프로파일을 보니 40%가 JSON 직렬화. protobuf로 전환하면 300m으로 줄일 수 있다"

인시던트 사후 분석:
"장애 시점의 CPU 프로파일을 보니 DB 커넥션 풀 대기에서 80%의 시간을 소비했다" → Postmortem의 Root Cause 근거

비용 절감:
CPU 프로파일에서 불필요한 처리(과도한 로깅, 불필요한 JSON 파싱)를 식별 → 리소스 request 축소 → 노드 수 감소
Go

Go pprof — Go 앱 프로파일링

Go는 pprof를 내장하여 프로덕션 프로파일링이 가장 쉬운 언어 중 하나
pprof 엔드포인트
HTTP 서버에 추가:
import _ "net/http/pprof"
go func() { http.ListenAndServe(":6060", nil) }()

프로파일 종류:
/debug/pprof/profile?seconds=30 — CPU 프로파일
/debug/pprof/heap — 힙 메모리 할당
/debug/pprof/goroutine — 고루틴 상태
/debug/pprof/block — 블로킹 프로파일
/debug/pprof/mutex — 뮤텍스 경합
/debug/pprof/trace?seconds=5 — 실행 트레이스
실전 사용
CPU 프로파일 분석:
go tool pprof http://localhost:6060/debug/pprof/profile?seconds=30
(pprof) top 20 — CPU 상위 20 함수
(pprof) web — 브라우저에서 그래프 열기
(pprof) list funcName — 소스 코드 라인별 CPU 소비

K8s에서:

Linux 60-Second Analysis

성능 문제가 보고되었을 때, 노드에 접속하자마자 실행하는 10개 명령
# 1. 시스템 부하 요약 — 지난 1/5/15분 load average uptime # 2. 커널 메시지 — OOM, 하드웨어 에러, TCP 드롭 dmesg -T | tail -20 # 3. CPU 사용률 + run queue vmstat 1 5 # r > CPU코어 수: CPU 포화 / si/so > 0: 스왑 발생 # 4. CPU 코어별 사용률 mpstat -P ALL 1 3 # 한 코어만 100%: 싱글스레드 병목 # 5. 프로세스별 CPU/Memory pidstat 1 3 # 6. 디스크 I/O iostat -xz 1 3 # %util > 80%: 디스크 포화 / await > 10ms: 느린 I/O # 7. 메모리 free -h # available이 중요 (free가 아님!) # 8. 네트워크 인터페이스 sar -n DEV 1 3 # 9. TCP 통계 — 재전송이 높으면 네트워크 문제 sar -n TCP,ETCP 1 3 # 10. 전체 요약 top # 또는 htop
💡 각 명령의 핵심 관찰 포인트
uptime: load average가 CPU 코어 수를 초과하면 포화. 급증 추세이면 장애 진행 중
dmesg: Out of memory, nf_conntrack: table full, NMI watchdog → 즉시 조치 필요
vmstat: r > 코어 수 → CPU 포화. si/so > 0 → 스왑 = 메모리 부족 = 성능 급락
iostat: %util > 80% 지속 → 디스크 병목. SSD인데 await > 10ms → 이상
free: available이 총 메모리의 10% 미만 → 메모리 압박

K8s 성능 진단 60초

K8s 환경에서의 성능 문제 초기 진단 체크리스트
# 1. 노드 리소스 사용률 kubectl top node # 2. Pod 리소스 사용률 (컨테이너별) kubectl top pod --containers -n production # 3. Pod 상태 확인 — Restart, OOMKilled kubectl get pods -n production --sort-by='.status.containerStatuses[0].restartCount' # 4. Node conditions — DiskPressure, MemoryPressure kubectl get nodes -o custom-columns='NAME:.metadata.name,CONDITIONS:.status.conditions[?(@.status=="True")].type' # 5. Events — 최근 문제 이벤트 kubectl get events -n production --sort-by='.lastTimestamp' | tail -20 # 6. CPU Throttling 확인 kubectl exec -it pod -- cat /sys/fs/cgroup/cpu.stat # nr_throttled > 0 이면 CPU limit에 의한 throttling 발생 # 7. 컨테이너 메모리 상세 kubectl exec -it pod -- cat /sys/fs/cgroup/memory.stat # 8. 노드 진입하여 USE Method kubectl debug node/my-node -it --image=ubuntu chroot /host uptime && vmstat 1 3 && iostat -xz 1 3 && free -h
⚠ CPU Throttling — 숨겨진 성능 킬러
CPU limit이 설정되어 있으면, 앱이 limit 이상의 CPU를 사용하려 할 때 커널이 강제로 실행을 중단(throttle)시킨다. CPU 사용률은 limit 이하로 보이지만 latency가 급증한다. container_cpu_cfs_throttled_periods_total이 높으면 limit 증가 또는 제거 고려. 많은 SRE 팀이 CPU limit을 설정하지 않는 전략을 채택하는 이유.
