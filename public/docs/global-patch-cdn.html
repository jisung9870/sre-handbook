● Overview

게임 패치 운영의 본질

게임 패치는 빌드 산출물을 전 세계 게이머의 디바이스로 안전하게 전달하는 일이다. 일반 웹 배포와 달리 대용량(GB 단위)·전 세계 동시 다운로드·플랫폼별 정책 차이가 핵심 제약이다.

게임 패치란?

일반 웹 배포와 다른 점

웹은 서버에 코드 push → 즉시 적용. 게임은 각 사용자 디바이스에 GB 단위 파일을 직접 전달해야 한다. CDN·차분 패치·점진 rollout 모두 이 제약에서 나온다.

두 종류의 패치

(1) 클라이언트 패치: 실행 파일·DLL·엔진 변경. 스토어 심사·재설치 필요. (2) 콘텐츠 패치: 에셋·데이터(아이템·이벤트)만 변경. 핫픽스가 가능해 운영의 핵심.

운영 사이클

정기 패치(주 1회·격주), 핫픽스(긴급), 시즌 패치(월·분기), 신작 런칭 패치(빅뱅). LiveOps 사이클과 직결되어 있어 패치 cadence가 곧 운영 리듬이다.

SRE 책임 영역

빌드 파이프라인 → CDN 업로드 → 점진 배포 → 다운로드 모니터링 → 롤백. 게임 코드 자체가 아니라 "전달 인프라"를 책임진다. 게임팀이 만든 빌드를 사용자에게 안전히 보내는 마지막 마일.

★ TIP
"게임 데이터(아이템·드롭률·이벤트)는 서버 사이드에서 핫픽스, 클라이언트 빌드는 정기 cadence" — 이 분리가 모바일·PC 양쪽에서 표준이다. 클라이언트 패치 한 번에 모든 걸 묶지 말고, 서버 데이터로 빼낼 수 있는 건 최대한 빼낸다.

핵심 KPI

패치 운영의 건강성을 측정하는 5대 지표. 각 KPI는 SLO로 정의해 두고 패치 후 자동으로 점검한다.

① 다운로드 성공률

패치 시도 대비 완료 비율. 완료 / 시작으로 계산. 낮으면 CDN 장애·디스크 부족·네트워크 단절 의심. 일반적으로 95% 이상 유지가 목표.

② 시작점유율 (Adoption Rate)

패치 N시간 후 신규 버전으로 게임을 시작한 사용자 비율. 모바일은 7일에 90%+, Steam은 24시간에 95%+가 일반적 목표.

③ 롤백 시간 (MTTR)

장애 감지 → 이전 버전 복귀까지의 시간. PC/Steam은 분 단위, 모바일은 단 시간~수 시간. 롤백이 빨라야 적극적 rollout이 가능하다.

④ 평균 패치 사이즈

업데이트 1회당 다운로드 용량. 차분 패치(delta)를 잘 쓰면 풀 빌드 대비 5~20%로 줄일 수 있다. 비용·완료율 모두에 직접 영향.

⑤ CDN 비용 / DAU

월 CDN 비용을 활성 사용자 수로 나눈 단가. 신작 런칭 시 일시적으로 치솟지만 정상 상태에서는 안정. 멀티 CDN·차분 패치·prefetch 최적화의 효과를 가장 직접적으로 보여준다.

● CDN Architecture

CDN 아키텍처

게임 패치 CDN은 일반 웹 CDN과 트래픽 패턴이 다르다. 대용량 파일·런칭 스파이크·전 세계 동시 다운로드에 맞춘 설계가 필요하다.

Origin · Edge 다단계 캐싱

CDN은 사용자에게 가까운 Edge 서버에서 콘텐츠를 캐시해 응답한다. 게임 패치는 파일이 크고 immutable(한 번 만들면 안 변함)이라 CDN과 매우 잘 맞는 워크로드다.

Origin Shield (Mid-tier)

모든 Edge가 Origin을 직접 때리면 캐시 미스가 폭주한다. Origin Shield 또는 Tiered Cache를 두면 Edge → Mid → Origin으로 적층, Origin 트래픽이 N분의 1로 줄어든다.

Cache TTL · 무효화

패치 파일은 immutable이므로 긴 TTL(1년+)이 정답. 새 빌드는 새 URL(파일명에 hash·BuildID 포함). 기존 URL을 invalidate할 일이 거의 없게 만드는 게 best practice.

HTTP Range Request

대용량 파일을 부분 다운로드하기 위한 핵심 기능. Range: bytes=N-M 요청을 받아 부분만 응답. 일시 중단·재개·병렬 다운로드 모두 이 위에 동작.

Prefetch · Pre-warm

런칭 직전 또는 패치 직전에 모든 Edge에 콘텐츠를 미리 푸시. "콜드 스타트 1분 안에 100만 명이 동시 접속" 시나리오에서 결정적. 대형 CDN은 prefetch API 제공.

멀티 CDN · GeoDNS · Anycast

단일 CDN은 (1) 장애 시 회피 불가 (2) 지역별 성능 편차 큼 (3) 가격 협상력 약함. 글로벌 게임은 보통 2~3개 CDN을 함께 쓴다.

GeoDNS 라우팅

DNS 응답을 사용자 위치별로 다르게 준다. 한국 사용자 → CDN A, 동남아 → CDN B, 유럽 → CDN C. DNS TTL을 짧게(60s) 잡아야 장애 시 빠른 전환이 가능.

Anycast (CDN 내부)

같은 IP를 전 세계 여러 PoP에 광고. BGP가 가장 가까운 곳으로 라우팅. CloudFlare·Fastly가 대표. DNS에 의존하지 않는 자동 페일오버가 강점.

RUM (Real User Monitoring)

클라이언트가 다운로드 속도·실패율을 텔레메트리로 보낸다. 이 데이터로 CDN별 성능을 측정해 GeoDNS 가중치를 동적으로 조정. 멀티 CDN의 핵심 기반.

Failover 전략

(1) 헬스체크 실패 시 자동 전환 (2) 다운로드 클라이언트가 1차 실패 시 2차 CDN URL로 retry (3) 매니페스트에 다중 CDN URL을 명시. 클라이언트 단의 fallback이 가장 빠르고 안정적.

★ NOTE
멀티 CDN으로 30~40% 비용 절감 사례가 많다. CloudFront(고정·통합) + BunnyCDN(저가) 같은 조합으로 평소엔 저가 CDN, 스파이크 시 메이저 CDN으로 분배하는 패턴이 일반적이다.

게임 패치 트래픽의 4가지 특성

웹 컨텐츠 CDN과 다른 게임 패치 CDN의 고유 특성. 이 4가지 때문에 일반 웹 CDN 설계로는 부족하다.

① 폭발적 동시 다운로드

점검 종료 시각·런칭 시각에 수십만 ~ 수백만 명이 동시 다운로드 시작. 평소 대비 100~1000배 트래픽 스파이크. 1Tbps 이상 피크가 나오는 경우도 있음.

② 파일이 크다

전체 빌드 30~150GB(AAA), 차분 패치 100MB ~ 5GB가 흔함. 일반 웹의 정적 자산(KB ~ MB) 대비 3~6 자릿수 큼. 단일 응답이 분 단위로 길어진다.

③ 글로벌 동시성

전 세계가 같은 시각에 같은 콘텐츠를 받는다. cache warmup이 실패하면 모든 region이 동시에 cold. Origin이 즉시 폭발한다.

④ Immutable

한 번 만든 빌드는 절대 안 변한다(URL hash로 식별). 캐시 hit률을 99%+ 유지하는 게 가능. 단, 빌드 단위 변경 시에는 새 URL → cold cache 위험.

⚠ PITFALL
점검 종료 시각을 똑같이 잡아두고 prefetch를 안 하면 모든 Edge가 동시에 cold cache → Origin이 모든 트래픽을 받음 → Origin 폭발. 점검 30분~1시간 전에 prefetch API로 모든 Edge에 콘텐츠를 미리 채워두는 게 표준.

● Mobile Patching

모바일 패치 운영

모바일은 스토어가 모든 배포 게이트를 쥔다. Apple App Store와 Google Play의 정책·rollout 메커니즘·in-app update API를 모두 이해해야 한다. 2026-05 기준 정책으로 정리.

스토어 심사 · 정책 (2026-05 기준)

스토어 심사는 패치 cadence를 결정짓는 가장 큰 외부 변수다. 일반 심사 시간과 지연 시 대응을 모두 알아야 한다.

Apple App Store

일반 심사: 24~48시간이 평균. 단, 정책 점검·자동 검수 큐 적체 시 7~30일+까지 늘어진 사례도 2026 보고됨. 긴급 심사(Expedited Review)는 신청 가능하지만 승인은 케이스별.

Google Play

심사: 보통 수 시간 ~ 1일. 정책·permission 변경 있으면 길어짐. Pre-launch report·App Bundle 검수가 자동으로 돈다. Google이 더 빠르지만 정책 위반 시 즉시 takedown 위험은 비슷.

긴급 패치 전략

(1) 핫픽스 데이터는 서버에서 즉시 (스토어 심사 0초). (2) 클라이언트 변경이 필수면 Apple Expedited Review 신청. (3) Google은 Internal Testing 트랙으로 1차 검증 후 Production 적용.

제출 사전 점검

크래시 + 디바이스 호환성 + permission rationale + privacy nutrition label/Data safety form. 한 번 reject되면 재심사 큐에 다시 들어가므로 제출 전 internal QA 체크리스트가 본 cadence를 좌우한다.

점진적 rollout 메커니즘

두 스토어 모두 점진 배포를 지원하지만 메커니즘이 다르다.

Apple Phased Release

7일에 걸쳐 자동 증가: 1% → 2% → 5% → 10% → 20% → 50% → 100%. 24시간마다 한 단계씩 올라간다. 문제 발견 시 최대 30일까지 pause 가능. 일시중지 후 풀어도 동일 사용자 그룹부터 재개.

Google Play Staged Rollout

비율을 임의로 직접 설정(예: 1%, 5%, 50%). 자동 증가 안 함 — 수동으로 비율을 올린다. Halt rollout 가능, resume 시 동일 그룹에 재개. inAppUpdatePriority(0~5)로 In-App Update 우선순위 지정.

표준 점진 rollout 운영 시퀀스

1%크래시·CS 모니터
5~10%지표 비교

★ TIP
각 단계에서 최소 24시간은 보고 다음 단계로 넘어가는 게 안전하다. 모바일은 사용자가 패치를 받는 시점이 분산되어 있어 4~8시간 내 데이터로는 통계적 유의성이 부족하다.

In-App Update API

스토어 외에 게임 자체가 업데이트를 유도하는 메커니즘. 강제 업데이트·권장 업데이트·핫픽스 안내에 핵심.

Google Play In-App Updates

Play Core 라이브러리 (Android 5.0+, 모바일·태블릿·ChromeOS).
· Immediate: 전체 화면 블로킹 → 강제 업데이트
· Flexible: 백그라운드 다운로드 → 사용자 게임 계속
· inAppUpdatePriority 0~5로 우선순위 표시

iOS — 자체 구현

Apple은 공식 In-App Update API 미제공. 게임이 직접 부팅 시 서버에 최소 버전 체크 → 미달이면 App Store 링크 띄우는 방식. SKStoreReviewController는 별개(평점 요청용).

강제 업데이트 정책 설계

서버에 버전 매트릭스(`min_required_version`, `recommended_version`, `latest_version`)를 두고 부팅 시 클라이언트가 조회한다. 미달이면 차단·권장이면 alert. 같은 빌드를 여러 region에 다른 정책으로 적용할 수 있어야 한다(예: 한국은 강제, 일본은 권장).

// 클라이언트 부팅 시 버전 체크 의사 코드
const {min, recommended, latest} = await fetchVersionPolicy(country, app);
if (current < min)         forceUpdateAndExit();   // 강제
else if (current < recommended) showSoftUpdate();      // 권장
else if (current < latest)      flexibleUpdate();      // In-App Update

Asset Bundle · Pak — 리소스 분리 패치

앱 본체 패치(스토어 심사)와 리소스 패치(자체 CDN)를 분리하는 핵심 메커니즘. 모든 글로벌 모바일 게임은 이 분리를 사용한다.

Unity Addressables

Asset Bundle의 차세대 방식. Remote Catalog + RemoteLoadPath 설정으로 CDN에서 로드. [BuildTarget] placeholder로 플랫폼별 분기. 빌드 후 "Update a Previous Build" 워크플로우로 차분 카탈로그 생성.

Unreal Pak / IoStore

UE5 IoStore: .ucas(데이터) + .utoc(메타) 분리. Pak 파일 mount 우선순위는 알파벳 순 + _p suffix로 patch 우선. 패치 빌드는 -basedonreleaseversion 옵션으로 생성.

분리의 장점

(1) 스토어 심사 우회 — 리소스만 갱신은 스토어 거치지 않음. (2) 초기 다운로드 작게 → 인스톨 컨버전 향상. (3) 이벤트마다 새 에셋을 즉시 푸시 가능.

분리의 함정

리소스가 클라이언트 코드와 incompatible 변경되면 실시간 버전 매칭 로직 필수. 카탈로그 무결성·서명 검증 안 하면 MITM·뒷주머니 어셋 주입 위험.

⚠ PITFALL
리소스 분리 패치는 "스토어 정책 회피"로 오해되기 쉽다. 게임 로직·코드 자체를 우회 패치하는 건 두 스토어 모두 정책 위반. 게임 데이터·에셋·텍스처·밸런스 수치까지가 안전 영역. JS 엔진·LuaVM으로 코드 패치를 시도하면 takedown 위험.

Steam · PC 패치 운영

Steam은 SteamPipe라는 자체 콘텐츠 배포 시스템으로 분 단위 즉시 배포가 가능하다. 모바일과 완전히 다른 운영 모델이라 별도 학습이 필요하다.

SteamPipe 아키텍처

2013년 Valve가 도입한 콘텐츠 배포 시스템. HTTP 기반이라 일반 CDN과 동일한 캐싱·proxy 적용이 가능하고, 1MB 블록 binary delta를 자동으로 계산한다.

App ID

Steam 상에서 게임을 식별하는 최상위 ID. DLC도 별도 App ID. 이 위에 빌드·branch·depot이 붙는다.

Depot

콘텐츠 묶음 단위. 일반적으로 OS별·언어별·DLC별로 분리. Windows depot, Linux depot, JP/CN 한정 depot 등. 사용자는 자기 환경에 맞는 depot만 받음.

Manifest

각 depot 빌드의 파일 목록 + 사이즈 + SHA1 + flags를 담은 64-bit ID. Steam 클라이언트는 매니페스트를 기준으로 어떤 chunk를 받아야 할지 결정.

BuildID

App 전체 빌드의 globally unique ID. 빌드 업로드 완료 시 자동 부여. branch에 어떤 BuildID를 live시키느냐로 사용자 노출 버전이 결정.

★ NOTE
SteamPipe는 HTTP 기반이라 제3자 HTTP 캐시·프록시·LAN 캐시가 모두 작동한다. 사내 OPS는 SteamPipe Local Content Server를 두고 사내 빌드 다운로드 트래픽을 외부로 안 새게 한다. PC방·인터넷 카페도 자체 LAN 캐시를 운영.

Branch · Build 라이프사이클

Steam은 빌드를 분 단위로 즉시 배포할 수 있지만, branch 분리·password 보호로 단계적 검증이 가능하다.

default branch

모든 사용자가 자동으로 받는 라이브 브랜치. 여기 set-live한 BuildID가 곧 정식 버전.

beta branch (password)

이름 붙인 branch + 비밀번호로 사전 접근 제한. 사용자가 라이브러리 → Properties → Betas에서 코드 입력. QA·인플루언서·PTR 운영의 표준 도구.

staging branch (internal)

사내 검증용. 비밀번호로 격리. 빌드 검수 후 default로 promote.

롤백

이전 BuildID를 default branch에 다시 set-live하면 끝. 사용자는 다음 게임 시작 시 자동 다운로드(차분 패치). 분 단위 rollback이 가능한 게 Steam의 핵심 강점.

Steamworks ContentBuilder 표준 워크플로우
# 1. 빌드 산출물을 ContentBuilder/content/ 하위에 배치
ContentBuilder/
  content/
    app_build_123456.vdf      # app·depot·branch 정의
    depot_123461_win64/         # Windows 64 빌드

# 2. SteamCMD로 업로드 → 자동 manifest 생성 + delta 계산
steamcmd +login partner_login \
         +run_app_build app_build_123456.vdf \
         +quit

# 3. 파트너 사이트에서 BuildID를 default 또는 beta branch에 set-live
# 4. 사용자는 다음 게임 시작 시 자동 다운로드 (delta only)
♥ KEY
Steam에는 모바일 같은 staged rollout(1%→5%→...) 메커니즘이 없다. 대신 beta branch로 사전 검증 → default로 push가 표준. PTR(Public Test Realm)을 운영하는 게임도 같은 메커니즘.

Delta 패치 — 1MB block 알고리즘

SteamPipe는 모든 빌드를 1MB chunk로 자르고 SHA1을 계산한다. 패치 시 변경된 chunk만 다운로드 — 클라이언트 입장에선 binary delta와 동일한 효과.

1MB chunk 경계

모든 파일이 1MB 단위로 분할 + SHA1 해시. 파일 변경 시 해당 chunk만 다른 SHA1 → Steam 클라이언트가 그것만 다운로드. 전체 빌드의 1% 변경이면 다운로드도 ~1%.

chunk align 최적화

대용량 파일(.pak, .ucas)이 1MB 경계에 정렬되어 있지 않으면 작은 변경에도 chunk가 줄줄이 깨진다. UE5 IoStore는 1MB align 옵션으로 SteamPipe 효율 극대화.

compression

chunk별 zstd 압축. 전송량과 디스크 모두 절약. Steam 클라이언트가 자동 처리해서 개발자가 신경 쓸 일은 거의 없음.

실패 시 재다운로드

SHA1이 안 맞으면 그 chunk만 다시 받음. 디스크 손상·다운로드 중단·MITM 시도 모두 자동 감지·복구.

★ TIP
Unity·Unreal로 게임 만들 때 빌드 옵션에서 "deterministic build" + "incremental" 옵션을 켜야 SteamPipe delta가 잘 먹는다. 매 빌드가 무작위로 재정렬되면 chunk가 모두 새 SHA1 → 사용자는 매번 풀 다운로드.

자체 PC 런처 패처

Steam을 거치지 않고 직접 배포하는 PC 게임은 자체 런처에 패처를 구현해야 한다. 한국 MMORPG·중국 게임이 흔히 이 모델.

구성 요소

(1) 매니페스트 서버(파일 목록·해시·CDN URL) (2) 차분 패치 서버(bsdiff/xdelta3) (3) 자체 CDN (4) 클라이언트 패처(런처 EXE).

패치 흐름

런처 시작 → 매니페스트 다운로드 → 로컬 파일과 해시 비교 → 변경된 파일만 다운로드 (또는 차분 적용) → 무결성 검증 → 게임 실행.

장점

스토어 수수료 0%. 패치 cadence 자유. 사용자 환경 정밀 통제(PC방 캐시·중국어 패치 분기·anti-cheat 통합).

단점

모든 인프라를 직접 운영(CDN·서명·anti-tamper). 사용자 발견성(discoverability) 낮음. Steam 대비 신규 유저 유입 채널이 작음.

♥ KEY
Steam과 자체 런처를 동시 운영하는 듀얼 채널 게임이 늘고 있다(예: 글로벌은 Steam, 한국·중국은 자체 런처). 두 채널의 빌드 동기화·매출 정산·CS 채널 분리가 운영 핵심 과제.
● Global Deployment

글로벌 배포 & 운영

플랫폼 메커니즘 위에서 실제 글로벌 패치를 안전하게 굴리는 운영 패턴. region 분할·canary·kill switch·차분 알고리즘·비용 최적화의 5축으로 본다.

Region별 점진 rollout

전 세계에 동시 배포하지 않고 region별로 시간차를 두면 (1) 문제를 작은 영향 범위에서 감지 (2) 시차 활용 — 한 region 장애 시 다른 region에 사전 학습.

Region 우선순위 결정

(1) 작은 트래픽 region을 먼저 (탐지 영향 작게). (2) 단, 대표성 있는 디바이스·언어 분포는 확보. KR·JP·SEA가 전형적 1차 region.

관찰 윈도우

region 간 최소 24시간 간격. 패치 후 24시간이 지나야 일·주간 사이클의 이벤트(접속자 시간대·BMT 결제 등)가 다 노출.

지표 비교

각 region에서 패치 전/후의 핵심 KPI를 비교: 크래시율, 결제 성공률, 다운로드 성공률, 로그인 실패율, 시작점유율. 1개라도 회귀하면 다음 region 차단.

cross-region 의존성

region별 데이터 격리가 약하면(공유 DB, 글로벌 CDN) region별 rollout의 효과가 줄어든다. region별 분리도가 곧 rollout 안전성.

Canary · Kill Switch · Rollback

패치를 안전하게 굴리는 3대 안전장치. 각각이 서로 다른 시점에 작동한다.

Canary

패치 직후 ~ 첫 1시간. 1% 사용자에 먼저 노출, 크래시·에러율을 실시간 비교. Apple Phased Release의 1% 단계, Google 1% Staged Rollout이 자동 canary 역할.

Kill Switch

"기능을 즉시 끄기". 클라이언트 코드는 그대로 두고 서버 플래그로 신규 기능을 비활성화. 클라이언트 패치 없이 분 단위로 동작 변경 가능. 롤백보다 빠르고 영향 적은 1차 방어선.

Rollback

이전 버전으로 복귀. Steam은 분 단위, 모바일은 staged rollout halt + 이전 버전 재제출(시간 소요). kill switch로 안 잡히는 근본 회귀 시 사용.

자동 트리거

크래시율 임계 초과·결제 실패율 급증·로그인 실패율 급증 등을 모니터링 대시보드에서 자동 감지 → 자동 halt + on-call 알림. 10분 안에 반응할 수 있는 자동화가 핵심.

⚠ PITFALL
"롤백하면 그만이지"는 위험한 사고방식. 모바일 패치 롤백은 이전 빌드를 다시 심사 → 다시 staged rollout이라 수 시간~수 일. 그 동안 영향이 누적된다. kill switch로 끌 수 있게 설계하는 게 본질이다.

Delta 알고리즘 비교

자체 런처·자체 패처를 만들 때 선택하는 binary delta 알고리즘. 각각 trade-off가 있어 워크로드에 맞게 고른다.

· Steam 사용 → SteamPipe만으로 충분

· 게임 데이터(텍스처·음원)만 → 파일 단위 교체로 충분

주의 — 압축 후 바이너리

이미 zstd·brotli·lz4로 압축된 파일에 bsdiff/xdelta를 돌리면 차분이 거의 안 줄어든다. 압축은 raw 데이터의 1bit 변화도 출력 전체로 전파시키기 때문. 차분은 raw 단계에서 만들고 그 다음 압축해야 한다.

비용 최적화

CDN 비용은 게임 라이프사이클 후반에 가장 비대해진다. DAU가 정체된 상태에서도 패치마다 GB가 나가기 때문. 4가지 축으로 최적화한다.

① 차분 패치 적극 사용

풀 빌드 5GB → 차분 200MB로 줄이면 96% 비용 절감. 전체 라이프타임에서 가장 큰 비용 요인. 이걸 못 하면 다른 최적화는 의미 없다.

② 멀티 CDN으로 단가 인하

CloudFront $0.085 + BunnyCDN $0.005 mix로 평균 단가 $0.03. 30~40% 절감 사례. 단, 운영 복잡도 ↑ — RUM·라우팅 자동화 필수.

③ Prefetch · LAN 캐시

점검 종료 시각 동시 트래픽을 분산. 사전 다운로드 옵션을 주면 일부 사용자가 미리 받음. 한국 PC방·일본 인터넷 카페·기업 환경에는 LAN 캐시 적극 권장.

④ region별 CDN 선택

한국 → 통신사 CDN, 동남아 → 지역 특화 CDN, NA·EU → 메이저 CDN. "글로벌 한 곳"보다 region별 mix가 더 싸고 빠른 경우가 많음.
