Payment & IAP
= Revenue Critical Path

● Overview

결제 & IAP 시스템

매출 critical path. 한 건의 사고가 매출 손실 + CS 폭주 + 컴플라이언스 위반으로 직결되는 가장 민감한 시스템. 정합성·가용성·환불·법규 4개 축에서 운영된다.

결제 시스템이란?

정의

유저가 게임 안에서 실제 돈으로 아이템·캐시·구독을 구매하는 흐름. 모바일은 IAP(In-App Purchase)가 강제, PC/Steam은 자체 wallet, 원스토어는 한국 특화.

구성 요소

스토어 결제 → 영수증 발급 → 서버 검증 → 아이템 지급 → 정합성 기록 → (필요 시) 환불 처리. 각 단계가 분산 시스템에서 정확히 한 번 일어나야 함.

관련 시스템

계정 플랫폼(P2, 결제는 user_id 단위), 인벤토리(아이템 지급), 텔레메트리(P6, 결제 이벤트 + LTV), CS 백엔드(환불·문의), 컴플라이언스(가챠 확률·청소년).

일반 e커머스와 다른 점

스토어 강제 IAP(Apple/Google) — 카드 직접 결제 불가능. 환불은 스토어가 결정 → 게임사는 사후 통보. chargeback도 게임사가 직접 막을 수단이 적음.

왜 가장 critical한가

① 매출 직접 영향

1시간 다운 = 매출 0 + 평소 결제하던 유저들의 좌절. 결제는 다른 시스템과 달리 "나중에 다시"가 잘 안 됨(타이밍성 이벤트, 세일 등).

② 정합성 사고의 사용자 영향

"돈은 빠져나갔는데 아이템이 안 옴" — 가장 분노가 큰 케이스. CS 폭주 + 평판 하락 + 환불 압박. 단 한 건이라도 빠지면 안 됨.

③ 법적 리스크

한국: 가챠 확률 표시 의무 (2024-03-22), 청소년 결제 한도, 전자상거래법 보관 의무 5년. 위반 시 시정명령·과태료·형사 처벌까지 가능.

④ 외부 의존도

Apple/Google/Steam 정책 변경, 결제 PG 장애, 환율 변동. 퍼블리셔가 통제 못하는 외부 요인이 매우 많다.

SRE 책임 영역

가용성 SLO

결제 API SLO를 가장 높게 (예: 99.99%+). 1초 단위 latency 모니터링. 장애 시 즉시 점검 페이지 + 사용자 안내.

정합성 모니터링

"결제 완료 vs 아이템 지급 완료" 차이를 분 단위로 추적. 차이 발생 시 자동 알림 + 백오피스 도구로 즉시 보정.

환불 워크플로우

스토어 환불 알림(Apple ASSN, Google RTDN, Steam GetReport) → 자동 아이템 회수 정책 → CS 알림. 처리 누락 모니터링.

컴플라이언스

가챠 확률 검증, 청소년 결제 한도 체크, 보관 기간 자동 정책. 법령 변경 추적은 법무팀과 협업.

영수증 검증 (서버 사이드 필수)

클라이언트가 보낸 "구매 완료" 신호만 믿으면 안 된다. 반드시 서버가 스토어 API에 직접 영수증 검증을 요청해야 함. 클라 검증은 우회·위변조에 쉽다.

영수증 검증 표준 흐름 (Apple StoreKit 2 기준)
# 1. 클라이언트가 구매 후 transaction ID 획득 // StoreKit 2: Transaction.currentEntitlements에서 받음 let transactionId = transaction.id # 2. 클라가 게임 서버에 transaction ID 전송 (영수증 통째로 X) POST /api/payment/verify { "transaction_id": "2000000123456789", "product_id": "com.game.gem_pack_100", "user_id": "u_xxx" } # 3. 게임 서버가 App Store Server API 호출 (JWT 인증) GET https://api.storekit.itunes.apple.com/inApps/v1/transactions/{transactionId} Authorization: "Bearer <JWT signed with In-App Purchase Key>" # 4. Apple 응답: signed transaction info (JWT) { "signedTransactionInfo": "eyJhbGc..." # 디코딩 시 productId, originalTransactionId, purchaseDate, ... } # 5. 게임 서버가 JWT 서명 검증 + 우리 DB의 product_id와 일치 확인 # 6. 멱등 키(transaction_id)로 중복 처리 차단 후 아이템 지급

왜 server-side?

클라이언트는 신뢰 불가. 메모리 변조·중간자 공격으로 가짜 "구매 성공" 신호를 만들 수 있음. 스토어 API로 직접 확인한 결과만 신뢰.

RTDN/ASSN 활용

Google RTDN, Apple App Store Server Notifications로 환불·취소·구독 변경 등 스토어 측 이벤트를 실시간 알림으로 받음. 폴링보다 정확하고 효율적.

중요 데이터 보존

original_transaction_id, purchase_date, product_id를 모두 DB에 저장. 환불·복구·정산·CS 모두 이 키로 추적.

StoreKit 2 마이그레이션

Apple은 기존 /verifyReceipt (StoreKit 1) → 새로운 App Store Server API (StoreKit 2)로 권장 방식 변경. 두 방식 동시 사용은 동기화 지연으로 비권장 — App Store Server API로 먼저 마이그레이션 권장.

멱등성 처리 (Idempotency)

같은 결제가 두 번 처리되면 아이템도 두 번 지급된다. 네트워크 재시도·중복 콜백·사용자 더블 탭 등 모든 상황에서 한 번만 처리가 보장되어야 함.

멱등 키 사용

스토어 영수증의 transaction_id(Apple), order_id(Google), orderid(Steam)를 멱등 키로. DB에 unique constraint를 걸어 두 번째 시도는 자연스럽게 거부.

상태 머신

거래 상태: INITIATED → VERIFIED → GRANTED → COMPLETED. 각 단계 전환은 트랜잭션. 중복 호출 시 현재 상태를 보고 idempotent하게 응답.

분산 락

같은 transaction_id로 동시에 두 요청이 들어올 수 있음. row-level lock 또는 분산 락(Redis SETNX)으로 직렬화.

스토어 콜백 멱등성

Apple ASSN/Google RTDN은 같은 알림을 여러 번 보낼 수 있음(at-least-once). 알림 ID 기반 dedupe 필수.

⚠ PITFALL
"이 결제 처리됐는지 확인 → 없으면 처리"의 두 단계로 나누면 race condition 발생 (같은 시간에 두 요청이 둘 다 "없음" 보고 둘 다 처리). 반드시 unique constraint·atomic upsert·분산 락 중 하나로 직렬화해야 안전.

미수령 아이템 복구

결제는 됐는데 아이템 지급이 실패하는 케이스. 네트워크 끊김·서버 다운·게임 강제 종료 등 다양한 원인. "앱 부팅 시 미지급 영수증 확인"이 표준 패턴.

● Platform Comparison

플랫폼별 결제

같은 IAP라도 플랫폼별 SDK·검증·환불·통보 메커니즘이 모두 다르다. 멀티 플랫폼 출시 게임은 4개 다 운영해야 한다.

Apple App Store IAP (StoreKit 2)

SDK

StoreKit 2 (iOS 15+, 2021년 도입). 클라가 transaction ID만 서버로 전달. 기존 StoreKit 1 (verifyReceipt)는 legacy로 deprecate 진행 중.

서버 검증

App Store Server API 사용. JWT 인증(In-App Purchase Key from App Store Connect). 응답은 signed transaction (JWT 형식, 서명 검증 필수).

이벤트 알림

App Store Server Notifications V2 (ASSN) — 환불, 구독 갱신·취소, gifting, 가격 변경 등 이벤트를 webhook으로 푸시. 페이로드는 signed JWT.

키 관리

App Store Connect에서 In-App Purchase Key 발급. 서버는 이 key로 JWT 서명 + Apple 서버 호출. 키 노출 = 위조 가능이라 KMS 등 안전 보관 필수.

Google Play Billing

SDK

Google Play Billing Library (현재 7.x 시리즈). Android 클라이언트에서 구매 후 purchaseToken 획득.

서버 검증

Google Play Developer API의 purchases.products:get, purchases.subscriptions:get으로 검증. 서비스 계정 + OAuth 2.0 토큰 인증.

이벤트 알림 (RTDN)

Real-time Developer Notifications — Google Cloud Pub/Sub 기반. 구매·환불·구독 상태 변경 시 즉시 알림. 단, 알림은 "변경됐다"만 알려주므로 Developer API로 다시 조회해 완전한 상태 확인 필수.

주의

구매 후 3일 안에 acknowledge() 호출해야 함. 안 하면 자동 환불됨. 일회성 구매는 그 후 consume 처리.

원스토어 (한국 안드로이드)

SDK

원스토어 In-App SDK (Android only). 한국 통신사·금융사 결제 통합. 갤럭시 스토어와는 별개.

서버 검증

원스토어 라이선스 서버에서 영수증 검증. publisher key 기반.

특징

한국 시장 점유율은 작지만 수수료가 낮아 (인앱 20%, 자체결제 5%, 원웹샵 8%) 한국 게임은 동시 출시 흔함. 일부 게임사는 한국 한정 원스토어 우대 가격.

운영 주의

원스토어 결제 다운 시 Google Play로 우회 가능 여부, 영수증 검증 SLA, 정산 주기 등 별도 모니터링 필요.

Steam Microtransactions (ISteamMicroTxn)

Steam에서 게임 안 구매(인게임 캐시·아이템). Steam Wallet에서만 결제 가능 — 외부 PG 사용 불가.

Steam IAP 흐름 (서버 주도)
# 1. 게임 서버가 ISteamMicroTxn/InitTxn 호출 POST https://partner.steam-api.com/ISteamMicroTxn/InitTxn/v3/ { "key": "PUBLISHER_API_KEY", "steamid": "7656...", "orderid": "unique_order_id", "itemid[0]": 100, ... } # 2. Steam이 사용자 클라이언트에 결제 승인 UI 띄움 # 사용자가 승인하면 Steam이 "트랜잭션 승인됨" 콜백 # 3. 게임 서버가 ISteamMicroTxn/FinalizeTxn 호출 → 거래 완료 POST https://partner.steam-api.com/ISteamMicroTxn/FinalizeTxn/v2/ { "key": "PUBLISHER_API_KEY", "orderid": "unique_order_id", "appid": 123456 } # 4. 정산 상태 확인: ISteamMicroTxn/GetReport (최소 일 1회) # 환불·chargeback 등 사후 변경을 reconcile
⚠ NOTE
모바일 IAP와 달리 Steam은 게임 서버가 결제를 주도. orderid는 게임 서버가 발행한 멱등 키. GetReport는 환불·정산 변경 동기화에 필수 — 폴링 누락 시 정합성 깨짐.

● Refunds & Chargeback

환불 & Chargeback

환불은 스토어가 결정하고 게임사는 사후 통보. 핵심은 환불 알림을 놓치지 않고 아이템을 회수하는 자동화. Chargeback은 더 어려운 케이스.

스토어별 환불 정책

Apple App Store

Apple Support에 사용자가 환불 신청. Apple이 케이스별 재량으로 결정. 결과는 ASSN V2의 REFUND 알림으로 통보.

Google Play

Google Play 또는 Play Console에서 환불. RTDN으로 알림. 구매 후 48시간 안 자동 환불 옵션도 있음.

Steam

2시간 미만 플레이 + 14일 이내 자동 승인. 그 외는 수동 검토. EA → 정식 전환 시 EA 플레이타임도 누적(2026 변경).

원스토어

한국 전자상거래법 준수. 디지털 콘텐츠는 사용 시 환불 제한 가능. CS 통한 케이스별 처리가 흔함.

환불 시 아이템 회수 정책

환불됐는데 게임 내 아이템은 그대로 두면 사용자가 매번 환불 후에도 아이템을 챙겨감. 그렇다고 무조건 회수하면 사용자 분노.

유형별 처리

미사용 캐시: 자동 회수. 이미 사용한 캐시·아이템: 회수 시도 (인벤토리에 남았으면 제거). 소비된 아이템(스킬 강화·1회성 부스터): 보통 회수 불가, 계정 마이너스 처리 또는 경고.

회수 불가 시

아이템이 이미 다른 형태로 변환됐거나(가챠 결과) 다른 사용자에게 양도됐으면 회수 어려움. 계정에 마이너스 잔액 + 다음 결제 시 차감, 또는 계정 정지 정책.

반복 환불 사용자

여러 번 환불 + 즉시 사용 패턴 = 의도적 어뷰징. 사용자 단위로 환불 횟수 추적 → 임계 초과 시 결제 차단 / 계정 검토.

SRE 자동화

환불 알림 수신 → 정책 매트릭스 조회 → 자동 회수 시도 → 실패 시 CS 큐에 이관. 모든 단계 감사 로그 + 사용자 통보.

Chargeback (지급 거절) 대응

사용자가 카드사에 직접 결제 취소 요청 → 스토어가 게임사에 chargeback 통보. 스토어 환불보다 후속 처리 부담이 크다.

처리 절차

스토어 chargeback 알림 → 자동 아이템 회수 + 계정 일시 정지 → CS가 본인 확인 (정당한 chargeback인지 사기인지) → 대응 결정.

친족 결제 사기

"자녀가 부모 카드로 결제했다"며 chargeback. 한국은 미성년자 결제 무효화 가능 → 스토어가 환불 인정. 게임사는 손실 + 아이템 회수 시도.

고의적 chargeback (사이버샵)

아이템 받은 뒤 카드사에 "도용됐다" 신고 → 환불받고 아이템 사용. 패턴 탐지 시스템으로 사전 차단해야 함.

스토어 정책 영향

chargeback 비율이 높으면 스토어가 게임사 계정에 페널티 부과 가능. chargeback 비율을 KPI로 관리하는 것이 일반적.

환불 사기 패턴

패턴설명탐지·대응
버스트 결제 + 환불한 번에 대량 결제 후 즉시 환불 신청일정 임계 초과 결제는 지연 지급
가챠 결과 후 환불고가 아이템 안 나오면 환불가챠는 결제 즉시 컨슘 처리, 환불 시 회수
아이템 양도 후 환불아이템을 다른 계정 이전 후 본인 계정 환불최근 결제 N일 내 양도 제한
친족 결제 + 환불아이 명의로 결제 후 부모가 환불 요청고가 결제 시 추가 본인 확인
다계정 어뷰징여러 계정으로 신규 보너스만 받고 환불디바이스 / 결제 수단 단위 추적
★ NOTE
완벽한 사기 차단은 불가능. 매출 보호와 사용자 경험의 균형이 핵심. 너무 빡빡하면 정상 사용자가 불편해짐. 환불 사기 비율 < N%를 KPI로 두고 정기 검토.
● Consistency & Compliance

정합성 & 컴플라이언스

분산 환경에서 결제 정합성을 보장하는 기법, 그리고 한국 시장의 법규 준수.

분산 트랜잭션 — 결제↔지급 정합성

결제 시스템과 인벤토리 시스템이 분리되어 있고 둘 다 다른 DB에 있을 때, "결제됐는데 아이템 안 들어옴" 또는 "아이템은 들어왔는데 결제 기록 없음"을 방지해야 함.

Saga 패턴

결제 → 아이템 지급을 일련의 단계로 처리. 각 단계 실패 시 보상 트랜잭션(compensating transaction) 실행. 실무에서 가장 흔한 패턴.

Outbox 패턴

결제 트랜잭션 안에서 outbox 테이블에 "지급 명령" 기록 → 별도 워커가 outbox 읽어 인벤토리 호출. 결제와 지급이 atomically 결정.

정합성 모니터링

"결제 완료된 transaction 중 N분 내 아이템 지급되지 않은 것" 쿼리를 분 단위로. 임계 초과 시 알림 + 백오피스 도구로 수동 보정.

일일 reconciliation

매일 새벽 (1) 스토어 정산 데이터 + (2) 자체 결제 DB + (3) 인벤토리 지급 로그를 cross-check. 차이 발견 시 즉시 검토. 회계·세무에도 사용.

⚠ PITFALL
"두 시스템에 순차로 쓰면 되겠지"는 흔한 함정. 첫 번째는 성공하고 두 번째 직전에 서버가 죽으면 정합성 깨짐. 분산 트랜잭션은 반드시 saga / outbox 등 정형 패턴으로 설계해야 한다.

한국 컴플라이언스 (2026-05 기준)

가챠 확률 표시 의무

2024-03-22 시행. 캡슐형·강화형·합성형 모든 확률형 아이템의 확률 정보를 게임 본체·홈페이지·광고 모두에 표시. 표시 vs 실제 일치 검증은 텔레메트리 책임 (P6 가이드 상세).

청소년 결제 한도

2019-06-27 성인 PC 온라인 게임 50만원 한도 폐지. 2026 시점에서는 청소년(만 18세 미만)은 월 7만원 한도 유지(웹보드 게임 등). 2024-01-02 청소년 정의가 변경(만 19세가 되는 해 1월 1일 미만)되어 한도 적용 연령 기준도 갱신.

전자상거래법 보관 의무

결제·공급 기록은 5년 보관 의무(전자상거래등에서의 소비자보호에 관한 법률). 게임은 escrow·소비자 피해 보상 보험 면제(실물 배송 없는 서비스). 보관 기간은 PIPA의 즉시 삭제 의무에 우선.

약관·고지

가입 시 결제 관련 약관 동의 필수. 결제 한도, 환불 정책, 미성년자 결제 시 부모 동의 안내 등을 명확히 게시. 약관 변경 시 30일 전 통지 의무.

★ NOTE
한국 컴플라이언스는 변동이 잦다. 2024년에 청소년 정의·확률 표시 의무가 바뀌었듯, 매년 갱신을 추적해야 함. 법무팀과의 정기 동기화 + 게임물관리위원회·문화체육관광부 공지 모니터링 필수. 글로벌 출시 시 region별 매트릭스 별도 관리.
