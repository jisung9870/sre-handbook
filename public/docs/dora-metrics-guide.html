● DORA Overview

DORA Metrics란?

Google Cloud의 DevOps Research and Assessment 팀이 7년 이상의 연구와 32,000명 이상의 전문가 설문을 기반으로 정립한 소프트웨어 딜리버리 성과 측정 프레임워크.

DORA가 뭔가요?

기원

Nicole Forsgren, Jez Humble, Gene Kim이 저술한 「Accelerate」 책에서 체계화. 현재는 Google Cloud 산하 팀으로 매년 State of DevOps Report를 발간한다.

핵심 목표

"얼마나 빠르게, 얼마나 안정적으로 소프트웨어를 딜리버리하는가?"를 데이터 기반으로 측정하고 개선의 방향을 제시한다.

두 축: 속도(Velocity) × 안정성(Stability)

DORA는 4개 메트릭을 2개 축으로 분류한다. 핵심은 두 축이 trade-off가 아니라는 것 — Elite 팀은 양쪽 모두 높다.

왜 필요한가?

객관적 성과 측정

"우리 팀 잘하고 있나?"를 감이 아닌 데이터로 답할 수 있다. 벤치마크 대비 자사 위치를 파악.

병목 식별

배포는 빠른데 실패율 높으면 테스트 부족, 리드 타임 길면 코드 리뷰 문제 등 개선 포인트를 특정.

투자 근거

CI/CD 개선, 모니터링 도입 등의 투자에 대해 경영진에게 정량적 근거 제시.

지속적 개선 문화

목표를 세우고 추이를 추적하면 팀이 자연스럽게 개선 방향에 정렬. 단, 팀 간 비교보다 자기 추이에 집중.

⚠ 주의
DORA 메트릭을 팀 간 순위 경쟁 도구로 사용하지 말 것. Google DORA 팀도 2023년부터 이를 명시적으로 경고했다. 리그 테이블은 부적절한 타겟팅과 역효과를 유발한다.

● Performance Levels

DORA 성과 레벨 벤치마크

각 메트릭에 대해 Elite / High / Medium / Low 4단계로 팀 성과를 분류한다.

벤치마크 기준표

메트릭EliteHighMediumLow
Deployment Frequency하루 여러 번하루~주 1회주~월 1회월~6개월 1회
Lead Time for Changes< 1시간1일~1주1주~1개월1개월~6개월
Change Failure Rate0~15%16~30%16~30%46~60%
Recovery Time (MTTR)< 1시간< 1일< 1주6개월 이상

해석 가이드

Elite 팀 특징

속도와 안정성 모두 최상위. CI/CD 완전 자동화, 변경이 작고 빈번하며 장애 시 즉시 자동 롤백. 조직 전체 성과가 2배 높은 것으로 연구됨.

흔한 안티패턴

배포 빈도만 높이고 CFR이 치솟는 경우 → 테스트 부재. 리드 타임은 짧지만 MTTR이 긴 경우 → 모니터링·알림 미비. 4개 메트릭을 함께 봐야 한다.

4개 메트릭은 독립이 아닌 연결

배포 빈도(DF)를 높이면 변경 단위가 작아지고, 작은 변경은 실패율(CFR)을 낮추며 복구(MTTR)도 빨라진다. 리드 타임(LT)이 짧으면 핫픽스도 빠르므로 MTTR에 기여. 하나를 개선하면 나머지가 따라오는 선순환 구조.

● Implementation

실무 적용 가이드

오픈소스 기반 LGTM 스택에서 DORA 메트릭을 수집·시각화하는 방법과 개선 전략.

측정 도구 매핑

메트릭데이터 소스수집 방법
Deployment FreqArgoCD Sync / Jenkins JobArgoCD Notification → Webhook → Prometheus push
Lead TimeGit commit + Deploy timestampCI에서 커밋 SHA의 authored_date와 deploy_time 차이 계산
Change Failure RateIncident 티켓 / RollbackGrafana Alerting → Incident 카운트 / ArgoCD rollback webhook
Recovery TimeAlert fired → resolvedGrafana Alerting의 alert duration 기록
💡 TIP
Grafana Alerting Webhook을 트리거로 사용하면 Alertmanager 없이도 인시던트 시작/종료 이벤트를 기록할 수 있다.

Prometheus 메트릭 설계 예시

# ── Deployment Frequency ── # CI/CD 파이프라인에서 배포 성공 시 push dora_deployments_total{ service="api-server", environment="production", result="success" } 142 # ── Lead Time (Histogram) ── dora_lead_time_seconds_bucket{ service="api-server", le="3600" } 128 # ── Change Failure Rate ── dora_deployments_total{result="failure"} 8 # CFR = failure / (success + failure) × 100 # ── Recovery Time (Histogram) ── dora_recovery_time_seconds_bucket{ service="api-server", le="3600" } 15
RECORDING RULES 예시
groups: - name: dora-metrics rules: # 일일 배포 빈도 - record: dora:deployment_frequency:rate1d expr: | increase(dora_deployments_total{ result="success"}[1d]) # 변경 실패율 (%) - record: dora:change_failure_rate:ratio expr: | increase(dora_deployments_total{ result="failure"}[30d]) / increase(dora_deployments_total[30d]) # 리드 타임 P50 - record: dora:lead_time:p50 expr: | histogram_quantile(0.5, rate(dora_lead_time_seconds_bucket[30d])) # 복구 시간 P50 - record: dora:recovery_time:p50 expr: | histogram_quantile(0.5, rate(dora_recovery_time_seconds_bucket[30d]))

메트릭별 개선 전략

DF 높이기

배포 단위를 작게 (마이크로 배포). Feature Flag로 릴리스 분리. GitOps(ArgoCD) 자동 Sync 활용.

LT 줄이기

Queue Time 줄이기가 핵심. PR 리뷰 SLA(4시간 이내). 트렁크 기반 개발. CI 병렬화 + 빌드 캐시.

CFR 낮추기

자동화 테스트 커버리지 확대. Canary / Blue-Green 배포. Staging 검증 필수화.

Recovery Time 줄이기

LGTM 기반 Observability. 자동 롤백 (ArgoCD). Runbook 정비. 온콜 체계 + 인시던트 Postmortem.

💡 핵심 인사이트
대부분의 병목은 "작업 처리 시간"이 아닌 "대기 시간(queue time)"에 있다. 코드 리뷰, 승인, 배포 대기 시간을 줄이는 것이 가장 효과적.

SLO와의 연결

CFR이 높으면 Error Budget을 빠르게 소진 → 배포 동결. SLO 기반 운영에서 DORA 메트릭은 Error Budget 소진의 선행 지표.

GitOps와의 연결

ArgoCD Sync 이벤트 = 배포 이벤트. Notification Controller Webhook으로 Prometheus에 push하면 DF/LT를 자동 수집.
