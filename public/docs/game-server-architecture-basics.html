● Overview

게임 서버 아키텍처 — 퍼블리셔 SRE의 진입점

퍼블리셔 SRE는 보통 게임사가 만든 서버 코드를 직접 수정하진 않지만, 장르·토폴로지·세션 모델·동기화 방식에 따라 운영 방식이 완전히 달라진다. 이 문서는 게임사와 협업할 때 필요한 최소한의 게임 서버 구조 지식을 정리한다. (2026-05 기준)

왜 퍼블리셔 SRE도 게임 서버 구조를 알아야 하나

운영 SLA가 다르다

MMO는 한 region에서 24/7 운영, FPS 매치는 5~30분짜리 짧은 인스턴스. 같은 다운타임이라도 영향이 완전히 다름.

스파이크 패턴이 다르다

MOBA·FPS는 "큐 진입 → 매칭 → 인스턴스 생성"의 갈고리형 트래픽, MMO 신규 서버 오픈은 채널/필드 동시 진입 폭증.

롤백·핫픽스 방식이 다르다

인스턴스 매치는 신규 인스턴스부터 새 빌드 적용 가능, persistent world는 점검 + 전체 롤아웃 강제.

관측 메트릭 자체가 다르다

HTTP 5xx로 게임 서버 health 못 잡음. ms-per-tick, AOI 부하, 패킷 손실, 큐 대기시간 같은 게임 도메인 메트릭이 핵심.

퍼블리셔의 위치

대부분의 한국 퍼블리셔는 (1) 게임사가 빌드한 dedicated 서버 바이너리를 받아 (2) 자사 클러스터(또는 매니지드: Agones/GameLift/Multiplay)에 배포하고 (3) 트래픽 매칭·결제·CDN·텔레메트리 같은 공통 플랫폼과 연동한다. 게임 코드를 못 고쳐도 "어떤 토폴로지의 게임이냐"에 따라 캐파시티·관측·온콜 전략을 바꿔야 한다.

토폴로지 분류 — Client-Server vs P2P

Client-Server (C/S)

중앙 서버가 권위(authoritative)를 가지고 모든 클라이언트의 입력을 검증·시뮬레이션. 대다수 상용 게임의 표준. 부정행위 방어와 일관성에 유리하지만 서버 비용·연산 부담이 크다.

Peer-to-Peer (P2P)

클라이언트끼리 직접 연결. 중앙 서버가 없어 비용이 낮고 latency가 가장 낮을 수 있지만 부정행위·NAT traversal·호스트 마이그레이션 문제. 격투(롤백 netcode)·소규모 협동·콘솔 LAN에 주로 사용.

Hosted Relay

P2P처럼 보이지만 중간에 클라우드 relay가 패킷을 중계 (예: Photon Cloud, Steam SDR P2P 모드). NAT 문제 해결 + IP 은닉. 권위는 호스트 클라이언트가 가짐 — Listen server 변형.

Hybrid

로비·매칭·인벤토리는 클라우드 C/S, 인게임 입력만 P2P/Relay (예: Among Us, 일부 격투). 비용 vs 보안 절충안.

★ 퍼블리셔 관점
P2P 게임을 운영해도 로비·매칭·결제·통계는 무조건 클라우드 인프라가 필요. "P2P니까 우리 서버 부담 없다"는 오해 — relay·NAT punching 실패율, 호스트 마이그레이션 사용자 경험 모두 SRE가 챙겨야 한다.

Dedicated 운영 포인트

오토스케일링 fleet, drain·rolling update, 노드 장애시 인스턴스 마이그레이션 필요. 장애 영향: 한 인스턴스 = 한 매치(수~수십 명). MMO는 한 채널 = 수백~수천 명.

Listen 운영 포인트

호스트 클라가 떠나면 호스트 마이그레이션 또는 매치 종료. 퍼블리셔는 서버 캐파시티는 안 들지만 "호스트 이탈 시 사용자 경험" 메트릭(매치 중단율) 추적 필요.

Relay 운영 포인트

Steam SDR 사용 시 Valve 백본이 알아서 routing. 자체 relay (Photon Bolt 등)면 PoP별 capacity·latency 모니터링.

혼합

FPS도 매칭/로비는 dedicated 서버 + 매치는 dedicated, 친구방은 listen, 외부 노출 IP는 SDR로 가리는 식의 조합이 일반적.

퍼블리셔 SRE의 관여 범위

퍼블리셔 책임

인프라 (k8s/VM/매니지드), 네트워크·CDN, 매칭·로비, 통합 계정, 결제·CS·BI, 모니터링·알림, 점검·공지, anti-cheat 통합

게임사 책임

게임 서버 코드, 시뮬레이션·물리, AOI·인벤토리 로직, 게임 DB 스키마, 패치 빌드, 게임 내 밸런싱

공동 책임 (회색지대)

매치메이킹 룰, 캐파시티 산정, 점검 윈도우 결정, 핫픽스 판단, 장애 RCA, 글로벌 region 운영 정책

★ 협업 원칙
게임 코드를 못 고쳐도 "어떤 모양의 트래픽인지"는 반드시 알아야 capacity·관측·온콜 전략이 나온다. 게임사와의 첫 미팅에서 (1) 토폴로지 (2) 틱 레이트 (3) 평균 세션 수명 (4) 매칭 정책 (5) 영속 데이터 모델 — 5가지를 반드시 확인하자.
● Simulation

게임 루프 & 시뮬레이션

게임 서버는 일반 백엔드와 다른 "시간 모델"을 가진다. tick 단위로 모든 것이 굴러가고, 한 tick 안에 끝나야 할 일을 못 끝내면 시뮬레이션이 밀린다. 이 섹션은 SRE가 알아야 할 최소한의 시뮬레이션 모델을 다룬다.

서버 게임 루프 — 한 tick 안에서 일어나는 일

Recv Input네트워크 큐
Validate입력 검증
Simulate물리·AI·게임 로직
AOI Cull가시 영역 추출
Snapshot스냅샷 직렬화
Send플레이어별 송신
★ 핵심
이 6단계가 한 tick (예: 30Hz면 33ms) 안에 끝나야 한다. 60Hz면 16ms, 128Hz는 7.8ms. 못 끝내면 다음 tick이 밀리며 클라에서는 lag·rubberbanding으로 보인다.

Fixed vs Variable timestep

Fixed timestep

매 tick 정확히 같은 시간(예: 16.67ms) 진행. 결정성 보장, 디버깅 쉬움. 서버는 거의 모두 fixed timestep. 부하가 몰려도 "한 tick 분량은 정해져 있다".

Variable timestep

실제 경과 시간만큼 진행. 클라이언트 렌더링에 주로 사용. 서버에서 사용 시 빠른 머신은 더 자주 시뮬레이션 → 결정성 깨짐. 거의 사용 안 함.

Fixed timestep 의사 코드
accumulator = 0
TICK_DT = 1/30      # 30Hz
while running:
    delta = now() - last
    accumulator += delta
    while accumulator >= TICK_DT:
        simulate(TICK_DT)        # 항상 같은 dt
        accumulator -= TICK_DT
    render_or_send()

Deterministic 시뮬레이션

같은 입력 → 같은 결과를 모든 머신에서 보장. RTS·격투·롤백 netcode의 전제 조건.

왜 어려운가

부동소수점(float) 비결정성, 컴파일러 최적화 차이, 멀티스레드 실행 순서, OS 콜 차이. CPU 아키텍처 (x86 vs ARM) 차이까지.

해결책

Fixed-point 산술 사용, 단일 스레드 시뮬레이션, 시드 고정 RNG. Photon Quantum이 이 모델의 대표 — Unity Verified Solution.

장점

대역폭 극소 (입력만 전송), 리플레이 = 입력 로그, 치팅 검출 강력 (모든 피어가 desync 발견).

단점

구현 난이도 매우 높음, "한 명 입력 안 오면 전체 대기" → lockstep 변형 (rollback) 필수.

AOI (Area of Interest) — 그리드 · 셀 · 옥트리

100명이 한 월드에 있어도 한 플레이어가 "보는" 건 주변 20~30명뿐. 전부 보내면 대역폭 폭발. AOI는 "이 플레이어에게 의미있는 다른 객체"만 추려내는 자료구조.

Grid (격자)

2D/3D 공간을 균일한 셀로 분할. 객체는 자기 셀 + 인접 8셀 검사. 구현 단순, MMO 평면 월드에 적합.

Quadtree / Octree

밀도에 따라 동적 분할. 객체가 한쪽에 몰려있으면 효율적. 구현 복잡, 동적 객체 갱신 비용.

Spatial Hash

좌표 → 해시 셀. O(1) 조회. 동적 환경 친화적, 충돌 처리에도 사용.

Interest Group

공간이 아닌 "그룹" (파티·길드·채팅방) 단위 관심. MMO에서 채팅·길드 알림에 사용.

★ 운영 시 보는 메트릭
AOI 결과 객체 수의 평균·max·p99. 한 플레이어가 인지하는 객체가 폭증 (예: 공성전 한 곳 모임)하면 송신 패킷이 폭증 → 서버 CPU·대역폭 동시 폭증. 알림 임계 설정 필수.

ECS (Entity-Component-System) — SRE가 알아야 할 만큼만

대규모 게임 서버의 표준 아키텍처. 객체(Entity)는 ID, 데이터(Component)는 별도 테이블, 로직(System)은 일괄 처리. 캐시 친화적이라 수만 entity 시뮬레이션에 유리.

운영 관점 1

"한 system이 한 번에 모든 entity 처리" → 처리 시간이 entity 수에 비례. 캐파시티 모델링이 직관적.

운영 관점 2

System별로 시간 측정 가능 → 어떤 게임 로직이 무거운지 SRE가 게임사에 피드백 줄 때 유용.

★ 코드 못 봐도
"이 system이 평균 8ms / p99 25ms 걸린다"는 메트릭만 있으면 SRE는 캐파시티 한계 추산 가능. 게임사에 "system별 prometheus 메트릭 노출해 달라" 요청은 합리적이다.

세션 · 룸 · 인스턴스 관리

"한 게임 서버 = 한 매치"인 인스턴스 모델부터 "한 서버 = 채널 = 수천 명"인 persistent world까지, 운영 모델이 완전히 다르다. 이 섹션은 게임 세션 lifecycle과 fleet 운영 도구를 정리한다.

Queue → Match latency

매치 메이킹 큐 평균 대기시간. 사용자 체감 1순위 메트릭. 보통 30초~3분 목표. 시간대별·MMR별 분포 추적.

Allocate latency

매치가 만들어진 뒤 서버 인스턴스 할당까지의 시간. Agones/GameLift는 sub-second 목표. 1초 넘으면 사용자 이탈.

Connect → Play

클라가 서버에 붙고 게임 시작까지. 로딩 화면 시간. CDN·맵 데이터 캐시·로딩 최적화 영향.

End → Settle

매치 결과 → BI/랭킹/보상 DB 반영 완료. 결제·인벤토리 일관성 보장 필요.

Allocator · Drain · Reservation 패턴

Allocate

매칭이 끝나면 fleet에서 "준비된 GameServer 인스턴스 1개 잡아줘". Agones는 GameServerAllocation CRD로 sub-second에 처리.

Reserve (예약)

매치가 만들어지기 전에 미리 인스턴스 예약. cold start 회피용. Agones 1.x부터 지원.

Drain (배수)

특정 노드의 GameServer를 새로 받지 않게 막고, 끝나는 대로 종료. 노드 점검·롤링 업그레이드 필수 패턴.

Counter / List

(Agones 신규) GameServer 내 상태(현재 플레이어 수 등)를 k8s API로 노출. 외부 매처가 "남은 자리 있는 서버" 검색 가능.

Fleet 오토스케일링

웹 서비스의 CPU 기반 HPA로는 게임 서버를 못 다룬다. 핵심 신호는 "비어있는 인스턴스 수 (Ready 상태)" — 신규 매치 수요를 흡수할 buffer.

Buffer-based

Ready 인스턴스가 N개 미만이면 fleet replicas 증가. Agones BufferPolicy 표준 패턴. 항상 일정한 매칭 수용 capacity 유지.

Capacity-based

전체 fleet의 사용률(allocated/total)이 임계 넘으면 확장. MOBA·BR처럼 매치 수명 짧으면 변동 큼.

Schedule-based

이벤트·런칭·평일 저녁 피크 등 시간대별 스케줄 기반 사전 확장. 신규 콘텐츠 패치일에 필수.

Spot/Preemptible 혼합

FleetIQ는 Spot으로 비용 70% 절감 가능. 단 매치 진행 중 회수되면 매치 종료 → critical 매치 (랭크전)는 on-demand fleet 분리.

★ Cold start 함정
Container 이미지 풀, 게임 데이터(asset) 로딩, 초기화 스크립트까지 합치면 한 GameServer가 Ready되는데 30초~수 분. Buffer 부족하면 매치 들어왔는데 서버 없는 상황. 항상 Ready 인스턴스를 buffer로 미리 띄워라.
● State Sync

상태 동기화 & 일관성

실시간 게임은 "서버 진실"을 클라에 어떻게 전파하느냐의 문제다. Snapshot, Prediction, Reconciliation, Lockstep, Rollback 같은 용어가 무엇이고, 어떤 게임 데이터에는 strong consistency가 필요한지 정리한다.

★ Lag compensation
FPS에서 "내 화면에 적이 보일 때 쏘면 맞춰주는 것"은 서버가 과거 시점(예: 80ms 전) 적 위치를 기억해 hit detection 해주는 별도 메커니즘. 사용자 ping이 들쭉날쭉하면 효과가 약해진다.

Deterministic Lockstep & Rollback Netcode

Lockstep

모든 피어가 같은 입력 받고 동시 진행. RTS·MOBA 일부 (StarCraft·LoL 일부 변형). 한 명 입력 안 오면 전체 대기 → 가장 느린 사람 기준.

Rollback

입력 안 와도 추정해서 진행, 늦게 도착하면 시점 되감기 후 재시뮬레이션. 격투 게임 표준 (GGPO, SF6, Tekken 8). 부드러운 경험, 결정성 필수.

★ Photon Quantum
Photon Quantum은 fully deterministic predict/rollback 엔진. e-sport·격투·RTS 같이 frame-perfect 동기화가 필요한 장르 대상. Photon Fusion(state transfer + client prediction)과는 정반대 방식 — 같은 회사 두 제품의 용도 차이 이해 필요.

★ 운영 함정

● Global Network

글로벌 분산 & 네트워크 토폴로지

플레이어와 서버가 같은 region에 있다고 보장 못 한다. 글로벌 라이브 게임은 region 선택, edge relay, cross-region matchmaking 모두 SRE가 설계한다.

Region 선택 전략 — 어떻게 결정하나

Latency-based

클라가 후보 region에 ping 보내고 가장 빠른 곳으로. FPS·격투·MOBA 표준. 클라가 측정·결정.

Geo-IP

클라 IP의 GeoIP 조회로 자동 매핑. 단순, 부정확 가능 (VPN·모바일 캐리어 NAT).

Player-choice

"한국 / 일본 / 동남아" 명시 선택. MMO 표준. 친구와 같은 서버 보장.

Hybrid (latency + queue health)

가장 빠른 region이 큐가 길면 차순위로 라우팅. FPS·BR에서 매칭 시간 vs latency 트레이드오프.

Edge / PoP relay — Steam Datagram Relay · Photon Cloud

Steam Datagram Relay (SDR)

Valve 백본을 통해 게임 트래픽 라우팅. 클라는 IP 모름 (DDoS 방어). 클라가 가까운 PoP 선택 → Valve 사설망 → 서버. 단, 서버는 AWS·Azure·GCP 데이터센터 안에 있어야 함.

Photon Cloud

Photon Realtime/Fusion이 글로벌 region cluster 운영. 클라는 region selector로 가장 빠른 곳 자동 선택. 모바일 협동 게임 사실상 표준 (Among Us·일부 한국 모바일).

AWS GameLift FleetIQ

Spot 인스턴스 가용성·가격을 region·zone별 추적해 최적 fleet 자동 할당. 비용 최적화에 강점.

Anycast (Cloudflare Spectrum 등)

같은 IP가 전 세계 여러 PoP에 advertise되어 BGP가 최적 경로로 라우팅. 일반 웹은 표준이지만 게임 UDP는 sticky session 보장 어려움.

★ SDR 활용 패턴
Steam에 출시하는 PvP 게임은 SDR 통합이 사실상 의무. (1) IP 노출 차단 → DDoS 방어 (2) 잡음 많은 ISP 라우팅 우회 → ping 개선. 단점: AWS/Azure/GCP 데이터센터에만 호스팅 가능 (베어메탈·온프레미스 직접 사용 어려움).

Cross-region matchmaking — 글로벌 매칭의 함정

한국·일본·동남아 사용자를 한 매치에 묶을 것인가? 매칭 시간 vs latency 영원한 트레이드오프.

Region-locked

같은 region 사용자끼리만. 매칭 시간 길어질 수 있음, latency 일정. 경쟁전 표준.

Adjacent region 허용

한국+일본, EU 내부 등 인접 region 매치. 매칭 풀 확장, 최대 RTT 80~100ms 정도 보장.

Global pool

전 세계 한 풀. 매칭 시간 가장 짧음, latency 무보장. 캐주얼·이벤트 모드.

Time-based 확장

큐 대기 시간이 일정 임계 넘으면 매칭 풀 확장. FlexMatch·Open Match에서 룰로 표현.

★ SRE가 통제 가능한 부분
네트워크 경로(region 선택, edge PoP)와 서버 처리(시뮬레이션·송신 큐)만이 SRE 통제 영역. 클라 모니터·라스트마일은 못 건드림. 따라서 region 배치·tick 부하 관리가 가장 큰 레버.

게임 특화 헬스체크 — Liveness/Readiness 게임 버전

Liveness

"프로세스가 살아있나" — 단순 ping은 부족. tick loop가 돌고 있나를 확인. 마지막 tick 타임스탬프가 N초 이전이면 dead로 간주.

Readiness

"새 매치 받을 준비 됐나" — 게임 데이터 로드 완료, anti-cheat 핸드셰이크 OK, 외부 서비스(매치메이킹·로비) 연결 확인까지.

Allocated 상태

매치 진행 중. k8s가 Pod 죽이면 매치 강제 종료 → 사용자 incident. Agones는 GameServer 상태 머신으로 이를 명시적 보호.

Drain 상태

새 매치 안 받음, 진행 중인 매치 끝나면 종료. 점검·롤링 업그레이드 패턴. Agones GameServer 상태 머신 표준.

★ k8s liveness 함정
일반 웹 서비스의 readinessProbe(HTTP 200)로 게임 서버 health를 잡으면 매치 중에 readiness 떨어졌다고 traffic 끊는 사고 발생. 게임 서버는 일반 webserver와 health 모델이 다르다 — Agones GameServer SDK가 표준 인터페이스 제공.

Crash · Replay · Desync 분석

Crash dump 수집

매치 진행 중 서버 프로세스 죽으면 core dump + 매치 ID·플레이어 ID·최근 tick 로그를 같이 저장. 원인 분석에 결정적.

Replay 시스템

매치 입력 로그를 저장하면 결정성 게임은 그대로 재현 가능. 어뷰징·어그로 분석·CS 응대에 사용. e-sport 송출용.

Desync 추적

Lockstep 게임에서 한 피어가 다른 결과 → 게임 종료. 발생 시 모든 피어의 상태 해시·입력 로그 수집해 원인 추적. 부동소수·플랫폼 차이 등.

플레이어 신고 → 매치 ID 연동

플레이어 신고가 매치 ID·시각·당사자 ID로 즉시 매치 로그 검색 가능해야 함. 운영자 도구 핵심.

점검 · 핫픽스 · 롤백 — 게임 서버 vs 클라 버전 정합

인스턴스 매치 게임

새 빌드는 신규 fleet에 배포 → 점차 라우팅 비율 증가 (canary). 기존 매치는 끝까지 구버전. 점검 없이 롤링 업데이트 가능.

Persistent world 게임

채널 단위 점검, DB 마이그레이션 동반 시 전체 점검. 점검 시간 = 매출 손실 = 사용자 이탈 → 점검 시간 단축이 SRE KPI.

클라 버전 강제 매칭

매칭 룰에 클라 버전 강제 — 구버전 클라는 신규 매치 못 들어감. 자동 패치 안내.

긴급 롤백

핫픽스가 더 큰 사고 만들면 즉시 이전 빌드로 fleet 교체. 새 빌드 배포 전 이전 빌드 이미지 보관은 기본.

★ 클라-서버 버전 매트릭스
"클라 v5.2.1 ~ v5.2.5는 서버 v5.2.x와 호환" 같은 호환 매트릭스를 명시 관리. 새 서버 배포가 구 클라를 끊어먹는 사고가 가장 흔한 incident. 점검 직후 사용자 폭증 시 OAuth·로비 부하도 따로 관찰.

Capacity 모델 — CCU per node, instance per region

"우리 게임은 한 노드에 몇 명 들어가나?"는 모든 capacity 산정의 출발. 게임사가 부하 테스트 데이터로 알려주는 게 정상.

캐파시티 산정 공식 (FPS·MOBA 인스턴스 모델)
예측 동시 매치 수 = (PCU × 매치 평균 수명) / (매치당 인원 × 60)
필요 인스턴스 수  = 예측 동시 매치 × (1 + buffer ratio)
필요 노드 수       = 필요 인스턴스 × (1 + 여유) / 노드당 인스턴스
권장 region 분산  = 필요 노드 × region별 PCU 비율

# 예: BR 게임, PCU 100,000명, 매치당 100명, 평균 수명 25분
동시 매치 ≈ (100000 × 25) / (100 × 60) ≈ 417개
buffer 30% 적용 → 542 인스턴스
한 노드 8 인스턴스 → 68 노드
KR 50% / JP 30% / SEA 20% 비율로 분산
★ MMO 캐파시티는 또 다르다
MMO는 채널당 동시 인원 (예: 500명)이 핵심. 한 채널 = 한 프로세스. 채널 수 = ceil(PCU / 채널당 인원) × 채널 capacity buffer. 채널 추가는 "노드 한 대 더" 수준이 아니라 게임사 운영 결정 사항.

퍼블리셔 운영팀 ↔ 게임 개발사 협업 모델

역할 분리

퍼블리셔 SRE = 인프라·플랫폼·통합 모니터링. 게임사 = 게임 코드·시뮬레이션·게임 DB·anti-cheat 룰. 회색지대는 RACI 명문화.

장애 핸드오프

1차 트리아지: 퍼블리셔 SRE (인프라·네트워크 원인 배제). 게임 코드 의심이면 게임사 온콜에 핸드오프. 핸드오프 SLA 명시 (예: 평일 30분, 야간 2시간).

공통 관측

Grafana·BI 대시보드는 양사 공유. "각자 보는 메트릭이 다르면 협업 불가". 매치 ID·플레이어 ID로 양 시스템 로그 추적 가능.

런칭 전 합동 훈련

부하 테스트 + 인시던트 시나리오 합동 훈련. 게임사 코드의 한계를 미리 파악해야 capacity·온콜 인원 산정 가능.

★ 협업의 첫 단추
킥오프 미팅에서 (1) 토폴로지 (2) 틱 레이트 (3) 평균 세션 수명 (4) 매칭 정책 (5) 영속 데이터 모델 + (6) 운영 도구(Agones/GameLift/Multiplay/자체) — 6가지를 표로 정리해 양사가 합의하는 데서 시작. 이게 capacity·관측·온콜 전부의 기반.
