● Service Mesh Fundamentals

왜 Service Mesh가 필요한가

서비스 간 통신의 복잡성을 인프라 레이어로 내려 앱 코드에서 분리하는 아키텍처 패턴.

Motivation

왜 Service Mesh인가?

마이크로서비스가 늘어나면 서비스 간 통신(East-West)이 가장 복잡한 문제가 된다
마이크로서비스 환경에서 각 서비스가 재시도, 타임아웃, 서킷 브레이커, mTLS, 메트릭 수집, 트레이싱을 직접 구현하면 모든 서비스에 같은 코드가 중복되고, 언어별로 다시 구현해야 하며, 일관성이 무너진다.

Service Mesh는 이 공통 기능을 앱 코드에서 분리하여 프록시(사이드카) 레이어에서 투명하게 처리한다. 앱은 비즈니스 로직에만 집중하고, 통신 정책은 인프라 팀이 중앙에서 관리한다.
Service Mesh가 해결하는 문제
Traffic Management: Canary, A/B, 가중치 라우팅, 서킷 브레이커, 재시도, 타임아웃
Security: 자동 mTLS (서비스 간 암호화), 인가 정책, 인증서 자동 로테이션
Observability: L7 메트릭 자동 생성 (RPS, latency, error rate), 분산 트레이싱, 서비스 토폴로지
Resilience: fault injection (chaos engineering), rate limiting, outlier detection
도입 판단 기준
필요한 경우:
• 서비스 10개 이상, 서비스 간 통신이 복잡
• mTLS가 컴플라이언스 요구사항
• 서비스 간 트래픽의 세밀한 제어 필요
• 다중 언어(Go, Java, Python) 서비스 혼재

불필요한 경우:
• 서비스 수가 적고 (5개 미만) 통신이 단순
• 팀이 Mesh 운영 역량을 감당할 수 없음
• 이미 앱 레벨에서 충분히 구현됨
• 성능 오버헤드가 허용 불가 (ultra-low latency)
Architecture

Data Plane vs Control Plane

Service Mesh의 2계층 구조 — 트래픽을 처리하는 층과 정책을 관리하는 층
Data Plane
역할: 실제 서비스 간 트래픽을 중개(proxy). 모든 요청이 이 프록시를 거침
구현: Envoy Proxy (사이드카 또는 노드별 프록시)

처리하는 것:
• L4/L7 로드밸런싱
• mTLS 핸드셰이크 / 인증서 관리
• 재시도, 타임아웃, 서킷 브레이커
• 메트릭 수집 (RPS, latency, status code)
• 트레이스 컨텍스트 전파
• 헤더 기반 라우팅, 트래픽 분할
Control Plane
역할: Data Plane의 동작을 설정하고 관리. 정책을 프록시에 배포
구현: Istiod (Istio), Linkerd Control Plane 등

처리하는 것:
• 서비스 디스커버리 (K8s API 감시)
• 인증서 발급 & 로테이션 (내장 CA)
• 설정 배포 (xDS API로 Envoy에 푸시)
• 정책 평가 (AuthorizationPolicy 등)
• CRD 감시 (VirtualService, DestinationRule 등)

Evolution

Sidecar 모델 vs Ambient Mesh

Istio Ambient — 사이드카 없이 Service Mesh를 구현하는 차세대 아키텍처
전통적 Service Mesh는 모든 Pod에 사이드카 프록시를 주입한다. Pod 수 × 프록시 리소스만큼 오버헤드가 발생하고, 사이드카 업그레이드 시 모든 Pod를 재시작해야 한다.

Istio Ambient Mesh는 사이드카를 제거하고, 노드별 ztunnel(L4)과 필요한 곳에만 waypoint proxy(L7)를 배치하는 새로운 모델이다.
Sidecar 모델
각 Pod에 Envoy 사이드카:
• 모든 in/out 트래픽이 사이드카를 통과
• Pod당 ~100MB 메모리, ~0.5 vCPU
• sidecar injection으로 자동 주입

장점:
• 완전한 L7 기능 (헤더 라우팅, fault injection 등)
• Pod별 격리된 정책 적용

단점:
• 리소스 오버헤드가 크다
• 사이드카 업그레이드 = Pod 재시작
• latency 추가 (~1ms per hop)
Ambient Mesh
ztunnel (DaemonSet, L4):
• 노드당 1개. mTLS와 L4 보안 오버레이 담당
• 투명한 트래픽 리다이렉션으로 서비스 통신을 중개
• 리소스 오버헤드가 상대적으로 작음

Waypoint Proxy (L7):
• L7 기능이 필요한 서비스나 네임스페이스에만 선택적으로 배치
• HTTP 라우팅, L7 인가, 트레이싱 같은 기능 처리
• Kubernetes Gateway 리소스를 통해 배포하고 Istio가 관리

장점:
• 사이드카 수가 줄어 리소스 절감 효과가 큼
• 데이터 플레인 업그레이드를 애플리케이션 재시작과 더 잘 분리 가능
• L4만 먼저 도입하고 필요한 곳에만 L7을 추가 가능

주의: Ambient의 세부 기능 범위와 운영 성숙도는 Istio 버전에 따라 달라질 수 있으니 도입 전 릴리스 노트와 공식 문서를 함께 확인하는 편이 안전하다.
💡 Sidecar vs Ambient 선택
Ambient 추천: 새로 도입하는 경우, 리소스 효율이 중요한 경우, L4(mTLS)만 우선 필요한 경우
Sidecar 유지: 이미 sidecar로 운영 중, Pod별 세밀한 L7 정책이 많은 경우, 검증된 안정성이 중요한 경우
Ambient가 Istio의 미래 방향이지만, sidecar 모델도 계속 지원됨

구성 요소 역할
Istiod: 단일 Control Plane 바이너리. 서비스 디스커버리, xDS 생성/배포, 설정 검증, 인증서 발급과 로테이션 같은 기능을 담당한다.
Envoy Proxy: C++로 작성된 고성능 L4/L7 프록시. Sidecar 또는 Gateway로 배포되며 xDS API로 Istiod에서 동적으로 설정을 수신한다.
Istio Ingress Gateway: 클러스터 외부 트래픽의 진입점. Envoy 기반이며 Gateway API 또는 Istio Gateway 리소스로 설정한다.
ztunnel (Ambient): Ambient 데이터 플레인의 L4 프록시. 노드 단위로 배포되며 보안 오버레이와 기본 트래픽 처리에 집중한다.
Proxy

Envoy Proxy 내부

Listener → Filter Chain → Route → Cluster → Endpoint — Envoy의 요청 처리 경로
핵심 개념
Listener: 포트를 바인딩하여 연결 수신. 여러 filter chain 가능
Filter Chain: 요청을 처리하는 필터들의 체인. HTTP connection manager, TCP proxy 등
Route: 요청의 호스트/경로/헤더를 기반으로 어디로 보낼지 결정
Cluster: 업스트림 서비스 그룹. 로드밸런싱 정책, 서킷 브레이커, 헬스체크 포함
Endpoint: 실제 업스트림 인스턴스 (Pod IP:Port)
xDS API
Envoy가 Istiod에서 설정을 동적으로 수신하는 API 세트:
LDS — Listener Discovery Service
RDS — Route Discovery Service
CDS — Cluster Discovery Service
EDS — Endpoint Discovery Service
SDS — Secret Discovery Service (인증서)

ADS (Aggregated): 모든 xDS를 단일 gRPC 스트림으로 통합. 설정 일관성 보장

CRD(VirtualService 등) 변경 → Istiod가 감지 → xDS로 Envoy에 push → Envoy가 즉시 반영 (Pod 재시작 없음)
Control

Istiod 내부 동작

서비스 디스커버리, xDS 배포, 인증서 발급을 담당하는 Control Plane의 핵심 기능
Pilot (트래픽 관리)
K8s API 감시: Service, Endpoint, Pod 변경을 실시간 추적
CRD 감시: VirtualService, DestinationRule 등 Istio CRD 변경 감지
xDS 변환: K8s 리소스 + Istio CRD를 Envoy 설정(xDS)으로 변환
Push: 변경된 설정을 연결된 모든 Envoy에 gRPC로 push

성능 포인트: 대규모 클러스터에서 Endpoint 변경이 잦으면 xDS push가 빈번 → Istiod CPU 부하. EDS 필터링으로 필요한 Envoy에만 push
보안/CA 기능
내장 CA: 각 서비스의 SPIFFE ID 기반 x509 인증서를 발급한다.
SDS: 인증서를 Envoy에 Secret Discovery Service로 배포한다.
자동 로테이션: 인증서는 주기적으로 갱신되며 설정으로 조정할 수 있다.
Root CA: 자체 생성하거나 외부 CA(cert-manager, Vault)와 연동할 수 있다.

SPIFFE ID 형식:
spiffe://cluster.local/ns/production/sa/api-server
네임스페이스와 ServiceAccount를 기반으로 서비스 신원을 식별한다.

● Traffic Management

트래픽 관리

VirtualService, DestinationRule, Canary, Fault Injection, Circuit Breaker — 서비스 간 트래픽의 세밀한 제어.

Core

VirtualService & DestinationRule

Istio 트래픽 관리의 핵심 2가지 CRD — "어디로" + "어떻게"
VirtualService는 "요청을 어디로 보낼 것인가"를 정의한다 (라우팅 규칙). DestinationRule은 "보내진 트래픽을 어떻게 처리할 것인가"를 정의한다 (LB, 서킷 브레이커, subset).
VirtualService
apiVersion: networking.istio.io/v1 kind: VirtualService spec: hosts: [api-server] http: - match: - headers: x-canary: { exact: "true" } route: - destination: host: api-server subset: canary - route: # default - destination: host: api-server subset: stable
DestinationRule
apiVersion: networking.istio.io/v1 kind: DestinationRule spec: host: api-server trafficPolicy: loadBalancer: simple: LEAST_REQUEST connectionPool: tcp: { maxConnections: 100 } http: { h2UpgradePolicy: UPGRADE } subsets: - name: stable labels: { version: v1 } - name: canary labels: { version: v2 }
Deployment

Canary & A/B Testing

VirtualService의 가중치(weight)와 매칭(match)으로 구현하는 점진적 배포
가중치 기반 Canary
route: - destination: { host: api, subset: stable } weight: 90 - destination: { host: api, subset: canary } weight: 10
점진적 확대: 10→25→50→100%
각 단계에서 canary subset의 메트릭을 stable과 비교하고, 문제 발견 시 canary weight를 0으로 내려 즉시 롤백할 수 있다.

Argo Rollouts + Istio: AnalysisTemplate으로 자동 weight 조절 가능
헤더 기반 A/B
match: - headers: x-user-group: { exact: "beta" } route: [{ destination: { subset: canary } }]
활용:
• 내부 직원만 canary 경험: x-internal: true • 특정 사용자 그룹만 새 기능: x-user-group: beta • 모바일 vs 웹: user-agent 매칭
vs Feature Flag: Mesh 라우팅은 인프라 레벨, Feature Flag는 앱 레벨이라 서로 보완적이다.
Chaos

Fault Injection

의도적으로 지연이나 에러를 주입하여 서비스의 복원력을 테스트
Chaos Engineering을 앱 코드 변경 없이 인프라 레벨에서 수행할 수 있다. Envoy가 요청을 가로채 지연을 삽입하거나 에러를 반환한다.
Delay Injection
http: - fault: delay: percentage: { value: 10 } # 10% 요청에 fixedDelay: 3s # 3초 지연 삽입 route: - destination: { host: payment-svc }

테스트: payment-svc가 3초 느려지면 호출자의 timeout/서킷 브레이커가 동작하는지 확인
Abort (에러 주입)
http: - fault: abort: percentage: { value: 5 } # 5% 요청에 httpStatus: 503 # 503 반환 route: - destination: { host: payment-svc }

테스트: 하류 서비스가 503을 반환할 때 호출자의 fallback/재시도가 정상 동작하는지 확인
Game Day에서 활용하면 실제 서비스 중단 없이 장애 대응 연습 가능
Resilience

Timeout & Retry (Mesh 레벨)

앱 코드 수정 없이 VirtualService에서 일괄 적용하는 타임아웃과 재시도
Timeout
http: - timeout: 5s route: - destination: { host: api-server }

주의: Mesh 타임아웃과 앱 타임아웃이 공존하면 짧은 쪽이 먼저 발동. Mesh 타임아웃은 앱 타임아웃보다 약간 길게 설정하여 앱이 먼저 에러를 처리하도록

Timeout Chain: Client → Gateway timeout → VirtualService timeout → App timeout. 외부에서 내부로 갈수록 짧게
Retry
http: - retries: attempts: 3 perTryTimeout: 2s retryOn: 5xx,reset,connect-failure retryRemoteLocalities: true route: - destination: { host: payment-svc }

핵심: 앱 레벨 retry와 Mesh retry가 겹치면 retry storm. 한 곳에서만 retry하도록 조율. Mesh에서 통합 관리가 일관성 측면에서 유리

retryRemoteLocalities: true → 실패한 인스턴스가 있는 AZ 대신 다른 AZ로 재시도
Protection

Circuit Breaker (Outlier Detection)

DestinationRule의 outlierDetection으로 장애 인스턴스를 자동 격리
# ── DestinationRule: Outlier Detection (Circuit Breaker) ── spec: host: api-server trafficPolicy: outlierDetection: consecutive5xxErrors: 5 # 연속 5xx 5회 발생 시 interval: 10s # 10초 간격으로 평가 baseEjectionTime: 30s # 30초간 격리 (backoff 증가) maxEjectionPercent: 50 # 최대 50%까지 격리 (전부 격리 방지) minHealthPercent: 30 # 정상 30% 미만이면 outlier detection 비활성 connectionPool: tcp: maxConnections: 100 # 업스트림 최대 커넥션 http: http1MaxPendingRequests: 100 # 대기 큐 최대 크기 http2MaxRequests: 1000 # 동시 요청 최대 수
⚠ maxEjectionPercent 주의
100%로 설정하면 모든 인스턴스가 격리되어 서비스 완전 중단. 50%로 제한하면 최소 절반은 항상 트래픽을 받음. minHealthPercent와 함께 사용하여 안전장치 구성.

Encryption

자동 mTLS

모든 서비스 간 통신을 자동으로 암호화하고 상호 인증 — Zero Trust의 기반
Istio는 앱 코드 변경 없이 모든 서비스 간 통신을 mTLS로 암호화한다. 각 서비스에 SPIFFE 기반 x509 인증서를 자동 발급하고, Envoy가 투명하게 TLS 핸드셰이크를 처리한다.
mTLS 모드
PERMISSIVE: mTLS와 평문을 모두 허용한다. 마이그레이션 단계에서 주로 쓴다.
STRICT: mTLS만 허용하고 평문 통신은 거부한다.
DISABLE: mTLS를 비활성화한다. 특수한 경우에만 사용한다.

PeerAuthentication으로 설정:
spec.mtls.mode: STRICT
범위:
• 메시 전체 • 네임스페이스별 • 워크로드별 (selector.matchLabels)
좁은 범위의 정책이 넓은 범위보다 우선한다.
인증서 관리
자동 발급: Istiod CA가 ServiceAccount 기반 SPIFFE ID로 인증서를 발급한다.
자동 로테이션: 인증서 수명과 갱신 시점은 설정으로 조정할 수 있다.
Root CA 교체: 외부 CA(cert-manager, Vault)와 연동할 수 있다.

인증서 확인:
istioctl proxy-config secret POD_NAME
현재 인증서 정보와 만료 시각, SAN 등을 확인할 수 있다.

SPIFFE ID 형식:
spiffe://cluster.local/ns/{ns}/sa/{sa}
이 ID가 AuthorizationPolicy에서 서비스 신원으로 쓰인다.
💡 mTLS 마이그레이션 전략
Phase 1: PERMISSIVE 모드로 시작. 모든 서비스를 메시에 편입
Phase 2: Kiali에서 mTLS 트래픽 비율 확인. 100%에 가까워지면
Phase 3: 네임스페이스별로 STRICT 전환. 문제 발생 시 즉시 PERMISSIVE 복귀
Phase 4: 메시 전체 STRICT 적용
Access Control

AuthorizationPolicy

L7 수준의 서비스 간 접근 제어 — "누가 이 서비스를 호출할 수 있는가?"
NetworkPolicy가 L3/L4(IP, Port)에서 제어한다면, AuthorizationPolicy는 L7(HTTP method, path, header, SPIFFE ID)에서 제어한다. 더 세밀하고, 서비스 신원 기반 제어가 가능하다.
정책 유형
ALLOW: 조건에 맞는 요청만 허용
DENY: 조건에 맞는 요청 거부 (ALLOW보다 우선)
CUSTOM: 외부 인가 서버에 위임 (OPA 등)

평가 순서: CUSTOM → DENY → ALLOW
DENY에 매칭되면 ALLOW와 무관하게 거부
ALLOW 정책이 있으면 매칭되지 않는 것은 모두 거부
실전 예시
# payment-svc는 api-server에서만 호출 가능 apiVersion: security.istio.io/v1 kind: AuthorizationPolicy metadata: name: payment-allow namespace: production spec: selector: matchLabels: { app: payment-svc } action: ALLOW rules: - from: - source: principals: - "cluster.local/ns/production/sa/api-server" to: - operation: methods: ["POST"] paths: ["/api/v1/payments/*"]
Authentication

PeerAuthentication & RequestAuthentication

서비스 간 인증(mTLS)과 최종 사용자 인증(JWT)을 분리
PeerAuthentication
"서비스가 서비스를 인증"
mTLS로 호출자의 SPIFFE ID(= ServiceAccount)를 검증

서비스 간 신뢰:
"이 요청이 정말 api-server ServiceAccount에서 온 것인가?"
→ Citadel이 발급한 인증서로 암호학적 검증

AuthorizationPolicy의 source.principals와 연동
RequestAuthentication
"최종 사용자를 인증"
HTTP 요청의 JWT 토큰을 검증

spec: jwtRules: - issuer: "https://auth.example.com" jwksUri: "https://auth.example.com/.well-known/jwks.json" forwardOriginalToken: true

활용: Istio Gateway에서 JWT 검증 → 유효하지 않은 토큰은 앱에 도달하기 전에 거부. 앱은 JWT 검증 로직 불필요

AuthorizationPolicy에서 source.requestPrincipals로 사용자 ID 기반 인가 가능

Metrics

자동 생성 메트릭

Envoy가 모든 요청에 대해 자동으로 생성하는 L7 메트릭 — 앱 계측 불필요
Service Mesh의 가장 즉각적인 가치. 앱에 한 줄의 코드도 추가하지 않고 모든 서비스 간 통신의 RPS, latency, error rate를 얻는다. Envoy가 모든 요청을 프록시하면서 자동으로 Prometheus 메트릭을 생성한다.
핵심 메트릭
주요 메트릭:
istio_requests_total{response_code, source_workload, destination_workload, ...} istio_request_duration_milliseconds_bucket{...} istio_request_bytes_bucket istio_response_bytes_bucket istio_tcp_connections_opened_total istio_tcp_sent_bytes_total
istio_request_duration_milliseconds_bucket는 histogram이라 P50/P99 계산에 쓸 수 있다.
LGTM 스택 연동
Prometheus: Envoy의 /stats/prometheus 엔드포인트를 scrape. ServiceMonitor 또는 PodMonitor 설정

Grafana 대시보드:
• Istio Mesh Dashboard (전체 메시 상태)
• Istio Service Dashboard (서비스별 상세)
• Istio Workload Dashboard (Pod별 상세)
→ Istio 공식 대시보드 JSON을 import

SLI로 활용:
istio_requests_total로 SLO Burn Rate 알림 구현 가능. 앱 메트릭 없어도 Mesh 메트릭만으로 SLO 운영 가능

카디널리티 주의: Istio 메트릭은 source×destination×method×status 등 라벨이 많아 시계열 폭발 가능. Telemetry CRD로 불필요한 라벨 제거
Tracing

분산 트레이싱 (Mesh 레벨)

Envoy가 Span을 자동 생성하고 Tempo로 전송 — 앱은 헤더 전파만 하면 됨
자동 Span 생성
Envoy가 자동으로 하는 것:
• 인바운드/아웃바운드 요청마다 Span 생성
• Span에 서비스명, 메서드, 경로, 상태 코드, latency 기록
• Trace ID가 없으면 새로 생성
• 설정된 백엔드(Tempo, Zipkin)로 Span 전송

앱이 해야 하는 것:
• HTTP 헤더 전파. Envoy가 인바운드에서 추출한 trace 헤더를 앱이 아웃바운드 요청에 포함해야 함
• x-request-id, traceparent, x-b3-* 등
• 헤더 전파가 없으면 trace가 서비스 경계에서 끊김
Tempo 연동
Istio 설정:
meshConfig.extensionProviders: - name: otel-tracing opentelemetry: service: alloy.monitoring port: 4317
Telemetry CRD:
spec.tracing: - providers: [{ name: otel-tracing }] randomSamplingPercentage: 10
파이프라인:
Envoy -> OTel collector / Alloy -> Tempo
Grafana에서 Tempo를 조회하면 Metric↔Trace 연결이 가능하고, LGTM 스택의 trace 파이프라인에 Mesh가 자연스럽게 통합된다.
Visualization

Kiali — 서비스 토폴로지

서비스 간 트래픽 흐름을 실시간 그래프로 시각화하는 Istio 전용 대시보드
핵심 기능
Service Graph: 서비스 간 연결을 실시간 시각화. 트래픽 양, 에러율, latency를 edge에 표시
Health 상태: 각 서비스의 건강 상태를 색상으로 (초록/노랑/빨강)
mTLS 확인: 어떤 연결이 mTLS인지 평문인지 아이콘으로 표시
Istio Config 검증: VirtualService, DestinationRule 등의 설정 오류 감지
Trace 연동: Grafana/Jaeger와 연동하여 trace drill-down
운영 활용
인시던트 시: 서비스 그래프의 빨간 edge를 보면 어떤 서비스 간 통신이 실패하는지 빠르게 파악할 수 있다.
mTLS 마이그레이션: 평문 트래픽이 남아있는 연결을 시각적으로 찾을 수 있다.
불필요한 의존성 발견: 예상치 못한 연결이 보이면 제거하거나 NetworkPolicy를 추가할 근거가 된다.

접속:
istioctl dashboard kiali
Kiali 설치 방식은 사용 중인 Istio 버전과 배포 방식에 맞춰 공식 add-on 문서를 확인하는 편이 안전하다.

Lifecycle

설치 & 업그레이드 전략

istioctl, Helm, Operator — 설치 방법별 특성과 무중단 업그레이드
설치 방법 비교
istioctl install: 가장 간단하게 시작할 수 있고 `IstioOperator` 형식의 설정 파일을 함께 쓸 수 있다.
istioctl install -f my-config.yaml
Helm: GitOps 친화적이라 ArgoCD와 함께 관리하기 좋다.
helm install istio-base istio/base -n istio-system helm install istiod istio/istiod -n istio-system
In-Cluster Operator: 공식 문서 기준으로 deprecated 되었고, 업그레이드 경로는 `istioctl` 또는 Helm으로 이동하는 방향이다.

추천: Helm + ArgoCD 조합이 GitOps 기반 생명주기 관리에 잘 맞는다.
Canary 업그레이드
Revision 기반 무중단 업그레이드:
1. 새 버전의 Istiod를 revision 이름으로 설치 helm install istiod-1-23 istio/istiod --set revision=1-23 2. 네임스페이스 레이블을 새 revision으로 변경 istio.io/rev: 1-23 3. 워크로드를 Rolling restart하여 새 사이드카 주입 kubectl rollout restart deployment -n production 4. 모든 워크로드가 새 revision이면 이전 Istiod 제거
Ambient: ztunnel과 waypoint는 애플리케이션 Pod와 분리되어 있어 업그레이드 전략이 sidecar 모드와 다르다. 다만 실제 영향 범위는 배포 형태와 사용 기능에 따라 확인해야 한다.
Performance

성능 오버헤드

Mesh가 추가하는 latency, 리소스, 처리량 — 트레이드오프를 수치로 이해
Sidecar 모드 오버헤드
Latency: P50 ~0.5ms, P99 ~2-5ms 추가 (hop당)
→ A→B 호출 시 Envoy 2개 거침 = ~1-10ms 추가

메모리: sidecar당 ~50-100MB
Pod 100개 = 5-10GB 추가 메모리

CPU: sidecar당 ~50-100m (idle), 부하 시 더 증가

처리량: 직접 통신 대비 ~10-15% 감소
(mTLS + L7 파싱 + 메트릭 수집 비용)
Ambient 모드 오버헤드
ztunnel (L4 only):
• Latency: ~0.1-0.3ms 추가 (사이드카의 1/3)
• 메모리: 노드당 ~30-50MB
• Pod 100개 있어도 노드 3개면 = 150MB

Waypoint (L7):
• 필요한 서비스에만 배포
• Envoy 기반이므로 sidecar와 유사한 L7 비용
• 하지만 서비스 수만큼만 (Pod 수가 아님)

절감 효과:
Sidecar: Pod 수 × 100MB
Ambient: 노드 수 × 50MB + L7 서비스 수 × 100MB
→ 대규모 클러스터에서 ~90% 리소스 절감

Strategy

도입 전략

점진적으로, 한 네임스페이스씩, 관측부터 시작 — 안전한 Mesh 도입 로드맵
단계별 도입
Phase 0 — 평가 (1~2주):
• 스테이징 환경에서 PoC
• Sidecar vs Ambient 선택
• 성능 오버헤드 측정 (k6 부하 테스트)
• 팀 학습

Phase 1 — 관측 먼저 (2~4주):
• non-critical 네임스페이스부터 sidecar/ztunnel 주입
• PERMISSIVE mTLS
• 메트릭, 트레이싱, Kiali 서비스 그래프 확인
• 이 단계에서는 트래픽 정책 없이 관측만
확장 단계
Phase 2 — 보안 (2~4주):
• mTLS STRICT 전환 (네임스페이스별)
• AuthorizationPolicy 적용 (핵심 서비스부터)
• NetworkPolicy와 AuthorizationPolicy 보완 관계 설정

Phase 3 — 트래픽 관리 (필요 시):
• Canary 배포 (VirtualService weight 기반)
• Retry/Timeout 중앙 관리
• Circuit Breaker (outlierDetection)
• Fault injection (Game Day)

Phase 4 — 프로덕션 전체:
• 모든 네임스페이스 확장
• 운영 Runbook 작성
• 알림 설정 (Envoy 메트릭 기반)
• 업그레이드 절차 수립
⚠ 도입하지 않아도 되는 경우
현재 Traefik + NetworkPolicy + cert-manager mTLS로 핵심 기능(라우팅, 보안, 인증서)을 이미 구현하고 있다면, Service Mesh 도입은 운영 복잡도 증가를 감수할 만한 추가 가치가 있는지 신중히 평가해야 한다. 서비스 수가 적고 통신 패턴이 단순하다면 현재 구성이 더 실용적일 수 있다.

Mesh가 확실히 가치 있는 경우: 서비스 10개+, 다중 언어, mTLS 컴플라이언스, 정교한 트래픽 제어, 앱 코드 변경 없는 관측이 필요할 때
