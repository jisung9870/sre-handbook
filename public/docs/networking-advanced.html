Architecture

VPC & 서브넷 설계

CIDR 계획, 퍼블릭/프라이빗 서브넷, AZ 분산 — 네트워크 기반 설계
VPC 설계는 나중에 바꾸기 어렵다. CIDR이 겹치면 피어링이 불가능하고, 서브넷이 작으면 Pod IP가 부족하다. 처음에 충분히 크게, 확장 가능하게 설계해야 한다.
CIDR 설계 원칙
VPC CIDR: /16 (65,536 IPs) 이상 권장
EKS Pod 네트워크를 고려하면 /16도 부족할 수 있음 → Secondary CIDR 추가 가능 (100.64.0.0/16 등)

겹침 방지: 멀티 VPC, 멀티 계정, 온프레미스와 CIDR이 겹치면 피어링/VPN 불가
→ 조직 전체 IP 주소 계획(IPAM) 필수

서브넷 분할:
• Public: /24 (254 IPs) × AZ 수
• Private (App): /20 (4,094 IPs) × AZ 수
• Private (Data): /22 (1,022 IPs) × AZ 수
• Private (EKS Pod): /18 (16,382 IPs) × AZ 수
3-Tier 서브넷
Public Subnet:
• Internet Gateway 연결. 퍼블릭 IP 할당 가능
• ALB/NLB, NAT Gateway, Bastion 배치
• EKS 노드는 여기에 두지 않음

Private Subnet (App):
• EKS 워커 노드 배치
• NAT Gateway를 통해 아웃바운드만 가능

Private Subnet (Data):
• RDS, ElastiCache, MSK 등 데이터 레이어
• 인바운드를 App 서브넷에서만 허용

태깅 필수:
kubernetes.io/role/elb: 1 (public, LB용)
kubernetes.io/role/internal-elb: 1 (private, internal LB용)
Egress

NAT Gateway & VPC Endpoints

아웃바운드 인터넷 접근과 AWS 서비스 접근을 최적화하는 두 가지 메커니즘
NAT Gateway
역할: Private 서브넷의 리소스가 인터넷에 아웃바운드 접근
비용: 시간당 과금 + 데이터 처리 비용. AZ당 1개 배치 (교차 AZ 비용 방지)

HA: NAT GW는 AZ 내에서 이중화됨. 하지만 해당 AZ 장애 시 NAT 불가
→ 각 AZ의 Private 서브넷이 같은 AZ의 NAT GW를 사용하도록 라우팅

대역폭: 기본 5Gbps, 최대 100Gbps 자동 확장
연결 제한: 동일 목적지 IP당 55,000 동시 연결. 초과 시 ErrorPortAllocation
→ 여러 NAT GW로 트래픽 분산 또는 VPC Endpoint 사용
VPC Endpoints
Gateway Endpoint (무료): S3, DynamoDB
라우팅 테이블에 추가. NAT 불필요. 비용 절감 + 보안 향상

Interface Endpoint (유료): ECR, CloudWatch, STS, SQS 등 대부분 AWS 서비스
Private 서브넷에 ENI 생성. PrivateLink 기반

EKS 필수 Endpoints:
• com.amazonaws.region.ecr.api — 이미지 메타데이터
• com.amazonaws.region.ecr.dkr — 이미지 pull
• com.amazonaws.region.s3 — ECR 이미지 레이어 (Gateway)
• com.amazonaws.region.sts — IRSA 토큰
• com.amazonaws.region.ec2 — ENI 관리

Private EKS: API Server endpoint를 Private으로 설정하면 추가 Endpoint 필수

Transit Gateway

여러 VPC와 온프레미스를 중앙 허브로 연결하는 네트워크 허브
VPC 피어링은 1:1 연결이므로 VPC가 많아지면 풀 메시(n×(n-1)/2 연결)가 필요하다. Transit Gateway는 허브-스포크 모델로 모든 VPC를 중앙에서 연결한다.
아키텍처
허브-스포크: TGW가 중앙 허브, 각 VPC가 스포크로 연결
VPC 간 라우팅을 TGW 라우팅 테이블로 중앙 관리

연결 유형:
• VPC Attachment: VPC↔TGW
• VPN Attachment: 온프레미스↔TGW (IPsec VPN)
• Direct Connect Attachment: 전용선↔TGW
• Peering Attachment: 다른 리전 TGW↔TGW

라우팅 도메인: 라우팅 테이블로 VPC 그룹별 격리 가능
예: Prod VPC끼리만 통신, Dev는 Prod에 접근 불가
비용 & 성능
비용: Attachment 시간당 + 데이터 처리량 과금
VPC 피어링보다 비쌈. 소수 VPC면 피어링이 경제적

대역폭: Attachment당 최대 50Gbps
VPC 피어링은 제한 없음 (같은 리전)

판단 기준:
• VPC 3개 이하 → VPC 피어링
• VPC 4개 이상 또는 온프레미스 연결 필요 → TGW
• 리전 간 연결 → TGW Peering 또는 VPC Peering
EKS

EKS 네트워킹 심화

AWS VPC CNI, Pod IP 관리, Security Group for Pods, 서비스 메시 없이 mTLS
VPC CNI 특수성
AWS VPC CNI: Pod에 실제 VPC IP를 할당. Overlay 없음
장점: VPC 네이티브 라우팅, Security Group 직접 적용, 성능 최적
단점: IP 소진 문제. ENI당 할당 가능 IP 수가 인스턴스 타입에 의존

IP 부족 대응:
• ENABLE_PREFIX_DELEGATION=true — /28 프리픽스 단위로 할당. IP 효율 16배 향상
• WARM_PREFIX_TARGET=1 — 여유 프리픽스 수
• Secondary CIDR 추가 (100.64.0.0/16)
• AWS_VPC_K8S_CNI_CUSTOM_NETWORK_CFG=true — Pod을 별도 서브넷에 배치
Security Group for Pods
Pod에 직접 Security Group을 적용하는 기능 (EKS 1.17+)
SecurityGroupPolicy CRD로 특정 Pod에 SG 매핑

활용:
• DB Pod에만 RDS SG 허용
• 외부 API 호출이 필요한 Pod에만 아웃바운드 허용
• NetworkPolicy보다 AWS 네이티브한 접근

⚠ 제약: Nitro 인스턴스만 지원. 추가 ENI 트렁킹 필요
ENABLE_POD_ENI=true VPC CNI 환경변수

NHN Cloud 네트워크 특수성

NHN Cloud의 네트워킹 특성, LB 설정, Proxy Protocol, 실무 이슈 패턴.

Platform

NHN Cloud 네트워크 특수성

AWS와 다른 NHN Cloud의 네트워킹 동작과 주의점
NHN Cloud(NKS)는 AWS EKS와 유사한 Managed K8s를 제공하지만, 네트워킹 동작에서 차이가 있다. 특히 LB, NAT, Proxy Protocol 동작이 AWS와 다르며, 이를 모르면 간헐적 장애를 겪을 수 있다.
네트워크 아키텍처 차이
VPC: AWS VPC와 유사하지만 서브넷 모델이 단순
Floating IP: AWS의 Elastic IP와 유사. 인스턴스에 할당
보안 그룹: 인스턴스 레벨. AWS SG와 유사하지만 규칙 적용에 지연이 있을 수 있음

NKS (NHN Kubernetes Service):
• 노드: NHN Cloud VM으로 프로비저닝
• CNI: 기본 Flannel 또는 Calico
• LB: NHN Cloud LB가 Service type: LoadBalancer를 프로비저닝
AWS 대비 핵심 차이
LB Annotations: AWS와 다른 NHN 전용 annotation 사용
NAT 동작: kube-proxy SNAT이 AWS와 다르게 동작할 수 있음
DNS: NHN Cloud 내부 DNS 서버 설정 필요
로드밸런서 Health Check: 경로/프로토콜 설정이 AWS ALB와 다름
인증서 관리: ACM 같은 네이티브 인증서 서비스 차이 → cert-manager 자체 운영 필요
Load Balancer

NHN Cloud LB & Proxy Protocol

LB 설정, Proxy Protocol 구성, 원본 IP 보존 — NHN Cloud 환경의 핵심 과제
Proxy Protocol 필요성
문제: NHN Cloud LB를 통과하면 원본 클라이언트 IP가 LB IP로 치환된다
Traefik이나 Nginx가 실제 클라이언트 IP를 알 수 없다

해결: LB에서 Proxy Protocol v2를 활성화
→ LB가 원본 IP를 Proxy Protocol 헤더에 실어 전달
→ Traefik에서 proxyProtocol.trustedIPs로 LB IP 범위를 지정

⚠ trustedIPs 설정이 빠지면 Traefik이 Proxy Protocol 파싱을 거부해 연결이 실패할 수 있다
externalTrafficPolicy
Cluster (기본): 모든 노드로 트래픽 분산 → kube-proxy가 SNAT → 원본 IP 소실
Local: 해당 Pod이 있는 노드로만 트래픽 → SNAT 안 함 → 원본 IP 보존

Local의 주의점:
• Pod이 없는 노드로 트래픽이 가면 드롭될 수 있음
• LB health check가 노드별 Pod 유무를 정확히 반영해야 함
• Pod 분포가 불균형하면 트래픽도 불균형

NHN Cloud에서의 권장:
Proxy Protocol + externalTrafficPolicy: Local 조합으로 원본 IP를 정확히 보존
Troubleshoot

NHN Cloud 실무 이슈 패턴

실제 경험한 문제 패턴과 해결 방법
CORS 403 간헐적 장애
증상: 특정 요청에서만 간헐적으로 CORS 403 에러 발생
원인: kube-proxy의 SNAT이 LB 소스 IP를 Pod CIDR 주소로 변환했다. Traefik의 proxyProtocol.trustedIPs에 Pod CIDR이 포함되지 않아 Proxy Protocol 파싱이 거부된 것이다
해결: externalTrafficPolicy: Local로 변경하여 SNAT 우회

교훈: LB → kube-proxy → Pod 경로에서 소스 IP가 어떻게 변환되는지 정확히 이해해야 한다
Keepalive/Timeout 체인
문제: WAF → LB → Traefik → Pod 경로에서 각 계층의 idle timeout이 불일치하면 조용한 연결 끊김(silent connection drop)이 발생할 수 있다

원칙: 외부에서 내부로 갈수록 timeout을 짧게
WAF idle timeout: 350s LB idle timeout: 300s Traefik keepalive: 270s App keepalive: 240s

LB의 idle timeout이 Traefik보다 길면:
Traefik이 먼저 연결을 닫았는데 LB는 아직 열려있다고 생각 → LB가 닫힌 연결에 요청 전송 → 502 에러
⚠ NHN Cloud LB Annotation 정리
NHN Cloud의 LB 관련 annotation은 AWS의 service.beta.kubernetes.io/aws-load-balancer-*와 다른 네이밍 체계를 사용한다. Helm chart에서 클라우드별 values 오버라이드를 분리하여 관리해야 한다. ArgoCD app-of-apps에서 NHN/AWS 환경별 values 파일을 Kustomize 또는 Helm valueFiles로 분기.

DNS

하이브리드 DNS

클라우드 간 내부 DNS 해석을 통합하는 전략
AWS의 서비스가 NHN Cloud의 내부 도메인을 해석하려면, 그리고 그 반대도 되려면 DNS 포워딩 체계가 필요하다. 각 클라우드의 내부 DNS는 기본적으로 상대 클라우드의 도메인을 알지 못한다.
DNS 포워딩 전략
AWS → NHN Cloud:
Route 53 Resolver Outbound Endpoint 생성
→ NHN Cloud DNS IP로 포워딩 규칙 설정
→ *.nhn.internal → NHN DNS IP

NHN Cloud → AWS:
NHN 측 DNS 서버에서 *.aws.internal을 Route 53 Resolver Inbound Endpoint IP로 포워딩

CoreDNS 활용:
K8s CoreDNS에서 특정 도메인을 상대 클라우드 DNS로 포워딩
forward nhn.internal 10.x.x.x
K8s 클러스터 간 DNS
같은 클러스터: service.namespace.svc.cluster.local 자동 해석
다른 클러스터: 기본적으로 해석 불가

해결 방법:
• ExternalName Service: 외부 FQDN을 내부 Service로 매핑
• ExternalDNS: Service를 외부 DNS(Route 53 등)에 자동 등록
• CoreDNS forward: 특정 도메인을 다른 클러스터의 CoreDNS로 포워딩
• Service Mesh (Istio): Multi-cluster Service Discovery
Pattern

멀티클라우드 아키텍처 패턴

왜 멀티클라우드인가, 어떤 패턴을 선택할 것인가
패턴 유형
① Active-Active (양쪽 모두 서빙):
각 클라우드가 독립적으로 트래픽 처리. DNS 기반 로드밸런싱
장점: 한쪽 장애 시 자동 전환. 단점: 데이터 동기화 복잡

② Active-Passive (DR):
평소 한쪽만 사용, 장애 시 다른 쪽으로 전환
장점: 단순. 단점: 전환 시간(RTO) 발생, 비용 낭비

③ Tiered (역할 분리):
AWS: 글로벌 서비스, CDN, ML. NHN: 한국 리전 서비스, 규제 대응
장점: 각 클라우드의 강점 활용. 단점: 관리 복잡도
공통 인프라 표준화
Terraform: 양쪽 클라우드를 동일 IaC로 관리. Provider만 분리
ArgoCD: 멀티클러스터 관리. ApplicationSet으로 환경별 자동 배포
Helm: 공통 차트 + 환경별 values 오버라이드
LGTM: 중앙 Observability로 양쪽 클러스터 통합 모니터링

핵심: 앱 레이어는 클라우드 비종속적으로, 인프라 레이어만 클라우드별 분기. K8s가 이 추상화 계층 역할
● TLS & Network Security

TLS 심화와 네트워크 보안

TLS Handshake, cert-manager, mTLS — 전송 계층 보안의 SRE 운영 가이드.

Protocol

TLS 1.3 심화

TLS Handshake 내부 동작, 성능 특성, SRE 관점의 모니터링 포인트
TLS 1.3 Handshake
1-RTT Handshake (TLS 1.3):
Client: ClientHello (지원 cipher + key share)
Server: ServerHello (선택된 cipher + key share + 인증서 + Finished)
Client: Finished → 앱 데이터 전송 시작

0-RTT (Resumption): 이전 세션의 PSK로 첫 요청에 데이터 포함
⚠ Replay Attack 위험 → 멱등하지 않은 요청에는 사용 금지

vs TLS 1.2: 2-RTT → 1-RTT. handshake latency 50% 감소
보안: 약한 cipher suite 제거 (RC4, SHA-1, static RSA 등)
SRE 모니터링 포인트
인증서 만료: ssl_certificate_expiry_seconds (Blackbox Exporter)
알림: 만료 30일 전 ticket, 7일 전 page

Handshake 에러: traefik_tls_handshake_errors_total

cert-manager — K8s 인증서 자동화

Let's Encrypt 등 CA에서 인증서를 자동 발급·갱신하는 K8s 네이티브 도구
인증서 만료는 완전히 예방 가능한 장애다. cert-manager는 인증서 생명주기(발급, 갱신, 교체)를 자동화하여 만료 장애를 원천 차단한다.
핵심 리소스
Issuer / ClusterIssuer: 인증서 발급 기관 설정
• Let's Encrypt (ACME)
• Self-signed (내부 mTLS용)
• Vault PKI (엔터프라이즈)

Certificate: 발급할 인증서 명세 (도메인, 비밀 키 이름)
CertificateRequest: 발급 요청 (자동 생성)
Secret: 발급된 인증서+키 저장 (tls.crt, tls.key)
ACME Challenge 유형
HTTP-01: /.well-known/acme-challenge/ 경로로 검증
장점: 단순. 단점: 80 포트 인바운드 필요, 와일드카드 불가

DNS-01: DNS TXT 레코드로 검증
장점: 와일드카드 가능, 내부 서비스도 가능
단점: DNS 프로바이더 연동 필요 (Route 53, CloudFlare 등)

NHN Cloud: DNS API가 제한적일 수 있어 HTTP-01이 더 실용적. 또는 외부 DNS(CloudFlare)를 사용하여 DNS-01
갱신 모니터링
certmanager_certificate_expiration_timestamp_seconds — 만료 시간
certmanager_certificate_ready_status — 인증서 준비 상태 (1=ready)
certmanager_certificate_renewal_timestamp_seconds — 다음 갱신 시간

알림 규칙:
• 인증서 만료까지 7일 미만 + ready=0 → Page
• 인증서 만료까지 30일 미만 → Ticket
• 갱신 실패 이벤트 → Ticket

mTLS — Mutual TLS

서버뿐 아니라 클라이언트도 인증서를 제시하여 양방향 인증을 수행하는 TLS
일반 TLS는 서버만 인증한다 (클라이언트가 서버를 검증). mTLS는 클라이언트도 인증서를 제시하여 서버가 클라이언트를 검증한다. Zero Trust 네트워크의 기본 구현체.
Service Mesh 없이 mTLS
cert-manager + Traefik:
1. cert-manager로 CA 생성 (self-signed Issuer)
2. 서비스별 클라이언트/서버 인증서 발급
3. Traefik TLS 설정에서 clientAuth 활성화

tls.options.default.clientAuth.clientAuthType: RequireAndVerifyClientCert
tls.options.default.clientAuth.caFiles: /certs/ca.crt

장점: Istio 같은 무거운 Service Mesh 없이 mTLS 달성
단점: 인증서 관리가 서비스마다 필요. 수가 많으면 복잡
SPIFFE/SPIRE
SPIFFE: 서비스 ID 표준 (spiffe://cluster/ns/default/sa/my-app)
SPIRE: SPIFFE 구현체. 서비스에 자동으로 X.509 인증서 발급

Istio의 mTLS도 내부적으로 SPIFFE ID를 사용
Cilium의 Mutual Authentication도 SPIFFE 기반

흐름:
1. SPIRE Agent가 각 노드에서 워크로드 인증 (K8s SA 확인)
2. SPIRE Server가 SVID(SPIFFE ID + 인증서) 발급
3. 서비스 간 통신 시 SVID로 상호 인증
4. 인증서 자동 갱신 (보통 1~24시간 수명)
💡 단계별 mTLS 도입
Phase 1: 외부 트래픽만 TLS (Ingress/Gateway 레벨). 이미 완료된 단계
Phase 2: 핵심 서비스 간 mTLS (cert-manager + Traefik clientAuth). DB, Redis 접근 등
Phase 3: 전체 서비스 mTLS (SPIRE 또는 Cilium Mutual Auth). Zero Trust 완성
→ Phase 2가 비용 대비 효과가 가장 크다. Service Mesh 없이 달성 가능.
