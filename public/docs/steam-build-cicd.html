● Overview

Steam 빌드 & 배포 자동화

모바일 스토어와 다른 Steam만의 강점은 "심사 없이 즉시 배포"다. 그만큼 빌드/배포 자동화가 라이브 운영의 직접 무기. SteamPipe, Depot, SteamCMD, Branch 운영을 통합한 CI 파이프라인이 핵심. 2026-05 기준

Steam 생태계 — 퍼블리셔 관점

Steam Direct

퍼블리셔가 게임을 등록하는 절차. $100/title 등록비 (revenue $1,000 도달 시 환급). 신원 확인 + 세금 정보 + 게임 정보 입력. 첫 검수 통과 후엔 추가 게임 등록도 같은 계정으로.

Revenue Share (2026-05)

기본 70/30 (개발사 70%, Valve 30%). 누적 매출 $10M 초과 후 분 75/25. 누적 $50M 초과 후 분 80/20. DLC, season pass, Steam 마이크로결제 모두 동일 비율.

Steamworks SDK

achievement, leaderboard, cloud save, friends, matchmaking, P2P 통신, anti-cheat(VAC) 등 게임 통합 API. C++ 라이브러리 + 각 엔진별 wrapper (Unity는 Steamworks.NET, Unreal은 Steam Online Subsystem).

Steamworks Web API

서버 측에서 Steam 시스템 호출 (도전과제 부여, 리더보드 조회, 인벤토리 관리). Publisher Web API Key가 필수 — 클라에서 절대 노출 금지.

SteamPipe — 본질을 이해하라

SteamPipe는 단순 "파일 업로드 도구"가 아니다. 1MB 블록 단위 컨텐츠 분배 시스템이다. 이 본질을 이해해야 빌드 사이즈, 패치 효율, depot 설계가 모두 자연스러워진다.

1MB 블록 모델

SteamCMD가 빌드 디렉토리를 스캔 → ~1MB(1,048,576 byte) 블록으로 쪼개기 → 각 블록을 압축·암호화 → CDN 업로드. 같은 블록은 한 번만 저장 (deduplication).

Delta 패치의 마법

이전 빌드에 있던 블록은 재사용. 새 패치에서 변경된 블록만 다운로드. 100MB 패치도 실제 다운로드는 5~20MB가 일반. CDN 비용 + 사용자 다운로드 시간 모두 절감.

1MB 블록과 빌드 최적화

한 가지 큰 함정: 파일이 약간 변경되면 그 파일이 속한 블록이 통째로 새 블록이 됨. 큰 단일 파일(에셋 archive)을 자주 변경하면 매번 모든 블록 무효화. Unreal Pak 정렬 패딩(1MB 배수)을 맞추면 cascading shift 방지 — 패치 사이즈 5~10x 차이 가능.

★ 운영 함의
큰 단일 파일을 쓰지 말고 여러 작은 파일로 분리하면 변경 시 영향 블록 최소화. Unreal은 Pak 정렬, Unity는 Addressables의 작은 group으로 패치 효율 ↑. 배포 후 첫 패치에서 "어 패치가 왜 이렇게 커?" 이슈가 흔함 — 빌드 구조를 처음부터 SteamPipe 친화적으로.

● Depot

Depot 설계 — 컨텐츠를 어떻게 묶을까

Depot은 사용자에게 다운로드되는 컨텐츠 묶음 단위. 한 게임이 여러 depot 가질 수 있고, 사용자별/플랫폼별/언어별 다른 depot 받음. Depot 설계가 패치 효율과 빌드 시간을 결정.

Depot이란 — App / Depot / Build 계층

App

게임 한 개. App ID(예: 12345)로 식별. 사용자 라이브러리에 보이는 단위. 하나의 App에 1~100개 depot.

Depot

컨텐츠 묶음. 각자 Depot ID(예: 12346, 12347...). 플랫폼/언어/DLC별로 분리. 사용자는 자기에게 해당하는 depot만 다운로드.

Build

특정 시점의 모든 depot 산출물 묶음. Build ID(예: 9876543)로 식별. 한 build이 여러 depot 포함. branch에 build를 set live하는 게 배포의 정의.

Depot 분할 전략

"한 depot에 다 넣을지, N개로 쪼갤지"는 게임 운영의 핵심 결정. 너무 적게 쪼개면 사용자가 불필요한 데이터 다운, 너무 많이 쪼개면 운영 복잡도 폭증.

분할 기준 ① 플랫폼

Windows / Mac / Linux 각자 별도 depot. 사용자가 자기 OS depot만 다운. Windows-only 사용자가 Mac 바이너리 5GB 받지 않게.

분할 기준 ② 언어 / 음성

일본어 음성 5GB, 한국어 음성 3GB, 영어 음성 4GB. 언어별 depot + 사용자 시스템 언어로 자동 선택. 12GB 게임이 사용자에게는 7GB로 다운.

분할 기준 ③ DLC

본편 + 각 DLC가 별도 depot. 구매한 DLC depot만 다운. App ID와 별도로 DLC App ID도 만들 수 있고, 본편 App에 DLC depot으로 추가도 가능.

분할 기준 ④ 옵션 컨텐츠

고해상도 텍스처팩 (10GB), 4K 컷씬 (5GB), 모드 SDK 등. 사용자가 "Properties → DLC"에서 켜고 끔. 디스크 공간 빡빡한 사용자에게 옵션 제공.

♥ 안 쪼개야 할 것
너무 잘게 쪼개면 메타데이터 오버헤드 + 운영 부담 ↑. 변경 빈도가 다른 컨텐츠끼리만 분리가 원칙. 매 패치마다 다 같이 변하는 컨텐츠를 5개 depot으로 쪼개는 건 의미 없음. 언어 음성처럼 분명히 사용자 패턴이 다른 것만 분리.

플랫폼 / 언어 분기 — content_root 패턴

SteamPipe 빌드 디렉토리 구조. 각 depot이 자기 컨텐츠 디렉토리를 가리킨다. CI에서 빌드 후 이 구조로 파일 배치.

표준 빌드 디렉토리 구조
build/
├── windows/                # depot 12346 (Windows)
│   ├── Game.exe
│   ├── Game_Data/
│   └── ...
├── macos/                  # depot 12347 (Mac)
│   └── Game.app/
├── lang_ko/                # depot 12349 (한국어 음성)
│   └── audio/ko/
├── lang_ja/                # depot 12350 (일본어 음성)
│   └── audio/ja/
└── lang_en/                # depot 12351 (영어 음성)
    └── audio/en/
depot_build_12346.vdf (Windows depot 정의)
"DepotBuildConfig"
{
  "DepotID" "12346"
  "FileMapping"
  {
    "LocalPath" "./build/windows/*"
    "DepotPath" "."
    "recursive" "1"
  }
  "FileExclusion" "*.pdb"     // debug symbol 제외
}

DLC depot — 본편과의 관계

DLC를 별도 App ID로 만들 수도, 본편 App의 추가 depot으로 만들 수도. 운영 패턴이 다르다.

별도 App ID

스토어 페이지가 별도. 본편과 별도 가격, 별도 리뷰. 큰 확장팩 (Cyberpunk 2077의 Phantom Liberty 같은) 패턴. 마케팅 가능, but 본편 + DLC 동시 구매 패키지 운영 복잡.

본편 App + DLC depot

본편 App ID 안에 추가 depot. 스토어에는 "DLC" 카드로 묶임. Season Pass, 작은 DLC, 코스튬 팩 패턴. 운영 단순.

DLC ownership 검증

클라에서 사용자가 DLC 구매했는지 검증: SteamApps.BIsDlcInstalled(dlcAppId). 네트워크 게임의 경우 서버에서 검증 필수 — 클라 코드 우회 가능. Publisher Web API의 CheckAppOwnership으로 서버 측 검증.

● SteamCMD

SteamCMD — CI 자동화의 핵심

Steam 빌드 업로드 + 다운로드 + 게임 서버 운영 전용 CLI. CI에서 사람 개입 없이 빌드 → upload → branch set live까지 한 번에 가능. 2FA 자동화가 가장 큰 함정.

SteamCMD 기본

설치

Valve 공식 다운로드 (Windows/Linux/Mac). Docker image도 공식 / 비공식 존재 (예: cm2network/steampipe). CI 러너에 미리 설치 또는 매 빌드 마다 다운.

주요 명령

+login: Steam 계정 로그인. +run_app_build: VDF로 빌드 업로드. +app_update: 게임 다운로드 (서버 운영용). +quit: 마지막에 항상 — 안 붙이면 인터랙티브 모드.

호출 패턴

모든 명령을 + prefix로 chained. CLI 한 줄로 실행. 예: steamcmd +login user pass +run_app_build app.vdf +quit. 마지막 +quit 누락 시 인터랙티브 prompt에서 멈춤 — CI에서 timeout.

VDF — Valve의 설정 언어

VDF (Valve Data Format)은 JSON 비슷한 KeyValues 포맷. SteamCMD의 빌드 정의가 모두 VDF. app_build.vdf(전체 빌드)와 depot_build.vdf(개별 depot)로 구성.

app_build_12345.vdf — 전체 빌드 정의
"AppBuild"
{
  "AppID" "12345"
  "Desc" "Build from CI ${BUILD_ID}, commit ${GIT_SHA:0:7}"
  "BuildOutput" "./output"      // 빌드 로그 / 결과 저장
  "ContentRoot" "./build"        // 빌드 산출물 root
  "SetLive" ""                    // "" = 어느 branch에도 live X (수동)
                                       // "internal" = internal branch에 즉시 live

  "Depots"
  {
    "12346" "depot_build_12346.vdf"   // Windows
    "12347" "depot_build_12347.vdf"   // Mac
    "12349" "depot_build_12349.vdf"   // 한국어
    "12350" "depot_build_12350.vdf"   // 일본어
  }
}

SetLive — 자동 배포

빈 문자열 ""은 업로드만, 어느 branch도 live X — 수동 검증 후 set live. 특정 branch 이름 (예: "internal")이면 업로드 + 즉시 그 branch에 set live. default(prod) branch는 SetLive 직접 X — Steam 정책상 prod는 항상 수동 또는 별도 절차.

FileExclusion / FileMapping

depot vdf에서 .pdb, .map, 디버그 심볼 등 제외. FileMapping으로 LocalPath → DepotPath 변환. 같은 파일을 여러 depot에 매핑 가능 (공통 리소스).

CI 로그인 — 2FA 자동화

2026 기준 Steam 계정은 사실상 Steam Guard 2FA 의무. CI에서 사람 개입 없이 어떻게 로그인할까. 패턴은 두 가지.

① ssfn 파일 + config.vdf 캐시

로컬에서 한 번 로그인 (2FA 코드 직접 입력) → ~/Steam/ssfn* 파일과 config/config.vdf 생성됨. 이 파일들을 CI에 시크릿으로 보관. CI는 매번 로그인하지 않고 이 파일을 사용해 인증된 세션 재사용.

② Steam Guard 코드 자동 입력

전용 빌드 봇 계정 + Steam Guard email → email API (예: Gmail API)로 자동 코드 추출. SteamGuardDog 같은 도구. 유지보수 부담 ↑, 권장 X.

운영 — 빌드 봇 계정 격리

(1) 사람 계정 쓰지 말 것 — 한 사람의 2FA 변경이 CI 통째로 깨짐. (2) 전용 빌드 봇 계정 만들기. (3) 그 계정에 게임 publish 권한만 부여 (Steamworks 권한 세분화). (4) ssfn 파일은 CI 시크릿(Vault, GitHub Secrets) 안전 보관. (5) 분기마다 권한 검토.

⚠ ssfn 파일 분실
ssfn 파일은 Steam의 "내가 신뢰하는 머신" 인증 토큰. 잃어버리거나 회전하면 다시 사람 개입 필요. 안전한 곳에 백업 + 회전 정책. 빌드 봇 계정은 사람 직접 로그인 안 하고 ssfn으로만 운영하면 안전성 ↑.

CI 파이프라인 통합 — GameCI 패턴

GitHub Actions로 Unity/Unreal Steam 빌드 + 업로드 자동화하는 표준 패턴. game-ci/steam-deploy가 사실상 표준 액션.

.github/workflows/steam-deploy.yml
name: Steam Deploy
on:
  push:
    tags: ['v*']                   # release tag만

jobs:
  build-and-upload:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Build with Unity
        uses: game-ci/unity-builder@v4
        env:
          UNITY_LICENSE: ${{ secrets.UNITY_LICENSE }}
        with:
          targetPlatform: StandaloneWindows64
          buildsPath: build/

      - name: Steam Deploy
        uses: game-ci/steam-deploy@v3
        with:
          username: ${{ secrets.STEAM_USERNAME }}
          configVdf: ${{ secrets.STEAM_CONFIG_VDF }}    # base64 ssfn
          appId: "12345"
          buildDescription: "v${{ github.ref_name }} from ${{ github.sha }}"
          rootPath: build
          depot1Path: build/StandaloneWindows64
          releaseBranch: "internal"             # 자동 set live는 internal만

표준 단계

(1) Build: Unity/Unreal 빌드. (2) Upload: SteamCMD로 SteamPipe 업로드. (3) Set Live (internal): 내부 검증 branch에 자동 live. (4) Manual gate: prod set live는 수동 (Slack 승인 → workflow_dispatch).

secrets 관리

4개 핵심 시크릿: STEAM_USERNAME, STEAM_CONFIG_VDF (base64), STEAM_SSFN (base64), STEAM_SSFN_NAME. repo secret이 아닌 environment secret + protection rule — prod 배포는 reviewer 필수.

● Branches

Branch 운영 — beta, demo, password

한 게임에 여러 branch를 동시 운영 가능. 사용자가 어느 branch를 받을지 선택. 검증 워크플로, 즉시 롤백, demo 배포 모두 branch 메커니즘 위에서 동작.

Branch 종류

Branch역할접근운영 패턴
default (public)실제 사용자 prod모든 사용자심사 통과한 안정 빌드만
internal내부 QApassword 필수매 CI 빌드 자동 set live
beta외부 베타 테스터password 또는 opt-in검증된 빌드를 베타에 push

archive_v1.0 등이전 버전 보관password (선택)롤백 + 호환성 디버깅

Branch 비밀번호

Steam Console에서 branch마다 password 설정 가능. password를 알아야 사용자가 그 branch로 전환. branch 이름조차 노출 안 됨 (password 알면 자동 노출). 변경 시 즉시 적용.

Demo App vs Demo Branch

구식 패턴은 default branch에 demo 빌드. 2026 권장은 별도 Demo App ID — 무료, 별도 라이브러리, 별도 도전과제. 본편 App과 연관 (구매 시 demo 자동 사라짐 옵션).

검증 워크플로 — internal → beta → default

Internal — 자동

매 main push → CI 빌드 → SteamPipe 업로드 → internal branch 자동 set live. QA가 즉시 새 빌드 받음. cadence: 일 3~10회.

Beta — 수동

QA 통과 후 사람이 internal build를 beta로 promote. 같은 build ID, branch만 변경. 외부 베타 테스터가 password 입력 → beta branch 받음. cadence: 주 1~2회.

Default — 수동 + 절차

Beta 1주 검증 → 사람이 수동 promote. Slack 승인 + 사후 모니터링 알림. cadence: 격주~월 1회 (대형 업데이트), 수시 (핫픽스). 핫픽스는 internal → default 직접 가는 패턴도 있음 (긴급 시).

즉시 롤백 — 이전 build set live

Steam의 가장 큰 강점. 이전 build를 default branch에 다시 set live하면 즉시 롤백. 모바일 스토어로는 수일 걸리는 작업이 분 단위.

롤백 절차 — Steamworks Console
# 1. Steamworks 사이트 → Builds 페이지
# 2. 이전 build 선택 (대부분 직전 build)
# 3. "Set build live for branch..." → "default"
# 4. "Preview Change" → 영향받는 사용자 수 확인
# 5. "Set Build Live Now" 클릭
# 6. Steam이 ~분 안에 새 default build 분배 시작

# 사용자 측: 다음 게임 시작 시 자동 다운로드
#   (Steam 옵션이 "자동 업데이트")

롤백의 함정 — 데이터 호환성

코드는 롤백 즉시지만, 저장 데이터·온라인 상태·DB 스키마는 롤백 안 됨. v2가 새 스키마 도입했으면 v1로 롤백 시 v2 저장 파일 못 읽어 크래시. 저장 포맷 호환 필수 — 핫픽스는 데이터 변경 없이.

API 자동 롤백

Steamworks Web API SetAppBuildLive로 롤백 자동화 가능. 모니터링 시스템이 crash spike 감지 시 자동 호출 — 분 단위 자동 복구. 단 false positive 위험 — 임계 보수적으로.

Branch 보존 — archive 패턴

중요 마일스톤 빌드(v1.0, 시즌 종료 시점)를 별도 branch로 보존: archive_v1.0, season1_final. 특정 시점 디버깅이나 호환성 테스트 시 사용자가 password로 그 branch 전환 가능. 디스크 비용은 SteamPipe deduplication으로 거의 추가 없음.

Release State 관리 — 점진적 출시

Steam은 "지역별 점진 출시" 같은 Play의 staged rollout 직접 기능 없음. 대신 price & availability + branch 조합으로 비슷하게 운영.

지역별 출시 (Region Lock)

Steamworks Console → Pricing & Distribution → 지역별 활성/비활성. 한국 먼저 → 일본 다음 주 → 글로벌 순차 출시 가능. price 별도 설정.

출시 시점 컨트롤

Release Date 설정 + 자동 활성화. "Coming Soon" 상태 → 출시 시점 자동 전환. price 미리 설정 가능. 마케팅 사전 캠페인과 sync.

출시일 = build live와 분리

출시 전 default branch에 build이 set live 되어 있어야 함. "Coming Soon" 상태에서 사용자가 wishlist 후, 출시 시점에 자동 다운로드. build이 안 올라간 상태로 출시일 도래하면 사용자가 "다운로드 안 됨" 사고 — 출시 D-7 이전 default branch 빌드 확정 권장.

● Steamworks Features

Steamworks 통합 — 도전과제 / 리더보드 / Cloud / Workshop

Steam 게임의 차별화 요소. 도전과제, 리더보드, Steam Cloud 저장, Workshop 모드. SDK 통합 + Web API 운영, 그리고 CI 파이프라인에서 함께 다뤄야 할 부분.

도전과제 (Achievements)

설정 — Steamworks Site

Steamworks → App Admin → Stats & Achievements. 도전과제 ID, 이름, 설명, 아이콘(unlocked/locked) 정의. 게임 출시 후에도 추가 가능 — 단 새 도전과제는 새 빌드 필요 X (Steamworks 측 정의만 변경).

SDK — 클라 측

ISteamUserStats::SetAchievement("ACH_BOSS_KILL") + StoreStats(). 비동기 — UserStatsReceived_t 콜백 대기. 네트워크 게임은 서버 측 검증: 클라 자체 unlock은 치팅 가능, Web API SetUserStatsForGame으로 서버에서 부여.

Hidden 도전과제

설명을 "???" 로 가리기. 스토리 spoiler 도전과제, 이스터에그용. Hidden flag는 Steamworks 측 정의에서 토글 — 운영자가 컨피그처럼 쉽게 변경 가능.

리더보드 (Leaderboards)

Steam 측 리더보드 시스템. 자동 ordering, 글로벌/친구 순위, UGC 첨부. 한 게임당 최대 10,000개 리더보드.

생성 — 자동 vs Console

Steam Console에서 미리 정의 또는 게임 코드에서 FindOrCreateLeaderboard로 자동 생성. 운영 편의는 Console 미리 정의 — 이름·정렬·display type 명시적.

Score 업로드 — Web API

치팅 방지 위해 Web API SetLeaderboardScore를 서버에서 호출. publisher Web API key 필수 — 클라에서 절대 노출 X. 클라가 점수를 서버에 전송 → 서버가 검증 → Steam Web API 호출.

UGC 첨부 — 리플레이 / 스크린샷

리더보드 entry에 UGC 첨부 가능. 리플레이 파일, 스크린샷, 게임 데이터. Steam Cloud 통해 업로드. "이 점수의 리플레이 보기" 기능 — 치팅 의심 시 검증, 또는 커뮤니티 공유.

Steam Cloud — 클라우드 저장

Steam이 자체 운영하는 저장 클라우드. 게임팀이 백엔드 안 만들고도 cross-device 저장 가능. 대부분의 SP 게임에 권장.

두 가지 방식

(1) Steam Auto-Cloud: 특정 디렉토리 패턴 지정 → Steam이 자동 동기화. 코드 변경 0. (2) ISteamRemoteStorage API: 코드에서 명시적으로 read/write. 더 정밀한 제어, 충돌 핸들링 가능.

용량 제한

App ID당 기본 100MB~ (게임마다 다름, Steamworks 신청). 큰 저장은 비추 — 저장 본질은 "진행 상태"지 게임 컨텐츠가 아님. 자체 백엔드와 함께 쓰는 케이스도 일반적 (Steam Cloud는 자동 백업, 자체 백엔드는 진행 상태).

충돌 (Conflict) 처리

다른 디바이스에서 동시 플레이 시 충돌. Steam Auto-Cloud는 최신 timestamp 우선. ISteamRemoteStorage API는 conflict resolution UI 노출 가능. 온라인 게임은 서버가 ground truth — Cloud는 보조 백업으로만.

Workshop — UGC 플랫폼

Steam의 user-generated content 플랫폼. 모드, 맵, 아이템, 시나리오. 커뮤니티가 곧 컨텐츠인 게임에 강력 — Skyrim, Steam, Cities Skylines 등.

Workshop 통합 옵션

Steamworks → App Admin → Workshop 활성. 게임팀이 어떤 형태의 UGC 받을지 정의 (mod, item, map, scenario). SDK ISteamUGC로 클라 측 통합.

운영 — 모더레이션

UGC는 커뮤니티 신고 + 퍼블리셔 모더레이션 조합. 부적절 컨텐츠, 저작권 침해, 악성 코드 모드 차단. Steam의 자동 스캔 + 인간 검토. CS 백엔드와 통합 운영.

도입 부담

Workshop 도입은 운영 부담 큼. (1) 모더레이션 인력, (2) 모드 호환성 (게임 패치 시 모드 깨짐), (3) 보안 검토 (악성 모드), (4) 저작권 분쟁. 신중히 결정 — 게임 성격 + 커뮤니티 규모 고려.
