● Process & Scheduling

프로세스 관리와 스케줄링

커널이 프로세스를 생성·관리·스케줄하는 내부 메커니즘. SRE가 CPU 병목과 응답 지연 급증을 이해하려면 필수.

Kernel Internals

Linux 프로세스 모델

task_struct — 커널이 보는 프로세스/스레드의 실체
Linux에서 프로세스와 스레드는 모두 task_struct 구조체로 표현된다. 커널 입장에서 프로세스와 스레드의 차이는 자원 공유 범위뿐이다. fork()는 모든 자원을 복제하고, clone()은 메모리·파일 디스크립터 등을 선택적으로 공유한다(스레드).

모든 프로세스는 PID 1(init/systemd)을 루트로 하는 트리 구조를 이룬다. 컨테이너 안에서는 PID namespace 덕분에 별도의 PID 1을 가진다.
프로세스 상태
R (Running): CPU에서 실행 중 또는 run queue 대기
S (Sleeping): I/O 등 이벤트 대기 (Interruptible)
D (Disk Sleep): I/O 대기 (Uninterruptible) — kill 불가
Z (Zombie): 종료했지만 부모가 wait() 안 함
T (Stopped): SIGSTOP으로 정지됨
SRE 관점 핵심
D state 프로세스 다수 → I/O 병목 (스토리지 문제)
Zombie 누적 → 부모 프로세스 버그 (리소스 누수)
높은 fork rate → 비효율적 프로세스 생성 패턴
cat /proc/[pid]/status로 상태 확인
fork() vs clone() vs exec()
fork(): 프로세스 복제. Copy-on-Write로 실제 복사 지연. 반환값으로 부모/자식 구분
clone(): 공유 범위를 플래그로 제어. CLONE_VM(메모리), CLONE_FILES(FD), CLONE_NEWPID(PID ns) 등
exec(): 현재 프로세스의 코드를 새 바이너리로 교체. fork()+exec()가 새 프로그램 실행의 표준 패턴
Copy-on-Write: fork() 시 메모리를 즉시 복사하지 않고, 쓰기가 발생할 때만 해당 페이지를 복사. 대형 프로세스의 fork도 빠른 이유
Scheduler

CFS (Completely Fair Scheduler)

Linux 기본 스케줄러. Red-Black Tree 기반 가상 실행시간 균등 배분
CFS는 각 태스크가 "공정하게" CPU 시간을 나눠 갖는 것을 목표로 한다. 핵심 개념은 vruntime(virtual runtime) — 각 태스크가 실제로 실행된 시간을 가중치(nice 값) 기반으로 정규화한 값이다. vruntime이 가장 작은 태스크가 다음에 실행된다.

태스크들은 Red-Black Tree에 vruntime 순으로 정렬되어 있어, 다음 실행할 태스크를 O(1)로 선택할 수 있다.
핵심 파라미터
sched_min_granularity_ns — 최소 타임슬라이스 (기본 0.75ms)
sched_latency_ns — 모든 태스크가 한 번씩 실행되는 주기 (기본 6ms)
sched_wakeup_granularity_ns — wakeup preemption 임계값
Scheduling Class 우선순위
1) SCHED_DEADLINE: 실시간 데드라인 기반 (최고 우선)
2) SCHED_FIFO/RR: 실시간 고정 우선순위
3) SCHED_NORMAL (CFS): 일반 프로세스 (대부분)
4) SCHED_BATCH: 배치 작업 (preemption 제한)
5) SCHED_IDLE: 시스템 유휴 시에만 실행
SRE 관점: CPU Throttling in K8s
K8s CPU limit은 CFS Bandwidth Control(cpu.cfs_quota_us / cpu.cfs_period_us)로 구현된다. limit 100m = 100ms/1000ms 주기 중 100ms만 사용 가능.
CPU throttling이 발생하면 프로세스가 quota를 소진한 나머지 시간 동안 강제 대기 → latency spike의 주범.
cat /sys/fs/cgroup/cpu/cpu.stat에서 nr_throttled, throttled_time 확인.

실무 팁: CPU limit을 지나치게 타이트하게 잡으면 burst 트래픽에서 P99 latency가 튄다. request는 설정하되 limit은 넉넉하게 또는 제거하는 전략도 고려.
Performance

Context Switch

CPU가 실행 중인 태스크를 전환하는 과정. 빈도가 높으면 오버헤드 발생
Context Switch 시 커널은 CPU 레지스터, 프로그램 카운터, 스택 포인터를 저장하고 다음 태스크의 것으로 복원한다. 프로세스 간 전환은 TLB flush가 추가로 발생하여 스레드 간 전환보다 비용이 크다.

자발적(voluntary) 전환은 I/O 대기 등으로 태스크가 스스로 CPU를 양보할 때, 비자발적(involuntary) 전환은 타임슬라이스 만료나 높은 우선순위 태스크에 의해 선점될 때 발생한다.
측정 방법
vmstat 1 — cs 열 (초당 전환 횟수)
pidstat -w 1 — 프로세스별 전환 횟수
cat /proc/[pid]/status — voluntary/nonvoluntary 누적
perf stat -e context-switches ./app
SRE 관점
초당 수만 회 이상이면 오버헤드 의심
involuntary cs 높음: CPU 부족 또는 과도한 스레드 수
voluntary cs 높음: I/O 대기 많음 (정상일 수 있음)
Go 런타임의 goroutine은 유저스페이스 스케줄링으로 커널 cs 최소화
Interface

System Call — 커널과 유저스페이스의 경계

유저 프로그램이 커널 기능(파일, 네트워크, 프로세스)을 사용하는 유일한 인터페이스
프로그램이 파일을 열거나(open()), 네트워크에 접속하거나(connect()), 프로세스를 생성할 때(fork()) 모두 system call을 통한다. syscall은 유저 모드 → 커널 모드 전환을 수반하므로 비용이 있다.

현대 Linux는 vDSO(virtual Dynamic Shared Object)를 통해 gettimeofday() 같은 빈번한 syscall을 커널 진입 없이 유저스페이스에서 처리하여 오버헤드를 줄인다.
핵심 syscall 카테고리
프로세스: fork, clone, execve, exit, wait
파일: open, read, write, close, stat, mmap
네트워크: socket, bind, listen, accept, connect, sendmsg
메모리: brk, mmap, munmap, madvise
시그널: kill, sigaction, sigprocmask
추적 도구
strace -c ./app — syscall 통계 요약
strace -T -e trace=network ./app — 네트워크 syscall만 시간 측정
perf trace ./app — 저오버헤드 syscall 추적
bpftrace -e 'tracepoint:syscalls:sys_enter_* { @[comm] = count(); }'
● Memory Management

메모리 관리 내부 동작

가상 메모리, 페이지 캐시, OOM Killer, NUMA — SRE가 메모리 관련 장애를 이해하고 튜닝하기 위한 핵심.

Core

Virtual Memory — 가상 메모리

프로세스마다 독립된 주소 공간을 제공하는 메모리 추상화 계층
모든 프로세스는 자신만의 가상 주소 공간을 가진다. 이 가상 주소는 Page Table을 통해 물리 메모리(RAM) 주소로 변환된다. CPU의 MMU(Memory Management Unit)가 이 변환을 하드웨어로 수행하고, TLB(Translation Lookaside Buffer)에 최근 변환 결과를 캐싱한다.

가상 메모리 덕분에 프로세스는 연속된 메모리를 사용하는 것처럼 보이지만, 실제 물리 메모리는 조각나 있을 수 있다. 또한 실제 물리 메모리보다 큰 주소 공간을 사용할 수 있다(overcommit).
주소 변환 과정
Virtual Addr → Page Table (4 level) → Physical Addr
└→ TLB hit? → 바로 물리 주소
└→ TLB miss → Page Table Walk → (Page Fault?)
Page Fault 종류
Minor Fault: 페이지가 메모리에 있지만 매핑 안 됨. 매핑만 하면 됨. 빠름
Major Fault: 페이지가 디스크에 있음(스왑). 디스크 I/O 필요. 매우 느림
ps -o min_flt,maj_flt -p PID로 확인
메모리 영역 (Process Address Space)
Text: 실행 코드 (읽기 전용) | Data: 초기화된 전역 변수 | BSS: 미초기화 전역 변수 | Heap: 동적 할당 (brk/mmap) ↑ 성장 | Memory-mapped files: mmap 영역 | Stack: 함수 호출 스택 ↓ 성장 | Kernel space: 상위 주소 (유저 접근 불가)
cat /proc/[pid]/maps로 영역별 매핑 확인
Performance

Page Cache

디스크 I/O를 줄이기 위해 커널이 자동으로 관리하는 메모리 캐시
파일을 읽으면 커널은 해당 데이터를 Page Cache에 보관한다. 같은 파일을 다시 읽을 때 디스크 대신 메모리에서 서빙하여 I/O 성능을 극적으로 향상시킨다.

free -h에서 buff/cache가 Page Cache다. "메모리를 많이 쓰고 있다"고 착각하기 쉽지만, 이것은 사용 가능한 메모리다. 커널은 메모리 압박 시 Page Cache를 즉시 회수한다.

SRE 핵심: 컨테이너 환경에서 cgroup memory limit에 Page Cache가 포함된다. 대량 파일 읽기가 Page Cache를 채우면 OOM Kill이 발생할 수 있다.
확인 명령어
free -h — buff/cache 열
vmstat 1 — cache 열, bi/bo (block I/O)
cat /proc/meminfo — Cached, Buffers 상세
pcstat filename — 특정 파일의 캐시 비율
Dirty Pages
쓰기 작업 후 아직 디스크에 반영 안 된 Page Cache.
커널이 주기적으로(또는 임계치 도달 시) writeback을 수행하여 디스크에 flush.
dirty_ratio 도달 시 동기 쓰기 → I/O 스파이크의 원인
Critical

OOM Killer

메모리가 완전히 고갈되었을 때 커널이 프로세스를 강제 종료하는 메커니즘
Linux는 기본적으로 메모리 오버커밋을 허용한다. 실제 물리 메모리보다 더 많은 메모리를 할당할 수 있게 한다. 문제는 실제 사용량이 물리 메모리를 초과할 때 — 커널은 OOM Killer를 호출하여 oom_score가 가장 높은 프로세스를 kill한다.

K8s에서 OOMKilled는 cgroup memory limit 도달 시 발생. 커널 전체 OOM과 달리 해당 cgroup 내에서만 동작. dmesg | grep -i oom으로 확인.
OOM Score 산정
oom_score: 메모리 사용량 기반 자동 점수 (0~1000)
oom_score_adj: 수동 조정 (-1000~1000)
-1000 = OOM 면제, 1000 = 최우선 kill
cat /proc/[pid]/oom_score
K8s: QoS Guaranteed → oom_score_adj=-997
Overcommit 정책
vm.overcommit_memory
0 (default): 휴리스틱 판단으로 과도한 할당 거부
1: 항상 허용 (위험하지만 Redis 등에서 필요)
2: swap + RAM × ratio 초과 시 거부
vm.overcommit_ratio = 기본 50%
⚠ 실무 사례
K8s Pod에서 memory request < limit으로 설정된 경우, 노드 메모리 압박 시 Burstable QoS인 Pod가 먼저 eviction 대상이 된다. request = limit (Guaranteed)으로 설정하면 oom_score_adj가 낮아져 보호됨.
Architecture

NUMA (Non-Uniform Memory Access)

멀티소켓 시스템에서 CPU와 메모리 사이의 접근 속도가 균일하지 않은 아키텍처
멀티소켓 서버에서 각 CPU 소켓은 자신에게 가까운 로컬 메모리에 빠르게 접근하고, 다른 소켓의 리모트 메모리에는 느리게 접근한다. 리모트 접근은 로컬 대비 1.5~3배 느릴 수 있다.

GPU 서버(TYAN A100 같은)에서 특히 중요. GPU와 가까운 NUMA 노드에 프로세스를 바인딩하면 성능이 크게 향상.
확인 & 제어
numactl --hardware — NUMA 토폴로지
numastat -p PID — 프로세스별 NUMA 할당
numactl --cpunodebind=0 --membind=0 ./app
K8s: topologyManager 정책으로 NUMA 인식 스케줄링
SRE 관점
NUMA misalignment는 latency 편차의 숨은 원인
메모리 집약 워크로드에서 리모트 접근 비율이 높으면
P99 latency가 불안정해진다. numastat의
other_node 값이 높으면 NUMA 바인딩 고려.
Tuning

메모리 튜닝 sysctl

프로덕션 서버에서 자주 조정하는 메모리 관련 커널 파라미터
메모리 튜닝은 워크로드 특성에 따라 달라진다. 데이터베이스는 Page Cache를 적극 활용하고, 컨테이너 환경은 cgroup limit과의 상호작용을 고려해야 한다.
파라미터기본값설명튜닝 가이드
vm.swappiness60스왑 사용 적극성 (0~200)DB/Redis: 1~10. K8s 노드: 0~10. 0이면 완전 비활성화 아님
vm.dirty_ratio20전체 메모리 대비 dirty page 비율 (동기 쓰기 트리거)SSD: 40~60. HDD: 15~20. 높으면 burst 쓰기 시 대기 발생
vm.dirty_background_ratio10백그라운드 writeback 시작 비율dirty_ratio의 1/2~1/3. SSD에서 5~10
vm.overcommit_memory0메모리 오버커밋 정책Redis fork 대비: 1. 보수적 운영: 0 또는 2
vm.min_free_kbytesauto커널이 유지할 최소 free 메모리대용량 서버: 128MB~1GB. 네트워크 버퍼 할당 실패 방지
vm.zone_reclaim_mode0NUMA zone 내에서 메모리 회수 정책NUMA 서버에서 1 설정 시 로컬 메모리 우선 회수 → latency 개선 가능
● I/O & Filesystem

I/O 서브시스템과 파일시스템

VFS 계층, I/O 스케줄러, Dirty Page writeback, 파일시스템 선택 — 디스크 성능의 핵심.

Architecture

VFS & I/O 경로

Virtual File System — 모든 파일시스템을 통일된 인터페이스로 추상화
Linux에서 "모든 것은 파일"이다. VFS는 ext4, xfs, tmpfs, procfs 등 다양한 파일시스템을 동일한 API(open/read/write)로 접근할 수 있게 하는 추상화 계층이다.

I/O 요청은 Application → VFS → Page Cache → Filesystem → Block Layer → Device Driver → Disk 순으로 내려간다. 이 경로의 각 단계에서 최적화와 병목이 발생할 수 있다.
▼ I/O 경로 스택
Applicationread()/write()
↓
VFS Layerinode, dentry, file ops
↓
Page CacheBuffered I/O (기본)
Direct I/OO_DIRECT (bypass)
↓
Filesystemext4 / xfs / btrfs
↓
Block LayerI/O Scheduler, merge, sort
↓
Device Driver → NVMe / SCSI / virtio
💡 Buffered vs Direct I/O
Buffered I/O (기본): Page Cache를 거침. 반복 읽기에 유리. 대부분의 워크로드에 적합.
Direct I/O (O_DIRECT): Page Cache를 우회하여 디스크에 직접 I/O. 자체 캐시가 있는 DB(MySQL InnoDB, PostgreSQL)에서 사용. 이중 캐싱 방지.
Scheduler

I/O Scheduler

블록 I/O 요청의 순서와 병합을 관리하여 디스크 성능을 최적화
I/O 스케줄러는 요청 병합(merge)과 순서 재배치(reorder)를 통해 디스크 접근 패턴을 최적화한다. 디스크 유형(HDD vs SSD/NVMe)에 따라 최적의 스케줄러가 다르다.
Linux 5.x+ 스케줄러
none (noop): 스케줄링 없음. NVMe에 적합 (하드웨어가 최적화)

Dirty Pages & Writeback

쓰기 데이터가 메모리에 머무르다가 디스크로 반영되는 과정
write() 호출 시 데이터는 즉시 디스크에 쓰이지 않고 Page Cache에 dirty로 마킹된다. 커널의 flush/writeback 스레드가 주기적으로 또는 임계치에 도달하면 디스크에 반영한다.

이 메커니즘 덕분에 쓰기가 빠르지만, dirty page가 과도하게 쌓이면 flush 시 I/O 스파이크가 발생한다. vm.dirty_ratio에 도달하면 동기 쓰기로 전환되어 프로세스가 blocking된다.
Writeback 트리거
1) dirty_background_ratio 도달 → 백그라운드 flush 시작
2) dirty_ratio 도달 → 쓰기 프로세스 block (동기)
3) dirty_expire_centisecs (30초) 경과 → 오래된 dirty page flush
4) dirty_writeback_centisecs (5초) 간격으로 flush 스레드 wake
모니터링
cat /proc/meminfo | grep Dirty
vmstat 1 — bo 열 (block out = 디스크 쓰기)
sar -b 1 — tps, bread/s, bwrtn/s
Dirty 값이 수 GB이면 flush storm 위험

Protocol

TCP 심화

3-Way Handshake, Congestion Control, TIME_WAIT, Keepalive — 네트워크 latency의 근원
TCP는 신뢰성 있는 연결 기반 프로토콜이다. SRE가 TCP를 깊게 이해해야 하는 이유는 대부분의 서비스 간 통신이 TCP이고, 연결 수립/해제/재전송 과정이 latency와 리소스 사용에 직접적으로 영향을 주기 때문이다.
3-Way Handshake & Backlog
SYN → SYN-ACK → ACK
서버 측에는 두 개의 큐가 존재:
SYN Queue (반개방): SYN 수신 후 SYN-ACK 전송. net.ipv4.tcp_max_syn_backlog
Accept Queue (전개방): 3-way 완료 후 accept() 대기. net.core.somaxconn
큐 overflow 시 SYN drop → 연결 실패/지연. ss -s에서 overflowed 확인
TIME_WAIT
연결 종료 후 2×MSL(60초) 동안 유지되는 상태.
목적: 지연된 패킷이 새 연결에 영향 주는 것 방지
많은 TIME_WAIT: 빈번한 단수명 연결 (HTTP 1.0 등)
해결: Connection Pooling, Keepalive 활성화
net.ipv4.tcp_tw_reuse=1 — 재사용 허용 (클라이언트에서)
ss -s → timewait 수 확인
Congestion Control
기본: cubic (loss-based). 패킷 손실 시 윈도우 축소
bbr: Google 개발. bandwidth+RTT 기반. 손실에 덜 민감
sysctl net.ipv4.tcp_congestion_control
클라우드/WAN 환경에서 BBR이 throughput 향상에 유리
sysctl -w net.ipv4.tcp_congestion_control=bbr
TCP Keepalive
유휴 연결이 살아있는지 확인하는 프로브 메커니즘
tcp_keepalive_time=7200 — 첫 프로브까지 대기 (기본 2시간, 너무 김)
tcp_keepalive_intvl=75 — 프로브 간격
tcp_keepalive_probes=9 — 실패 허용 횟수
K8s/LB 환경: LB의 idle timeout(보통 60~350초)보다 짧게 설정해야 silent connection drop 방지
TCP Retransmission & SRE
재전송은 네트워크 품질 저하의 직접적 신호. ss -ti로 retrans 값 확인. 재전송이 발생하면 RTT의 2배, 4배, 8배...로 기하급수적으로 대기(exponential backoff) → P99 latency 급등.
nstat -a | grep Retrans — 시스템 전체 재전송 통계. TcpRetransSegs가 지속적으로 높으면 네트워크 경로 문제 의심.

Conntrack & NAT

Netfilter의 연결 추적 모듈. K8s Service, SNAT/DNAT의 기반
Conntrack은 모든 네트워크 연결의 상태를 추적하는 커널 모듈이다. iptables의 -m state --state ESTABLISHED가 동작하려면 conntrack이 필요하다. K8s에서 kube-proxy(iptables 모드)는 conntrack에 의존한다.

문제는 conntrack 테이블이 가득 차면 새 연결이 드롭된다는 것. 고트래픽 K8s 클러스터에서 빈번하게 발생하는 장애 원인 중 하나.
핵심 sysctl
net.nf_conntrack_max — 최대 추적 연결 수 (기본 65536, 부족할 수 있음)
net.netfilter.nf_conntrack_tcp_timeout_established — TCP 연결 타임아웃 (기본 432000초 = 5일)
net.netfilter.nf_conntrack_tcp_timeout_time_wait — TIME_WAIT 타임아웃
K8s와 SNAT 문제
kube-proxy iptables SNAT: NodePort/ClusterIP 접근 시 소스 IP가 Pod CIDR로 변환됨
→ 원본 IP 소실. Traefik proxyProtocol.trustedIPs 등에서 문제 발생
해결: externalTrafficPolicy: Local로 SNAT 우회
또는 Proxy Protocol 활성화로 원본 IP 전달
모니터링
conntrack -C — 현재 추적 수
conntrack -S — 통계 (drop, insert_failed 확인)
cat /proc/sys/net/nf_conntrack_max — 최대값
Prometheus: node_nf_conntrack_entries / node_nf_conntrack_entries_limit
알림 설정: 사용률 80% 이상이면 경고
Tuning

Socket Buffer 튜닝

소켓 송수신 버퍼 크기가 throughput과 latency에 직접 영향
TCP 소켓은 송신 버퍼(wmem)와 수신 버퍼(rmem)를 가진다. 이 버퍼가 작으면 bandwidth-delay product를 채울 수 없어 throughput이 제한된다. 버퍼가 너무 크면 메모리를 과도하게 사용하고, 버퍼블로트(bufferbloat)가 발생할 수 있다.

net.core.wmem_max212992소켓 송신 버퍼 최대값16777216 (16MB)
net.ipv4.tcp_rmem4096 131072 6291456TCP 수신 버퍼 min/default/max4096 87380 16777216

DNS 내부 동작

DNS 조회 과정과 K8s CoreDNS에서의 문제점
DNS는 모든 서비스 통신의 첫 단계다. DNS 지연 = 모든 요청의 latency 증가. K8s 환경에서는 CoreDNS가 클러스터 DNS를 담당하며, ndots:5 기본 설정이 불필요한 DNS 쿼리를 5배로 증폭시키는 주요 성능 이슈다.
K8s DNS 해석 순서
my-svc 조회 시 ndots:5에 의해:
1. my-svc.ns.svc.cluster.local ← 대부분 여기서 해결
2. my-svc.svc.cluster.local
3. my-svc.cluster.local
4. my-svc.<search domain>
5. my-svc. (절대 도메인)
외부 도메인도 이 과정을 모두 거침 → 5x 쿼리 증폭
최적화
FQDN에 점(.) 붙이기: api.external.com. → search 우회
ndots:1 설정: Pod spec의 dnsConfig로 변경
NodeLocal DNSCache: 각 노드에 DNS 캐시 DaemonSet
DNS 캐시 TTL: CoreDNS 설정에서 cache 30 등 추가
A+AAAA 동시 쿼리: IPv6 불필요 시 single-request-reopen
Load Balancing

L4/L7 로드밸런싱 & kube-proxy

K8s Service의 네트워크 구현체와 각 모드의 트레이드오프
K8s Service는 kube-proxy가 각 노드에서 iptables/IPVS 규칙을 생성하여 구현한다. 모드에 따라 성능, 기능, 운영 복잡도가 크게 다르다.
iptables 모드
기본 모드. Service당 iptables 규칙 생성
장점: 단순, 안정적, 추가 컴포넌트 불필요
단점: O(n) 규칙 매칭 — Service 수가 많아지면 latency 증가
규칙 수 만 개 이상 시 iptables-restore 지연
연결 기반 분산이 아닌 random 분산
IPVS 모드
L4 로드밸런서를 커널에 내장
장점: O(1) 해시 기반 매칭, 다양한 분산 알고리즘 (rr, lc, sh 등)
conntrack 의존은 동일하지만 규칙 성능이 월등
단점: IPVS 커널 모듈 필요, 디버깅이 iptables보다 어려움
ipvsadm -Ln으로 규칙 확인
eBPF/Cilium 모드 (차세대)
iptables/conntrack을 완전히 우회하고 eBPF 프로그램이 패킷을 직접 처리. kube-proxy replacement 가능.
장점: conntrack 테이블 병목 제거, 낮은 latency, Socket-level LB (커널 네트워크 스택 단축)
단점: 최신 커널(5.10+) 필요, 학습 곡선, 생태계 성숙도
Cilium의 bpf.masquerade=true + kubeProxyReplacement=strict

Resource Control

cgroups v2

프로세스 그룹의 리소스(CPU, 메모리, I/O) 사용량을 제한하고 측정하는 커널 기능
cgroups는 K8s의 resource requests/limits가 실제로 구현되는 메커니즘이다. v2는 통합된 계층 구조를 제공하여 v1의 컨트롤러별 분리 문제를 해결했다. K8s 1.25+에서는 cgroups v2가 기본에 가깝다.
CPU 제어
cpu.max — "quota period" 형식. 100000 100000 = 1코어 100%
cpu.weight — CFS 가중치 (1~10000, 기본 100)
K8s CPU limit 100m = cpu.max "10000 100000"
K8s CPU request → cpu.weight로 변환
cpu.stat — usage_usec, nr_throttled, throttled_usec
Memory 제어
memory.max — 하드 리밋. 초과 시 OOM 발생 가능
memory.high — 소프트 리밋. 초과 시 throttle (reclaim 강제)
memory.current — 현재 사용량 (Page Cache 포함!)
memory.stat — anon, file, slab 등 상세 분류
K8s memory limit → memory.max로 설정
I/O 제어
io.max — 디바이스별 BPS/IOPS 제한. "MAJ:MIN rbps=X wbps=Y riops=Z wiops=W"
io.weight — I/O 우선순위 가중치
io.stat — 디바이스별 I/O 통계
K8s는 I/O limit을 직접 설정하지 않지만, cgroups v2에서 직접 제어 가능

Networking

CNI & Pod 네트워킹

Container Network Interface — K8s Pod에게 네트워크를 할당하는 표준 인터페이스
K8s의 네트워킹 모델: 모든 Pod는 고유한 IP를 가지며, NAT 없이 서로 통신할 수 있어야 한다. 이를 구현하는 것이 CNI 플러그인이다.
Pod 네트워크 구조
veth pair: 호스트↔Pod 연결. 한쪽은 Pod net ns 안, 한쪽은 호스트 브릿지에 연결
pause 컨테이너: net ns를 유지하는 역할. 같은 Pod의 컨테이너가 이 ns를 공유
브릿지: 같은 노드 내 Pod 간 L2 통신
CNI 플러그인 비교
Calico: BGP 기반 L3 라우팅. NetworkPolicy 지원. 가장 범용
Cilium: eBPF 기반. 고성능, 풍부한 정책, L7 관찰. 차세대
Flannel: VXLAN overlay. 단순하지만 기능 제한적
AWS VPC CNI: VPC IP를 Pod에 직접 할당. EKS 기본
패킷 경로: Pod A → Pod B (다른 노드)
Pod A eth0 → veth → 호스트 브릿지/라우팅 → [VXLAN encap / BGP route] → 물리 NIC → 네트워크 → 대상 노드 NIC → [VXLAN decap / route lookup] → veth → Pod B eth0
Overlay(VXLAN)는 캡슐화 오버헤드가 있음. 네이티브 라우팅(BGP)은 오버헤드 없지만 네트워크 인프라 지원 필요.
Advanced

eBPF — Extended Berkeley Packet Filter

커널 내에서 안전하게 프로그램을 실행하는 기술. 네트워킹, 보안, 관측의 게임 체인저
eBPF는 커널을 수정하지 않고 커널 내부에서 코드를 실행할 수 있게 하는 기술이다. 네트워킹(Cilium), 보안(Falco), 관측(bpftrace, Pixie) 등 SRE의 핵심 영역을 혁신하고 있다.

SRE에게 eBPF가 중요한 이유: 기존에는 커널 수준 성능 분석을 위해 커널 모듈을 작성하거나 perf/ftrace를 사용해야 했다. eBPF는 안전하고 동적으로 커널 이벤트를 관찰하고 조작할 수 있다.
활용 영역
Networking: XDP, TC hook으로 패킷 고속 처리. kube-proxy 대체 (Cilium)

Brendan Gregg의 USE Method와 도구 매핑

💡 60초 체크리스트
Brendan Gregg의 "Linux Performance Analysis in 60 Seconds":
uptime → dmesg -T | tail → vmstat 1 → mpstat -P ALL 1 → pidstat 1 → iostat -xz 1 → free -h → sar -n DEV 1 → sar -n TCP 1 → top
이 10개 명령으로 CPU, 메모리, I/O, 네트워크 상태를 빠르게 파악할 수 있다.
